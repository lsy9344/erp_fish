## 3. 핵심 조사 결과

### 3.1 재고 엑셀 단가

- 엑셀의 `재고 단가`는 정상적으로 파싱되고 `InventoryOpeningSnapshot.unitPrice`에 저장된다.
- 문제는 저장이 아니라 **지점장 표시용 DTO 누락**이다.
- 현재 작은 단가 문구는 실제 매입 이력의 `purchasePrice`만 사용한다.
- 본사 데이터에는 월초 단가 fallback이 있지만 지점장 응답에서는 내부 `unitPrice`가 제거되므로 표시되지 않는다.
- DB나 엑셀 파서를 다시 만들지 말고, 승인된 표시용 중첩 DTO에 월초 단가 출처를 연결한다.

관련 파일:

- `src/features/inventory/opening-import.ts`
- `src/features/inventory/opening-import-actions.ts`
- `src/features/inventory/queries.ts`
- `src/features/inventory/types.ts`
- `src/features/inventory/response-shaping.ts`
- `src/features/inventory/components/inventory-step-client.tsx`

### 3.2 판매계획가

- 재고 단계 입력, 필수 검증, 저장, 기존 손실액 재동기화는 이미 구현돼 있다.
- 중복 구현하지 않는다.
- 사용자에게 보이는 용어만 `판매한 가격`으로 통일한다.
- 호환성을 위해 이번 작업에서는 DB의 `StoreSalesPricePlan`, `plannedUnitPrice` 이름을 변경하지 않는다.
- 품목별 실제 POS 데이터가 없으므로 관련 매출 문구의 `추정` 표시는 유지한다.

### 3.3 손실 단계

- 현재 단계 순서는 `매입 → 손실 → 재고 → 지출 → 근무 → 매출 → 검토`다.
- 재고 저장은 `lossReviewedAt`을 선행 조건으로 요구한다.
- 기준재고와 당일 판매량 계산도 손실 수량을 사용한다.
- 따라서 이번 작업에서는 손실 단계를 재고 뒤로 이동하지 않는다.
- 손실 화면 안내만 “재고 단계에서 판매한 가격을 저장하면 손실액이 자동 확정된다”는 의미로 보완한다.

### 3.4 이월 매출

- 현재 DB, DTO, UI, 저장 액션, 계산기에 이월 매출 필드가 없다.
- `DailyLedger`에는 총매출, 현금, 카드, 기타 결제만 있다.
- 총매출은 매출이익, 실제 마진율, 영업이익, 생산성, 매출차액, 관제, 마감 점검, 기간·월간 리포트에 재사용된다.
- 입력칸만 추가하면 안 되며 **마감 정산 매출**과 **영업 성과 매출**을 분리해야 한다.

권장 계약 A:

```text
장부 마감 매출 = totalSalesAmount
이월 매출 = carryoverSalesAmount
영업 매출 합계 = totalSalesAmount + carryoverSalesAmount
마감 정산 차액 = totalSalesAmount - 현금 - 카드 - 기타 - 지출
```

단, 구현 전에 실제 2영업일 예시로 다음을 확인해야 한다.

1. 총매출이 장부 마감 뒤 매출을 포함하는가?
2. 전일 이월 매출이 다음 날 총매출에 포함되는가?
3. 이월 매출의 결제금액이 당일 현금·카드·기타에 포함되는가?

전일 이월이 다음 날 총매출에 포함된다면 계약 B를 사용한다.

```text
영업 매출 합계(D)
= totalSalesAmount(D)
- carryoverSalesAmount(D-1)
+ carryoverSalesAmount(D)
```

계약 A/B 확정 전에는 이월 매출 리포트 계산을 구현하지 않는다.

### 3.5 재고 저장 UX

- 현재 성공 메시지는 일반적인 `저장됐습니다.`뿐이다.
- 현재 `수정됨` 배지는 이번 저장 여부가 아니라 전일재고와 당일재고가 다른지를 뜻한다.
- 저장 직전 draft와 서버 기준값을 비교해 이번 저장에서 변경된 행 목록을 보존한다.
- 성공 후 `방금 저장한 재고 N건`과 `품목명 기존값 → 저장값`을 표시한다.
- 해당 행에 `방금 저장됨` 상태를 표시하고, 사용자가 다시 편집하면 그 행의 상태만 제거한다.
- 변경 없는 저장은 변경 저장과 구분한다.

### 3.6 Enter 포커스

- 현재 당일재고 input에 Enter 처리기가 없어 form submit으로 이어질 수 있다.
- Enter 시 `preventDefault()` 후 현재 탭·정렬 기준 다음 행의 당일재고 input으로 이동한다.
- 페이지 경계를 넘으면 다음 페이지를 먼저 연 뒤 포커스한다.
- 가격·사유 입력칸은 건너뛴다.
- IME 조합 중인 Enter는 처리하지 않는다.
- 마지막 행에서는 저장하지 않고 현재 포커스를 유지하며 완료 안내만 제공한다.

## 5. 작업 시 주의사항

- 기존 `InventoryOpeningSnapshot.unitPrice`를 재사용하고 새 단가 저장 구조를 만들지 않는다.
- 지점장 DTO에 FIFO 원가, 재고금액 등 민감한 내부 금액을 함께 노출하지 않는다.
- 실제 매입단가가 있으면 월초 재고단가보다 우선한다.
- 판매한 가격 명칭 변경 때문에 내부 DB 컬럼 rename 마이그레이션을 만들지 않는다.
- 손실 단계 순서를 이번 작업에서 변경하지 않는다.
- 이월 매출을 기존 마감 정산 차액에 무조건 더하지 않는다.
- 이월 매출 변경 시 본사 수정, 정정 overlay, 감사로그, 리포트 export 누락 여부를 확인한다.
- Enter 이동은 저장, version 증가, 감사로그 생성을 유발하면 안 된다.
- 범위 밖 리팩터링이나 새 의존성 추가를 피한다.
