## 목적

기존 FIFO 정책 문서에는 OQ-7이 "일부 품목 적용"으로 남아 있다. 실제 운영 방식은 매일 아침 본사가 이카운트 엑셀처럼 지점에 출고하고, 지점장은 그 물량을 현장에서 판매 예정가로 판매한 뒤, 영업 종료 후 장부에 현재 재고와 판매 예정가를 입력하는 흐름이다.

따라서 FIFO 정책은 특정 품목만 고르는 방식이 아니라 모든 재고품목을 대상으로 해야 한다. 다만 품목 매핑, 단가, 수량, 입고 근거가 빠진 행은 FIFO 대상에서 제외하는 것이 아니라 `기준 확인 필요`, `데이터 부족`, `계산 불가` 상태로 남긴다.

## 핵심 결정

### OQ-7. FIFO 적용 범위

FIFO 적용 범위는 모든 재고품목이다.

| 항목 | 결정 |
| --- | --- |
| 적용 대상 | 모든 재고품목 |
| 품목별 예외 | 정책상 예외 품목을 두지 않는다. |
| 계산 불가 row | FIFO 제외가 아니라 상태값으로 분리한다. |
| 상태값 | `기준 확인 필요`, `데이터 부족`, `검토 필요`, `재검증 필요`, `계산 불가` |
| 지점장 노출 | 원가, FIFO 재고금액, lot 근거는 기존 민감 필드 정책에 따라 차단한다. |

기존 문서의 "승인된 정규 품목 중 lot 근거가 완전한 품목에만 적용" 표현은 바꾼다. 새 표현은 "모든 재고품목이 FIFO 계산 대상이며, 근거가 불완전한 행은 확정 계산 대신 상태값으로 반환한다"로 통일한다.

### OQ-17. FIFO 처리 순서

사용자 운영 방식 기준으로 FIFO 처리 순서를 아래처럼 고정한다.

| 순서 | 이벤트 | 의미 |
| --- | --- | --- |
| 0 | 전일 확정 이월 lot 로드 | 전날 마감 후 남은 재고를 오늘 시작 재고로 가져온다. |
| 1 | 당일 본사 출고/지점 입고 lot 생성 | 아침 이카운트 출고분은 당일 판매 전에 입고된 것으로 본다. |
| 2 | 당일 승인된 추가 입고 또는 조정 증가 반영 | 본사 승인 또는 명확한 근거가 있는 증가분만 새 lot으로 더한다. |
| 3 | 반품/void 입고 취소 반영 | 당일 입고 자체가 취소된 경우 판매 차감 전에 lot에서 제거한다. |
| 4 | 판매 수량 역산 | 사용 가능 수량에서 마감 재고, 손실, 폐기, 떨이, 조정 감소를 뺀 값을 판매 수량으로 본다. |
| 5 | 판매 수량 FIFO 차감 | 오래된 lot부터 판매된 것으로 보고 판매원가 후보를 계산한다. |
| 6 | 손실/폐기/떨이 FIFO 차감 | 판매 후 남은 오래된 lot부터 비정상 차감 수량을 뺀다. |
| 7 | 조정 감소 반영 | 승인된 재고 감소 조정을 마지막 차감으로 반영한다. |
| 8 | 마감 snapshot 생성 | 남은 lot 잔량과 재고금액 후보를 다음 날 이월 기준으로 저장한다. |

판매 수량 역산식:

```text
판매 수량 =
  전일 이월 수량
  + 당일 본사 출고/지점 입고 수량
  + 승인된 조정 증가 수량
  - 마감 현재 재고 수량
  - 손실/폐기/떨이 수량
  - 승인된 조정 감소 수량
```

판매 예정가와 FIFO 원가는 분리한다.

- 판매 예정가: 지점장이 칠판이나 장부에 적는 판매 가격이다. 예: 고등어 1,300원
- FIFO 원가: 본사 출고/입고 단가나 이월 lot 단가다. 예: 고등어 원가 1,000원
- 매출 후보: 판매 수량 x 판매 예정가
- 판매원가 후보: FIFO로 차감된 lot 원가 합계
- 재고금액 후보: 마감 후 남은 lot 원가 합계

## 예시 시뮬레이션

### 예시 1. 기본 판매

아침 기준:

| 구분 | 수량 | 원가 |
| --- | ---: | ---: |
| 전일 이월 고등어 | 30개 | 900원 |
| 당일 본사 출고/지점 입고 고등어 | 70개 | 1,000원 |

지점장 입력:

| 입력 | 값 |
| --- | ---: |
| 판매 예정가 | 1,300원 |
| 마감 현재 재고 | 15개 |
| 폐기 | 5개 |

판매 수량:

```text
30 + 70 - 15 - 5 = 80개
```

FIFO 판매원가:

```text
전일 이월 30개 x 900원 = 27,000원
당일 입고 50개 x 1,000원 = 50,000원
판매원가 후보 = 77,000원
```

매출 후보:

```text
80개 x 1,300원 = 104,000원
```

마감 후 재고:

```text
당일 입고 lot 15개 x 1,000원 = 15,000원
```

폐기 5개는 판매 차감 후 남은 lot에서 차감한다. 이 예시에서는 당일 입고 lot 5개가 폐기 원가 후보가 된다.

### 예시 2. 같은 날 단가가 다른 입고가 있는 경우

아침 기준:

| 구분 | 수량 | 원가 |
| --- | ---: | ---: |
| 전일 이월 | 10개 | 900원 |
| 본사 출고 A | 40개 | 1,000원 |
| 본사 출고 B | 50개 | 1,100원 |

지점장 입력:

| 입력 | 값 |
| --- | ---: |
| 판매 예정가 | 1,300원 |
| 마감 현재 재고 | 20개 |
| 손실/폐기/떨이 | 0개 |

판매 수량:

```text
10 + 40 + 50 - 20 = 80개
```

FIFO 판매원가:

```text
전일 이월 10개 x 900원 = 9,000원
본사 출고 A 40개 x 1,000원 = 40,000원
본사 출고 B 30개 x 1,100원 = 33,000원
판매원가 후보 = 82,000원
```

마감 후 재고:

```text
본사 출고 B 20개 x 1,100원 = 22,000원
```

## 구현 지시

### Task 1. 정책 문서의 OQ-7 범위 수정

**문제**

현재 정책 문서에는 FIFO가 일부 품목에만 적용된다고 되어 있다. 사용자 결정은 모든 재고품목 적용이다.

**작업**

- `OQ-7 적용 범위`를 모든 재고품목으로 바꾼다.
- `일부 적용`, `승인된 정규 품목 중 lot 근거가 완전한 품목에만 적용` 같은 표현을 제거하거나 교정한다.
- 근거가 부족한 품목은 "적용 제외"가 아니라 "확정 계산 불가 상태"로 분리한다.
- 상태 분류는 다음과 같이 유지한다.
  - `기준 확인 필요`
  - `데이터 부족`
  - `검토 필요`
  - `재검증 필요`
  - `계산 불가`

**완료 기준**

- 문서에서 OQ-7이 모든 재고품목 적용으로 읽힌다.
- 단가/수량/lot 근거 부족은 정책 제외가 아니라 계산 상태로 표현된다.

### Task 2. OQ-17 처리 순서를 운영 흐름 기준으로 수정

**문제**

기존 OQ-17 순서는 일반적인 lot 이벤트 순서에 가깝다. 실제 의뢰자 운영은 "본사 아침 출고 -> 지점 판매 -> 마감 재고 입력" 흐름이다.

**작업**

- 본사 아침 이카운트 출고분을 당일 판매 전 입고 lot으로 처리한다.
- 판매 수량은 지점장이 매번 판매 행을 입력하는 방식이 아니라 마감 입력값으로 역산한다.
- 판매 예정가는 매출 후보 계산에만 사용하고, FIFO 원가 계산에는 사용하지 않는다.
- FIFO 차감 순서는 다음으로 고정한다.
  1. 전일 확정 이월 lot
  2. 당일 본사 출고/지점 입고 lot
  3. 승인된 조정 증가 lot
  4. 반품/void 입고 취소
  5. 판매 수량 FIFO 차감
  6. 손실/폐기/떨이 FIFO 차감
  7. 승인된 조정 감소
  8. 마감 snapshot

**완료 기준**

- 문서와 테스트 설명이 "아침 출고는 당일 판매 전 입고"라고 명시한다.
- 판매 수량 역산식이 문서화되어 있다.
- 판매 예정가와 FIFO 원가가 섞이지 않는다.

### Task 3. FIFO 계산 테스트 시나리오 추가

**작업**

- 모든 재고품목이 FIFO 계산 후보가 되는 테스트를 추가한다.
- 단가가 다른 이월 lot과 당일 본사 출고 lot이 있을 때 오래된 lot부터 차감되는지 검증한다.
- 마감 재고와 폐기 수량으로 판매 수량을 역산하는 테스트를 추가한다.
- 판매 예정가가 FIFO 원가 계산에 쓰이지 않는지 검증한다.
- 지점장 응답에는 원가, 재고금액, lot 근거가 노출되지 않는지 기존 민감 필드 테스트를 유지 또는 보강한다.

**권장 테스트 케이스**

```text
전일 이월 30개 @ 900원
당일 본사 출고 70개 @ 1,000원
판매 예정가 1,300원
마감 재고 15개
폐기 5개

판매 수량 = 80개
판매원가 후보 = 77,000원
마감 재고금액 후보 = 15,000원
폐기 원가 후보 = 5,000원
```

**완료 기준**

- 위 시나리오가 자동 테스트로 고정된다.
- 판매 예정가를 바꿔도 FIFO 원가와 재고금액은 바뀌지 않는다.

### Task 4. Policy gate 해제 조건 재정리

**작업**

  - 지점장 민감 필드 차단 테스트가 통과함
  - 기존 재고 저장, 매입 저장, 손실 저장 후 FIFO refresh 동작이 통과함

**완료 기준**

- 정책 문서만 바뀐 상태에서는 여전히 `기준 확인 필요`가 유지된다.
- 구현 story 완료 후에만 FIFO 확정 재고금액/판매원가를 `ok`로 승격할 수 있다.

```text
재고 금액은 본사 기준 확인 후 확정됩니다.
```

- 본사 또는 관리자 상세 화면에는 내부 추적용으로 OQ id를 남길 수 있다.

**완료 기준**

- 지점장은 내부 정책 코드명을 몰라도 현재 상태를 이해할 수 있다.
- 본사/개발자는 OQ-7/OQ-17 추적이 가능하다.

## 승인 기준

이 작업은 다음 조건을 만족하면 완료로 본다.

- OQ-7 정책 문서가 모든 재고품목 적용으로 바뀐다.
- OQ-17 정책 문서가 실제 운영 흐름인 아침 출고, 지점 판매, 마감 재고 입력 기준으로 바뀐다.
- 판매 예정가와 FIFO 원가가 문서와 테스트에서 분리된다.
- FIFO 계산 엔진이 아직 미완료라면 사용자 화면에는 확정 금액 대신 쉬운 `기준 확인 필요` 문구가 표시된다.
- 지점장 민감 필드 차단 정책은 약화되지 않는다.
