# 과거재고 이월 보존 및 복구 설계

## 배경

과거재고 엑셀의 `2026-07-10` 재고는 다음 영업일인 `2026-07-11` 장부의 전일재고로 사용한다. 업로드 자체는 성공했고 현재 DB에도 `2026-07` 월초 스냅샷 71건이 남아 있다. 그러나 지점이 재고 단계를 처음 저장할 때 변경하지 않은 이월 행이 저장 대상에서 빠지며, 이후 조회는 저장된 장부 행만 반환해 빠진 스냅샷을 다시 보여주지 않는다.

감사 로그에서 최초 저장 전후의 월초 이월 행 수가 다음과 같이 줄었다.

- 제일수산: 25 → 20
- 삼국유통: 25 → 10
- 강서수산: 21 → 17

## 목표

1. 과거재고 또는 전일 장부라는 근거가 있는 이월 행은 사용자가 값을 바꾸지 않아도 장부 저장 시 보존한다.
2. `10일 마감재고 → 11일 장부 전일재고` 의미를 유지한다.
3. 기존 7월 11일 장부에서 누락되거나 수동 행으로 다시 만들어진 과거재고 기준을 복구하되, 사용자가 입력한 당일재고와 이후 업무 입력은 보존한다.
4. 이미 장부 재고가 저장된 뒤 과거재고 파일을 다시 올렸을 때 기존 장부가 자동으로 덮어써지지 않는다는 사실을 업로드 결과에 명확히 표시한다.
5. 회귀 테스트, 정적 검사, 빌드, 배포 검증을 통과한다.

## 비목표

- 과거 날짜 장부를 업로드 과정에서 자동 생성하지 않는다.
- 월 단위 `InventoryOpeningSnapshot` 모델을 날짜별 이력 모델로 전면 교체하지 않는다.
- 마감된 장부나 사용자가 입력한 당일재고를 업로드 값으로 자동 덮어쓰지 않는다.
- 이번 문제와 관계없는 재고 계산식이나 화면 구성을 바꾸지 않는다.

## 검토한 접근

### 1. 근거 있는 이월 행 보존 + 제한된 데이터 복구 (채택)

저장 정책이 월초 스냅샷 또는 전일 장부에서 온 seed 행을 실제 장부 행으로 보존하도록 한다. 현재 누락 데이터는 감사 로그의 최초 저장 직전 값과 현재 스냅샷을 교차검증해 한 번만 복구한다. 기존 사용자 입력을 건드리지 않으면서 원인을 한 곳에서 막는 가장 작은 변경이다.

### 2. 조회할 때마다 월 스냅샷 강제 병합 (미채택)

구현은 간단하지만 월 중간에 이미 판매되었거나 0으로 소진된 품목이 다시 살아날 수 있다. 장부가 권위 있는 상태가 된 이후에도 월초 값을 계속 주입하므로 데이터 의미가 깨진다.

### 3. 날짜별 스냅샷 모델 도입 (미채택)

장기적으로 가장 표현력이 높지만 스키마 마이그레이션, 과거 데이터 보정, 조회 우선순위 재설계가 필요하다. 이번 결함을 해결하는 데 필요한 범위를 크게 넘는다.

## 설계

### 저장 정책

`shouldPersistInventoryLine`은 기존 DB 행, 사용자가 명시 입력한 매입·손실 seed 행뿐 아니라 근거가 있는 이월 seed 행도 저장 대상으로 판정한다. 근거 있음은 `carryoverSource`가 `OPENING_SNAPSHOT`, `PREVIOUS_CLOSED_LEDGER`, `PREVIOUS_SAVED_LEDGER` 중 하나인 경우다. 근거가 없는 `MANUAL` seed 행은 기존처럼 사용자가 값을 입력한 경우에만 저장한다.

지점 저장과 본사 저장은 같은 정책 함수를 계속 사용한다. 두 경로가 서로 다른 규칙을 갖지 않도록 호출부에서 동일한 이월 출처 정보를 넘긴다.

### 업로드 결과 안내

업로드는 스냅샷 upsert까지만 수행한다. 동일 대상의 다음 날 장부에 재고 행이 이미 존재한다면 그 장부를 자동 갱신하지 않는다. 대신 결과에 기존 장부 존재 여부와 “스냅샷은 갱신됐지만 작성된 장부는 덮어쓰지 않음” 안내를 포함한다. 이를 통해 성공 메시지가 장부 자동 반영까지 의미하는 오해를 막는다.

### 현재 데이터 복구

복구는 `2026-07-11` 장부만 대상으로 하는 명시적이고 재실행 가능한 작업으로 수행한다.

1. 각 장부의 첫 `ledger.inventory.saved` 감사 로그에서 저장 직전 `OPENING_SNAPSHOT` 행을 읽는다.
2. 현재 `InventoryOpeningSnapshot`의 지점·품목·월과 일치하는지 확인한다.
3. 현재 장부에 없는 행은 감사 로그의 원래 기준값으로 생성한다.
4. 현재 장부에 수동 행으로 다시 존재하는 품목은 당일재고·수량·사용자 입력을 유지하고 전일재고와 이월 출처만 원래 스냅샷 기준으로 복구한다.
5. 이월 상세, 재고금액, FIFO 및 조정 상태를 현재 장부 데이터에 맞게 다시 계산한다.
6. 변경 전후와 복구 건수를 감사 로그에 남긴다.

복구 전에는 dry-run으로 지점별 생성·수정·건너뜀 건수를 출력한다. 실제 실행은 원격 DB 대상 확인과 명시적 확인 플래그가 모두 있어야 하며 하나의 트랜잭션으로 처리한다. 마감 장부 발견, 스냅샷 불일치, 감사 근거 누락 시 전체 실행을 중단한다.

## 오류 처리

- 업로드 파싱 또는 매핑 실패는 기존처럼 전체 트랜잭션을 취소한다.
- 업로드 후 기존 장부를 발견해도 업로드 자체는 성공으로 유지하되 자동 미반영 안내를 함께 반환한다.
- 복구는 예상 장부·스냅샷·감사 건수가 맞지 않으면 쓰기 전에 실패한다.
- 복구 후 대상 이외 테이블 및 사용자 입력 필드가 바뀌면 트랜잭션을 롤백한다.

## 테스트

1. 순수 정책 테스트: 변경하지 않은 `OPENING_SNAPSHOT` 및 전일 이월 seed 행이 저장 대상으로 판정된다.
2. 정책 회귀 테스트: 근거 없는 `MANUAL` seed 행은 계속 저장되지 않는다.
3. 저장 경로 테스트: 지점·본사 저장 호출부가 이월 출처를 정책 함수에 전달한다.
4. 업로드 결과 테스트: 기존 다음 날 장부가 있는 경우 자동 미반영 안내가 반환·표시된다.
5. 통합 시나리오: 과거재고 업로드 → 일부 품목만 당일재고 입력 → 저장 → 재조회 후 모든 과거재고 행이 유지된다.
6. 복구 dry-run 테스트: 생성·수정·건너뜀 분류가 결정적이며 재실행 시 추가 변경이 없다.

현재 `main` 기준 단위 테스트는 479개 중 4개가 기존부터 실패한다. 재고 변경 검증과 별도로 원인을 확인하되, 최종 배포 전 전체 검사·빌드·필수 테스트가 통과하도록 필요한 배포 차단 오류를 해결한다.

## 배포 및 성공 기준

- 최신 `main` 기반 `codex/fix-opening-inventory-carryover` 브랜치에서 작업한다.
- 변경 검증 후 의도한 파일만 커밋하고 원격에 푸시한다.
- 프로젝트의 운영 배포 절차로 배포하며 마이그레이션이 없음을 확인한다.
- 배포 후 업로드 화면, 지점 재고 화면, DB 행 수와 감사 로그를 점검한다.
- 배포 실패 시 로그의 첫 원인을 해결하고 동일 검증을 반복한다.

성공은 다음 조건을 모두 만족할 때다.

- 업로드된 71개 월초 스냅샷이 유지된다.
- 7월 11일 장부의 누락된 월초 이월 근거가 복구된다.
- 사용자가 입력한 당일재고와 장부 상태는 유지된다.
- 이후 저장·새로고침에서도 근거 있는 이월 행이 사라지지 않는다.
- 업로드 결과가 기존 장부 자동 미반영 여부를 정확히 알려준다.
