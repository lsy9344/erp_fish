# 재고 흐름 수량 소수점 첫째 자리 입력 설계

## 배경

현재 시스템은 재고·매입·손실 수량을 소수점 둘째 자리까지 허용한다. 지점/본사 재고 입력, 이카운트 매입 업로드, 재고 시작 스냅샷 업로드와 서버 스키마가 이 규칙을 공유하지만, 실제 배포용 과거 재고 엑셀 양식은 아직 수량을 정수로만 제한한다. 또한 본사 마감 후 정정 경로는 수량을 정수로만 처리해 일반 입력 경로와 일치하지 않는다.

사용자는 재고뿐 아니라 매입·손실을 포함해 재고 흐름에 들어가는 모든 신규 수량 입력을 소수점 첫째 자리까지만 허용하도록 확정했다.

## 목표

- 재고·매입·손실 수량 입력에서 정수와 소수점 첫째 자리 값을 허용한다.
- 소수점 둘째 자리 이상인 신규 입력은 화면, 서버 액션, 엑셀 업로드, 마감 후 정정에서 모두 거부한다.
- 실제 과거 재고 엑셀 양식과 생성 스크립트가 `0.1` 단위 입력을 안내하고 검증하게 한다.
- 기존에 저장된 소수점 둘째 자리 데이터는 반올림하거나 삭제하지 않는다.
- 금액, 단가, 근무인원처럼 정수 정책을 유지해야 하는 값에는 소수 허용이 번지지 않게 한다.

## 비목표

- 기존 `Decimal(12,2)` 데이터를 한 자리로 일괄 반올림하거나 정리하지 않는다.
- Prisma 컬럼 정밀도를 `Decimal(12,1)`로 축소하지 않는다.
- 수량 표시 전체를 항상 한 자리로 강제하지 않는다. 기존 두 자리 데이터는 그대로 조회할 수 있다.
- 재고와 무관한 금액·단가·근무인원 입력 정책을 바꾸지 않는다.
- 엑셀 양식의 디자인이나 시트 구조를 재설계하지 않는다.

## 선택한 설계

### 1. 입력 경계에서 한 자리 정밀도 적용

공용 수량 검증은 다음 계약을 따른다.

- 허용: `0`, `1`, `1.0`, `1.5`
- 거부: 음수, 쉼표가 포함된 문자열, 숫자가 아닌 값, `1.25`처럼 소수점 둘째 자리 이상인 값
- 최대값: 기존 열의 정수 자릿수를 유지하는 `9_999_999_999.9`

`src/lib/validation.ts`에 한 자리 수량 반올림과 검증을 한 곳에 둔다. 재고·매입·손실 서버 스키마와 다음 엑셀 파서는 이 공용 규칙을 사용한다.

- `src/features/inventory/opening-import.ts`
- `src/features/ledger/ecount-purchase-import.ts`
- `src/features/ledger/ecount-supply-import.ts`

FIFO와 기존 장부 계산의 내부 `roundToTwoDecimals` 처리는 유지한다. 이 계산 경로는 과거 두 자리 데이터도 정확히 읽고 계산해야 하며, 신규 입력 경계에서 이미 한 자리로 제한되므로 새 데이터가 둘째 자리로 들어오는 우회로가 되지 않는다.

### 2. 화면 입력과 오류 문구 통일

지점/본사 재고, 매입, 손실 화면은 현재 문자열 상태와 `inputMode="decimal"`을 사용한다. 각 화면의 클라이언트 파서가 소수 한 자리 정규식과 공용 최대값을 따르도록 바꾸고, 서버 오류 문구를 모두 “소수점 첫째 자리까지”로 맞춘다.

적용 대상은 다음과 같다.

- `src/features/inventory/components/inventory-step-client.tsx`
- `src/features/ledger/components/purchase-step-client.tsx`
- `src/features/losses/components/loss-step-client.tsx`
- `src/features/inventory/schemas.ts`
- `src/features/ledger/schemas.ts`
- `src/features/losses/schemas.ts`
- `src/features/losses/hq-edit-actions.ts`

클라이언트 검증은 빠른 피드백을 제공하고, 서버 검증은 직접 호출이나 조작된 요청도 같은 규칙으로 차단한다.

### 3. 마감 후 정정은 대상별로 검증

정정 값의 `kind="quantity"`는 재고·손실 수량뿐 아니라 `workerCount`에도 사용된다. 따라서 모든 `quantity`를 소수 허용으로 바꾸지 않고 대상과 필드에 따라 나눈다.

- `INVENTORY_ROW.currentQuantity`, `INVENTORY_ROW.quantity`: 소수점 첫째 자리까지
- `LOSS_ROW.quantity`: 소수점 첫째 자리까지
- `LEDGER_FIELD.workerCount`: 계속 정수만
- `money`: 계속 정수만

`src/features/corrections/schemas.ts`, `src/features/corrections/actions.ts`, `src/server/calculations/ledger.ts`에서 같은 대상별 계약을 적용한다. `src/features/corrections/components/correction-panel.tsx`는 재고·손실 수량 대상에 `inputMode="decimal"`을 사용한다.

매입 행 정정은 현재 정책대로 지원하지 않으므로 새 기능을 추가하지 않는다.

### 4. DB 정밀도는 유지

Prisma와 PostgreSQL의 실제 재고 흐름 수량 컬럼은 이미 `Decimal(12,2)`/`NUMERIC(12,2)`다. 이 저장 형식은 한 자리 값을 손실 없이 저장하며, 기존 두 자리 값도 보존한다.

컬럼을 한 자리로 축소하면 기존 `2.28` 같은 값이 반올림될 수 있으므로 마이그레이션을 만들지 않는다. 신규 값의 정책은 앱의 모든 쓰기 경계에서 강제한다.

### 5. 과거 재고 엑셀 양식

다음 추적 파일을 함께 수정한다.

- `outputs/inventory_import_template/build-simple-inventory-template.mjs`
- `outputs/inventory_import_template/build-inventory-template.mjs`
- `outputs/inventory_import_template/과거_재고_간단_입력_양식.xlsx`

수량 셀은 `#,##0.0` 형식으로 표시하고, 0 이상이며 `수량 × 10`이 정수인 값만 허용하는 사용자 지정 데이터 검증을 적용한다. 단가와 금액은 기존 정수 검증을 유지한다. 작성 안내는 수량은 소수점 첫째 자리까지, 단가는 정수라는 점을 분리해서 설명한다.

간단 양식에서는 `재고입력`의 남은 수량과 `입고별잔량_선택`의 남은 수량을 바꾼다. 전체 양식에서는 재고·입고·손실 흐름의 모든 수량 열을 같은 규칙으로 바꾸고 금액·단가 열은 건드리지 않는다.

완성된 `.xlsx`는 원본 스타일과 시트 구조를 유지한 채 값·수식·데이터 검증을 검사하고 모든 표시 시트를 렌더링해 육안 확인한다.

## 오류 처리

- 화면과 서버는 잘못된 수량을 해당 행의 필드 오류로 표시한다.
- 엑셀 업로드는 잘못된 수량이 있는 행 번호와 열 이름을 포함해 파일 전체를 거부한다.
- 마감 후 정정은 잘못된 값이면 정정 기록을 저장하지 않는다.
- 엑셀 자체 데이터 검증을 우회해도 업로드 서버가 다시 검증한다.

## 테스트 전략

TDD로 기존 둘째 자리 허용 사례를 먼저 새 계약에 맞는 실패 테스트로 바꾼다.

- 공용/화면/서버 스키마: `2.2`와 `1.5`는 통과하고 `2.28`은 실패한다.
- 재고 시작 업로드: 한 자리 수량과 합계는 유지하고 둘째 자리 수량은 행 오류로 거부한다.
- 이카운트 매입·공급 업로드: 한 자리 수량은 보존하고 둘째 자리 수량은 거부한다.
- 마감 후 정정: 재고·손실 `1.5`는 적용되고 `1.25`는 거부되며 근무인원 `1.5`는 계속 거부된다.
- E2E: 실제 화면에서 한 자리 수량을 저장·재조회하고 둘째 자리 입력 오류를 확인한다.
- 엑셀: 안내 문구, 수량 형식, 데이터 검증, 수식 오류 없음, 시각 렌더링을 확인한다.
- 회귀: 전체 단위 테스트, 타입 검사, 린트, 빌드와 관련 Playwright 테스트를 실행한다.
