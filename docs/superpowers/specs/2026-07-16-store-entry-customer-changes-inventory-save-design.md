# 지점장 장부 수정 및 재고 저장 오류 설계

## 배경

2026년 7월 16일 고객 수정 요청 6건과 재고 저장 오류 1건을 현재 장부 입력 구조에 맞춰 반영한다. 추가 요청으로 본사 리포트의 `냉동/생물 매출 (추정)` 차트 카드도 제거한다.

이 설계는 기존 DB와 본사 급여 데이터를 유지하면서 화면과 공통 재고 판정의 잘못된 부분만 고치는 최소 변경을 사용한다.

## 범위

- 장부 입력 흐름의 `비용` 사용자 문구를 `지출`로 변경
- 5단계 이름을 `근무인원/이름`으로 변경
- 지점장 근무 화면에서 급여 UI 제거 및 근무자 입력 유지
- 지점장 매입 화면의 상세/펼치기 박스 제거
- 지점장 재고 행에 당일 또는 최근 실제 매입 거래일의 1박스 단가 표시
- 상단 `전날 재고 보기` 버튼 강조
- 부족 방향의 정상 판매가 재고 차이 사유 검증에 막히는 오류 수정
- 본사 리포트 3곳의 `냉동/생물 매출 (추정)` 차트 카드 제거

## 범위 밖

- DB 스키마 또는 기존 급여 데이터 삭제
- 본사 급여 입력·조회·월간 집계 변경
- FIFO 원가, 기본 단가, 원본 이카운트 단가의 지점장 노출
- 최근 매입단가를 재고 테이블에 별도 저장하거나 과거 데이터를 백필하는 작업
- 하단 `전날 재고 보기` 버튼 추가
- 냉동/생물 분류나 리포트 계산식 변경

## 사용자 문구

장부 입력과 검토 흐름에서 사용자에게 보이는 `비용`을 `지출`로 통일한다.

- `4단계: 지출`
- `지출 항목`
- `지출 금액`
- `입력 중 지출 합계`
- `마지막 서버 저장 지출 합계`
- `지출 단계로 이동`
- `5단계: 근무인원/이름`

내부 식별자와 DB 이름인 `expense`, `expenses`, `EXPENSE_ITEM` 등은 변경하지 않는다.

## 화면 설계

### 지출 단계

단계 내비게이션, 저장 상태, 입력 화면, 검토 요약, 누락 안내에서 같은 `지출` 용어를 사용한다. 내부 저장 구조와 계산은 그대로 유지한다.

### 근무인원/이름 단계

지점장 화면은 근무인원, 직원 추가·연결·이름, 지각·조퇴·특이사항을 한 카드 안에서 보여준다.

지점장에게는 다음 급여 표현을 노출하지 않는다.

- `급여 / 인건비`
- 급여 금액
- 입력 중 급여 합계
- 마지막 서버 저장 급여 합계
- 급여 행 기준 참고 인원
- `급여 저장`

기존 근무 정보 저장과 근무자 행 저장 경계는 유지한다. 근무자 행 저장 버튼은 급여 용어가 없는 `근무자 저장`으로 표시한다. 본사 화면은 기존 급여 금액, 합계, 저장 기능을 유지한다.

### 매입 단계

지점장 화면에서 품목 아래의 `상세`, 참고 단가, 펼치기·접기 박스를 제거한다. 품목을 선택하면 기존처럼 품목 스냅샷 값이 내부에 채워지며 다음 핵심 입력은 유지한다.

- 품목
- 매입 단가
- 수량
- 오늘 팔 가격(예상)
- 매입금액

본사 보완 입력에 필요한 기존 데이터와 서버 검증은 삭제하지 않는다.

### 재고 단계

각 재고 행의 품목·수량 흐름 근처에 표시용 매입단가를 추가한다.

- 장부 날짜에 매입이 있으면 `당일 매입단가 · YYYY-MM-DD · 12,000원/1박스`
- 당일 매입이 없으면 `최근 매입단가 · YYYY-MM-DD · 12,000원/1박스`
- 과거를 포함해 매입 이력이 없으면 `매입 이력 없음`

날짜는 레코드 생성 시각이 아니라 해당 매입이 속한 장부의 실제 거래일을 사용한다.

상단 `전날 재고 보기` 버튼은 하나만 유지한다. 기본 배경, 테두리, 글자 굵기, 호버, 키보드 포커스를 주변 보조 버튼보다 분명하게 만든다.

### 본사 리포트

다음 3곳에서 `냉동/생물 매출 (추정)` 제목, 설명, 차트 카드를 제거한다.

- 일별 리포트
- 품목 검토
- 월간 리포트

다른 리포트 카드, 표, 필터, export와 냉동/생물 계산 데이터는 변경하지 않는다. 세 화면에서 제거한 뒤 차트 컴포넌트에 호출자가 남지 않으면 해당 컴포넌트도 삭제한다.

## 매입단가 데이터 흐름

재고 조회는 대상 지점, 품목, 장부 날짜를 기준으로 저장된 `LedgerPurchaseItem` 이력을 사용한다.

1. 장부 날짜와 같은 거래일의 매입을 먼저 찾는다.
2. 당일 매입이 없으면 장부 날짜보다 이전인 실제 매입 거래일 중 가장 최근 날짜를 찾는다.
3. 선택된 거래일에 같은 품목의 매입이 여러 건이면 `매입금액 합계 / 매입수량 합계`로 1박스 가중평균 단가를 계산한다.
4. 표시 단가는 원 단위 정수로 반올림한다.
5. 표시용 `{ kind, businessDate, unitPrice }`만 재고 DTO에 추가한다.

`kind`는 `TODAY` 또는 `RECENT`이며 매입 이력이 없으면 전체 값은 `null`이다. 이 필드는 고객이 승인한 당일·최근 실제 매입단가만을 위한 좁은 예외다. 기존 지점장 응답에서 차단하는 FIFO 단가, 재고 평가 단가, 원본 단가, 매입금액은 계속 제거한다.

## 재고 저장 정책

기준수량은 기존 계산식 `전일재고 + 당일매입 - 손실`을 사용한다.

- `당일재고 <= 기준수량`: 정상 판매로 보고 조정 사유 없이 저장
- `당일재고 > 기준수량`: 과재고로 보고 `고친 이유`가 있어야 저장
- 당일재고 미입력, 허용 범위 밖 수량, 손실 단계 미검토 등 다른 검증은 기존대로 유지

클라이언트 검증, 서버 저장 가드, 조정 생성·정리, 검토 신호가 같은 공통 판정을 사용한다. 소수 수량은 기존 정규화·정밀도 규칙을 거친 값으로 비교해 부동소수점 차이로 과재고가 되지 않게 한다.

과재고 사유가 들어오면 기존 조정 레코드 저장 흐름을 사용한다. 부족 방향에는 새 조정 레코드를 만들지 않으며, 이전의 잘못된 부족 조정 레코드는 재조정 시 정리한다.

## 오류와 입력값 유지

검증 오류가 발생하면 클라이언트의 재고 행 상태를 서버 응답으로 다시 만들지 않는다. 사용자가 입력한 모든 수량과 사유를 그대로 유지한다.

- 클라이언트 검증 오류: 첫 오류 행의 필요한 입력칸으로 이동
- 서버 필드 오류: 서버의 행 인덱스를 현재 품목에 연결해 첫 오류 칸으로 이동
- 과재고 사유 누락: 해당 행의 `고친 이유` 입력칸으로 이동
- 저장 충돌: 기존 충돌 안내를 유지하며 사용자가 명시적으로 최신값을 다시 불러오기 전에는 초안을 버리지 않음

## 테스트

### 단위 테스트

- 장부 입력·검토 용어가 `지출`, `근무인원/이름`으로 통일되는 계약
- 지점장 근무 화면의 급여 UI 미노출과 근무자 입력 유지
- 본사 근무 화면의 급여 기능 유지
- 당일 매입단가 계산
- 최근 실제 거래일 선택
- 같은 거래일 복수 매입의 수량 가중평균
- 매입 이력 없음
- 이월 판매, 당일 매입 판매, 손실 혼재 부족 방향 통과
- 실제 과재고 사유 요구
- 소수 수량 경계
- 본사 리포트 3곳에서 냉동/생물 추정 차트 렌더 제거

### E2E 테스트

- 지출·근무인원/이름 문구와 단계 이동
- 지점장 근무 화면에서 직원 추가·이름·특이사항 저장
- 본사 급여 화면 회귀
- 매입 상세/펼치기 박스 제거와 핵심 매입 저장
- 재고 행의 당일 단가, 최근 단가와 실제 거래일 표시
- 상단 전날 재고 버튼 하나만 존재하고 마우스·키보드로 동작
- 여러 품목 수량 입력 후 과재고 오류가 나도 모든 입력값 유지
- 첫 과재고 사유 입력칸 포커스
- 사유를 입력한 과재고 저장과 정상 판매 저장
- 본사 일별·품목 검토·월간 리포트에서 냉동/생물 추정 차트 미노출

## 완료 기준

- 요청된 화면 문구와 제거 대상만 바뀌고 기존 장부 데이터는 보존된다.
- 지점장에게 급여 금액은 보이지 않지만 근무자 정보는 계속 저장된다.
- 재고 행에서 단가의 종류, 실제 거래일, 1박스 단가를 구분할 수 있다.
- 부족 방향 정상 판매는 저장되고 실제 과재고는 사유 없이 저장되지 않는다.
- 어떤 검증 오류에서도 사용자가 입력한 재고 값이 유지된다.
- `전날 재고 보기`는 상단 강조 버튼 하나만 남는다.
- 본사 리포트 3곳에서 `냉동/생물 매출 (추정)` 차트 카드가 보이지 않는다.
