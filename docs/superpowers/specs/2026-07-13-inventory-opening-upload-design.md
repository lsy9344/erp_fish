# 재고 엑셀 자동 품목 생성과 안전 반영 설계

## 배경

과거 재고 엑셀 업로드는 현재 활성 품목과 정확히 일치하는 행만 허용한다. 품목이 없으면 파일 전체가 취소되며, 업로드가 성공해도 `InventoryOpeningSnapshot`만 저장되고 기존 장부는 직접 바뀌지 않는다. 현재 작업 브랜치는 `main`에 이미 포함된 월초 이월 행 보존과 기존 장부 안내 수정도 포함하지 않는다.

## 목표

- 재고 엑셀에 처음 등장한 품목을 업로드 트랜잭션 안에서 자동 생성한다.
- 기존 장부 재고를 자동으로 덮어쓰지 않는다.
- 월초 스냅샷을 사용자가 수정하지 않고 저장해도 장부 행이 사라지지 않게 한다.
- 품목 생성, 스냅샷 저장, 감사 기록이 함께 성공하거나 함께 취소되게 한다.
- 업로드 결과와 오류가 운영자가 이해할 수 있는 지점·날짜·품목 정보를 제공하게 한다.

## 비목표

- 마감 장부나 이미 작성된 장부의 재고를 자동 병합하거나 소급 수정하지 않는다.
- 비활성 품목을 자동 재활성화하지 않는다.
- 엑셀 재고 단가를 품목 마스터의 `defaultUnitPrice`로 저장하지 않는다.
- 과거 사고 데이터 복구 절차를 일반 업로드 기능으로 바꾸지 않는다.

## 선택한 설계

### 1. 기준 코드 동기화

구현 전에 현재 브랜치를 `main` 기준으로 갱신한다. `main`에 있는 다음 검증된 동작을 재사용한다.

- `OPENING_SNAPSHOT`, `PREVIOUS_CLOSED_LEDGER`, `PREVIOUS_SAVED_LEDGER` 행은 수량을 바꾸지 않아도 장부 저장 시 유지한다.
- 기존 장부 재고와 월초 스냅샷의 충돌 정보를 업로드 결과에 노출한다.
- 필요한 과거 데이터 복구는 별도의 감사 가능한 복구 명령으로만 수행한다.

### 2. 업로드 전 기존 장부 충돌 검사

각 엑셀 행의 입력 날짜 다음 날을 대상 장부 날짜로 계산한다. 대상 지점·날짜의 `DailyLedger`에 `LedgerInventoryItem`이 하나라도 있으면 충돌로 본다.

충돌이 하나라도 있으면 파일 전체를 `VALIDATION_ERROR`로 거부한다. 오류에는 고유한 지점명과 대상 날짜를 표시한다. 이 검사는 품목 생성과 스냅샷 upsert가 일어나는 같은 트랜잭션에서 수행하므로, 충돌 파일은 어떤 데이터도 남기지 않는다.

### 3. 자동 품목 생성

품목 키는 정규화된 `(name, category, spec)`이다. `앱 품목명/앱 규격`이 있으면 파서가 선택한 앱 값을 사용하고, 없으면 원본 값을 사용한다.

- 활성 품목이 있으면 재사용한다.
- 품목이 전혀 없으면 `Product`를 `isActive=true`, `defaultUnitPrice=null`, `updatedById=업로더`로 생성한다.
- 같은 신규 품목이 여러 지점 행에 있어도 한 번만 생성하고 모든 행이 같은 `productId`를 사용한다.
- 같은 키의 비활성 품목이 있으면 자동 재활성화하지 않고 해당 엑셀 행을 명시해 파일 전체를 거부한다.
- 새 품목마다 같은 트랜잭션에서 `product.created` 감사 로그를 남기고 사유를 `재고 스냅샷 업로드 미등록 품목 자동 생성`으로 기록한다.

`UPLOAD_COMMIT` 권한은 재고 업로드의 부수 작업인 품목 생성까지 허용한다. 별도 `SETTINGS_MANAGE` 권한을 추가로 요구하지 않는다. 모든 자동 생성을 감사 로그로 추적한다.

### 4. 스냅샷 저장과 결과

장부 충돌과 지점 검증이 통과한 뒤 `(storeId, yearMonth, productId)`로 `InventoryOpeningSnapshot`을 upsert한다. 업로드 결과에는 기존 생성·갱신·동일 건수 외에 `productCreatedCount`를 포함한다. UI는 자동 추가된 품목 수를 성공 요약에 표시한다.

동일 파일을 다시 올리면 기존 활성 품목과 스냅샷을 재사용하므로 중복 품목이나 중복 스냅샷은 생기지 않는다. 별도의 파일 해시 중복 차단은 추가하지 않는다.

### 5. 장부 반영 의미

업로드 성공은 안전한 월초 스냅샷 저장을 뜻한다. 대상일 장부가 아직 작성되지 않았거나 재고 행이 비어 있으면 지점 재고 화면이 스냅샷을 불러온다. 사용자가 수량을 그대로 저장해도 근거 있는 이월 행을 모두 `LedgerInventoryItem`으로 보존한다.

이미 작성된 장부는 업로드로 바뀌지 않는다. 운영자는 충돌 오류를 보고 기존 장부를 먼저 확인해야 하며, 필요한 소급 수정은 일반 업로드가 아니라 별도 감사 절차를 사용한다.

## 오류와 원자성

다음 조건은 파일 전체를 취소한다.

- 지점명이 활성 지점과 일치하지 않음
- 동일 키의 비활성 품목 존재
- 같은 기준월·지점·품목 중복
- 대상일 장부에 기존 재고 행 존재
- 파일 구조, 날짜, 수량, 단가 검증 실패

품목 생성, 품목 감사 로그, 스냅샷 upsert, 업로드 감사 로그는 하나의 DB 트랜잭션에서 처리한다. 어떤 검증이나 저장이 실패해도 일부 품목만 남는 상태를 허용하지 않는다.

## 테스트 전략

- 신규 품목 업로드가 품목과 `product.created` 감사를 한 번만 생성하는지 검증한다.
- 기존 활성 품목 재사용과 재업로드 중복 방지를 검증한다.
- 같은 신규 품목의 여러 지점 행이 하나의 품목을 공유하는지 검증한다.
- 비활성 품목, 잘못된 지점, 파일 내부 중복, 기존 장부 충돌이 전체 트랜잭션을 취소하는지 검증한다.
- 자동 품목의 `defaultUnitPrice`가 `null`인지 검증한다.
- `2026-06-30` 입력이 `2026-07-01` 대상일과 `2026-07` 스냅샷으로 이어지는지 검증한다.
- 실제 xlsx 업로드 후 스냅샷을 불러오고, 수량을 수정하지 않은 채 장부 저장·재조회해도 모든 행이 유지되는지 E2E로 검증한다.
- 기존 장부가 있는 지점은 업로드가 거부되고 장부 값이 변하지 않는지 검증한다.

## 배포와 데이터 보호

- 사용자 제공 미추적 엑셀과 조사 문서는 커밋에 포함하지 않는다.
- 원격 운영 DB를 사용하는 E2E나 복구 명령은 실행하지 않는다.
- 구현 커밋은 설계 문서 커밋 이후 별도로 만들고, 전체 단위 테스트·타입 검사·안전한 테스트 DB E2E가 통과한 뒤 현재 브랜치를 푸시한다.
