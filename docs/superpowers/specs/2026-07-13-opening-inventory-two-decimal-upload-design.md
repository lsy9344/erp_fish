# 과거 재고 업로드 수량 소수점 둘째 자리 허용 설계

## 배경

과거 재고 엑셀 양식의 `남은 수량`에는 `0.62`, `0.71`처럼 소수점 둘째 자리 값이 들어 있다. 엑셀 양식은 이 값을 허용하도록 수정됐지만, 웹 업로드 파서는 공용 한 자리 수량 검증을 사용해 `5행 남은 수량 값을 확인해 주세요.` 오류를 반환한다.

현재 `main`과 `feat/rev_02` 모두 같은 한 자리 업로드 규칙을 사용하므로 기존 `main`을 다시 배포하는 것만으로는 해결되지 않는다.

## 목표

- 과거 재고 엑셀 업로드의 `남은 수량`에서 0 이상 소수점 둘째 자리까지 허용한다.
- `0`, `0.2`, `0.62`, `0.71`, `1.5`를 원래 값 그대로 읽는다.
- 음수와 소수점 셋째 자리 이상 값은 행 번호와 열 이름을 포함한 기존 오류 형식으로 거부한다.
- 사용자 제공 파일의 `1.375`는 `1.38`로 반올림해 실제 업로드가 끝까지 통과하게 한다.
- 수정한 양식을 프로젝트의 추적 파일과 전달용 파일에 동일하게 반영한다.

## 비목표

- 재고·매입·손실 화면의 일반 신규 입력 정책을 두 자리로 넓히지 않는다.
- 이카운트 매입·공급 업로드의 한 자리 정책을 바꾸지 않는다.
- 공용 `isNonNegativeDecimalInRange` 또는 `roundToOneDecimal`의 의미를 바꾸지 않는다.
- Prisma `Decimal(12, 2)` 컬럼을 변경하거나 데이터베이스 마이그레이션을 만들지 않는다.
- 단가와 금액의 정수 검증을 바꾸지 않는다.

## 설계

### 웹 업로드 파서

`src/features/inventory/opening-import.ts`의 과거 재고 수량 경계만 두 자리 규칙을 사용한다. 값은 유한한 숫자이고 0 이상이며 기존 최대값 이하여야 한다. `값 × 100`이 허용 오차 안에서 정수일 때만 통과시키고, 반환값은 `roundToTwoDecimals`로 정규화한다.

기존 공용 한 자리 검증은 다른 입력 경로가 계속 사용하므로 수정하지 않는다. 이렇게 하면 과거 재고 업로드 요구만 해결하고 매입·손실 정책에 영향을 주지 않는다.

### 엑셀 양식

`재고입력` 시트의 `남은 수량`과 `입고별잔량_선택` 시트의 `남은 수량`은 다음 계약을 유지한다.

- 표시 형식: `#,##0.00`
- 데이터 검증: 0 이상, 소수점 둘째 자리까지
- 허용 예: `0.2`, `0.62`, `0.71`, `1.5`
- 거부 예: `-0.1`, `0.001`

사용자 제공 데이터 중 셋째 자리 값 `1.375`는 DB 정밀도와 업로드 계약에 맞춰 `1.38`로 수정한다. 다른 값·수식·시트 구조·열 너비·행 높이·화면 서식은 유지한다.

프로젝트의 `outputs/inventory_import_template/과거_재고_간단_입력_양식.xlsx`와 전달용 `outputs/inventory_template_from_123123/123123_소수입력_수정본.xlsx`는 같은 파일 해시를 갖게 한다.

### 테스트

- 단위 테스트에서 `0.2`, `0.62`, `0.71`이 그대로 파싱되는지 확인한다.
- `0.001`과 음수가 기존 행 오류로 거부되는지 확인한다.
- 실제 추적 엑셀 파일을 웹 파서로 읽어 전체 행이 성공하는지 확인한다.
- 엑셀 자체 검증에서 `0.2`, `0.62`, `0.71`은 허용되고 `0.001`, 음수는 거부되는지 Excel로 확인한다.
- 관련 단위 테스트, 타입 검사, 린트, 프로덕션 빌드를 실행한다.

## 병합과 배포

구현은 최신 `main`에서 분기한 격리 브랜치에서 진행한다. 검증이 끝나면 변경 파일만 커밋하고 최신 `main`에 fast-forward 병합한 뒤 `origin/main`에 푸시한다. 저장소의 기존 배포 방식이 `main` 푸시 자동 배포이면 해당 배포 상태와 서비스 응답을 확인한다.
