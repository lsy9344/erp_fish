# 아침회의 리포트 재배치·재구성 계획

- 작성일: 2026-07-18
- 대상 화면: `/app/reports/daily`
- 대상 사용자: 본사 리포트 조회 권한 보유자
- 범위: 화면 재배치, 매출 분석 3종, 직원 근태 현황 추가

## 1. 결론

새 리포트 라우트나 신규 DB 모델을 만들지 않고 기존 `아침 회의 리포트` 한 페이지를 다음 5개 영역으로 재편한다.

1. 지점별 매출·이익률
2. 매출 분석
   - 전일 대비 매출액 증감률
   - 재고비율
   - 매장 매출 포지션
3. 품목별 판매 현황
4. 직원 근태 현황
5. 마감·이상 신호 현황(기존 내용)

기존의 지점 차트, 품목 판매순위, 요약 카드, 상세 표를 재사용한다. 추가 개발은 매출 분석 집계와 직원 근태 요약에만 집중한다.

## 2. 현재 구현과 변경 판정

| 요청 항목 | 현재 상태 | 계획 |
| --- | --- | --- |
| 지점별 매출·이익률 | `StoreDailyPerformanceChart` 구현 완료 | 제목을 간결히 정리하고 첫 번째 본문 영역으로 유지 |
| 전일 대비 매출액 증감률 | 없음 | 정정 반영 매출 기준으로 추가 |
| 재고비율 | 기간·월간 계산 규칙은 있으나 일별 화면에 없음 | 선택일 재고금액 ÷ 선택일 매출액으로 추가 |
| 매장 매출 포지션 | 매출순 정렬만 가능 | 권한 범위 내 순위와 전체 매출 비중을 함께 표시 |
| 품목별 판매 현황 | `품목 판매순위` 구현 완료 | 섹션명만 요청 용어로 변경하고 기존 상위 10개+검색 유지 |
| 직원 근태 현황 | 장부에 직원명·지각·조퇴·특이사항 저장 중, 일별 리포트 없음 | 기존 `LedgerLaborItem`을 읽어 요약·목록 추가 |
| 기존 내용 | 상태 요약 카드 + 지점별 상세 표 | `마감·이상 신호 현황`으로 묶어 페이지 하단으로 이동 |

## 3. 제안 화면 구조

```text
[페이지 제목 / 리포트 이동 / 날짜 / Excel·CSV]

[지점별 매출·이익률]
  └ 기존 매출 막대 + 실제·예상 이익률 + 정렬

[매출 분석]
  ├ 전일 대비 매출액 증감률
  ├ 재고비율
  └ 매장 매출 포지션

[품목별 판매 현황]
  └ 기존 판매수량 상위 10개 + 품목/규격 검색

[직원 근태 현황]
  ├ 정상 / 지각 / 조퇴 / 특이사항 / 미연결 요약
  └ 지점·직원별 상세

[마감·이상 신호 현황]
  ├ 기존 활성 지점/마감/검토 대기/미입력/손실 카드
  └ 기존 지점별 상태·매출·이익률·손실·이상 신호 표
```

페이지 상단의 기존 리포트 이동 버튼과 날짜 선택 방식은 변경하지 않는다. 데스크톱·모바일 모두 동일한 섹션 순서를 사용한다.

## 4. 항목별 표시·계산 규칙

### 4.1 지점별 매출·이익률

- 기존 `StoreDailyPerformanceChart` 데이터와 정렬 기능을 그대로 사용한다.
- 제목은 `지점별 매출·이익률`로 표준화한다.
- 매출은 장부의 정정 반영 총매출, 이익률은 기존 매출원가 기준 실제 이익률을 사용한다.
- 예상 이익률은 오늘 팔 가격이 누락된 품목이 있으면 기존처럼 `데이터 부족`으로 표시한다.

### 4.2 전일 대비 매출액 증감률

```text
증감액 = 선택일 정정 반영 매출 - 전일 정정 반영 매출
증감률 = 증감액 / 전일 정정 반영 매출
```

- 지점별 `선택일 매출 / 전일 매출 / 증감액 / 증감률`을 표시한다.
- 기본 정렬은 증감률 절대값이 큰 지점 순으로 한다.
- 전일은 선택일의 직전 달력일을 사용한다. 영업일 캘린더를 새로 만들지 않는다.
- 전일이 휴무·미입력이거나 매출이 0원이면 0%로 보이지 않고 `계산 불가`와 사유를 표시한다.
- 전일 장부의 정정도 동일하게 반영한다.

### 4.3 재고비율

```text
재고비율 = 선택일 FIFO 기준 재고금액 / 선택일 정정 반영 매출액
```

- 기존 기간 리포트의 `평균재고 / 평균매출` 계산 의미를 1일 범위에 적용한다.
- 지점별 `재고금액 / 매출액 / 재고비율`을 표시한다.
- 재고금액은 단순 `수량 × 단가`를 새로 계산하지 않고 기존 FIFO `inventoryAmount`를 사용한다.
- 한 품목이라도 재고금액을 구할 수 없으면 부분합을 정상값처럼 표시하지 않고 지점 전체를 `계산 불가`로 표시한다.
- 매출 0원은 0%가 아니라 `계산 불가`로 표시한다.

### 4.4 매장 매출 포지션

1차 구현의 `포지션`은 본사 사용자가 조회할 수 있는 활성 지점 내 매출 순위와 매출 비중을 뜻한다.

```text
매출 비중 = 지점 매출 / 권한 범위 내 전체 지점 매출 합계
```

- `순위 / 지점 / 매출액 / 전체 비중 / 전체 평균 대비`를 가로 막대와 함께 표시한다.
- 기본 정렬은 매출 내림차순, 동률은 지점명 순으로 고정해 재조회 시 순서가 흔들리지 않게 한다.
- 미입력·휴무는 순위에 포함하지 않고 제외 사유를 별도 표시한다.
- 2축 사분면 차트는 축의 업무 정의가 확정된 뒤에만 추가한다.

### 4.5 품목별 판매 현황

- 기존 `ProductProfitabilityReport` 테이블 모드를 유지한다.
- 섹션 제목은 `품목별 판매 현황`, 컬럼은 `품목 / 규격 / 판매수량`으로 표시한다.
- 기본은 판매수량 상위 10개, 검색 시에는 전체 결과에서 품목명·규격을 찾는 기존 동작을 유지한다.
- 판매수량은 POS 확정값이 아니므로 설명에 `재고 흐름으로 계산한 판매수량`을 유지한다.

### 4.6 직원 근태 현황

신규 출·퇴근 시간 모델을 만들지 않고 선택일 장부의 `LedgerLaborItem`을 사용한다.

| 상태 | 판정 기준 |
| --- | --- |
| 정상 | 직원 행이 있고 지각·조퇴·특이사항 메모가 모두 없음 |
| 지각 | `lateMemo`가 있음 |
| 조퇴 | `earlyLeaveMemo`가 있음 |
| 특이사항 | `specialMemo`가 있음 |
| 직원 미연결 | 자유 입력 `workerName`만 있고 `employeeId`가 없음 |
| 명단 부족 | 장부 `workerCount`가 직원 행 수보다 큼 |

- 요약 카드: `총 근무자 / 지각 / 조퇴 / 특이사항 / 명단 부족`.
- 상세 표: `지점 / 직원 / 상태 / 지각 메모 / 조퇴 메모 / 특이사항`.
- 한 직원에 복수 상태가 있으면 모두 배지로 표시한다.
- 급여·인건비 금액은 아침회의 리포트에 노출하지 않는다.
- 현재 데이터로 판별할 수 없는 결근·휴가·정확한 출퇴근 시각은 표시하지 않는다.

### 4.7 마감·이상 신호 현황(기존 내용)

- 기존 5개 요약 카드를 이 섹션 상단으로 이동한다.
- 기존 `DailyMeetingReportTable`을 요약 카드 아래에 그대로 유지한다.
- 정정 근거, 상세 장부 이동, 이상 신호, 모바일 카드 동작을 변경하지 않는다.

## 5. 데이터·권한 경계

- 모든 매출·재고·이익률은 기존 `requireReportAccess` 및 `getHeadquartersStoreScope` 범위 안에서만 조회한다.
- 선택일과 전일 매출은 모두 정정 반영값을 사용한다.
- 미입력, 휴무, 계산 불가를 0으로 바꿔 순위·평균·비율에 넣지 않는다.
- 직원 이름과 근태 메모는 기존 본사 리포트 권한 안에서만 노출한다.
- 직원 근태에서 급여금액, 직원 ID, 내부 정정 키는 클라이언트 데이터에 넣지 않는다.

## 6. 구현 계획

### 1단계: 일별 리포트 데이터 계약 확장

**수정 파일**

- `src/features/reports/types.ts`
- `src/features/reports/queries.ts`

**작업**

1. `DailyMeetingReportData`에 `salesAnalysis`, `attendance` 읽기 전용 DTO를 추가한다.
2. 기존 선택일 장부 조회에 `ledgerLaborItems` 필드만 추가한다.
3. 전일 매출은 기존 정정 반영 계산 헬퍼를 재사용하여 조회한다.
4. 재고비율은 현재 일별 장부 조회에 이미 포함된 `inventoryAmount`를 지점별로 합산한다.
5. 포지션 순위·비중·평균 대비는 서버 DTO 생성 단계에서 계산하고 UI에서 업무 계산을 반복하지 않는다.

신규 DB 모델, 마이그레이션, 외부 차트 라이브러리는 필요하지 않다.

### 2단계: 섹션 컴포넌트 추가 및 기존 섹션 재배치

**생성 파일**

- `src/features/reports/components/daily-sales-analysis.tsx`
- `src/features/reports/components/daily-attendance-report.tsx`

**수정 파일**

- `src/app/app/reports/daily/page.tsx`
- `src/app/app/reports/daily/loading.tsx`

**작업**

1. 매출 분석 3개를 한 컴포넌트의 하위 영역으로 구성한다.
2. 직원 근태는 데스크톱 표와 모바일 카드를 동일 DTO로 표시한다.
3. 기존 차트·품목 표·상태 표는 이동만 하고 내부 업무 로직을 바꾸지 않는다.
4. 로딩 화면도 실제 섹션 순서와 같게 재배치한다.

### 3단계: 검증

**수정 파일**

- `tests/unit/hq-reports.test.mjs`
- `tests/e2e/hq-reports.spec.ts`

**필수 검증**

1. 선택일·전일 정정 반영 매출로 증감률이 계산된다.
2. 전일 미입력·휴무·0원은 `계산 불가`로 보인다.
3. FIFO 재고금액이 하나라도 누락되면 재고비율을 부분 계산하지 않는다.
4. 지정 지점 본사 계정은 권한 범위 밖 지점의 순위·비중·근태를 보지 못한다.
5. 지각·조퇴·특이사항·직원 미연결·명단 부족이 구분된다.
6. 모바일에서 섹션 순서가 동일하고 가로 넘침이 없다.
7. 기존 정정 근거, 장부 상세 이동, Excel·CSV, 리포트 간 이동이 계속 동작한다.

## 8. 1차 범위에서 제외

- 신규 출퇴근 체크 기능·근태 DB 모델
- 결근·휴가 자동 판정
- 매출 포지션 2축 사분면 차트
- 사용자별 위젯 순서 설정
- 신규 차트 의존성
- Excel·CSV 시트/컬럼 확장

화면 검수 후 수치를 내보내야 하는 요구가 확정되면 기존 export 구조에 매출 분석·근태 시트를 별도 추가한다.

## 9. 구현 전 확인 1건

`매장 매출 포지션`을 1차 계획의 `매출 순위 + 전체 매출 비중`이 아니라 사분면으로 보려는 경우, 두 축을 먼저 확정해야 한다. 별도 지시가 없으면 본 계획의 순위·비중 방식으로 구현한다.
