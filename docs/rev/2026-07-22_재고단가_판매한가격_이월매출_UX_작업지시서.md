## 2. 현행 코드 확인 결과

### 2.1 재고 엑셀 단가는 저장되지만 지점장에게 표시되지 않는다

**Evidence**

1. 재고 엑셀은 `재고 단가`를 필수 헤더로 읽는다 (`src/features/inventory/opening-import.ts:9-17`).
2. 파서가 `재고 단가`를 정수로 읽고 `InventoryOpeningImportRow.unitPrice`에 넣는다 (`src/features/inventory/opening-import.ts:567-593`).
3. 업로드 액션은 단가를 `InventoryOpeningSnapshot.unitPrice`에 생성·갱신한다 (`src/features/inventory/opening-import-actions.ts:350-394`).
4. 재고 조회는 월초 스냅샷의 `unitPrice`를 재고 기준 행의 내부 단가로 사용한다 (`src/features/inventory/queries.ts:829-864`).
5. 재고 행의 표시용 `purchasePrice`는 `LedgerPurchaseItem` 이력만 조회한다. 월초 스냅샷 단가는 이 표시 DTO에 붙이지 않는다 (`src/features/inventory/queries.ts:1061-1129`).
6. 지점장 응답은 최상위 `unitPrice`를 제거하고 중첩 `purchasePrice`만 허용한다 (`src/features/inventory/types.ts:124-149`, `src/features/inventory/response-shaping.ts:26-60`).
7. 화면의 월초 재고단가 fallback은 `hasSensitiveInventoryAmounts(item)` 조건 때문에 내부 `unitPrice`를 가진 본사 데이터에서만 동작한다 (`src/features/inventory/components/inventory-step-client.tsx:1769-1774`).
8. 단위 테스트도 이 동작을 `HQ inventory shows opening stock price fallback`으로 고정하고 있다 (`tests/unit/ledger-inventory.test.mjs:1485-1506`).

**판정**

- 엑셀 단가 유실 문제는 아니다.
- 업로드·DB 저장은 정상이고, **지점장 표시용 DTO로 전달되지 않는 것이 원인**이다.
- 따라서 재고 엑셀 파서나 DB 컬럼을 새로 만들 필요는 없다.

### 2.2 판매계획가의 재고 단계 이동은 이미 완료됐다

**Evidence**

- 재고 DTO에 `plannedUnitPrice`가 있고 (`src/features/inventory/types.ts:18-42`), 재고 조회가 당일 `StoreSalesPricePlan`을 붙인다 (`src/features/inventory/queries.ts:1096-1127`).
- 지점장 재고 행에 `판매계획가` 입력과 `계획 마진율` output이 있다 (`src/features/inventory/components/inventory-step-client.tsx:2009-2071`).
- 빈 값을 막고 모든 표시 행의 가격을 요구한다 (`src/features/inventory/components/inventory-step-client.tsx:603-631`).
- 서버 스키마도 지점장 재고 저장에서 가격을 필수 정수로 검증한다 (`src/features/inventory/schemas.ts:153-176`).
- 재고 저장 트랜잭션은 가격을 upsert하고 기존 손실 행을 다시 계산한다 (`src/features/inventory/actions.ts:627-639`).
- 매입 단계가 판매계획가를 더 이상 다루지 않는 회귀 테스트가 있다 (`tests/e2e/store-ledger-purchase.spec.ts:222-261`).

**판정**

- “재고 단계에서 입력 가능”은 신규 개발 대상이 아니다.
- 이번 작업에서는 명칭 변경, 엑셀 단가 표시, UX 개선과의 결합 회귀만 다룬다.

### 2.3 손실 단계를 재고 뒤로 옮기면 현재 계약이 깨진다

**Evidence**

- 현재 순서는 `매입 → 손실/폐기 → 재고 → 지출 → 근무 → 매출 → 검토`다 (`src/features/ledger/components/store-entry-step-navigation.tsx:22-30`).
- 손실 저장은 `lossReviewedAt`을 기록한다 (`src/features/losses/actions.ts:439-450`).
- 재고 저장은 손실 검토 완료가 없으면 서버에서 거부한다 (`src/features/inventory/actions.ts:436-446`).
- 재고 기준수량은 `전일재고 + 매입 - 손실`이고, 당일 판매량은 그 값에서 당일재고를 뺀다 (`src/features/inventory/components/inventory-step-client.tsx:977-983`, `:1252-1265`).
- 손실 수량 자체도 전일·매입 수량을 기준으로 초과 여부를 검증한다 (`src/features/losses/actions.ts:383-437`).
- 손실 저장 당시 가격이 없어도 0원·미산정으로 저장하고, 이후 재고 가격 저장이 손실 금액을 재동기화한다 (`src/features/losses/actions.ts:356-380`, `src/features/losses/planned-price-sync.ts:7-103`).

**판정**

- 현재 순서는 단순 화면 순서가 아니라 서버 저장 가드와 계산 순서다.
- `판매한 가격`을 먼저 보이게 하려는 이유만으로 손실 단계를 재고 뒤로 이동하지 않는다.
- 향후 순서 변경은 `lossReviewedAt` 가드 제거, 손실 저장 후 재고 완료 재판정, 신규 손실 품목의 가격 누락 처리, 재고·FIFO·검토 재계산을 포함한 별도 프로젝트로 분리한다.

### 2.4 이월 매출은 데이터 모델과 계산에 없다

**Evidence**

- `DailyLedger` 매출/결제 필드는 `totalSalesAmount`, `cashAmount`, `cardAmount`, `otherPaymentAmount`뿐이다 (`prisma/schema.prisma:420-440`).
- 매출 입력 DTO와 저장 payload도 같은 네 필드만 가진다 (`src/features/ledger/types.ts:7-27`, `src/features/ledger/components/sales-payment-step-client.tsx:177-196`).
- 매출 저장 액션과 본사 수정 액션도 네 필드만 갱신한다 (`src/features/ledger/actions.ts:825-837`, `src/features/ledger/hq-edit-actions.ts:426-436`).
- 마감 정산 차액은 `총매출 - 현금 - 카드 - 기타 - 지출`이다 (`src/server/calculations/ledger.ts:27-47`).
- 총매출은 매출이익, 마진율, 영업이익, 생산성, 매출차액, 계획 대비 차이에 직접 사용된다 (`src/server/calculations/ledger.ts:582-667`, `:717-731`).
- 같은 계산기는 검토, 본사 관제, 마감 preflight, 기간·월간 리포트에서 재사용된다 (`src/features/ledger/review-queries.ts:545-580`, `src/features/dashboard/queries.ts:661-689`, `src/features/ledger/hq-close-preflight.ts:271-304`, `src/features/reports/queries.ts:3368-3400`).

**판정**

- 입력칸만 추가하면 안 된다.
- 어떤 합계에는 포함하고 어떤 정산에는 제외할지 먼저 고정해야 한다.

### 2.5 저장 UX와 Enter 포커스는 요청을 충족하지 않는다

**Evidence**

- 재고 저장 성공 메시지는 `저장됐습니다.`와 `재고 정보를 저장했습니다.`뿐이다 (`src/features/inventory/components/inventory-step-client.tsx:832-840`).
- `수정됨` 배지는 이번 저장 여부가 아니라 `당일재고 != 전일재고`인지로 계산한다 (`src/features/inventory/components/inventory-step-client.tsx:1143-1153`, `:1793-1804`).
- 저장 뒤 모든 행 상태를 서버 응답으로 다시 채우므로 “이번 저장에서 바뀐 행” 집합은 보존하지 않는다 (`src/features/inventory/components/inventory-step-client.tsx:832-834`).
- 당일재고 input에는 `onChange`, `onFocus`, `onBlur`만 있고 Enter 처리기가 없다 (`src/features/inventory/components/inventory-step-client.tsx:1928-1971`).
- input은 `<form>` 안에 있으므로 별도 처리하지 않은 Enter는 폼 제출로 이어질 수 있다 (`src/features/inventory/components/inventory-step-client.tsx:2427-2431`).
- 행이 30개를 넘으면 50개 단위 페이지가 생기므로 다음 행 포커스는 페이지 전환까지 고려해야 한다 (`src/features/inventory/components/inventory-step-client.tsx:117-120`, `:1724-1731`, `:2246-2284`).

## 3. 확정 구현 정책

### 3.1 재고 행의 단가 표시 우선순위

지점장 재고 행의 작은 단가 문구는 아래 순서로 하나만 표시한다.

1. 당일 실제 매입이 있으면 `당일 매입단가 · YYYY-MM-DD · N원/1박스`
2. 당일 매입이 없고 최근 실제 매입이 있으면 `최근 매입단가 · YYYY-MM-DD · N원/1박스`
3. 매입 이력이 없고 현재 재고 근거가 월초 엑셀 스냅샷이면 `월초 재고단가 · YYYY-MM · N원/1박스`
4. 모두 없으면 `단가 근거 없음`

구현 원칙:

- `InventoryOpeningSnapshot.unitPrice`를 지점장에게 허용되는 **표시용 단가 출처**로 명시한다.
- 내부 `InventoryStepLine.unitPrice`, FIFO lot 단가·금액, 재고금액을 지점장 DTO에 다시 노출하지 않는다.
- 기존 `InventoryPurchasePrice`를 `InventoryDisplayUnitPrice` 성격으로 확장하거나, `kind: "OPENING"`을 추가해 중첩 allowlist로 전달한다.
- 표시용 DTO는 `kind`, 기준일/기준월, `unitPrice`만 포함한다.
- 단가 우선순위는 조회 계층에서 한 번 결정하고 UI에서 내부 단가를 다시 추론하지 않는다.
- DB 마이그레이션은 필요 없다.

### 3.2 사용자 용어를 `판매한 가격`으로 변경

사용자에게 보이는 대표 용어를 다음과 같이 변경한다.

| 현재                           | 변경                            |
| ------------------------------ | ------------------------------- |
| 판매계획가                     | 판매한 가격                     |
| 계획 마진율                    | 판매가 기준 마진율              |
| 계획 판매가 기준 추정 매출     | 판매한 가격 기준 추정 매출      |
| 판매가 계획 없음               | 판매한 가격 없음                |
| 3단계 재고에서 판매계획가 입력 | 3단계 재고에서 판매한 가격 입력 |

의미 계약:

- 한 품목의 영업일별 `1박스 기준 대표 판매가격`이다.
- 품목별 실제 POS 라인이 없으므로 품목별 판매량과 품목 매출은 계속 **추정**으로 표기한다.
- 손실액 계산은 `판매한 가격 × 손실수량 - 떨이로 실제 판매한 금액` 계약을 유지한다.
- 사용자 용어만 바꾼다고 실제 POS 품목별 매출이 생긴 것으로 표현하지 않는다.

내부 호환성:

- 이번 작업에서 `StoreSalesPricePlan` 테이블과 `plannedUnitPrice` DB 컬럼은 이름을 바꾸지 않는다 (`prisma/schema.prisma:769-793`).
- DB rename/migration보다 사용자 계약과 계산 회귀 수정에 집중한다.
- 코드 주석에는 `legacy internal name; user label is 판매한 가격`을 남겨 새 기능에서 계획값으로 오해하지 않게 한다.
- 충돌 비교, 감사 로그, 검증 오류, 저장 상태, 접근성 label은 모두 새 사용자 용어를 사용한다.

### 3.3 손실 단계 순서 유지

이번 작업의 단계 순서는 유지한다.

```text
1 매입 → 2 손실/폐기/떨이 → 3 재고(판매한 가격) → 4 지출 → 5 근무 → 6 매출/결제 → 7 검토
```

손실 화면은 다음과 같이 안내한다.

- 첫 입력 시: `손실 수량과 떨이 판매액을 먼저 저장하세요. 3단계 재고에서 판매한 가격을 저장하면 손실액이 자동 확정됩니다.`
- 가격 저장 후 재방문 시: `손실액 산정 기준 판매한 가격 N원`
- 가격 미입력 상태는 `판매한 가격 없음 · 손실액 미산정`으로 표시한다.

### 3.4 이월 매출 계산 계약

#### 3.4.1 구현 전 P0 확인

저장소에는 이월 매출의 기존 정의가 없다. 개발 착수 전에 실제 영업일 2일치 예시로 아래를 확인한다.

1. `총매출` 입력값이 장부 마감 뒤 발생한 매출을 포함하는가?
2. 전일 이월 매출이 다음 날 `총매출`에 포함되는가?
3. 이월 매출 결제금액이 당일 `현금/카드/기타`에 포함되는가?

이 문서의 기본 구현은 다음 **권장 계약 A**를 전제로 한다.

#### 3.4.2 권장 계약 A — 총매출에 포함되지 않은 추가 영업 매출

```text
장부 마감 매출 = totalSalesAmount
이월 매출 = carryoverSalesAmount
영업 매출 합계 = totalSalesAmount + carryoverSalesAmount
마감 정산 차액 = totalSalesAmount - 현금 - 카드 - 기타 - 지출
```

- `carryoverSalesAmount`는 장부 마감 뒤 팔려 `totalSalesAmount`와 당일 결제 정산에 포함되지 않은 금액이다.
- 영업 성과 계산은 `영업 매출 합계`를 사용한다.
- 마감 정산 차액은 기존 `totalSalesAmount`만 사용한다. 이월 매출을 더하면 정상적인 마감도 불일치로 표시되기 때문이다.
- `carryoverSalesAmount` 기본값은 `0`, 허용 범위는 `0..2,147,483,647` 정수다.
- 합산 시에도 정수 범위를 넘지 않는지 검사하고, 초과하면 저장을 거부한다.

#### 3.4.3 계약 B — 다음 날 총매출에 전일 이월이 포함되는 경우

P0 확인에서 다음 날 총매출에 전일 이월이 포함되는 것으로 확인되면 단순 합산을 사용하지 않는다.

```text
영업 매출 합계(D)
= totalSalesAmount(D)
- carryoverSalesAmount(D-1)
+ carryoverSalesAmount(D)
```

- 전일 값을 자동 조회하고 현재 화면에 `전일 이월 차감`으로 읽기 전용 표시한다.
- 전일 장부 없음, 휴무, 데이터 누락의 처리 상태를 `0`과 구분한다.
- 계약 A/B가 확정되기 전에 리포트 계산을 구현하지 않는다.

#### 3.4.4 영업 매출 합계를 사용하는 계산

계약 A 기준으로 아래 계산의 `totalSalesAmount` 입력을 `operatingSalesAmount`로 바꾼다.

- 검토 요약의 총매출 표시
- 매출원가 대비 매출이익
- 실제 마진율
- 영업이익
- 인당 생산성
- 매출차액
- `판매한 가격 기준 추정 매출` 대비 장부 매출 차이
- 본사 관제판의 매출·마진·이상 신호
- 본사 마감 preflight
- 일별·기간·월간 매출, 평균매출, 손익 리포트
- CSV/XLSX의 매출·이익·마진 컬럼

#### 3.4.5 기존 마감 정산 값을 사용하는 계산

아래는 기존 `totalSalesAmount`를 유지한다.

- 현금·카드·기타·지출 합계와의 마감 정산 차액
- 마감 정산 불일치 경고
- 결제수단 합계 비교

#### 3.4.6 화면 표시

6단계 입력 순서는 다음으로 한다.

1. `총매출` — 도움말: `장부 마감 시점에 기록된 매출입니다.`
2. `이월 매출` — 도움말: `장부 마감 뒤 추가로 판매되어 총매출과 당일 결제 정산에 포함되지 않은 금액입니다.`
3. 읽기 전용 `영업 매출 합계` — `총매출 + 이월 매출`
4. 현금, 카드, 지출 합계, 기타 결제수단

본사 검토·리포트에서는 세 값을 구분한다.

- 장부 마감 매출
- 이월 매출
- 영업 매출 합계

### 3.5 재고 저장 영수증 UX

저장 직전 서버 기준값과 현재 입력값을 비교해 `이번 저장 변경 집합`을 만든다.

포함 대상:

- 당일재고 값이 바뀐 품목
- 새로 직접 추가해 저장한 품목
- 판매한 가격이 바뀐 품목
- 바꾼 이유가 새로 저장된 품목

저장 성공 후:

1. 상단/하단 상태 영역에 `방금 저장한 재고 N건`을 표시한다.
2. 당일재고가 바뀐 품목은 `품목명 3 → 2.5` 형식으로 표시한다.
3. 목록이 길면 앞 3건과 `외 N건`을 표시하고 전체 목록을 펼칠 수 있게 한다.
4. 해당 행에는 `방금 저장됨` 배지와 접근성 상태를 붙인다.
5. 사용자가 그 행을 다시 수정하면 `방금 저장됨`을 제거하고 `수정 중` 상태로 바꾼다.
6. 다음 저장 전까지 또는 화면을 떠날 때까지 저장 영수증을 유지한다. 임의 시간 후 사라지게 하지 않는다.
7. 변경값 없이 저장한 경우 `변경된 재고 없이 확인 저장했습니다.`로 구분한다.

기존 `수정됨`은 전일재고 대비 차이 의미이므로 삭제하지 않는다. `방금 저장됨`과 의미를 분리한다.

### 3.6 Enter 포커스 이동

당일재고 input의 Enter 처리 규칙:

1. 한글 IME 조합 중(`isComposing`)에는 처리하지 않는다.
2. Enter 기본 동작을 막아 폼 전체 저장이 실행되지 않게 한다.
3. 현재 활성 탭의 화면 순서에서 다음 행을 찾는다.
4. 다음 행의 `당일재고` input으로만 포커스를 이동한다. 판매한 가격·사유 input을 거치지 않는다.
5. 현재 페이지 마지막 행이고 다음 페이지가 있으면 다음 페이지로 이동한 뒤 첫 당일재고에 포커스한다.
6. 카테고리 탭에서는 해당 카테고리 안의 다음 번호로 이동한다.
7. 전체 탭에서는 전체 행 순서로 이동한다.
8. 마지막 행에는 포커스를 유지하고 `마지막 재고 항목입니다.`를 polite live region으로 알린다.
9. 읽기 전용·저장 중·본사 마감 상태에서는 기존 disabled 동작을 유지한다.

## 5. 필수 수용 기준

### 5.1 재고 엑셀 단가

- [ ] `재고 단가 12,000원`인 월초 엑셀 행이 DB에 12,000원으로 저장된다.
- [ ] 해당 품목에 매입 이력이 없으면 지점장 재고 행에 `월초 재고단가 · YYYY-MM · 12,000원/1박스`가 보인다.
- [ ] 같은 품목에 당일 매입이 있으면 월초 단가가 아니라 당일 가중평균 단가가 보인다.
- [ ] 당일 매입이 없고 최근 매입이 있으면 최근 매입단가가 보인다.
- [ ] 지점장 응답에 최상위 `unitPrice`, FIFO 단가·금액, 재고금액이 새로 노출되지 않는다.
- [ ] 본사 재고 화면의 기존 단가 표시가 유지된다.

### 5.2 판매한 가격과 손실

- [ ] 재고 화면의 모든 사용자 노출 `판매계획가`가 `판매한 가격`으로 바뀐다.
- [ ] 빈 값·음수·소수·정수 상한 초과 검증이 새 문구와 연결된다.
- [ ] 저장 DB는 기존 `StoreSalesPricePlan` 한 행을 계속 사용한다.
- [ ] 판매한 가격 변경 후 기존 손실 행의 `unitPrice`, `amount`, `usedPlannedPrice`가 같은 트랜잭션에서 갱신된다.
- [ ] 손실 단계는 계속 2단계이고 재고는 3단계다.
- [ ] 손실 0건이어도 손실 단계 검토 저장 후에만 재고 저장이 가능하다.
- [ ] 리포트의 품목별 매출은 실제 POS 확정값이 아니라 추정값으로 계속 표시된다.

### 5.3 이월 매출

- [ ] 기존 장부는 마이그레이션 후 `carryoverSalesAmount = 0`이다.
- [ ] 이월 매출은 저장·재방문·본사 수정에서 유지된다.
- [ ] 계약 A에서 총매출 100,000원, 이월 매출 10,000원이면 영업 매출 합계는 110,000원이다.
- [ ] 같은 값에서 현금 40,000원, 카드 50,000원, 기타 0원, 지출 10,000원이면 마감 정산 차액은 0원이다. 이월 10,000원 때문에 불일치가 생기지 않는다.
- [ ] 매출원가 60,000원이면 매출이익은 50,000원이고 마진율은 `50,000 / 110,000`이다.
- [ ] 본사 관제·검토·일별·기간·월간·export의 영업 매출이 모두 110,000원으로 일치한다.
- [ ] raw export에는 장부 마감 매출 100,000원, 이월 매출 10,000원, 영업 매출 합계 110,000원이 분리된다.
- [ ] 본사 마감 후 이월 매출 정정 시 관제·preflight·리포트가 정정 반영값으로 다시 계산된다.
- [ ] 합산이 정수 상한을 넘으면 저장 전체를 거부하고 이월 매출 input으로 포커스한다.

### 5.4 재고 저장 UX

- [ ] 두 품목의 당일재고를 바꾸고 저장하면 `방금 저장한 재고 2건`이 보인다.
- [ ] 각 품목명과 저장된 값이 요약 또는 펼침 목록에 보인다.
- [ ] 두 행에 `방금 저장됨` 배지가 보인다.
- [ ] 한 행을 다시 수정하면 그 행의 `방금 저장됨`이 사라지고 미저장 상태가 보인다.

## 7. 구현 순서

1. 이월 매출 계약 A/B를 실제 2영업일 예시로 확정한다.
2. 실패 테스트로 엑셀 단가 표시, 새 용어, 이월 매출 계산, 저장 receipt, Enter 포커스를 고정한다.
3. 재고 표시 단가 DTO를 확장하고 민감정보 차단을 검증한다.
4. 사용자 용어를 `판매한 가격`으로 변경하고 손실 단계 안내를 정리한다.
5. `carryoverSalesAmount` 마이그레이션·저장·본사 수정·감사 계약을 추가한다.
6. 공용 계산기에서 영업 매출과 마감 정산을 분리한다.
7. 검토·관제·preflight·리포트·export·정정에 같은 계산을 연결한다.
8. 재고 저장 receipt와 Enter 이동을 구현한다.
9. 단위 → 재고/손실/매출 E2E → 본사 관제/리포트 E2E → 전체 정적 검증 순으로 실행한다.

## 8. 위험과 방지책

| 위험                                                  | 방지책                                                                 |
| ----------------------------------------------------- | ---------------------------------------------------------------------- |
| 재고 엑셀 단가를 열면서 내부 FIFO 원가까지 노출       | 중첩 표시 DTO allowlist만 확장하고 최상위 `unitPrice` 차단 테스트 유지 |
| `판매한 가격`을 실제 POS 품목 매출로 오해             | 품목 판매량·매출에는 계속 `추정` 표기, 대표 단가 도움말 제공           |
| 손실 단계를 옮겨 재고 계산 순환 발생                  | 이번 범위에서 단계 순서·`lossReviewedAt` 가드 유지                     |
| 이월 매출을 총매출과 결제 정산에 모두 더해 경고 왜곡  | `operatingSalesAmount`와 `paymentDifference` 입력을 분리               |
| 다음 날 총매출에 전일 이월이 포함돼 월 매출 이중 집계 | Task 0에서 계약 A/B를 실제 2일 예시로 확정                             |
| 기존 장부/정정이 새 필드 때문에 계산 불가             | DB default 0, correction overlay와 raw/original/applied 계산 동시 수정 |
| Enter가 저장을 실행                                   | `preventDefault`, E2E에서 version·audit 불변 검증                      |
| 저장 배지가 과거 상태와 혼동                          | `수정됨`과 `방금 저장됨`을 별도 상태·문구로 유지                       |

## 9. 범위 제외

- 손실 단계를 재고 다음으로 실제 이동하는 구조 개편
- `StoreSalesPricePlan` 테이블/컬럼 물리 이름 변경
- 품목별 POS 판매 라인 도입
- 이월 매출의 품목별 수량 입력
- 새 UI/상태관리 라이브러리 추가

## 10. 완료 정의

다음 조건을 모두 만족해야 완료다.

1. 재고 엑셀 단가가 정해진 우선순위로 지점장 재고 행에 표시된다.
2. 사용자 용어가 `판매한 가격`으로 통일되고 손실 계산은 기존 동기화 계약을 유지한다.
3. 손실 단계는 2단계로 유지되며 재고·판매량 계산 회귀가 없다.
4. 이월 매출의 포함 관계가 운영 예시로 확정되고 모든 성과 계산·정산·리포트가 같은 계약을 사용한다.
5. 재고 저장 후 사용자가 방금 저장한 품목과 값을 식별할 수 있다.
6. Enter가 저장하지 않고 다음 번호 당일재고로 이동한다.
7. 관련 단위·E2E·타입·린트·빌드가 모두 통과하고 민감 단가 노출 회귀가 없다.
