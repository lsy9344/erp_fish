# 당일 현금매출·지출 정산 수정 Implementation Plan

> **구현 담당자:** 이 문서는 `docs/rev/3/KakaoTalk_20260717_143045859_01.jpg`와 2026-07-17 추가 확인 내용을 기준으로 한다. 기존 `docs/rev/2026-07-17_rev2_rev3_수정요청_상세_작업지시서.md`의 **Task 5를 이 문서로 대체**한다.

**Goal:** 6단계의 현금 입력을 “당일 지출 후 남은 당일 현금매출”로 명확히 하고, 4단계에서 저장한 지출을 마감 정산에 포함한다. 6단계의 주황색 `차액` 박스는 완전히 삭제한다.

**Architecture:** `총매출`과 현금·카드·기타 입력값은 기존 DB 컬럼을 그대로 사용한다. 지출 합계는 `LedgerExpense`의 마지막 서버 저장값으로 계산하며, 서버 공통 계산·검토·감사 로그가 같은 공식을 사용하게 한다. 6단계는 지출 합계를 읽기 전용으로 보여 주되 차액을 화면에서 실시간 계산하지 않는다. 본사 상세의 강제 마운트 탭 간에는 `updatedAt`, `version`, `expenseTotal`을 함께 동기화한다.

**Tech Stack:** Next.js 15, React 19, TypeScript, Prisma, shadcn `Field`/`Input`, Node test runner, Playwright

## 1. 확정 정책

### 1.1 입력값의 뜻

| 항목 | 확정 의미 | 저장 방식 |
| --- | --- | --- |
| 총매출 | 당일 전체 매출 | 기존 `DailyLedger.totalSalesAmount` 직접 입력 유지 |
| 현금 | **당일 현금지출을 하고 남은 당일 현금매출** | 기존 `DailyLedger.cashAmount` 사용 |
| 카드 | 당일 카드 매출 | 기존 `DailyLedger.cardAmount` 사용 |
| 기타 결제수단 | 당일 기타 결제 매출 | 기존 `DailyLedger.otherPaymentAmount` 사용 |
| 4단계 지출 합계 | 당일 현금매출에서 이미 빠져나간 현금지출 | 기존 `LedgerExpense.amount` 합계, 6단계에서는 읽기 전용 |

`총매출`을 결제수단 값으로 자동 산출하거나 덮어쓰지 않는다. 사용자가 입력한 총매출은 기존과 같이 독립된 원본값으로 보존한다.

### 1.2 마감 정산 공식

```text
마감 정산 합계 = 현금(지출 후) + 카드 + 기타 결제수단 + 4단계 지출 합계
마감 정산 차액 = 총매출 - 마감 정산 합계
```

정상 예시:

```text
총매출          1,000,000원
현금(지출 후)     400,000원
카드              500,000원
기타                    0원
4단계 지출         100,000원
마감 정산 차액           0원
```

지출은 매출이나 이익에 두 번 더하는 값이 아니다. 현금 입력값이 이미 지출만큼 줄어 있기 때문에, **총매출 원본과 현금 흐름을 맞춰 보는 마감 검산에서만 복원하여 더한다.** 영업이익 계산에서는 지금처럼 지출을 뺀다.

### 1.3 주황색 `차액` 박스 처리

- 6단계의 주황색 `결제 합계 차액` 박스를 값이 0원이든 아니든 **항상 삭제**한다.
- 같은 위치에 회색 박스, 성공 박스, 인라인 차액 문구 등 다른 차액 표시를 새로 만들지 않는다.
- 클라이언트에서 입력 중 차액을 계산하는 지역 함수와 상태도 삭제한다.
- 서버의 정산 검증값은 잘못 입력된 장부를 본사 검토·마감에서 찾기 위해 유지한다. 내부 호환성을 위해 `paymentDifference`와 `paymentDifferenceAmount` 식별자는 바꾸지 않아도 되지만, 사용자 문구는 `마감 정산 차액`으로 바꾼다.

### 1.4 변경하지 않는 계산

다음 값은 이번 수정 대상이 아니다.

- `총매출`, 매출원가, 매출이익, 마진율, 영업이익의 기존 의미
- `영업이익 = 매출이익 - 지출 합계`
- 재고·손실을 기준으로 계산하는 별도 `salesDifference`와 대시보드 이상 신호
- 매입, 재고, 손실, 근무 단계의 업무 규칙
- 지출 결제수단 구분 컬럼 추가

따라서 Prisma 스키마 변경과 DB 마이그레이션은 하지 않는다.

## 2. 완료 조건

- 6단계 현금 라벨 또는 바로 아래 설명에 `당일 지출 후 남은 당일 현금매출`이라는 뜻이 보인다.
- 카드와 기타 결제수단 사이에 `4단계 지출 합계`가 읽기 전용으로 표시된다.
- 지출 합계는 4단계의 **마지막 서버 저장값**이며, 6단계에서 수정할 수 없다.
- 6단계 DOM과 화면 어디에도 `결제 합계 차액` 주황색 박스가 없다.
- 서버 검산은 `총매출 - (현금 + 카드 + 기타 + 지출)`을 사용한다.
- 4단계 지출을 저장한 직후 본사 상세의 6단계 탭에 새 지출 합계와 새 장부 버전이 반영된다.
- 검토·마감 화면은 `결제 합계`가 아니라 `마감 정산`이라는 말로 공식을 설명한다.
- 지점장에게 숨기던 민감 지표 범위는 넓어지지 않는다.
- 기존 총매출·영업이익·재고 기반 차액 테스트가 그대로 통과한다.

### 배경

본사 장부 상세는 지출과 매출/결제 탭을 `forceMount`하며 같은 `initialLedger`를 각각의 클라이언트 state에 복사한다. 현재 이벤트는 `updatedAt`만 전파한다. 이 상태에서 4단계 지출을 저장하면 6단계는 이전 `expenseTotal`과 이전 `version`을 계속 가지고 있어, 화면 값이 낡거나 다음 저장이 충돌한다.

- [ ] **Step 1: 동기화 snapshot 확장**

이벤트와 메모리 map이 다음 최소 snapshot을 보관하도록 바꾼다.

```ts
type LedgerUpdatedSnapshot = {
  id: string;
  updatedAt: string;
  version: number;
  expenseTotal: number;
};
```

`notifyLedgerUpdated()`는 이 snapshot 하나를 받게 하고, 구독 callback도 snapshot을 받게 한다. 기존 `useLedgerUpdatedAtSync`는 실제 역할에 맞게 `useLedgerSync`로 이름을 바꾸고 모든 import와 호출부를 한 번에 고친다.

- [ ] **Step 2: 모든 저장 성공 시 최신 snapshot 전파**

지출·매입·근무·매출 저장 action은 이미 최신 장부 데이터를 반환한다. 각 성공 지점에서 `notifyLedgerUpdated(next)`를 호출해 최신 `version`, `updatedAt`, `expenseTotal`을 전파한다.

- [ ] **Step 3: 각 탭은 공통 장부 메타만 병합**

구독한 탭은 자신의 입력 중 draft를 덮어쓰지 않고 다음 공통 필드만 갱신한다.

```ts
setLedger((current) => ({
  ...current,
  updatedAt: snapshot.updatedAt,
  version: snapshot.version,
  expenseTotal: snapshot.expenseTotal,
}));
```

`hq-ledger-close-dialog.tsx`는 snapshot에서 `updatedAt`만 사용하면 된다.

- [ ] **Step 4: 본사 교차 탭 E2E 추가**

`tests/e2e/hq-ledger-edit.spec.ts`에 다음 회귀 시나리오를 추가한다.

1. 편집 가능한 본사 장부 상세를 연다.
2. `지출` 탭에서 합계를 4,000원으로 저장한다.
3. reload 없이 `매출/결제` 탭을 연다.
4. `4단계 지출 합계`가 즉시 `4,000원`으로 보인다.
5. 매출/결제를 저장한다.
6. 저장 충돌 dialog가 뜨지 않고 저장 성공 상태가 보인다.

단순 `router.refresh()`만 호출하는 방식은 사용하지 않는다. 현재 탭 컴포넌트 key에 `version`이 없고 각 컴포넌트가 초기 props를 state로 복사하므로, refresh만으로는 낡은 로컬 state가 확실히 교체되지 않는다.

## 8. Task 6 — seed와 fixture를 새 현금 정의에 맞춤

**Files:**

- [ ] **Step 1: 잘못된 seed 조합 수정**

현재 `seed-main.mjs`는 `cash + card + other = totalSales`을 먼저 만든 뒤 지출을 별도로 생성한다. 새 공식에서는 모든 지출 장부가 거짓 불일치가 된다.

seed는 그날의 지출 목록과 합계를 먼저 결정하고, 현금 배분액에서 지출 합계를 뺀 값을 `cashAmount`로 저장한다.

```text
grossCashAllocation = 총매출의 현금 결제 배분액
cashAmount = grossCashAllocation - expenseTotal
cardAmount + otherPaymentAmount = 총매출 - grossCashAllocation
```

seed 지출 합계가 현금 배분액보다 커지지 않도록 기존 seed 범위 안에서 금액을 제한한다. 음수 현금을 `Math.max(0, ...)`로 조용히 숨기지 말고 fixture 생성 조건 자체를 맞춘다.

- [ ] **Step 2: 실제 데이터 자동 보정 금지**

이번 변경으로 운영 장부의 `cashAmount`를 일괄 수정하지 않는다. 현금 원본이 지출 전 금액인지 지출 후 금액인지는 숫자만으로 확정할 수 없기 때문이다. 사용자가 확정한 운영 규칙은 지출 후 금액이므로 계산만 새 공식으로 전환한다.

- [ ] **Step 3: 배포 전 읽기 전용 표본 점검**

운영 DB를 수정하지 않는 조회로 다음 두 값을 비교한다.

```text
기존 차이 = 총매출 - (현금 + 카드 + 기타)
새 정산 차이 = 총매출 - (현금 + 카드 + 기타 + 지출)
```

새 정산 차이가 0원인 표본이 업무 설명과 일치하는지 확인한다. 반대 사례가 있으면 자동 보정하지 말고 해당 영업일·지점만 운영자 확인 대상으로 남긴다.
