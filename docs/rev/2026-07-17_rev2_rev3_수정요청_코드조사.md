## 2. rev 2 원인 조사

### 2.1 같은 품목의 오늘 팔 가격은 하루에 하나만 입력

사진에는 같은 품목의 여러 매입 행에 서로 다른 오늘 팔 가격이 들어가 있습니다. 서버 스키마는 같은 `productId`의 첫 가격과 다른 값을 가진 행을 오류로 처리합니다.

- 검증 규칙: `src/features/ledger/schemas.ts:389-415`
- DB도 지점·영업일·품목별 한 건만 허용: `prisma/schema.prisma:768-788`
- 저장 시 품목별 한 가격으로 upsert: `src/features/ledger/actions.ts:601-670`
- 회귀 테스트: `tests/e2e/store-ledger-purchase.spec.ts:314-355`

판정: **오류 자체는 의도된 데이터 규칙**입니다. 다만 품목 선택 시에는 형제 행에 가격을 복사하지만, 이후 한 행의 가격을 수정할 때는 그 행만 바뀝니다 (`src/features/ledger/components/purchase-step-client.tsx:496-529`, `:1054-1057`). 같은 품목이 여러 행이면 사용자가 쉽게 충돌 상태를 만들 수 있으므로, 동일 품목 행의 가격을 함께 바꾸거나 대표 입력 하나만 보여주는 쪽이 맞습니다.

사진 오른쪽의 `270,000원`, `28,000원`, `480,000원`은 오늘 팔 가격의 결과가 아니라 **매입금액**입니다.

```text
매입금액 = round(매입 단가 × 수량)

활문어 18,000 × 15 = 270,000원
활문어 14,000 ×  2 =  28,000원
전복   16,000 × 30 = 480,000원
```

- 화면 계산·표시: `src/features/ledger/components/purchase-step-client.tsx:161`, `:1082-1086`
- 서버 재계산: `src/server/calculations/inventory.ts:31-45`

`25,000`, `21,000`, `20,500`은 별도 판매가 계획이며 이후 분석 매출 계산에 사용됩니다.

### 2.2 손실 저장의 “재고 흐름을 확인할 수 없습니다”

손실 저장은 해당 품목의 재고 행을 찾은 뒤 다음 값을 확인합니다.

```text
손실 후 가능 재고 = 전일재고 + 오늘매입 - 손실수량
```

- 손실 저장 검증: `src/features/losses/actions.ts:384-429`
- 오류 문구 선택: `src/features/losses/quantity-error.ts:15-31`
- 수량 계산: `src/server/calculations/inventory.ts:48-77`

사진의 문구는 손실 수량이 단순히 재고보다 많을 때가 아니라, 선택한 `참소라 / 중`의 `productId`에 대응하는 재고 행 자체가 없을 때 나옵니다. 즉 당일 매입, 전일 저장 재고, 월초 스냅샷 중 어느 근거에도 품목이 연결되지 않은 상태입니다.

판정: **재고 근거가 없는 손실을 막는 검증은 의도된 동작**입니다. 실제 재고가 있었던 품목이라면 계산식 버그보다 다음 상류 데이터를 먼저 확인해야 합니다.

- 참소라/중의 품목 ID가 매입·재고의 품목 ID와 같은지
- 전일 장부 또는 월초 재고가 정상 이월됐는지
- 재고 단계에서 직접 추가한 품목이 저장됐는지

단계 순서가 손실 → 재고인데 오류는 재고 단계 확인을 요구하므로, 사용 흐름은 다듬을 필요가 있습니다.

### 2.3 값을 입력한 뒤에도 남는 “당일재고를 입력해 주세요”

매입·손실이 있는 신규 재고 행은 사용자가 남은 수량을 직접 확인하도록 빈칸으로 시작합니다. 빈칸 저장을 막는 규칙 자체는 의도입니다.

- 클라이언트 필수 입력 검사: `src/features/inventory/components/inventory-step-client.tsx:452-489`
- 서버의 같은 검사: `src/features/inventory/adjustment-save-guard.ts:39-94`

그러나 수량 입력의 `onChange`는 값만 바꾸고 기존 `fieldErrors`와 `formError`를 지우지 않습니다 (`src/features/inventory/components/inventory-step-client.tsx:693-701`). 따라서 다음 순서로 사진 상태가 재현됩니다.

1. 빈칸 상태에서 저장해 오류가 생김
2. 고등어 당일재고에 `4` 입력
3. 다시 저장하기 전까지 이전 오류가 화면에 계속 남음

사진의 기준재고 6, 당일재고 4는 정상 판매 2개로 계산됩니다. 값 `4`가 있는 상태에서 검사를 다시 실행하면 통과합니다.

판정: **입력 후 오래된 오류가 남는 UI 버그**입니다.

### 2.4 `0.6000000000000001` 표시

사진의 값은 JavaScript 이진 부동소수점 계산에서 `2 - 1.4`를 그대로 문자열로 넣어 생겼습니다. 이는 업무 수량이 아니라 표시 버그입니다.

사진 당시 코드는 기준재고와 당일재고 차이를 반올림하지 않고 오류 문장에 넣었습니다. 2026-07-16의 `5a8da87` 커밋 이후 기준재고 이하 수량은 정상 판매로 분류하므로 사진의 `2 → 1.4` 상황에서는 더 이상 사유를 요구하지 않습니다.

- 현재 정상/과재고 판정: `src/features/inventory/inventory-persist-policy.ts:84-114`
- 현재 기준재고 계산의 소수 정규화: `src/server/calculations/inventory.ts:48-77`
- 현재 사유 문구의 원시 뺄셈: `src/features/inventory/adjustment-save-guard.ts:14-22`

판정: **사진의 잘못된 부족재고 차단은 수정됐지만 표시 정밀도 문제는 완전히 끝나지 않았습니다.** 현재도 실제 과재고가 `1.4 - 0.8`처럼 계산되면 `0.5999999999999999`가 문구에 나올 수 있습니다. 문구에 넣기 전에 공통 수량 반올림·포맷을 써야 합니다.

### 3.2 6단계 매출/결제에 4단계 지출 합계 반영

사진 요청은 다음과 같습니다.

- 6단계 `카드` 입력칸 바로 아래, `기타 결제수단` 입력칸 위에 4단계에서 저장한 지출 총액을 수정 불가(읽기 전용)로 표시
- 정합식: `총매출 = 결제 합계 + 지출 합계`
- 차액: `총매출 - 결제 합계 - 지출 합계`

현재 공식은 아래와 같습니다.

```text
현재 결제 차액 = 총매출 - (현금 + 카드 + 기타 결제수단)
```

- 클라이언트 계산: `src/features/ledger/components/sales-payment-step-client.tsx:37-44`, `:117-127`
- 공통 서버 계산: `src/server/calculations/ledger.ts:602-622`
- 현재 화면: `src/features/ledger/components/sales-payment-step-client.tsx:379-464`

지출 합계는 이미 6단계에 전달되는 `initialLedger.expenseTotal`에 있으므로 조회를 새로 만들 필요는 없습니다 (`src/features/ledger/types.ts:77-101`, `src/features/ledger/queries.ts:295-337`). 표시 연결과 공통 차액 공식을 함께 바꿔야 합니다.

주의: 이는 총매출 값을 지출만큼 늘리라는 뜻보다, **결제 후 지출로 빠진 현금을 포함해 총매출과 맞추는 정합식**으로 해석하는 편이 안전합니다. `totalSalesAmount` 자체의 뜻을 바꾸면 대시보드 매출·마진까지 모두 변합니다.

### 3.3 대시보드 마진 표기와 “금액의 근거”

사진의 `35.9% / 90.0%`는 현재 **실제마진율 / 예상마진율**이 아닙니다.

```text
35.9% = 현재 장부 실제 마진율
90.0% = 본사 이상징후 설정의 기준 마진율
```

판매계획가와 재고 흐름으로 계산한 예상마진율은 그 아래의 작은 `분석 ...` 줄입니다 (`src/features/dashboard/components/hq-dashboard-table.tsx:770-790`). 요청대로라면 주 표기를 `실제마진율 / 예상마진율`로 바꾸고, 경보 기준 90%는 별도 표시해야 합니다.

#### 실제마진율

```text
매출원가 = 품목별 소비 원가 합계
매출이익 = 총매출 - 매출원가
실제마진율 = 매출이익 / 총매출
```

- 공통 계산: `src/server/calculations/ledger.ts:438-465`, `:651-656`
- FIFO 소비금액을 모두 구할 수 없으면 `round((전일재고 + 당일매입 - 당일재고) × 장부 적용 단가)`를 품목별로 합산합니다.

사진 데이터는 다음과 같이 재현됩니다.

| 지점 | 총매출 | 매출원가 | 매출이익 | 실제마진율 |
|---|---:|---:|---:|---:|
| 강서수산 | 2,573,000 | 1,650,000 | 923,000 | 35.8725% → 35.9% |
| 삼국유통 | 3,410,000 | 1,695,400 | 1,714,600 | 50.2815% → 50.3% |
| 제일수산 | 2,861,000 | 1,440,700 | 1,420,300 | 49.6435% → 49.6% |

#### 예상마진율

```text
예상 판매수량 = 전일재고 + 당일매입 - 손실수량 - 당일재고
예상 매출 = Σ round(예상 판매수량 × 오늘 팔 가격)
예상 매출이익 = 예상 매출 - 매출원가
예상마진율 = 예상 매출이익 / 예상 매출
```

- 판매수량·예상 매출: `src/server/calculations/ledger.ts:533-577`
- 예상 이익·마진율: `src/server/calculations/ledger.ts:712-722`

판매된 품목 중 오늘 팔 가격이 하나라도 없으면 `분석 데이터 부족`으로 표시합니다.

#### 노란 금액의 정확한 의미

`1,392,700원`은 매출이나 예상매출이 아니라 **현재 총매출에서 90% 기준 마진까지 부족한 매출이익 금액**입니다.

```text
미달 금액 = 총매출 × (기준 마진율 - 실제 마진율)
          = 기준 마진율 × 총매출 - 현재 매출이익

강서수산 = 2,573,000 × 90% - 923,000
          = 2,315,700 - 923,000
          = 1,392,700원
```

같은 공식으로 삼국유통은 `1,354,400원`, 제일수산은 `1,154,600원`입니다.

- 계산: `src/server/calculations/anomaly.ts:103-137`
- 화면의 현재/기준 조립: `src/features/dashboard/queries.ts:1315-1350`
- 기준 저장: `prisma/schema.prisma:385-388`

핵심 브리핑:

- 90%는 코드에 박힌 예상마진이 아니라 현재 활성화된 전역 경보 설정값입니다.
- 금액이 큰 이유는 실제마진이 약 36~50%인데 경보 기준이 90%로 설정돼 있기 때문입니다.
- 화면의 마진율은 소수 첫째 자리로 보이지만 금액은 반올림 전 비율로 계산되므로 표시값만 곱하면 몇백 원 정도 다를 수 있습니다.
- 요청대로 실제/예상 마진을 나란히 보이면, 이 경보 미달 금액은 `90% 기준 미달액`처럼 기준을 이름에 써야 오해가 없습니다.

### 3.4 손실 입력 용어 변경

- 도움말 예: 100마리 중 10마리 폐기 시 `0.1`, 10바구니 중 2바구니 폐기 시 `0.2`

```text
손실액 = 오늘 팔 가격 × 박스단위 손실수량 - 떨이 판매금액
```

- 계산: `src/features/losses/amount.ts:10-31`

### 3.5 작성자 표시명 박스 강조

작성자 입력은 현재 일반 흰 배경 필드입니다 (`src/features/ledger/components/purchase-step-client.tsx:612-651`). 저장 정책 변경 없이 이 영역만 기존 파란 강조 계열로 바꾸는 UI 요청입니다. 최초 작성자는 한 번 저장되면 잠기는 현재 정책을 유지합니다.

### 3.6 지점별 매출·마진 그래프

사진 요청:

- 이익률/매출액 전환 그래프를 하나의 매출액 기준 그래프로 합침
- 막대 옆에 매출액과 실제마진율 표시
- 매출액 버튼은 매출 내림차순, 마진율 버튼은 마진 내림차순 정렬
- 실제마진율과 예상마진율 차이가 절대 `1.5%p` 이상이면 빨간색
- `추정` 문구 제거

현재 차트는 버튼으로 한 지표만 표시하고 정렬하지 않습니다 (`src/features/reports/components/store-daily-performance-chart.tsx:45-90`, `:102-137`).

중요한 코드 불일치가 있습니다. 화면 제목·범례는 `추정`이라고 쓰지만 실제 차트 값은 이미 지점장이 입력한 `총매출`과 그 장부의 실제마진율입니다 (`src/features/reports/components/store-daily-performance-chart.tsx:28-31`, `:50-61`; `src/features/reports/queries.ts:3847-3859`). 따라서 이 그래프의 `추정` 제거는 데이터 변경 없이 문구를 바로잡는 일입니다. 예상마진율도 서버에 이미 있으므로 비교값을 차트 DTO에 연결하면 됩니다.

### 3.7 품목 판매순위 표와 그래프 제거

사진 요청을 합치면 다음 결과입니다.

- 품목 그래프 삭제
- 판매순위 표를 기존 그래프 위치로 이동
- 기본 표는 상위 10개만 표시
- 우측 검색창으로 나머지 품목 검색
- 컬럼은 `품목 / 규격 / 판매수량`만 유지
- `분류 / 추정 판매액 / 추정 원가 / 추정 마진 / 추정 이익률 / 상태` 삭제

현재 모든 품목과 모든 컬럼을 한 번에 보여주며 검색이 없습니다 (`src/features/reports/components/product-profitability-report.tsx:135-235`). 데이터는 이미 추정 판매액 내림차순입니다 (`src/features/reports/queries.ts:693-721`). 전체 데이터가 클라이언트에 있으므로 10개 제한과 검색은 새 API 없이 처리할 수 있습니다.

단, 사진에서 말하는 `상위` 기준은 확정이 필요합니다.

- 현재 순서 유지: 추정 판매액 상위 10개
- 단순 판매순위 의미: 추정 판매수량 상위 10개

또한 품목별 POS 실판매 수량은 없습니다. 현재 판매수량은 아래 식의 추정값입니다.

```text
전일재고 + 당일매입 - 손실수량 - 당일재고
```

따라서 컬럼에서 `추정`을 지우더라도 표 설명에는 `재고 흐름으로 계산한 판매수량`임을 남겨야 실제 판매수량으로 오해하지 않습니다.

## 4. 권장 처리 순서와 확인 사항

### 먼저 고칠 오류

1. 재고 수량을 입력하면 기존 필드 오류를 즉시 지우기
2. 모든 재고 차이 문구에 공통 소수 반올림·포맷 적용
3. 동일 품목의 오늘 팔 가격 수정 시 중복 행 동기화

### 적용 전 정책 확인

1. 6단계 차액만 지출을 반영할지, `totalSalesAmount`의 저장 의미까지 바꿀지
2. 대시보드에서 경보 기준 90%를 실제/예상 마진 옆 어디에 표시할지
3. 품목 상위 10개의 기준을 판매액으로 할지 판매수량으로 할지
4. 품목 그래프 삭제 범위를 사진의 일별 리포트만으로 할지, 같은 컴포넌트를 쓰는 매출 검토·품목 검토까지 포함할지
5. 손실 품목에 실제 재고가 있었던 경우 참소라/중의 품목 ID와 이월 데이터 확인
