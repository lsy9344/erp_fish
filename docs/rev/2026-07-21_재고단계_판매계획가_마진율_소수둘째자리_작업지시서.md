# 재고 단계 판매계획가·계획 마진율·소수 둘째 자리 수정 작업지시서

- 작성일: 2026-07-21
- 요청 근거: `KakaoTalk_20260720_181135262.jpg`
- 대상 사용자: 지점장
- 대상 화면: `3단계: 재고`
- 문서 범위: 이미지 요구사항 분석, 현재 코드 확인, 구현 작업 지시
- 코드 수정 여부: 이 문서 작성 시점에는 코드 수정 없음

## 1. 결론

이미지의 요청은 다음 세 가지로 확정한다.

1. 현재 `1단계: 매입`에 있는 `오늘 팔 가격(예상)` 입력을 `3단계: 재고`의 각 품목 행으로 이동한다.
2. 재고 행에서 `당일/최근 매입단가`, 사용자가 작성한 `판매계획가`, 두 가격으로 자동 계산한 `계획 마진율`을 함께 보여준다. 계획 마진율은 읽기 전용이며 사용자가 수정할 수 없다.
3. 지점장 `당일재고` 입력은 `1.23`처럼 소수점 둘째 자리까지 저장할 수 있게 한다.

현재 구현은 이 요구를 충족하지 않는다.

- 판매계획가는 재고 단계가 아니라 매입 단계에 있으며 빈칸을 허용한다.
- 재고 단계에는 계획 마진율 표시가 없다.
- `1.23`은 입력란에 타이핑할 수 있지만 저장 시 검증 오류로 거부된다.
- DB는 이미 소수점 둘째 자리 저장이 가능하므로 재고 수량용 DB 마이그레이션은 필요 없다.

## 2. 이미지 요구사항 해석

이미지에 표시된 한 품목 행의 목표 배치는 다음과 같다.

```text
품목/규격 및 당일·최근 매입단가
당일재고 | 판매계획가 | 계획 마진율(자동·수정 불가) | 당일재고 바꾼 이유
```

이미지의 메모는 다음 정책으로 해석한다.

- 판매계획가는 매일 바뀔 수 있으므로 매입 단계가 아니라 재고를 확인하는 단계에서 작성한다.
- 재고 화면에 표시된 모든 품목의 판매계획가가 작성돼야 재고 저장과 다음 단계 이동이 가능하다.
- 마진율은 매입단가와 판매계획가로 즉시 계산해 표시하고 별도 입력값으로 저장하지 않는다.
- `당일재고 바꾼 이유`의 기존 필수 조건은 유지한다. 현재 정책상 기준재고보다 많은 초과재고일 때 사유가 필요하다.

### 2.1 용어 구분

이 작업의 `계획 마진율`은 지점 전체 장부의 실제 마진율과 다른 값이다.

```text
계획 마진율 = (판매계획가 - 표시 매입단가) / 판매계획가 × 100
```

- 품목 한 행의 가격 비교용 지표다.
- 기존 본사 리포트의 `실제 마진율`, `예상 마진율`, 경보 기준 마진율을 대체하지 않는다.
- 혼동을 막기 위해 화면 라벨은 단순 `마진율`보다 `계획 마진율`을 사용한다.
- 판매계획가가 `0`이거나 표시할 매입 이력이 없으면 `계산 불가`로 표시한다.
- 퍼센트는 소수 첫째 자리까지 표시한다. 예: `30.0%`.

## 3. 현재 코드 확인 결과

### 3.1 단계와 이동

- 현재 단계 순서는 `매입 → 손실/폐기 → 재고 → 지출 → 근무인원/이름 → 매출/결제 → 검토/제출`이다 (`src/features/ledger/components/store-entry-step-navigation.tsx:18-25`).
- 재고 저장 후 다음 경로는 4단계 지출이다 (`src/features/inventory/components/inventory-step-client.tsx:370-374`).
- 하단 `다음 단계로 →` 버튼은 재고 저장 성공 직후에만 나타난다 (`src/features/inventory/components/inventory-step-client.tsx:2265-2286`).
- 상단 단계 링크는 완료 여부와 무관하게 이동할 수 있다 (`src/features/ledger/components/store-entry-step-navigation.tsx:57-97`).
- 현재 재고 완료 표시는 재고 저장 행이 한 건 이상인지로만 판정하며 판매계획가 완성 여부를 보지 않는다 (`src/features/ledger/step-completion.ts:22-38`).

따라서 하단 버튼만 제어해서는 “모두 작성 전 다음 단계 차단”을 만족하지 못한다. 상단 단계 링크를 통한 전진도 같은 규칙으로 막아야 한다.

### 3.2 판매계획가의 현재 위치와 저장

- 현재 `1단계: 매입`의 품목 행에 `오늘 팔 가격(예상)` 입력이 있다 (`src/features/ledger/components/purchase-step-client.tsx:1055-1102`).

- 현재 스키마는 빈칸을 `null`로 처리하는 선택 입력이다 (`src/features/ledger/schemas.ts:292-298`).
- 매입 화면에서 값을 비우고 저장하면 해당 화면 품목의 기존 판매계획을 삭제한다 (`src/features/ledger/actions.ts:624-637`).
- 재고 DTO와 조회에는 `plannedUnitPrice`가 없다 (`src/features/inventory/types.ts:18-41`, `src/features/inventory/queries.ts:1056-1108`).

### 3.3 재고 화면의 매입단가와 사유

- 재고 화면은 당일 매입단가를 우선하고, 없으면 가장 최근 실제 거래일의 매입단가를 조회한다 (`src/features/inventory/queries.ts:1056-1108`).

### 3.4 계획 마진율의 현재 상태

- 지점장 재고 화면에는 품목별 계획 마진율 계산이나 표시가 없다.
- 기존 타입 주석과 응답 정책은 지점장 재고 화면에서 내부 원가·마진을 제외한다고 명시한다 (`src/features/inventory/types.ts:122-131`).
- 이번 요청은 이미 허용된 `purchasePrice`와 지점장이 직접 작성한 `plannedUnitPrice`만으로 품목별 계획 마진율을 보여주는 제한적 예외다.
- 내부 `unitPrice`, FIFO lot 단가·금액, 재고금액은 계속 노출하지 않는다.

### 3.5 지점장 재고 `1.23` 저장 여부

판정: **현재 저장 불가**다. 입력값을 `1.2`로 반올림하거나 절삭하지 않고 검증 오류로 거부한다.

근거는 다음과 같다.

- 입력은 `inputMode="decimal"`인 text 입력이므로 `1.23` 문자열 자체는 타이핑할 수 있다 (`src/features/inventory/components/inventory-step-client.tsx:1771-1814`, `src/components/ui/input.tsx:5-15`).
- 입력 상태는 소수 문자열을 임의로 바꾸지 않는다 (`src/features/inventory/components/inventory-step-client.tsx:769-789`).
- 신규 입력 공용 파서는 정규식 `^\d+(?:\.\d)?$`로 소수 첫째 자리까지만 허용한다 (`src/lib/decimal.ts:15-36`).
- 서버 검증도 10배 정수 여부와 소수 한 자리 정규식을 사용하고 `roundToOneDecimal()`로 반환한다 (`src/lib/validation.ts:134-206`).
- 재고 스키마 오류 문구도 `소수점 첫째 자리까지`로 고정돼 있다 (`src/features/inventory/schemas.ts:10-35`).
- E2E 테스트는 `1.25` 저장 실패, 오류 포커스, `aria-invalid`를 정상 동작으로 검증한다 (`tests/e2e/store-ledger-inventory.spec.ts:1116-1167`).
- 단위 테스트도 신규 `2.28`을 거부하고, 과거 DB에 저장된 정확히 같은 `2.28` 문자열만 보존 예외로 둔다 (`tests/unit/ledger-inventory.test.mjs:505-528`).

저장소는 이미 둘째 자리를 지원한다.

- `LedgerInventoryItem.currentQuantity`와 `quantity`는 `Decimal(12,2)`다 (`prisma/schema.prisma:552-565`).
- 관련 재고·조정·FIFO 수량 컬럼은 이미 `NUMERIC(12,2)`로 변경됐다 (`prisma/migrations/20260706090000_decimal_inventory_quantities/migration.sql:1-39`).
- 서버 계산과 FIFO는 두 자리 반올림 경계를 이미 사용한다 (`src/server/calculations/inventory.ts`, `src/features/inventory/fifo-lots.ts`).

관련 단위 테스트를 새로 실행했으며 `tests/unit/ledger-inventory.test.mjs`, `tests/unit/ledger-validation.test.mjs`의 47개 테스트가 모두 통과했다. 즉 현재의 “신규 입력 한 자리 제한”은 우연한 현상이 아니라 테스트로 고정된 계약이다.

## 4. 확정 구현 정책

### 4.1 판매계획가 대상

- 재고 화면에 실제로 렌더링된 모든 품목 행을 필수 입력 대상으로 한다.
- 매입 품목, 전일 이월 품목, 손실 품목, 사용자가 재고 화면에서 직접 추가한 품목을 포함한다.
- 아직 화면에 추가하지 않은 `manualProductOptions`까지 강제로 작성하게 하지는 않는다.
- 지점·영업일·품목별 판매계획가는 기존 `StoreSalesPricePlan` 한 행을 계속 사용한다.
- 같은 품목이 재고 화면에 중복되지 않는 현재 구조를 유지한다.

### 4.2 판매계획가 값 계약

- 기존 데이터 계약을 유지해 `0원 이상의 정수`만 허용한다.
- 빈칸은 허용하지 않고 행별 필드 오류로 처리한다.
- `0`은 작성된 값으로 보되 계획 마진율은 `계산 불가`로 표시한다.
- 판매계획가를 금액 문자열 형식으로 보여주더라도 서버에는 쉼표 없는 정수 값으로 전달한다.

### 4.3 저장과 다음 단계 차단

- 클라이언트는 당일재고 필수 검증, 판매계획가 필수·형식 검증, 초과재고 사유 검증을 모두 통과한 뒤 저장 액션을 호출한다.
- 서버는 화면 우회나 조작된 요청을 막기 위해 현재 재고 대상 품목과 제출된 판매계획가를 다시 대조한다.
- 하나라도 누락되면 재고 수량, 사유, 다른 행의 판매계획가를 저장하지 않는다.
- 첫 누락 판매계획가로 포커스를 이동하고 입력 초안은 유지한다.
- 재고 저장과 `StoreSalesPricePlan` upsert는 같은 DB 트랜잭션에서 처리한다.
- 모든 필수값이 저장된 뒤에만 하단 다음 단계 버튼을 활성화한다.
- 상단 단계 이동은 이전 단계 이동은 허용하되, 재고보다 뒤 단계로의 전진은 같은 필수 검증을 통과해야 한다.
- `stepCompletion.inventory`는 단순 재고 행 존재 여부가 아니라 `재고 저장 + 화면 대상 판매계획가 완성`을 반영한다.

### 4.4 손실 금액 동기화

현재 단계 순서에서는 손실 입력이 재고 입력보다 먼저다. 판매계획가를 재고 단계로 옮기면 2단계 손실 저장 시점에는 당일 판매계획가가 없을 수 있다.

- 2단계에서는 현 정책대로 판매계획가가 없으면 손실액을 미산정 상태로 저장할 수 있다.
- 3단계 재고에서 판매계획가를 저장한 직후, 기존 `syncLedgerLossItemsWithSalesPricePlansInTx()` 계약을 같은 트랜잭션에서 실행한다.
- 재고 저장 응답과 7단계 검토·본사 리포트에는 동기화된 손실 금액이 반영돼야 한다.
- 재고 저장 후 손실 화면 캐시도 무효화해 뒤로 이동했을 때 최신 판매계획가와 손실 금액을 보여준다.

### 4.5 계획 마진율 계산과 보안 경계

- 계획 마진율은 별도 DB 컬럼으로 저장하지 않는다.
- 이미 지점장에게 허용된 `purchasePrice.unitPrice`와 입력 중인 `plannedUnitPrice`만으로 계산한다.
- 값 변경 시 즉시 다시 계산한다.
- 마진율 입력 요소를 만들지 않고 읽기 전용 텍스트 또는 `output` 성격의 표시 요소를 사용한다.
- `purchasePrice`가 없거나 판매계획가가 0이거나 입력 형식이 잘못됐으면 `계산 불가`를 표시한다.
- 내부 `unitPrice`, FIFO lot 단가·금액, 재고금액, 지점 전체 실제 마진율은 지점장 응답에 추가하지 않는다.
- 민감정보 테스트의 “마진율 문자열 전면 금지”가 있다면 품목별 `계획 마진율` 예외와 내부 회계 마진 차단을 구분하도록 좁힌다.

### 4.6 당일재고 소수 둘째 자리

- 지점장 재고의 `currentQuantity`와 같은 화면에서 함께 보내는 `quantity`에 한해 최대 소수 둘째 자리까지 허용한다.
- 허용 예: `0`, `1`, `1.2`, `1.23`.
- 거부 예: `-1`, `.5`, `1.`, `1.234`, `1,000`, `1e2`, 공백만 있는 값, 상한 초과.
- 최대값은 DB `Decimal(12,2)`와 화면 상수에 맞춰 `9,999,999,999.99`로 통일한다.
- 오류 문구는 `재고 수량은 0 이상이고 소수점 둘째 자리까지 입력할 수 있습니다.`로 바꾼다.
- 매입·손실 신규 수량 입력은 기존 소수 첫째 자리 정책을 유지한다.
- 공용 검증 정규식만 전역으로 두 자리로 넓히지 않는다. 자리수 인자를 기본값 1로 추가하거나 재고 전용 2자리 parser를 둔다.
- 기존 DB의 두 자리 수량을 수정하지 않고 다시 저장하는 호환성도 유지한다.

## 7. 필수 테스트와 합격 기준

### 7.1 소수 둘째 자리

- [ ] 지점장 당일재고 `1.23` 저장이 성공한다.
- [ ] 저장 후 새로고침해 입력값이 `1.23`으로 다시 표시된다.
- [ ] DB의 `currentQuantity`와 `quantity`가 `1.23`이다.
- [ ] `1.234`, 음수, 쉼표, 지수표기, 상한 초과는 행 오류로 거부된다.
- [ ] 잘못된 값을 자동으로 반올림하거나 절삭하지 않는다.
- [ ] 기존 저장 두 자리 값은 무변경 저장과 재조회에서 보존된다.
- [ ] 매입과 손실의 신규 `1.25` 입력은 계속 거부된다.

### 7.2 판매계획가 이동과 필수 처리

- [ ] 1단계 매입 화면에 판매계획가 입력이 없다.
- [ ] 3단계 재고의 모든 표시 행에 판매계획가 입력이 있다.
- [ ] 기존 당일 판매계획가가 있으면 재고 화면에 채워진다.
- [ ] 빈 판매계획가가 한 건이라도 있으면 저장하지 않고 첫 누락 필드로 포커스한다.
- [ ] 누락 오류 뒤 다른 재고 수량·사유·판매계획가 초안이 사라지지 않는다.
- [ ] 모든 표시 행을 채우면 재고와 판매계획이 한 번에 저장된다.
- [ ] 동일 지점·영업일·품목은 `StoreSalesPricePlan` 한 건만 유지한다.
- [ ] 값을 바꾸고 다시 저장하면 기존 행이 upsert로 갱신된다.
- [ ] 재고보다 뒤 단계로 가는 상단 링크도 누락 상태를 우회하지 못한다.
- [ ] 이전 단계로 돌아가는 이동은 허용한다.

### 7.3 계획 마진율

- [ ] 매입단가 50,000원, 판매계획가 71,429원은 약 `30.0%`로 표시된다.
- [ ] 판매계획가를 바꾸면 저장 전에도 즉시 다시 계산된다.
- [ ] 계획 마진율은 키보드 포커스 가능한 편집 input이 아니다.
- [ ] 판매계획가 0원 또는 매입 이력 없음은 `계산 불가`다.
- [ ] 계획 마진율을 DB에 별도 저장하지 않는다.
- [ ] 내부 FIFO 단가·금액, 재고금액, 지점 전체 실제 마진율은 지점장 응답에 새로 노출되지 않는다.

### 7.4 기존 업무 회귀

- [ ] 초과재고의 `당일재고 바꾼 이유` 필수 규칙이 유지된다.
- [ ] 정상 판매로 해석되는 기준재고 이하 입력은 사유 없이 저장된다.
- [ ] 재고 저장 뒤 손실 금액이 새 판매계획가 기준으로 다시 계산된다.
- [ ] 7단계 검토와 본사 리포트가 새 판매계획가를 사용한다.
- [ ] 저장 충돌 시 재고·판매계획가 초안을 유지하고 최신값 다시 불러오기를 제공한다.
- [ ] 휴무·본사 마감 장부의 읽기 전용 정책이 유지된다.
