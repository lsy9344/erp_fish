## 2. 현행 코드 검토 결과

### 2.1 본사에서 지점 인건비를 보는 전용 진입점이 없다

**Evidence**

- 본사 사이드바에는 홈, 리포트, 본사 지출, 기준정보 등이 있으나 인건비 전용 항목이 없다 (`src/components/app-sidebar.tsx:20-97`).
- 리포트 상단 메뉴에도 아침 회의, 통합, 비교, 월간, 재고 등의 항목만 있고 인건비 항목은 없다 (`src/features/reports/components/reports-nav.tsx:5-63`).
- 지점장이 입력한 인건비 원본은 `LedgerLaborItem`의 `workerName`, `amount`, 지각/조퇴/특이사항으로 저장된다 (`prisma/schema.prisma:724-750`).
- `DailyLedger`에는 영업일, 지점, 상태, 근무인원과 `ledgerLaborItems` 관계가 있다 (`prisma/schema.prisma:420-470`).
- 월간 손익은 이미 `LedgerLaborItem.amount`를 지점별로 합산한다 (`src/features/reports/monthly-profit-loss.ts:203-255`). 데이터는 있으나 상세 조회 화면이 없는 상태다.
- 직원 급여 화면은 `employeeId`가 연결된 행을 직원별로 묶는 별도 기능이므로, 지점장이 자유 입력한 미연결 근무자까지 보여줘야 하는 이번 요구를 대신하지 못한다 (`src/features/labor/employees-queries.ts:132-176`).
- 지점장 응답에서는 급여 금액과 합계를 제거한다 (`src/features/ledger/types.ts:82-96`, `src/features/ledger/response-shaping.ts`). 새 화면은 반드시 본사 전용 조회 계약을 사용해야 한다.

**판정**

- 새 DB 테이블은 필요 없다.
- 읽기 전용 리포트이므로 기존 `REPORT_VIEW` 권한을 사용한다.
- `getHeadquartersStoreScope()`의 `ALL_STORES`/`ASSIGNED_STORES` 범위를 적용하지 않으면 담당 외 지점의 급여가 노출될 수 있다 (`src/server/authz.ts:253-287`).

### 2.2 손실 수량은 애플리케이션에서 소수 첫째 자리로 제한한다

**Evidence**

- 손실 수량 입력은 decimal 키보드를 사용하지만 저장 직전에 `toStockQuantitySaveInput()`을 사용한다 (`src/features/losses/components/loss-step-client.tsx:299-302,679-703`).
- 공용 초안 파서는 `^\d+(?:\.\d)?$` 정규식으로 소수 첫째 자리까지만 허용한다 (`src/lib/decimal.ts:15-38`).
- 서버 파서도 10배 정수 검사와 `roundToOneDecimal()`을 사용한다 (`src/lib/validation.ts:134-206`).
- 손실 오류 문구도 `소수점 첫째 자리까지`로 고정돼 있다 (`src/features/losses/terms.ts:21-22`).
- 본사 장부 수정 액션은 손실 수량을 마지막 단계에서 정수로 다시 검사해 더 좁은 제한을 둔다 (`src/features/losses/hq-edit-actions.ts:156-188`).
- 반면 `LedgerLossItem.quantity`는 이미 `Decimal(12,2)`이므로 소수 둘째 자리 저장이 가능하다 (`prisma/schema.prisma:688-699`).
- 재고 입력에는 별도의 소수 둘째 자리 초안 파서가 이미 있다 (`src/lib/decimal.ts:16,53-69`).

**판정**

- DB 마이그레이션은 필요 없다.
- 공용 한 자리 파서를 전역으로 넓히면 매입 수량 정책까지 바뀌므로, 손실 전용 두 자리 계약을 추가해야 한다.
- 지점장이 `1.25`를 저장한 뒤 본사에서 같은 장부를 수정할 수 있어야 하므로 본사 수정 경로도 같은 두 자리 계약을 사용해야 한다.

### 2.3 지점장 손실 품목 목록은 본사 전체 활성 품목이다

**Evidence**

- 손실 조회는 `Product.isActive = true`인 전체 품목을 지점 구분 없이 조회한다 (`src/features/losses/queries.ts:71-104`).
- 지점장 화면은 전달받은 `data.productOptions` 전체를 `<select>`에 표시한다 (`src/features/losses/components/loss-step-client.tsx:536-600`).
- 저장 액션은 선택된 품목이 활성 품목인지만 먼저 확인한다 (`src/features/losses/actions.ts:255-290`).
- 이후에야 `getInventoryStepDataInTx()` 결과와 손실 수량을 대조해 재고가 없는 품목 또는 재고 초과 수량을 거부한다 (`src/features/losses/actions.ts:383-437`).

**판정**

- 현재는 화면에서 전체 품목을 보여주고 저장 시점에만 재고를 검사해 사용자가 불필요한 품목을 고를 수 있다.
- 화면 옵션과 저장 검증이 같은 재고 계산을 사용하도록 정리해야 한다.
- 본사 수정 화면은 과거 행 복원과 예외 정정이 필요하므로 전체 활성 품목 정책을 유지하고, 지점장 응답만 제한한다.

### 2.4 판매한 가격은 당일 날짜의 행만 조회한다

**Evidence**

- 재고 화면은 `item.plannedUnitPrice`를 입력 초깃값으로 사용하며 사용자가 수정할 수 있다 (`src/features/inventory/components/inventory-step-client.tsx:205-216,2141-2165`).
- 재고 조회는 `storeId + ledger.closingDate + productId`가 정확히 일치하는 당일 `StoreSalesPricePlan`만 조회한다 (`src/features/inventory/queries.ts:1079-1137`).
- 다음 영업일 장부 생성은 빈 `DailyLedger`만 만들고 판매한 가격 행을 복사하지 않는다 (`src/features/ledger/queries.ts:440-470`).
- 재고 수량 이월은 최근 장부의 재고 행을 사용하지만 판매한 가격을 이월하지 않는다 (`src/features/inventory/queries.ts:733-825`).
- 저장 시에는 `(storeId, businessDate, productId)` 기준으로 당일 가격을 upsert한다 (`src/features/inventory/actions.ts:175-207,627-639`).
- `StoreSalesPricePlan`은 영업일별 독립 행이며 지점·영업일·품목 unique 제약이 있다 (`prisma/schema.prisma:773-800`).

**판정**

- 다음날 빈칸이 되는 직접 원인은 당일 exact-date 조회만 하는 데 있다.
- 조회 시 전일 값을 초깃값으로 사용하고 저장 시 당일 행을 만드는 방식이면 기존 영업일별 이력과 수정 가능 요구를 모두 유지할 수 있다.
- 본사 대시보드·리포트 계산의 exact-date 정책은 이번 요청 범위가 아니다. 지점장 재고 화면에서 당일 저장을 완료하면 기존 리포트 조회는 당일 행을 그대로 사용한다.

## 3. 확정 구현 정책

### 3.1 본사 `인건비 현황`

#### 진입점과 권한

- 새 경로는 `/app/reports/labor`로 한다.
- 본사 사이드바의 `리포트` 바로 아래에 `인건비 현황` 직접 메뉴를 추가한다.
- 리포트 상단 `ReportsNav`의 `기간 분석` 그룹에도 `인건비` 항목을 추가한다.
- 두 메뉴와 페이지 모두 `PermissionAction.REPORT_VIEW`를 사용한다.
- 페이지 진입 시 `requireReportAccess()`와 `getHeadquartersStoreScope()`를 적용한다.
- `ASSIGNED_STORES` 사용자는 배정 지점만 조회할 수 있고 URL 검색조건을 조작해도 다른 지점 데이터가 나오면 안 된다.

#### 조회 범위

- 기본 기간은 현재 월 1일~말일이다.
- 필터는 `월`, `지점(전체/단일)`, `장부 상태`를 제공한다.
- “입력한 값 확인” 목적이므로 `IN_PROGRESS`, `IN_REVIEW`, `HEADQUARTERS_CLOSED`를 조회하고 `HOLIDAY`는 제외한다.
- 미완료 장부는 숨기지 않고 `작성 중` 상태 배지를 표시해 확정값과 구분한다.
- `employeeId`가 없는 자유 입력 근무자도 `workerName` 스냅샷 기준으로 반드시 표시한다.

#### 표시 항목

- 상단 요약: 조회기간 인건비 합계, 조회 지점 수, 근무 기록 수.
- 지점 요약: 지점명, 근무일 수, 근무인원 합계, 인건비 합계.
- 상세: 영업일, 지점, 장부 상태, 근무자명, 인건비, 지각, 조퇴, 특이사항.
- 금액은 정수 원화 형식으로 표시한다.
- 같은 지점/영업일의 상세 합계는 해당 장부의 `LedgerLaborItem.amount` 합계와 같아야 한다.
- 이번 화면은 읽기 전용이다. 수정, 삭제, 본사 강제 변경 기능과 export는 추가하지 않는다.

### 3.2 손실 `박스단위 수량` 소수 둘째 자리

- 허용 예: `0`, `1`, `1.2`, `1.25`, `0.01`, `9999999999.99`.
- 거부 예: `-1`, `.5`, `1.`, `1.234`, `1,000`, `1e2`, 공백만 있는 값, `9999999999.991`.
- 허용 상한은 DB `Decimal(12,2)`에 맞춰 `9,999,999,999.99`로 한다.
- 저장값은 절삭하지 않는다. 허용된 두 자리 값을 `roundToTwoDecimals()` 경계에서 정규화한다.
- 오류 문구는 `박스단위 수량은 0 이상이고 소수점 둘째 자리까지 입력할 수 있습니다.`로 바꾼다.
- 지점장 손실 저장과 본사 손실 수정에서 동일한 두 자리 검증을 사용한다.
- `떨이로 실제 판매한 금액`은 기존과 같이 0원 이상의 정수만 허용한다.
- 매입 수량과 그 밖의 기존 `parseStockQuantityDraft()` 사용처는 소수 첫째 자리 정책을 유지한다.
- 기존 `0.1` 기반 설명 예시는 유효하므로 도움말은 유지하되, 둘째 자리도 가능하다는 짧은 안내를 덧붙인다.

### 3.3 지점장 손실 품목을 보유 재고로 제한

#### 지점 보유 재고 정의

지점장 신규 선택 목록에는 아래 조건을 모두 만족하는 품목만 포함한다.

1. 해당 영업일 재고 계산 대상 품목이다.
2. `전일 이월 + 당일 매입 - 현재 저장된 손실`로 계산한 가용 박스 수량이 `0`보다 크다.
3. 품목이 현재 active 상태다.

재고 수량 계산은 `getInventoryStepDataInTx()`와 `calculateSystemInventoryQuantity()`가 사용하는 기존 기준을 재사용한다. 손실 화면만의 별도 수량 공식을 만들지 않는다.

#### 기존 행 보존

- 이미 저장된 손실 행의 품목은 현재 재고가 0이거나 품목이 비활성이어도 해당 행에서 계속 보인다.
- 이런 기존 행은 수량/사유 수정 또는 삭제가 가능해야 한다.
- 새 행에서 재고가 없는 품목을 선택하게 하지는 않는다.
- 조작된 요청으로 신규 행 또는 품목 변경 행에 비대상 품목 ID를 보내면 서버가 `VALIDATION_ERROR`로 거부한다.
- 기존 행이 같은 `id + productId`를 유지하는 경우에는 과거 스냅샷 복원 정책을 유지한다.
- 기존 총 손실 수량이 가용 재고를 초과하지 않는다는 서버 검증은 삭제하지 않는다.

#### 본사 화면 분리

- `getLossStepDataByLedgerId()`를 사용하는 본사 장부 수정 화면의 전체 활성 품목 옵션은 유지한다.
- 지점장용 `StoreManagerLossStepData.productOptions`만 보유 재고 기준으로 제한한다.
- 지점장 응답에서 단가·손실액을 제거하는 현재 보안 shaping도 유지한다.

### 3.4 다음 영업일 판매한 가격 이월

#### 이월 원본

- 현재 영업일보다 이전인 장부 중 상태가 `IN_REVIEW` 또는 `HEADQUARTERS_CLOSED`인 가장 최근 영업일 한 건을 찾는다.
- 휴무일과 `IN_PROGRESS` 초안은 이월 원본으로 사용하지 않는다.
- 월 경계가 바뀌어도 가장 최근 제출 영업일이면 이월할 수 있다.
- 원본 영업일의 `StoreSalesPricePlan` 중 현재 재고 화면에 표시되는 같은 품목의 값만 사용한다.
- 새 품목처럼 원본 영업일에 값이 없던 품목은 빈칸으로 두고 현재 필수 검증을 유지한다.

#### 조회 우선순위

품목별 표시값은 다음 순서로 결정한다.

1. 현재 영업일에 이미 저장된 `StoreSalesPricePlan.plannedUnitPrice`.
2. 현재 값이 없을 때만 최근 제출 영업일의 같은 품목 `plannedUnitPrice`.
3. 둘 다 없으면 `null`/빈칸.

현재 영업일 값을 한 번이라도 저장했다면 이후 재방문에서는 전일 값보다 현재 값이 항상 우선한다.

#### 저장과 수정

- 페이지 조회만으로 다음날 계획 행을 자동 생성하지 않는다.
- 사용자가 전일 값을 그대로 사용하거나 수정한 뒤 `재고 저장`을 누르면 기존 `upsertInventorySalesPricePlansInTx()`를 통해 현재 영업일 행으로 저장한다.
- 현재 영업일 저장은 원본 영업일의 행을 수정하지 않는다.
- 모든 표시 재고 행의 판매한 가격 필수 검증, 손실액 재동기화, 충돌 처리, 감사 로그는 현재 계약을 유지한다.
- `getInventoryPlanGateForLedgerInTx()`는 당일 저장 완료 여부를 계속 exact-date 행으로 판단한다. 전일 값이 화면에 보이기만 한 상태를 저장 완료로 간주하지 않는다.
- 본사 대시보드와 리포트의 판매한 가격 조회에 임의 전일 fallback을 추가하지 않는다.

## 5. 필수 수용 기준

### 5.1 본사 인건비

- [ ] `REPORT_VIEW` 권한이 있는 본사 사용자는 사이드바와 리포트 메뉴에서 `인건비 현황`을 볼 수 있다.
- [ ] 권한이 없는 사용자는 메뉴를 볼 수 없고 직접 URL 접근도 거부된다.
- [ ] 배정 지점 사용자는 담당 지점만 조회한다.
- [ ] 작성 중/검토 중/본사 마감 상태가 구분되어 보인다.
- [ ] `employeeId = null`인 지점장 자유 입력 근무자도 누락되지 않는다.
- [ ] 일자 상세 합계, 지점 합계, 전체 합계가 원본 `LedgerLaborItem.amount` 합계와 일치한다.
- [ ] 화면에 수정/삭제 기능이 없다.
- [ ] 지점장 화면에서는 기존처럼 개인별 인건비 금액이 노출되지 않는다.

### 5.2 손실 수량

- [ ] 지점장 손실 화면에서 `0.01`, `1.25`를 저장하고 다시 열면 같은 값이 보인다.
- [ ] `1.234`, 음수, 지수 표기, 쉼표 값, 상한 초과는 클라이언트와 서버가 모두 거부한다.
- [ ] 오류 문구가 소수점 둘째 자리 기준이다.
- [ ] 본사 손실 수정도 두 자리 값을 저장할 수 있다.
- [ ] 떨이 실제 판매 금액은 계속 정수만 허용한다.
- [ ] 매입 수량의 신규 `1.25`는 기존 정책대로 거부된다.
- [ ] DB migration이 추가되지 않는다.

### 5.3 지점 재고 품목 선택

- [ ] A지점 손실 품목 목록에 B지점만 보유한 품목이 나오지 않는다.
- [ ] A지점의 현재 가용 재고가 0인 품목은 신규 선택 목록에 나오지 않는다.
- [ ] 전일 이월 또는 당일 매입으로 가용 재고가 생긴 품목은 목록에 나온다.
- [ ] 기존 저장 손실 행은 재고 소진/품목 비활성 후에도 이름과 규격이 보이며 수정 또는 삭제할 수 있다.
- [ ] 비대상 productId를 조작해 전송하면 서버가 저장을 거부한다.
- [ ] 본사 장부 수정 화면의 전체 활성 품목 옵션은 유지된다.

### 5.4 판매한 가격 이월

- [ ] D일 리포트를 제출한 뒤 D+1 재고 화면을 열면 공통 품목의 D일 판매한 가격이 입력값으로 보인다.
- [ ] D+1 화면을 열기만 해서는 D+1 `StoreSalesPricePlan` 행이 생성되지 않는다.
- [ ] D+1에서 값을 그대로 저장하면 D+1 날짜의 별도 행이 생성된다.
- [ ] D+1에서 값을 바꿔 저장하면 D+1에는 새 값, D일에는 원래 값이 유지된다.
- [ ] D+1 값을 저장한 뒤 재방문하면 전일 값이 아니라 D+1 저장값이 보인다.
- [ ] D일에 없던 신규 품목은 빈칸이며 기존 필수 검증을 받는다.
- [ ] 이전 장부가 휴무 또는 작성 중이면 그 값은 이월 원본으로 쓰지 않는다.
- [ ] 월이 바뀌어도 최근 제출 영업일의 값은 이월된다.
- [ ] 전일 fallback만 표시한 상태는 재고 단계 완료로 판정되지 않으며, 당일 저장 후 완료된다.
- [ ] 본사 리포트와 대시보드의 exact-date 계산 계약은 임의로 변경되지 않는다.

- Task 1~5의 변경과 수용 기준이 모두 구현돼 있다.
- 신규 동작과 회귀 동작을 단위 테스트와 E2E로 증명했다.
- lint, typecheck, build가 통과했다.
- 새 DB migration과 dependency가 추가되지 않았다.
- 본사 담당 지점 scope와 지점장 민감정보 redaction이 유지된다.
- 구현 결과 문서에 변경 파일, 테스트 결과, 남은 운영 정책 이슈를 기록했다.
