## 1. 오늘 리뷰 목적

오늘 리뷰는 "새 기능 설명회"라기보다, 지난번 회의에서 나온 요청이 실제 화면과 업무 흐름에 어떻게 반영됐는지 같이 확인하는 자리다.

의뢰자에게는 아래 흐름으로 설명한다.

> 지난번 회의에서 대표님이 말씀하신 핵심은, 지점별 장부를 매일 쉽게 입력하고 본사는 전체 지점의 이상 신호를 빨리 보는 것이었습니다.  
> 그래서 이번에는 지점장은 복잡한 회계 화면을 보지 않고 입력만 쉽게 하고, 본사는 이카운트, 재고, 손실, 근무, 리포트, 알림까지 한 화면 흐름으로 관리하는 방식으로 반영했습니다.

## 2. 전체 사용 컨셉

### 한 줄 컨셉

**지점장은 매일 장부를 쉽게 입력하고, 본사는 전체 지점의 숫자와 이상 신호를 확인해서 바로 조치하는 시스템**이다.

### 쉽게 말하면

- 예전 방식: 본사가 여러 지점의 엑셀과 장부를 하나씩 열어보고, 이상한 숫자를 사람이 찾아야 했다.
- 바뀐 방식: 지점이 매일 같은 순서로 입력하면, 본사는 관제판과 리포트에서 미입력, 마감 대기, 손실, 마진 문제, 재고 문제를 바로 본다.
- 중요한 원칙: 지점장 화면은 단순하게, 본사 화면은 판단 근거까지 자세하게 만든다.

## 3. 리뷰 진행 순서

1. 로그인과 권한 분리 확인
2. 본사 관제판에서 지점 상태와 이상 신호 확인
3. 지점장 장부 입력 7단계 확인
4. 이카운트 업로드와 지점 입고 반영 흐름 확인
5. 재고, FIFO, 판매 예정가, 손실 계산 컨셉 확인
6. 리포트, 냉동/생물 분석, 품목별 이익률 추정 확인
7. 직원/근무/인건비, 본사 지출, LINE 알림 확인
8. 오늘 의뢰자에게 최종 확인받을 운영 기준 정리

## 4. 발표용 핵심 설명

### 4.1 본사와 지점장을 나눴습니다

지난번 회의에서 본사는 전체 지점 데이터를 보고, 지점장은 자기 지점 입력만 해야 한다는 이야기가 있었다. 그래서 계정과 권한을 분리했다.

- 본사는 지점 등록, 사용자 권한, 이카운트 업로드, 리포트, 본사 지출, 변경 이력을 본다.
- 지점장은 장부, 재고, 손실 입력만 한다.
- 폐업하거나 사용하지 않는 지점은 본사에서 비활성화할 수 있다.

사용 컨셉:

> 본사는 관제실이고, 지점장은 매일 입력하는 현장 화면입니다.

리뷰 체크:

- [ ] 본사 계정으로 로그인했을 때 본사 메뉴가 보이는지 확인
- [ ] 지점장 계정으로 로그인했을 때 지점장 메뉴만 보이는지 확인
- [ ] 사용자/권한에서 지점장 계정과 지점 연결을 확인
- [ ] 지점 활성/비활성 처리가 가능한지 확인

### 4.2 본사 관제판은 "전 지점 오늘 상태판"으로 정리했습니다

지난번 회의에서 본사가 전날 실적과 문제를 한눈에 보고 싶다고 했다. 그래서 본사 홈은 지점별 상태와 이상 신호를 보는 관제판 역할로 정리했다.

반영 내용:

- 지점별 상태를 `미입력`, `입력 중`, `검토 대기`, `본사 마감`처럼 구분한다.
- 문제가 있는 지점은 이상 신호로 드러난다.
- 대표님이 말씀하신 직관적인 문구를 반영했다.
  - 매출 기준 확인 필요
  - 이익률 계산 불가
  - 매출 차액 계산 불가
  - 재고 입력 필요
- 화면은 자유 드래그 방식 대신 운영 안정성이 높은 화면 밀도 조절과 컬럼 폭 조절 방식으로 정리했다.

사용 컨셉:

> 본사는 매일 아침 이 화면을 보고, 어느 지점이 아직 안 넣었는지, 어느 지점 숫자가 이상한지 먼저 확인합니다.

리뷰 체크:

- [ ] 지점별 장부 상태가 구분되어 보이는지 확인
- [ ] 미입력 지점과 검토 대기 지점이 구분되는지 확인
- [ ] 이상 신호 문구가 일반인이 봐도 이해되는지 확인
- [ ] 마진 기준 또는 이상 신호 기준값을 조정할 수 있는지 확인
- [ ] 표 컬럼과 화면 밀도가 실제 사무실 모니터에서 보기 좋은지 확인

### 4.3 지점장 장부 입력은 7단계로 고정했습니다

지난번 회의에서 지점장들이 전문 회계 화면을 쓰는 것이 아니라, 매일 필요한 것만 순서대로 입력해야 한다는 흐름이 있었다. 그래서 지점장 입력 화면을 7단계로 정리했다.

반영된 7단계:

| 단계 | 화면 | 사용 목적 |
| --- | --- | --- |
| 1단계 | 매출/결제 | 총매출, 결제수단, 작성자 표시명 입력 |
| 2단계 | 비용 | 식대, 얼음, 주유비, 비품 등 현장 비용 입력 |
| 3단계 | 매입 | 당일 입고/매입 품목과 `오늘 팔 가격(예상)` 입력 |
| 4단계 | 손실/폐기 | 폐기, 떨이, 손실 수량과 금액 확인 |
| 5단계 | 재고 | 당일 남은 재고 입력, 당일 판매량 확인 |
| 6단계 | 근무/인건비 | 근무인원, 직원별 급여/인건비, 특이사항 입력 |
| 7단계 | 검토/제출 | 저장한 숫자와 경고를 보고 본사에 제출 |

작성자 표시명 반영:

- 1단계에서 작성자 표시명을 필수로 받는다.
- 한 번 저장된 최초 작성자명은 이후 단계와 재방문에서도 유지된다.
- 본사 수정자는 별도 변경 이력으로 추적한다.

사용 컨셉:

> 지점장은 매일 같은 순서로 입력하고 마지막에 제출만 하면 됩니다. 누가 처음 작성했는지도 남습니다.

리뷰 체크:

- [ ] 1단계에서 작성자 표시명 없이 저장이 막히는지 확인
- [ ] 작성자 표시명이 2~7단계에서도 유지되는지 확인
- [ ] 3단계 매입 화면에서 `오늘 팔 가격(예상)`을 입력할 수 있는지 확인
- [ ] 6단계가 `근무/인건비`로 보이고, 근무인원과 급여 행 기준 참고 인원이 구분되는지 확인
- [ ] 7단계에서 제출 전 확인 문구가 이해되는지 확인

### 4.4 지점장에게 민감한 회계 숫자는 줄였습니다

지난번 회의에서 지점장이 원가, 영업이익, 생산성 같은 숫자를 보면 오해하거나 숫자를 맞추려 할 수 있다는 이야기가 있었다. 그래서 지점장 화면은 현장 운영에 필요한 값 위주로 줄였다.

반영 내용:

- 지점장에게는 현장 입력값과 요약 정보 중심으로 보여준다.
- 매출원가, 매출이익, 영업이익, 인당 생산성, FIFO 원가 근거 같은 본사 판단용 값은 지점장 화면에서 숨긴다.
- 7단계 경고도 조작을 부르는 말투 대신, 확인해야 할 내용 중심으로 보여준다.
- 손실이 섞인 재고 부족은 `재고가 틀렸다`가 아니라 `판매 추정 확인`처럼 현장 의미에 맞게 표현한다.

사용 컨셉:

> 지점장은 입력과 현장 확인에 집중하고, 민감한 수익성 판단은 본사가 봅니다.

리뷰 체크:

- [ ] 지점장 검토 화면에 불필요한 원가/영업이익이 보이지 않는지 확인
- [ ] 본사 장부 상세에서는 필요한 판단 근거를 볼 수 있는지 확인
- [ ] `판매 추정 확인` 같은 문구가 현장 직원에게 자연스러운지 확인

### 4.5 이카운트 파일은 "본사 출고 / 지점 입고 원장"으로 반영했습니다

지난번 회의에서 수산물은 단가가 매일 바뀌고, 같은 품목도 지점과 날짜에 따라 금액이 달라질 수 있다는 이야기가 있었다. 그래서 이카운트 파일을 단순한 품목별 기준 단가표로 보지 않고, 본사가 어느 지점에 무엇을 얼마에 보냈는지 남기는 원장으로 반영했다.

반영 내용:

- 본사 메뉴에 `이카운트 업로드`가 있다.
- 하루 전체 이카운트 파일을 올리면 지점별 입고 라인으로 미리보기한다.
- 거래처명과 지점명, 품목명과 앱 품목이 다를 수 있으므로 매핑을 둔다.
- 매핑이 끝나야 장부에 반영할 수 있다.
- 같은 파일을 다시 올리는 중복 업로드는 막는다.
- 원본 이카운트 단가와 장부 적용 단가를 분리한다.
- 원본은 보존하고, 본사 보정이나 적용 단가 변경은 이력으로 남긴다.

사용 컨셉:

> 이카운트 파일은 그냥 가격표가 아니라, 본사가 지점에 보낸 물건의 증빙입니다. 그래서 원본을 보존하고 지점 장부에 연결합니다.

리뷰 체크:

- [ ] 이카운트 업로드 메뉴가 본사에서만 보이는지 확인
- [ ] 실제 파일을 올렸을 때 미리보기, 매핑 필요, commit 가능 상태가 이해되는지 확인
- [ ] 거래처명과 지점명 매핑 방식이 실제 운영 파일과 맞는지 확인
- [ ] 품목명/규격 매핑이 실제 품목 마스터와 맞는지 확인
- [ ] 원본 단가와 장부 적용 단가를 구분해서 설명할 수 있는지 확인

### 4.6 재고와 FIFO는 "오래된 재고부터 팔린 것으로 본다"는 기준을 잡았습니다

지난번 회의에서 같은 품목도 단가가 다를 수 있고, 재고 금액을 선입선출로 잡을지 평균으로 잡을지 이야기했다. 현재 기준은 이카운트와 비슷하게 **먼저 들어온 물건이 먼저 팔린 것으로 보는 선입선출(FIFO)** 기준이다.

반영 내용:

- FIFO 기준은 모든 재고 품목에 적용하는 방향으로 정책을 정리했다.
- 아침 본사 출고/지점 입고 물량은 당일 판매 전 들어온 물건으로 본다.
- 판매 수량은 `전일 재고 + 당일 매입 - 손실 - 당일 재고` 흐름으로 역산한다.
- 판매 예정가와 FIFO 원가는 서로 다른 역할로 분리했다.
  - 판매 예정가: 오늘 얼마에 팔려고 했는지
  - FIFO 원가: 그 물건이 얼마짜리 재고였는지
- 지점장 화면에는 복잡한 원가 근거를 노출하지 않고, 본사 판단이나 리포트에서 근거를 확인한다.

사용 컨셉:

> 재고는 "몇 개 남았나"만 보는 것이 아니라, 어떤 단가의 물건이 남았는지까지 본사가 추적할 수 있게 만드는 방향입니다.

리뷰 체크:

- [ ] 재고 화면에서 `당일 판매량` 표현이 자연스러운지 확인
- [ ] 정상 판매인데 조정 사유를 요구하지 않는지 확인
- [ ] 초과 재고, 손실이 섞인 차이, 매입 없는 이월 차이는 조정 사유를 요구하는지 확인
- [ ] 직접 추가해야 할 품목과 기본 표에 보이는 품목 구분이 이해되는지 확인
- [ ] FIFO 설명이 대표님에게 너무 복잡하지 않은지 확인

### 4.7 손실과 판매 예정가는 "어디서 새는지 보기 위한 기준"으로 연결했습니다

지난번 회의에서 손실액은 단순 매입원가가 아니라, 원래 팔려고 했던 가격과의 차이를 봐야 한다는 요구가 있었다. 그래서 지점장이 3단계 매입에서 `오늘 팔 가격(예상)`을 입력하게 했다.

반영 내용:

- `오늘 팔 가격(예상)`은 3단계 매입 화면에 통합했다.
- 별도 판매가 계획 메뉴를 찾아갈 필요가 없다.
- 7단계 검토 화면의 `오늘 많이 팔린 품목`과 추정 매출 계산에 판매 예정가를 우선 사용한다.
- 판매 예정가가 없으면 매입 단가로 대체하되, `판매가 미반영`처럼 표시해 확정값처럼 보이지 않게 했다.

사용 컨셉:

> 아침에 "이 품목은 오늘 얼마에 팔자"라고 정하고, 저녁에 실제 장부와 비교해서 어디서 차이가 났는지 보는 구조입니다.

리뷰 체크:

- [ ] 3단계에서 매입 단가, 수량, 오늘 팔 가격이 한 줄 흐름으로 이해되는지 확인
- [ ] 같은 품목의 판매 예정가를 하루 하나로 관리하는 정책이 맞는지 확인
- [ ] 7단계에서 판매가 미반영 품목을 다시 3단계로 고칠 수 있는지 확인

### 4.8 리포트는 "확정값"과 "추정값"을 구분했습니다

지난번 회의에서 냉동/생물 구분, 품목별 이익률, 하루 전체 이익률을 그래프로 보고 싶다는 요청이 있었다. 다만 현재 시스템은 POS처럼 품목별 실제 판매 단가를 직접 받지는 않는다. 그래서 확정 매출처럼 쓰지 않고 `추정`으로 표시한다.

반영 내용:

- 일별 리포트에 `아침 회의 리포트`가 있다.
- 일별/월간 리포트에서 `냉동/생물 매출 (추정)`을 볼 수 있다.
- 일별 리포트에서 `품목별 이익률 (추정)`을 볼 수 있다.
- 추정 계산은 재고 흐름, 판매 예정가, FIFO 원가를 사용한다.
- 품목별 POS 매출이 없다는 설명을 화면에 같이 표시한다.
- 남은 재고 리포트에서 지점별/품목별 잔여 재고를 모아볼 수 있다.
- 이카운트 출고/입고 리포트에서 업로드로 반영된 지점별 입고 라인을 확인할 수 있다.

사용 컨셉:

> POS처럼 품목별 실제 판매가를 받은 것은 아니기 때문에, 지금은 "확정 수익"이 아니라 "장부와 재고 흐름으로 추정한 경영 판단 자료"로 봅니다.

리뷰 체크:

- [ ] 일별 아침 회의 리포트가 대표님이 매일 볼 내용으로 충분한지 확인
- [ ] 냉동/생물 매출 차트가 보기 좋은지 확인
- [ ] 품목별 이익률 추정 표가 너무 복잡하지 않은지 확인
- [ ] `추정`이라는 표현이 충분히 보이는지 확인
- [ ] 남은 재고 리포트가 본사 발주 판단에 쓸 수 있는지 확인

### 4.9 직원, 근무, 인건비는 월말 정산과 생산성 분석의 기초로 반영했습니다

지난번 회의에서 직원이 여러 지점을 오가며 일하고, 근태와 급여를 관리하고 싶다는 요구가 있었다. 그래서 근무인원과 급여/인건비 행을 분리했다.

반영 내용:

- 6단계 화면 이름을 `근무/인건비`로 정리했다.
- `근무인원`은 실제 일한 사람 수를 입력한다.
- `급여/인건비`는 직원별 금액과 지각, 조퇴, 특이사항 메모를 남긴다.
- 급여 행 기준 참고 인원과 근무인원이 다를 수 있음을 안내한다.
- 본사에는 직원 관리 메뉴가 있어 직원 정보를 따로 관리할 수 있다.

사용 컨셉:

> 근무인원은 생산성 판단용 숫자이고, 급여/인건비 행은 월말 정산 자료입니다. 둘을 무조건 같은 숫자로 맞추지 않습니다.

리뷰 체크:

- [ ] 근무인원과 급여/인건비가 왜 분리되어 있는지 설명이 이해되는지 확인
- [ ] 직원명, 금액, 지각/조퇴/특이사항 입력이 실제 운영에 맞는지 확인
- [ ] 지점 이동 근무자를 본사에서 어떻게 볼지 추가 요구가 있는지 확인

### 4.10 본사 지출은 지점장 화면과 분리했습니다

지난번 회의에서 지점장들은 월세, 세금 같은 본사 고정비를 모르기 때문에 영업이익을 잘못 이해할 수 있다는 이야기가 있었다. 그래서 본사 지출은 본사 전용 메뉴로 분리했다.

반영 내용:

- 본사 메뉴에 `본사 지출`이 있다.
- 본사 지출은 지점장 화면에 보이지 않는다.
- 날짜, 지점, 항목, 금액, 메모를 본사가 관리한다.
- 월 손익이나 리포트 판단 자료로 활용할 수 있는 구조다.

사용 컨셉:

> 현장 비용은 지점 장부에, 월세나 세금 같은 본사 관리 비용은 본사 지출에 따로 넣습니다.

리뷰 체크:

- [ ] 본사 지출 항목 구성이 실제 비용 관리 방식과 맞는지 확인
- [ ] 지점장에게 보이면 안 되는 비용이 지점 화면에 보이지 않는지 확인
- [ ] 월간 리포트에서 본사 지출을 어떤 방식으로 보고 싶은지 확인

### 4.11 LINE 아침 요약 알림을 준비했습니다

지난번 회의에서 매일 아침 대표님이나 임원에게 요약 알림을 보내는 이야기가 있었다. 카카오톡은 자동 발송 제약이 있어 LINE 기준으로 정리했다.

반영 내용:

- 매일 오전 8시 기준으로 전날 핵심 요약을 보낼 수 있는 구조가 있다.
- LINE 채널 토큰, 수신자 ID, 내부 인증값을 설정하면 발송된다.
- 수신자는 정확히 3명으로 고정하지 않고, 1명 이상이면 발송 가능하게 했다.
- 운영 권장값은 임원/지출 권한 관리자 3명이다.
- 발송 결과는 로그로 남는다.

사용 컨셉:

> 대표님이 아침마다 시스템에 직접 들어가기 전에, 문제 있는 지점과 확인할 내용을 LINE으로 먼저 받아보는 기능입니다.

리뷰 체크:

- [ ] LINE을 실제로 사용할지 최종 확인
- [ ] 받을 사람 명단을 확정
- [ ] 알림에 꼭 들어가야 할 항목을 확정
- [ ] 오전 8시 발송 시간이 맞는지 확인
- [ ] 실패했을 때 재발송 또는 수동 확인 방식이 필요한지 확인

### 4.12 변경 이력과 감사 로그를 남기게 했습니다

지난번 회의에서 본사가 숫자를 강제 수정할 수 있어야 한다는 이야기가 있었다. 대신 누가 언제 무엇을 바꿨는지도 남아야 한다. 그래서 변경 이력과 감사 로그 흐름을 반영했다.

반영 내용:

- 본사는 지점장이 입력한 장부를 수정할 수 있다.
- 수정 전 값과 수정 후 값을 추적한다.
- 최초 작성자 표시명은 보존한다.
- 본사 수정자는 변경 이력으로 따로 남긴다.
- 사용자/지점/기준정보 변경도 이력에서 확인할 수 있다.

사용 컨셉:

> 숫자는 본사가 바로잡을 수 있지만, 누가 무엇을 바꿨는지는 남깁니다.

리뷰 체크:

- [ ] 본사에서 장부 수정이 가능한지 확인
- [ ] 수정 사유 또는 변경 이력이 남는지 확인
- [ ] 의뢰자가 원하는 변경 이력 표현 방식이 있는지 확인

## 6. 리뷰 자리에서 바로 쓸 말

### 시작 멘트

> 지난번 회의에서 나온 내용은 단순히 장부 입력 화면을 만드는 것이 아니라, 본사가 여러 지점을 매일 확인하기 쉽게 만드는 것이 핵심이었습니다.  
> 그래서 지점장은 7단계로 쉽게 입력하고, 본사는 관제판, 리포트, 이카운트, 재고, 손실, 근무, 본사 지출, 알림까지 이어서 볼 수 있게 정리했습니다.

### 지점장 화면 설명

> 지점장님들은 회계 프로그램을 쓰는 느낌이 아니라, 오늘 매출, 비용, 들어온 물건, 손실, 남은 재고, 근무인원만 순서대로 입력하면 됩니다.  
> 마지막 검토 화면에서 빠진 것과 이상한 것을 보고 제출하면 본사에서 검토합니다.

### 본사 화면 설명

> 본사는 모든 지점이 입력했는지, 어느 지점이 이상한지, 어떤 품목이 많이 팔렸는지, 냉동/생물 구분으로 어떤 흐름인지 볼 수 있습니다.  
> 잘못된 값은 본사에서 수정할 수 있고, 수정 이력도 남습니다.

### 이카운트 설명

> 이카운트 파일은 품목별 기준 단가표가 아니라, 본사가 지점에 보낸 물건의 원장으로 봅니다.  
> 그래서 원본을 보존하고, 지점과 품목 매핑을 거쳐 각 지점 장부에 반영합니다.

### 추정값 설명

> 품목별 실제 POS 판매 데이터가 아직 있는 것은 아니기 때문에, 냉동/생물 매출이나 품목별 이익률은 확정값이 아니라 추정값으로 표시했습니다.  
> 대신 판매 예정가와 재고 흐름을 사용해서 대표님이 판단할 수 있는 자료로 보이게 했습니다.

### 마무리 멘트

> 오늘은 기능이 있는지 없는지보다, 실제 사장님 업무 방식에 맞게 흐름이 맞는지를 확인하면 됩니다.  
> 화면 문구, 지점명, 품목명, 알림 내용처럼 현장에서 쓰는 말은 오늘 보시고 바로 조정하면 됩니다.

## 8. 주의해서 말할 부분

- `추정`이라고 표시된 값은 POS 확정 매출이 아니다.
- FIFO 원가와 재고금액은 본사 판단용이며, 지점장에게 복잡하게 노출하지 않는다.
- LINE 알림은 환경값과 실제 수신자 설정이 완료되어야 운영 발송된다.
- 이카운트 업로드는 실제 운영 파일의 거래처명/품목명 매핑 확인이 중요하다.
- 화면 문구와 품목명은 오늘 리뷰 후 현장 용어에 맞춰 조정할 수 있다.
