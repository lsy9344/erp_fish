## 우선 미완료로 봐야 할 항목

### 1. 월별 손익의 `조정사유`와 `메모` 분리 — ✅ 해결(2026-06-30)

월별 손익 계획 문서에는 `조정사유`와 `메모`를 분리하라고 되어 있었고, 이전 코드는 본사조정의 조정사유를 `expense.memo`에서 가져왔다.

처리 상태:

- `HeadquartersExpense.adjustmentReason`(nullable) 컬럼을 추가하고 입력/검증/감사/조회/화면에 분리했다. 본사조정일 때만 저장한다.
- migration이 기존 본사조정 행의 `memo`를 `adjustmentReason`로 backfill한다(배포 후 조정사유 유실 방지).
- 월별 손익 builder는 `adjustmentReason`을 조정사유로, `memo`는 메모로만 쓴다.
- `tests/unit/monthly-profit-loss.test.mjs`에 둘이 섞이지 않는 소스 계약 테스트를 추가했다.

### 2. 월별 xlsx의 `품목매출` 시트가 기간 합산이 아님 — ✅ 해결(2026-06-30)

회의에서는 기간별 조회와 xlsx 피벗 분석이 중요하다고 나왔고, 이전 export route는 월 마지막 날의 일별 리포트 `productProfitability`를 대표값으로 썼다.

처리 상태:

- `getHqProductSalesReportForRange(startDate, endDate, storeId)`로 조회 기간 전체를 store×product 단위로 합산하는 쿼리를 추가했다.
- `품목매출` 시트는 확정 컬럼(조회 시작/종료일·지점·원가·이익·손실·재고)으로 바뀌었고, route가 이 기간 쿼리를 쓴다.
- 월별손익/기간조회와 같은 모집단을 쓰도록 `status IN (IN_REVIEW, HEADQUARTERS_CLOSED)`만 집계한다(작성 중·휴무 장부 제외).
- 분류 미확정(`기준 미정`) 품목도 누락하지 않고 시트에 포함한다.
- `tests/api/report-export.spec.ts`가 시트 헤더(확정 컬럼)를, `tests/unit/monthly-profit-loss.test.mjs`가 route/쿼리 계약(기간 쿼리·status·기준 미정 포함)을 검증한다.

### 3. 냉동/생물 기준 검수

음성 내용에서는 냉동/생물 구분이 문자열 추측으로 끝나면 안 되고, 의뢰자가 가진 기초 자료를 기준으로 해야 한다고 정리됐다. 기준자료(`냉동_생물_자료.xlsx`) 도착으로 최종 용어는 `냉동/생물`로 확정했고 코드/문서도 이 용어로 통일한다. 남은 일은 자료 부재가 아니라 분류 규칙 검수다.

필요 조치:

- 최종 용어는 `냉동/생물`로 확정(이전 미확정 표현은 실행 문서/코드에서 제거).
- 현재 코드의 `냉)` 접두 분류 규칙이 제공 자료와 실제 운영 품목을 충분히 덮는지 확인한다.
- 품목명, 규격, 표준품목, 분류, 장기재고 기준일을 기준표와 화면/export 용어에 맞춘다.
- 품목 분류 자체가 애매한 신규 품목은 `기준 미정`으로 둔다.
- 장기재고 기준일이 없는 품목군은 `기준 확인 필요`로 표시하고 LINE 알림 대상에서 제외한다.

### 4. 장부 전체 셀 반영 검증표의 빈칸 — ✅ 해결(2026-06-30)

`AE4/AE5`, `C5/C17`은 이미 확인됐고, 하위 셀이 `앱 확인 필요` 상태였다.

처리 상태:

- `C15`, `AG36/AG63/AG76`, `AI36/AI63/AI76`, `C22/C23/C24`, `C36`의 원본 값/수식을 ExcelJS로 추출해 `ledger-cell-mapping-review-2026-06-28.md` 표에 채웠다. `앱 확인 필요` 행은 남지 않았다.
- 각 셀은 본사 홈 직접 노출값이 아니라 `AE4/AE5/C5/C17`의 원천 셀이라 `확인됨`/`제외`로 정리하고 제외 사유를 적었다.
- 원본 값으로 본사 홈 4개 값(분석 매출 3,514,811 / 장부 매출 3,491,000 등)이 재현되는지 교차 검증했다.
