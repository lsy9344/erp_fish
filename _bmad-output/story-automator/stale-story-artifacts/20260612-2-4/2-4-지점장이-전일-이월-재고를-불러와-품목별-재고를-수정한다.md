# Story 2.4: 지점장이 전일 이월 재고를 불러와 품목별 재고를 수정한다

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 지점장,
I want 전일 이월 재고를 불러온 뒤 변경된 품목만 수정하고 싶다,
so that 30개 이상의 냉동/생물 품목 재고를 빠르게 입력할 수 있다.

## Acceptance Criteria

1. **Given** 지점장이 입력중 장부의 재고 단계에 접근할 때  
   **When** 해당 지점+일자의 전일재고 기준을 확인하면  
   **Then** 월 첫 장부는 월초 재고 스냅샷에서 전일재고를 가져와야 한다  
   **And** 이후 영업일은 직전 본사 마감 장부의 당일재고를 가져와야 한다.

2. **Given** 전일재고 기준을 불러올 수 있을 때  
   **When** 재고 단계가 열리면  
   **Then** 품목별 전일재고가 프리필되어야 한다  
   **And** "전일 재고를 불러왔습니다. 변경된 품목만 수정하세요." 안내가 표시되어야 한다.

3. **Given** 전일 장부가 마감되지 않아 자동 이월이 불가할 때  
   **When** 재고 단계가 열리면  
   **Then** 직접 입력하거나 본사에 문의하라는 한국어 안내가 표시되어야 한다  
   **And** 사용자는 수동으로 재고를 입력할 수 있어야 한다.

4. **Given** 지점장이 재고 품목을 볼 때  
   **When** 재고 테이블이 표시되면  
   **Then** 냉동/생물 탭으로 품목을 나눠 볼 수 있어야 한다  
   **And** 품목, 규격, 전일재고, 매입, 당일재고, 재고금액, 수량 컬럼이 제공되어야 한다.

5. **Given** 지점장이 품목의 당일재고 또는 수량 값을 수정할 때  
   **When** 값을 저장하면  
   **Then** 수정된 값이 장부에 저장되어야 한다  
   **And** 수정된 행은 시각적으로 강조되어야 하며 색상만으로 의미를 전달하면 안 된다.

6. **Given** 재고 품목이 30개 이상일 때  
   **When** 모바일 화면에서 재고를 입력하면  
   **Then** 가로 스크롤 또는 50행 단위 페이지/가상 목록 중 하나로 입력 가능해야 한다  
   **And** 주요 편집 셀은 44x44px 이상의 터치 대상을 가져야 한다.

7. **Given** 재고 수량이나 금액에 잘못된 값이 입력될 때  
   **When** 저장을 시도하면  
   **Then** 필드 하단 또는 행 안에 한국어 검증 메시지가 표시되어야 한다  
   **And** 오류가 있는 첫 번째 편집 위치로 이동해야 한다.

8. **Given** 재고금액이 계산될 때  
   **When** 수량과 단가가 모두 존재하면  
   **Then** 재고금액은 수량과 단가를 기준으로 표시되어야 한다  
   **And** 값이 없거나 계산 불가한 경우 숨은 오류 대신 "계산 불가" 상태를 표시해야 한다.

## Dev Notes

### Source Context

- Story 2.4는 Epic 2의 네 번째 입력 단계이며, 매출/비용/매입 뒤 재고를 입력하고 손실/검토 단계로 이어진다.
- 현재 저장소에는 `/app/store-entry/inventory/page.tsx`가 있지만 아직 "재고 입력 준비" placeholder만 표시한다.
- `StoreManagerShell`은 모바일 하단 탭으로 `장부`, `재고`, `손실`을 제공하고, 재고 탭은 `/app/store-entry/inventory`를 canonical route로 사용한다.
- `src/features/ledger`에는 2.1~2.3에서 만든 `queries.ts`, `actions.ts`, `schemas.ts`, `types.ts`, `components/*step-client.tsx` 패턴이 있다.
- Story 2.3 구현 흔적 기준으로 `LedgerPurchaseItem`, `saveLedgerPurchases`, `calculatePurchaseTotal`, `PurchaseStepClient`, `tests/unit/ledger-purchase.test.mjs`, `tests/e2e/store-ledger-purchase.spec.ts`가 존재한다.

### Architecture Guardrails

- 스택은 Next.js App Router + Server Components/Server Actions + Prisma + PostgreSQL + Zod + shadcn/ui + Tailwind CSS를 유지한다.
- `src/components/ui`에는 shadcn primitive만 두고, 재고 업무 컴포넌트는 `src/features/inventory/components` 아래에 둔다.
- 권한은 클라이언트에서만 확인하지 않는다. 모든 조회/저장 경로는 서버에서 `requireStoreAccess`를 통과해야 한다.
- 데이터 변경은 `db.$transaction` 안에서 장부/재고/감사 로그를 함께 처리한다.
- 마스터 데이터 변경은 과거 장부 값을 바꾸면 안 된다. 재고 행에도 품목명/구분/규격/단가 스냅샷을 저장한다.
- 돈과 수량은 정수로 저장한다. `"1,000원"` 같은 표시 문자열은 DB에 저장하지 않는다.
- 계산 규칙은 `src/server/calculations/*`의 서버 헬퍼로 모으고, 화면/리포트/검토 단계가 같은 규칙을 재사용하게 한다.

  - 재고 화면은 매입 데이터를 읽어야 하지만 매입 저장 책임을 가져가면 안 된다.

### UX Guardrails

- UX 요구사항은 `InventoryTable`, `InventoryRowEdit`, 냉동/생물 탭, 전일 이월 프리필, 변경된 품목만 수정하는 흐름을 요구한다.
- 수정된 재고 행은 primary 기반 강조, 신규 품목 행은 success 기반 강조를 사용한다.
- 색상만으로 의미를 전달하지 않는다. 행에 "수정됨" 텍스트 또는 `aria-label`을 함께 둔다.
- 모바일 재고 테이블은 가로 스크롤을 허용하고, 태블릿 landscape에서는 주요 컬럼(품목, 전일재고, 당일재고, 재고금액)을 고정하는 방향이 UX 문서와 맞다.
- 숫자 데이터는 `tabular-nums` 또는 동등한 numeric font feature를 적용한다.
- key inventory mockup의 핵심 상태는 "재고 입력 단계 - 냉동 탭, 전일 이월 프리필, 일부 행 수정됨"이다.

  [External: `https://www.prisma.io/docs/orm/prisma-client/queries/transactions`]
- Zod 공식 문서는 field별 오류 구조를 제공하는 error formatting을 안내한다. 현재 저장소의 `toFieldErrors` 패턴을 유지한다.
