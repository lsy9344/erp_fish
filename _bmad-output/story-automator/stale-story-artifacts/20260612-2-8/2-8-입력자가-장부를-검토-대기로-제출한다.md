# Story 2.8: 입력자가 장부를 검토 대기로 제출한다

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 지점장 또는 본사 사용자,
I want 입력한 장부를 검토 대기로 제출하고 싶다,
so that 본사가 장부를 확인하고 필요한 경우 보완한 뒤 마감할 수 있다.

## Acceptance Criteria

1. **Given** 장부 상태가 `입력중`이고 사용자가 검토/제출 단계에 있을 때
   **When** 사용자가 제출 버튼을 누르면
   **Then** 장부 상태가 `검토대기`로 변경되어야 한다
   **And** 제출자와 제출 시각이 기록되어야 한다.

2. **Given** 누락 항목이나 이상 후보가 있는 장부일 때
   **When** 사용자가 제출을 시도하면
   **Then** 누락과 이상 후보가 다시 표시되어야 한다
   **And** 이 경고는 제출을 자동 차단하지 않아야 한다.

3. **Given** 장부가 `검토대기` 상태가 되었을 때
   **When** 지점장 또는 권한 있는 본사 사용자가 장부를 다시 열면
   **Then** 본사 마감 전까지 원본 장부를 수정할 수 있어야 한다
   **And** `검토대기`는 원본 잠금으로 처리되면 안 된다.

4. **Given** 제출이 완료될 때
   **When** 장부 상태가 변경되면
   **Then** actor, timestamp, target, before, after를 포함하는 감사 기록이 남아야 한다
   **And** 관련 관제판/장부 화면은 새 상태를 보여주도록 갱신되어야 한다.

5. **Given** 사용자가 이미 `검토대기` 상태인 장부에서 제출을 다시 시도할 때
   **When** 제출 요청이 처리되면
   **Then** 중복 제출 기록이 무의미하게 쌓이지 않아야 한다
   **And** 사용자는 이미 검토 대기 상태라는 안내를 봐야 한다.

6. **Given** 제출 요청 중 서버 오류가 발생할 때
   **When** 사용자가 제출 버튼을 누르면
   **Then** 장부 상태는 기존 상태로 유지되어야 한다
   **And** 한국어 오류 메시지와 재시도 동작이 제공되어야 한다.

7. **Given** 390px 모바일 화면에서 제출 결과가 표시될 때
   **When** 제출이 성공하면
   **Then** "장부를 제출했습니다." 피드백과 현재 상태 배지가 표시되어야 한다
   **And** 상태 배지는 텍스트로 의미를 전달해야 한다.

### Architecture Guardrails

- 스택은 Next.js App Router + Server Components/Server Actions + Prisma + PostgreSQL + Zod + shadcn/ui + Tailwind CSS를 유지한다. [Source: `C:\\Code\\Project\\erp_fish\\_bmad-output\\planning-artifacts\\architecture.md#Core-Architectural-Decisions`]
- 내부 mutation은 Route Handler가 아니라 Server Action으로 구현한다. 공개 API가 필요한 요구사항이 아니다.
- 모든 민감한 장부 mutation은 서버에서 `requireStoreAccess`를 호출한다.
- 감사 로그는 업무 변경과 같은 transaction 안에서 쓴다. 상태만 바뀌고 감사 로그가 빠지는 중간 상태를 만들지 않는다.
- 돈과 수량 계산 규칙은 기존 shared calculation/query 흐름을 재사용한다. 제출 action에서 계산을 새로 만들지 않는다.
- `IN_REVIEW`는 수정 가능한 상태다. 잠금 로직은 `HEADQUARTERS_CLOSED`에만 둔다.
- 중복 제출은 멱등적으로 처리한다. 같은 장부에 `ledger.review.submitted` 로그가 여러 건 쌓이면 안 된다.

### UX Guardrails

- 제출 버튼은 기존 검토 화면의 마지막 행동으로 둔다. 랜딩형 설명, 큰 hero, 장식 layout을 만들지 않는다.
- 누락과 이상 후보는 제출 전 다시 보이게 하되, 차단 UI처럼 보이면 안 된다.
- 상태 배지는 텍스트 의미를 포함한다. 색상만으로 `검토대기`를 전달하지 않는다.

- 오류 메시지는 한국어로 짧게 제공하고, 다시 시도 버튼은 같은 submit action을 재호출한다.
- 모바일 390px에서 제출 버튼, 상태 배지, 성공/오류 문구, 경고 목록이 서로 겹치거나 카드 밖으로 넘치면 실패다.
