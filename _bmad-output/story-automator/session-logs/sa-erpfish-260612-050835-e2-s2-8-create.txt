+# Story 2.8: 근무인원/특이사항과 검토 제출
+
+Status: ready-for-dev
+
+<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->
+
+## Story
+
+As a 지점장,
+I want 근무인원과 특이사항을 입력하고 장부를 검토 대기 상태로 제출할 수 있기를 원한다,
+so that 본사가 회의 전 장부의 운영 맥락과 제출 상태를 확인할 수 있다.
+
+## Acceptance Criteria
+
+1. Given 사용자가 근무인원/특이사항 단계에 있을 때, when 근무인원과 특이사항을 입력한다, then 값은 해당 장부에 저장되어야 한다, and 근무인원은 승인된 기본 계산에서 사용할 수 있어야 한다.
+2. Given 일반 직원 로그인 기능이나 직원별 근태 관리가 요구될 때, when 이 스토리를 구현한다, then 직원별 선택, 지각/조퇴 상세, 실제 급여 지급 확정 기능은 포함하지 않아야 한다, and MVP는 근무인원/특이
사항 기록 수준으로 제한해야 한다.
+3. Given 사용자가 검토/제출 단계에 도달했을 때, when 제출 전 검토 화면이 표시된다, then 입력된 매출, 비용, 매입, 재고, 손실, 근무 정보의 요약과 저장 상태가 보여야 한다, and 필수 누락이나 `기준 확인
필요` 상태가 있는 항목은 숨기지 않아야 한다.
+4. Given 사용자가 장부를 제출할 때, when 서버 검증이 통과한다, then 장부 상태는 `검토대기`로 변경되어야 한다, and 지점장 제출은 원본 잠금이 아니며, 본사 마감 전까지 권한 있는 사용자가 수정할 수 있어
야 한다.
+5. Given 제출 시 필수 값이 누락되었을 때, when 서버가 제출 요청을 검증한다, then 제출은 거부되어야 한다, and 사용자는 어느 단계의 어떤 필드를 고쳐야 하는지 알 수 있어야 한다.
+6. Given 장부가 제출되거나 제출 상태에서 수정될 때, when 저장 요청이 성공한다, then 제출자/수정자, 시각, 변경 전/후 상태는 감사 로그에 남아야 한다, and 장부 version 또는 edit token이 갱신되어야 한다.

+# Story 2.8: 근무인원/특이사항과 검토 제출
+
+Status: ready-for-dev
+
+<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->
+
+## Story
+
+As a 지점장,
+I want 근무인원과 특이사항을 입력하고 장부를 검토 대기 상태로 제출할 수 있기를 원한다,
+so that 본사가 회의 전 장부의 운영 맥락과 제출 상태를 확인할 수 있다.
+
+## Acceptance Criteria
+
+1. Given 사용자가 근무인원/특이사항 단계에 있을 때, when 근무인원과 특이사항을 입력한다, then 값은 해당 장부에 저장되어야 한다, and 근무인원은 승인된 기본 계산에서 사용할 수 있어야 한다.
+2. Given 일반 직원 로그인 기능이나 직원별 근태 관리가 요구될 때, when 이 스토리를 구현한다, then 직원별 선택, 지각/조퇴 상세, 실제 급여 지급 확정 기능은 포함하지 않아야 한다, and MVP는 근무인원/특이
사항 기록 수준으로 제한해야 한다.
+3. Given 사용자가 검토/제출 단계에 도달했을 때, when 제출 전 검토 화면이 표시된다, then 입력된 매출, 비용, 매입, 재고, 손실, 근무 정보의 요약과 저장 상태가 보여야 한다, and 필수 누락이나 `기준 확인
필요` 상태가 있는 항목은 숨기지 않아야 한다.
+4. Given 사용자가 장부를 제출할 때, when 서버 검증이 통과한다, then 장부 상태는 `검토대기`로 변경되어야 한다, and 지점장 제출은 원본 잠금이 아니며, 본사 마감 전까지 권한 있는 사용자가 수정할 수 있어
야 한다.
+5. Given 제출 시 필수 값이 누락되었을 때, when 서버가 제출 요청을 검증한다, then 제출은 거부되어야 한다, and 사용자는 어느 단계의 어떤 필드를 고쳐야 하는지 알 수 있어야 한다.
+6. Given 장부가 제출되거나 제출 상태에서 수정될 때, when 저장 요청이 성공한다, then 제출자/수정자, 시각, 변경 전/후 상태는 감사 로그에 남아야 한다, and 장부 version 또는 edit token이 갱신되어야 한다.

# Story 2.8: 근무인원/특이사항과 검토 제출

Status: ready-for-dev

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 지점장,
I want 근무인원과 특이사항을 입력하고 장부를 검토 대기 상태로 제출할 수 있기를 원한다,
so that 본사가 회의 전 장부의 운영 맥락과 제출 상태를 확인할 수 있다.

## Acceptance Criteria

1. Given 사용자가 근무인원/특이사항 단계에 있을 때, when 근무인원과 특이사항을 입력한다, then 값은 해당 장부에 저장되어야 한다, and 근무인원은 승인된 기본 계산에서 사용할 수 있어야 한다.
2. Given 일반 직원 로그인 기능이나 직원별 근태 관리가 요구될 때, when 이 스토리를 구현한다, then 직원별 선택, 지각/조퇴 상세, 실제 급여 지급 확정 기능은 포함하지 않아야 한다, and MVP는 근무인원/특이사
항 기록 수준으로 제한해야 한다.
3. Given 사용자가 검토/제출 단계에 도달했을 때, when 제출 전 검토 화면이 표시된다, then 입력된 매출, 비용, 매입, 재고, 손실, 근무 정보의 요약과 저장 상태가 보여야 한다, and 필수 누락이나 `기준 확인 필
요` 상태가 있는 항목은 숨기지 않아야 한다.
4. Given 사용자가 장부를 제출할 때, when 서버 검증이 통과한다, then 장부 상태는 `검토대기`로 변경되어야 한다, and 지점장 제출은 원본 잠금이 아니며, 본사 마감 전까지 권한 있는 사용자가 수정할 수 있어야
 한다.
5. Given 제출 시 필수 값이 누락되었을 때, when 서버가 제출 요청을 검증한다, then 제출은 거부되어야 한다, and 사용자는 어느 단계의 어떤 필드를 고쳐야 하는지 알 수 있어야 한다.
6. Given 장부가 제출되거나 제출 상태에서 수정될 때, when 저장 요청이 성공한다, then 제출자/수정자, 시각, 변경 전/후 상태는 감사 로그에 남아야 한다, and 장부 version 또는 edit token이 갱신되어야 한다.

- Critical 2: `tests/e2e/store-ledger-review.spec.ts`가 현재 "누락과 경고가 있어도 제출"을 기대한다. AC 5에 맞게 누락 차단으로 바꾸고, 비차단 경고는 별도 허용 기준으로 유지해야 한다.
- Critical 3: 검토/제출 E2E marker/store가 `story-2-7`로 남아 있어 Dev agent와 report가 current sprint 기준을 혼동할 수 있다. Story 2.8 명명으로 정리한다.
- Critical 4: 지점장 review response에서 `productivity`와 signal amount를 빼고 있지만, submit validation을 만들면서 full review data를 client로 넘기면 민감 필드가 누출될 수 있다.
- Critical 5: 제출은 원본 잠금이 아니다. `IN_REVIEW` 수정 가능성과 각 save action의 status 유지 회귀를 테스트해야 한다.
- Critical 6: duplicate submit은 감사 로그를 중복 생성하거나 `submittedAt`을 갱신하면 안 된다.
- Critical 7: 근무인원 0/null 처리 기준을 흐리면 productivity 계산과 missing 판단이 달라진다. 저장은 0을 허용하되 제출 필수 기준은 `workerCount > 0` 여부로 명확히 한다.
- Critical 8: 검토 화면의 `계산 불가`/`기준 확인 필요` 상태를 정상 또는 0으로 숨기면 본사 회의 전 누락/정책 미확정 항목을 놓친다.

+## Story
+
+As a 지점장,
+I want 근무인원과 특이사항을 입력하고 장부를 검토 대기 상태로 제출할 수 있기를 원한다,
+so that 본사가 회의 전 장부의 운영 맥락과 제출 상태를 확인할 수 있다.
+
+## Acceptance Criteria
+
+1. Given 사용자가 근무인원/특이사항 단계에 있을 때, when 근무인원과 특이사항을 입력한다, then 값은 해당 장부에 저장되어야 한다, and 근무인원은 승인된 기본 계산에서 사용할 수 있어야 한다.
+2. Given 일반 직원 로그인 기능이나 직원별 근태 관리가 요구될 때, when 이 스토리를 구현한다, then 직원별 선택, 지각/조퇴 상세, 실제 급여 지급 확정 기능은 포함하지 않아야 한다, and MVP는 근무인원/특이
사항 기록 수준으로 제한해야 한다.
+3. Given 사용자가 검토/제출 단계에 도달했을 때, when 제출 전 검토 화면이 표시된다, then 입력된 매출, 비용, 매입, 재고, 손실, 근무 정보의 요약과 저장 상태가 보여야 한다, and 필수 누락이나 `기준 확인
필요` 상태가 있는 항목은 숨기지 않아야 한다.
+4. Given 사용자가 장부를 제출할 때, when 서버 검증이 통과한다, then 장부 상태는 `검토대기`로 변경되어야 한다, and 지점장 제출은 원본 잠금이 아니며, 본사 마감 전까지 권한 있는 사용자가 수정할 수 있어

+5. Given 제출 시 필수 값이 누락되었을 때, when 서버가 제출 요청을 검증한다, then 제출은 거부되어야 한다, and 사용자는 어느 단계의 어떤 필드를 고쳐야 하는지 알 수 있어야 한다.
+6. Given 장부가 제출되거나 제출 상태에서 수정될 때, when 저장 요청이 성공한다, then 제출자/수정자, 시각, 변경 전/후 상태는 감사 로그에 남아야 한다, and 장부 version 또는 edit token이 갱신되어야 한다.

+# Story 2.8: 근무인원/특이사항과 검토 제출
+
+Status: ready-for-dev
+
+<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

+## Story
+
+As a 지점장,
+I want 근무인원과 특이사항을 입력하고 장부를 검토 대기 상태로 제출할 수 있기를 원한다,
+so that 본사가 회의 전 장부의 운영 맥락과 제출 상태를 확인할 수 있다.
+
+## Acceptance Criteria
+
+1. Given 사용자가 근무인원/특이사항 단계에 있을 때, when 근무인원과 특이사항을 입력한다, then 값은 해당 장부에 저장되어야 한다, and 근무인원은 승인된 기본 계산에서 사용할 수 있어야 한다.
+2. Given 일반 직원 로그인 기능이나 직원별 근태 관리가 요구될 때, when 이 스토리를 구현한다, then 직원별 선택, 지각/조퇴 상세, 실제 급여 지급 확정 기능은 포함하지 않아야 한다, and MVP는 근무인원/특이
사항 기록 수준으로 제한해야 한다.
+3. Given 사용자가 검토/제출 단계에 도달했을 때, when 제출 전 검토 화면이 표시된다, then 입력된 매출, 비용, 매입, 재고, 손실, 근무 정보의 요약과 저장 상태가 보여야 한다, and 필수 누락이나 `기준 확인
필요` 상태가 있는 항목은 숨기지 않아야 한다.
+4. Given 사용자가 장부를 제출할 때, when 서버 검증이 통과한다, then 장부 상태는 `검토대기`로 변경되어야 한다, and 지점장 제출은 원본 잠금이 아니며, 본사 마감 전까지 권한 있는 사용자가 수정할 수 있어
야 한다.
+5. Given 제출 시 필수 값이 누락되었을 때, when 서버가 제출 요청을 검증한다, then 제출은 거부되어야 한다, and 사용자는 어느 단계의 어떤 필드를 고쳐야 하는지 알 수 있어야 한다.
+6. Given 장부가 제출되거나 제출 상태에서 수정될 때, when 저장 요청이 성공한다, then 제출자/수정자, 시각, 변경 전/후 상태는 감사 로그에 남아야 한다, and 장부 version 또는 edit token이 갱신되어야 한다.

+# Story 2.8: 근무인원/특이사항과 검토 제출
+
+Status: ready-for-dev
+
+<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

+## Story
+
+As a 지점장,
+I want 근무인원과 특이사항을 입력하고 장부를 검토 대기 상태로 제출할 수 있기를 원한다,
+so that 본사가 회의 전 장부의 운영 맥락과 제출 상태를 확인할 수 있다.
+
+## Acceptance Criteria
+
+1. Given 사용자가 근무인원/특이사항 단계에 있을 때, when 근무인원과 특이사항을 입력한다, then 값은 해당 장부에 저장되어야 한다, and 근무인원은 승인된 기본 계산에서 사용할 수 있어야 한다.
+2. Given 일반 직원 로그인 기능이나 직원별 근태 관리가 요구될 때, when 이 스토리를 구현한다, then 직원별 선택, 지각/조퇴 상세, 실제 급여 지급 확정 기능은 포함하지 않아야 한다, and MVP는 근무인원/특이
사항 기록 수준으로 제한해야 한다.
+3. Given 사용자가 검토/제출 단계에 도달했을 때, when 제출 전 검토 화면이 표시된다, then 입력된 매출, 비용, 매입, 재고, 손실, 근무 정보의 요약과 저장 상태가 보여야 한다, and 필수 누락이나 `기준 확인
필요` 상태가 있는 항목은 숨기지 않아야 한다.
+4. Given 사용자가 장부를 제출할 때, when 서버 검증이 통과한다, then 장부 상태는 `검토대기`로 변경되어야 한다, and 지점장 제출은 원본 잠금이 아니며, 본사 마감 전까지 권한 있는 사용자가 수정할 수 있어
야 한다.
+5. Given 제출 시 필수 값이 누락되었을 때, when 서버가 제출 요청을 검증한다, then 제출은 거부되어야 한다, and 사용자는 어느 단계의 어떤 필드를 고쳐야 하는지 알 수 있어야 한다.
+6. Given 장부가 제출되거나 제출 상태에서 수정될 때, when 저장 요청이 성공한다, then 제출자/수정자, 시각, 변경 전/후 상태는 감사 로그에 남아야 한다, and 장부 version 또는 edit token이 갱신되어야 한다.
