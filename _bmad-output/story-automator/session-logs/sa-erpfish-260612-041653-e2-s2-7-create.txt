+# Story 2.7: 손실/폐기/떨이 입력
+
+Status: ready-for-dev
+
+<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->
+
+## Story
+
+As a 지점장,
+I want 손실, 폐기, 떨이 항목을 품목과 사유와 함께 입력할 수 있기를 원한다,
+so that 본사가 재고 감소와 현장 손실의 원인을 확인할 수 있다.
+
+## Acceptance Criteria
+
+1. Given 사용자가 손실/폐기/떨이 단계에 있을 때, when 품목, 수량, 금액, 처리 유형, 사유/특이사항을 입력한다, then 손실 라인은 해당 장부에 저장되어야 한다, and 처리 유형은 관리 코드 기준을 따라야 한다
.
+2. Given 사용자가 필수 값 없이 손실 라인을 저장하려 할 때, when 서버가 요청을 검증한다, then 저장은 거부되어야 한다, and UI는 누락 필드를 사용자가 고칠 수 있게 표시해야 한다.
+3. Given 손실 라인이 저장되었을 때, when 재고 흐름을 조회한다, then 손실 수량은 해당 품목의 재고 흐름에 반영되어야 한다, and 손실 리포트에서 사용할 수 있는 구조화 데이터로 보존되어야 한다.
+4. Given 희망 판매가 기준 손실액 정책 OQ-9가 아직 닫히지 않았을 때, when 이 스토리를 구현한다, then 희망 판매가 기준 손실액 계산은 포함하지 않아야 한다, and 사용자가 입력한 기존 손실 유형, 수량, 금액
, 사유 기록에 집중해야 한다.
+5. Given 손실 라인이 수정되거나 삭제될 때, when 저장 요청이 성공한다, then 변경 전/후 값과 사유는 감사 로그에 남아야 한다, and 장부 version 또는 edit token이 갱신되어야 한다.
+6. Given 지점장 권한으로 손실 정보를 조회할 때, when 서버가 응답을 만든다, then 원가나 이익을 역추정할 수 있는 민감 필드는 제외되어야 한다, and 현장 입력에 필요한 품목, 수량, 금액, 유형, 사유 중심으
로 표시되어야 한다.

+### Architecture Guardrails
+
+- 선택 스택은 Create T3 App 기반 Next.js App Router, TypeScript, Prisma, PostgreSQL, NextAuth/Auth.js, Tailwind CSS, shadcn/ui, Vercel이다. Story 2.7에서 ORM, auth framework, UI primitive 체계를 바꾸
지 않는다. [Source: `_bmad-output/planning-artifacts/architecture.md#Starter Template Evaluation`]
+- Prisma schema and migrations are the source of truth for persisted data. schema 변경은 새 migration으로 반영하고 기존 migration을 수정하지 않는다. [Source: `_bmad-output/planning-artifacts/architec
ture.md#Data Architecture`]
+- 대부분의 제품 동작은 Server Components와 Server Actions를 사용한다. 손실 저장을 위해 public API route를 만들지 않는다. [Source: `_bmad-output/planning-artifacts/architecture.md#API & Communication
Patterns`]
+- 모든 mutation은 Zod validation, shared authorization helper, transaction, audit, revalidation을 따라야 한다. [Source: `_bmad-output/planning-artifacts/architecture.md#Process Patterns`]
+- Store manager responses must omit sensitive accounting fields. Story 2.7은 지점장에게 원가/이익/마진/FIFO 근거를 보내면 안 되지만, 사용자가 입력한 손실 금액은 AC상 현장 입력 필드이므로 노출 정책을
명확히 해야 한다. [Source: `_bmad-output/planning-artifacts/architecture.md#Authorization Pattern`]
+- `확인 필요`, `계산 불가`, `데이터 부족`은 shared status enum 또는 display helper로 맞춰야 하며 각 화면이 다른 라벨을 임의로 만들면 안 된다. [Source: `_bmad-output/planning-artifacts/architecture.md
#Error Handling Pattern`]
