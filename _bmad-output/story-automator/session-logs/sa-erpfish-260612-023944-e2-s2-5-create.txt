+# Story 2.5: 재고 입력과 이월 상태 표시
+
+Status: ready-for-dev
+
+<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->
+
+## Story
+
+As a 지점장,
+I want 품목별 전일재고, 매입, 손실, 당일재고를 입력하고 이월 상태를 확인할 수 있기를 원한다,
+so that 본사가 재고 흐름의 누락과 재확인 대상을 파악할 수 있다.
+
+## Acceptance Criteria
+
+1. Given 사용자가 재고 단계에 있을 때, when 품목+규격 단위 재고 행을 입력한다, then 전일재고, 매입, 손실, 당일재고, 수량, 상태를 저장할 수 있어야 한다, and 냉동/생물 또는 동등한 품목 그룹 탭으로 구분
할 수 있어야 한다.
+2. Given 직전 장부의 당일재고가 존재할 때, when 사용자가 다음 영업일 재고 단계를 연다, then 시스템은 전일재고 후보를 보여줄 수 있어야 한다, and 직전 장부가 본사 마감 전이면 확정값이 아니라 `검토 필요
` 상태로 표시해야 한다.
+3. Given 전일 장부가 없거나 이월 근거가 부족할 때, when 재고 행이 표시된다, then `이월 공백` 또는 동등한 상태를 보여줘야 한다, and 사용자가 부족한 근거를 0으로 오해하지 않게 해야 한다.
+4. Given 본사 마감 후 정정이 이월 값에 영향을 줄 수 있을 때, when 다음 장부 재고 화면이 열린다, then `이월 재확인 필요` 상태를 표시할 수 있어야 한다, and 기존 값을 조용히 덮어쓰면 안 된다.
+5. Given 지점장 권한으로 재고 화면을 볼 때, when 서버가 재고 응답을 만든다, then 재고금액, 원가, lot 근거 같은 민감 필드는 반환하지 않아야 한다, and 품목, 규격, 전일재고 후보, 매입/손실, 당일재고, 상
태 중심으로 표시해야 한다.
+6. Given FIFO, `30%단가`, 재고금액 정책이 아직 확정되지 않았을 때, when 이 스토리를 구현한다, then FIFO 원가 계산, `30%단가` 파생 표시, 확정 재고금액 계산은 포함하지 않아야 한다, and 필요한 경우 `기
준 확인 필요` 또는 `데이터 부족` 상태로 표시해야 한다.
+7. Given 모바일 재고 테이블에 30개 이상 행이 있을 때, when 사용자가 재고를 입력한다, then sticky header와 가상화 또는 50행 단위 페이지 처리를 제공해야 한다, and 현재 편집 셀이 모바일 키패드에 가려지
지 않아야 한다.

+### Architecture Guardrails
+
+- 선택 스택은 Create T3 App 기반 Next.js App Router, TypeScript, Prisma, PostgreSQL, NextAuth/Auth.js, Tailwind CSS, shadcn/ui, Vercel이다. Story 2.5에서 ORM, auth framework, UI primitive 체계를 바꾸
지 않는다. [Source: `_bmad-output/planning-artifacts/architecture.md#Starter Template Evaluation`]
+- Prisma schema and migrations are the source of truth for persisted data. schema 변경은 새 migration으로 반영한다. 기존 migration을 수정하지 않는다. [Source: `_bmad-output/planning-artifacts/archite
cture.md#Data Architecture`]
+- 대부분의 제품 동작은 Server Components와 Server Actions를 사용한다. 재고 저장을 위해 public API route를 만들지 않는다. [Source: `_bmad-output/planning-artifacts/architecture.md#API & Communication
Patterns`]
+- 모든 mutation은 Zod validation, shared authorization helper, transaction, audit, revalidation을 따라야 한다. [Source: `_bmad-output/planning-artifacts/architecture.md#Process Patterns`]
+- Store manager responses must omit sensitive accounting fields. Story 2.5는 재고 수량/상태 중심이며, 지점장 응답에 재고금액, 원가, lot 근거, 마진/이익 파생값을 보내면 안 된다. [Source: `_bmad-output
/planning-artifacts/architecture.md#Authorization Pattern`]
+- `확인 필요`, `계산 불가`, `데이터 부족`은 shared status enum 또는 display helper로 맞춰야 하며 각 화면이 다른 라벨을 임의로 만들면 안 된다. [Source: `_bmad-output/planning-artifacts/architecture.md
#Error Handling Pattern`]
