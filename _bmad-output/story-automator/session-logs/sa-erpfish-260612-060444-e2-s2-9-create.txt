+# Story 2.9: 장부 저장 충돌과 미저장 변경 처리
+
+Status: ready-for-dev
+
+<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->
+
+## Story
+
+As a 지점장 또는 본사 사용자,
+I want 같은 장부를 여러 사람이 다룰 때 내 입력이 조용히 덮어써지지 않기를 원한다,
+so that 본사와 지점장이 함께 수정해도 장부 데이터 신뢰를 잃지 않는다.
+
+## Acceptance Criteria
+
+1. Given 사용자가 장부를 불러올 때, when 서버가 장부 데이터를 반환한다, then 장부 또는 입력 섹션의 version/edit token이 함께 제공되어야 한다, and 저장 요청은 마지막으로 읽은 token을 함께 보내야 한다.
+2. Given 다른 사용자가 같은 섹션의 같은 필드를 먼저 저장했을 때, when 사용자가 오래된 token으로 저장을 시도한다, then 서버는 저장을 거부해야 한다, and 내 입력값, 서버 최신값, 마지막 수정자, 수정 시각
을 비교할 수 있는 충돌 정보를 반환해야 한다.
+3. Given 서로 다른 섹션의 변경이 충돌 없이 병합 가능한 경우, when 서버가 저장 요청을 처리한다, then 병합은 감사 로그를 남긴 뒤 허용할 수 있어야 한다, and 변경된 섹션, 필드, actor, 시각이 추적 가능해
야 한다.
+4. Given 본사가 강제 수정 중인 장부 또는 섹션이 있을 때, when 지점장이 해당 영역을 저장하려 한다, then 서버는 최신 상태 재확인을 요구해야 한다, and UI는 `본사 수정 중` 또는 동등한 상태를 표시할 수 있
어야 한다.
+5. Given 사용자가 단계 이동 또는 하단 탭 이동을 시도할 때, when 현재 단계에 미저장 변경이 있다, then 저장, 취소, 계속 편집 중 하나를 선택하게 해야 한다, and 선택 전에는 입력값을 버리거나 자동으로 덮
어쓰면 안 된다.
+6. Given 모바일 또는 브라우저 세션이 오래된 장부 상태를 가지고 있을 때, when 사용자가 저장을 시도한다, then 서버는 오래된 저장을 거부하고 최신 장부를 다시 불러오게 해야 한다, and 사용자가 어떤 값이
저장되지 않았는지 알 수 있어야 한다.

+- 선택 스택은 Next.js App Router, TypeScript, Prisma, PostgreSQL, NextAuth/Auth.js, Tailwind CSS, shadcn/ui, Vercel이다. Story 2.9에서 ORM, auth framework, UI primitive 체계를 바꾸지 않는다. [Source:
 `_bmad-output/planning-artifacts/architecture.md#Starter Template Evaluation`]

 Patterns`]

architecture.md#Authorization Pattern`]
