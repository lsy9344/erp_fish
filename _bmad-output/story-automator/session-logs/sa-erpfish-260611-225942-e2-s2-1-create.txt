+# Story 2.1: 지점+일자 장부 생성과 상태 관리
+
+Status: ready-for-dev
+
+<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->
+
+## Story
+
+As a 지점장 또는 본사 사용자,
+I want 특정 지점과 날짜의 일일 장부를 열거나 만들 수 있기를 원한다,
+so that 하루 운영 기록을 하나의 기준 장부에 입력할 수 있다.
+
+## Acceptance Criteria
+
+1. Given 사용자가 자기 권한 범위 안의 지점과 날짜를 선택했을 때, when 해당 지점+일자 장부가 아직 없으면, then 시스템은 새 일일 장부를 생성할 수 있어야 한다, and 같은 지점+일자에는 하나의 장부만 존재
해야 한다.
+2. Given 같은 지점+일자의 장부가 이미 있을 때, when 사용자가 해당 날짜 장부를 다시 연다, then 기존 장부가 열려야 한다, and 중복 장부가 생성되면 안 된다.
+3. Given 장부가 생성되거나 수정될 때, when 시스템이 장부 상태를 저장한다, then 장부는 `미입력`, `입력중`, `검토대기`, `본사마감`, `휴무` 중 하나의 상태를 가져야 한다, and 상태 배지는 UX 기준에 따라
텍스트 pill로 표시되어야 한다.
+4. Given 지점장 사용자가 다른 지점 장부를 생성하거나 열려고 할 때, when 서버가 권한을 확인한다, then 요청은 거부되어야 한다, and 권한 밖 지점 데이터는 응답에 포함되지 않아야 한다.
+5. Given 장부가 본사 마감 상태일 때, when 사용자가 원본 입력값을 수정하려 한다, then 원본 수정은 차단되어야 한다, and 마감 후 변경은 이후 정정 기록 흐름에서만 처리되어야 한다.
+6. Given 개발자가 데이터 모델을 구현할 때, when 일일 장부 테이블을 정의한다, then `storeId + closingDate` 유일성이 보장되어야 한다, and 장부 저장 충돌 방지를 위한 version 또는 edit token 필드가 있어
야 한다.

+### Data and Security Guardrails
+
+- 장부 생성/열기/저장 대상은 `storeId + closingDate`다. `id`만 믿고 저장하면 권한 밖 ledger id 주입을 놓칠 수 있으므로 server action에서 store access와 ledger store/date를 함께 확인한다.
+- `closingDate`는 `Asia/Seoul` 업무일 기준으로 normalize한다. `new Date("2026-06-11")`의 로컬/UTC 해석 차이를 UI와 서버에 섞지 않는다.
+- `미입력`은 "장부 row 없음"이다. 미입력 표시를 위해 row를 만들면 HQ dashboard summary와 morning meeting 상태가 오염된다.
+- `HEADQUARTERS_CLOSED`는 원본 수정 불가다. 기존 `CorrectionReadonlySummary`와 correction flow를 우회해 ledger 원본 필드를 업데이트하지 않는다.
+- `HOLIDAY` 원본 수정 정책은 현재 action guard에서 수정 불가로 취급한다. 휴무 전환 기능을 넣는다면 별도 status action으로 만들고 audit/version guard를 적용한다.
+- audit payload에는 ledger id, store id, closing date, status, 변경 전/후 값과 actor context를 남긴다. password/session/token/raw credential은 절대 넣지 않는다.
