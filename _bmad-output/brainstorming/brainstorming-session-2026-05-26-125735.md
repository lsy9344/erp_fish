## Technique Execution Working Notes

### Role Playing: 지점장 / 매장 직원

**User-confirmed daily input data:**

- 오늘 받은 상품별 수량
- 상품별 판매 수량
- 상품별 남은 재고
- 폐기/감모 수량
- 총 매출액
- 각 지점장이 등록한 그날 매출, 상세 항목 포함

**[Role Playing #1]: 지점 일일 마감 입력 묶음**
_Concept_: 지점장은 하루 마감 시 상품 흐름 데이터와 매출 데이터를 함께 입력한다. 상품 흐름은 받은 수량, 판매 수량, 남은 재고, 폐기/감모 수량으로 구성되고, 매출은 총액뿐 아니라 상세 항목까지 등록한다.
_Novelty_: 단순 매출 보고가 아니라 `상품 흐름 + 매출 상세 + 재고 검증`을 한 번에 묶어 본사 확인 시간을 줄이는 운영 데이터 구조로 본다.

**[Role Playing #2]: 상품 행 단위 매출/재고 동시 입력**
_Concept_: 지점장은 상품별로 판매 수량과 상품별 매출을 같은 행에서 입력한다. 같은 화면에서 오늘 받은 수량, 판매 수량, 폐기/감모, 남은 재고, 상품별 매출액을 함께 보게 하면 재고 검산과 매출 검산을 동시에 처리할 수 있다.
_Novelty_: 매출 화면과 재고 화면을 분리하지 않고, 상품별 운영 결과를 한 줄에서 끝내는 방식으로 입력 시간을 줄인다.

**Clarification:** 상품별 매출/수량에는 가격 정보가 포함된다. 따라서 상품별 판매 데이터는 최소한 판매 수량, 판매 가격 또는 단가, 상품별 매출액을 함께 다뤄야 한다.

**[Role Playing #2-1]: 가격 포함 상품별 판매 입력**
_Concept_: 지점장은 상품별 판매 수량과 가격을 입력하고, 시스템은 상품별 매출액을 자동 계산하거나 입력값과 비교한다. 본사가 기준 가격을 등록해두면 지점 입력 가격과의 차이도 예외로 잡을 수 있다.
_Novelty_: 단순 매출 합계가 아니라 `상품별 수량 × 가격 = 매출` 구조를 만들어, 재고 이상과 매출 이상을 같은 상품 기준으로 추적한다.

**Clarification:** 가격 정책은 본사가 상품별 기준 가격을 등록하되, 지점장이 할인, 시세, 상품 상태, 크기 차이 등에 따라 실제 판매 가격을 조정할 수 있는 방식으로 본다.

**[Role Playing #2-2]: 기준가 대비 판매가 예외 표시**
_Concept_: 본사가 기준 가격을 등록하고 지점장은 실제 판매 가격을 입력하거나 조정한다. 시스템은 기준가와 실제 판매가의 차이가 큰 상품을 예외로 표시하고, 필요한 경우 할인/상태불량/시세변동/서비스 등 사유를 받는다.
_Novelty_: 수산물처럼 가격 변동이 자연스러운 업종에서 모든 가격 차이를 오류로 보지 않고, 설명이 필요한 차이만 본사 확인 대상으로 올린다.

**Clarification:** MVP에서는 가격 변경 자체는 막지 않고, 기준가 대비 일정 비율 이상 차이가 날 때 사유를 입력하게 한다. 본사 승인까지는 초기 필수 흐름으로 두지 않는다.

**[Role Playing #2-3]: 가격 차이 사유 입력 임계값**
_Concept_: 본사는 기준가 대비 허용 차이율을 설정하고, 지점장이 입력한 실제 판매가가 이 범위를 벗어나면 사유 입력을 요구한다. 예외는 본사 대시보드에서 가격 차이 항목으로 모아 보여준다.
_Novelty_: 현장 가격 조정의 유연성은 살리면서, 대표/본사가 확인해야 할 가격 이상만 자동으로 추려낸다.

### Role Playing: 본사 스태프

**Clarification:** 본사는 지점별로 오늘 출고/전달한 상품과 수량을 미리 입력한다. 지점장은 본사가 등록한 오늘의 상품 목록과 수량을 기준으로 실제 수령, 판매, 폐기/감모, 잔여 재고, 가격/매출 데이터를 입력한다.

**[Role Playing #8]: 본사 출고 기반 지점 마감표 자동 생성**
_Concept_: 본사가 지점별로 오늘 보낸 상품과 수량을 입력하면, 해당 지점의 일일 마감 화면이 자동 생성된다. 지점장은 빈 화면에서 상품을 하나씩 추가하지 않고, 본사가 내려준 상품 목록을 확인하며 판매 수량, 가격, 매출, 폐기/감모, 남은 재고를 채운다.
_Novelty_: 지점장의 입력 시간을 줄이고, 본사 출고 데이터와 지점 마감 데이터를 같은 기준으로 비교할 수 있게 만든다.

**Clarification:** 지점장은 본사 출고 수량과 실제 수령 수량이 다를 경우 다르게 입력할 수 있어야 하며, 차이가 있으면 사유를 입력한다. 초기에는 본사 승인보다 사유 기록과 예외 표시를 우선한다.

**[Role Playing #9]: 출고-수령 차이 예외 기록**
_Concept_: 본사가 입력한 출고 수량과 지점장이 확인한 실제 수령 수량을 분리해서 저장한다. 두 수량이 다르면 지점장은 사유를 입력하고, 본사는 대시보드에서 수령 차이 예외를 확인한다.
_Novelty_: 본사 데이터가 항상 정답이라는 전제를 버리고, 현장에서 발생하는 누락, 배송 차이, 검수 차이를 감사 가능한 데이터로 남긴다.

**Clarification:** 수산물은 시세 변동이 있으므로, 본사는 지점별 출고 입력 시 상품, 수량, 그날 기준가를 함께 내려보낼 수 있어야 한다. 기준가는 상품 마스터의 기본값을 가져오되 출고 당일 조정 가능하게 본다.

**[Role Playing #10]: 출고일 기준가 스냅샷**
_Concept_: 본사가 지점별 출고를 입력할 때 상품, 출고 수량, 그날 기준가를 함께 저장한다. 상품 마스터의 기본 기준가는 유지하되, 실제 마감 검산에는 출고일에 내려간 기준가 스냅샷을 사용한다.
_Novelty_: 가격이 자주 바뀌는 수산물 특성을 반영해, 과거 마감 데이터가 나중에 기준가 변경 때문에 흔들리지 않게 만든다.

### Role Playing: 지점장 입력 시간 단축 UX

**[UX #11]: 예외만 입력하는 빠른 마감 모드**
_Concept_: 본사가 내려준 상품 목록, 출고 수량, 기준가가 기본으로 채워져 있고 지점장은 실제로 달라진 값만 수정한다. 판매 수량, 남은 재고, 폐기/감모만 빠르게 입력하고, 기준과 다른 부분에만 사유를 넣는다.
_Novelty_: 모든 칸을 새로 쓰는 방식이 아니라, “오늘 본사가 알고 있는 값에서 무엇이 달라졌는가”만 입력하게 해서 시간을 줄인다.

**Clarification:** 지점장 입력 화면은 모바일/태블릿에서는 상품 카드형, PC에서는 표 형태를 우선한다.

**[UX #12]: 기기별 마감 입력 레이아웃**
_Concept_: 모바일과 태블릿에서는 상품 하나씩 카드로 넘기며 수량, 가격, 폐기/감모, 잔여 재고를 입력하고, PC에서는 여러 상품을 표 형태로 빠르게 입력한다. 같은 데이터 구조를 유지하되 기기별 입력 방식만 다르게 제공한다.
_Novelty_: 현장 입력과 사무실 입력의 맥락을 분리해, 지점장의 입력 부담을 줄이면서 본사 검토에는 구조화된 데이터를 제공한다.

**[UX #13]: 남은 재고 또는 판매 수량 역산**
_Concept_: 지점장이 실제 수령 수량, 폐기/감모, 남은 재고를 입력하면 시스템이 판매 수량을 역산할 수 있고, 반대로 판매 수량을 입력하면 남은 재고를 계산할 수 있다. 품목별로 어느 값을 직접 입력할지 다르게 둘 수 있다.
_Novelty_: 현장에서는 어떤 값이 더 세기 쉬운지 상품마다 다르므로, 하나의 입력 방식만 강제하지 않는다.

**Clarification:** 지점장이 더 쉽게 아는 값은 상품마다 다르다. 따라서 품목별 또는 마감 행별로 입력 기준을 유연하게 지원해야 한다.

**[UX #14]: 품목별 기본 입력 모드**
_Concept_: 본사는 상품별로 기본 입력 모드를 설정한다. 예를 들어 어떤 상품은 판매 수량 중심, 어떤 상품은 잔여 재고 중심, 어떤 상품은 중량 중심으로 입력하게 하고 시스템이 나머지 값을 계산한다.
_Novelty_: 수산물 품목의 단위 혼합 문제를 입력 UX에서 흡수해, 지점장이 매번 계산 방식을 고민하지 않게 한다.

### Role Playing: 대표님 / 본사 대시보드

**[HQ #15]: 매장 상태 신호등 대시보드**
_Concept_: 본사는 매장별로 정상, 주의, 확인 필요, 미마감 상태만 먼저 본다. 정상 매장은 접어두고, 확인 필요한 매장만 상단에 노출한다.
_Novelty_: 대표님이 장부 전체를 읽는 시간을 줄이고, 아침에는 판단이 필요한 매장만 보게 한다.

**Clarification:** 대표님은 아침에 문제가 있는 매장 순서로 먼저 보고 싶어한다.

**[HQ #16]: 문제 매장 우선 정렬**
_Concept_: 본사 대시보드는 매장 목록을 단순 가나다순이나 매출순이 아니라 문제 심각도 순으로 정렬한다. 수령 차이, 재고 차이, 매출 계산 차이, 가격 예외, 미마감 등을 종합해 문제가 큰 매장을 위에 배치한다.
_Novelty_: 대표님이 “어디를 먼저 봐야 하는지” 판단하는 시간을 줄이고, 매장 수가 늘어도 확인 순서가 자동으로 만들어진다.

**Clarification:** 문제 매장 정렬에서 가장 심각한 기준은 재고 차이다.

**[HQ #17]: 재고 차이 최우선 예외 점수**
_Concept_: 본사 대시보드는 재고 차이가 있는 매장을 가장 높은 우선순위로 올린다. 미마감, 수령 차이, 가격 차이도 중요하지만, 재고 차이는 상품 흐름과 매출 신뢰성 전체를 흔드는 신호로 보고 가장 먼저 확인하게 한다.
_Novelty_: 단순 예외 개수 기준이 아니라, 수산물 운영에서 가장 손실과 오류 가능성이 큰 재고 차이를 대표님 확인 순서의 핵심 기준으로 삼는다.

**Clarification:** 재고 차이 확인 화면은 먼저 매장별로 보고, 필요하면 상품별 상세로 내려가는 방식이 적합하다.

**[HQ #18]: 매장 우선, 상품 드릴다운**
_Concept_: 대표님은 첫 화면에서 문제가 있는 매장을 우선 보고, 특정 매장을 클릭하면 해당 매장의 문제 상품 목록을 확인한다. 상품별 상세에서는 출고 수량, 실제 수령, 판매 수량, 폐기/감모, 남은 재고, 차이, 사유를 한 번에 본다.
_Novelty_: 대표님의 판단 단위인 매장을 먼저 유지하면서, 상세 분석은 필요한 경우에만 열어 확인 시간을 줄인다.

**[HQ #19]: 확인 완료 처리**
_Concept_: 본사가 예외를 확인한 뒤 `확인 완료` 상태를 남길 수 있다. MVP에서는 추가 요청, 보정 요청, 다단계 승인까지 확장하지 않고, 확인한 예외와 아직 확인하지 않은 예외를 구분하는 데 집중한다.
_Novelty_: 대시보드를 단순 조회 화면에서 최소한의 업무 처리 화면으로 바꾸되, 초기 제품의 복잡도를 낮춘다.

**[Role Playing #3]: 상품 흐름 자동 검산**
_Concept_: 지점장이 상품별로 받은 수량, 판매 수량, 남은 재고, 폐기/감모를 입력하면 시스템이 자동으로 `받은 수량 - 판매 수량 - 폐기/감모 - 남은 재고 = 차이`를 계산한다. 차이가 0이 아니면 즉시 표시하고, 지점장이 사유를 입력해야 마감할 수 있게 한다.
_Novelty_: 대표님이 아침에 재고 이상을 찾는 게 아니라, 지점장이 마감하는 순간 이상을 먼저 잡게 만든다.
