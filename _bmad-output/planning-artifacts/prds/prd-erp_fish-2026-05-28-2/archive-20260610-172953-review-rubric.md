# PRD Quality Review — ERP Fish PRD

## Overall verdict
이 PRD는 "본사 관제탑"이라는 제품 중심축, 장부 상태 모델, 본사 마감/정정 원칙, 지점장 입력 흐름을 잘 잡고 있어 MVP 방향을 공유하기에는 충분히 쓸 수 있다. 다만 §8의 "추가 구현 범위"가 §6의 제외 범위와 한 문서 안에서 겹치고, CAP 요구사항들이 성공 지표와 완료 기준에 충분히 연결되지 않아 이후 에픽/스토리 작성자는 범위와 우선순위를 다시 해석해야 한다. 결론적으로 MVP 기준 PRD로는 adequate, 확장 범위까지 포함한 빌드 지시서로는 보강이 필요하다.

## Decision-readiness — adequate
의사결정자가 MVP의 핵심 선택은 읽고 판단할 수 있다. §1은 제품을 "모든 지점의 일일 장부를 확인"하고 "최종 마감할 수 있는 관제탑"으로 정의하고, §3.1은 미입력/입력 중/검토 대기/본사 마감의 상태와 수정 가능 여부를 표로 닫는다. §10도 18개 Open Questions에 `스토리 작성 전 필수`, `Epic 6 구현 전 필수`, `계약 전 필수` 같은 결정 등급을 붙여 미확정 항목을 숨기지 않는다.

하지만 "무엇을 지금 만들 것인가"라는 결정은 §6, §7, §8을 함께 읽어야만 이해된다. §6은 텔레그램 알림, 이카운트 연동, 특수 기간 비교, 직원 마스터를 제외한다고 쓰고, §8은 같은 축의 LINE/텔레그램 알림, 이카운트 엑셀 업로드, 특수기간 비교, 직원 마스터를 "추가 구현 범위"로 포함한다. §6의 "§6을 1차 범위 방어선으로 사용"이라는 문장과 §8의 "추가 견적 기준으로 ... 포함된 요구사항"이 모두 맞더라도, downstream reader에게는 릴리스 라인이 분리되어 보이지 않는다.

### Findings
- **high** MVP 제외와 추가 구현 범위가 같은 항목을 서로 다른 방식으로 말함 (§6, §7.2, §8) — §6은 "텔레그램 알림", "이카운트 연동", "설날 등 특수 기간 비교 리포트", "직원 마스터"를 제외하지만, §8은 CAP-11, CAP-6, CAP-2, CAP-1로 이를 추가 구현 범위에 넣는다. §6이 "1차 범위 방어선"이라고 되어 있어 범위 승인자가 어느 문장을 기준으로 계약/스토리를 자를지 흔들린다. *Fix:* §6/§7/§8 앞에 릴리스 구분 표를 추가해 각 항목을 `MVP`, `추가 구현 승인`, `계약/운영 별도`, `후순위 검토`로 한 번에 정리하고, §6의 제외 목록에는 "MVP 제외, §8에서 추가 구현으로 승격된 항목은 별도"라고 명시한다.
- **medium** CAP별 미결정 질문 연결이 약함 (§8, §10) — CAP-7은 FIFO 범위를 다루지만 OQ-7/OQ-17과 직접 연결하지 않고, CAP-14는 희망 판매가 손실액을 요구하지만 OQ-9와 연결하지 않는다. CAP-11도 알림 채널과 장기 체화 기준이 OQ-13/OQ-16에 남아 있다. *Fix:* 각 CAP 끝에 `Blockers/Open Questions` 줄을 붙여 관련 OQ ID와 결정 전까지 가능한 설계 범위를 명시한다.

## Substance over theater — strong
대부분의 내용은 실제 업무 관찰에서 나온 요구로 보인다. §2.1의 JTBD는 "매일 아침 회의 전에", "본사 마감 전까지", "엑셀 수식이나 서식을 깨뜨릴 걱정 없이"처럼 구체적인 운영 상황을 담고, addendum도 `total_sales.xlsx`, `장부-202605현대 (1).xlsx`, `docs/meeting/change.md`에서 추출한 관찰을 본문과 분리해 보존한다.

NFR도 복사한 문구보다 제품 특화 기준이 많다. §5의 "본사 마감 후 원본 장부는 보존", "최소 390px 폭의 모바일 화면", "10개 내외 지점 기준 3초"는 이 ERP Fish의 업무 환경과 연결된다. CAP-13의 민감 지표 숨김도 단순 권한 문구가 아니라 "본사 전용 고정비와 손익 구조를 모르는 지점 직원"이라는 이유를 갖고 있다.

### Findings

## Strategic coherence — adequate
MVP의 전략은 선명하다. §1의 "매일 아침 지점장 회의의 기준 자료"가 FR-15 관제판, FR-16 이상 신호, FR-19 본사 마감, FR-20 정정 기록, FR-27 아침 회의 리포트를 하나의 이야기로 묶는다. §9의 SM-1~SM-3도 아침 회의 준비, 본사 마감 통제, 마감 후 추적 가능성으로 같은 thesis를 검증한다.

전략적 약점은 §8이 MVP thesis 밖으로 크게 넓어진 뒤 별도 성공 기준을 받지 않는다는 점이다. 직원/급여, FIFO 원가, 본사 통합 재고, 리포트 고도화, LINE/텔레그램, 유지보수 계약까지 들어오지만 §9 성공 지표는 여전히 FR-15~FR-27 중심이다. 추가 구현 범위가 승인된 범위라면, 그것이 본사 관제탑 thesis를 어떻게 강화하는지 또는 어떤 phase thesis를 갖는지 더 분명해야 한다.

### Findings
- **high** 추가 구현 범위의 성공 기준이 없음 (§8, §9) — §8은 CAP-1~CAP-19까지 넓은 구현 범위를 추가하지만 §9는 SM-1~SM-5 모두 MVP FR만 참조한다. 예를 들어 CAP-6 업로드 매입, CAP-7 FIFO, CAP-11 알림, CAP-10 월 손익은 성공/역효과 지표가 없어 기능 완료 후 "좋은 결과"를 판단하기 어렵다. *Fix:* §9에 `추가 구현 범위 성공 지표`를 추가하고 CAP 묶음별로 운영 결과와 counter-metric을 둔다. 예: 업로드 매입은 누락/오입력 감소, FIFO는 원가 근거 확인 가능성, 알림은 위험 탐지 누락 방지와 알림 과다 방지.
- **medium** CAP 우선순위와 phase 논리가 ID나 배치에서 드러나지 않음 (§8) — §8.1은 CAP-1 뒤에 CAP-9가 오고, §8.3은 CAP-8 뒤에 CAP-4가 온다. 번호가 입력 순서나 epic 순서일 수는 있지만, PRD만 보면 어떤 CAP이 먼저 구현되어야 하는지 알기 어렵다. *Fix:* CAP 번호를 유지하더라도 각 CAP에 `phase`, `depends on`, `priority rationale`를 추가하거나, 별도 표로 구현 순서와 의존성을 요약한다.

## Done-ness clarity — adequate
FR-1~FR-29에는 대부분 검증 조건이 있고, 많은 조건이 테스트 가능한 형태다. 예를 들어 FR-2는 지점장이 다른 지점 목록과 상세에 접근할 수 없다고 하고, FR-19는 본사 마감 후 원본 장부 입력값을 수정할 수 없다고 한다. §3.1의 상태 모델과 §4.3의 계산 기준 표도 완료 기준을 만드는 데 도움이 된다.

약한 부분은 §8 CAPs와 일부 운영/외부 연동 조건이다. CAP-11의 "재시도 정책을 적용할 수 있다"와 "운영 토큰과 외부 API 키는 안전하게 관리한다"는 맞는 방향이지만, 몇 번 재시도하는지, 실패 상태를 어디서 보는지, 어떤 보안 기준을 통과해야 하는지 알 수 없다. CAP-12의 "중요한 운영 데이터"도 좋은 의도지만 어떤 필드가 구조화되어야 하는지 확정하지 않는다.

### Findings
- **medium** 외부 알림과 보안 완료 기준이 검증 가능하게 닫히지 않음 (§8.6 CAP-11) — CAP-11은 "발송 실패가 로그에 남고 재시도 정책을 적용할 수 있다", "운영 토큰과 외부 API 키는 안전하게 관리한다"고 하지만 재시도 횟수, 실패 노출 방식, 토큰 저장/회전/권한 기준이 없다. *Fix:* 알림 실패 상태, 재시도 횟수와 간격, 운영자 확인 화면, 비밀값 저장 방식, 권한/감사 로그 기준을 검증 조건으로 추가한다.
- **medium** 분석 확장성 요구가 데이터 계약으로 내려오지 않음 (§8.6 CAP-12) — CAP-12는 "중요한 운영 데이터를 텍스트 메모에만 묻어두지 않는다"고 하지만 어떤 특이사항, 근무 배치, 직원/지점/일자 관계가 구조화 필드인지 정하지 않는다. *Fix:* 향후 AI가 아니라도 리포트/검색에 필요한 최소 구조화 필드 목록과 메모로 남겨도 되는 항목을 표로 분리한다.
- **low** 성능 NFR이 MVP 관제판 한 지점 수 기준에만 묶임 (§5) — §5는 "10개 내외 지점 기준 3초"를 제시해 좋은 출발점이지만, §8의 통합 재고, 기간 비교, FIFO 근거 조회 같은 무거운 화면에는 성능 기준이 없다. *Fix:* 주요 조회 유형별 기준 데이터량과 목표 응답 시간을 별도 NFR로 추가한다.

## Scope honesty — adequate
문서는 범위를 숨기지 않으려는 태도가 좋다. §6은 1차 제외 범위를 노골적으로 열거하고, §8은 "기존 §6 제외 범위가 삭제되는 것은 아니며"라고 전제해 MVP와 추가 구현을 구분하려 한다. §11도 inline `[ASSUMPTION]`이 없고 남은 결정은 §10에 분리했다고 명시한다.

다만 범위가 "정직하지만 읽기 어렵다." 제외, 포함, 추가 견적, 계약 조건이 한 PRD 안에 들어오면서 범위 판단이 문장 해석에 의존한다. CAP-19는 유지보수/서버 운영을 다루며 "제품 요구사항이 아니라 계약 조건"이라고 쓰는데, 같은 섹션의 검증 조건처럼 보인다.

### Findings
- **medium** 계약/운영 조건이 제품 요구사항과 같은 레벨에 놓임 (§8.7 CAP-19) — CAP-19는 "제품 요구사항이 아니라 계약 조건으로 별도 확정"한다고 하면서도 §8 추가 구현 범위의 CAP로 들어가 있다. 이 상태면 에픽/스토리 생성 시 제품 기능으로 오해될 수 있다. *Fix:* CAP-19를 `운영/계약 결정사항` 섹션으로 분리하고, 제품에 남길 것은 로그/설정/점검 화면 같은 시스템 요구사항만 FR/CAP로 남긴다.

## Downstream usability — adequate
하위 문서가 source-extract하기 좋은 기반은 있다. FR-1~FR-29는 연속이고, UJ-1~UJ-4는 이름 있는 주체를 가진다. 용어집도 본사 마감, 정정 기록, 전일재고 후보, 확정 이월 기준, 업로드 매입처럼 데이터 모델에 필요한 명사를 많이 잡아 둔다.

하지만 §8 CAP 묶음은 FR/SM/UJ와 연결이 약하고, CAP ID 순서도 읽기 부담을 만든다. 에픽/스토리 작성자는 CAP-6이 FR-8/FR-25와 어떤 관계인지, CAP-13이 FR-13/FR-28의 지점장 화면/API 응답과 어디까지 충돌하는지 별도로 추적해야 한다. 용어도 본문에서는 `지점`이 표준인데 일부 리포트와 계약 문장에는 `매장`이 남아 있다.

### Findings
- **medium** CAP 요구사항이 FR/SM/UJ에 연결되지 않아 추출 비용이 큼 (§4, §8, §9) — CAP-6 업로드 매입은 FR-8/FR-25와, CAP-13 민감 지표 숨김은 FR-13/FR-28/FR-29와, CAP-15 일괄 마감은 FR-19와 강하게 연결되지만 문서에는 cross-reference가 없다. *Fix:* CAP별로 `Related FR`, `Related SM`, `Related OQ`를 붙여 후속 UX/architecture/story 작업이 같은 requirement를 중복 해석하지 않게 한다.
- **low** 제품 용어 `지점`과 원본/보고 용어 `매장`이 섞임 (§3, §8.5, addendum §2) — §3은 `지점`을 정의하지만 CAP-10은 "매장별 월 고정비", §8.5는 "매장 손익계산서"라고 쓴다. addendum의 원본 표현 때문일 수 있지만 데이터 모델 용어로는 흔들릴 수 있다. *Fix:* 제품 표준 용어는 `지점`으로 통일하고, 원본 엑셀/고객 문서의 `매장`은 `매장=지점` 매핑으로 설명한다.
- **low** CAP ID 순서가 downstream 참조성을 떨어뜨림 (§8) — CAP-1, CAP-9, CAP-5, CAP-6, CAP-7처럼 번호가 섹션 순서와 맞지 않아 "다음 CAP"를 따라가기 어렵다. *Fix:* 기존 ID를 유지해야 한다면 §8 첫머리에 CAP 목록 표를 두고 번호, 제목, 섹션, phase를 한눈에 보이게 한다.

## Shape fit — adequate
내부 ERP, 본사/지점장 양면 업무 도구라는 제품 형태에는 대체로 잘 맞는다. PRD는 랜딩 페이지식 설명이 아니라 capability spec과 검증 조건 중심이고, 사용자 여정도 네 개로 제한되어 과하지 않다. Brownfield 성격도 addendum에 원본 엑셀 관찰을 보존하고 본문에서는 "엑셀 화면을 그대로 복제하지 않고" 업무 목적과 데이터 흐름으로 재정의한다.

형태상 가장 아쉬운 점은 하나의 final PRD가 MVP 기준선, 추가 구현 견적 범위, 운영 계약 조건을 모두 담는 혼합 문서가 되었다는 점이다. §8이 별도 phase extension으로 잘 표시되면 괜찮지만, 지금은 본문 길이와 CAP 범위가 MVP의 선명한 모양을 흐린다.

### Findings
- **medium** PRD가 MVP spec과 phase-extension annex 사이에 걸쳐 있음 (§0, §6, §8) — §0은 "1차 제품 범위"를 정의한다고 하면서 동시에 "2026-06-10에 승인된 추가 구현 범위"를 다룬다. §8은 실제로 phase 2+ backlog/견적 범위에 가까워 MVP PRD의 shape를 넓힌다. *Fix:* 문서 제목 또는 §0에 `MVP baseline + approved extension`이라고 명명하고, §8을 별도 extension annex처럼 읽히게 요약 표, phase, dependency, success metric을 추가한다.

## Mechanical notes
- FR ID는 FR-1~FR-29까지 연속이고 중복이 보이지 않는다.
- UJ ID는 UJ-1~UJ-4까지 연속이며 각 여정에 김관리자, 현대 지점장, 이스태프처럼 주체가 있다.
- SM ID는 SM-1~SM-5와 SM-C1~SM-C2로 구분되어 있고, 주요 SM은 관련 FR을 참조한다.
- CAP ID는 CAP-1~CAP-19가 모두 존재하지만 문서 순서가 숫자 순서와 다르다.
- Assumptions Index는 inline `[ASSUMPTION]` 태그가 없다는 현재 상태와 맞다.
- §10 Open Questions는 결정 등급과 담당/재확인 시점을 갖고 있어 이전보다 downstream 사용성이 좋아졌다.
