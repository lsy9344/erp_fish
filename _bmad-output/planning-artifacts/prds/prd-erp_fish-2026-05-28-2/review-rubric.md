# PRD Quality Review — ERP Fish PRD

## Overall verdict
이 PRD는 제품의 중심 논리인 "본사 관제탑"과 1차 범위, 사용자 여정, 기능 ID가 잘 맞물려 있어 기획 방향을 공유하기에는 충분히 강하다. 다만 이상 신호와 리포트의 핵심이 되는 계산식, 기준값, 상태 정의 일부가 아직 Open Questions에 남아 있어 에픽/스토리 작성과 구현 착수 단계에서는 재해석 비용이 크다. 결론적으로 방향성 PRD로는 좋지만, 빌드 지시서로 쓰려면 계산/상태/운영 기준을 먼저 닫아야 한다.

## Decision-readiness — adequate
의사결정자가 1차 제품의 방향은 이해할 수 있다. §1은 제품을 "모든 지점의 일일 장부를 확인"하고 "최종 마감할 수 있는 관제탑"으로 정의하고, §6과 §7은 텔레그램, POS/카드 매출 자동 연동, AI 이미지 식별 등 1차 제외 범위를 분명히 둔다.

하지만 핵심 판단 기준 일부가 아직 결정되지 않았다. 특히 §3은 `매출차액`을 "정확한 계산 정의는 Open Questions에 남긴다"고 하고, §9는 `매출차액`, `30%단가`, `전일 이월금액`, 휴무일/미입력일, `이월` 시트 역할을 모두 열어 둔다. 이 항목들은 단순 후속 질문이 아니라 관제판, 이상 신호, 리포트 숫자를 좌우하는 결정이다.

### Findings
- **high** 핵심 운영 숫자의 결정이 아직 열려 있음 (§3, §4.3, §9; addendum §4) — PRD가 `매출차액`을 관제판과 이상 신호의 핵심으로 반복 사용하지만, §3은 "정확한 계산 정의는 Open Questions에 남긴다"고 하고 §9도 계산식과 허용 기준을 미확정으로 둔다. addendum §4의 `30%단가`, `전일 이월금액`, 휴무일/미입력일도 같은 성격이다. *Fix:* MVP 착수 전 필수 결정으로 묶어 계산식, 입력 필드, 허용 기준, 휴무/미입력 상태값을 표로 확정하고 관련 FR의 검증 조건에 반영한다.

## Substance over theater — strong
내용은 대체로 실제 업무에서 나온 요구로 보인다. §2.1의 JTBD는 본사 회의 전 확인, 본사 직접 수정/마감, 지점장 일일 입력처럼 구체적인 업무 동기로 쓰여 있고, §2.4의 UJ-1~UJ-4도 본사와 지점장의 실제 하루 흐름을 기준으로 한다.

차별화나 비전 문구가 과장되어 있지 않고, addendum도 엑셀 원본에서 관찰한 내용을 본문에 억지로 모두 넣지 않고 보존 자료로 분리했다. §0의 "엑셀 화면을 그대로 복제하지 않고"라는 선택도 제품 방향을 실제 제약과 연결한다.

### Findings

## Strategic coherence — strong
PRD의 논리는 선명하다. 핵심 베팅은 엑셀 장부를 단순 화면 복제가 아니라 본사 회의 전 관제, 이상 탐지, 마감/정정 추적 흐름으로 바꾸는 것이다. §1의 "매일 아침 지점장 회의의 기준 자료"라는 문장이 기능 묶음의 기준점 역할을 한다.

기능도 이 논리와 대체로 맞는다. FR-14~FR-16은 관제판과 이상 신호, FR-17~FR-20은 본사 수정/마감/정정, FR-26은 아침 회의 리포트를 담당한다. §8의 SM-1~SM-3과 Counter-metrics도 "빠른 입력"보다 검증과 이상 확인을 우선한다는 방향을 보강한다.

### Findings

## Done-ness clarity — thin
각 FR에 "검증 조건"이 붙어 있어 기본적인 완료 기준은 있다. 예를 들어 FR-18은 "본사 마감 후 원본 장부 입력값은 수정할 수 없다"고 하며, FR-19는 정정 기록에 작성자, 시각, 원본값, 정정값, 사유가 남아야 한다고 적어 테스트 가능한 조건을 준다.

반면 계산, 리포트, 비기능 요구사항은 아직 구현자가 같은 결과를 만들 만큼 닫혀 있지 않다. FR-12는 여러 지표를 "계산한다"고 하지만 계산 기준 미확정 항목은 Open Questions로 남긴다고 하며, FR-27과 FR-28은 비교/요약 항목은 말하지만 기간 기준, 집계 방식, 정정 반영 방식, 빈 장부 처리 같은 완료 조건은 부족하다. §5의 "지연 없이"도 측정 가능한 성능 기준이 아니다.

### Findings
- **medium** 리포트 FR의 완료 조건이 기능명 수준에 머묾 (§4.7) — FR-27은 "선택 기간의 지점별 실적을 비교"하고 FR-28은 "월간 실적, 마감 현황, 주요 이상 항목"을 보여준다고 하지만, 기간 경계, 미마감 장부 포함 방식, 정정 반영값 적용 여부, 비교 기준일 같은 검증 조건이 없다. *Fix:* 각 리포트별 입력 조건, 집계 기준, 정정 반영 규칙, 빈/미마감 데이터 표시 규칙을 검증 조건으로 추가한다.
- **medium** 비기능 요구사항이 측정 가능한 경계로 닫히지 않음 (§5) — "지연 없이 보여주는 것을 목표"는 테스트할 수 없고, 반응형 웹도 최소 지원 해상도나 필수 입력 흐름 기준이 없다. *Fix:* 관제판 초기 조회 목표 시간, 지점 수 기준, 모바일/태블릿 최소 폭, 서버 권한 강제 검증 항목처럼 측정 가능한 경계로 바꾼다.

## Scope honesty — adequate
범위 방어는 좋다. §6은 1차 제외 범위를 명확히 열거하고, §7.2는 이를 "UX, 아키텍처, 에픽/스토리 작성 시 §6을 1차 범위 방어선"으로 쓰라고 한다. §10도 확정되지 않은 추론을 가정 태그로 숨기지 않고 §9 Open Questions로 분리했다고 밝힌다.

다만 Open Questions가 제품의 핵심 계산과 데이터 상태를 포함하는데, 어떤 질문이 MVP 착수 전 차단 항목인지, 어떤 질문이 구현 중 상세화 가능한 항목인지 구분되어 있지 않다. 이 상태에서는 후속 담당자가 질문 목록의 위험도를 다시 판단해야 한다.

### Findings
- **medium** Open Questions의 차단 여부가 구분되지 않음 (§9) — §9의 8개 질문 중 `매출차액` 계산식, 휴무일/미입력일 상태, `이월` 시트 역할은 구현 모델과 화면 검증을 막을 수 있지만 모두 같은 무게의 목록으로 보인다. *Fix:* 각 질문에 `MVP 착수 전 필수`, `스토리 작성 전 필수`, `구현 중 확인 가능` 같은 결정 등급과 담당자/재확인 시점을 붙인다.

## Downstream usability — adequate
하위 작업으로 넘기기 쉬운 구조가 있다. FR-1~FR-28은 연속 ID를 가지고, UJ-1~UJ-4와 SM-1~SM-5도 고유 ID가 있다. §8의 성공 지표는 관련 FR을 직접 참조하고, §3 용어는 본사 마감, 정정 기록, 정정 반영값, 이상 신호 같은 핵심 도메인 명사를 한곳에 모은다.

다만 용어와 데이터 경계는 조금 더 다듬어야 한다. 본문은 주로 `지점`을 쓰지만 §4.7과 §8에는 `매장별`이 섞여 있고, addendum은 엑셀 원본의 `매장`, `지점`, `근무표`, `이월` 표현을 함께 보존한다. 지금 수준은 읽을 수 있지만, 데이터 모델과 API 이름을 뽑는 단계에서는 통일이 필요하다.

### Findings
- **low** `지점`과 `매장` 용어가 혼용됨 (§3, §4.7, §8; addendum §2~§3) — PRD 용어집은 `지점`을 정의하지만 FR-27은 "매장별 기간 비교", FR-28은 "월간 매장 요약"이라고 쓴다. addendum도 원본 표현상 `매장`과 `지점`이 같이 등장한다. *Fix:* 제품 용어는 `지점`으로 통일하고, 원본 엑셀 명칭을 보존해야 할 때만 addendum 또는 매핑 표에 `매장=지점`을 명시한다.

## Shape fit — strong
제품 형태에 잘 맞는 PRD다. 내부 ERP이고 본사/지점장이라는 두 역할이 함께 쓰는 업무 도구라서, §2.4의 사용자 여정은 과하지 않으면서도 핵심 작업 흐름을 보여준다. 동시에 본문 대부분은 FR과 검증 조건 중심의 capability spec 형태라 내부 도구에 맞는다.

Brownfield 성격도 무리 없이 처리했다. 기존 원드라이브 엑셀의 구조와 관찰은 addendum에 남기고, 본문은 "엑셀 화면을 그대로 복제하지 않고" 업무 목적과 데이터 흐름을 기준으로 요구사항을 정리한다. 이는 화면 복제 요구와 제품화 요구 사이의 중요한 선택을 잘 반영한다.

### Findings

## Mechanical notes
- FR ID는 FR-1~FR-28까지 연속이고 중복이 보이지 않는다.
- UJ ID는 UJ-1~UJ-4까지 연속이며 각 여정에 김관리자, 현대 지점장, 이스태프처럼 주체가 있다.
- SM ID는 SM-1~SM-5와 SM-C1~SM-C2로 구분되어 있고, 주요 SM은 관련 FR을 참조한다.
- Assumptions Index는 inline `[ASSUMPTION]` 태그가 없다는 현재 상태와 맞다.
- `매출차액`, `30%단가`, `전일 이월금액`, `이월`은 후속 문서에서 정의 확정 전까지 같은 의미로 쓰이지 않도록 주의가 필요하다.
