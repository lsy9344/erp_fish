# PRD Quality Review — ERP Fish PRD

## Overall verdict
이 PRD는 본사 관제탑 중심의 MVP와 2026-06-10 승인 추가 구현 범위를 비교적 정직하게 분리했고, FR/CAP별 검증 조건도 대부분 후속 UX, 아키텍처, 스토리 작성자가 추출할 수 있는 수준이다. 다만 문서 스스로 `draft`와 구현 게이트를 선언하듯이, 계산/FIFO/권한/리포트 계약의 일부 결정은 아직 닫히지 않았고, 승인 추가 구현 범위의 일부 CAP 추적성이 끊겨 최종 구현 승인용 PRD로 쓰기에는 추가 정리가 필요하다.

## Decision-readiness — adequate
PRD는 상태를 숨기지 않는다. §0.1은 현재 문서를 "**draft** 상태"라고 못박고, "그대로 에픽/스토리 생성 가능한 최종 PRD"로 보지 말라고 경고한다. 또한 G1~G5 게이트가 릴리스 경계, 계산 정책, FIFO, 권한/감사, 리포트 데이터 계약을 구현 전 확인 대상으로 분리한다.

결정 기준도 꽤 분명하다. §0.2는 MVP 필수, 승인 추가 구현, 계약/운영 별도, 후순위/명시 제외를 표로 나누고, §10은 Open Questions마다 결정 등급과 영향 범위를 둔다. 예를 들어 OQ-1~OQ-3은 `스토리 작성 전 필수`로 지정되어 있어 어떤 결정이 스토리 생성을 막는지 알 수 있다.

약점은 승인 추가 구현 범위의 배포 판단이 아직 에픽 계획으로 밀려 있다는 점이다. §0.2는 CAP-1~CAP-18에 대해 "MVP와 같은 배포에 묶을지, 후속 배포로 뺄지는 에픽 계획에서 결정"한다고 한다. 이는 정직하지만, 의사결정자가 이 PRD만 보고 MVP와 추가 범위의 실제 출시 경계를 확정하기에는 부족하다.

### Findings
- **high** 승인 추가 구현의 실제 릴리스 경계가 아직 PRD 밖에 있다 (§0.2) — CAP-1~CAP-18은 승인 범위로 정리되었지만, "MVP와 같은 배포에 묶을지, 후속 배포로 뺄지는 에픽 계획에서 결정"한다고 되어 있다. 이 상태에서는 견적/일정/스토리 분할 결정이 PRD 바깥에서 다시 흔들릴 수 있다. *Fix:* CAP별로 `MVP와 동시`, `MVP 직후`, `별도 계약/운영`, `후순위` 같은 릴리스 판정을 PRD 안에 추가하고, 아직 미정인 항목은 OQ 또는 게이트에 연결한다.
- **medium** 게이트가 닫히는 절차가 결과 조건 중심이라 실행 소유자가 약하다 (§0.1) — G2~G5는 "정책이 확정된다", "계약이 반영된다"처럼 닫히는 조건은 말하지만, 누가 어떤 산출물로 닫는지는 약하다. *Fix:* 각 게이트에 owner와 증빙 산출물, 예: `PM 승인 계산표`, `API 권한 매트릭스`, `리포트 컬럼 계약`, `FIFO 정책 메모`를 붙인다.

## Substance over theater — strong
문서 내용은 템플릿을 채우기 위한 장식보다 실제 업무에서 나온 요구에 가깝다. 비전은 "본사가 모든 지점의 일일 장부를 확인하고... 최종 마감할 수 있는 관제탑"이라는 구체적 운영 역할로 좁혀져 있고, 사용자 여정도 김관리자, 현대 지점장, 이스태프의 실제 업무 흐름을 기준으로 한다.

NFR과 확장 요구도 제품 특화도가 높다. 예를 들어 §5는 10개 내외 지점, 3초 조회, 390px 모바일 폭, 최신 Chrome/Edge와 모바일 Chrome/Safari 같은 검증 조건을 둔다. §8.2의 이카운트 업로드는 `.xlsx`, 필수 논리 필드, 헤더 매핑, 파일 해시 또는 행 내용 기반 중복 키까지 들어 있어 단순한 "업로드 가능" 요구가 아니다.

일부 문장은 여전히 일반적이다. CAP-12의 "향후 분석 가능한 구조로 저장한다"는 앞선 구조화 필드 목록이 있어 완전히 빈말은 아니지만, 어디까지 저장해야 충분한지 판단하기 어렵다. 그래도 전체적으로는 고객 자료와 엑셀 관찰에서 추출된 실무 요구가 중심이다.

### Findings
- **low** CAP-12의 마지막 검증 조건이 앞 조건을 반복한다 (§8.6) — "향후 분석 가능한 구조로 저장한다"는 CAP-12의 설명과 앞 검증 조건을 다시 말할 뿐, 별도 완료 기준을 만들지 않는다. *Fix:* 구조화 필드의 최소 목록 또는 검색/필터 가능한 화면/API 결과를 검증 조건으로 바꾼다.

## Strategic coherence — adequate
전략 축은 명확하다. §1은 1차 제품을 본사 관제탑으로 정의하고, 성공 기준을 "매일 아침 회의 전에 전체 지점 장부를 확인하고, 문제 가능성이 높은 지점을 먼저 파악"하는 것으로 둔다. FR-15~FR-17, FR-19~FR-21, FR-27~FR-29는 이 thesis를 직접 지원한다.

성공 지표도 단순 활동량보다 운영 성과에 가깝다. SM-1은 3초 조회와 미제출/지연/미마감 100% 표시를, SM-2와 SM-3은 마감과 정정 추적을, SM-C1~SM-C5는 빠른 입력, 조용한 화면, 알림량, 엑셀 일치, 민감 지표 숨김의 부작용을 counter-metric으로 잡는다.

하지만 §8이 들어오면서 문서의 전략 초점이 넓어졌다. 직원/급여 참고, FIFO, 상품 분석, 월 손익, 외부 알림, AI 대비 구조화까지 승인 추가 구현이 모두 들어오며, 본사 관제탑 MVP와 후속 확장 플랫폼의 경계가 독자가 계속 확인해야 하는 구조가 되었다. §0.2와 §8의 phase 표가 이를 완화하지만, 전략 우선순위가 CAP별로 "왜 이 순서인지"까지 충분히 설명되지는 않는다.

### Findings
- **medium** CAP phase 순서의 전략 근거가 약하다 (§8) — Extension A~D로 묶었지만, 왜 권한/통제 기반이 먼저이고 리포트/알림이 뒤인지, 어떤 위험이나 가치 가설 때문에 이 순서인지 설명이 짧다. *Fix:* §8의 CAP 구현 순서 표 앞에 phase별 목표와 다음 phase로 넘어가는 조건을 1~2문장씩 추가한다.
- **medium** 승인 추가 구현 성공 지표가 운영 결과보다 기능 존재에 더 가까운 항목이 있다 (§9) — SM-7~SM-10은 preview, 추적, 차단, 계약 보유처럼 검증 가능하지만, 실제 운영 개선 목표는 약하다. *Fix:* 업로드 재작업률, FIFO `확인 필요` 비율, 민감 지표 접근 차단 로그 0건, 리포트 수동 보정 건수 같은 운영 지표를 선택적으로 보강한다.

## Done-ness clarity — adequate
MVP FR 대부분은 완료 기준이 좋다. FR-3은 변경 전/후 값, 이벤트 유형, 검색/export까지 요구하고, FR-9는 휴무일, 전일 장부 누락, 늦은 마감, 마감 후 정정, 월 전환의 이월 상태 규칙을 표로 닫았다. FR-19~FR-21도 마감, replacement 정정, 무효화, 정정 반영값 표시를 검증 조건으로 둔다.

계산과 리포트 역시 이전보다 선명하다. §4.3은 원화 정수 저장, 소수점 입력 오류, 비율 표시 자리수, 0 나누기 `계산 불가`, VAT 제외 범위를 명시한다. §4.7은 FR-27~FR-29와 CAP-2/CAP-3/CAP-10의 필수 컬럼, 필터, export/차트 기준을 표로 제시한다.

그래도 일부 CAP와 UI 요구는 아직 story-ready하지 않다. CAP-18은 "데이터를 보기 좋은 크기"처럼 주관적이고, CAP-4의 "그래프 또는 차트"는 차트 종류나 최소 시각화 기준이 없다. CAP-14는 손실액 계산을 말하지만 OQ-9가 핵심 정책을 열어 두고 있어, 구현 전에 계산 기준을 닫아야 한다.

### Findings
- **medium** 일부 UI/CAP 검증 조건이 주관적이다 (§8.3, §8.4) — CAP-18의 "보기 좋은 크기", CAP-4의 "그래프 또는 차트"는 구현자가 완료 여부를 다르게 해석할 수 있다. *Fix:* 리사이징 대상 컬럼, 저장 여부, 최소/최대 폭, 차트 유형, 필수 축/범례/필터를 지정한다.
- **medium** CAP-14 손실액 계산은 핵심 정책이 열려 있어 완료 기준이 반쪽이다 (§8.4, §10) — CAP-14는 희망 판매가와 실제 처리 금액 차이를 기준으로 한다고 하지만, OQ-9는 미입력 또는 영업 중 변경 시 기준을 묻는다. *Fix:* CAP-14 앞에 `OQ-9 닫힘 전 구현 불가`를 명시하고, 닫힌 뒤 희망 판매가 버전 관리와 변경 적용 시점을 검증 조건에 넣는다.
- **low** FR-16의 표시 기준이 시각적으로 느슨하다 (§4.4) — "눈에 띄게 표시된다"는 접근성, 우선순위, 다중 이상 신호 표시 방식을 닫지 않는다. *Fix:* 색상 외 아이콘/텍스트 병행, 다중 신호 정렬, 필터링, 접근성 기준을 최소 검증 조건으로 추가한다.

## Scope honesty — strong
범위 정직성은 이 문서의 강점이다. §6은 MVP 제외 범위를 직접 말하고, §7.2는 "§6을 MVP 범위 방어선으로 사용"하라고 한다. §8은 제외 범위 중 일부가 승인 추가 구현으로 승격되었음을 숨기지 않고, "기존 §6 제외 범위가 삭제되는 것은 아니다"라고 설명한다.

Open Questions도 실제로 영향 범위와 결정 시점을 가진다. OQ-7, OQ-15, OQ-17은 FIFO와 업로드 흐름을 막는 질문으로 연결되고, OQ-18은 유지보수 대응 시간과 비용을 계약 전 필수로 분리한다. §11은 별도 `[ASSUMPTION]` 태그가 없고 남은 결정은 §10으로 분리했다고 밝힌다.

특히 CAP-19를 제품 기능 스토리로 자동 변환하지 않는 계약/운영 항목으로 둔 것은 좋은 범위 통제다. 유지보수 월 비용 같은 회의 입력값을 제품 요구로 끌어들이지 않고 계약 문서로 분리한다.

### Findings
- **medium** Open Questions 수가 많은 상태에서 `draft`임을 알리지만, 최종화 전 필수 질문의 우선순위 묶음은 더 필요하다 (§10) — OQ는 18개이며, `스토리 작성 전 필수`, `Epic 구현 전 필수`, `구현 중 확인 가능`, `계약 전 필수`가 섞여 있다. 분류는 좋지만 실제 다음 액션 순서가 한눈에 보이지 않는다. *Fix:* §10 위에 `최종 PRD 전 필수`, `MVP 스토리 전 필수`, `확장 Epic 전 필수`, `계약 전 필수`의 짧은 실행 순서 목록을 추가한다.

## Downstream usability — adequate
후속 작업자가 추출하기 좋은 장치가 많다. 용어 섹션은 장부 상태, 전일재고 후보, 확정 이월 기준, 업로드 매입, 매출차액 등을 정의하고, FR/SM/OQ ID는 대부분 고유하게 연결된다. UJ도 네 개 모두 김관리자, 현대 지점장, 이스태프 같은 named protagonist를 가진다.

리포트와 계산 계약은 아키텍처/스토리 작성에 특히 유용하다. §4.3의 계산 공통 규칙, §4.7의 리포트별 최소 데이터 계약, §8의 CAP 구현 순서와 Related FR/SM/OQ 표는 source-extract가 가능하다.

가장 큰 문제는 CAP 추적 누락이다. §8.4에는 **CAP-14: 희망 판매가 기준 손실액 계산**이 있지만, §8의 CAP 구현 순서와 추적 기준 표에는 CAP-14가 없다. OQ-9는 CAP-14를 영향 범위로 참조하므로, 표에서 빠진 것은 단순 정렬 문제가 아니라 downstream planning의 누락 위험이다.

### Findings
- **high** CAP-14가 CAP 구현 순서/추적 표에서 빠져 있다 (§8, §8.4, §10) — §8.4에는 CAP-14가 있고 OQ-9도 CAP-14를 참조하지만, §8 표는 CAP-13 다음 CAP-15로 넘어간다. 에픽/스토리 생성기가 표를 기준으로 추출하면 손실액 계산 CAP가 누락될 수 있다. *Fix:* CAP-14를 §8 표에 phase, 의존/선행, Related FR/SM/OQ와 함께 추가한다.
- **medium** CAP ID 순서가 구현 순서와 숫자 순서를 동시에 표현해 읽는 비용이 있다 (§8) — 표가 구현 순서라는 점은 명시되어 있지만 CAP-13, 15, 16, 17, 18 다음 CAP-5로 이동해 처음 읽는 사람은 누락과 의도적 정렬을 구분하기 어렵다. *Fix:* 표 제목을 `구현 권장 순서`로 더 명확히 하고, 별도 `CAP ID 인덱스`를 숫자 순으로 추가한다.
- **low** decision log와 PRD 간 OQ 변경 이력이 완전히 직관적이지 않다 (.decision-log.md §2026-06-10, PRD §10) — decision log는 OQ-19/OQ-20을 제거했다고 설명하지만 현재 PRD에는 해당 ID가 없다. 결과는 맞지만, 후속 검토자가 왜 OQ 번호가 18에서 끝나는지 이해하려면 로그를 읽어야 한다. *Fix:* §10 또는 decision log에 제거된 OQ가 본문 규칙으로 흡수되었다는 짧은 주석을 남긴다.

## Shape fit — strong
제품 형태에 맞는 PRD다. ERP Fish는 내부 운영 도구이자 brownfield 전환 프로젝트이므로, 장식적인 시장/페르소나보다 업무 흐름, 권한, 감사, 계산, 리포트 계약이 중요하다. 이 PRD는 바로 그 방향으로 구성되어 있다.

UJ는 과하지 않다. 네 개 여정은 관제, 지점 입력, 본사 보완/마감, 마감 후 정정이라는 핵심 업무를 보여주고, 나머지는 FR/CAP 중심 capability spec으로 처리한다. 이는 단일 운영 도구에 불필요한 UX 서사를 많이 붙이지 않는 적절한 형태다.

addendum도 잘 쓰였다. 엑셀 시트 관찰, 정책 확인 항목, 회의 변경 요구를 본문에 모두 밀어 넣지 않고 별도 추출 메모로 보존한다. 본문은 확정 방향과 요구사항, addendum은 근거와 세부 관찰로 분리되어 있다.

### Findings
- **low** brownfield 기준의 현재 시스템/엑셀 대체 범위가 FR별로 완전히 붙어 있지는 않다 (§4, addendum) — addendum은 기존 엑셀 시트와 후보 요구를 잘 보존하지만, 모든 FR이 어떤 기존 엑셀 흐름을 대체하는지까지 연결되지는 않는다. *Fix:* 구현 착수 전 필요한 경우 FR 묶음별로 `대체하는 기존 엑셀 흐름` 한 줄을 추가한다. 단, 현재 PRD의 형태를 망칠 만큼 큰 매트릭스는 필요 없다.
