# PRD: ERP Fish

## 0. 문서 목적

이 PRD는 ERP Fish의 **MVP 기준선과 2026-06-10 승인 추가 구현 범위**를 함께 정의한다. 대상 독자는 PM, 개발자, UX/화면 설계 담당자, 아키텍처 담당자, 에픽/스토리 작성자다.

제품은 원드라이브 엑셀로 운영하던 지점별 일일 장부를 웹 기반 내부 ERP로 전환한다. 이 문서는 엑셀 화면을 그대로 복제하지 않고, 실제 업무 목적과 데이터 흐름을 기준으로 요구사항을 정리한다.

입력 자료 추출 메모는 `addendum.md`에 보존한다. 본문은 확정된 제품 방향, MVP 요구사항, 2026-06-10에 승인된 추가 구현 범위를 구분해 다룬다. `docs/meeting/change.md`에서 보강된 회의 요구사항은 기존 MVP 기준선과 충돌하지 않는 범위에서 §8 추가 구현 범위와 관련 검증 조건에 반영한다.

### 0.2 릴리스 기준선

§6은 **MVP 기준선의 제외 범위**이고, §8은 고객 회의와 추가 견적 기준으로 **MVP 이후 승인된 추가 구현 범위**다. 두 섹션은 서로를 삭제하지 않는다. 구현 계획, 견적, 에픽/스토리 작성은 아래 기준선을 먼저 따른다.

| 구분 | 포함 항목 | 구현 판단 기준 |
| --- | --- | --- |
| MVP 필수 | FR-1~FR-29. 본사/지점장 로그인, 일일 장부 입력, 기본 계산/검증, 본사 관제판, 마감/정정, 마스터 관리, 기본 리포트, 지점장 민감 지표 서버 노출 차단 최소 기준. | MVP 오픈 전 완료되어야 한다. 단, §10의 `스토리 작성 전 필수` 질문이 연결된 기능은 해당 질문을 먼저 닫는다. |
| 승인 추가 구현 | CAP-1~CAP-18. 직원/급여 참고, 이카운트 업로드, FIFO 원가, 통합 재고, 상품 분석, 민감 지표 노출 정책 고도화, 일괄 마감, 특수기간/고도화 리포트, 외부 알림, AI 대비 데이터 구조화. | 각 CAP의 배포 판단은 §8의 `릴리스 버킷`을 따른다. 버킷 변경은 PM 승인 change proposal 또는 동등한 결정 기록으로 남긴다. |
| 계약/운영 별도 | CAP-19 유지보수 및 서버 인프라 운영 대행, 서버 이용료 포함 여부, 대응 시간, 월 비용. | 제품 기능 요구사항이 아니라 계약/운영 문서에서 확정한다. 제품 PRD에는 로그, 설정, 점검 화면처럼 시스템이 제공해야 할 동작만 남긴다. |
| 후순위/명시 제외 | POS/카드 매출 자동 연동, AI 이미지 식별, 과거 엑셀 데이터 일괄 이관, 실제 급여 지급 확정, AI 화면/API/프롬프트/분석 결과 저장. | 별도 결정 전까지 MVP와 승인 추가 구현 범위에 포함하지 않는다. 필요 시 새 PRD 또는 change proposal로 승격한다. |

**MVP FR 스토리 작성 준비 상태:**

MVP 필수 FR이라도 일부는 §10의 Open Question이 닫혀야 구현 스토리로 확정할 수 있다. 에픽/스토리 작성자는 아래 분류를 따른다. `스토리 작성 가능`은 바로 구현 스토리로 분해해도 되는 범위이고, `OQ 차단`은 연결된 OQ가 닫히기 전에는 구현 스토리 대신 정책 확정용 discovery story(질문 종결 + 정책 메모)로만 다룬다. 하나의 FR 안에서도 계산·노출·리포트 slice가 나뉘면 준비 상태를 분리한다.

스토리 생성 작업은 별도 산출물 `mvp-story-extraction-checklist.md`를 입력 게이트로 사용한다. 아래 표는 PRD 안의 요약 기준이고, 실제 에픽/스토리 생성 여부는 체크리스트 파일의 slice별 `may generate implementation story` 값과 승인 상태를 따른다. OQ가 닫히지 않은 slice는 구현 스토리로 추출하지 않는다.

| 상태 | FR 또는 기능 slice | 차단 OQ | 처리 |
| --- | --- | --- | --- |
| 스토리 작성 가능 | FR-1~FR-5, FR-7, FR-8(MVP 수동 매입), FR-10~FR-12(희망 판매가 손실 제외), FR-15, FR-18~FR-23, FR-25~FR-27 | — | 바로 구현 스토리로 분해한다. |
| 스토리 작성 가능 | FR-13 중 총매출, 비용합계, 근무인원, 평균매출처럼 가격·원가·민감 노출 정책에 의존하지 않는 기본 합계/상태 계산 | — | 민감 지표 노출 차단 최소 기준(§4.1)을 함께 적용한다. |
| 스토리 작성 가능 | FR-28/FR-29 중 본사 권한 전용 조회와 export | — | 지점장, 외부 공유 링크, 캐시 응답에는 민감 지표가 노출되지 않는다는 §4.1 최소 기준을 먼저 적용한다. |
| OQ 차단 | FR-6, FR-14, FR-16, FR-17 | OQ-1 (매출차액 이상 신호 기준) | OQ-1이 닫히기 전에는 매출차액 임계값을 확정 스토리에 넣지 않고, 차이값 표시까지만 구현한다. 임계값/이상 신호 판정은 discovery story로 분리한다. |
| OQ 차단 | FR-9, FR-13의 `30%단가` 파생 표시 또는 계산 | OQ-2 (`30%단가` 의미) | 재고 입력·기본 계산 자체는 구현 가능하나 `30%단가` 관련 화면/계산은 discovery story로 분리한다. |
| OQ 차단 | FR-24의 초기 품목 마스터 적재/정규화 | OQ-3 (품목 마스터 분리 기준) | 품목 선택·입력 기능은 구현 가능하나 초기 마스터 적재/정규화는 discovery story로 분리한다. |
| OQ 차단 | FR-13의 FIFO 기준 매출원가, 재고금액, 매출이익, 이익률, 재고비율 | OQ-7, OQ-17 | FIFO 정책이 닫히기 전에는 MVP 기본 계산값 또는 `기준 확인 필요`로만 표시하고 FIFO 확정 계산으로 스토리화하지 않는다. |
| OQ 차단 | FR-11/FR-13의 희망 판매가 기준 손실액 | OQ-9 | 희망 판매가 기준 손실 계산은 CAP-14 discovery/정책 확정 후 구현한다. MVP 손실 기록은 기존 손실 유형·수량·금액 기록으로 제한한다. |
| OQ 차단 | FR-13/FR-28/FR-29의 지점장 노출, 공유 링크, export, 캐시 응답에 포함될 민감 파생 지표 | OQ-10A | MVP 최소 노출 차단 기준이 승인되기 전에는 지점장 경로에서 숨김을 기본값으로 삼고, 본사 전용 화면/API에만 노출한다. |
| OQ 차단 | FR-9/FR-13의 `차이` → `당일 판매량` 의미 변경 | OQ-14 | 단순 라벨 변경은 가능하나 계산 의미 변경은 재고/리포트 계산 정책 확정 후 구현한다. |

체크리스트 승인 기준:
- `implementation story`: 연결된 OQ가 없거나, 본문 요구사항과 증빙 산출물만으로 구현자가 새 정책을 해석하지 않아도 된다.
- `discovery story`: 질문 종결, 정책 메모, 승인 산출물 생성이 목표이며 제품 동작 구현을 포함하지 않는다.
- `blocked`: owner와 재확인 시점이 정해졌지만 아직 구현 또는 discovery story로 넘길 근거가 없다.

## 1. 비전

ERP Fish의 1차 제품은 본사가 모든 지점의 일일 장부를 확인하고, 필요한 데이터를 직접 입력하거나 수정하고, 최종 마감할 수 있는 관제탑이다.

지점장은 로그인 후 자기 지점의 일일 장부를 입력한다. 본사는 전체 지점의 장부 상태, 매출/이익률 급락, 매출차액, 재고/손실 이상을 매일 아침 한 화면에서 확인한다. 이 데이터는 매일 아침 지점장 회의의 기준 자료로 사용된다.

1차 성공 기준은 “본사가 매일 아침 회의 전에 전체 지점 장부를 확인하고, 문제 가능성이 높은 지점을 먼저 파악하며, 필요한 장부를 마감할 수 있는 상태”다.

## 2. 대상 사용자

### 2.1 Jobs To Be Done

- 본사 사용자는 매일 아침 회의 전에 모든 지점의 장부 상태와 이상 신호를 빠르게 확인하고 싶다.
- 본사 사용자는 지점 데이터가 누락되거나 잘못된 경우 직접 입력하거나 수정한 뒤 마감하고 싶다.
- 본사 사용자는 마감 후 오류를 발견해도 원본 장부를 보존하면서 정정 기록으로 회의/리포트 숫자를 바로잡고 싶다.
- 지점장은 자기 지점의 매출, 결제, 비용, 매입, 재고, 손실, 근무 정보를 하루 단위로 입력하고 싶다.
- 지점장은 엑셀 수식이나 서식을 깨뜨릴 걱정 없이 장부를 제출하고 싶다.

### 2.2 1차 사용자

- **본사 사용자**: 전체 지점 조회, 직접 입력, 수정, 마감, 정정 기록, 마스터 관리, 이상 신호 기준값 설정 권한을 가진다.
- **지점장**: 자기 지점의 일일 장부를 입력하고, 본사 마감 전까지 수정할 수 있다.

### 2.3 1차 비사용자

- 일반 직원은 1차에서 로그인 사용자로 보지 않는다. 직원 정보는 근무인원/특이사항 기록에 필요한 데이터로만 다룬다.
- 외부 회계 시스템, POS, 카드사 시스템은 1차에서 직접 사용자나 필수 연동 대상으로 보지 않는다.

### 2.4 핵심 사용자 여정

- **UJ-1. 본사 김관리자가 아침 회의 전 문제 지점을 고른다.**
  - **상황:** 김관리자는 매일 아침 지점장 회의 전에 전 지점 장부를 확인해야 한다.
  - **진입:** 본사 계정으로 로그인 후 관제판을 연다.
  - **흐름:** 오늘/어제 기준 전체 지점 현황을 보고, 매출/이익률 급락, 매출차액, 재고/손실 이상이 있는 지점을 먼저 연다.
  - **가치 순간:** 문제 가능성이 높은 지점과 사유가 한눈에 보인다.
  - **결과:** 김관리자는 회의에서 어떤 지점을 먼저 이야기할지 결정한다.

- **UJ-2. 현대 지점장이 하루 장부를 입력한다.**
  - **상황:** 현대 지점장은 퇴근 전 당일 장부를 입력해야 한다.
  - **진입:** 지점장 계정으로 로그인 후 자기 지점의 오늘 장부를 연다.
  - **흐름:** 매출/결제, 비용, 매입, 재고, 손실/폐기/떨이, 근무인원/특이사항을 단계형 화면에서 입력한다.
  - **가치 순간:** 마지막 검토 화면에서 누락 항목과 계산 요약을 확인한다.
  - **결과:** 장부가 본사 검토 대상으로 올라간다.

- **UJ-3. 본사 이스태프가 누락 매입을 보완하고 마감한다.**
  - **상황:** 특정 지점 장부에 매입 정보가 빠졌다.
  - **진입:** 본사 계정으로 해당 지점 장부 상세를 연다.
  - **흐름:** 변경 이력을 확인하고, 매입 정보를 직접 입력한 뒤 계산값과 이상 신호를 다시 확인한다.
  - **가치 순간:** 본사 입력/수정 내역이 지점장 입력과 구분되어 남는다.
  - **결과:** 본사가 장부를 마감하면 원본 장부가 잠긴다.

- **UJ-4. 본사 김관리자가 마감 후 오류를 정정 기록으로 반영한다.**
  - **상황:** 마감 후 회의 준비 중 매출차액 오류가 발견됐다.
  - **진입:** 본사 계정으로 마감된 장부 상세를 연다.
  - **흐름:** 원본 장부는 수정하지 않고 정정 기록을 추가한다.
  - **가치 순간:** 리포트와 관제판 숫자는 정정 반영값으로 바뀌고, 원본값과 정정 내역은 추적 가능하다.
  - **결과:** 회의 자료는 최신 운영 숫자를 보여주면서도 원본 감사 추적을 잃지 않는다.

## 3. 용어

- **본사 사용자** — 전체 지점 데이터를 조회, 입력, 수정, 마감, 정정할 수 있는 사용자.
- **지점장** — 자기 지점의 일일 장부를 입력하고 본사 마감 전까지 수정할 수 있는 사용자.
- **지점** — 장부가 작성되는 매장 단위.
- **일일 장부** — 특정 지점과 특정 일자에 대한 매출, 결제, 비용, 매입, 재고, 손실, 근무 기록.
- **장부 상태** — 일일 장부의 진행 상태. 1차 상태는 미입력, 입력 중, 검토 대기, 본사 마감이다.
- **영업 상태** — 특정 지점+일자의 영업 여부. 1차 상태는 영업일, 휴무일이다.
- **본사 마감** — 본사 사용자가 일일 장부를 확정해 원본 수정을 잠그는 행위.
- **정정 기록** — 본사 마감 후 원본 장부를 바꾸지 않고 추가하는 수정 기록.
- **정정 반영값** — 원본 장부 값에 정정 기록을 반영한 운영 기준 숫자.
- **관제판** — 본사가 전체 지점의 장부 상태와 이상 신호를 한 화면에서 보는 화면.
- **이상 신호** — 본사가 설정한 기준값에 따라 표시되는 주의 표시. 1차 이상 신호는 매출/이익률 급락, 매출차액, 재고/손실 이상이다.
- **품목 마스터** — 본사가 관리하는 품목명, 구분, 규격, 기본 단가 정보.
- **매입 기준** — 본사가 관리하는 매입 관련 기준 정보.
- **재고 흐름** — 전일재고, 매입, 판매/차감, 손실, 당일재고로 이어지는 품목 단위 흐름.
- **재고 조정** — 시스템 계산 재고와 실제 재고가 다를 때 조정 전/후 값과 차이 사유를 기록하는 행위.
- **월초 재고 스냅샷** — 월 첫 장부의 전일재고를 만들기 위한 기준 재고. 기존 엑셀의 `이월` 시트 역할을 대체한다.
- **전일재고 후보** — 직전 저장 장부의 당일재고를 다음 영업일 전일재고로 미리 가져온 값. 직전 장부가 본사 마감 전이면 확정값이 아니라 검토 필요 상태로 다룬다.
- **확정 이월 기준** — 본사 마감된 장부의 당일재고를 다음 영업일 전일재고의 확정 기준으로 사용하는 정책.
- **업로드 매입** — 본사가 이카운트 엑셀 등 외부 파일을 업로드해 장부에 생성하는 매입 라인.
- **출고 단가** — 본사 품목 마스터의 기준 가격과 별개로, 실제 매입/출고 상황에 맞춰 업로드 검수 과정에서 수동 조정할 수 있는 단가.
- **당일 판매량** — 기존 재고 테이블의 `차이` 항목을 현장 사용자가 이해하기 쉽게 바꾼 명칭. 단순 라벨 변경인지 계산 의미 변경인지는 §10 Open Questions에서 확정한다.
- **장기 체화 재고** — 일정 기간 이상 판매 또는 출고되지 않고 지점 재고로 남아 운영 위험 알림 대상이 되는 재고.
- **전일 이월금액** — 총매출에 포함하지 않고 별도 조정/이월 항목으로 저장하는 금액.
- **매출차액** — 총매출과 상품별 판매금액 합계에서 손실/조정 반영액을 차감한 값 사이의 차이. 전일 이월금액은 총매출에 포함하지 않는다.
- **아침 회의 리포트** — 본사가 매일 아침 지점장 회의에서 사용하는 일별 요약 리포트.

**운영자 표시용 기술 용어:**

| 기술 용어 | 운영자에게 보일 말 |
| --- | --- |
| API response | 서버 응답 |
| mutation | 저장/변경 요청 |
| preview | 업로드 미리검토 |
| commit | 업로드 확정 반영 |
| rollback | 자동 되돌림 |
| idempotency | 같은 행 중복 생성 방지 |
| export | 내보내기 |
| cache | 임시 저장 응답 |

### 3.1 장부 상태 모델

| 상태 | 의미 | 수정 가능 여부 |
| --- | --- | --- |
| 미입력 | 해당 지점+일자 장부가 아직 작성되지 않은 상태 | 본사와 해당 지점장 입력 가능 |
| 입력 중 | 일부 값이 저장되었지만 검토 대기 전 상태 | 본사와 해당 지점장 수정 가능 |
| 검토 대기 | 지점장 또는 본사가 입력을 마치고 본사 검토 대상으로 올린 상태 | 본사 마감 전까지 본사와 해당 지점장 수정 가능 |
| 본사 마감 | 본사가 최종 마감한 상태 | 원본 수정 불가, 본사 정정 기록만 추가 가능 |

지점장 제출 또는 검토 대기 전환은 원본 잠금이 아니다. 1차 정책에서 원본 장부가 잠기는 시점은 본사 마감이다.

### 3.2 동시 편집과 저장 충돌

본사와 지점장이 같은 지점+일자 장부를 본사 마감 전 함께 다룰 수 있으므로, 시스템은 마지막 저장값으로 조용히 덮어쓰지 않는다.

| 항목 | 기준 |
| --- | --- |
| 편집 토큰 | 장부에는 서버 버전 또는 edit token이 있어야 하며, 부분 저장과 최종 제출은 마지막으로 읽은 토큰을 함께 보낸다. |
| 충돌 단위 | 기본 충돌 단위는 입력 단계/섹션이며, 같은 섹션 안의 같은 필드가 서버에서 먼저 바뀐 경우 충돌로 본다. 서로 다른 섹션의 변경은 감사 로그를 남기고 병합할 수 있다. |
| 비교/해결 | 충돌이 발생하면 사용자에게 내 입력값, 서버 최신값, 마지막 수정자, 수정 시각을 보여주고 다시 저장할 값을 선택하게 한다. |
| 본사 강제 수정 | 본사 강제 수정은 해당 장부 또는 섹션에 `본사 수정 중` 상태를 표시할 수 있으며, 지점장 저장은 최신값 재확인 후에만 허용한다. |
| 오래된 세션 | 모바일 또는 브라우저 세션이 오래된 토큰으로 저장하려 하면 서버는 저장을 거부하고 최신 장부를 다시 불러오게 한다. |
| 부분 저장 | 부분 저장은 입력 중 손실을 막기 위한 장치이며, 다른 사용자의 확정 저장을 덮어쓸 권한이 아니다. |

## 4. 기능 요구사항

### 4.1 인증, 권한, 변경 이력

**설명:** 본사 사용자와 지점장은 로그인 역할에 따라 접근 범위와 입력 기능이 달라진다. 공동 입력 구조이므로 입력자, 수정자, 수정 시각, 변경 이력은 제품 신뢰의 핵심이다.

#### 권한 프로파일 기준

`본사 사용자`는 단일 권한이 아니라 여러 운영 프로파일의 상위 표현이다. 구현 시 화면 표시, API 응답, mutation 권한, 데이터 범위를 아래 기준으로 나누고, 한 사용자가 여러 프로파일을 가질 수 있게 한다.

| 권한 프로파일 | 주요 권한 | 제한/주의 |
| --- | --- | --- |
| 대표/소유자 | 전체 조회, 민감 지표 조회, 권한 위임, 계약/운영 설정 확인 | 최소 인원에게만 부여한다. |
| 본사 관리자 | 전체 지점 조회, 직접 입력/수정, 마감, 정정, 리포트 조회 | 권한/계정 변경은 별도 설정 관리자 권한이 필요하다. |
| 본사 스텝 | 전체 지점 조회, 누락 데이터 보완, 리포트 확인 | 마감, 권한 변경, 민감 설정 변경은 기본 차단한다. |
| 업로드 담당자 | 이카운트 엑셀 업로드, 업로드 검수, 업로드 취소/재처리 요청 | 업로드 반영은 preview/commit 절차와 감사 로그를 통과해야 한다. |
| 마감 담당자 | 개별 마감, 일괄 마감 dry run, 마감 실행 | 일괄 마감은 CAP-15의 preflight와 사유 입력을 통과해야 한다. |
| 설정 관리자 | 지점, 사용자, 권한, 코드, 이상 신호 기준값 관리 | 설정 변경은 변경 전/후 값과 사유가 감사 로그에 남아야 한다. |
| 조회 전용 본사 | 전체 또는 지정 지점 조회, 리포트 조회 | 입력/수정/마감/정정/API mutation은 차단한다. |
| 지점장 | 연결된 지점의 장부 입력/조회 | 타 지점, 민감 지표, 본사 전용 리포트, 본사 설정 mutation은 차단한다. |

모든 권한 제한은 UI 숨김만으로 처리하지 않는다. 서버 API 응답과 mutation 권한에서 같은 규칙이 강제되어야 한다.

#### 권한 프로파일 x 주요 action 기준

`본사 사용자`라는 말은 요구사항 설명을 위한 상위 표현이며, 구현 권한은 아래 action 단위로 판단한다.

| 작업 | 허용 프로파일 | 제한/감사 기준 |
| --- | --- | --- |
| 장부 생성/일반 수정 | 본사 관리자, 허용된 본사 스텝, 해당 지점장 | 지점장은 자기 지점과 본사 마감 전 장부만 가능하다. |
| 본사 강제 수정 | 본사 관리자 또는 직접 수정 mutation 권한을 받은 프로파일 | 사유, 대상 필드, 변경 전/후 값, 실행자를 감사 로그에 남긴다. |
| 개별 본사 마감 | 본사 관리자, 마감 담당자 | §4.5 마감 가능성 표를 통과해야 한다. |
| 마감 후 정정 추가 | 본사 관리자, 정정 권한을 받은 마감 담당자 | 원본 수정이 아니라 정정 기록으로만 처리한다. |
| 정정 영향 확정 | 본사 관리자, 마감 담당자 | 영향 범위와 확정 시각을 감사 로그에 남긴다. |
| 업로드 preview/commit/void/reprocess | 업로드 담당자, 본사 관리자 | commit/void/reprocess는 배치 id와 사유를 남긴다. |
| 지점/사용자/권한/코드 관리 | 설정 관리자 | 변경 전/후 값, 변경자, 시각, 사유가 필요하다. |
| 리포트 조회/export | 대표/소유자, 본사 관리자, 조회 전용 본사(허용 범위) | 민감 지표는 권한 프로파일별로 서버 응답과 export에서 차단한다. |

#### MVP 민감 데이터 노출 최소 기준

CAP-13은 민감 지표 통제의 고도화 요구사항이지만, 지점장에게 원가·이익 구조가 새는 것을 막는 최소 보안 기준은 MVP 필수다.

- 지점장 권한의 화면, 서버 응답, export, 공유 링크, 캐시 데이터에는 매출원가, 매출이익, 영업이익, 인당생산성, 재고금액, 원가 근거처럼 본사 전용 또는 원가/이익을 역추정할 수 있는 값이 기본 노출되지 않는다.
- OQ-10A가 닫히기 전까지 애매한 파생 지표는 지점장 경로에서 숨김을 기본값으로 본다.
- 본사 전용 리포트 URL이나 export 파일은 지점장 세션에서 열 수 없어야 하며, 저장된 응답을 재사용하더라도 권한 재검사를 통과해야 한다.
- 민감 지표 차단은 UI 표시만이 아니라 서버 응답 필드, export 컬럼, 알림 템플릿, 캐시 키 설계에서 함께 적용한다.

**민감 필드 노출 기본 매트릭스:**

상태: `draft baseline`. OQ-10A 승인 전까지는 아래 매트릭스를 지점장 경로의 최소 차단 기본값으로 사용하되, 승인본으로 오해하지 않는다.

| surface / 권한 | 기본 허용 필드 | 기본 차단 필드 | 예외 조건 |
| --- | --- | --- | --- |
| 지점장 화면/API | 자기 지점의 입력값, 장부 상태, 제출/마감 여부, 현장 전달사항, 비민감 검증 메시지 | 매출원가, 매출이익, 영업이익, 이익률, 인당생산성, 재고금액, 원가 lot 근거, 본사 고정비, 타 지점 비교 | OQ-10A의 MVP 최소 노출 승인 전에는 예외 없음 |
| 본사 조회 전용 | 허용된 지점/기간의 장부 상태, 리포트 요약, 본사 회의용 지표 | 권한 범위 밖 지점, 권한 없는 민감 컬럼, 설정 mutation | 대표/소유자 또는 본사 관리자 승인으로 권한 범위를 부여한 경우 |
| 본사 관리자/대표 | 전체 지점 장부, 민감 지표, 정정 반영값, 리포트/export | 권한 관리, 계약/운영 설정처럼 별도 프로파일이 필요한 작업 | 설정 관리자 또는 대표/소유자 프로파일이 함께 있을 때 |
| export 파일 | 화면 리포트의 허용 컬럼과 원본/정정 반영 구분 | 지점장 권한의 민감 지표, 권한 밖 지점, 숨김 처리된 가격/원가 근거 | export 생성 시점의 권한으로 서버에서 재검사 |
| 알림 템플릿 | 수신자 권한에 맞는 상태, 제출 지연, 이상 신호 요약 | 원가, 이익, 마진율, 재고금액, 본사 고정비, 가격 신뢰 상세 | 본사 전용 알림 채널과 권한 프로파일이 명시된 템플릿 |

#### FR-1: 역할 기반 로그인

사용자는 본사 사용자 또는 지점장 역할로 로그인할 수 있다.

**검증 조건:**
- 본사 사용자는 전체 지점 메뉴에 접근할 수 있다.
- 지점장은 자기 지점 장부 메뉴에 접근할 수 있다.
- 로그인하지 않은 사용자는 장부 데이터에 접근할 수 없다.
- 시스템 최초 기동 또는 초기 세팅 시 본사 마스터 계정과 지점장 마스터 계정이 준비되어야 한다.
- 초기 마스터 계정은 운영 전 비밀번호 변경, 담당자 지정, 권한 점검을 거쳐야 한다.
- 비밀번호 변경/초기화, 계정 비활성화, 세션 만료, 로그인 실패 제한 정책은 제품에서 최소 동작을 제공하고 운영 설정 또는 운영 매뉴얼에 정의되어야 한다.
- 비밀번호는 평문으로 저장하지 않으며, 마스터 계정의 초기 비밀번호는 운영 전 반드시 교체되어야 한다.
- 초기 비밀번호 또는 관리자 초기화 비밀번호로 로그인한 계정은 첫 사용 전에 비밀번호를 바꿔야 한다.
- 로그인 실패는 일정 횟수 이상 반복되면 지연, 잠금, 또는 관리자 확인 같은 제한이 적용되어야 한다.
- 세션 만료 시간이 지나면 저장/변경 요청이 차단되고 다시 로그인해야 한다.
- 계정 비활성화는 신규 로그인뿐 아니라 기존 활성 세션의 저장/변경 요청도 차단해야 한다.
- 관리자 비밀번호 초기화와 계정 잠금 해제는 감사 로그에 남아야 한다.

**계정 보안 기본값:**

| 항목 | MVP 기본값 | 변경 기준 |
| --- | --- | --- |
| 로그인 실패 제한 | 같은 계정 기준 연속 5회 실패 시 15분 잠금 또는 설정 관리자 해제 필요 | 더 강한 값은 운영 설정으로 조정 가능, 더 약한 값은 PM + 개발 리드 승인 필요 |
| 유휴 세션 만료 | 2시간 동안 화면/API 활동이 없으면 저장/변경 요청 차단 후 재로그인 | 공동 PC 사용 지점은 더 짧게 설정 가능 |
| 장기 세션 만료 | 로그인 후 최대 12시간이 지나면 재로그인 | 운영 편의로 연장하려면 대표/소유자 승인 필요 |
| 비밀번호 초기화 토큰 | 발급 후 30분 만료, 1회 사용 후 폐기 | 관리자 초기화는 감사 로그 필수 |
| 초기 비밀번호 | 첫 로그인 전 변경 필수, 운영 오픈 전 미변경 계정은 저장/변경 요청 차단 | 임시 오픈 시에도 본사 승인 예외 기록 필요 |

#### FR-2: 지점 접근 제한

지점장은 자기 지점의 일일 장부만 조회하고 입력할 수 있다.

**검증 조건:**
- 지점장은 다른 지점 장부 목록과 상세에 접근할 수 없다.
- 본사 사용자는 모든 지점 장부 목록과 상세에 접근할 수 있다.
- 지점 접근 제한은 목록 화면, 상세 화면, 리포트, export, API 응답, mutation 요청에서 동일하게 적용된다.

#### FR-3: 입력/수정 이력 기록

시스템은 주요 입력과 수정에 대해 입력자, 수정자, 수정 시각, 변경 전 값, 변경 후 값을 기록한다.

**검증 조건:**
- 지점장 입력과 본사 입력/수정 내역이 구분되어 표시된다.
- 장부 상세에서 변경 이력을 확인할 수 있다.
- 본사 마감 후 정정 기록은 일반 수정 이력과 구분된다.
- 변경 이력에는 수정 주체의 이름 또는 아이디, 수정 일시, 기존 값, 변경된 값이 남는다.
- 변경 이력은 내부 코드만 보여주는 방식이 아니라 운영자가 이해할 수 있는 텍스트로 누적 표시되어야 한다.
- 변경 이력은 생성, 수정, 마감, 정정, 업로드, 기준값 변경, 권한 변경, 일괄 마감 이벤트를 구분해야 한다.
- 감사 로그는 이벤트 발생 후 일반 사용자 화면에서 수정/삭제할 수 없어야 한다. 잘못된 로그가 있으면 별도 정정 또는 무효화 이벤트로 남긴다.
- 감사 로그는 지점, 일자, 사용자, 이벤트 유형, 대상 필드 기준으로 검색하거나 export할 수 있어야 한다.
- 자유 입력한 작성자 이름은 감사 로그의 인증 계정 또는 실행자를 대체하지 않는다. 인증 계정, 화면 표시용 작성자 이름, 실제 입력자가 다르면 장부 상세와 감사 로그에서 구분되어야 한다.

#### 감사 이벤트 공통 계약

| 이벤트 유형 | 필수 필드 | 추가 규칙 |
| --- | --- | --- |

| 본사 마감 | event id, 장부 id, 실행자, 실행 시각, 마감 전 상태, 검증 결과 요약 | 마감 사유 또는 확인 메모를 남길 수 있어야 한다. |
| 정정 기록 | event id, 장부 id, 대상 필드, 원본값, 직전 정정 반영값, 새 정정값, 사유, 실행자, 시각 | 삭제 대신 추가 정정 또는 무효화 이벤트로 처리한다. |

### 4.2 일일 장부 입력

**설명:** 일일 장부는 지점+일자 단위로 작성된다. 입력 화면은 엑셀식 단일 장표 복제가 아니라 단계형 입력 흐름을 기본으로 한다.

#### FR-4: 지점+일자 장부 생성

본사 사용자와 지점장은 지점+일자 기준으로 일일 장부를 열 수 있다.

**검증 조건:**
- 같은 지점+일자에는 하나의 일일 장부가 존재한다.
- 장부가 없으면 입력 시작 시 생성된다.
- 이미 생성된 장부를 다시 열면 기존 입력값이 유지된다.

#### FR-5: 단계형 입력 흐름

일일 장부 입력은 매출/결제, 비용, 매입, 재고, 손실/폐기/떨이, 근무인원/특이사항, 검토/제출 순서의 7단계 흐름으로 제공된다.

**검증 조건:**
- 입력자는 각 단계를 이동할 수 있다.
- 검토/제출 화면에서 주요 누락 항목과 계산 요약을 확인할 수 있다.
- 저장 중인 장부는 입력 중 상태로 표시된다.
- 검토 대기 상태는 본사 마감 전 상태이며, 원본 잠금으로 처리하지 않는다.
- 지점 입력 화면은 PC 장애 시에도 사용할 수 있도록 모바일 세로형 화면에서 7단계 흐름이 끊기지 않아야 한다.
- 단계 이동 시 저장되지 않은 변경사항이 있으면 저장, 취소, 계속 편집 중 하나를 선택하게 한다.
- 모바일 입력 중 네트워크 오류가 나면 사용자가 잃을 수 있는 입력값과 재시도 방법을 명확히 보여준다.
- 각 단계의 완료 여부, 필수 누락, 검증 오류는 검토/제출 화면에서 한 번에 확인할 수 있어야 한다.

#### FR-6: 매출/결제 입력

입력자는 총매출, 현금, 카드, 기타 결제수단, 매출차액 관련 값을 입력할 수 있다.

**검증 조건:**
- 결제수단별 금액과 총매출이 장부에 저장된다.
- 시스템은 매출차액을 계산하거나 표시할 수 있다.
- 매출차액이 본사 기준값을 넘으면 이상 신호 후보로 표시된다.

#### FR-7: 비용 입력

입력자는 비용 항목, 금액, 메모를 입력할 수 있다.

**검증 조건:**
- 하나의 일일 장부에 여러 비용 항목을 등록할 수 있다.
- 비용 합계는 영업이익 계산에 반영된다.

#### FR-8: 매입 입력

입력자는 품목, 규격, 단가, 수량, 매입금액을 입력할 수 있다.

**검증 조건:**
- MVP에서는 지점장과 본사 사용자 모두 매입 정보를 수동 입력할 수 있다.
- CAP-6이 활성화되는 릴리스에서는 본사 업로드 또는 본사 작성 매입을 기본값으로 삼고, 지점장은 기본적으로 조회 전용이다.
- CAP-6 릴리스에서 지점장 비상 입력을 허용하려면 본사 허용 여부, 사유, 작성자, 작성 시각, 대상 장부가 감사 로그에 남아야 한다.
- 본사 사용자가 관리하는 매입 기준 또는 품목 마스터를 선택해 입력할 수 있다.
- 입력된 매입은 재고 흐름과 일일 요약 계산에 반영된다.

#### FR-9: 품목 단위 재고 입력

입력자는 품목, 규격, 단가, 전일재고, 매입, 판매/차감, 당일재고, 재고금액, 수량을 다룰 수 있다.

**검증 조건:**
- 재고는 단순 금액 요약이 아니라 품목+규격 단위로 저장된다.
- 냉동/생물 같은 품목 구분을 기록할 수 있다.
- 전일재고, 매입, 판매/차감, 손실, 당일재고 흐름을 일자별로 추적할 수 있다.
- 시스템 계산 재고와 실제 입력 재고의 차이를 확인할 수 있다.
- 월 첫 장부의 전일재고는 월초 재고 스냅샷에서 가져온다.
- 이후 영업일의 전일재고는 직전 저장 장부의 당일재고를 전일재고 후보로 자동 이월한다.
- 직전 장부가 본사 마감 전이면 확정 이월값이 아니라 `검토 필요` 상태로 표시한다.
- 본사 마감 이후에는 해당 마감 장부의 당일재고를 확정 이월 기준으로 사용한다.

**재고 이월 상태 규칙:**

| 상황 | 전일재고 후보 | 확정 기준 | 표시/처리 |
| --- | --- | --- | --- |
| 정상 영업일 | 직전 저장 장부의 최신 당일재고 | 직전 본사 마감 장부의 당일재고 | 후보와 확정값이 다르면 `검토 필요`로 표시한다. |
| 휴무일 다음 영업일 | 마지막 저장 영업일의 당일재고 | 마지막 본사 마감 영업일의 당일재고 | 휴무일은 재고 흐름을 생성하지 않으며 리포트에서 휴무로 구분한다. |
| 전일 장부 누락 | 가장 최근 저장 장부의 당일재고가 있으면 후보로 사용 | 가장 최근 본사 마감 장부의 당일재고 | 누락 기간을 `이월 공백`으로 표시하고 본사 확인 전 마감 위험으로 표시한다. |
| 전일 장부가 여러 번 저장됨 | 최신 저장 시점의 당일재고 | 본사 마감 시점의 당일재고 | 저장 시각과 입력자를 감사 이력에서 확인할 수 있어야 한다. |
| 늦은 마감 | 기존 후보를 유지하되 확정 기준이 바뀔 수 있음을 표시 | 마감 완료 후 확정 기준 갱신 | 영향을 받는 이후 장부는 `이월 재확인 필요`로 표시한다. |
| 마감 후 정정 | 원본 후보는 보존 | 정정 반영 운영값은 별도 계산값으로 표시 | 원본 이월과 정정 반영 이월을 구분하고, 본사 확인 후 리포트에 반영한다. |
| 월 전환 | 월 첫 장부는 월초 재고 스냅샷을 우선 사용 | 월초 스냅샷 또는 본사 확정 이월 정책 | 월초 스냅샷과 전월 마지막 마감 장부가 다르면 차이 사유가 필요하다. |

#### FR-10: 재고 조정 기록

본사 사용자와 지점장은 시스템 계산 재고와 실제 재고가 다를 때 재고 조정을 기록할 수 있다.

**검증 조건:**
- 재고 조정에는 품목, 규격, 조정 전 수량/금액, 조정 후 수량/금액, 차이 수량/금액, 차이 사유, 작성자, 작성 시각이 남는다.
- 차이 사유는 필수로 입력하거나 선택해야 한다.
- 재고 조정은 재고 흐름, 이상 신호, 리포트에 반영된다.
- 본사 마감 후 재고 조정은 원본 수정이 아니라 정정 기록으로만 추가된다.

#### FR-11: 손실/폐기/떨이 입력

입력자는 손실, 폐기, 떨이 항목을 품목, 수량, 금액, 처리 유형, 사유/특이사항과 함께 입력할 수 있다.

**검증 조건:**
- 손실 항목은 재고 흐름과 손실 리포트에 반영된다.
- 손실액 또는 손실 수량이 본사 기준값을 넘으면 이상 신호 후보로 표시된다.

#### FR-12: 근무인원/특이사항 입력

입력자는 근무인원과 근무 관련 특이사항을 기록할 수 있다.

**검증 조건:**
- 근무인원은 인당생산성 계산에 반영된다.
- MVP에서 저장한 근무인원 숫자는 직원별 근무 기록으로 자동 복원된다는 전제를 두지 않는다.
- 1차 범위는 직원 마스터, 근무자별 선택, 월간 근무일수 자동 집계, 지각/조퇴, 출퇴근/급여 수준의 상세 근태 관리를 포함하지 않는다.

### 4.3 계산과 검증

**설명:** 시스템은 장부 입력값을 기준으로 본사 회의에 필요한 핵심 지표를 계산하고, 이상 신호 후보를 만든다.

1차 계산 기준은 운영 확인을 위한 기본 계산값이다. FIFO 매입 lot, 잔여 수량, 재고 원가 근거가 도입되기 전에는 매출원가, 매출이익, 이익률, 영업이익, 재고금액을 FIFO 기준 확정 원가처럼 표시하지 않는다. 화면과 리포트는 기본 계산값, FIFO 기준 계산값, `기준 확인 필요` 상태를 구분해야 한다. `30%단가`의 업무 의미처럼 아직 확인이 필요한 항목은 §10 Open Questions에서 별도 관리한다.

| 지표 | 1차 기준 |
| --- | --- |
| 총매출 | 현금 + 카드 + 기타매출 + 당일 조정 매출. 전일 이월금액은 제외 |
| 매출원가 | 전일재고 + 매입 - 당일재고 |
| 매출이익 | 총매출 - 매출원가 |
| 이익률 | 매출이익 / 총매출 |
| 영업이익 | 매출이익 - 비용합계 |
| 인당생산성 | 총매출 / 근무인원 |
| 재고금액 | 품목별 수량 * 단가 |
| 평균재고 | 선택 기간의 일별 재고 스냅샷 평균 |
| 평균매출 | 선택 기간의 일별 총매출 평균 |
| 매출대비 재고비율 | 평균재고 / 평균매출 |
| 최고매출품목 | 상품별 판매금액이 가장 큰 품목 |
| 매출차액 | 총매출 - 상품별 판매금액 합계 - 손실/당일 조정 반영액 |

**MVP 단가/가격 기준:**

FIFO 원가가 도입되기 전에도 본사 회의에는 비용·이익 참고값이 필요하다. MVP의 단가 기준은 아래 우선순위를 따른다. 이 값은 운영 확인용 기본값이며, FIFO 기준 확정 원가로 부르지 않는다.

| 우선순위 | 단가 출처 | 적용 기준 |
| --- | --- | --- |
| 1 | 장부 라인에 사용자가 직접 입력한 실제 단가 | 해당 장부 라인의 운영 기준값으로 사용한다. |
| 2 | 업로드 commit 또는 본사 작성 매입 라인의 단가 | CAP-6이 적용된 범위에서 해당 지점+일자+품목 라인에 사용한다. |
| 3 | 적용일이 유효한 품목 마스터 기본 단가 | 실제 단가가 없을 때 참고 기본값으로 사용하되, 화면에 `마스터 단가 기준`으로 표시한다. |
| 4 | 없음 | 0으로 대체하지 않고 `기준 확인 필요`로 표시한다. |

모든 장부 라인 단가는 가격 신뢰 상태를 함께 가진다. 단가가 매출원가, 재고금액, 매출이익, 이익률, 영업이익에 영향을 주면 마감 요약과 본사 리포트에 이 상태가 표시되어야 한다.

| 가격 신뢰 상태 | 의미 | 필수 기록 | 리포트 표시 |
| --- | --- | --- | --- |
| `approved` | 본사 승인 단가, 확정 업로드 단가, 또는 적용일이 유효한 마스터 단가가 검증된 상태 | source, approved_by, effective_date, approval_event_id | 기본 표시 |
| `manual_override` | 장부 라인 직접 입력, 본사 강제 수정, 업로드 preview 편집처럼 사람이 단가를 바꾼 상태 | source, actor, manual_override_reason, changed_at | `수동 단가 포함` 표시와 건수 집계 |
| `pending_review` | 업로드 preview, 신규 마스터 적용 전 검토처럼 아직 최종 반영되지 않은 상태 | source, batch_id 또는 change_request_id, reviewer | 마감 전 확인 필요, 일괄 마감 차단 |
| `basis_missing` | 단가 기준이 없어 파생 계산을 확정할 수 없는 상태 | missing_reason, affected_fields, close_exception_reason(마감 시) | `기준 확인 필요` 표시와 월간 별도 집계 |

- 가격 기준이 없는 품목이 매출원가, 재고금액, 이익률, 영업이익에 영향을 주면 해당 파생 지표는 `기준 확인 필요`로 표시한다.
- 가격 기준이 없는 장부는 저장할 수 있지만, 개별 마감은 사유 입력이 필요하고 일괄 마감은 §4.5 마감 가능성 표를 따른다.
- 지점장 화면에서는 가격 기준이 있더라도 §4.1의 민감 데이터 노출 최소 기준에 따라 원가·이익 파생 지표를 숨긴다.
- `30%단가`는 OQ-2가 닫히기 전까지 단가 우선순위에 포함하지 않는다.

**계산 공통 규칙:**

| 항목 | 기준 |
| --- | --- |
| 계산 주체 | 관제판, 리포트, export는 서버 계산값을 기준으로 한다. 화면에서 임시 계산을 하더라도 저장/마감/리포트 기준은 서버 계산값이다. |
| 금액 단위 | 원화 금액은 정수 원 단위로 저장한다. 사용자가 소수점 금액을 입력하면 저장하지 않고 검증 오류로 표시한다. |
| 비율 표시 | 이익률, 하락률, 재고비율은 반올림 전 계산값을 보존한다. 화면 기본 표시는 소수점 1자리, export 기본 표시는 소수점 2자리로 한다. 0으로 나누는 경우 `계산 불가`로 표시한다. |
| 음수/반품/환불 | 음수 금액은 반품, 환불, 정정, 조정처럼 허용된 유형과 사유가 있을 때만 저장한다. 허용되지 않은 음수는 검증 오류로 표시한다. |
| 세금/VAT | MVP와 승인 추가 구현 범위의 금액은 사용자가 입력한 세금 포함 운영 금액으로 저장한다. VAT 자동 분리, 세금계산서 기준 분개, 부가세 신고용 리포트는 별도 범위다. |
| 정정 반영 | 기본 운영 숫자는 정정 반영값을 사용하되, 장부 상세와 감사 로그에서는 원본값, 직전 정정 반영값, 최신 정정 반영값을 구분한다. |
| FIFO 전환 | FIFO가 도입되기 전의 MVP 계산값과 FIFO 기준 계산값은 화면/리포트/export에서 같은 이름으로 섞지 않는다. |

**매출차액 정의와 부호/표시 규칙:**

매출차액은 단일 지표이며, `매출차액 = 총매출 - 상품별 판매금액 합계 - 손실/당일 조정 반영액` 한 식으로 계산한다. 구성 항목은 결제 차액(현금/카드/기타 결제 합계와 총매출의 차이), 상품 합계 차액(총매출과 상품별 판매금액 합계의 차이), 손실/조정 차액(손실·폐기·떨이·당일 조정 반영액)으로 나누어 상세에서 분해해 볼 수 있다. 부호와 표시 규칙은 다음과 같다.

| 항목 | 기준 |
| --- | --- |
| 부호 의미 | 양수(+)는 총매출이 상품·손실·조정 반영 합계보다 큰 경우(설명되지 않은 초과 매출), 음수(-)는 총매출이 합계보다 작은 경우(설명되지 않은 누락/과다 차감)다. |
| 영(0) 처리 | 매출차액 0은 입력값이 서로 일치함을 뜻하며, 미입력 장부의 빈 값과 구분해 표시한다. |
| 임계값 | 이상 신호 기준 임계값(금액 기준인지 비율 기준인지)은 OQ-1에서 확정하기 전까지 확정 임계값으로 표시하지 않고 차이값 자체만 보여준다. |
| 표시 | 화면은 정수 원 단위 + 부호, export는 정수 원 단위 + 부호로 표시한다. 정정 반영 시 원본 매출차액과 정정 반영 매출차액을 구분한다. |

매출차액 입력 예시(금액 단위: 원):

| # | 총매출 | 상품 판매금액 합계 | 손실/조정 반영액 | 매출차액 | 해석 |
| --- | --- | --- | --- | --- | --- |
| 예시 1 | 1,000,000 | 1,000,000 | 0 | 0 | 결제·상품·손실이 모두 일치한 정상 장부 |
| 예시 2 | 1,000,000 | 980,000 | 0 | +20,000 | 총매출이 상품 합계보다 2만 원 많음(설명되지 않은 초과) |
| 예시 3 | 1,000,000 | 1,030,000 | 0 | -30,000 | 총매출이 상품 합계보다 3만 원 적음(상품 입력 과다 또는 매출 누락) |
| 예시 4 | 1,000,000 | 950,000 | 50,000 | 0 | 상품 합계 부족분이 손실/떨이 반영액으로 설명됨 |
| 예시 5 | 1,000,000 | 950,000 | 30,000 | +20,000 | 손실 반영 후에도 2만 원 초과가 남음(추가 확인 필요) |

#### FR-13: 핵심 지표 계산

시스템은 매출, 매출원가, 매출이익, 이익률, 영업이익, 인당생산성, 평균재고, 평균매출, 매출대비 재고비율, 최고매출품목, 최고매출품목 매출액, 매출차액, 재고/손실 관련 지표를 계산한다.

**검증 조건:**
- 각 일일 장부 상세에서 계산값을 확인할 수 있다.
- 관제판과 리포트는 같은 계산 기준을 사용한다.
- FIFO 기준 원가가 도입되기 전의 MVP 계산값은 운영 확인용 기본 계산값으로 표시된다.
- FIFO 기준 원가와 MVP 기본 계산값의 적용 범위가 화면과 리포트에서 구분된다.
- 계산 기준이 확정되지 않은 항목은 §10 Open Questions의 결정 등급에 따라 처리한다.
- FR-13은 스토리 작성 시 기본 합계/상태 계산, 가격·원가·이익 계산, 민감 지표 노출, FIFO 계산, 희망 판매가 손실 계산으로 slice를 나누어 다룬다.

#### FR-14: 입력 검증

시스템은 필수 입력값 누락, 결제수단 합계와 총매출 차이, 상품별 판매금액 합계와 총매출 차이, 재고/손실 이상 후보를 표시한다.

**검증 조건:**
- 검토/제출 화면에서 주요 누락과 이상 후보를 확인할 수 있다.
- 이상 후보가 있어도 본사 사용자는 장부를 열어 판단할 수 있다.
- 검증 결과는 본사 마감 여부와 구분되어 표시된다.
- 영업 상태가 휴무일인 장부는 미입력 장부와 구분되어 표시된다.

### 4.4 본사 관제판과 이상 신호

**설명:** 본사 첫 화면은 오늘/어제 기준 전체 지점 관제판이다. 본사는 전체 순위보다 문제 가능성이 높은 지점부터 확인해야 한다.

#### FR-15: 전체 지점 관제판

본사 사용자는 오늘/어제 기준 전체 지점의 장부 상태를 한 화면에서 볼 수 있다.

**검증 조건:**
- 각 지점은 한 줄로 표시된다.
- 각 행에는 영업 상태, 장부 상태, 이상 신호, 매출, 이익률, 매출차액, 손실 여부, 마지막 수정자, 마지막 수정 시각, 본사 마감 여부가 표시된다.
- 장부 상태는 미입력, 입력 중, 검토 대기, 본사 마감의 4단계 상태 인디케이터로 직관적으로 표시된다.
- 상태 인디케이터는 현재 조회 시점의 최신 장부 상태를 반영해야 한다.
- 본사 사용자는 지점 행을 선택해 일일 장부 상세로 이동할 수 있다.

#### FR-16: 이상 신호 표시

시스템은 매출/이익률 급락, 매출차액, 재고/손실 이상을 이상 신호로 표시한다.

**검증 조건:**
- 이상 신호는 지점 행에서 눈에 띄게 표시된다.
- 한 지점에 여러 이상 신호가 동시에 표시될 수 있다.
- 이상 신호가 없는 지점도 관제판에서 상태를 확인할 수 있다.

#### FR-17: 이상 신호 기준값 설정

본사 사용자는 이상 신호 기준값을 설정할 수 있다.

**검증 조건:**
- 본사는 매출 하락률, 이익률 하락폭, 매출차액 금액, 손실액, 재고 차이 기준을 설정할 수 있다.
- 기준값 변경 후 새 관제판 조회에는 변경된 기준이 반영된다.
- 기준값 변경 이력이 남는다.
- 기준값은 고정 코드가 아니라 본사 운영 설정으로 저장된다.

### 4.5 본사 입력, 마감, 정정

**설명:** 본사는 관제탑 역할을 하며, 지점 입력 데이터만 보는 사용자가 아니다. 본사는 직접 입력/수정하고 최종 마감한다.

#### FR-18: 본사 직접 입력/수정

본사 관리자 또는 직접 수정 mutation 권한을 가진 본사 프로파일은 모든 지점의 본사 마감 전 일일 장부를 직접 입력하거나 수정할 수 있다.

**검증 조건:**
- 허용된 본사 프로파일은 지점장 입력값을 보완하거나 수정할 수 있다.
- 허용된 본사 프로파일은 매출, 비용, 매입, 재고, 손실, 근무 정보 등 본사 마감 전 장부의 주요 데이터 항목을 강제 입력하거나 수정할 수 있다.
- 본사 수정은 변경 이력에 본사 사용자 수정으로 기록된다.
- 수정 후 계산값과 이상 신호가 갱신된다.
- 조회 전용 본사, 본사 스텝 기본 권한, 업로드 담당자 기본 권한은 직접 수정 mutation을 갖지 않는다.

#### FR-19: 본사 마감과 원본 잠금

본사 관리자, 마감 담당자, 또는 마감 mutation 권한을 가진 본사 프로파일은 일일 장부를 본사 마감 처리할 수 있다.

**검증 조건:**
- 본사 마감 후 원본 장부 입력값은 수정할 수 없다.
- 본사 마감 상태는 관제판과 장부 상세에서 표시된다.
- 지점장은 본사 마감된 장부를 수정할 수 없다.
- 마감 전에는 필수 누락, 이상 신호, 이월 공백, 검증 오류, 미확정 정정 반영 여부가 요약되어야 한다.
- 검증 오류가 있어도 마감할 수 있는 예외 상황은 §4.5 마감 가능성 표를 따르며, 사유 입력과 감사 로그가 필요하다.

**마감 가능성 표:**

| 조건 | 개별 마감 | 일괄 마감 | 예외 허용 | 필수 감사 필드 |
| --- | --- | --- | --- | --- |
| 로그인 없음 또는 마감 권한 없음 | 차단 | 차단 | 불가 | 사용자, 권한 실패 사유 |
| 장부 id, 지점, 일자 식별 불일치 | 차단 | 차단 | 불가 | 요청값, 서버 기준값 |
| 이미 본사 마감된 장부 | 차단 | 차단 | 불가 | 장부 id, 기존 마감 이벤트 |
| 숫자 형식 오류 또는 서버 계산 실패 | 차단 | 차단 | 불가 | 오류 필드, 계산 오류 코드/문장 |
| 업로드 확정 반영 실패 또는 대기 배치가 장부 숫자에 영향 | 차단 | 차단 | 불가 | 배치 id, 실패 사유 |
| 미확정 정정 반영 또는 정정 영향 재확인 상태 | 차단 | 차단 | 불가 | 정정 event id, 영향 범위 |
| 필수 입력값 누락 | 사유 입력 시 예외 허용 | 차단 | 개별 마감만 가능 | 누락 항목, 사유, 실행자 |
| 이월 공백 또는 가격 기준 없음 | 사유 입력 시 예외 허용 | 차단 | 개별 마감만 가능 | 공백/가격 누락 항목, 사유, 실행자 |
| 이상 신호 존재 | 경고만 | 사유 입력 시 예외 허용 | 가능 | 이상 신호 요약, 확인 메모 |
| 휴무일 또는 미영업일 | 경고만 | 경고만 | 가능 | 영업 상태, 확인자 |

마감 요약에는 가격 신뢰 상태별 라인 수와 영향 지표를 포함한다. `manual_override` 또는 `basis_missing` 상태가 매출원가, 재고금액, 이익률, 영업이익에 영향을 주면 본사 마감자는 해당 상태를 확인해야 하며, `basis_missing` 사유로 개별 마감된 장부는 월간 리포트에서 별도 건수와 대상 지표로 집계한다.

#### FR-20: 마감 후 정정 기록

본사 마감 후 오류가 발견되면 본사 관리자 또는 정정 mutation 권한을 가진 본사 프로파일은 원본을 수정하지 않고 정정 기록을 추가할 수 있다.

**검증 조건:**
- 정정 기록에는 작성자, 작성 시각, 대상 항목, 원본값, 정정값, 사유가 남는다.
- 정정 기록은 삭제되거나 원본 수정으로 합쳐지지 않는다.
- 장부 상세에서 원본값과 정정 반영값을 구분해 볼 수 있다.
- 정정 기록은 대상 필드의 값을 바꾸는 replacement 방식으로 저장한다. 수량/금액 증감만 입력하는 delta 정정은 별도 유형으로 명시될 때만 허용한다.
- 같은 필드에 여러 정정이 있으면 작성 시각과 이벤트 순서에 따라 최신 유효 정정을 기본 정정 반영값으로 사용한다.
- 잘못된 정정은 삭제하지 않고 `정정 무효화` 이벤트 또는 새 정정 기록으로 바로잡는다.
- 정정에는 사유가 필수이며, 권한 프로파일에 따라 승인 또는 2인 확인이 필요한 항목을 설정할 수 있어야 한다.

#### FR-21: 정정 반영값 사용

관제판과 리포트의 기본 숫자는 정정 반영값을 사용한다.

**검증 조건:**
- 정정 기록 추가 후 관제판과 리포트 숫자가 정정 반영값으로 표시된다.
- 사용자는 상세에서 원본값과 정정 반영값의 차이를 확인할 수 있다.
- 리포트와 export는 원본 기준, 정정 반영 기준, 차이 금액/수량을 구분해 표시할 수 있어야 한다.
- 정정 반영값이 재고 이월, FIFO, 알림 조건에 영향을 주면 영향받는 장부와 리포트에 `정정 반영 재확인` 상태를 표시한다.

**정정 연쇄 전파 상태 전이:**

마감 후 정정이 이후 장부(전일/당일재고, FIFO 잔량, 리포트)에 미치는 영향은 아래 상태 전이로 처리한다. 잘못 마감한 장부를 자동으로 재마감하는 기능은 포함하지 않으며(CAP-15 참조), 대신 영향 범위 식별과 본사 확정 이벤트로 처리한다.

| 단계 | 상태 | 처리 기준 |
| --- | --- | --- |
| 1. 정정 기록 | `정정 반영 대기` | 정정 기록 추가 시 대상 장부와 항목을 표시한다. 원본값은 보존한다. |
| 2. 영향 범위 식별 | `정정 영향 식별` | 정정 대상 항목이 재고 이월·FIFO 잔량·리포트 집계에 쓰이는 경우, 영향받는 날짜 범위(정정 일자부터 가장 최근 마감/저장 장부까지)와 영향받는 후속 장부 목록을 산정한다. |
| 3. 재계산 대기 | `정정 반영 재확인` | 영향받는 후속 장부·리포트에 `정정 반영 재확인` 상태를 표시한다. 본사 확인 전에는 관제판·리포트에서 원본 기준값과 정정 반영 후보값을 함께 노출한다. |
| 4. 본사 확정 | `정정 반영 확정` | 정정 영향 확정 권한을 가진 본사 프로파일이 영향 범위를 확인하고 확정하면 정정 반영값이 관제판·리포트·이월 후보의 기본 운영 숫자가 된다. 확정은 작성자·시각·대상 범위와 함께 감사 로그에 남는다. |
| 5. 확정 보류/무효화 | `정정 무효화` | 정정이 잘못되었으면 새 정정 또는 `정정 무효화` 이벤트로 바로잡으며, 영향받았던 후속 장부의 상태도 다시 산정한다. |

- 영향받는 날짜 범위 산정과 후속 장부 재계산은 즉시 동기 처리하지 않아도 되며, 재계산 큐로 비동기 처리할 수 있다. 다만 재계산 완료 전까지 해당 장부·리포트는 `정정 반영 재확인` 상태를 유지한다.
- 정정으로 마감 장부의 확정 이월 기준(FR-9)이 바뀌면, 영향받는 이후 장부는 `이월 재확인 필요`와 `정정 반영 재확인`을 함께 표시한다.

### 4.6 마스터와 관리 설정

**설명:** 본사는 지점, 사용자, 품목, 매입 기준, 코드, 이상 신호 기준을 관리한다. 지점장은 본사 마스터를 기준으로 장부를 입력한다.

#### FR-22: 지점 마스터 관리

본사 사용자는 지점명과 활성/비활성 상태를 관리할 수 있다.

**검증 조건:**
- 본사는 최소 10개 이상의 지점을 등록하고 운영할 수 있어야 한다.
- 비활성 지점은 신규 장부 입력 대상에서 제외할 수 있다.
- 과거 장부 조회를 위해 비활성 지점 데이터는 보존된다.
- 지점 생성, 활성화, 비활성화, 권한 연결 절차는 개발사가 제공하는 운영 매뉴얼에 포함되어야 한다.
- 지점명 중복, 지점 코드 중복, 지점명 변경 이력, 비활성화 적용일을 관리할 수 있어야 한다.

#### FR-23: 사용자/권한 관리

본사 사용자는 본사 사용자와 지점장 계정, 지점 접근 권한을 관리할 수 있다.

**검증 조건:**
- 지점장 계정은 하나 이상의 지점에 연결된다.
- 권한 변경 후 접근 범위가 즉시 반영된다.
- 권한 변경 전/후 프로파일, 변경자, 변경 시각, 사유가 감사 로그에 남는다.
- 계정 비활성화 후에는 신규 로그인과 mutation이 차단되며, 과거 이력의 사용자 표시는 보존된다.

#### FR-24: 품목 마스터 관리

본사 사용자는 품목명, 구분, 규격, 기본 단가를 관리할 수 있다.

**검증 조건:**
- 장부 입력자는 품목 마스터에서 품목을 선택할 수 있다.
- 장부 입력 시 실제 단가와 수량은 장부 상황에 맞게 입력/조정할 수 있다.
- 품목 마스터 변경은 과거 장부의 원본 입력값을 임의로 바꾸지 않는다.
- 품목명, 규격, 별칭, 활성/비활성, 적용 시작일을 관리할 수 있어야 한다.
- 품목 병합 또는 별칭 매핑이 필요한 경우 과거 원본명과 분석용 품목을 구분해 보존한다.

#### FR-25: 매입 기준 관리

본사 사용자는 매입 관련 기준 또는 기본 정보를 관리할 수 있다.

**검증 조건:**
- 매입 입력 시 본사 기준 정보를 선택하거나 참조할 수 있다.
- MVP에서는 본사와 지점장 모두 장부에 매입 정보를 입력할 수 있다.
- CAP-6 릴리스에서는 본사 업로드 또는 본사 작성 매입을 기본 흐름으로 삼고, 지점장 입력은 본사 허용 조건을 따른다.
- 매입 기준 변경은 과거 장부의 원본 매입 라인을 임의 변경하지 않는다.

#### FR-26: 코드 관리

본사 사용자는 결제수단, 비용 항목, 손실 유형 같은 장부 입력 코드를 관리할 수 있다.

**검증 조건:**
- 장부 입력 화면에서 관리 코드가 선택지로 제공된다.
- 비활성 코드는 신규 입력 선택지에서 제외할 수 있다.
- 과거 장부의 기존 코드 표시는 유지된다.
- 코드 순서, 표시명, 활성/비활성, 적용 시작일을 관리할 수 있어야 한다.

#### 마스터 적용일 공통 규칙

마스터 변경은 과거 장부와 업로드 trace를 임의로 다시 쓰지 않는다.

| 마스터 유형 | 적용 기준 | 과거/진행 중 데이터 처리 |
| --- | --- | --- |
| 지점 | 활성/비활성 적용일 이후 신규 장부 생성과 목록 표시 기준에 적용 | 과거 장부의 지점명과 코드는 보존하고 변경 이력을 표시한다. |
| 사용자/권한 | 저장 즉시 신규 요청과 활성 세션 권한 재검사에 적용 | 과거 감사 로그의 사용자 표시는 당시 인증 계정 기준으로 보존한다. |
| 품목/규격/별칭 | 적용 시작일 이후 신규 장부와 신규 업로드 매핑에 적용 | 기존 장부의 원문 품목명과 당시 분석용 품목 mapping id를 보존한다. |
| 기본 단가/매입 기준 | 적용 시작일 이후 새 입력의 기본값으로만 적용 | 이미 저장된 장부 라인 단가를 자동 변경하지 않는다. |
| 코드 | 적용 시작일 이후 신규 선택지에 적용 | 비활성 코드는 신규 입력에서 숨기되, 기존 장부와 정정 화면에서는 원래 값을 볼 수 있어야 한다. |
| 업로드 매핑 | preview 생성 시점의 mapping version을 고정 | preview 이후 매핑이 바뀌면 재검증 없이는 commit할 수 없다. |

### 4.7 리포트

**설명:** 1차 리포트는 운영 회의와 월간 관리에 필요한 범위로 제한한다.

모든 리포트는 아래 계약을 갖는다.

| 계약 항목 | 기준 |
| --- | --- |
| 필수 컬럼 | 지점, 일자/기간, 장부 상태, 영업 상태, 마감 상태, 정정 반영 여부, 주요 금액/수량, 이상 신호를 리포트 성격에 맞게 명시한다. |
| 필터/정렬 | 기간, 지점, 상태, 이상 신호, 품목/구분 필터와 기본 정렬 기준을 리포트별로 정의한다. |

#### FR-27: 일별 아침 회의 리포트

본사 사용자는 전체 지점의 일별 회의용 요약을 볼 수 있다.

**검증 조건:**
- 리포트는 지점별 마감 상태, 이상 신호, 매출, 이익률, 매출차액, 손실 현황을 보여준다.
- 기본 숫자는 정정 반영값을 사용한다.
- 원본값과 정정 반영값 차이는 상세에서 확인할 수 있다.
- 회의 기준 시각은 본사가 설정하며, 기본값은 오전 8시다.
- 기준 시각에 제출되지 않은 지점은 `지연 제출` 또는 `미제출`로 표시한다.
- 기준 시각 이후 수정되거나 마감된 장부는 리포트에서 최신 반영 시각과 함께 표시한다.

#### FR-28: 지점별 기간 비교

본사 사용자는 선택 기간의 지점별 실적을 비교할 수 있다.

**검증 조건:**
- 비교 항목은 매출, 매출이익, 이익률, 영업이익, 인당생산성, 평균재고, 평균매출, 매출대비 재고비율, 손실을 포함한다.
- 지점별 비교가 가능해야 한다.
- 1차 리포트는 시스템에 입력된 신규 데이터를 기준으로 하며, 과거 엑셀 일괄 이관 데이터는 1차 필수 전제로 삼지 않는다.
- 미입력, 입력 중, 검토 대기, 본사 마감, 휴무일 데이터는 구분되어 집계된다.
- 과거 데이터가 부족한 기간은 `데이터 부족`으로 표시하고, 0과 미입력을 같은 값으로 취급하지 않는다.

#### FR-29: 월간 지점 요약

본사 사용자는 지점별 월간 실적, 마감 현황, 주요 이상 항목, 손실/재고 흐름 요약을 볼 수 있다.

**검증 조건:**
- 월과 지점을 선택해 요약을 볼 수 있다.
- 마감되지 않은 장부가 있으면 요약에서 구분된다.
- 월간 요약은 최고매출품목, 최고매출품목 매출액, 평균재고, 평균매출, 매출대비 재고비율을 포함한다.
- 월간 요약은 월 내 휴무일, 미입력일, 마감 후 정정 반영일을 구분해 보여준다.

## 5. 전역 비기능 요구사항

- **권한 보안:** 본사 사용자와 지점장의 접근 범위는 서버 기준으로 강제되어야 한다.
- **감사 추적:** 마감, 정정, 주요 입력/수정, 기준값 변경은 누가 언제 무엇을 바꿨는지 추적 가능해야 한다.
- **데이터 보존:** 본사 마감 후 원본 장부는 보존되어야 하며, 정정 기록으로 덮어쓰면 안 된다.
- **반응형 웹:** 지점장 입력 화면은 PC, 태블릿, 모바일 웹에서 사용할 수 있어야 한다.
- **운영 속도:** 본사 관제판 기본 조회는 10개 내외 지점 기준 3초 안에 지점 목록과 이상 신호를 표시하는 것을 목표로 한다.
- **모바일 사용성:** 단계형 장부 입력의 핵심 흐름은 최소 390px 폭의 모바일 화면에서 사용할 수 있어야 한다.
- **엑셀 의존 감소:** 1차 제품은 기존 엑셀의 목적을 대체하되, 엑셀 수식이나 서식 구조를 그대로 복제하는 것을 목표로 하지 않는다.
- **운영 인수인계:** 지점, 사용자, 권한, 업로드, 마감, 알림 같은 운영자가 직접 관리할 기능은 사용 매뉴얼 또는 운영 가이드가 제공되어야 한다.
- **운영 대응:** 서버 세팅, 장애 대응, 시인성 개선 요청 같은 운영 업무는 제품 기능과 별도 계약 범위로 관리하되, 시스템 설계 시 운영 담당자가 점검할 수 있는 로그와 설정 화면을 남겨야 한다.
- **지원 환경:** 본사 화면은 최신 Chrome/Edge 데스크톱 브라우저를 기준으로 검증하고, 지점장 입력 화면은 모바일 Chrome/Safari 최신 버전에서 핵심 7단계 입력 흐름을 검증한다.
- **백업/복구:** 일일 장부, 마감, 정정, 감사 로그, 업로드 이력은 운영 백업 대상이며, 제품은 복구 대상 데이터를 식별 가능하게 저장해야 한다. 제품 설계가 보장해야 할 최소 기준은 다음과 같다. 정확한 백업 주기·보존 정책·인프라 비용·운영 대응 시간은 CAP-19 운영 계약에서 구체화하되, 아래 최소선 미만으로 내려갈 수 없다.
  - **RPO(복구 시점 목표):** 영업시간 중 장부·마감·정정·업로드 데이터베이스 변경의 최대 허용 손실은 1시간 이내를 최소 기준으로 한다. 일 1회 백업만으로는 MVP 최소선을 충족하지 못한다.
  - **RTO(복구 소요 목표):** 영업시간 장애는 4시간 이내에 최신 복구 지점 기준으로 읽기 또는 운영 재개 가능한 상태를 목표로 한다. 영업외 장애는 다음 영업일 아침 회의 전까지 복구 가능해야 한다.
  - **감사 로그 보존:** 마감, 정정, 권한 변경, 업로드, 일괄 마감의 감사 로그는 최소 24개월 이상 보존하며, 보존 기간 내 일반 사용자 화면이나 운영자 실수로 삭제·변조되지 않도록 저장한다.
  - **업로드 원본/파싱 결과 보존:** 이카운트 업로드 원본 파일과 파싱 결과(preview/commit/void 이력 포함)는 감사 로그 보존 기준과 동일하게 최소 24개월 보존한다.
  - **복구 훈련:** 오픈 전 최소 1회는 장부, 마감 이벤트, 정정 이벤트, 업로드 원본, 업로드 preview/commit/void/reprocess 이력, 감사 로그 검색을 포함한 복구 리허설을 수행한다. 복구 리허설은 각 대상의 조회 가능 여부, 정정 반영값 재계산 가능 여부, 감사 로그 검색 가능 여부를 확인해야 성공으로 본다.
- **동시 사용:** 10개 내외 지점의 지점장 동시 입력과 본사 사용자 복수 조회를 기본 운영 규모로 본다. 동시 저장 충돌이 발생하면 §3.2의 edit token과 충돌 해결 규칙을 따른다.
- **모니터링:** 업로드 실패, 알림 실패, 마감 실패, 권한 오류, 서버 계산 오류는 운영자가 확인할 수 있는 로그 또는 상태 화면에 남아야 한다.

## 6. 1차 제외 범위

다음 항목은 **MVP 필수 범위**에서 제외한다. 일부 항목은 §8의 승인 추가 구현 범위로 승격되었지만, MVP 오픈 차단 항목으로 보지는 않는다.

- LINE/텔레그램 자동 알림. 단, §8 CAP-11은 승인 추가 구현으로 별도 관리한다.
- 이카운트 직접 API 연동. 단, §8 CAP-6의 이카운트 엑셀 업로드는 승인 추가 구현으로 별도 관리한다.
- POS/카드 매출 자동 연동
- AI 이미지 식별
- 설날 등 특수 기간 비교 리포트. 단, §8 CAP-2는 승인 추가 구현으로 별도 관리한다.
- 출퇴근/급여 수준의 상세 근태 관리
- 직원 마스터, 근무자별 선택, 월간 근무일수 자동 집계, 지각/조퇴 상세 관리. 단, §8 CAP-1/CAP-9는 급여 지급 확정이 아닌 참고 자료 범위로 별도 관리한다.
- 고급 재고 신선도/입고일 추적. 단, §8 CAP-7의 FIFO 매입 lot 추적과 CAP-11의 장기 체화 재고 알림은 승인 추가 구현으로 별도 관리한다.
- 과거 엑셀 데이터 일괄 이관

이 항목들은 1차 MVP 기준으로는 후순위 또는 추후 검토 항목이며, 1차 확정 기능으로 보지 않는다. 1차는 수동 입력, 검증, 본사 관제, 마감, 정정 기록, 기본 리포트를 안정화하는 데 집중한다.

단, 2026-06-08 회의와 고객 제공 엑셀 확인 이후 일부 항목은 §8의 추가 구현 범위로 승격되었다. §6은 1차 MVP 기준선이며, §8은 MVP 이후 추가 범위를 뜻한다.

## 7. MVP 범위

### 7.1 포함 범위

아래 항목은 MVP 제품 범위다. 다만 `MVP 범위`가 곧 `즉시 구현 스토리 작성 가능`을 뜻하지는 않는다. 에픽/스토리 생성 전에는 §0.2와 `mvp-story-extraction-checklist.md`의 slice 상태, 그리고 §10 Open Question gate를 먼저 적용한다.

- 본사/지점장 로그인과 역할 기반 권한
- 지점+일자 일일 장부 생성
- 단계형 장부 입력
- 매출/결제, 비용, 매입, 재고, 손실/폐기/떨이, 근무인원/특이사항 입력
- 품목/규격/단가/재고/수량 기반 재고 관리
- 본사 관제판
- 매출/이익률 급락, 매출차액, 재고/손실 이상 신호
- 재고 조정 기록과 차이 사유 필수 입력
- 본사 기준값 설정
- 본사 직접 입력/수정
- 본사 마감과 원본 잠금
- 마감 후 정정 기록
- 정정 반영값 기반 관제판/리포트
- 지점/사용자/품목/매입 기준/코드 관리
- 일별 아침 회의 리포트
- 지점별 기간 비교
- 월간 지점 요약

### 7.2 제외 범위

MVP 제외 범위는 §6과 동일하다. UX, 아키텍처, 에픽/스토리 작성 시 §6을 MVP 범위 방어선으로 사용하고, §8 항목은 §0.2의 승인 추가 구현 기준에 따라 별도 phase로 다룬다.

## 8. 2026-06-10 추가 구현 범위

이 섹션은 2026-06-08 회의, 고객 제공 엑셀, 추가 견적 기준으로 1차 MVP 이후 구현 범위에 포함된 요구사항을 정리한다. 기존 §6 제외 범위가 삭제되는 것은 아니며, 1차 MVP 기준선과 추가 구현 범위를 구분하기 위한 보강이다.

추가 구현 범위의 공통 원칙은 다음과 같다.

- 기존 장부 원본값은 덮어쓰지 않는다.
- 본사 마감 후 원본 수정 제한과 정정 기록 원칙은 유지한다.
- 본사 수정, 정정, 업로드, 일괄 마감은 감사 로그를 남긴다.
- 지점장 권한 제한은 UI 숨김이 아니라 서버 응답과 mutation 권한에서 강제한다.
- 엑셀 수식은 그대로 복사하지 않고 shared server calculation으로 재정의한다.
- 계산 기준이 불명확한 항목은 임의 확정하지 않고 `기준 확인 필요`로 표시한다.
- AI 분석 화면, AI API 호출, AI 프롬프트, AI 분석 결과 저장은 이번 범위에 포함하지 않는다.
- 운영자가 보는 변경 이력과 오류 사유는 사람이 이해할 수 있는 문장 또는 표기 형태로 제공한다.

**CAP 약속 원장:**

약속 수준은 아래 5개 commitment label만 사용한다. 다른 표현은 설명란에만 쓸 수 있고, 구현 착수 판단에는 쓰지 않는다.

- `contracted build`: MVP 또는 승인 계약/견적 안에서 구현 약속으로 승격된 범위.
- `approved discovery only`: 정책 확정, 자료 확인, 설계 메모 작성만 승인된 범위. 구현 story로 만들 수 없다.

- `contract/ops only`: 제품 기능 구현이 아니라 운영 계약, 비용, 대응 조건 문서에서 다룬다.

### 8.1 직원, 근무, 급여 정산

**CAP-1: 직원 마스터와 월간 근무일수 집계**

본사와 지점은 직원 정보를 관리하고, 일일 장부의 근무자 선택 기록을 월별 근무일수로 집계할 수 있어야 한다. 직원 관리는 정규 시간제 근태보다 일당제 또는 일급 운영에 맞춘다.

**검증 조건:**
- 본사는 직원 등록, 수정, 비활성, 소속 지점을 관리할 수 있다.
- 지점장은 일일 장부에서 근무자를 선택할 수 있다.
- 같은 직원이 여러 지점에서 근무해도 기록이 저장된다.
- 본사는 월별 직원별 근무일수를 볼 수 있다.
- CAP-1 도입 전 MVP 장부에 저장된 근무인원 숫자는 특정 직원의 근무 기록으로 자동 분해하지 않는다.
- 직원은 이름만으로 식별하지 않고 내부 직원 id 또는 중복 구분값을 가진다.
- 직원 정보 생성/수정/비활성 권한은 본사 권한 프로파일로 제한한다.
- 지점장은 자기 지점 근무 입력에 필요한 직원 선택 정보만 볼 수 있으며, 타 지점 전용 직원이나 급여 참고 값은 기본 노출하지 않는다.
- 비활성 또는 퇴사 처리된 직원은 신규 근무 입력 선택지에서 제외하되, 과거 장부와 월 집계에서는 당시 표시명을 보존한다.

**CAP-9: 인사/급여 정산 참고 자료**

본사는 직원별 월 근무 내역, 특수 근태 메모, 급여 차액을 관리하고 여러 지점 근무를 월 정산 참고 자료로 합산할 수 있어야 한다. 급여 차감은 분초 단위 자동 계산이 아니라, 관리자가 근태 메모를 확인한 뒤 급여 차액을 수동 입력하는 간소화 방식으로 처리한다.

**검증 조건:**
- 본사는 직원 입사일, 지각, 조퇴, 특수 상황 메모를 관리할 수 있다.
- 본사는 급여 차액을 수동으로 입력할 수 있다.
- 여러 지점에서 근무한 직원의 월 정산 참고 자료가 한 페이지에서 합산된다.
- CAP-9 도입 전 장부의 근무인원 숫자는 필요한 경우 본사 수동 보정 또는 별도 이관 상태로 표시한다.
- 실제 지급 확정 기능은 별도 범위로 둔다.
- 급여 차액과 특수 근태 메모는 권한 프로파일별로 조회/수정 권한을 제한한다.
- 급여 차액, 특수 근태 메모, 입사/퇴사 관련 정보는 field-level 권한을 갖고, export 가능 프로파일을 별도로 제한한다.
- 직원/급여 참고 자료의 보존, 삭제, 익명화 기준은 OQ-12 확정 시 함께 닫는다.

**직원/급여 개인정보 보존 기본표:**

아래 표는 CAP-1/CAP-9 에픽 작성 전 승인해야 하는 제품 기본값이다. 법정·계약·노무 정책이 더 엄격하면 그 기준을 우선한다.

| 데이터 | 기본 보존/삭제 기준 | 권한/export 기준 |
| --- | --- | --- |
| 직원 id, 이름, 소속 지점, 입사/퇴사 상태 | 활성 기간 동안 보존하고, 퇴사 또는 비활성 후 3년이 지나면 분석용 id만 남기고 화면 표시명은 익명화 후보로 전환 | 본사 관리자 이상 조회, 지점장은 자기 지점 근무 입력에 필요한 이름만 조회 |

| 급여 차액 수동 입력값 | 정산 참고 근거로 36개월 보존을 기본으로 하며, 실제 지급 확정 데이터로 쓰지 않는다 | 지정 급여 담당자와 대표/소유자만 조회/export |

### 8.2 품목 정규화, 매입 업로드, FIFO 재고 원가

**CAP-5: 품목명/규격 정규화**

시스템은 원문 품목명과 분석용 품목/규격을 분리해, 다르게 입력된 같은 상품을 하나로 묶어 분석할 수 있어야 한다.

**검증 조건:**
- 원문 품목명은 과거 장부와 업로드 trace에 보존된다.
- 본사는 품목 별칭과 매입 품목/재고 품목 매핑을 관리할 수 있다.
- 매핑 실패 품목은 본사 검수 화면에 남는다.
- 같은 상품이 다르게 입력되어도 분석용 품목으로 묶을 수 있다.

**CAP-6: 이카운트 엑셀 업로드 기반 매입 자동 생성**

본사는 이카운트 엑셀을 업로드해 검증, 파싱, 품목 매핑을 거친 매입 라인을 해당 장부에 자동 생성할 수 있어야 한다. 추가 구현 범위에서는 지점 입력 누락과 오입력 위험을 줄이기 위해, 본사 업로드 매입을 기본 흐름으로 본다.

**검증 조건:**
- 본사는 이카운트 엑셀을 업로드할 수 있다.
- 지원 파일은 `.xlsx` 형식의 이카운트 매입/출고 엑셀을 기본으로 한다. CSV, 직접 API 연동, 복수 파일 묶음 업로드는 별도 범위다.
- 업로드 필수 논리 필드는 매입일자, 지점 또는 출고처 식별자, 원문 품목명, 규격, 수량, 단가, 매입금액이다. 공급처, 비고, 세금 구분은 선택 필드로 받을 수 있다.
- 실제 엑셀 헤더명이 다른 경우 본사 설정에서 헤더 매핑을 관리한다. 필수 논리 필드로 매핑되지 않은 파일은 commit할 수 없다.
- 업로드 파일의 품목명, 규격, 단가, 수량 같은 주요 매입 정보가 preview 단계에서 파싱된다.
- 업로드 preview는 지점/일자 매칭, 품목 매핑 실패, 중복 가능성, 필수 컬럼 누락, 숫자 형식 오류를 보여준다.
- 같은 파일, 같은 source row, 또는 같은 사업 문서로 보이는 중복 업로드는 감지되어야 하며, 재업로드 시 덮어쓰기/추가/취소 정책을 사용자가 선택할 수 있어야 한다.
- 파일 단위 중복 식별은 파일명만으로 판단하지 않고 파일 해시, 업로드 배치 id, source sheet, source row number를 함께 사용한다.
- 업로드 실패 사유가 사용자에게 보이며 감사 로그에 남는다.
- 업로드된 매입은 preview 확인 후 commit 단계에서 해당 지점/일자 장부에 반영된다.
- 업로드 검수 과정에서 본사는 품목 마스터의 기준 가격에 구애받지 않고 출고 단가와 수량을 직접 수정하거나 강제 입력할 수 있다.
- 지점장은 본사 작성 또는 이카운트 업로드 매입을 기본적으로 보기만 할 수 있으며, 임의 수정할 수 없다.
- 비상용 지점 강제 입력창을 제공하는 경우에는 본사 허용 여부, 사유, 작성자, 작성 시각을 감사 로그에 남긴다.
- 업로드 이력과 감사 로그가 저장된다.
- 업로드 라인과 수동 매입 라인은 같은 장부에서 출처가 구분되어야 한다.

**업로드 상태 모델과 취소/재처리 규칙:**

각 업로드 배치는 아래 상태를 갖는다. commit 이후의 취소·재처리도 원본 이력을 보존하는 추가 이벤트로만 처리하며, 마감 장부에 대한 우회 수정은 허용하지 않는다.

| 상태 | 의미 | 마감 전후 규칙 |
| --- | --- | --- |
| `preview` | 파싱·매핑 검증 후 commit 대기 상태. 아직 장부에 매입 라인을 생성하지 않는다. | 마감과 무관. preview는 장부 숫자에 영향을 주지 않는다. |
| `committed` | preview 확인 후 매입 라인이 장부에 반영된 상태. | 대상 장부가 본사 마감 전이면 정상 반영된다. 이미 마감된 장부에는 직접 commit할 수 없고, FR-20 정정 기록 흐름으로만 처리한다. |
| `voided` | commit된 업로드를 취소한 상태. 생성된 매입 라인을 무효 처리하되 원본 업로드 이력은 보존한다. | 마감 전 장부만 직접 void 가능하다. 마감 후에는 void 대신 정정 기록으로 처리한다. void에는 사유·작성자·시각이 감사 로그에 남는다. |
| `reprocessed` | 같은 원본 파일 또는 수정한 파일을 다시 처리한 상태. 직전 commit과의 관계(덮어쓰기/추가/취소 후 재생성)를 명시한다. | 재처리는 새 배치로 기록하며 이전 배치 상태(`committed`/`voided`)와 연결해 추적한다. |
| `failed` | 검증·파싱·commit 중 실패한 상태. 실패 사유가 사용자와 감사 로그에 남는다. | 실패 배치는 장부에 라인을 남기지 않는다. |

- **행 단위 idempotency:** 행 identity는 업로드 배치 id, source sheet, source row number, 원본 문서번호/공급처/출고문서(있을 때), 정규화 품목 mapping id를 우선 사용한다. 수량·단가·금액·원문 행 해시는 보조 중복 신호이며, 이것만으로 서로 다른 실제 매입 행을 자동 병합하지 않는다.
- **preview 편집 identity:** preview 단계에서 본사가 단가, 수량, 품목 매핑을 고치더라도 원본 source identity와 preview edit version을 함께 보존한다. 한 원본 행을 여러 매입 라인으로 나누면 child line id를 부여한다.
- **취소 감사 이벤트:** commit·void·reprocess는 각각 별도 감사 이벤트로 남으며, 대상 배치 id, 영향받은 매입 라인, 사유, 작성자, 시각을 포함한다.

**CAP-7: FIFO 재고 금액 계산과 매입 잔량 이력**

시스템은 같은 품목의 매입 단가가 바뀌어도 선입선출 방식으로 재고 금액과 매입 잔량 근거를 추적할 수 있어야 한다. 이카운트의 FIFO 원가 추적 방식과 맞추는 것을 목표로 하되, 반품/조정/폐기/떨이 처리 순서는 별도 정책으로 확정한다.

**검증 조건:**
- 매입 라인별 잔여 수량, 매입일, 단가, 수량, 잔량이 저장된다.
- 같은 품목의 단가가 여러 번 바뀌어도 잔량별 원가가 추적된다.
- 본사는 특정 재고 금액의 매입 근거를 기본 1개월 범위와 기간 필터로 볼 수 있다.
- 본사 대시보드나 재고 화면에서 전일재고 또는 재고금액을 선택하면 해당 품목의 매입일, 단가, 수량, 잔량 근거를 팝업 또는 상세 패널로 확인할 수 있다.
- FIFO 기준 계산값과 MVP 기본 계산값의 적용 범위가 구분된다.
- FIFO 계산이 불가능한 데이터는 `확인 필요`로 표시된다.
- FIFO 적용 품목, 반품, 재고 조정, 폐기, 떨이 처리 순서는 OQ-7/OQ-17이 닫히기 전까지 확정 계산으로 표시하지 않는다.

### 8.3 본사 재고와 상품 분석

**CAP-8: 본사 통합 전체 재고 뷰**

본사는 모든 지점의 품목별 재고와 전체 합계를 한 화면에서 확인할 수 있어야 한다. 이 화면은 본사 매입 발주 계획의 기초 자료로 쓰인다.

**검증 조건:**
- 본사는 특정 품목이 각 지점에 얼마나 남았는지 한 화면에서 본다.
- 본사는 전체 지점 합산 수량을 볼 수 있다.
- 10개 이상 지점의 품목별 잔여 수량과 전체 합산 수량을 확인할 수 있다.
- 냉동/생물, 품목, 기간 필터를 사용할 수 있다.

**CAP-4: 상품별 관리자 분석**

본사는 장부별 또는 기간별로 상품 판매, 마진, 최고매출품목을 분석할 수 있어야 한다.

**검증 조건:**
- 본사는 품목규격 기준 전재고, 매입, 판매원가, 이익률, 판매금액, 판매량을 볼 수 있다.
- 최고매출품목이 자동 계산된다.
- 냉동품과 생물 그룹별 총판매액과 이익률을 당일 및 기간 기준으로 볼 수 있다.
- 냉동품과 생물 그룹 분석은 표뿐 아니라 그래프 또는 차트로도 확인할 수 있다.
- 계산 기준이 불확실한 항목은 `기준 확인 필요`로 표시된다.

### 8.4 권한, 마감, 손실, 입력 흐름 보완

**CAP-13: 지점장 민감 회계 지표 노출 정책 고도화**

지점장 화면과 서버 응답에서 민감 회계 지표를 기본 차단하는 최소 기준은 §4.1 MVP 필수다. CAP-13은 그 위에 본사가 지표별 노출 정책을 조정하고, 차단/허용 이력을 감사할 수 있게 하는 고도화 범위다.

숨길 지표:

- 매출원가
- 매출이익
- 인당생산성
- 영업이익

**검증 조건:**
- 지점장 권한으로 민감 지표를 화면이나 API 응답에서 볼 수 없는 최소 기준은 MVP에서 먼저 충족한다.
- 본사 권한은 필요한 회계 지표를 볼 수 있다.
- 지점장 화면에는 현장 입력값과 비민감 운영 요약만 표시한다.
- 지점장에게 마진율, 재고 금액, 상품별 원가 근거처럼 원가/이익을 역추정할 수 있는 값은 OQ-10A에서 MVP 최소 노출 차단 기준이 확정되기 전까지 숨김으로 본다.
- 민감 지표 숨김은 화면뿐 아니라 API 응답, export, 리포트 링크, 캐시 데이터에서도 적용된다.
- CAP-13 고도화에서는 지표별 허용/차단 정책, 변경자, 변경 사유, 적용 시작일, 차단된 export/API 요청 감사 로그를 제공한다.
- OQ-10B가 닫히기 전에는 CAP-13 고도화를 노출 허용 기능으로 구현하지 않고 차단 기본값과 감사 강화만 구현할 수 있다.

**CAP-14: 희망 판매가 기준 손실액 계산**

손실액 계산 정책은 단순 매입원가가 아니라 지점장이 당일 영업 개시 전에 설정한 희망 판매가와 실제 처리 금액의 차이를 반영할 수 있어야 한다. 희망 판매가는 권장가 또는 목표가 성격의 값으로 다룬다.

**검증 조건:**
- 희망 판매가, 실제 판매/떨이/폐기 처리 금액, 손실 사유를 저장할 수 있다.
- 희망 판매가는 당일 영업 개시 전 지점장 입력값을 기본 기준으로 삼는다.
- 희망 판매가와 실제 처리 금액의 차이를 통해 당일 누수 금액과 실제 마진율 차이를 추적할 수 있다.
- 손실액 계산 기준은 화면과 리포트에서 일관되게 적용된다.
- CAP-14는 OQ-9가 닫히기 전까지 구현 스토리로 확정하지 않는다.
- OQ-9 확정 시 영업 개시 시각, 가격 잠금 시각, 늦은 입력 처리, 영업 중 변경 version, 본사 override 권한, 희망 판매가 미입력 시 closeability를 함께 정의한다.

**CAP-15: 본사 일괄 마감 또는 검토 생략 토글**

본사는 지점 장부를 개별 검토하지 않고 바로 마감할 수 있는 옵션을 가져야 한다. 이 옵션은 본사 판단에 따라 수동 검토 단계를 생략하는 프로세스 간소화 장치다.

**검증 조건:**
- 본사 권한만 일괄 마감을 실행할 수 있다.
- 본사는 검토 생략 또는 즉시 일괄 마감 옵션을 명시적으로 켜고 끌 수 있다.
- 일괄 마감 전 dry run으로 대상 지점/일자, 현재 상태, 검증 오류, 이상 신호, 누락 장부, 휴무일, 이미 마감된 장부를 요약한다.
- 검증 오류나 이월 공백이 있는 장부를 포함해 마감하려면 사유 입력이 필요하다.
- 일괄 마감은 장부 단위 독립 처리로 본다. 일부 장부가 실패해도 이미 성공한 장부의 마감을 자동 rollback하지 않는다.
- 부분 실패가 발생하면 성공/실패 건수와 실패 사유가 표시되고 감사 로그에 남는다. 재시도는 실패한 장부 중 여전히 마감 가능 상태인 장부만 대상으로 한다.
- 잘못 마감한 장부를 되돌리는 기능은 MVP/승인 추가 구현 범위에 포함하지 않는다. 마감 후 오류는 FR-20 정정 기록으로 처리한다.
- 일괄 마감 실행 전 위험 알림과 확인 요약이 표시된다.
- 실행 내역은 대상 장부 목록, 실행자, 실행 시각, 사유, dry run 결과와 함께 감사 로그에 남는다.
- 기존 마감 후 원본 수정 제한은 유지된다.
- 일괄 마감은 §4.5 마감 가능성 표를 따르며, 개별 마감보다 느슨한 예외 처리를 허용하지 않는다.

**CAP-16: 작성자 이름 세션 캐싱**

장부 작성 과정에서 1단계에 입력한 작성자 이름이 7단계 전체 저장까지 유지되어야 한다.

**검증 조건:**
- 작성자 이름을 한 번 입력하면 같은 장부 저장 흐름 안에서 다시 입력하지 않아도 된다.
- 7단계 진행 중 부분 저장이나 부분 수정이 발생해도 작성자 이름이 유지된다.
- 최종 저장 데이터에는 작성자명이 누락되지 않는다.
- 작성자 이름은 화면 표시용 값이며 인증 계정, 감사 로그 실행자, 실제 입력자와 구분된다.
- 공동 기기나 대리 입력 상황에서 작성자 이름과 인증 계정이 다르면 장부 상세에서 둘 다 확인할 수 있어야 한다.

**CAP-17: 재고 테이블 문구 변경**

기존 `차이` 항목 명칭은 사용자가 이해하기 쉬운 `당일 판매량`으로 바꾼다.

**검증 조건:**
- 단순 라벨 변경인지 계산 의미 변경인지 먼저 확인한다.
- 계산 의미가 바뀌면 관련 리포트와 이상 신호 계산도 함께 맞춘다.

**CAP-18: 대시보드 그리드 리사이징**

본사는 대시보드 테이블의 컬럼, 영역, 주요 레이아웃 크기를 업무와 시인성에 맞게 조절할 수 있어야 한다.

**검증 조건:**
- 본사 대시보드에서 대상 컬럼 또는 영역 크기를 조절할 수 있다.
- 조절값은 기본적으로 사용자별로 저장되며, `기본값으로 되돌리기`가 제공된다.
- 지점명, 장부 상태, 마감 상태, 이상 신호, 최신 반영 시각 같은 핵심 컬럼은 숨기거나 읽을 수 없을 정도로 줄일 수 없다.
- 데스크톱 기준 최소/최대 폭과 overflow 처리 기준을 UX 설계에서 숫자로 확정한다. 모바일 화면에서는 리사이징보다 기본 반응형 배치를 우선한다.
- 키보드 또는 보조기기 사용자가 조절 가능한 영역을 인지할 수 있어야 한다.

### 8.5 리포트 고도화와 월 손익

**CAP-2: 특수기간 비교 리포트**

본사는 설날 같은 특수기간을 기준으로 지점 실적을 연도별로 비교할 수 있어야 한다.

**검증 조건:**
- 본사가 특수기간과 기준일을 등록할 수 있다.
- 본사는 선택한 특수기간의 연도별 지점 실적을 볼 수 있다.
- D-day 기준으로 매출, 마진, 근무인원을 비교할 수 있다.
- 특수기간 리포트는 기준 기간, 비교 기간, 지점, 품목/그룹 필터와 미마감/휴무 처리 규칙을 가진다.

**CAP-3: 기존 엑셀 보고 항목 매핑 기반 리포트 고도화**

본사는 기존 엑셀에서 쓰던 주요 보고 항목을 서버 계산 기준으로 다시 매핑해 기간, 월, 연도 기준의 지점 실적을 비교할 수 있어야 한다. 엑셀의 화면 서식, 모든 수식, 차트 모양, 파일 구조를 그대로 맞추는 것은 목표가 아니다.

**검증 조건:**
- 본사가 기간 두 개를 선택해 지점별 증감을 볼 수 있다.
- 본사가 연도별/월별 지점 실적을 볼 수 있다.
- 숫자 기준은 기존 서버 계산 로직을 사용한다.
- 기존 엑셀 항목과 신규 리포트 항목 매칭을 확인할 수 있으나, 엑셀 수식 자체를 권위 있는 계산 기준으로 보지 않는다.
- 리포트별 필수 컬럼, 필터, 정렬, export 컬럼, 차트 유무를 정의한다.

**CAP-10: 본사 고정비 입력과 월 손익계산서**

본사는 매장별 월 고정비를 입력하고 일일 비용과 구분해 월 손익을 계산할 수 있어야 한다. 지점장이 매일 입력하는 식대, 얼음, 주유비, 비품비 같은 일일 현장 비용과 본사가 관리하는 월세, 세금 같은 월 고정비는 분리한다.

**검증 조건:**
- 본사는 지점별 월세, 세금 등 고정비를 입력할 수 있다.
- 월 손익계산서에는 일일 비용과 월 고정비가 구분된다.
- 기간별 또는 월별 지점 손익계산서를 조회할 수 있다.
- 지점장 권한으로 본사 전용 손익 지표가 반환되면 안 된다.
- 월 손익계산서는 정정 반영값, 미마감 장부 포함 여부, 고정비 적용월, export 권한을 명시한다.

### 8.6 외부 알림과 AI 제외 원칙

**CAP-11: LINE 또는 텔레그램 자동 알림**

시스템은 정해진 시간에 본사나 담당자에게 주요 운영 위험을 자동으로 알려야 한다. 예시 시간은 오전 8시이며, 실제 발송 시간은 본사가 설정한다.

LINE을 쓰는 경우 `LINE Notify`를 사용하지 않는다. LINE Official Account와 LINE Messaging API를 기준으로 검토한다.

**검증 조건:**
- 지정 시간에 요약 알림이 발송된다.
- 알림 요약에는 당일 적자 발생 지점, 목표 마진율 미달 지점, 장기 체화 재고 같은 운영 위험 항목을 포함할 수 있다.
- 알림 수신자, 채널, 발송 시간, 조건, 임계값, 중복 억제 기간을 본사가 설정할 수 있다.
- 발송 실패가 로그에 남고 재시도 횟수/간격/최종 실패 표시 정책을 적용할 수 있다.
- 운영자는 test send로 수신자와 메시지 형식을 확인할 수 있다.
- 같은 위험 항목이 반복될 때 알림 피로를 줄이기 위해 중복 알림 억제 또는 escalation 정책을 설정할 수 있다.
- 본사는 알림 조건을 확인하고 설정할 수 있다.
- LINE 사용 시 알림 대상은 Official Account 친구이거나, 알림용 그룹/room에 Official Account가 참여 중이어야 한다.
- 운영 토큰과 외부 API 키는 안전하게 관리한다.
- 알림 템플릿은 수신자 권한 프로파일별 허용 필드를 가진다. 지점장 또는 지점 그룹 알림에는 원가, 이익, 마진율, 재고금액, 본사 고정비 같은 민감 지표를 넣지 않는다.
- 본사 대상 알림과 지점 대상 알림은 다른 템플릿을 사용하며, 새 템플릿이나 새 수신 그룹은 본사 승인 후 활성화한다.
- 알림 로그는 발송 성공/실패, 수신자, 템플릿 id, 주요 이벤트 id를 남기되, 민감한 메시지 본문 전체를 장기 보존하지 않는다.

**CAP-12: AI 기반 경영 분석 확장성**

이번 기능을 만들 때 향후 분석에 필요한 데이터 구조는 남기되, AI 기능 자체는 구현하지 않는다. 예를 들어 향후 “최근 3년간 매출이 가장 높았던 날의 특이사항이나 팀장 배치 현황”을 분석할 수 있도록, 특이사항과 근무 배치 데이터를 구조화해 보존한다.

**검증 조건:**
- 새 기능이 중요한 운영 데이터를 텍스트 메모에만 묻어두지 않는다.
- 근무자, 소속 지점, 근무일, 특수 근태 유형, 손실/폐기/떨이 사유, 이상 신호 유형, 알림 대상, 장기 체화 기준처럼 리포트/검색에 필요한 항목은 구조화 필드로 저장한다.
- 자유 메모는 보존하되, 리포트/검색/필터에 쓰일 핵심 값은 별도 코드나 태그로 저장한다.
- 본사는 작업자, 지점, 근무일, 이상 신호 유형, 손실 사유, 장기 체화 기준으로 검색/필터 가능한 데이터를 확보한다.
- AI 화면, AI API 호출, AI 프롬프트, AI 분석 결과 저장 기능이 추가되지 않는다.

### 8.7 운영 정책과 인프라

**CAP-19: 유지보수 및 서버 인프라 운영 대행**

제품 오픈 후 운영을 위해 서버 세팅, 장애 대응, 사용 중 발생하는 시인성 개선 요청을 처리할 수 있는 유지보수 체계를 별도 계약 범위로 정의해야 한다. CAP-19는 제품 기능 스토리로 자동 변환하지 않는 계약/운영 항목이다.

**검증 조건:**
- 서버 인프라 세팅과 운영 책임 범위가 계약 또는 운영 문서에 명시된다.
- 새벽 시간 등 비상 상황의 장애 대응 연락 경로와 대응 기준이 정의된다.
- 단순 UI/화면 변경이나 시인성 개선 요청의 처리 범위가 유지보수 정책에 포함되는지 명확히 한다.
- 회의 입력의 비용 기준은 월 7~8만 원 수준, 서버 이용료 포함 10만 원 이하이며, 실제 월 유지보수 비용과 서버 이용료 포함 여부는 제품 요구사항이 아니라 계약 조건으로 별도 확정한다.

## 9. 성공 지표

**MVP Primary**

- **SM-1: 아침 회의 준비 가능성** — 본사가 회의 기준 시각 전 전체 활성 지점의 장부 상태, 마감 상태, 최신 반영 시각, 이상 신호를 확인할 수 있다. 목표: 기본 관제판 조회 3초 이내, 미제출/지연/미마감 지점 100% 표시. 검증: FR-15, FR-16, FR-27.
- **SM-2: 본사 마감 통제** — 본사가 각 지점 장부를 검토하고, 필요한 수정 후 마감할 수 있다. 목표: 모든 마감 이벤트가 실행자, 시각, 대상 장부, 검증 요약과 함께 감사 로그에 남는다. 검증: FR-18, FR-19.
- **SM-3: 마감 후 추적 가능성** — 마감 후 오류는 원본 수정이 아니라 정정 기록으로만 반영된다. 목표: 모든 정정은 원본값, 직전 정정 반영값, 새 정정값, 사유를 보존한다. 검증: FR-20, FR-21.

**MVP Operational**

- **SM-4: 지점 입력 완결성** — 지점장 또는 본사가 1차 장부 입력 범위 전체를 한 장부 안에서 입력할 수 있다. 목표: 검토/제출 화면에서 필수 누락, 이상 후보, 이월 공백을 100% 확인할 수 있다. 검증: FR-5~FR-12.
- **SM-5: 본사 기준 운영** — 본사가 품목, 권한, 코드, 이상 신호 기준을 관리할 수 있다. 목표: 마스터/기준 변경은 변경 전/후 값과 변경자가 감사 로그에 남는다. 검증: FR-17, FR-22~FR-26.
- **SM-6: 엑셀 의존 감소** — 오픈 후 운영 장부의 기준 입력/조회 위치가 엑셀이 아니라 ERP Fish가 된다. 측정: 지점별 시스템 장부 작성률, 엑셀 fallback 발생 건수, fallback 사유.

**Approved Extension**

- **SM-7: 업로드 매입 안정성** — 이카운트 엑셀 업로드는 preview/commit 흐름으로 중복, 매핑 실패, 필수 컬럼 누락을 실행 전 보여준다. 측정: 업로드 재작업률, commit 실패율, 중복 후보 중 사용자 확인 후 실제 중복으로 확정된 비율. 검증: CAP-5, CAP-6.
- **SM-8: FIFO/재고 근거 추적성** — 본사는 재고금액 또는 전일재고에서 매입 lot, 단가, 수량, 잔량 근거를 추적할 수 있다. 측정: FIFO `확인 필요` 비율, 근거 조회 성공률, 재고 재확인 건수. 검증: CAP-7, CAP-8.
- **SM-9: 민감 지표 통제** — 지점장 권한은 민감 회계 지표와 역추정 가능한 API/export 값을 받을 수 없다. 측정: 민감 export/API 차단 건수, 잘못된 노출 신고 건수, 노출 정책 변경 감사 로그 완결성. 검증: §4.1, CAP-13.
- **SM-10: 리포트 고도화 판단 가능성** — 특수기간, 기간 비교, 월 손익 리포트는 필수 컬럼, 필터, 집계 규칙, 정정/미마감 처리 기준을 갖는다. 측정: 수동 리포트 보정 건수, `데이터 부족` 표시 건수, 리포트 숫자 이의 제기 건수. 검증: CAP-2, CAP-3, CAP-10.
- **SM-11: 알림 유용성** — 자동 알림은 운영 위험을 놓치지 않되 과다 발송을 통제한다. 측정: 알림 발송 건수, 중복 억제 건수, 실패/재시도 건수, test send 성공 여부. 검증: CAP-11.

**Counter-metrics**

- **SM-C1: 빠른 입력만을 위해 필수 검증을 줄이지 않는다.** 장부 입력 속도는 중요하지만, 매출차액과 재고/손실 이상 확인을 건너뛰는 방향으로 최적화하지 않는다.
- **SM-C2: 이상 신호 수를 줄이는 것을 성공으로 보지 않는다.** 이상 신호는 운영 위험을 찾기 위한 장치이며, 기준값을 느슨하게 해서 조용한 화면을 만드는 것은 성공이 아니다.
- **SM-C3: 알림 발송량 자체를 성공으로 보지 않는다.** 알림은 누락 방지가 목적이며, 중복/피로/오발송은 실패 신호다.
- **SM-C4: 엑셀과 숫자가 같다는 이유만으로 성공으로 보지 않는다.** 기존 엑셀 수식은 참고 자료이며, 서버 계산 정책과 감사 가능한 근거가 우선이다.
- **SM-C5: 민감 지표를 숨기기 위해 본사 운영 판단에 필요한 근거까지 제거하지 않는다.** 지점장 제한과 본사 분석 가능성을 분리해 평가한다.
