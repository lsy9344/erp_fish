# Markdown 원천 문서 대조 리뷰

검토일: 2026-05-28

대상 입력:

- `docs/reference_from_customer/desc.md`
- `docs/reference_from_customer/feature_analysis.md`
- `_bmad-output/planning-artifacts/prds/prd-erp_fish-2026-05-28-2/prd.md`
- `_bmad-output/planning-artifacts/prds/prd-erp_fish-2026-05-28-2/addendum.md`

## 1. 총평

현재 PRD와 addendum은 고객 원문 Markdown의 큰 방향을 대부분 반영하고 있다. 특히 원드라이브 엑셀 수동 운영을 웹 ERP로 바꾸는 목적, 지점장 입력, 본사 관제판, 마감/잠금, 매출차액과 이익률 급락 같은 이상 신호, 품목 단위 재고, 손실/폐기/떨이, 이카운트/POS/텔레그램/AI 이미지 식별의 후순위 분리는 잘 잡혀 있다.

다만 최종화 전에 막아야 할 공백이 있다. 공백의 성격은 새 기능을 더 넣자는 뜻이 아니라, 고객 원문에 있는 핵심 운영 규칙이 현재 PRD에서 확정 범위인지, Open Questions인지, 또는 의도적 제외인지 분명하지 않다는 점이다.

결론: blocker 있음.

## 2. Blocker gaps

### B1. 지점장 제출 후 잠금과 본사 마감 잠금의 권한 경계가 충돌한다

원문 근거:

- `desc.md`는 "당일 마감 장부 입력 완료 시, 지점장이 데이터를 임의로 조작하거나 사후 수정할 수 없도록 마감 락"을 요구한다.
- 현재 PRD는 지점장이 "본사 마감 전까지 수정"할 수 있고, 본사 마감 후 원본 잠금이 걸리는 구조다.

현재 반영 상태:

- PRD FR-18은 본사 마감 후 원본 잠금을 명확히 정의한다.
- 그러나 지점장 제출 직후 지점장 수정 제한이 필요한지, 본사 검토 전에는 지점장이 계속 수정 가능한지에 대한 정책 충돌이 남아 있다.

왜 막아야 하는가:

- 이 권한 경계는 화면 흐름, 상태값, 감사 이력, 수정 요청 UX, 데이터 무결성에 직접 영향을 준다.
- 최종화 전 `지점 제출 후 잠금`, `본사 검토 전 수정 허용`, `수정 요청/재제출` 중 어떤 운영 모델인지 확정해야 한다.

권장 처리:

- PRD에 상태 모델을 명확히 추가하거나 Open Questions에 phase-blocker로 올린다.

### B2. 재고 조정 메뉴와 차이 사유 필수 기록이 요구사항으로 충분히 분리되지 않았다

원문 근거:

- `desc.md`는 실재고와 장부 재고가 다를 때 강제로 맞추는 "재고 조정" 메뉴와 차이 사유 필수 기록을 별도 요구한다.
- `feature_analysis.md`도 "재고 차이 기록", "재고 이월 정합성", "재고 비교" 검증을 요구한다.

현재 반영 상태:

- PRD FR-9는 품목 단위 재고 흐름을 다룬다.
- FR-13/FR-15는 재고 이상 후보를 표시한다.
- 하지만 `재고 조정`이라는 별도 행위, 조정 전/후 값, 조정 사유 필수, 조정 이력, 조정 승인 여부가 명확한 FR로 없다.

왜 막아야 하는가:

- 고객 원문에서 재고 조정은 생선 유통 특성 때문에 핵심 문제 해결 수단이다.
- 단순 재고 입력과 재고 조정은 감사 추적과 리포트 의미가 다르다.

권장 처리:

- PRD 1차 범위에 `재고 조정 기록` FR을 추가하거나, 의도적 제외라면 제외 범위와 Open Questions에 분명히 남긴다.

### B3. 핵심 계산식 일부가 원문에는 있는데 PRD/addendum에는 행동 기준으로 남지 않았다

원문 근거:

- `feature_analysis.md`는 총매출, 매출원가, 매출이익, 이익률, 영업이익, 인당생산성, 재고금액, 상품별 판매금액, 매출차액, 평균 판매단가 계산 정의를 제시한다.
- 특히 `매출원가 = 전일재고 + 매입 - 당일재고`, `영업이익 = 매출이익 - 비용합계`, `인당생산성 = 총매출 / 근무인원`은 시스템 기본 동작에 가깝다.

현재 반영 상태:

- PRD FR-12는 핵심 지표 계산을 요구하지만 계산식은 거의 Open Questions로 밀려 있다.
- Open Questions에는 매출차액, 30%단가, 전일 이월금액, 손실액 저장 방식만 있고, 원문에서 비교적 명확한 기본 계산식 보존은 약하다.
- addendum도 엑셀 관찰 중심이며 계산식 전체를 보존하지 않는다.

왜 막아야 하는가:

- ERP의 핵심은 숫자 기준이다. 계산식이 불명확하면 UX, 백엔드, QA, 이관 검증이 서로 다른 해석을 할 수 있다.

권장 처리:

- 원문에서 확정 가능한 계산식은 PRD 또는 addendum에 기준으로 보존한다.
- 정말 미확정인 항목만 Open Questions로 분리한다.

### B4. 과거 엑셀 데이터 이관 범위와 결측값 정책이 1차 범위에서 불명확하다

원문 근거:

- `feature_analysis.md`는 `total_sales.xlsx`의 2,772행, 기간 2025-01-01~2026-01-31, 결제수단/최고매출품목/매출차액 결측 현황을 기록한다.
- 2차 필수에 "엑셀 과거 데이터 이관"이 포함되어 있고, 과거 결제수단 공란을 미입력으로 둘지 0으로 둘지 정책 질문도 있다.

현재 반영 상태:

- addendum에는 과거 파일 구조와 일부 관찰이 남아 있다.
- PRD는 기간 비교와 월간 매장 요약을 1차에 포함하지만, 과거 데이터 이관 여부와 이관 전까지 리포트가 신규 데이터만 대상으로 동작하는지 명확하지 않다.
- 제외 범위에도 과거 데이터 이관이 없다.

왜 막아야 하는가:

- 기간 비교 리포트가 MVP에 있으므로, 비교 대상 데이터가 신규 입력분만인지 과거 엑셀 포함인지 결정해야 한다.
- 이관 여부에 따라 개발 범위와 QA 기준이 크게 달라진다.

권장 처리:

- PRD에 "1차 리포트는 서비스 시작 후 입력 데이터 기준" 또는 "과거 엑셀 이관 포함"을 명시한다.
- 과거 이관이 후순위라면 제외 범위 또는 Open Questions에 올린다.

## 3. Non-blocker gaps 또는 addendum/Open Questions로 충분한 항목

### N1. 텔레그램 알림

`desc.md`는 텔레그램 자동 메시지를 요구하지만, 현재 PRD는 1차 제외 범위에 넣었고 addendum에도 확장 후보로 보존했다. 의도적 후순위로 보이므로 blocker는 아니다.

### N2. 이카운트 연동, POS/카드 매출 자동 연동, 입고일/신선도 추적, AI 이미지 식별

원문에서 모두 확장 또는 미확정으로 표현되어 있다. 현재 PRD의 1차 제외 범위와 addendum 보존 방식이 적절하다.

### N3. 설날 등 특수 기간 비교

`feature_analysis.md`에는 설날 비교가 있지만 현재 PRD는 제외 범위에 명시했다. 최종화 가능하다.

### N4. 예산 150만 원, 기간 2~3주, 다음 주 월/화 결정

`desc.md`에 있지만 제품 요구사항 본문보다는 제안/계약 맥락이다. PRD나 addendum에 필수로 넣지 않아도 된다.

### N5. 엑셀 오류 건수와 세부 데이터 품질 수치

`feature_analysis.md`의 `#REF!`, `#DIV/0!`, `#VALUE!` 건수는 addendum에 일부 요약으로 충분하다. 단, 이관을 1차에 포함한다면 별도 이관 명세에서 다시 필요하다.

## 4. 원문별 반영 상태

### `desc.md`

잘 반영됨:

- 원드라이브 엑셀 기반 수동 정산을 웹 ERP로 전환한다는 목적
- 7~8개 지점 운영 맥락
- 지점장 계정 기반 웹 입력
- 수식/포맷 파손 방지
- 본사/대표의 문제 매장 중심 모니터링
- 매출차액, 이익률 급락 등 이상 신호
- 생선 유통 특성에 맞는 재고/손실 관리
- 이카운트, 입고일 추적, AI 이미지 식별의 후순위 분리

부족함:

- 지점장 제출 후 잠금 정책과 본사 마감 정책의 관계
- 재고 조정 메뉴와 차이 사유 필수 기록의 별도 요구사항화
- 텔레그램을 제외한 이유와 후순위 조건은 충분하지만, 알림 전체의 1차/2차 경계는 조금 더 명확하면 좋다.

### `feature_analysis.md`

잘 반영됨:

- 엑셀 화면 복제가 아니라 업무 기능 분리라는 방향
- 일일 장부 입력, 결제, 비용, 매입, 재고, 손실, 근무, 본사 관제, 리포트
- 냉동/생물 품목 구분과 품목+규격 단위 재고
- 본사 직접 입력/수정, 마감, 정정 기록
- 기간 비교, 월간 요약, 문제 매장 대시보드
- 상품/품목 마스터, 코드 관리
- 30%단가, 전일 이월금액, 매출차액, 손실액 저장 방식 등 정책 질문

부족함:

- 원문 계산식 보존이 약하다.
- 직원 마스터와 월간 근무일수 집계가 PRD에서는 `근무인원/특이사항` 수준으로 축소되어 있다. 의도적 축소라면 명시가 필요하다.
- 본사 전달사항/검토 메모, 지점 확인 처리, 재검토 상태 관리가 PRD FR로 명확하지 않다.
- 과거 엑셀 데이터 이관과 결측값 정책이 PRD 범위에서 불명확하다.

## 5. 최종화 판단

최종화 blocker가 있다.

최소 해결 조건:

1. 지점장 제출 후 잠금과 본사 마감 잠금의 상태/권한 정책을 확정한다.
2. 재고 조정과 차이 사유 필수 기록을 1차 FR 또는 명시적 제외/Open Question으로 분류한다.
3. 기본 계산식 중 확정 가능한 항목을 PRD/addendum에 보존하고, 미확정 항목만 Open Questions로 남긴다.
4. 과거 엑셀 데이터 이관이 1차 리포트에 포함되는지 명시한다.

이 네 가지가 정리되면 나머지 공백은 Open Questions, addendum, 또는 후순위 범위로 유지해도 무리가 없다.
