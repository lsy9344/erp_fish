# erp_fish - Epic Breakdown

## Overview

This document provides the complete epic and story breakdown for erp_fish, decomposing the requirements from the PRD, UX Design if it exists, and Architecture requirements into implementable stories.

## Requirements Inventory

### Functional Requirements

FR1: 사용자는 본사 사용자 또는 지점장 역할로 로그인할 수 있어야 하며, 역할에 따라 접근 가능한 메뉴와 장부 데이터가 달라져야 한다.

FR2: 지점장은 자기 지점의 일일 장부만 조회하고 입력할 수 있어야 하며, 본사 사용자는 모든 지점의 장부 목록과 상세를 조회할 수 있어야 한다.

FR3: 시스템은 주요 입력과 수정에 대해 입력자, 수정자, 수정 시각, 변경 전 값, 변경 후 값을 기록하고, 지점장 입력과 본사 입력/수정을 구분해 보여줘야 한다.

FR4: 본사 사용자와 지점장은 지점+일자 기준으로 일일 장부를 열 수 있어야 하며, 같은 지점+일자에는 하나의 일일 장부만 존재해야 한다.

FR5: 일일 장부 입력은 매출/결제, 비용, 매입, 재고, 손실/폐기/떨이, 근무인원/특이사항, 검토/제출 순서의 단계형 흐름으로 제공되어야 한다.

FR6: 입력자는 총매출, 현금, 카드, 기타 결제수단, 매출차액 관련 값을 입력할 수 있어야 하며, 시스템은 매출차액을 계산하거나 표시할 수 있어야 한다.

FR7: 입력자는 하나의 일일 장부에 여러 비용 항목, 금액, 메모를 입력할 수 있어야 하며, 비용 합계는 영업이익 계산에 반영되어야 한다.

FR8: 입력자는 품목, 규격, 단가, 수량, 매입금액을 입력할 수 있어야 하며, 본사 매입 기준 또는 품목 마스터를 참조할 수 있어야 한다.

FR9: 입력자는 품목+규격 단위로 전일재고, 매입, 판매/차감, 손실, 당일재고, 재고금액, 수량을 다룰 수 있어야 하며, 월초 재고 스냅샷과 직전 마감 장부의 당일재고를 기준으로 전일재고가 이월되어야 한다.

FR10: 본사 사용자와 지점장은 시스템 계산 재고와 실제 재고가 다를 때 재고 조정을 기록할 수 있어야 하며, 조정 전/후 값, 차이, 사유, 작성자, 작성 시각이 남아야 한다.

FR11: 입력자는 손실, 폐기, 떨이 항목을 품목, 수량, 금액, 처리 유형, 사유/특이사항과 함께 입력할 수 있어야 하며, 손실은 재고 흐름과 손실 리포트에 반영되어야 한다.

FR12: 입력자는 근무인원과 근무 관련 특이사항을 기록할 수 있어야 하며, 근무인원은 인당생산성 계산에 반영되어야 한다.

FR13: 시스템은 매출, 매출원가, 매출이익, 이익률, 영업이익, 인당생산성, 평균재고, 평균매출, 매출대비 재고비율, 최고매출품목, 최고매출품목 매출액, 매출차액, 재고/손실 관련 지표를 계산해야 한다.

FR14: 시스템은 필수 입력값 누락, 결제수단 합계와 총매출 차이, 상품별 판매금액 합계와 총매출 차이, 재고/손실 이상 후보를 표시해야 한다.

FR15: 본사 사용자는 오늘/어제 기준 전체 지점의 영업 상태, 장부 상태, 이상 신호, 매출, 이익률, 매출차액, 손실 여부, 마지막 수정자, 마지막 수정 시각, 본사 마감 여부를 한 화면에서 볼 수 있어야 한다.

FR16: 시스템은 매출/이익률 급락, 매출차액, 재고/손실 이상을 이상 신호로 표시해야 하며, 한 지점에 여러 이상 신호가 동시에 표시될 수 있어야 한다.

FR17: 본사 사용자는 매출 하락률, 이익률 하락폭, 매출차액 금액, 손실액, 재고 차이 기준 등 이상 신호 기준값을 설정할 수 있어야 하며, 변경 이력이 남아야 한다.

FR18: 본사 사용자는 모든 지점의 본사 마감 전 일일 장부를 직접 입력하거나 수정할 수 있어야 하며, 수정 후 계산값과 이상 신호가 갱신되어야 한다.

FR19: 본사 사용자는 일일 장부를 본사 마감 처리할 수 있어야 하며, 본사 마감 후 원본 장부 입력값은 수정할 수 없어야 한다.

FR20: 본사 마감 후 오류가 발견되면 본사 사용자는 원본을 수정하지 않고 정정 기록을 추가할 수 있어야 하며, 정정 기록에는 대상 항목, 원본값, 정정값, 사유, 작성자, 작성 시각이 남아야 한다.

FR21: 관제판과 리포트의 기본 숫자는 정정 반영값을 사용해야 하며, 사용자는 상세에서 원본값과 정정 반영값의 차이를 확인할 수 있어야 한다.

FR22: 본사 사용자는 지점명과 활성/비활성 상태를 관리할 수 있어야 하며, 비활성 지점의 과거 장부 데이터는 보존되어야 한다.

FR23: 본사 사용자는 본사 사용자와 지점장 계정, 지점 접근 권한을 관리할 수 있어야 하며, 권한 변경 후 접근 범위가 즉시 반영되어야 한다.

FR24: 본사 사용자는 품목명, 구분, 규격, 기본 단가를 관리할 수 있어야 하며, 품목 마스터 변경은 과거 장부의 원본 입력값을 임의로 바꾸지 않아야 한다.

FR25: 본사 사용자는 매입 관련 기준 또는 기본 정보를 관리할 수 있어야 하며, 매입 입력 시 본사 기준 정보를 선택하거나 참조할 수 있어야 한다.

FR26: 본사 사용자는 결제수단, 비용 항목, 손실 유형 같은 장부 입력 코드를 관리할 수 있어야 하며, 비활성 코드는 신규 입력 선택지에서 제외할 수 있어야 한다.

FR27: 본사 사용자는 전체 지점의 일별 회의용 요약을 볼 수 있어야 하며, 리포트는 지점별 마감 상태, 이상 신호, 매출, 이익률, 매출차액, 손실 현황을 보여줘야 한다.

FR28: 본사 사용자는 선택 기간의 지점별 실적을 비교할 수 있어야 하며, 비교 항목은 매출, 매출이익, 이익률, 영업이익, 인당생산성, 평균재고, 평균매출, 매출대비 재고비율, 손실을 포함해야 한다.

FR29: 본사 사용자는 지점별 월간 실적, 마감 현황, 주요 이상 항목, 손실/재고 흐름 요약을 볼 수 있어야 하며, 월간 요약은 최고매출품목, 최고매출품목 매출액, 평균재고, 평균매출, 매출대비 재고비율을 포함해야 한다.

### NonFunctional Requirements

NFR1: 본사 사용자와 지점장의 접근 범위는 서버 기준으로 강제되어야 한다.

NFR2: 마감, 정정, 주요 입력/수정, 기준값 변경은 누가 언제 무엇을 바꿨는지 추적 가능해야 한다.

NFR3: 본사 마감 후 원본 장부는 보존되어야 하며, 정정 기록으로 덮어쓰면 안 된다.

NFR4: 지점장 입력 화면은 PC, 태블릿, 모바일 웹에서 사용할 수 있어야 한다.

NFR5: 본사 관제판 기본 조회는 10개 내외 지점 기준 3초 안에 지점 목록과 이상 신호를 표시하는 것을 목표로 한다.

NFR6: 단계형 장부 입력의 핵심 흐름은 최소 390px 폭의 모바일 화면에서 사용할 수 있어야 한다.

NFR7: 1차 제품은 기존 엑셀의 목적을 대체하되, 엑셀 수식이나 서식 구조를 그대로 복제하는 것을 목표로 하지 않는다.

### Additional Requirements

- 첫 구현 스토리는 `pnpm create t3-app@latest erp_fish`로 프로젝트를 초기화하고, NextAuth.js, Prisma, Tailwind CSS, PostgreSQL을 선택해야 한다.
- shadcn/ui는 프로젝트 스캐폴드 후 초기화해야 하며, 기본 ERP 컴포넌트 세트(`button`, `card`, `table`, `form`, `input`, `select`, `dialog`, `sheet`, `tabs`, `badge`, `alert`, `separator`, `skeleton`, `sonner`, `sidebar`, `chart`)를 추가해야 한다.
- MVP에서는 tRPC를 포함하지 않으며, 클라이언트 API 표면이 커질 때만 재검토한다.
- PostgreSQL과 Prisma Migrate를 기본 데이터 아키텍처로 사용해야 한다.
- `storeId + closingDate`는 일일 장부에서 유일해야 한다.
- store, user, product, closing, correction, audit 관계는 중요한 참조에 외래 키를 사용해야 한다.
- 금액과 수량 값은 서버 경계에서 검증하고, 가능한 범위에서 데이터베이스 제약으로 보호해야 한다.
- 마스터 데이터 변경은 과거 장부 원본값을 다시 쓰지 않아야 한다.
- JSON 필드는 좁은 메타데이터, 원본 import trace, 보고에 의존하지 않는 비핵심 메모에만 제한적으로 사용해야 한다.
- 모든 mutation 입력은 Zod로 검증하고, 데이터베이스 제약을 두 번째 방어선으로 사용해야 한다.
- 본사 마감 후 원본 장부는 보존하고, `AuditLog`와 `CorrectionRecord`는 append-only로 유지해야 한다.
- 관제판, 장부 상세, 리포트는 같은 서버 측 계산 함수를 공유해야 하며 UI 전용 계산 규칙을 중복 구현하면 안 된다.
- HQ 관리 내부 계정을 사용하고, 공개 self-signup은 MVP 범위에 포함하지 않는다.
- 모든 장부, 리포트, 마스터 데이터, 마감, 정정, 설정 작업은 서버에서 세션과 역할/지점 접근 권한을 확인해야 한다.
- `HEADQUARTERS` 사용자는 모든 지점에 접근하고 마감, 정정, 마스터 데이터, 기준값 관리를 할 수 있어야 한다.
- `STORE_MANAGER` 사용자는 배정된 지점에만 접근할 수 있어야 하며, 배정 관계는 명시적으로 모델링해야 한다.
- 입력, 수정, 마감, 정정, 이상 신호 기준 변경, 마스터 데이터 변경은 actor, timestamp, target, before, after, reason을 포함하는 감사 기록을 남겨야 한다.
- 감사 기록은 가능하면 비즈니스 변경과 같은 데이터베이스 트랜잭션 안에서 작성되어야 한다.
- 인증 secret, database URL, provider secret은 환경 변수에 두고 저장소에 커밋하면 안 된다.
- MVP 읽기는 Server Components와 서버 쿼리 헬퍼를 사용하고, 내부 mutation은 Server Actions를 사용해야 한다.
- Route Handlers는 인증, export, health check, 향후 webhook/integration처럼 실제 HTTP endpoint가 필요한 경우에만 사용해야 한다.
- 장부, 마감, 정정, 기준값, 마스터 데이터 변경 후에는 Next.js revalidation을 사용해 관제판과 리포트가 낡은 값을 보여주지 않게 해야 한다.
- shadcn/ui primitive를 먼저 사용하고, 커스텀 컴포넌트는 `LedgerStatusBadge`, `ClosingStepForm`, `InventoryFlowTable`, `CorrectionSummary`, `AnomalySignalList` 같은 도메인 조합 컴포넌트로 제한해야 한다.
- 복잡한 폼은 React Hook Form + Zod resolver 또는 Server Action form state를 사용하되, 서버 검증을 권위 있는 검증으로 유지해야 한다.
- MVP 상태 관리는 URL state, server state, local form state를 우선 사용하고, 반복 복잡성이 확인되기 전에는 전역 client state library를 도입하지 않는다.
- 단순 테이블은 shadcn Table을 사용하고, 정렬/필터/페이지네이션/컬럼 표시/행 선택이 충분히 복잡해질 때만 TanStack Table을 추가한다.
- 월간 요약, 지점 비교, 리포트 시각화는 shadcn Chart/Recharts를 사용한다.
- HQ dashboard query는 아침 회의에 필요한 status row 데이터에 집중하고, 무거운 상세/감사/리포트 drilldown은 별도 페이지나 패널로 분리해야 한다.
- Vercel Local, Preview, Production 환경을 분리해 배포하고, Vercel Marketplace의 Neon Postgres 또는 동등한 managed PostgreSQL을 사용해야 한다.
- Production 배포 전 managed PostgreSQL의 connection pooling, backup, point-in-time recovery 역량을 확인해야 한다.
- Prisma migration은 release process에 포함해야 하며, production migration은 특히 ledger, correction, audit, master-data 테이블에 대해 명시적으로 검토해야 한다.
- 인프라/런타임 로그는 Vercel Logs와 Observability를 사용하고, 비즈니스 감사 추적은 애플리케이션 데이터베이스에 보관해야 한다.
- MVP에는 영구 파일 업로드 저장소가 필요 없으며, 리포트 export는 생성 다운로드로 시작한다.
- 돈은 integer KRW 값으로 저장하고, 날짜/시간은 DateTime으로 저장하며, JSON 경계에서는 ISO 문자열을 사용한다.
- 의도적으로 값이 없을 때는 optional 값을 `null`로 표현한다.
- Server Action 결과는 `{ ok: true, data }` 또는 `{ ok: false, error: { code, message, fieldErrors } }` 형태의 표준 `ActionResult<T>`를 사용해야 한다.
- 비즈니스 이벤트 이름은 `ledger.updated`, `ledger.closed`, `correction.created`, `threshold.updated` 같은 dot notation을 사용한다.
- 소스 구조는 `src/features/{domain}`, `src/server`, `src/lib`, `src/components/ui`, `src/components` 경계를 따라야 한다.
- 모든 민감한 server action/query는 shared authorization helper를 호출해야 하며, 숨겨진 UI 컨트롤을 보안 경계로 삼으면 안 된다.
- shared authorization, validation, audit, calculation helper를 사용하고 one-off 변형을 만들지 않아야 한다.
- `검토 대기`는 원본 잠금 상태가 아니며, 원본 장부는 `본사 마감` 때만 잠긴다.
- 구현 스토리는 NextAuth.js/Auth.js 용어를 혼용하지 않도록 하나의 인증 어휘를 일관되게 사용해야 한다.

### 2026-06-10 Additional Scope Requirements

CAP-1: 본사와 지점은 직원 마스터를 관리하고, 일일 장부의 근무자 선택 기록을 월별 근무일수로 집계할 수 있어야 한다.

CAP-2: 본사는 설날 같은 특수기간 템플릿을 만들고 D-day 기준으로 연도별 지점 실적을 비교할 수 있어야 한다.

CAP-3: 본사는 기존 엑셀 보고 양식 수준으로 대조 기간/현재 기간, 연도별, 월별 지점 실적을 비교할 수 있어야 한다.

CAP-4: 본사는 품목규격 기준으로 전재고, 매입, 판매원가, 이익률, 판매금액, 판매량, 최고매출품목을 분석할 수 있어야 한다.

CAP-5: 시스템은 원문 품목명과 분석용 품목/규격을 분리하고, 별칭과 매핑 실패 검수 흐름을 제공해야 한다.

CAP-6: 본사는 이카운트 엑셀을 업로드해 검증, 파싱, 품목 매핑을 거친 매입 라인을 장부에 자동 생성할 수 있어야 한다.

CAP-7: 시스템은 매입 라인별 잔여 수량을 기준으로 FIFO 재고 금액과 매입 잔량 근거를 추적할 수 있어야 한다.

CAP-8: 본사는 모든 지점의 품목별 재고, 지점별 잔량, 전체 합계를 한 화면에서 확인할 수 있어야 한다.

CAP-9: 본사는 직원별 월 근무 내역, 특수 근태 메모, 급여 차액, 여러 지점 근무 합산을 월 정산 참고 자료로 관리할 수 있어야 한다.

CAP-10: 본사는 매장별 월 고정비를 입력하고 일일 비용과 구분해 월 손익계산서를 계산할 수 있어야 한다.

CAP-11: 시스템은 LINE Messaging API 또는 텔레그램 API로 주요 운영 위험을 정해진 시간에 발송하고 실패/재시도/로그를 남겨야 한다.

CAP-12: 새 기능은 향후 분석 가능한 구조화 데이터를 남기되, AI 화면, AI API 호출, AI 프롬프트, AI 분석 결과 저장은 구현하지 않아야 한다.

CAP-13: 지점장 화면과 서버 응답은 매출원가, 매출이익, 인당생산성, 영업이익 같은 민감 회계 지표를 노출하지 않아야 한다.

CAP-14: 손실액 계산은 희망 판매가와 실제 처리 금액 차이를 반영할 수 있어야 하며, 정책 미확정 항목은 임의 계산하지 않아야 한다.

CAP-15: 본사는 개별 검토 없이 장부를 일괄 마감할 수 있는 옵션을 갖되, 권한 제한, 위험 알림, 감사 로그를 반드시 적용해야 한다.

CAP-16: 장부 작성 과정에서 1단계에 입력한 작성자 이름은 7단계 전체 저장까지 유지되어야 한다.

CAP-17: 재고 테이블의 `차이` 항목은 `당일 판매량`으로 바꾸되, 계산 의미 변경 여부는 리포트와 이상 신호까지 함께 확인해야 한다.

CAP-18: 본사는 대시보드 테이블의 컬럼 또는 영역 크기를 업무에 맞게 조절할 수 있어야 한다.

### 공통 구현 가드레일

CAP-1~CAP-18 구현 시 날짜, 지점, 직원, 품목, 금액, 상태, 이력 데이터는 조회 가능한 구조로 저장한다. 중요한 운영 데이터를 텍스트 메모에만 묻어두지 않는다.

이번 범위에는 AI 챗봇, 자연어 질의, AI API 호출, 프롬프트 저장, AI 분석 결과 저장을 포함하지 않는다. 구현 중 AI 기능처럼 보이는 요구가 생기면 별도 고도화 범위로 분리한다.

### 공통 구현 체크리스트

모든 UI/리포트/장부 화면 구현자는 다음을 확인한다.

- 접근성: WCAG 2.2 AA, 키보드 이동, `aria-label`, `scope="col"`, Dialog focus trap을 확인한다.
- 상태 문구: `확인 필요`, `계산 불가`, `데이터 부족`은 공통 상태 enum 또는 공통 표시 helper를 사용한다.
- 디자인 토큰: `DESIGN.md`의 primary/accent/warning 오버라이드, light/dark 색상 쌍, tabular-nums, radius 규칙을 `globals.css` 또는 지정된 테마 파일에서 적용한다.
- 범위 경계: MVP와 추가 구현 범위를 섞지 않는다. 직원/급여, 특수기간, 이카운트, FIFO, 알림은 Epic 6~11 순서를 따른다.
- Epic 10 리포트 구현 전에는 기간 비교, 연도/월별, 특수기간, 월 손익, 엑셀 항목 매칭의 우선순위를 다시 확인한다.

### UX Design Requirements

UX-DR1: shadcn/ui + Tailwind CSS를 UI 시스템으로 사용하고, `DESIGN.md`를 시각적 정체성의 단일 출처로 삼아야 한다.

UX-DR2: shadcn 기본 토큰 중 `primary`, `accent`, 커스텀 `warning`만 브랜드 레이어에서 오버라이드하고, `background`, `foreground`, `muted`, `border`, `card`, `popover`, `destructive` 등 나머지 토큰은 shadcn 기본값을 유지해야 한다.

UX-DR3: 라이트/다크 모드를 모두 지원하고, `primary`, `accent`, `warning`에 대해 light/dark 색상 쌍을 구현해야 한다.

UX-DR4: body font-family에는 `Inter`, `"Apple SD Gothic Neo"`, `"Noto Sans KR"`, `system-ui`, `sans-serif` fallback을 포함해야 한다.

UX-DR5: 매출, 재고, 이익률 같은 숫자 데이터에는 `tabular-nums` 또는 동등한 numeric font feature를 적용해야 한다.

UX-DR6: radius는 업무 도구 느낌에 맞게 `sm 4px`, `md 6px`, `lg 8px`를 사용하고, pill 형태는 배지와 신호 칩에만 사용해야 한다.

UX-DR7: 장부 상태 배지 `StatusBadge`는 미입력, 입력중, 검토대기, 본사마감, 휴무 5종을 텍스트만으로 표시하고 각 상태의 semantic style을 적용해야 한다.

UX-DR8: 이상 신호 칩 `SignalChip`은 아이콘과 짧은 레이블을 함께 표시해야 하며, 색상만으로 의미를 전달하면 안 된다.

UX-DR9: 이상 신호는 경고 등급(amber)과 심각 등급(red)을 구분해야 하며, 툴팁에 구체적 수치를 표시할 수 있어야 한다.

UX-DR10: 관제판 행 `BranchRow`는 지점당 한 행으로 표시하고, 행 전체 클릭으로 장부 상세에 진입해야 하며, 지점명, 상태 배지, 이상 신호 칩, 매출, 이익률, 마지막 수정자, 마감 여부를 보여줘야 한다.

UX-DR11: 이상 신호가 있는 관제판 행은 좌측 컬러 보더와 배경 강조를 사용하고, hover 시 `bg-muted/50`을 적용해야 한다.

UX-DR12: 장부 상세에는 품목 재고 테이블 `InventoryTable`이 포함되어야 하며, 냉동/생물 탭, 품목, 규격, 전일재고, 매입, 당일재고, 재고금액, 수량 컬럼을 제공해야 한다.

UX-DR13: 수정된 재고 행은 primary 기반 강조 스타일을, 신규 품목 행은 success 기반 강조 스타일을 적용해야 한다.

UX-DR14: 재고 입력은 전일 이월값을 프리필하고, 변경된 품목만 수정하는 흐름을 기본으로 해야 한다.

UX-DR15: 재고 행 인라인 편집 `InventoryRowEdit`은 셀 클릭/탭으로 편집에 진입하고, 변경 즉시 수정 표시를 하며, 수량 0 입력 시 확인 프롬프트를 제공해야 한다.

UX-DR16: 한 지점 하루 장부의 재고 품목은 냉동+생물 합산 30개 이상을 고려해야 하며, 모바일에서는 가상 스크롤 또는 50행 단위 페이지 처리를 제공해야 한다.

UX-DR17: 단계형 입력 폼 `StepForm`은 매출/결제, 비용, 매입, 재고, 손실/폐기, 근무인원, 검토/제출의 7단계를 제공해야 한다.

UX-DR18: 단계형 입력은 단계 간 자유 이동을 허용하고, 저장된 데이터는 단계 이동 후에도 유지되어야 한다.

UX-DR19: 단계형 입력 폼 상단에는 완료, 현재, 대기 상태를 구분하는 Step Indicator를 표시하고, 현재 단계에는 `aria-current="step"`을 적용해야 한다.

UX-DR20: 검토/제출 요약 `ReviewSummary`는 총매출, 매출이익, 이익률, 영업이익, 인당생산성, 재고금액 같은 핵심 계산값과 누락 항목, 이상 신호 후보를 보여줘야 한다.

UX-DR21: 이상 후보는 제출 또는 마감을 자동 차단하지 않고, 사용자가 확인할 수 있는 경고로 표시해야 한다.

UX-DR22: 본사 마감 확인 `ClosingDialog`는 shadcn Dialog로 구현하고, 마감 후 지점장이 원본을 수정할 수 없으며 정정 기록만 추가할 수 있음을 명확히 알려야 한다.

UX-DR23: 정정 기록 패널 `CorrectionPanel`은 원본값과 정정값을 나란히 표시하고, 정정 사유 필수 입력과 정정 이력 타임라인을 제공해야 한다.

UX-DR24: 마감된 장부 상세의 원본 입력 필드는 disabled 상태여야 하며, 정정 기록 추가만 가능해야 한다.

UX-DR25: 본사 사용자는 데스크탑 좌측 사이드바를 통해 관제판, 리포트, 기준정보, 설정에 접근해야 한다.

UX-DR26: 지점장은 모바일 하단 탭 바를 통해 장부, 재고, 손실에 접근해야 하며, 탭 이동은 현재 입력 단계를 유지해야 한다.

UX-DR27: 데스크탑 사이드바는 240px, `md` 구간에서는 56px 아이콘 전용 collapsed 상태, 모바일/태블릿 지점장 화면은 56px 하단 탭 바와 safe-area inset을 적용해야 한다.

UX-DR28: Dialog 위에 Dialog를 쌓지 않고, 모달 스택은 1단계 이하로 제한해야 한다.

UX-DR29: 장부 상세 진입은 데스크탑에서 Sheet 또는 페이지, 모바일에서 새 페이지 방식으로 제공해야 한다.

UX-DR30: 모바일/태블릿 터치 타겟은 재고 행 편집 셀을 포함해 최소 44x44px를 확보해야 한다.

UX-DR31: 숫자 입력 필드는 `inputmode="numeric"` 또는 `inputmode="decimal"`을 사용해 모바일에서 숫자 키패드를 띄워야 한다.

UX-DR32: 데스크탑 키보드 사용자는 `Tab`, `Shift+Tab`, `Enter`, `Esc`로 폼 이동, 편집 확정, Dialog/Sheet 닫기, 인라인 편집 취소를 할 수 있어야 한다.

UX-DR33: v1에서는 단축키, 커맨드 팔레트, 드래그 재정렬, 브라우저 push 알림, 스와이프 제스처를 도입하지 않는다.

UX-DR34: 데이터 로딩 상태는 shadcn Skeleton으로 실제 레이아웃과 같은 형태를 보여줘야 한다.

UX-DR35: 저장 성공/실패는 shadcn Toast/Sonner로 보여주고, 실패 toast에는 재시도 버튼을 포함해야 한다.

UX-DR36: 폼 유효성 오류는 필드 하단 인라인 메시지로 보여주고, 제출 시 첫 번째 오류 필드로 포커스를 이동해야 한다.

UX-DR37: 권한 없는 직접 URL 접근은 빈 에러 페이지가 아니라 "접근 권한이 없습니다." 화면으로 처리해야 한다.

UX-DR38: 전 서피스는 WCAG 2.2 AA를 목표로 해야 하며, 색상 대비와 키보드 접근성을 검증해야 한다.

UX-DR39: 장부 상태 배지는 스크린 리더용 aria-label을 제공해야 한다. 예: "현대점 장부 상태: 검토 대기".

UX-DR40: 재고 테이블 컬럼 헤더는 `scope="col"`을 사용하고, 수정된 행에는 "수정됨"을 전달하는 aria-label을 추가해야 한다.

UX-DR41: Dialog, Sheet, Popover는 접근 가능한 제목, focus trap, 닫힌 후 trigger로 포커스 반환을 제공해야 한다.

UX-DR42: `lg` 이상에서는 본사 관제판 테이블 전체 컬럼을 표시하고, `md`에서는 일부 컬럼을 숨기며, `< md`에서는 하단 탭과 간소 테이블 또는 카드형 관제판을 제공해야 한다.

UX-DR43: 모바일 재고 테이블은 가로 스크롤을 허용하고, 태블릿 landscape에서는 주요 컬럼(품목, 전일재고, 당일재고, 재고금액)을 고정해야 한다.

UX-DR44: 마이크로카피는 간결한 한국어 업무 문장으로 작성하고, 오류 코드나 영문 alert 문구를 그대로 노출하지 않아야 한다.

UX-DR45: 금액은 천 단위 콤마와 원 단위 표기(예: `₩3,420,000`)를 사용하고, 이익률은 소수 첫째 자리까지 표시해야 한다.

UX-DR46: key dashboard, key inventory, key closing detail 목업은 참조하되, 목업과 Experience Spine이 충돌하면 Experience Spine을 우선해야 한다.

**And** 사용자는 로그인 화면으로 안내되어야 한다.

**And** 좌측 사이드바에는 홈, 리포트, 기준정보, 설정 진입점이 보여야 한다.

**Given** 로그인한 본사 사용자가 업무 셸을 볼 때
**When** 화면이 렌더링되면
**Then** shadcn/ui 기반 레이아웃과 ERP Fish 기본 디자인 토큰이 적용되어야 한다
**And** 라이트/다크 모드에서 기본 텍스트와 주요 버튼 대비가 깨지지 않아야 한다.

**When** 사용자가 로그인을 시도하면
**Then** 간결한 한국어 오류 메시지가 표시되어야 한다
**And** 오류는 필드 또는 폼 수준에서 접근 가능하게 연결되어야 한다.

**And** 사용자는 "접근 권한이 없습니다." 화면을 봐야 한다.

**Given** 지점장 권한 검사가 수행될 때
**When** 장부, 재고, 손실 화면의 데이터를 조회하면
**Then** 서버는 세션의 사용자 역할과 지점 배정을 기준으로 접근을 허용하거나 거부해야 한다
**And** 클라이언트에서 숨긴 메뉴만을 보안 경계로 삼으면 안 된다.

**Given** 지점장 계정에 배정된 지점이 없거나 비활성 지점만 배정되어 있을 때

**And** 목록은 지점명이 많아져도 찾을 수 있도록 검색 또는 필터를 제공해야 한다.

**Given** 본사 사용자가 새 지점을 등록할 때
**When** 필수 값인 지점명을 입력하고 저장하면
**Then** 새 지점이 활성 상태로 생성되어야 한다
**And** 이후 장부 입력 대상 옵션에 사용할 수 있는 active 지점 목록에 포함되어야 한다.

**Given** 본사 사용자가 기존 지점명을 수정할 때
**When** 변경 내용을 저장하면
**Then** 새 지점명이 지점 목록에 표시되어야 한다
**And** 과거 장부와 연결된 지점 참조는 끊기지 않아야 한다.

**Then** 해당 지점은 신규 장부 입력 대상에서 제외될 수 있어야 한다
**And** 과거 장부 조회를 위해 지점 데이터는 삭제되지 않아야 한다.

**When** 생성, 이름 변경, 활성 상태 변경이 완료되면

**Given** 지점 관리 폼에 잘못된 값이 입력될 때
**When** 저장을 시도하면

## Epic 2: 지점 일일 장부 입력과 검토 제출

**When** 숫자 입력 필드가 표시되면

### Story 2.3: 지점장이 매입 품목을 기준정보에서 선택해 입력한다

As a 지점장,
I want 매입 품목을 기준정보에서 선택해 수량과 단가를 입력하고 싶다,
So that 매입 내역이 장부와 이후 재고 흐름에 반영될 수 있다.

**Acceptance Criteria:**

**Given** 지점장이 입력중 장부의 매입 단계에 접근할 때
**When** 화면이 열리면
**Then** active 품목 목록과 active 매입 기준을 선택지로 사용할 수 있어야 한다
**And** 비활성 품목이나 비활성 매입 기준은 신규 입력 선택지에서 제외되어야 한다.

**Given** 지점장이 매입 품목을 선택할 때
**When** 품목을 고르면
**Then** 품목명, 구분, 규격, 기본 단가 또는 매입 기준 참조 정보가 입력 보조값으로 표시되어야 한다
**And** 실제 장부 단가와 수량은 사용자가 장부 상황에 맞게 입력하거나 조정할 수 있어야 한다.

**Given** 지점장이 품목, 규격, 단가, 수량을 입력할 때
**When** 매입 항목을 저장하면
**Then** 매입금액이 계산되어 장부에 저장되어야 한다
**And** 하나의 장부에 여러 매입 항목을 저장할 수 있어야 한다.

**Given** 기존 매입 항목이 저장되어 있을 때
**When** 매입 단계가 다시 열리면
**Then** 저장된 항목과 매입 합계가 표시되어야 한다.

**Given** 품목 마스터가 이후 변경되었을 때
**When** 과거 장부의 매입 항목을 조회하면
**Then** 장부에 저장된 원본 품목명, 규격, 단가 값이 유지되어야 한다.

**Given** 단가나 수량에 잘못된 값이 입력될 때
**When** 저장을 시도하면
**Then** 필드 하단에 한국어 검증 메시지가 표시되어야 한다
**And** 첫 번째 오류 필드로 포커스가 이동해야 한다.

**Given** 390px 모바일 화면에서 매입 단계를 사용할 때
**When** 품목 검색과 수량/단가 입력을 수행하면
**Then** 터치 가능한 선택지와 숫자 입력 모드가 제공되어야 한다
**And** 저장 후 성공 또는 실패 피드백이 표시되어야 한다.

### Story 2.4: 지점장이 전일 이월 재고를 불러와 품목별 재고를 수정한다

As a 지점장,
I want 전일 이월 재고를 불러온 뒤 변경된 품목만 수정하고 싶다,
So that 30개 이상의 냉동/생물 품목 재고를 빠르게 입력할 수 있다.

**Acceptance Criteria:**

**Given** 지점장이 입력중 장부의 재고 단계에 접근할 때
**When** 해당 지점+일자의 전일재고 기준을 확인하면
**Then** 월 첫 장부는 월초 재고 스냅샷에서 전일재고를 가져와야 한다
**And** 이후 영업일은 직전 본사 마감 장부의 당일재고를 가져와야 한다.

**Given** 전일재고 기준을 불러올 수 있을 때
**When** 재고 단계가 열리면
**Then** 품목별 전일재고가 프리필되어야 한다
**And** "전일 재고를 불러왔습니다. 변경된 품목만 수정하세요." 안내가 표시되어야 한다.

**Given** 전일 장부가 마감되지 않아 자동 이월이 불가할 때
**When** 재고 단계가 열리면
**Then** 직접 입력하거나 본사에 문의하라는 한국어 안내가 표시되어야 한다
**And** 사용자는 수동으로 재고를 입력할 수 있어야 한다.

**Given** 지점장이 재고 품목을 볼 때
**When** 재고 테이블이 표시되면
**Then** 냉동/생물 탭으로 품목을 나눠 볼 수 있어야 한다
**And** 품목, 규격, 전일재고, 매입, 당일재고, 재고금액, 수량 컬럼이 제공되어야 한다.

**Given** 지점장이 품목의 당일재고 또는 수량 값을 수정할 때
**When** 값을 저장하면
**Then** 수정된 값이 장부에 저장되어야 한다
**And** 수정된 행은 시각적으로 강조되어야 하며 색상만으로 의미를 전달하면 안 된다.

**Given** 재고 품목이 30개 이상일 때
**When** 모바일 화면에서 재고를 입력하면
**Then** 가로 스크롤 또는 50행 단위 페이지/가상 목록 중 하나로 입력 가능해야 한다
**And** 주요 편집 셀은 44x44px 이상의 터치 대상을 가져야 한다.

**Given** 재고 수량이나 금액에 잘못된 값이 입력될 때
**When** 저장을 시도하면
**Then** 필드 하단 또는 행 안에 한국어 검증 메시지가 표시되어야 한다
**And** 오류가 있는 첫 번째 편집 위치로 이동해야 한다.

**Given** 재고금액이 계산될 때
**When** 수량과 단가가 모두 존재하면
**Then** 재고금액은 수량과 단가를 기준으로 표시되어야 한다
**And** 값이 없거나 계산 불가한 경우 숨은 오류 대신 "계산 불가" 상태를 표시해야 한다.

### Story 2.5: 지점장 또는 본사가 실제 재고 차이를 조정 사유와 함께 기록한다

As a 지점장 또는 본사 사용자,
I want 시스템 계산 재고와 실제 재고가 다를 때 조정 사유와 함께 기록하고 싶다,
So that 재고 차이를 숨기지 않고 장부와 리포트에 추적 가능한 값으로 반영할 수 있다.

**Acceptance Criteria:**

**Given** 입력중 또는 검토 대기 장부의 재고 단계에서 시스템 계산 재고와 실제 입력 재고가 다를 때
**When** 사용자가 재고 조정을 추가하면
**Then** 품목, 규격, 조정 전 수량/금액, 조정 후 수량/금액, 차이 수량/금액이 기록되어야 한다
**And** 작성자와 작성 시각이 함께 저장되어야 한다.

**Given** 사용자가 재고 조정을 저장하려 할 때
**When** 차이 사유가 비어 있으면
**Then** 저장이 차단되어야 한다
**And** 사유 필드에 한국어 검증 메시지가 표시되어야 한다.

**Given** 재고 조정이 저장될 때
**When** 장부 재고 흐름을 다시 계산하면
**Then** 조정값이 재고 흐름과 재고/손실 이상 후보에 반영되어야 한다.

**Given** 재고 조정이 저장된 장부를 다시 열 때
**When** 사용자가 해당 품목 행을 보면
**Then** 조정 전 값과 조정 후 값, 차이 사유를 확인할 수 있어야 한다
**And** 조정된 행은 색상만이 아니라 텍스트 또는 아이콘 레이블로도 구분되어야 한다.

**Given** 본사 마감 전 장부일 때
**When** 권한 있는 지점장 또는 본사 사용자가 조정을 추가하거나 수정하면
**Then** 원본 장부의 재고 조정 기록으로 저장되어야 한다.

**Given** 본사 마감된 장부일 때
**When** 사용자가 재고 조정을 시도하면
**Then** 이 스토리의 원본 재고 조정 기능으로는 저장되지 않아야 한다
**And** 마감 후 정정 기록 흐름을 사용해야 한다는 안내가 표시되어야 한다.

**Given** 재고 조정이 생성되거나 변경될 때
**When** 저장이 완료되면
**Then** actor, timestamp, target, before, after, reason을 포함하는 감사 기록이 남아야 한다.

### Story 2.6: 지점장이 손실/폐기/떨이를 입력하고 재고 흐름에 반영한다

As a 지점장,
I want 손실, 폐기, 떨이 항목을 품목별로 입력하고 싶다,
So that 실제 재고 감소와 손실 금액이 장부와 검토 요약에 반영될 수 있다.

**Acceptance Criteria:**

**Given** 지점장이 입력중 장부의 손실/폐기/떨이 단계에 접근할 때
**When** 화면이 열리면
**Then** active 품목 목록과 active 손실 유형 코드 목록을 선택지로 사용할 수 있어야 한다
**And** 비활성 품목이나 비활성 손실 유형은 신규 입력 선택지에서 제외되어야 한다.

**Given** 지점장이 손실 항목을 입력할 때
**When** 품목, 수량, 금액, 처리 유형, 사유/특이사항을 저장하면
**Then** 해당 손실 항목이 장부에 저장되어야 한다
**And** 하나의 장부에 여러 손실 항목을 저장할 수 있어야 한다.

**Given** 손실 항목이 저장될 때
**When** 재고 흐름을 다시 계산하면
**Then** 손실 수량과 금액이 품목별 재고 흐름에 반영되어야 한다
**And** 손실 합계는 검토 요약과 이후 손실 리포트에 사용할 수 있어야 한다.

**Given** 손실액 또는 손실 수량이 본사 기준값을 넘을 때
**When** 손실 단계 또는 검토 요약이 표시되면
**Then** 재고/손실 이상 후보로 표시될 수 있어야 한다
**And** 이 경고는 저장을 막지 않아야 한다.

**Given** 사유/특이사항이 필요한 처리 유형일 때
**When** 사용자가 사유 없이 저장을 시도하면
**Then** 저장이 차단되어야 한다
**And** 사유 필드에 한국어 검증 메시지가 표시되어야 한다.

**Given** 기존 손실 항목이 저장되어 있을 때
**When** 손실 단계가 다시 열리면
**Then** 저장된 손실 항목과 손실 합계가 표시되어야 한다.

**Given** 390px 모바일 화면에서 손실 단계를 사용할 때
**When** 품목 선택, 유형 선택, 수량/금액 입력을 수행하면
**Then** 터치 가능한 선택지와 숫자 입력 모드가 제공되어야 한다
**And** 저장 후 성공 또는 실패 피드백이 표시되어야 한다.

### Story 2.7: 입력자가 검토 화면에서 역할에 맞는 계산값과 이상 후보를 확인한다

As a 지점장 또는 본사 사용자,
I want 검토 화면에서 역할에 맞는 장부 계산값과 누락/이상 후보를 확인하고 싶다,
So that 제출 전에 장부의 주요 숫자와 문제 가능성을 한 번에 점검할 수 있다.

**Acceptance Criteria:**

**Given** 입력중 장부에 매출, 비용, 매입, 재고, 손실, 근무 정보가 일부 또는 전체 저장되어 있을 때
**When** 사용자가 검토/제출 단계에 접근하면
**Then** 본사 권한에는 총매출, 매출원가, 매출이익, 이익률, 영업이익, 인당생산성, 재고금액, 매출차액이 표시될 수 있다
**And** 지점장 권한에는 매출원가, 매출이익, 영업이익, 인당생산성이 서버 응답과 화면에 포함되면 안 된다
**And** 지점장 권한에는 현장 입력값, 허용된 매출 마진율/재고 금액, 누락 항목, 이상 후보, 확인 필요 상태를 표시한다
**And** 계산은 서버 측 공유 계산 함수와 역할별 응답 shaping을 함께 사용해야 한다.

**Given** 상품별 판매금액 합계 또는 판매/차감 값이 계산에 필요한 경우
**When** 해당 값이 아직 입력되지 않았거나 정책이 미정이면
**Then** 숨은 임의 계산 대신 "계산 기준 확인 필요" 또는 "계산 불가" 상태가 표시되어야 한다.

**Given** 총매출이 0이거나 비어 있을 때
**When** 이익률을 계산하면
**Then** 나누기 오류나 잘못된 무한대 값 대신 "계산 불가" 상태가 표시되어야 한다.

**Given** 근무인원이 0이거나 비어 있을 때
**When** 인당생산성을 계산하면
**Then** 나누기 오류 대신 "계산 불가" 상태가 표시되어야 한다.

**Given** 필수 입력값이 누락되어 있을 때
**When** 검토 요약이 표시되면
**Then** 누락 항목 목록이 표시되어야 한다
**And** 각 항목은 사용자가 해당 입력 단계로 돌아갈 수 있게 연결되어야 한다.

**Given** 결제수단 합계와 총매출이 다르거나 상품별 판매금액 합계와 총매출이 다를 때
**When** 검토 요약이 표시되면
**Then** 합계 불일치 경고가 표시되어야 한다
**And** 경고는 제출을 자동 차단하지 않아야 한다.

**Given** 매출차액, 재고 차이, 손실액 또는 손실 수량이 기준값을 넘을 때
**When** 검토 요약이 표시되면
**Then** 이상 후보가 아이콘과 레이블로 표시되어야 한다
**And** 색상만으로 의미를 전달하면 안 된다.

**Given** 사용자가 모바일 화면에서 검토 요약을 볼 때
**When** 계산값, 누락 항목, 이상 후보가 표시되면
**Then** 390px 폭에서도 내용이 겹치지 않고 읽을 수 있어야 한다.

### Story 2.8: 입력자가 장부를 검토 대기로 제출한다

As a 지점장 또는 본사 사용자,
I want 입력한 장부를 검토 대기로 제출하고 싶다,
So that 본사가 장부를 확인하고 필요한 경우 보완한 뒤 마감할 수 있다.

**Acceptance Criteria:**

**Given** 장부 상태가 `입력중`이고 사용자가 검토/제출 단계에 있을 때
**When** 사용자가 제출 버튼을 누르면
**Then** 장부 상태가 `검토대기`로 변경되어야 한다
**And** 제출자와 제출 시각이 기록되어야 한다.

**Given** 누락 항목이나 이상 후보가 있는 장부일 때
**When** 사용자가 제출을 시도하면
**Then** 누락과 이상 후보가 다시 표시되어야 한다
**And** 이 경고는 제출을 자동 차단하지 않아야 한다.

**Given** 장부가 `검토대기` 상태가 되었을 때
**When** 지점장 또는 권한 있는 본사 사용자가 장부를 다시 열면
**Then** 본사 마감 전까지 원본 장부를 수정할 수 있어야 한다
**And** `검토대기`는 원본 잠금으로 처리되면 안 된다.

**Given** 제출이 완료될 때
**When** 장부 상태가 변경되면
**Then** actor, timestamp, target, before, after를 포함하는 감사 기록이 남아야 한다
**And** 관련 관제판/장부 화면은 새 상태를 보여주도록 갱신되어야 한다.

**Given** 사용자가 이미 `검토대기` 상태인 장부에서 제출을 다시 시도할 때
**When** 제출 요청이 처리되면
**Then** 중복 제출 기록이 무의미하게 쌓이지 않아야 한다
**And** 사용자는 이미 검토 대기 상태라는 안내를 봐야 한다.

**Given** 제출 요청 중 서버 오류가 발생할 때
**When** 사용자가 제출 버튼을 누르면
**Then** 장부 상태는 기존 상태로 유지되어야 한다
**And** 한국어 오류 메시지와 재시도 동작이 제공되어야 한다.

**Given** 390px 모바일 화면에서 제출 결과가 표시될 때
**When** 제출이 성공하면
**Then** "장부를 제출했습니다." 피드백과 현재 상태 배지가 표시되어야 한다
**And** 상태 배지는 텍스트로 의미를 전달해야 한다.

### Story 2.9: 지점장 장부 입력/검토 서버 응답에서 민감 회계 지표를 차단한다

As a 본사 사용자,
I want 지점장 권한의 장부 입력/검토 응답에서 민감 회계 지표가 빠지길 원한다,
So that 이미 구현된 Epic 2 장부 흐름이 CAP-13 보안 요구와 충돌하지 않는다.

**Acceptance Criteria:**

**Given** 지점장 권한으로 비용/근무 단계와 검토/제출 단계를 조회할 때
**When** 서버가 응답을 만들면
**Then** 매출원가, 매출이익, 영업이익, 인당생산성은 응답에 포함되면 안 된다
**And** UI 숨김만으로 보안을 처리하면 안 된다.

**Given** 지점장 화면에 장부 요약이 표시될 때
**When** 비용/근무 또는 검토/제출 단계가 열리면
**Then** 현장 입력값, 비용 합계, 허용된 매출 마진율/재고 금액, 누락 항목, 이상 후보, 확인 필요 상태만 표시되어야 한다.

**Given** 본사 권한으로 같은 장부를 조회할 때
**When** 본사 전용 장부 상세 또는 검토 화면이 표시되면
**Then** 본사 운영에 필요한 회계 지표를 볼 수 있어야 한다.

**Given** 기존 Story 2.2와 Story 2.7 테스트를 보정할 때
**When** 지점장 권한 E2E와 서버 계약 테스트가 실행되면
**Then** 지점장 화면과 응답에서 민감 회계 지표가 보이지 않는 회귀 테스트가 포함되어야 한다.

**Given** 역할별 응답 shaping을 구현할 때
**When** 장부 입력/검토 query 또는 action result가 반환되면
**Then** 공통 응답 shaping helper 또는 동등한 서버-side mapper를 사용해야 한다
**And** 화면 컴포넌트가 민감 필드를 받은 뒤 숨기는 방식으로 처리하면 안 된다.

## Epic 3: 본사 관제판과 이상 신호 운영

본사가 아침 회의 전에 전체 지점 상태를 한 화면에서 보고, 매출/이익률/매출차액/재고/손실 이상 신호를 확인하며, 기준값을 운영 설정으로 조정할 수 있다.

### Story 3.1: 본사가 전체 지점 장부 상태를 관제판에서 본다

As a 본사 사용자,
I want 오늘 또는 어제 기준 전체 지점의 장부 상태를 한 화면에서 보고 싶다,
So that 아침 회의 전에 어떤 지점부터 확인할지 빠르게 판단할 수 있다.

**Acceptance Criteria:**

**Given** 본사 사용자가 관제판에 접근할 때
**When** 화면이 열리면
**Then** 오늘 날짜 기준 전체 활성 지점의 장부 상태 행이 표시되어야 한다
**And** 사용자는 오늘/어제 기준을 전환할 수 있어야 한다.

**Given** 관제판 행이 표시될 때
**When** 지점별 데이터가 로드되면
**Then** 각 행에는 영업 상태, 장부 상태, 매출, 이익률, 매출차액, 손실 여부, 마지막 수정자, 마지막 수정 시각, 본사 마감 여부가 표시되어야 한다.

**Given** 지점+일자 장부가 아직 없을 때
**When** 해당 지점 행이 표시되면
**Then** 장부 상태는 `미입력`으로 표시되어야 한다
**And** 휴무일 상태와 시각적으로 구분되어야 한다.

**Given** 장부 상태가 입력중, 검토대기, 본사마감, 휴무 중 하나일 때
**When** 상태 배지가 표시되면
**Then** 상태 텍스트가 명확히 표시되어야 한다
**And** 색상만으로 상태 의미를 전달하면 안 된다.

**Given** 관제판 데이터가 로딩 중일 때
**When** 화면이 데이터를 기다리면
**Then** 실제 테이블 행과 비슷한 Skeleton 상태가 표시되어야 한다.

**Given** 활성 지점이 없거나 표시할 데이터가 없을 때
**When** 관제판이 열리면
**Then** 빈 표만 보여주는 대신 간결한 한국어 빈 상태 메시지를 표시해야 한다.

**Given** 본사 관제판 기본 조회가 수행될 때
**When** 활성 지점이 10개 내외이면
**Then** 3초 안에 지점 목록과 기본 상태 정보가 표시되는 것을 목표로 해야 한다
**And** 상세 감사 이력이나 무거운 drilldown 데이터는 기본 조회에 포함하지 않아야 한다.

**Given** 390px 모바일 또는 좁은 화면에서 관제판을 볼 때
**When** 화면이 렌더링되면
**Then** 지점명, 상태, 이상 신호 요약, 매출 같은 핵심 정보가 겹치지 않게 표시되어야 한다
**And** 숨겨진 컬럼은 상세 화면에서 확인할 수 있어야 한다.

### Story 3.2: 본사가 이상 신호 기준값을 설정한다

As a 본사 사용자,
I want 이상 신호 기준값을 운영 상황에 맞게 설정하고 싶다,
So that 관제판이 본사 기준에 따라 문제 가능성이 높은 지점을 표시할 수 있다.

**Acceptance Criteria:**

**Given** 본사 사용자가 이상 신호 기준값 설정 화면에 접근할 때
**When** 화면이 열리면
**Then** 매출 하락률, 이익률 하락폭, 매출차액 금액, 손실액, 재고 차이 기준을 볼 수 있어야 한다
**And** 각 기준값에는 단위와 설명이 표시되어야 한다.

**Given** 매출차액 기준을 설정할 때
**When** 금액 기준을 입력하고 저장하면
**Then** 이후 관제판 조회는 변경된 금액 기준을 사용해야 한다
**And** 비율 기준이 필요하면 별도 Open Question으로 남겨야 하며 임의 구현하면 안 된다.

**Given** 매출/이익률 급락 기준을 설정할 때
**When** 매출 하락률 또는 이익률 하락폭을 저장하면
**Then** 이후 관제판 조회는 변경된 기준을 사용해야 한다
**And** 비교 기준일 정책이 미정이면 "기준일 정책 확인 필요"로 표시해야 한다.

**Given** 손실액 또는 재고 차이 기준을 설정할 때
**When** 기준값을 저장하면
**Then** 이후 재고/손실 이상 신호 계산에 반영되어야 한다.

**Given** 기준값에 잘못된 숫자나 허용 범위 밖 값이 입력될 때
**When** 저장을 시도하면
**Then** 필드 하단에 한국어 검증 메시지가 표시되어야 한다
**And** 잘못된 기준값은 저장되지 않아야 한다.

**Given** 기준값이 변경될 때
**When** 저장이 완료되면
**Then** actor, timestamp, target, before, after를 포함하는 감사 기록이 남아야 한다
**And** 다음 관제판 조회부터 변경된 기준이 적용되어야 한다.

**Given** 지점장 또는 권한 없는 사용자가 기준값 설정 화면에 접근할 때
**When** 서버가 요청을 처리하면
**Then** 기준값 데이터가 반환되지 않아야 한다
**And** 사용자는 "접근 권한이 없습니다." 화면을 봐야 한다.

### Story 3.3: 본사가 매출/이익률/매출차액 이상 신호를 본다

As a 본사 사용자,
I want 관제판에서 매출, 이익률, 매출차액 이상 신호를 확인하고 싶다,
So that 회의 전에 매출 흐름이나 계산 차이가 큰 지점을 먼저 볼 수 있다.

**Acceptance Criteria:**

**Given** 본사 사용자가 관제판을 열 때
**When** 장부의 매출, 이익률, 매출차액 데이터와 기준값이 존재하면
**Then** 매출 급락, 이익률 급락, 매출차액 이상 여부가 서버 측 공유 이상 신호 계산으로 평가되어야 한다
**And** UI 전용 계산으로 신호를 따로 만들면 안 된다.

**Given** 매출 하락률 또는 이익률 하락폭이 본사 기준값을 넘을 때
**When** 관제판 행이 표시되면
**Then** 해당 지점 행에 경고 등급 이상 신호 칩이 표시되어야 한다
**And** 칩에는 아이콘과 한국어 레이블이 함께 표시되어야 한다.

**Given** 매출차액 금액이 본사 기준값을 넘을 때
**When** 관제판 행이 표시되면
**Then** 매출차액 이상 신호 칩이 표시되어야 한다
**And** 칩 또는 상세에는 기준값과 실제 매출차액이 표시되어야 한다.

**Given** 한 지점에 매출 급락, 이익률 급락, 매출차액 이상이 동시에 발생할 때
**When** 관제판 행이 표시되면
**Then** 여러 신호 칩이 동시에 표시될 수 있어야 한다
**And** 행 강조는 가장 높은 심각도를 기준으로 적용되어야 한다.

**Given** 비교 기준일 데이터가 없거나 기준 정책이 미정일 때
**When** 매출/이익률 급락 신호를 평가하면
**Then** 임의로 정상 처리하지 않아야 한다
**And** "계산 기준 확인 필요" 또는 "계산 불가" 상태를 표시해야 한다.

**Given** 총매출이 0이거나 이익률 계산이 불가능한 장부일 때
**When** 이상 신호를 평가하면
**Then** 나누기 오류나 잘못된 무한대 값이 표시되면 안 된다
**And** 계산 불가 상태와 실제 입력 상태가 구분되어야 한다.

**Given** 사용자가 이상 신호가 있는 관제판 행을 선택할 때
**When** 행을 클릭하거나 키보드로 활성화하면
**Then** 해당 지점+일자의 장부 상세로 이동하거나 Sheet가 열려야 한다
**And** 상세에도 같은 이상 신호 요약이 표시되어야 한다.

### Story 3.4: 본사가 재고/손실 이상 신호를 본다

As a 본사 사용자,
I want 관제판에서 재고 차이와 손실 이상 신호를 확인하고 싶다,
So that 재고 흐름이나 손실이 비정상적인 지점을 먼저 점검할 수 있다.

**Acceptance Criteria:**

**Given** 본사 사용자가 관제판을 열 때
**When** 장부의 재고 조정, 실제 재고, 손실 데이터와 기준값이 존재하면
**Then** 재고 차이와 손실 이상 여부가 서버 측 공유 이상 신호 계산으로 평가되어야 한다
**And** UI 전용 계산으로 신호를 따로 만들면 안 된다.

**Given** 재고 차이 수량 또는 금액이 본사 기준값을 넘을 때
**When** 관제판 행이 표시되면
**Then** 재고 이상 신호 칩이 표시되어야 한다
**And** 칩 또는 상세에는 기준값과 실제 차이값이 표시되어야 한다.

**Given** 손실액 또는 손실 수량이 본사 기준값을 넘을 때
**When** 관제판 행이 표시되면
**Then** 손실 이상 신호 칩이 표시되어야 한다
**And** 칩에는 아이콘과 한국어 레이블이 함께 표시되어야 한다.

**Given** 재고/손실 신호가 심각 등급으로 분류될 때
**When** 관제판 행이 표시되면
**Then** destructive 스타일의 신호 칩과 행 강조가 적용되어야 한다
**And** 색상만으로 의미를 전달하면 안 된다.

**Given** 재고 또는 손실 데이터가 아직 미입력일 때
**When** 이상 신호를 평가하면
**Then** 미입력 상태와 정상 0 값을 구분해야 한다
**And** 미입력을 정상으로 취급해 조용히 숨기면 안 된다.

**Given** 재고 계산에 필요한 전일재고, 매입, 당일재고, 조정값 중 일부가 없을 때
**When** 재고 이상 신호를 평가하면
**Then** "계산 불가" 또는 "입력 필요" 상태가 표시되어야 한다
**And** 임의의 0으로 대체해 정상 처리하면 안 된다.

**Given** 한 지점에 재고 이상과 손실 이상이 동시에 발생할 때
**When** 관제판 행이 표시되면
**Then** 두 신호 칩이 모두 표시될 수 있어야 한다
**And** 행 선택 시 장부 상세에서 관련 재고/손실 근거를 확인할 수 있어야 한다.

### Story 3.5: 본사가 회의 전에 문제 지점을 빠르게 추적한다

As a 본사 사용자,
I want 관제판에서 문제 가능성이 높은 지점을 우선적으로 보고 상세로 빠르게 이동하고 싶다,
So that 아침 회의 전에 확인 순서를 정하고 필요한 장부를 먼저 열 수 있다.

**Acceptance Criteria:**

**Given** 관제판에 여러 지점 행이 표시될 때
**When** 이상 신호 또는 검토 대기/미마감 상태가 있는 지점이 존재하면
**Then** 문제 가능성이 높은 지점이 먼저 눈에 띄도록 정렬 또는 우선 표시되어야 한다
**And** 정렬 기준은 사용자가 이해할 수 있는 한국어 레이블로 표시되어야 한다.

**Given** 심각 등급과 경고 등급 신호가 함께 있을 때
**When** 관제판 행 우선순위를 정하면
**Then** 심각 등급 신호가 있는 지점이 경고 등급만 있는 지점보다 우선되어야 한다
**And** 같은 등급 안에서는 미마감/검토대기 상태를 구분할 수 있어야 한다.

**Given** 본사 사용자가 특정 지점 행을 선택할 때
**When** 데스크탑에서 행을 클릭하면
**Then** 장부 상세 Sheet 또는 상세 페이지로 이동해야 한다
**And** 사용자가 다시 관제판으로 돌아왔을 때 기존 날짜 기준과 필터/정렬 상태가 유지되어야 한다.

**Given** 본사 사용자가 태블릿이나 모바일 폭에서 관제판을 볼 때
**When** 지점 행을 선택하면
**Then** 상세 페이지로 이동할 수 있어야 한다
**And** 핵심 상태와 이상 신호 요약은 이동 전 목록에서도 확인 가능해야 한다.

**Given** 관제판 기본 조회가 수행될 때
**When** 활성 지점이 10개 내외이면
**Then** 상태, 계산 요약, 이상 신호, 우선순위 정보가 3초 안에 표시되는 것을 목표로 해야 한다
**And** 지점별 상세 감사 이력, 전체 재고 행, 정정 상세 같은 무거운 데이터는 기본 조회에 포함하지 않아야 한다.

**Given** 관제판 행이 키보드로 탐색될 때
**When** 사용자가 Tab, Enter, Space를 사용하면
**Then** 문제 지점 행 또는 명확한 상세 보기 컨트롤에 접근할 수 있어야 한다
**And** 포커스 표시가 보여야 한다.

**Given** 이상 신호가 없는 지점도 관제판에 표시될 때
**When** 사용자가 전체 지점 상태를 확인하면
**Then** 정상 또는 입력 상태를 구분해 볼 수 있어야 한다
**And** 이상 신호가 없다는 이유로 지점 행이 숨겨지면 안 된다.

## Epic 4: 본사 검토, 마감, 정정 기록

본사가 장부를 직접 보완하거나 수정하고, 최종 마감해 원본을 잠그며, 마감 후에는 원본을 보존한 채 정정 기록과 정정 반영값으로 운영 숫자를 바로잡을 수 있다.

### Story 4.1: 본사가 검토 대기 장부를 보완하고 수정한다

As a 본사 사용자,
I want 검토 대기 또는 입력중 장부를 직접 보완하고 수정하고 싶다,
So that 지점 데이터가 누락되거나 잘못되어도 본사가 회의 전에 장부를 정리할 수 있다.

**Acceptance Criteria:**

**Given** 본사 사용자가 입력중 또는 검토대기 장부 상세에 접근할 때
**When** 장부 상세가 열리면
**Then** 본사는 매출/결제, 비용, 매입, 재고, 재고 조정, 손실, 근무 정보를 조회하고 수정할 수 있어야 한다
**And** 서버는 본사 권한을 확인해야 한다.

**Given** 본사 사용자가 지점장 입력값을 보완하거나 수정할 때
**When** 변경 내용을 저장하면
**Then** 장부 값이 갱신되어야 한다
**And** 수정 주체는 본사 사용자로 기록되어야 한다.

**Given** 본사 수정이 저장될 때
**When** 저장이 완료되면
**Then** actor, timestamp, target, before, after를 포함하는 감사 기록이 남아야 한다
**And** 지점장 입력/수정 이력과 본사 입력/수정 이력이 구분되어야 한다.

**Given** 본사 수정 후 계산값에 영향이 있을 때
**When** 장부 상세 또는 관제판이 다시 조회되면
**Then** shared server calculation을 사용한 계산값과 이상 신호가 갱신되어야 한다.

**Given** 본사 사용자가 권한 없는 경로나 잘못된 장부에 접근할 때
**When** 서버가 요청을 처리하면
**Then** 장부 데이터가 반환되지 않아야 한다
**And** "접근 권한이 없습니다." 또는 찾을 수 없다는 한국어 안내가 표시되어야 한다.

**Given** 장부가 이미 본사마감 상태일 때
**When** 본사 사용자가 원본 입력 필드를 수정하려 하면
**Then** 원본 수정은 허용되지 않아야 한다
**And** 정정 기록을 추가해야 한다는 안내가 표시되어야 한다.

**Given** 본사 수정 폼에 잘못된 값이 입력될 때
**When** 저장을 시도하면
**Then** 필드 하단 또는 행 안에 한국어 검증 메시지가 표시되어야 한다
**And** 첫 번째 오류 위치로 포커스가 이동해야 한다.

### Story 4.2: 본사가 장부를 마감하고 원본을 잠근다

As a 본사 사용자,
I want 검토한 일일 장부를 본사 마감하고 원본을 잠그고 싶다,
So that 회의와 리포트 기준이 되는 원본 장부를 보존할 수 있다.

**Acceptance Criteria:**

**Given** 본사 사용자가 입력중 또는 검토대기 장부 상세에 접근할 때
**When** 마감 버튼을 선택하면
**Then** "장부를 마감합니다" 확인 Dialog가 표시되어야 한다
**And** 마감 후 지점장이 원본을 수정할 수 없고 정정 기록만 추가할 수 있다는 안내가 표시되어야 한다.

**Given** 본사 사용자가 마감 확인 Dialog에서 마감 확정을 선택할 때
**When** 서버가 요청을 처리하면
**Then** 장부 상태는 `본사마감`으로 변경되어야 한다
**And** `closedBy`, `closedAt`이 저장되어야 한다.

**Given** 장부가 `본사마감` 상태일 때
**When** 지점장 또는 본사 사용자가 원본 입력 저장 Server Action을 직접 호출해도
**Then** 서버에서 원본 수정이 거부되어야 한다
**And** UI disabled 상태만으로 잠금을 처리하면 안 된다.

**Given** 장부가 마감될 때
**When** 상태 변경과 감사 기록을 저장하면
**Then** 마감 상태 변경과 감사 기록은 같은 데이터베이스 트랜잭션 안에서 저장되어야 한다
**And** 감사 기록에는 actor, timestamp, target, before, after가 포함되어야 한다.

**Given** 이미 `본사마감` 상태인 장부에 마감 요청이 다시 들어올 때
**When** 서버가 요청을 처리하면
**Then** 중복 마감 기록이 무의미하게 쌓이지 않아야 한다
**And** 사용자는 이미 마감된 장부라는 안내를 봐야 한다.

**Given** 두 본사 사용자가 같은 장부를 거의 동시에 마감하려 할 때
**When** 첫 번째 요청이 성공한 뒤 두 번째 요청이 처리되면
**Then** 두 번째 요청은 현재 상태를 기준으로 안전하게 거부되거나 이미 마감됨으로 처리되어야 한다
**And** 원본 잠금 상태가 깨지면 안 된다.

**Given** 마감 Dialog가 열릴 때
**When** 키보드 사용자가 조작하면
**Then** Dialog에는 접근 가능한 제목, focus trap, 취소/확정 버튼, 닫힌 후 trigger 포커스 반환이 있어야 한다.

### Story 4.3: 본사가 마감된 장부에 정정 기록을 추가한다

As a 본사 사용자,
I want 마감된 장부에 원본을 바꾸지 않는 정정 기록을 추가하고 싶다,
So that 원본 감사 추적을 보존하면서 운영 숫자를 바로잡을 수 있다.

**Acceptance Criteria:**

**Given** 장부가 `본사마감` 상태일 때
**When** 본사 사용자가 정정 기록 추가를 선택하면
**Then** 정정 입력 영역이 열려야 한다
**And** 원본 입력 필드는 계속 disabled 상태여야 한다.

**Given** 본사 사용자가 정정 기록을 작성할 때
**When** 대상 항목, 원본값, 정정값, 사유를 입력하고 저장하면
**Then** append-only `CorrectionRecord`가 생성되어야 한다
**And** 원본 장부 값은 변경되면 안 된다.

**Given** 정정 대상 항목을 선택할 때
**When** 사용자가 정정 대상을 지정하면
**Then** 대상은 안정적인 식별자로 저장되어야 한다
**And** 대상 유형은 장부 필드, 결제/비용/매입/재고/손실 행, 계산 표시값 중 어떤 범위인지 구분되어야 한다.

**Given** 정정 기록이 저장될 때
**When** 원본값을 기록하면
**Then** 정정 당시의 원본값 또는 이전 반영값을 비교할 수 있도록 저장해야 한다
**And** 원본 장부 자체를 덮어쓰면 안 된다.

**Given** 같은 대상 항목에 여러 정정 기록이 있을 때
**When** 정정 반영값을 계산하면
**Then** 최신 정정 기록의 `정정값`을 해당 대상의 최종 반영값으로 사용해야 한다
**And** 이전 정정 기록은 삭제되지 않고 타임라인에 남아야 한다.

**Given** 정정 사유가 비어 있을 때
**When** 저장을 시도하면
**Then** 저장이 차단되어야 한다
**And** 사유 필드에 한국어 검증 메시지가 표시되어야 한다.

**Given** 지점장 또는 권한 없는 사용자가 정정 기록 추가를 시도할 때
**When** 서버가 요청을 처리하면
**Then** 정정 기록은 생성되지 않아야 한다
**And** 사용자는 "접근 권한이 없습니다." 화면 또는 권한 오류 안내를 봐야 한다.

**Given** 정정 기록이 생성될 때
**When** 저장이 완료되면
**Then** 정정 기록과 감사 기록은 같은 데이터베이스 트랜잭션 안에서 저장되어야 한다
**And** 감사 기록에는 actor, timestamp, target, before, after, reason이 포함되어야 한다.

### Story 4.4: 본사가 원본값과 정정 반영값을 구분해 본다

As a 본사 사용자,
I want 마감된 장부 상세에서 원본값과 정정 반영값을 구분해 보고 싶다,
So that 회의 숫자는 최신 운영값으로 보면서도 원본 감사 추적을 잃지 않을 수 있다.

**Acceptance Criteria:**

**Given** 본사 사용자가 정정 기록이 있는 마감 장부 상세를 열 때
**When** 대상 항목이 표시되면
**Then** 원본값과 정정 반영값이 구분되어 표시되어야 한다
**And** 정정 반영값에는 "정정 반영" 레이블이 표시되어야 한다.

**Given** 사용자가 정정 기록 패널을 열 때
**When** 정정 이력을 확인하면
**Then** 대상 항목, 원본값, 정정값, 사유, 작성자, 작성 시각이 시간순 또는 최신순 타임라인으로 표시되어야 한다.

**Given** 장부 상세가 Sheet로 열려 있는 상태일 때
**When** 사용자가 정정 기록 패널을 열면
**Then** Sheet 위에 또 다른 Sheet를 중첩하지 않아야 한다
**And** 상세 안 확장 영역, 하단 패널, 또는 별도 페이지 중 하나로 표시해야 한다.

**Given** 원본 입력 필드가 마감 상태에서 표시될 때
**When** 사용자가 장부 상세를 보면
**Then** 원본 입력 필드는 disabled 상태여야 한다
**And** 정정 기록 추가가 가능한 항목에는 명확한 정정 액션이 제공되어야 한다.

**Given** 같은 대상 항목에 여러 정정 기록이 있을 때
**When** 사용자가 상세를 확인하면
**Then** 최신 정정값이 현재 정정 반영값으로 표시되어야 한다
**And** 이전 정정값도 타임라인에서 확인할 수 있어야 한다.

**Given** 정정 반영값과 원본값이 모두 표시될 때
**When** 스크린 리더나 키보드 사용자가 탐색하면
**Then** 두 값의 의미가 텍스트 레이블로 구분되어야 한다
**And** 색상이나 취소선만으로 차이를 전달하면 안 된다.

**Given** 정정 기록이 없는 마감 장부를 볼 때
**When** 정정 패널을 열면
**Then** 정정 기록이 없다는 빈 상태 메시지를 표시해야 한다
**And** 본사 사용자는 새 정정 기록 추가 동작을 시작할 수 있어야 한다.

### Story 4.5: 관제판이 정정 반영값을 기본으로 사용한다

As a 본사 사용자,
I want 정정 기록을 추가한 뒤 관제판 숫자와 이상 신호가 정정 반영값으로 갱신되길 원한다,
So that 리포트 화면이 없어도 본사 운영 판단에 최신 보정 숫자를 사용할 수 있다.

**Acceptance Criteria:**

**Given** 마감 장부에 정정 기록이 추가되어 있을 때
**When** 관제판 행이 표시되면
**Then** 매출, 이익률, 매출차액, 손실 여부, 이상 신호 계산은 정정 반영값을 기본으로 사용해야 한다
**And** 원본값이 필요한 경우 상세에서 구분해 확인할 수 있어야 한다.

**Given** 정정 기록이 새로 저장될 때
**When** 저장이 완료되면
**Then** 관련 장부 상세와 관제판 경로가 갱신되어야 한다
**And** 사용자가 낡은 계산값을 계속 보지 않도록 revalidation 또는 동등한 갱신 처리가 수행되어야 한다.

**Given** 정정 반영값을 계산할 때
**When** 같은 대상에 여러 정정 기록이 있으면
**Then** 최신 정정 기록의 `정정값`을 최종 반영값으로 사용해야 한다
**And** 이전 정정 기록은 계산에서 제외되더라도 이력에서는 보존되어야 한다.

**Given** 정정 기록 대상이 계산 지표에 영향을 줄 때
**When** 매출이익, 이익률, 영업이익, 매출차액, 이상 신호를 계산하면
**Then** shared server calculation이 원본값과 정정 반영값을 명시적으로 구분해 계산해야 한다
**And** UI에서 별도 보정 계산을 만들면 안 된다.

**Given** 정정 대상이나 정정값 형식이 계산에 적용할 수 없을 때
**When** 관제판이 계산을 수행하면
**Then** 임의로 원본값을 정상값처럼 사용하지 않아야 한다
**And** "정정 확인 필요" 또는 계산 불가 상태를 표시해야 한다.

**Given** 지점장이 마감 장부를 조회할 때
**When** 정정 반영값이 있는 항목을 보면
**Then** 원본 수정은 할 수 없어야 한다
**And** 정정 반영값과 본사 정정 이력을 읽기 전용으로 확인할 수 있어야 한다.

## Epic 5: 회의와 기간/월간 리포트

본사가 일별 아침 회의 리포트, 지점별 기간 비교, 월간 지점 요약을 보고 마감 상태, 이상 신호, 실적, 손실/재고 흐름을 운영 회의 기준으로 확인할 수 있다.

### Story 5.1: 본사가 일별 아침 회의 리포트를 본다

As a 본사 사용자,
I want 특정 일자의 전체 지점 회의용 요약을 보고 싶다,
So that 아침 회의에서 마감 상태와 이상 항목을 기준으로 지점별 이야기를 진행할 수 있다.

**Acceptance Criteria:**

**Given** 본사 사용자가 아침 회의 리포트 화면에 접근할 때
**When** 날짜를 선택하면
**Then** 해당 일자의 전체 지점 요약이 표시되어야 한다
**And** 기본 날짜는 오늘 또는 운영 기준일로 설정되어야 한다.

**Given** 일별 회의 리포트가 표시될 때
**When** 지점별 행이 로드되면
**Then** 지점별 마감 상태, 이상 신호, 매출, 이익률, 매출차액, 손실 현황이 표시되어야 한다.

**Given** 장부에 정정 기록이 있을 때
**When** 리포트 숫자가 계산되면
**Then** 기본 숫자는 정정 반영값을 사용해야 한다
**And** 원본값과 정정 반영값 차이가 있으면 상세 확인 경로가 제공되어야 한다.

**Given** 장부가 미입력, 입력중, 검토대기, 본사마감, 휴무일 중 하나일 때
**When** 리포트 행이 표시되면
**Then** 상태가 명확한 텍스트 배지로 표시되어야 한다
**And** 미입력과 휴무일은 구분되어야 한다.

**Given** 이상 신호가 있는 지점이 있을 때
**When** 리포트 행이 표시되면
**Then** 이상 신호는 아이콘과 레이블로 표시되어야 한다
**And** 색상만으로 의미를 전달하면 안 된다.

**Given** 리포트 계산이 수행될 때
**When** 매출, 이익률, 매출차액, 손실 현황을 계산하면
**Then** 관제판/장부 상세와 같은 shared server calculation을 사용해야 한다
**And** UI 전용 계산식을 만들면 안 된다.

**Given** 리포트 데이터가 로딩 중일 때
**When** 화면이 데이터를 기다리면
**Then** 실제 표와 유사한 Skeleton 상태가 표시되어야 한다.

**Given** 선택한 날짜에 표시할 지점 데이터가 없을 때
**When** 리포트가 열리면
**Then** 간결한 한국어 빈 상태 메시지가 표시되어야 한다.

### Story 5.2: 본사가 리포트 숫자의 근거와 정정 차이를 추적한다

As a 본사 사용자,
I want 리포트 숫자의 원본값, 정정 반영값, 계산 상태를 확인하고 싶다,
So that 회의에서 사용하는 숫자를 신뢰하고 필요하면 근거를 추적할 수 있다.

**Acceptance Criteria:**

**Given** 리포트 숫자에 정정 기록이 반영되어 있을 때
**When** 본사 사용자가 해당 숫자의 상세를 열면
**Then** 원본값, 정정 반영값, 정정 여부가 구분되어 표시되어야 한다
**And** 정정 반영값에는 "정정 반영" 레이블이 표시되어야 한다.

**Given** 리포트 숫자가 계산 불가 상태일 때
**When** 본사 사용자가 상세를 열면
**Then** 계산 불가 사유가 한국어로 표시되어야 한다
**And** 미입력, 휴무, 0, 데이터 부족 상태가 구분되어야 한다.

**Given** 리포트 숫자가 장부 상세 데이터에서 나온 경우
**When** 본사 사용자가 근거 확인 경로를 선택하면
**Then** 해당 지점+일자의 장부 상세 또는 정정 타임라인으로 이동할 수 있어야 한다.

**Given** 같은 항목에 여러 정정 기록이 있을 때
**When** 리포트 상세가 표시되면
**Then** 최신 정정값이 현재 정정 반영값으로 표시되어야 한다
**And** 이전 정정 기록은 타임라인에서 확인할 수 있어야 한다.

**Given** 리포트 계산이 수행될 때
**When** 원본값과 정정 반영값을 비교하면
**Then** shared server calculation이 두 값을 명시적으로 구분해 제공해야 한다
**And** UI에서 별도 보정 계산을 만들면 안 된다.

**Given** 좁은 화면에서 숫자 근거 상세를 볼 때
**When** 원본값과 정정 반영값이 함께 표시되면
**Then** 두 값이 겹치지 않고 텍스트 레이블로 구분되어야 한다
**And** 색상이나 취소선만으로 의미를 전달하면 안 된다.

### Story 5.3: 본사가 선택 기간의 지점별 실적을 비교한다

As a 본사 사용자,
I want 선택 기간의 지점별 실적을 비교하고 싶다,
So that 어느 지점의 매출, 이익, 생산성, 재고 흐름이 좋은지 운영 기준으로 판단할 수 있다.

**Acceptance Criteria:**

**Given** 본사 사용자가 기간 비교 리포트 화면에 접근할 때
**When** 시작일과 종료일을 선택하면
**Then** 선택 기간의 지점별 실적 비교 표가 표시되어야 한다
**And** 기간 필터는 URL 상태로 유지되어 새로고침해도 같은 조건을 볼 수 있어야 한다.

**Given** 기간 비교 표가 표시될 때
**When** 지점별 데이터가 로드되면
**Then** 매출, 매출이익, 이익률, 영업이익, 인당생산성, 평균재고, 평균매출, 매출대비 재고비율, 손실이 표시되어야 한다.

**Given** 선택 기간에 정정 기록이 있는 장부가 포함될 때
**When** 기간 비교 지표가 계산되면
**Then** 기본 숫자는 정정 반영값을 사용해야 한다
**And** 원본값과 정정 반영값 차이가 있는 지표는 Story 5.2의 상세 확인 경로를 제공해야 한다.

**Given** 선택 기간에 미입력, 입력중, 검토대기, 본사마감, 휴무일 데이터가 섞여 있을 때
**When** 지표가 집계되면
**Then** 각 상태는 구분되어 집계 또는 표시되어야 한다
**And** 미입력과 휴무일을 같은 값으로 취급하면 안 된다.

**Given** 평균재고, 평균매출, 재고비율을 계산할 때
**When** 필요한 일별 데이터가 부족하거나 평균매출이 0이면
**Then** 나누기 오류 대신 "계산 불가" 또는 "데이터 부족" 상태가 표시되어야 한다.

**Given** 기간 비교 계산이 수행될 때
**When** 리포트 지표를 계산하면
**Then** shared server calculation을 사용해야 한다
**And** 관제판이나 월간 리포트와 다른 UI 전용 계산식을 만들면 안 된다.

**Given** 본사 사용자가 비교 표를 볼 때
**When** 숫자 컬럼이 표시되면
**Then** 금액은 원 단위와 천 단위 콤마로, 이익률은 소수 첫째 자리까지 표시되어야 한다
**And** 숫자 데이터는 tabular-nums로 정렬되어야 한다.

**Given** 좁은 화면에서 기간 비교 리포트를 볼 때
**When** 전체 컬럼을 표시하기 어렵다면
**Then** 핵심 컬럼을 우선 표시하고 나머지는 가로 스크롤 또는 상세로 확인할 수 있어야 한다.

### Story 5.4: 본사가 월간 지점별 마감과 이상 현황을 본다

As a 본사 사용자,
I want 월과 지점을 선택해 마감 현황과 주요 이상 항목을 보고 싶다,
So that 월간 운영 점검에서 누락 장부와 반복 문제를 확인할 수 있다.

**Acceptance Criteria:**

**Given** 본사 사용자가 월간 요약 리포트 화면에 접근할 때
**When** 월과 지점을 선택하면
**Then** 선택한 지점의 월간 마감 현황과 주요 이상 항목이 표시되어야 한다
**And** 월/지점 필터는 URL 상태로 유지되어야 한다.

**Given** 월간 마감 현황이 표시될 때
**When** 해당 월의 일별 장부 상태를 집계하면
**Then** 본사마감, 검토대기, 입력중, 미입력, 휴무일이 구분되어 표시되어야 한다
**And** 마감되지 않은 장부가 있으면 명확히 구분되어야 한다.

**Given** 선택한 월에 이상 신호가 있는 장부가 있을 때
**When** 월간 주요 이상 항목이 표시되면
**Then** 매출/이익률 급락, 매출차액, 재고/손실 이상 항목이 날짜와 지점 맥락으로 표시되어야 한다
**And** 이상 신호는 아이콘과 텍스트 레이블로 표시되어야 한다.

**Given** 월간 이상 항목을 계산할 때
**When** 장부에 정정 기록이 포함되어 있으면
**Then** 기본 숫자는 정정 반영값을 사용해야 한다
**And** 원본값과 정정 반영값 차이가 있는 항목은 Story 5.2의 상세 확인 경로를 제공해야 한다.

**Given** 월간 상태 또는 이상 항목이 표시될 때
**When** 사용자가 특정 날짜를 선택하면
**Then** 해당 일자의 장부 상세로 이동할 수 있어야 한다.

**Given** 선택한 월에 장부 데이터가 없을 때
**When** 월간 요약 화면이 열리면
**Then** 간결한 한국어 빈 상태 메시지가 표시되어야 한다.

**Given** 월간 리포트 계산이 수행될 때
**When** 마감 현황과 이상 항목을 계산하면
**Then** 관제판/장부 상세와 같은 shared server calculation과 이상 신호 평가를 사용해야 한다
**And** UI 전용 계산식을 만들면 안 된다.

### Story 5.5: 본사가 월간 손실/재고 흐름과 핵심 성과를 본다

As a 본사 사용자,
I want 월간 손실/재고 흐름과 핵심 성과 지표를 보고 싶다,
So that 한 지점의 월간 운영 품질과 재고 흐름을 요약해서 판단할 수 있다.

**Acceptance Criteria:**

**Given** 본사 사용자가 월간 요약 리포트에서 월과 지점을 선택할 때
**When** 데이터가 로드되면
**Then** 월간 매출, 매출이익, 이익률, 영업이익, 손실 합계, 평균재고, 평균매출, 매출대비 재고비율이 표시되어야 한다.

**Given** 월간 손실/재고 흐름이 표시될 때
**When** 해당 월의 장부 데이터를 집계하면
**Then** 손실/폐기/떨이 합계와 재고 흐름 요약이 표시되어야 한다
**And** 손실 유형별 구분이 가능한 경우 유형별 요약을 볼 수 있어야 한다.

**Given** 월간 최고매출품목을 계산할 때
**When** 상품별 판매금액 데이터가 충분히 존재하면
**Then** 최고매출품목과 최고매출품목 매출액이 표시되어야 한다
**And** 동률이 있으면 일관된 tie-breaking 규칙 또는 "공동" 표시를 사용해야 한다.

**Given** 최고매출품목 계산에 필요한 상품별 판매금액 산출 기준이 미정이거나 데이터가 부족할 때
**When** 월간 요약이 표시되면
**Then** 임의로 품목을 선택하지 않아야 한다
**And** "계산 기준 확인 필요" 또는 "데이터 부족" 상태를 표시해야 한다.

**Given** 월간 평균재고, 평균매출, 매출대비 재고비율을 계산할 때
**When** 마감되지 않은 장부, 미입력, 휴무일이 포함되어 있으면
**Then** 어떤 일자가 계산에 포함 또는 제외됐는지 확인할 수 있어야 한다
**And** 평균매출이 0이면 나누기 오류 대신 "계산 불가" 상태를 표시해야 한다.

**Given** 월간 지표에 정정 기록이 반영될 때
**When** 월간 요약 숫자가 계산되면
**Then** 기본 숫자는 정정 반영값을 사용해야 한다
**And** 원본값과 정정 반영값 차이가 있는 항목은 Story 5.2의 상세 확인 경로를 제공해야 한다.

**Given** 월간 리포트 계산이 수행될 때
**When** 손실, 재고, 최고매출품목, 평균 지표를 계산하면
**Then** shared server calculation을 사용해야 한다
**And** 기간 비교 리포트와 다른 UI 전용 계산식을 만들면 안 된다.

**Given** 월간 요약을 좁은 화면에서 볼 때
**When** 핵심 성과와 손실/재고 흐름이 표시되면
**Then** 주요 숫자는 겹치지 않고 읽을 수 있어야 한다
**And** 세부 표는 가로 스크롤 또는 상세 영역으로 확인할 수 있어야 한다.

## Epic 6: 품목 정규화와 매입/FIFO 재고 기반

본사가 원문 품목명과 분석용 품목/규격을 분리하고, 이카운트 엑셀 업로드 매입을 장부에 자동 반영하며, FIFO 재고 금액과 매입 잔량 근거를 추적할 수 있다.

### Story 6.1: 본사가 품목명/규격 별칭과 매핑 실패를 관리한다

As a 본사 사용자,
I want 원문 품목명과 분석용 품목/규격 매핑을 관리하고 싶다,
So that 다르게 입력된 같은 상품을 하나로 묶어 분석할 수 있다.

**Acceptance Criteria:**

**Given** 본사 사용자가 품목 정규화 화면에 접근할 때
**When** 화면이 열리면
**Then** 원문 품목명, 분석용 품목, 규격, 별칭, 매핑 상태가 표시되어야 한다
**And** 매핑 실패 항목만 필터링할 수 있어야 한다.

**Given** 본사 사용자가 원문 품목명을 분석용 품목/규격에 연결할 때
**When** 매핑을 저장하면
**Then** 이후 업로드와 리포트 계산은 분석용 품목/규격을 사용할 수 있어야 한다
**And** 과거 장부의 원문 품목명은 덮어쓰면 안 된다.

**Given** 품목 매핑이 생성되거나 변경될 때
**When** 저장이 완료되면
**Then** actor, timestamp, target, before, after, reason 또는 memo를 포함하는 감사 기록이 남아야 한다.

**Given** 매핑할 수 없는 원문 품목이 있을 때
**When** 업로드나 장부 계산이 실행되면
**Then** 해당 항목은 조용히 제외되지 않아야 한다
**And** `매핑 실패` 또는 `확인 필요` 상태로 본사 검수 목록에 남아야 한다.

### Story 6.2: 본사가 이카운트 엑셀 업로드로 매입 라인을 자동 생성한다

As a 본사 사용자,
I want 이카운트 엑셀을 업로드해 장부 매입 데이터를 자동 생성하고 싶다,
So that 지점 입력 누락과 오기입을 줄일 수 있다.

**Acceptance Criteria:**

**Given** 본사 사용자가 이카운트 업로드 화면에 접근할 때
**When** 엑셀 파일을 선택하고 업로드하면
**Then** 파일 형식, 필수 컬럼, 날짜, 지점, 품목명, 규격, 단가, 수량이 검증되어야 한다
**And** 검증 실패 사유는 한국어로 표시되어야 한다.

**Given** 업로드 파일의 품목명이 매핑되어 있을 때
**When** 업로드를 확정하면
**Then** 해당 지점/일자 장부에 매입 라인이 자동 생성되어야 한다
**And** 본사 업로드 출처, 업로드 배치 ID, 원문 파일명 또는 trace가 저장되어야 한다.

**Given** 업로드 파일에 매핑 실패 품목이 있을 때
**When** 본사 사용자가 업로드 결과를 확인하면
**Then** 실패 행과 사유가 표시되어야 한다
**And** 매핑 완료 전에는 해당 행을 정상 매입처럼 반영하면 안 된다.

**Given** 이카운트 업로드로 생성된 매입 라인이 지점장 화면에 표시될 때
**When** 지점장이 매입 단계를 열면
**Then** 해당 매입은 읽기 전용으로 표시되어야 한다
**And** 지점장은 본사 업로드 매입을 임의 수정할 수 없어야 한다.

**Given** 업로드가 확정되거나 취소될 때
**When** 상태가 변경되면
**Then** 업로드 이력과 감사 로그가 남아야 한다
**And** 장부 계산과 관제판 revalidation이 수행되어야 한다.

### Story 6.3: 시스템이 매입 라인별 잔량과 FIFO 재고 금액을 계산한다

As a 본사 사용자,
I want 매입 라인별 잔여 수량과 FIFO 재고 금액을 추적하고 싶다,
So that 같은 품목의 단가가 바뀌어도 재고 금액 근거를 확인할 수 있다.

**Acceptance Criteria:**

**Given** 같은 품목/규격의 매입 단가가 여러 번 바뀌었을 때
**When** 재고 금액을 계산하면
**Then** 매입일, 단가, 수량, 잔량 기준으로 FIFO 원가가 계산되어야 한다
**And** 계산은 shared server calculation에서 수행되어야 한다.

**Given** 판매/차감, 손실, 재고 조정이 저장될 때
**When** 재고 흐름을 다시 계산하면
**Then** 관련 매입 라인의 잔여 수량이 추적되어야 한다
**And** 원본 매입 라인은 감사 추적을 위해 보존되어야 한다.

**Given** FIFO 계산에 필요한 매핑, 수량, 단가, 이월 기준이 부족할 때
**When** 장부 상세, 관제판, 리포트가 계산을 요청하면
**Then** 임의로 0원 또는 기존 단가를 정상값처럼 사용하면 안 된다
**And** `확인 필요` 또는 `계산 불가` 상태를 표시해야 한다.

**Given** FIFO 계산 결과가 리포트 지표에 사용될 때
**When** 매출원가, 매출이익, 이익률, 영업이익을 계산하면
**Then** 원본값과 정정 반영값을 구분해야 한다
**And** UI 전용 계산식을 만들면 안 된다.

### Story 6.4: 본사가 재고 금액의 매입 잔량 근거를 확인한다

As a 본사 사용자,
I want 전일 재고 또는 재고 금액의 매입 잔량 근거를 보고 싶다,
So that 재고 금액이 어떤 매입 단가와 잔량에서 나온 값인지 설명할 수 있다.

**Acceptance Criteria:**

**Given** 본사 사용자가 재고 금액이나 전일 재고 값을 선택할 때
**When** 근거 확인 패널을 열면
**Then** 기본 1개월 범위의 매입일, 단가, 매입 수량, 잔여 수량, 반영 금액이 표시되어야 한다
**And** 기간 필터를 바꿔 조회할 수 있어야 한다.

**Given** 조회 대상 품목에 여러 지점 또는 여러 매입 배치가 섞여 있을 때
**When** 근거 목록이 표시되면
**Then** 지점, 장부일, 업로드/수기 출처, 매핑 상태가 구분되어야 한다.

**Given** 근거 데이터를 만들 수 없는 항목이 있을 때
**When** 사용자가 상세를 확인하면
**Then** 누락 원인이 한국어로 표시되어야 한다
**And** 매핑 실패, 데이터 부족, 정정 확인 필요 상태가 구분되어야 한다.

## Epic 7: 본사 통합 재고와 상품별 관리자 분석

본사가 전체 지점 품목별 재고를 한 화면에서 보고, 품목규격 기준 판매/원가/마진/최고매출품목을 분석할 수 있다.

### Story 7.1: 본사가 전체 지점 품목별 재고를 한 화면에서 본다

As a 본사 사용자,
I want 전체 지점의 품목별 재고와 합계를 한 화면에서 보고 싶다,
So that 본사 발주 판단에 필요한 재고 현황을 빠르게 확인할 수 있다.

**Acceptance Criteria:**

**Given** 본사 사용자가 전체 재고 화면에 접근할 때
**When** 품목별 재고 데이터가 로드되면
**Then** 품목/규격별 전체 합계와 지점별 잔량이 표시되어야 한다
**And** 본사 사용자만 이 화면에 접근할 수 있어야 한다.

**Given** 본사 사용자가 냉동/생물, 품목, 지점, 기간 필터를 변경할 때
**When** 조회 조건이 바뀌면
**Then** 재고 합계와 지점별 잔량이 같은 조건으로 다시 계산되어야 한다
**And** 필터 상태는 URL 또는 동등한 공유 가능한 상태로 유지되어야 한다.

**Given** 재고 계산에 매핑 실패나 FIFO 계산 불가 항목이 포함될 때
**When** 재고 뷰가 표시되면
**Then** 해당 품목은 `확인 필요`로 표시되어야 한다
**And** 사용자가 근거 확인 화면으로 이동할 수 있어야 한다.

### Story 7.2: 본사가 상품별 판매/원가/마진 분석을 본다

As a 본사 사용자,
I want 품목규격 기준 상품별 분석을 보고 싶다,
So that 최고매출품목과 상품별 이익률을 운영 판단에 사용할 수 있다.

**Acceptance Criteria:**

**Given** 본사 사용자가 상품 분석 화면에 접근할 때
**When** 장부 또는 기간을 선택하면
**Then** 품목규격별 전재고, 매입, 판매원가, 이익률, 판매금액, 판매량이 표시되어야 한다.

**Given** 상품별 판매금액 데이터가 충분할 때
**When** 분석이 계산되면
**Then** 최고매출품목과 최고매출액이 자동 산출되어야 한다
**And** 동률이면 일관된 tie-breaking 규칙 또는 `공동` 표시를 사용해야 한다.

**Given** 상품별 계산 기준이 불확실하거나 데이터가 부족할 때
**When** 분석 화면이 표시되면
**Then** 임의 계산값을 정상값처럼 표시하면 안 된다
**And** `기준 확인 필요` 또는 `데이터 부족` 상태를 표시해야 한다.

**Given** 상품 분석이 관제판 또는 월간 리포트와 같은 지표를 사용할 때
**When** 계산이 수행되면
**Then** shared server calculation을 사용해야 한다
**And** UI마다 다른 계산식을 만들면 안 된다.

### Story 7.3: 본사가 상품 분석과 통합 재고의 근거를 추적한다

As a 본사 사용자,
I want 상품 분석 숫자의 원본과 계산 근거를 확인하고 싶다,
So that 회의에서 숫자를 설명하고 필요한 항목을 정정할 수 있다.

**Acceptance Criteria:**

**Given** 본사 사용자가 상품 분석 숫자를 선택할 때
**When** 근거 상세가 열리면
**Then** 장부, 매입 라인, 재고 스냅샷, 손실/조정, 정정 반영 여부가 구분되어야 한다.

**Given** 분석 숫자에 정정 기록이 반영되었을 때
**When** 근거 상세가 표시되면
**Then** 원본값과 정정 반영값이 텍스트 레이블로 구분되어야 한다
**And** 색상만으로 차이를 전달하면 안 된다.

**Given** 상품 분석 데이터가 없는 조건을 조회할 때
**When** 화면이 열리면
**Then** 간결한 한국어 빈 상태와 다음 행동이 표시되어야 한다.

## Epic 8: 지점장 권한, 마감, 손실, 입력 흐름 보완

지점장에게 민감 회계 지표를 노출하지 않고, 본사 일괄 마감, 작성자명 유지, 손실액 정책, 재고 테이블 문구, 대시보드 리사이징을 운영 수준으로 보완한다.

### Story 8.1: 지점장 민감 회계 지표 제한을 추가 범위 화면까지 검증/강화한다

As a 본사 사용자,
I want 추가 범위 화면에서도 지점장 권한의 민감 회계 지표 제한이 유지되길 원한다,
So that 새 기능이 늘어나도 지점 직원에게 제한된 회계 정보가 노출되지 않는다.

**Acceptance Criteria:**

**Given** 지점장 권한으로 추가 범위 화면의 장부 상세, 관제판, 리포트성 요약을 조회할 때
**When** 서버가 응답을 만들면
**Then** 매출원가, 매출이익, 인당생산성, 영업이익은 응답에 포함되면 안 된다
**And** UI 숨김만으로 보안을 처리하면 안 된다.

**Given** 본사 권한으로 같은 데이터를 조회할 때
**When** 서버가 응답을 만들면
**Then** 본사 운영에 필요한 회계 지표는 표시될 수 있어야 한다.

**Given** 지점장 화면에서 추가 범위의 운영 숫자를 표시할 때
**When** 근무자 선택, 매입 업로드 확인, 재고, 손실, 리포트성 요약 화면이 열리면
**Then** 현장 입력값, 매출 마진율, 재고 금액 중심으로 표시되어야 한다
**And** 지점장에게 보여줄 마진율 기준이 불확실하면 `기준 확인 필요`로 남겨야 한다.

**Given** Story 2.9에서 만든 민감 지표 응답 shaping 기준이 있을 때
**When** 추가 범위 query와 화면을 구현하면
**Then** 같은 기준을 재사용하거나 확장해야 한다
**And** 별도 화면마다 one-off 필터링을 만들면 안 된다.

### Story 8.2: 본사가 일괄 마감 또는 검토 생략을 실행한다

As a 본사 사용자,
I want 여러 장부를 개별 검토 없이 일괄 마감할 수 있는 옵션을 갖고 싶다,
So that 본사 판단에 따라 마감 업무를 빠르게 처리할 수 있다.

**Acceptance Criteria:**

**Given** 본사 사용자가 일괄 마감 화면에 접근할 때
**When** 마감 대상 장부를 선택하면
**Then** 대상 지점, 날짜, 현재 상태, 이상 신호, 미확인 항목이 표시되어야 한다.

**Given** 일괄 마감 대상에 이상 신호나 계산 불가 항목이 있을 때
**When** 본사 사용자가 실행을 시도하면
**Then** 위험 알림이 표시되어야 한다
**And** 사용자가 명시 확인한 뒤에만 실행되어야 한다.

**Given** 일괄 마감이 실행될 때
**When** 서버가 요청을 처리하면
**Then** 본사 권한만 허용되어야 한다
**And** 각 장부의 원본 잠금, 감사 로그, revalidation이 처리되어야 한다.

**Given** 지점장 또는 권한 없는 사용자가 일괄 마감을 시도할 때
**When** 서버가 요청을 처리하면
**Then** 마감은 실행되지 않아야 한다
**And** 권한 오류가 표시되어야 한다.

### Story 8.3: 장부 작성자 이름을 입력 흐름 전체에 유지한다

As a 지점장,
I want 장부 작성자 이름을 한 번만 입력하고 싶다,
So that 7단계 저장 과정에서 같은 이름을 반복 입력하지 않아도 된다.

**Acceptance Criteria:**

**Given** 지점장이 1단계에서 작성자 이름을 입력할 때
**When** 다음 단계로 이동하거나 부분 저장을 수행하면
**Then** 작성자 이름은 같은 장부 저장 흐름 전체에 유지되어야 한다.

**Given** 입력자가 7단계 검토/제출까지 진행할 때
**When** 최종 저장 데이터가 생성되면
**Then** 작성자명이 누락되면 안 된다
**And** 누락 시 저장 전에 한국어 검증 메시지를 표시해야 한다.

**Given** 사용자가 같은 장부를 다시 열 때
**When** 작성자명이 이미 저장되어 있으면
**Then** 기존 작성자명이 표시되어야 한다
**And** 변경 시 변경 이력이 남아야 한다.

### Story 8.4: 재고 테이블의 `차이` 문구를 `당일 판매량`으로 정리한다

As a 지점장,
I want 재고 테이블 용어를 현장 업무에 맞게 이해하고 싶다,
So that 재고 입력 중 어떤 값이 당일 판매량인지 헷갈리지 않을 수 있다.

**Acceptance Criteria:**

**Given** 메시지 발송이 실패할 때
**When** 외부 API 오류나 네트워크 오류가 발생하면
**Then** 실패 로그와 오류 사유가 저장되어야 한다
**And** 설정된 정책에 따라 재시도할 수 있어야 한다.

**Given** 알림 계산에 필요한 리포트 데이터가 부족할 때
**When** 알림 메시지를 생성하면
**Then** 임의 정상값으로 숨기면 안 된다
**And** `확인 필요` 상태를 메시지나 발송 로그에 남겨야 한다.

**Given** 알림 발송 이력을 조회할 때
**When** 본사 사용자가 로그 화면을 열면
**Then** 발송 시간, 대상, 채널, 성공/실패, 재시도 횟수, 오류 사유를 볼 수 있어야 한다.
