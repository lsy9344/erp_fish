# erp_fish - Epic Breakdown

## Overview

This document provides the complete epic and story breakdown for erp_fish, decomposing the requirements from the PRD, UX Design, Architecture, and the approved MVP story extraction gate into implementable stories.

## Requirements Inventory

### Functional Requirements

FR1: 사용자는 본사 사용자 또는 지점장 역할로 로그인할 수 있어야 하며, 역할에 따라 접근 가능한 메뉴와 장부 데이터가 달라져야 한다.

FR2: 지점장은 자기 지점의 일일 장부만 조회하고 입력할 수 있어야 하며, 본사 사용자는 권한 범위에 따라 전체 또는 지정 지점의 장부 목록과 상세를 조회할 수 있어야 한다.

FR3: 시스템은 주요 입력, 수정, 마감, 정정, 권한 변경, 기준값 변경에 대해 실행자, 시각, 대상, 변경 전 값, 변경 후 값, 사유를 감사 가능하게 기록해야 한다.

FR4: 본사 사용자와 지점장은 지점+일자 기준으로 일일 장부를 열 수 있어야 하며, 같은 지점+일자에는 하나의 일일 장부만 존재해야 한다.

FR5: 일일 장부 입력은 매출/결제, 비용, 매입, 재고, 손실/폐기/떨이, 근무인원/특이사항, 검토/제출 순서의 7단계 흐름으로 제공되어야 한다.

FR6: 입력자는 총매출, 현금, 카드, 기타 결제수단, 매출차액 관련 값을 입력할 수 있어야 하며, 시스템은 매출차액을 계산하거나 표시할 수 있어야 한다.

FR7: 입력자는 하나의 일일 장부에 여러 비용 항목, 금액, 메모를 입력할 수 있어야 하며, 비용 합계는 영업이익 계산에 반영되어야 한다.

FR8: 입력자는 품목, 규격, 단가, 수량, 매입금액을 입력할 수 있어야 하며, 본사 매입 기준 또는 품목 마스터를 참조할 수 있어야 한다.

FR9: 입력자는 품목+규격 단위로 전일재고, 매입, 판매/차감, 손실, 당일재고, 재고금액, 수량을 다룰 수 있어야 하며, 월초 재고 스냅샷과 직전 마감 장부의 당일재고를 기준으로 전일재고가 이월되어야 한다.

FR10: 본사 사용자와 지점장은 시스템 계산 재고와 실제 재고가 다를 때 재고 조정을 기록할 수 있어야 하며, 조정 전/후 값, 차이, 사유, 작성자, 작성 시각이 남아야 한다.

FR11: 입력자는 손실, 폐기, 떨이 항목을 품목, 수량, 금액, 처리 유형, 사유/특이사항과 함께 입력할 수 있어야 하며, 손실은 재고 흐름과 손실 리포트에 반영되어야 한다.

FR12: 입력자는 근무인원과 근무 관련 특이사항을 기록할 수 있어야 하며, 근무인원은 인당생산성 계산에 반영되어야 한다.

FR13: 시스템은 매출, 매출원가, 매출이익, 이익률, 영업이익, 인당생산성, 평균재고, 평균매출, 매출대비 재고비율, 최고매출품목, 최고매출품목 매출액, 매출차액, 재고/손실 관련 지표를 계산해야 한다.

FR14: 시스템은 필수 입력값 누락, 결제수단 합계와 총매출 차이, 상품별 판매금액 합계와 총매출 차이, 재고/손실 이상 후보를 표시해야 한다.

FR15: 본사 사용자는 오늘/어제 기준 전체 지점의 영업 상태, 장부 상태, 이상 신호, 매출, 이익률, 매출차액, 손실 여부, 마지막 수정자, 마지막 수정 시각, 본사 마감 여부를 한 화면에서 볼 수 있어야 한다.

FR16: 시스템은 매출/이익률 급락, 매출차액, 재고/손실 이상을 이상 신호로 표시해야 하며, 한 지점에 여러 이상 신호가 동시에 표시될 수 있어야 한다.

FR17: 본사 사용자는 매출 하락률, 이익률 하락폭, 매출차액 금액, 손실액, 재고 차이 기준 등 이상 신호 기준값을 설정할 수 있어야 하며, 변경 이력이 남아야 한다.

FR18: 권한 있는 본사 사용자는 모든 지점의 본사 마감 전 일일 장부를 직접 입력하거나 수정할 수 있어야 하며, 수정 후 계산값과 이상 신호가 갱신되어야 한다.

FR19: 본사 사용자는 일일 장부를 본사 마감 처리할 수 있어야 하며, 본사 마감 후 원본 장부 입력값은 수정할 수 없어야 한다.

FR20: 본사 마감 후 오류가 발견되면 본사 사용자는 원본을 수정하지 않고 정정 기록을 추가할 수 있어야 하며, 정정 기록에는 대상 항목, 원본값, 정정값, 사유, 작성자, 작성 시각이 남아야 한다.

FR21: 관제판과 리포트의 기본 숫자는 정정 반영값을 사용해야 하며, 사용자는 상세에서 원본값과 정정 반영값의 차이를 확인할 수 있어야 한다.

FR22: 본사 사용자는 지점명과 활성/비활성 상태를 관리할 수 있어야 하며, 비활성 지점의 과거 장부 데이터는 보존되어야 한다.

FR23: 본사 사용자는 본사 사용자와 지점장 계정, 권한 프로파일, 지점 접근 권한을 관리할 수 있어야 하며, 권한 변경 후 접근 범위가 즉시 반영되어야 한다.

FR24: 본사 사용자는 품목명, 구분, 규격, 기본 단가를 관리할 수 있어야 하며, 품목 마스터 변경은 과거 장부의 원본 입력값을 임의로 바꾸지 않아야 한다.

FR25: 본사 사용자는 매입 관련 기준 또는 기본 정보를 관리할 수 있어야 하며, 매입 입력 시 본사 기준 정보를 선택하거나 참조할 수 있어야 한다.

FR26: 본사 사용자는 결제수단, 비용 항목, 손실 유형 같은 장부 입력 코드를 관리할 수 있어야 하며, 비활성 코드는 신규 입력 선택지에서 제외할 수 있어야 한다.

FR27: 본사 사용자는 전체 지점의 일별 회의용 요약을 볼 수 있어야 하며, 리포트는 지점별 마감 상태, 이상 신호, 매출, 이익률, 매출차액, 손실 현황을 보여줘야 한다.

FR28: 본사 사용자는 선택 기간의 지점별 실적을 비교할 수 있어야 하며, 비교 항목은 매출, 매출이익, 이익률, 영업이익, 인당생산성, 평균재고, 평균매출, 매출대비 재고비율, 손실을 포함해야 한다.

FR29: 본사 사용자는 지점별 월간 실적, 마감 현황, 주요 이상 항목, 손실/재고 흐름 요약을 볼 수 있어야 하며, 월간 요약은 최고매출품목, 최고매출품목 매출액, 평균재고, 평균매출, 매출대비 재고비율을 포함해야 한다.

### NonFunctional Requirements

NFR1: 권한 제한은 UI 숨김이 아니라 서버 응답, export, 캐시 응답, 알림 템플릿, mutation 권한에서 강제되어야 한다.

NFR2: 본사 마감 후 원본 장부는 보존되어야 하며, 마감 후 변경은 append-only 정정 기록으로만 반영되어야 한다.

NFR3: 입력, 수정, 마감, 정정, 기준값 변경, 권한 변경, 업로드 commit/void/reprocess는 감사 로그에 남아야 한다.

NFR4: 본사 관제판은 10개 내외 지점 기준 기본 조회를 3초 안에 표시하는 것을 목표로 한다.

NFR5: 지점장 장부 입력 핵심 흐름은 PC, 태블릿, 모바일 웹에서 동작해야 하며 최소 390px 폭 모바일을 지원해야 한다.

NFR6: 본사 화면은 최신 Chrome/Edge, 지점장 입력은 모바일 Chrome/Safari 최신 버전 기준으로 검증해야 한다.

NFR7: 영업시간 데이터 손실 목표는 RPO 1시간 이내, 장애 복구 목표는 RTO 4시간 이내로 둔다.

NFR8: 감사 로그, 업로드 원본, 파싱 결과, 정정 기록은 최소 24개월 이상 보존해야 한다.

NFR9: 장부 동시 편집은 조용한 마지막 저장 덮어쓰기를 허용하지 않고 edit token 또는 서버 version 기반 충돌 해결을 적용해야 한다.

NFR10: 계산 결과는 서버 계산 로직을 기준으로 하며, 관제판, 상세, 리포트, export가 같은 계산 함수를 공유해야 한다.

NFR11: 정책 미정, 데이터 부족, 계산 실패는 `확인 필요`, `데이터 부족`, `계산 불가`로 명시해야 하며 0이나 오래된 값으로 숨기면 안 된다.

NFR12: 기존 엑셀 수식/서식 복제가 아니라 업무 목적, 데이터 흐름, 서버 계산 기준으로 시스템을 재정의해야 한다.

NFR13: POS/카드 매출 자동 연동, AI 기능, 과거 엑셀 일괄 이관, 실제 급여 지급 확정은 현재 구현 범위가 아니다.

NFR14: 외부 토큰, Auth secret, DB URL, API key는 환경 변수로 관리하고 저장소에 커밋하지 않아야 한다.

NFR15: 접근성 기준은 WCAG 2.2 AA를 기본으로 하며, 색상만으로 상태를 전달하지 않아야 한다.

NFR16: MVP 구현은 신선한 운영 데이터와 감사 가능성을 우선하며, 장부 상태와 정정 반영값에 공격적인 캐싱을 적용하지 않는다.

### Additional Requirements

AR1: Epic/story generation must obey the approved G6 gate in `mvp-story-extraction-checklist.md`.

AR2: Only `MVP-S01`, `MVP-S02`, and `MVP-S03` may generate implementation stories at this stage.

AR3: `MVP-S04` through `MVP-S10` must remain discovery or policy work until the linked OQs and closure artifacts are approved.

AR4: `MVP 필수` does not automatically mean `implementation story allowed`; story generation must check each slice's `may generate implementation story` value.

AR5: The first implementation work must initialize Create T3 App with TypeScript, Next.js, Prisma, PostgreSQL, Auth/NextAuth foundation, Tailwind CSS, shadcn/ui, and Vercel-compatible environment setup.

AR6: shadcn/ui must be initialized early, with baseline ERP primitives for buttons, forms, tables, dialogs, sheets, tabs, badges, alerts, separators, skeletons, toasts, sidebar, and charts.

AR7: Prisma schema and migrations are the source of truth for persisted data; `storeId + closingDate` must be unique for daily ledgers.

AR8: Domain models must cover stores, users, permission profiles, daily ledgers, ledger sections, inventory, purchases, losses, corrections, audit logs, master data, report/export metadata, and later extension models when promoted.

AR9: Server Actions and Route Handlers must validate input with Zod, check authorization, run database writes in transactions where needed, write audit logs, and trigger revalidation.

AR10: Authorization helpers must support action-level checks, not just a broad headquarters/store-manager role check.

AR11: Store-manager responses must be shaped on the server to remove sensitive accounting fields before data reaches the client.

AR12: Ledger saves must use optimistic concurrency checks with edit token or version values.

AR13: Audit logs and correction records must be append-only; agents must not overwrite original ledger values after HQ close.

AR14: Shared server calculation modules must cover basic non-sensitive MVP calculations first and later FIFO, product analysis, special-period reports, monthly P&L, and notifications only when their gates close.

AR15: CAP-1, CAP-9, CAP-5, CAP-6, CAP-7, CAP-8, CAP-4, CAP-13, CAP-14, CAP-15, CAP-16, CAP-17, CAP-18, CAP-2, CAP-3, CAP-10, CAP-11, and CAP-12 are approved extension requirements, but their story readiness is governed by release bucket and OQ gates.

AR16: CAP-19 is contract/operations scope and must not become a product implementation story unless a later PRD or change proposal defines concrete product surfaces.

AR17: OQ-1, OQ-2, OQ-3, and OQ-10A must close before their related MVP blocked slices can become implementation stories.

AR18: OQ-7, OQ-8, OQ-15, and OQ-17 must close before FIFO/eCount/product mapping implementation stories are generated.

AR19: OQ-10B must close before configurable sensitive-field exposure, CAP-13 advanced behavior, CAP-4 sensitive analysis, CAP-10 sensitive monthly P&L exposure, or alert template exceptions are implemented.

AR20: Epics must separate implementation stories from discovery/policy stories so agents do not build unapproved calculations or exposure policies.

### UX Design Requirements

UX-DR1: 전체 UI는 shadcn/ui + Tailwind 기반의 내부 ERP/SaaS 톤을 유지하고, 낮은 장식성, 빠른 데이터 스캔, 반복 업무 효율을 우선해야 한다.

UX-DR2: 색상 토큰은 회색 앱 캔버스, 흰 작업 표면, Primary Blue, Sky Accent, Warning Amber, Destructive 계열을 구분해 사용해야 한다.

UX-DR3: 본사 기본 화면은 데스크탑 관제판이며, 오늘/어제 기준 지점별 장부 상태, 마감 여부, 최신 반영 시각, 이상 신호를 행 단위로 보여줘야 한다.

UX-DR4: 관제판 지점 행 전체 클릭은 장부 상세 페이지로 이동해야 하며, 본사 장부 상세는 Sheet가 아니라 `/app/ledgers/[ledgerId]` 같은 페이지 라우트로 처리해야 한다.

UX-DR5: 지점장 장부 입력은 모바일 퍼스트 7단계 폼으로 제공해야 한다.

UX-DR6: 단계 이동이나 하단 탭 이동 전에 미저장 변경이 있으면 저장, 취소, 계속 편집 중 하나를 선택하게 해야 한다.

UX-DR7: 작성자 표시명은 인증 계정 및 감사 로그 actor와 분리하고, 장부 입력 흐름 전체에서 유지해야 한다.

UX-DR8: 장부 상태 배지는 `미입력`, `입력중`, `검토대기`, `본사마감`, `휴무` 5종을 텍스트 pill로 표시해야 한다.

UX-DR9: 이상 신호는 SignalChip으로 표시하고 색상만 쓰지 않으며, 아이콘, 레이블, 툴팁 또는 보조 텍스트를 함께 제공해야 한다.

UX-DR10: OQ가 열린 기준은 `초과`가 아니라 `기준 확인 필요`로 표현해야 한다.

UX-DR11: 재고 테이블은 냉동/생물 탭, 전일재고 후보, 확정 이월 기준, 매입, 손실, 당일재고, 상태를 다루고 수정 행을 즉시 강조해야 한다.

UX-DR12: 지점장에게 원가, 이익, 마진율, 재고금액, lot 근거, 타 지점 비교 같은 민감 지표를 표시하지 않아야 한다.

UX-DR13: `기준 확인 필요`, `데이터 부족`, `계산 불가`는 서로 다른 상태로 표시해야 한다.

UX-DR14: 마감 전 ClosePreflight는 권한, 이미 마감, 필수 누락, 숫자 오류, 이월 공백, 가격 기준 없음, 이상 신호 등을 표로 요약해야 한다.

**When** `.env.example`을 확인한다

**And** 장부나 본사 데이터는 반환되지 않아야 한다.

**Given** 개발자가 Prisma를 사용해 앱을 실행할 때
**When** Prisma client generation과 기본 DB 연결 확인을 수행한다
**Then** 앱은 DB 연결 실패를 명확한 서버 오류로 기록해야 한다
**And** 인증/권한/감사 모델을 이후 스토리에서 확장할 수 있는 구조여야 한다.

**Given** 개발자가 기본 UI를 확인할 때
**When** 로그인 화면과 앱 shell을 렌더링한다
**Then** shadcn/ui와 Tailwind 기반 스타일이 적용되어야 한다

As a 지점장 또는 본사 사용자,

**Then** 사용자는 로그인 화면 또는 인증 흐름으로 이동해야 한다
**And** 보호 데이터는 서버 응답에 포함되지 않아야 한다.

**Given** 지점장 계정이 로그인되어 있을 때
**When** 지점장 앱 영역을 연다
**Then** 연결된 지점의 장부 입력/조회 메뉴만 접근 가능해야 한다

**When** 본사 앱 영역을 연다
**Then** 권한 프로파일에 맞는 본사 메뉴가 표시되어야 한다
**And** 권한이 없는 설정, 마감, export 같은 action은 UI와 서버에서 모두 차단되어야 한다.

**Given** 사용자가 URL을 직접 입력해 권한 밖 지점 또는 본사 화면에 접근할 때

**And** 권한 없는 데이터가 클라이언트로 전달되지 않아야 한다.

**When** 사용자가 다음 요청을 보낸다
**Then** 변경된 권한이 서버 기준으로 즉시 반영되어야 한다
**And** 오래된 클라이언트 UI 상태만 믿고 접근을 허용하면 안 된다.

### Story 1.4: 서버 권한 헬퍼와 민감 필드 응답 차단

As a 개발자,
I want 권한 확인과 민감 필드 차단을 공통 서버 헬퍼로 사용할 수 있기를 원한다,
So that 이후 장부, 리포트, export 기능이 각자 다른 방식으로 보안을 구현하지 않는다.

**Acceptance Criteria:**

**Given** 서버 코드가 사용자 권한을 확인해야 할 때
**When** 기능 코드가 공통 authorization helper를 호출한다
**Then** helper는 로그인 여부, 권한 프로파일, 지점 접근 범위, action 권한을 함께 확인해야 한다
**And** 권한 없는 요청은 업무 로직 실행 전에 중단되어야 한다.

**Given** 본사 action 권한이 필요한 요청이 있을 때
**When** close, correction, upload, settings, report/export 같은 action을 검사한다
**Then** 각 action은 명시된 permission key 또는 동등한 구조로 확인되어야 한다
**And** 단순히 `본사 사용자`라는 큰 역할만으로 모든 action을 허용하면 안 된다.

**Given** 지점장 경로에서 장부나 리포트 데이터를 조회할 때
**When** 서버가 응답 데이터를 만든다
**Then** 매출원가, 매출이익, 영업이익, 이익률, 인당생산성, 재고금액, lot 근거, 본사 고정비, 타 지점 비교 같은 민감 필드는 응답에서 제거되어야 한다
**And** 클라이언트에서 숨기는 방식에 의존하면 안 된다.

**Given** 본사 조회 전용 권한이 있는 사용자가 리포트나 export를 요청할 때
**When** 서버가 응답 필드와 대상 지점을 결정한다
**Then** 허용된 지점/기간/컬럼만 포함해야 한다
**And** 권한 밖 지점이나 권한 없는 민감 컬럼은 반환하지 않아야 한다.

**Given** 새로운 feature action이 추가될 때
**When** 개발자가 권한 검사를 구현한다
**Then** 기존 공통 helper 패턴을 재사용해야 한다
**And** one-off role check를 만들면 코드 리뷰에서 차단할 수 있어야 한다.

### Story 1.5: 감사 로그 기반 구축

As a 본사 관리자,
I want 주요 변경 이력이 누가 언제 무엇을 바꿨는지 남기를 원한다,
So that 장부, 권한, 기준값 변경을 나중에 신뢰할 수 있게 추적할 수 있다.

**Acceptance Criteria:**

**Given** 감사 대상 이벤트가 발생할 때
**When** 서버 mutation이 실행된다
**Then** 감사 로그에는 actor 계정, 권한 컨텍스트, 대상 엔티티, 이벤트 유형, 발생 시각이 기록되어야 한다
**And** 가능한 경우 변경 전 값, 변경 후 값, 사유도 함께 기록되어야 한다.

**Given** 권한 변경이나 지점 접근권 변경이 발생할 때
**When** 본사 설정 관리자가 변경을 저장한다
**Then** 변경 전/후 권한과 대상 사용자가 감사 로그에 남아야 한다
**And** 변경 실행자와 변경 시각을 확인할 수 있어야 한다.

**Given** 아직 장부 입력 기능이 완성되지 않았을 때
**When** 감사 로그 기반 스토리가 완료된다
**Then** 사용자/권한/지점 접근 변경 이벤트로 감사 로그 기능을 독립 검증할 수 있어야 한다
**And** 장부 전체 모델을 미리 구현하지 않아야 한다.

**Given** 본사 사용자가 감사 로그를 확인할 때
**When** 감사 로그 목록을 조회한다
**Then** 최근 이벤트를 사람이 읽기 쉬운 문장과 기본 필터로 볼 수 있어야 한다
**And** 권한 없는 사용자는 감사 로그에 접근할 수 없어야 한다.

**Given** 향후 장부, 마감, 정정, 기준값 변경 기능이 추가될 때
**When** 개발자가 새 mutation을 구현한다
**Then** 기존 감사 로그 helper를 재사용할 수 있어야 한다
**And** 각 기능이 서로 다른 감사 로그 형식을 만들지 않아야 한다.

## Epic 2: 지점 일일 장부 입력

지점장 또는 본사가 하루 장부를 만들고, 매출/비용/매입/재고/손실/근무 정보를 모바일 우선 7단계 흐름으로 입력한다.

### Story 2.1: 지점+일자 장부 생성과 상태 관리

As a 지점장 또는 본사 사용자,
I want 특정 지점과 날짜의 일일 장부를 열거나 만들 수 있기를 원한다,
So that 하루 운영 기록을 하나의 기준 장부에 입력할 수 있다.

**Acceptance Criteria:**

**Given** 사용자가 자기 권한 범위 안의 지점과 날짜를 선택했을 때
**When** 해당 지점+일자 장부가 아직 없으면
**Then** 시스템은 새 일일 장부를 생성할 수 있어야 한다
**And** 같은 지점+일자에는 하나의 장부만 존재해야 한다.

**Given** 같은 지점+일자의 장부가 이미 있을 때
**When** 사용자가 해당 날짜 장부를 다시 연다
**Then** 기존 장부가 열려야 한다
**And** 중복 장부가 생성되면 안 된다.

**Given** 장부가 생성되거나 수정될 때
**When** 시스템이 장부 상태를 저장한다
**Then** 장부는 `미입력`, `입력중`, `검토대기`, `본사마감`, `휴무` 중 하나의 상태를 가져야 한다
**And** 상태 배지는 UX 기준에 따라 텍스트 pill로 표시되어야 한다.

**Given** 지점장 사용자가 다른 지점 장부를 생성하거나 열려고 할 때
**When** 서버가 권한을 확인한다
**Then** 요청은 거부되어야 한다
**And** 권한 밖 지점 데이터는 응답에 포함되지 않아야 한다.

**Given** 장부가 본사 마감 상태일 때
**When** 사용자가 원본 입력값을 수정하려 한다
**Then** 원본 수정은 차단되어야 한다
**And** 마감 후 변경은 이후 정정 기록 흐름에서만 처리되어야 한다.

**Given** 개발자가 데이터 모델을 구현할 때
**When** 일일 장부 테이블을 정의한다
**Then** `storeId + closingDate` 유일성이 보장되어야 한다
**And** 장부 저장 충돌 방지를 위한 version 또는 edit token 필드가 있어야 한다.

### Story 2.2: 장부 입력 7단계 폼과 작성자 표시명 유지

As a 지점장,
I want 하루 장부를 7단계 흐름으로 입력하고 작성자 표시명이 유지되기를 원한다,
So that 모바일에서도 빠뜨리지 않고 장부를 끝까지 작성할 수 있다.

**Acceptance Criteria:**

**Given** 지점장이 작성 가능한 장부를 열었을 때
**When** 장부 입력 화면이 표시된다
**Then** 입력 흐름은 매출/결제, 비용, 매입, 재고, 손실/폐기/떨이, 근무인원/특이사항, 검토/제출의 7단계로 구성되어야 한다
**And** 현재 단계와 저장 상태를 사용자가 알 수 있어야 한다.

**Given** 사용자가 1단계에서 작성자 표시명을 입력했을 때
**When** 2~7단계로 이동하거나 부분 저장을 수행한다
**Then** 작성자 표시명은 같은 장부 입력 흐름 안에서 유지되어야 한다
**And** 최종 저장 데이터에서 누락되면 안 된다.

**Given** 작성자 표시명과 인증 계정이 다를 때
**When** 장부 상세 또는 감사 관련 정보가 표시된다
**Then** 작성자 표시명과 인증 계정 actor는 구분되어야 한다
**And** 작성자 표시명이 감사 로그 실행자를 대체하면 안 된다.

**Given** 모바일 390px 폭 화면에서 입력할 때
**When** 사용자가 각 단계를 이동한다
**Then** 핵심 입력 필드와 단계 이동 버튼은 겹치지 않아야 한다
**And** 터치 대상은 최소 44x44px을 만족해야 한다.

**Given** 사용자가 단계 이동 중 미저장 변경을 가진 상태일 때
**When** 다음 단계로 이동하려 한다
**Then** 저장, 취소, 계속 편집 중 하나를 선택하게 해야 한다
**And** 사용자가 선택하기 전에는 입력값을 조용히 버리면 안 된다.

**Given** 네트워크 오류나 저장 실패가 발생했을 때
**When** 사용자가 현재 단계에 머문다
**Then** 마지막 저장 시각, 저장되지 않았을 수 있는 단계/필드, 재시도 액션을 보여줘야 한다
**And** 사용자가 입력값 손실 가능성을 이해할 수 있어야 한다.

### Story 2.3: 매출/결제와 비용 입력

As a 지점장,
I want 총매출, 결제수단, 비용 항목을 입력할 수 있기를 원한다,
So that 하루 장부의 기본 매출과 비용 정보를 본사에 제출할 수 있다.

**Acceptance Criteria:**

**Given** 사용자가 매출/결제 단계에 있을 때
**When** 총매출, 현금, 카드, 기타 결제수단 값을 입력한다
**Then** 값은 장부에 저장되어야 한다
**And** 금액은 서버에서 숫자/범위 검증을 거쳐야 한다.

**Given** 결제수단 합계와 총매출이 다를 때
**When** 사용자가 매출/결제 단계를 저장하거나 검토 단계로 이동한다
**Then** 시스템은 부호 있는 차이값을 표시해야 한다
**And** OQ-1이 닫히기 전에는 이상 신호 임계값 초과 판정으로 처리하지 않아야 한다.

**Given** 사용자가 비용 단계에 있을 때
**When** 비용 항목, 금액, 메모를 여러 줄 입력한다
**Then** 각 비용 라인은 장부에 저장되어야 한다
**And** 비용 합계는 서버 기준으로 계산되어야 한다.

**Given** 비용 항목 코드가 비활성 상태일 때
**When** 사용자가 새 비용 라인을 추가한다
**Then** 비활성 코드는 신규 선택지에 표시되지 않아야 한다
**And** 과거 장부에 이미 저장된 비활성 코드는 원본 표시를 유지해야 한다.

**Given** 지점장 또는 본사가 매출/비용 입력값을 수정할 때
**When** 저장 요청이 성공한다
**Then** 변경 이력은 감사 로그 helper를 통해 남아야 한다
**And** 장부의 version 또는 edit token이 갱신되어야 한다.

**Given** 사용자가 권한 없는 지점의 매출/비용 저장을 시도할 때
**When** 서버 action이 요청을 처리한다
**Then** 저장은 거부되어야 한다
**And** 권한 밖 장부 데이터는 반환되지 않아야 한다.

### Story 2.4: MVP 수동 매입 입력

As a 지점장 또는 본사 사용자,
I want 장부에 품목별 매입 정보를 수동으로 입력할 수 있기를 원한다,
So that 이카운트 업로드 전에도 당일 매입 내역을 장부에 반영할 수 있다.

**Acceptance Criteria:**

**Given** 사용자가 매입 단계에 있을 때
**When** 품목, 규격, 단가, 수량, 매입금액을 입력한다
**Then** 매입 라인은 해당 장부에 저장되어야 한다
**And** 숫자 값은 서버에서 검증되어야 한다.

**Given** 본사 매입 기준 또는 품목 마스터가 존재할 때
**When** 사용자가 매입 라인을 추가한다
**Then** 사용자는 기준 정보를 선택하거나 참조할 수 있어야 한다
**And** 기준 정보가 없더라도 MVP 수동 입력 자체는 막히지 않아야 한다.

**Given** 품목 마스터 정규화 OQ-3이 아직 닫히지 않았을 때
**When** 사용자가 품목/규격을 입력한다
**Then** 시스템은 원문 입력값을 보존해야 한다
**And** 초기 품목명/구분/규격 정규화 확정 구현을 이 스토리에 포함하지 않아야 한다.

**Given** 사용자가 매입 라인을 수정하거나 삭제할 때
**When** 저장 요청이 성공한다
**Then** 변경 전/후 값은 감사 로그에 남아야 한다
**And** 장부 version 또는 edit token이 갱신되어야 한다.

**Given** 권한 없는 사용자가 매입 정보를 저장하려 할 때
**When** 서버 action이 요청을 처리한다
**Then** 저장은 거부되어야 한다
**And** 권한 밖 장부의 매입 라인은 반환되지 않아야 한다.

**Given** 이카운트 업로드 요구가 존재할 때
**When** 이 스토리를 구현한다
**Then** CAP-6 업로드 preview/commit/reprocess 기능은 구현하지 않아야 한다
**And** 나중에 업로드 매입 라인과 수동 매입 라인을 구분할 수 있는 확장 여지는 남겨야 한다.

### Story 2.5: 재고 입력과 이월 상태 표시

As a 지점장,
I want 품목별 전일재고, 매입, 손실, 당일재고를 입력하고 이월 상태를 확인할 수 있기를 원한다,
So that 본사가 재고 흐름의 누락과 재확인 대상을 파악할 수 있다.

**Acceptance Criteria:**

**Given** 사용자가 재고 단계에 있을 때
**When** 품목+규격 단위 재고 행을 입력한다
**Then** 전일재고, 매입, 손실, 당일재고, 수량, 상태를 저장할 수 있어야 한다
**And** 냉동/생물 또는 동등한 품목 그룹 탭으로 구분할 수 있어야 한다.

**Given** 직전 장부의 당일재고가 존재할 때
**When** 사용자가 다음 영업일 재고 단계를 연다
**Then** 시스템은 전일재고 후보를 보여줄 수 있어야 한다
**And** 직전 장부가 본사 마감 전이면 확정값이 아니라 `검토 필요` 상태로 표시해야 한다.

**Given** 전일 장부가 없거나 이월 근거가 부족할 때
**When** 재고 행이 표시된다
**Then** `이월 공백` 또는 동등한 상태를 보여줘야 한다
**And** 사용자가 부족한 근거를 0으로 오해하지 않게 해야 한다.

**Given** 본사 마감 후 정정이 이월 값에 영향을 줄 수 있을 때
**When** 다음 장부 재고 화면이 열린다
**Then** `이월 재확인 필요` 상태를 표시할 수 있어야 한다
**And** 기존 값을 조용히 덮어쓰면 안 된다.

**Given** 지점장 권한으로 재고 화면을 볼 때

### Story 2.8: 근무인원/특이사항과 검토 제출

**Given** 사용자가 검토/제출 단계에 도달했을 때

**When** 서버 검증이 통과한다

**Given** 제출 시 필수 값이 누락되었을 때

**Then** 제출은 거부되어야 한다
**And** 사용자는 어느 단계의 어떤 필드를 고쳐야 하는지 알 수 있어야 한다.

**Given** 장부가 제출되거나 제출 상태에서 수정될 때
**When** 저장 요청이 성공한다

### Story 2.9: 장부 저장 충돌과 미저장 변경 처리

As a 지점장 또는 본사 사용자,

**Given** 서로 다른 섹션의 변경이 충돌 없이 병합 가능한 경우
**When** 서버가 저장 요청을 처리한다
**Then** 병합은 감사 로그를 남긴 뒤 허용할 수 있어야 한다
**And** 변경된 섹션, 필드, actor, 시각이 추적 가능해야 한다.

**Given** 본사가 강제 수정 중인 장부 또는 섹션이 있을 때
**When** 지점장이 해당 영역을 저장하려 한다

**And** UI는 `본사 수정 중` 또는 동등한 상태를 표시할 수 있어야 한다.

**Given** 사용자가 단계 이동 또는 하단 탭 이동을 시도할 때
**When** 현재 단계에 미저장 변경이 있다
**Then** 저장, 취소, 계속 편집 중 하나를 선택하게 해야 한다
**And** 선택 전에는 입력값을 버리거나 자동으로 덮어쓰면 안 된다.

**Given** 모바일 또는 브라우저 세션이 오래된 장부 상태를 가지고 있을 때
**When** 사용자가 저장을 시도한다
**Then** 서버는 오래된 저장을 거부하고 최신 장부를 다시 불러오게 해야 한다
**And** 사용자가 어떤 값이 저장되지 않았는지 알 수 있어야 한다.

## Epic 3: 신뢰 가능한 기본 계산과 검증

본사와 지점장이 누락, 기본 합계, 저장 충돌, 계산 불가 상태를 명확히 보고, 승인된 기본 계산만 같은 서버 기준으로 확인한다.

### Story 3.1: 기본 계산 모듈과 상태 표현

As a 본사 사용자,
I want 승인된 기본 계산값이 서버 기준으로 일관되게 계산되기를 원한다,
So that 장부 상세, 관제판, 리포트가 서로 다른 숫자를 보여주지 않는다.

**Acceptance Criteria:**

**Given** 장부에 매출/결제, 비용, 근무인원 데이터가 저장되어 있을 때
**When** 서버 계산 모듈이 기본 계산을 수행한다
**Then** 총매출, 결제수단 합계, 비용합계, 근무인원, 평균매출 같은 OQ 비의존 계산값을 반환해야 한다
**And** 같은 계산 함수는 장부 상세, 관제판, 리포트에서 재사용 가능해야 한다.

**Given** 계산에 필요한 입력값이 누락되었을 때
**When** 서버 계산 모듈이 결과를 만든다
**Then** 누락된 계산은 `데이터 부족` 상태로 반환해야 한다
**And** 0이나 이전 값을 조용히 대체하면 안 된다.

**Given** 정책이 확정되지 않은 계산 항목이 있을 때
**When** 서버 계산 모듈이 결과를 만든다
**Then** 해당 항목은 `확인 필요` 상태로 반환해야 한다
**And** 확정 계산값처럼 표시되면 안 된다.

**Given** 계산 중 예상하지 못한 오류가 발생했을 때
**When** 서버가 결과를 반환한다
**Then** 해당 계산은 `계산 불가` 상태로 반환되어야 한다
**And** 서버에는 디버깅 가능한 오류 로그가 남아야 한다.

**Given** 금액 값이 표시될 때
**When** UI가 계산 결과를 렌더링한다
**Then** 내부 값은 integer KRW 기준이어야 한다
**And** 화면 표시 포맷은 공통 formatter를 통해 처리되어야 한다.

**Given** 개발자가 계산 테스트를 작성할 때
**When** 기본 계산 함수의 unit test를 실행한다
**Then** 정상값, 누락값, 정책 미정, 오류 상태를 각각 검증해야 한다
**And** UI 전용 계산 복제 로직을 만들면 안 된다.

### Story 3.2: 장부 단계별 입력 검증

As a 지점장,
I want 각 입력 단계에서 잘못된 값과 누락을 바로 확인할 수 있기를 원한다,

**Then** 서버는 저장 또는 제출을 거부해야 한다
**And** 응답은 어느 단계의 어느 필드가 문제인지 알려야 한다.

**Given** 사용자가 금액, 수량, 인원 같은 숫자 필드를 입력할 때
**When** 음수, 문자, 허용 범위 밖 값 등 잘못된 값이 들어온다
**Then** 서버 검증은 요청을 거부해야 한다
**And** UI는 해당 필드에 연결된 오류 메시지를 보여줘야 한다.

**Given** 폼 오류가 화면에 표시될 때
**When** 사용자가 제출을 시도한다
**Then** 첫 번째 오류 필드로 포커스가 이동해야 한다
**And** 오류 메시지는 `aria-describedby` 또는 동등한 접근성 연결을 가져야 한다.

**Given** 지점장이 권한 밖 장부 값을 저장하려 할 때
**When** 서버가 요청을 검증한다
**Then** 권한 오류는 입력값 검증보다 먼저 처리되어야 한다
**And** 권한 밖 장부 데이터는 응답에 포함되지 않아야 한다.

**Given** 검증 오류와 계산 상태가 동시에 존재할 때
**When** UI가 검토 정보를 표시한다
**Then** 필수 오류, `기준 확인 필요`, `데이터 부족`, `계산 불가`는 구분되어야 한다
**And** 한 상태가 다른 상태를 덮어 숨기면 안 된다.

**Given** 개발자가 검증 스키마를 구현할 때
**When** 새로운 장부 입력 필드를 추가한다
**Then** Zod 또는 동등한 서버 검증 스키마를 함께 갱신해야 한다
**And** 클라이언트 검증만으로 저장을 허용하면 안 된다.

### Story 3.3: 검토 화면의 계산/검증 요약

As a 지점장 또는 본사 사용자,
I want 제출 전 검토 화면에서 장부의 기본 계산과 검증 상태를 한눈에 보고 싶다,
So that 누락과 이상 후보를 확인한 뒤 제출하거나 보완할 수 있다.

**Acceptance Criteria:**

**Given** 사용자가 검토/제출 단계에 도달했을 때
**When** 검토 화면이 렌더링된다
**Then** 매출/결제, 비용, 매입, 재고, 손실, 근무 정보의 저장 상태와 기본 요약이 표시되어야 한다
**And** 각 요약은 서버 계산/검증 결과를 기준으로 해야 한다.

**Given** 결제수단 합계와 총매출이 다를 때
**When** 검토 화면이 차이값을 표시한다
**Then** 부호 있는 차이값과 설명을 보여줘야 한다
**And** OQ-1 전에는 이상 신호 임계값 초과로 표현하지 않아야 한다.

**Given** 필수 누락 또는 검증 오류가 있을 때
**When** 검토 화면이 표시된다
**Then** 사용자는 어느 단계로 돌아가야 하는지 알 수 있어야 한다
**And** 오류 항목을 눌러 해당 단계/필드로 이동할 수 있어야 한다.

**Given** 정책 미정 또는 데이터 부족 계산이 있을 때
**When** 검토 화면이 결과를 표시한다
**Then** `기준 확인 필요`, `데이터 부족`, `계산 불가` 상태를 그대로 보여줘야 한다
**And** 0원 또는 빈 값으로 숨기면 안 된다.

**Given** 지점장 권한으로 검토 화면을 볼 때
**When** 서버가 요약 응답을 만든다
**Then** 민감 회계 지표나 원가 기반 파생값은 응답에서 제외되어야 한다
**And** 지점장에게 필요한 입력 확인 정보만 제공되어야 한다.

**Given** 본사 사용자가 검토 화면을 볼 때
**When** 권한 있는 본사 계정이 접근한다
**Then** 본사 권한에 허용된 요약과 검증 상태를 볼 수 있어야 한다
**And** 권한 프로파일에 없는 action은 수행할 수 없어야 한다.

### Story 3.4: 계산 정책 미정 항목 차단과 테스트 기준

As a 개발 리드,
I want 정책이 확정되지 않은 계산이 구현 스토리에 섞이지 않기를 원한다,
So that 잘못된 원가, 이익, 민감 지표가 제품에 먼저 들어가지 않는다.

**Acceptance Criteria:**

**Given** 개발자가 계산 모듈을 구현할 때
**When** FIFO 원가, `30%단가`, 희망 판매가 기준 손실액, 민감 파생 지표 계산을 발견한다
**Then** 해당 계산은 구현하지 않아야 한다
**And** 관련 OQ가 닫히기 전에는 `확인 필요` 또는 discovery/policy story로 분리해야 한다.

**Given** OQ-gated 계산 항목이 UI에 표시될 수 있을 때
**When** 서버 응답이 계산 결과를 만든다
**Then** 확정 숫자 대신 상태값과 설명을 반환해야 한다
**And** 화면은 확정 계산처럼 보이게 표시하면 안 된다.

**Given** 지점장 권한으로 계산 요약을 조회할 때
**When** 민감 파생 지표가 포함될 수 있다
**Then** 서버는 해당 필드를 반환하지 않아야 한다
**And** OQ-10A 승인 전에는 애매한 원가/이익 기반 값도 숨김을 기본값으로 해야 한다.

**Given** 본사 권한으로 계산 요약을 조회할 때
**When** OQ가 닫히지 않은 계산 항목이 있다
**Then** 본사에도 확정 계산값처럼 제공하면 안 된다
**And** `기준 확인 필요`와 필요한 정책 질문을 함께 표시해야 한다.

**Given** 개발자가 테스트를 작성할 때
**When** OQ-gated 항목에 대한 테스트를 실행한다
**Then** 임시 계산값이 아니라 차단 상태가 반환되는지 검증해야 한다
**And** 승인되지 않은 계산 로직이 추가되면 테스트가 실패해야 한다.

**Given** 이후 정책이 승인되어 계산이 구현 가능해질 때
**When** 새 구현 스토리가 생성된다
**Then** 이 스토리의 차단 규칙은 승인 문서와 테스트 기준에 맞게 갱신되어야 한다
**And** 기존 차단을 조용히 우회하면 안 된다.

## Epic 4: 본사 관제, 보완 수정, 마감과 정정

본사가 아침 회의 전 전체 지점 상태를 보고, 마감 전 보완 수정과 마감, 마감 후 정정 기록을 처리한다.

### Story 4.1: 본사 전체 지점 관제판 조회

As a 본사 사용자,
I want 오늘/어제 기준 전체 지점 장부 상태를 한 화면에서 보고 싶다,
So that 아침 회의 전에 문제 가능성이 있는 지점을 빠르게 찾을 수 있다.

**Acceptance Criteria:**

**Given** 본사 사용자가 관제판에 접근할 때
**When** 권한 있는 세션으로 오늘/어제 기준 관제판을 연다
**Then** 지점별 영업 상태, 장부 상태, 본사 마감 여부, 최신 반영 시각, 마지막 수정자, 기본 요약이 표시되어야 한다
**And** 지점 행 전체 클릭 시 해당 장부 상세 페이지로 이동해야 한다.

**Given** 조회 전용 본사 권한 사용자가 관제판을 볼 때
**When** 서버가 관제판 데이터를 반환한다
**Then** 허용된 지점과 컬럼만 포함해야 한다
**And** 저장/마감/정정 같은 mutation action은 수행할 수 없어야 한다.

**Given** 본사 관제판이 기본 조회를 수행할 때
**When** 활성 지점이 10개 내외인 경우
**Then** 지점 목록과 기본 상태는 3초 안에 표시되는 것을 목표로 해야 한다
**And** 무거운 감사 내역이나 리포트 drilldown은 별도 페이지나 패널로 분리해야 한다.

**Given** 관제판 데이터 로딩이 3초를 초과할 때
**When** 화면이 대기 상태를 보여준다
**Then** 부분 로드, 마지막 갱신 시각, 재시도 액션을 제공해야 한다
**And** 사용자가 빈 화면을 오류 없는 정상 상태로 오해하면 안 된다.

**Given** 지점장 권한 사용자가 본사 관제판 URL에 접근할 때
**When** 서버가 요청을 처리한다
**Then** 접근은 거부되어야 한다
**And** 전체 지점 목록 또는 타 지점 데이터는 반환되지 않아야 한다.

**Given** 관제판 행에 숫자 컬럼이 표시될 때
**When** UI가 금액이나 수량을 렌더링한다
**Then** 숫자 컬럼은 `tabular-nums` 또는 동등한 고정폭 숫자 표시를 사용해야 한다
**And** 상태 배지는 텍스트만으로도 의미를 알 수 있어야 한다.

### Story 4.2: 관제판 이상 상태와 기준 확인 필요 표시

As a 본사 사용자,
I want 관제판에서 문제 가능성이 있는 지점과 기준 미확정 상태를 구분해서 보고 싶다,
So that 확정된 이상 신호와 아직 정책 확인이 필요한 항목을 혼동하지 않는다.

**Acceptance Criteria:**

**Given** 관제판 행에 장부 상태와 기본 계산 결과가 있을 때
**When** 서버가 표시 상태를 계산한다
**Then** 필수 누락, 데이터 부족, 계산 불가, 기준 확인 필요 상태를 구분해야 한다
**And** 하나의 상태가 다른 상태를 덮어 숨기면 안 된다.

**Given** 매출차액이나 이익률 급락 기준 OQ-1이 아직 닫히지 않았을 때
**When** 관제판이 관련 항목을 표시한다
**Then** 임계값 초과나 확정 이상 신호로 표시하지 않아야 한다
**And** `기준 확인 필요` 또는 동등한 문구로 정책 미정 상태를 보여줘야 한다.

**Given** 재고/손실 이상 후보가 있을 때
**When** 정책이 확정되지 않은 계산에 의존한다
**Then** 확정 이상으로 표시하지 않아야 한다
**And** 필요한 정책 질문 또는 관련 OQ를 확인할 수 있어야 한다.

**Given** SignalChip이 화면에 표시될 때
**When** 사용자가 행을 스캔한다
**Then** 색상만이 아니라 텍스트, 아이콘 또는 툴팁으로 의미를 알 수 있어야 한다
**And** 접근성 이름이 제공되어야 한다.

**Given** 한 지점에 여러 상태가 동시에 있을 때
**When** 관제판 행이 렌더링된다
**Then** 여러 SignalChip을 함께 표시할 수 있어야 한다
**And** 우선순위가 정해지지 않은 항목을 임의로 숨기면 안 된다.

**Given** 지점장 경로에서 상태 요약이 재사용될 때
**When** 서버가 응답을 만든다
**Then** 민감 지표나 타 지점 비교 정보는 포함하지 않아야 한다
**And** 본사 관제판 데이터를 그대로 내려보내면 안 된다.

### Story 4.3: 본사 마감 전 장부 보완 수정

As a 권한 있는 본사 사용자,
I want 마감 전 지점 장부의 누락 또는 오류를 직접 보완 수정할 수 있기를 원한다,
So that 아침 회의 전에 장부를 본사 기준으로 정리할 수 있다.

**Acceptance Criteria:**

**Given** 본사 사용자가 마감 전 장부 상세를 열었을 때
**When** 권한 프로파일이 직접 수정 action을 허용한다
**Then** 본사는 매출/비용/매입/재고/손실/근무 입력값을 보완 수정할 수 있어야 한다
**And** 권한이 없는 본사 계정은 수정 action을 수행할 수 없어야 한다.

**Given** 본사가 장부 값을 수정할 때
**When** 저장 요청이 성공한다
**Then** 변경 전/후 값, 수정자, 시각, 사유가 감사 로그에 남아야 한다
**And** 지점장 입력과 본사 수정은 구분되어야 한다.

**Given** 본사가 특정 장부 또는 섹션을 수정 중일 때
**When** 지점장이 같은 영역을 저장하려 한다
**Then** 서버는 최신 상태 재확인을 요구해야 한다
**And** UI는 `본사 수정 중` 또는 동등한 상태를 표시할 수 있어야 한다.

**Given** 본사가 오래된 version/edit token으로 저장하려 할 때
**When** 서버가 저장 요청을 처리한다
**Then** 저장은 거부되어야 한다
**And** 서버 최신값과 본사 입력값을 비교할 수 있는 충돌 정보를 제공해야 한다.

**Given** 장부가 이미 본사 마감 상태일 때
**When** 본사 사용자가 원본 필드를 직접 수정하려 한다
**Then** 원본 수정은 차단되어야 한다
**And** 마감 후 변경은 정정 기록 흐름으로 안내되어야 한다.

**Given** 보완 수정 후 계산/검증 결과가 바뀔 때
**When** 장부 상세 또는 관제판이 다시 조회된다
**Then** 승인된 기본 계산과 검증 상태는 최신 값 기준으로 갱신되어야 한다
**And** OQ-gated 계산은 여전히 `기준 확인 필요`로 유지되어야 한다.

### Story 4.4: 마감 전 ClosePreflight 검토

As a 본사 마감 담당자,
I want 장부를 마감하기 전에 마감 가능 여부와 위험 항목을 확인하고 싶다,
So that 누락이나 정책 미정 항목을 알고도 잘못 마감하지 않을 수 있다.

**Acceptance Criteria:**

**Given** 본사 사용자가 장부 마감을 시도할 때
**When** ClosePreflight가 실행된다
**Then** 권한, 이미 마감 여부, 필수 누락, 숫자 오류, 이월 공백, 기준 확인 필요, 계산 불가 상태를 표로 요약해야 한다
**And** 마감 담당자는 어떤 항목이 차단이고 어떤 항목이 경고인지 구분할 수 있어야 한다.

**Given** 필수 입력 누락이나 저장 오류가 있을 때
**When** ClosePreflight 결과가 반환된다
**Then** 마감은 차단되어야 한다
**And** 사용자는 어느 단계/필드를 보완해야 하는지 알 수 있어야 한다.

**Given** OQ-gated 항목이 `기준 확인 필요` 상태일 때
**When** ClosePreflight가 표시된다
**Then** 확정 이상이나 확정 계산처럼 보이면 안 된다
**And** 마감 가능 여부에 미치는 기준은 PRD 정책 또는 승인된 설정에 따라야 한다.

**Given** 권한 없는 본사 계정이 ClosePreflight 또는 마감을 실행하려 할 때
**When** 서버가 요청을 처리한다
**Then** 요청은 거부되어야 한다
**And** 장부의 민감 데이터나 마감 가능성 상세를 반환하지 않아야 한다.

**Given** ClosePreflight가 성공적으로 실행될 때
**When** 사용자가 결과를 본다
**Then** 마감 대상 장부, 실행자, 실행 시각, 주요 검증 요약이 확인 가능해야 한다
**And** 이후 마감 action에서 같은 장부 version/edit token을 기준으로 재검증할 수 있어야 한다.

**Given** ClosePreflight 결과가 화면에 표시될 때
**When** 사용자가 키보드나 보조기기로 확인한다
**Then** 차단/경고 상태는 색상만이 아니라 텍스트로도 구분되어야 한다
**And** 표의 상태와 설명은 읽기 순서가 자연스러워야 한다.

### Story 4.5: 본사 마감과 원본 잠금

As a 본사 마감 담당자,
I want 검토가 끝난 일일 장부를 본사 마감 처리할 수 있기를 원한다,
So that 원본 장부가 보존되고 이후 운영 숫자는 정정 기록으로만 바뀌게 할 수 있다.

**Acceptance Criteria:**

**Given** 권한 있는 본사 사용자가 마감 가능한 장부를 열었을 때
**When** ClosePreflight가 통과한 장부를 마감한다
**Then** 장부 상태는 `본사마감`으로 변경되어야 한다
**And** 마감 실행자, 시각, 대상 장부, 검증 요약이 감사 로그에 남아야 한다.

**Given** 마감 요청이 들어왔을 때
**When** 서버가 요청을 처리한다
**Then** 권한, 장부 상태, version/edit token, 필수 검증을 다시 확인해야 한다
**And** ClosePreflight만 통과했다는 이유로 재검증을 생략하면 안 된다.

**Given** 장부가 본사 마감 상태가 된 뒤

**When** 서버 action이 요청을 처리한다
**Then** 요청은 거부되어야 한다

**Given** 정정 기록이 추가되었을 때
**When** 장부 상세를 조회한다
**Then** 원본값과 정정 반영값의 차이를 확인할 수 있어야 한다
**And** 정정 이력은 시간순으로 추적 가능해야 한다.

**Given** 관제판이나 리포트가 장부 숫자를 조회할 때
**When** 정정 기록이 존재한다
**Then** 기본 숫자는 정정 반영값을 사용해야 한다
**And** 원본값이 필요한 상세 화면에서는 원본값도 함께 확인할 수 있어야 한다.

**Given** 권한 없는 본사 사용자 또는 지점장이 정정 기록을 추가하려 할 때
**When** 서버가 요청을 처리한다

**Given** 정정 기록이 계산에 영향을 줄 때

## Epic 5: 본사 기준 정보와 운영 설정

**And** 변경 전/후 지점명은 감사 로그에 남아야 한다.

**And** 사용자는 비활성화를 사용해야 한다는 안내를 받아야 한다.

**Given** 지점명 또는 상태 값이 잘못 입력되었을 때

## Epic 6: 본사 리포트와 안전한 Export

본사가 일별 회의 리포트, 기간 비교, 월간 요약을 본사 권한으로 조회하고 export한다.

### Story 6.1: 일별 아침 회의 리포트

As a 본사 사용자,
I want 특정 일자의 전체 지점 회의용 요약을 볼 수 있기를 원한다,
So that 아침 회의 전에 지점별 상태와 주요 확인 대상을 빠르게 파악할 수 있다.

**Acceptance Criteria:**

**Given** 본사 사용자가 일별 리포트 화면에 접근할 때
**When** 날짜를 선택한다
**Then** 지점별 마감 상태, 장부 상태, 최신 반영 시각, 기본 매출 요약, 손실 여부, 상태 메시지가 표시되어야 한다
**And** 권한 있는 지점/컬럼만 포함되어야 한다.

**Given** 정정 기록이 있는 장부가 있을 때
**When** 리포트가 숫자를 표시한다
**Then** 기본 숫자는 정정 반영값을 사용해야 한다
**And** 원본값 확인은 장부 상세에서 가능해야 한다.

**Given** OQ-gated 계산이나 이상 기준이 필요한 항목이 있을 때
**When** 리포트가 해당 항목을 표시한다
**Then** 확정 숫자나 확정 이상으로 표시하지 않아야 한다
**And** `기준 확인 필요`, `데이터 부족`, `계산 불가` 상태를 구분해 보여야 한다.

**Given** 지점장 권한 사용자가 일별 본사 리포트 URL에 접근할 때
**When** 서버가 요청을 처리한다
**Then** 접근은 거부되어야 한다
**And** 전체 지점 리포트 데이터는 반환되지 않아야 한다.

**Given** 리포트가 조회될 때
**When** 서버가 데이터를 만든다
**Then** 장부 상세/관제판과 같은 서버 계산 함수를 사용해야 한다
**And** 리포트 전용으로 계산식을 복제하면 안 된다.

**Given** 일별 리포트 화면이 렌더링될 때
**When** 숫자 컬럼과 상태 배지가 표시된다
**Then** 숫자 컬럼은 읽기 쉽게 정렬되어야 한다
**And** 상태는 색상만이 아니라 텍스트로도 의미를 알 수 있어야 한다.

### Story 6.2: 지점별 기간 비교 리포트

As a 본사 사용자,
I want 선택한 기간의 지점별 실적을 비교할 수 있기를 원한다,
So that 지점별 흐름과 차이를 회의나 운영 점검에 활용할 수 있다.

**Acceptance Criteria:**

**Given** 본사 사용자가 기간 비교 리포트 화면에 접근할 때
**When** 시작일, 종료일, 지점 필터를 선택한다
**Then** 권한 있는 지점의 기간별 기본 실적 요약이 표시되어야 한다
**And** 권한 밖 지점 데이터는 포함되지 않아야 한다.

**Given** 정정 기록이 있는 장부가 기간 안에 포함될 때
**When** 리포트가 집계 값을 계산한다
**Then** 기본 집계는 정정 반영값을 사용해야 한다
**And** 원본값과 정정 차이는 장부 상세에서 추적 가능해야 한다.

**Given** 비교 항목에 OQ-gated 계산이 포함될 수 있을 때
**When** 리포트가 결과를 표시한다
**Then** 매출이익, 이익률, 영업이익, 인당생산성, 재고금액 등 민감하거나 정책 미정인 항목은 승인 전 확정값으로 표시하지 않아야 한다
**And** `기준 확인 필요` 또는 권한 차단 상태로 표시해야 한다.

**Given** 지점장 권한 사용자가 기간 비교 리포트에 접근할 때
**When** 서버가 요청을 처리한다
**Then** 접근은 거부되어야 한다
**And** 타 지점 비교나 민감 지표는 반환되지 않아야 한다.

**Given** 본사 조회 전용 권한 사용자가 기간 비교를 조회할 때
**When** 서버가 응답을 만든다
**Then** 허용된 지점/기간/컬럼만 반환해야 한다
**And** export나 상세 drilldown action은 별도 권한이 없으면 수행할 수 없어야 한다.

**Given** 기간에 미마감 장부가 포함될 때
**When** 리포트가 집계를 표시한다
**Then** 미마감 포함 여부와 해당 상태를 표시해야 한다
**And** 마감 장부와 미마감 장부가 섞여 있음을 사용자가 알 수 있어야 한다.

### Story 6.3: 월간 지점 요약 리포트

As a 본사 사용자,
I want 지점별 월간 마감 현황과 운영 요약을 볼 수 있기를 원한다,
So that 월 단위로 지점 운영 상태를 점검할 수 있다.

**Acceptance Criteria:**

**Given** 본사 사용자가 월간 지점 요약 화면에 접근할 때
**When** 대상 월과 지점 필터를 선택한다
**Then** 권한 있는 지점의 월간 마감 현황, 주요 상태, 손실/재고 흐름 요약이 표시되어야 한다
**And** 권한 밖 지점 데이터는 포함되지 않아야 한다.

**Given** 해당 월에 마감 장부와 미마감 장부가 섞여 있을 때
**When** 월간 요약이 표시된다
**Then** 마감/미마감 건수와 미마감 포함 여부를 보여줘야 한다
**And** 사용자가 숫자의 확정 상태를 알 수 있어야 한다.

**Given** 정정 기록이 있는 장부가 월간 집계에 포함될 때
**When** 리포트가 기본 숫자를 계산한다
**Then** 정정 반영값을 사용해야 한다
**And** 원본값 확인은 장부 상세에서 가능해야 한다.

**Given** 최고매출품목, 평균재고, 재고비율 같은 항목이 정책 미정 또는 데이터 부족일 때
**When** 월간 요약이 표시된다
**Then** `기준 확인 필요`, `데이터 부족`, `계산 불가` 상태를 구분해 보여야 한다
**And** 확정 숫자처럼 표시하면 안 된다.

**Given** 지점장 권한 사용자가 월간 본사 요약에 접근할 때
**When** 서버가 요청을 처리한다
**Then** 접근은 거부되어야 한다
**And** 타 지점 비교 또는 민감 지표는 반환되지 않아야 한다.

**Given** 월간 요약 화면이 렌더링될 때
**When** 숫자와 상태가 표시된다
**Then** 숫자 컬럼은 읽기 쉽게 정렬되어야 한다
**And** 상태는 색상만이 아니라 텍스트로도 의미를 알 수 있어야 한다.

### Story 6.4: 본사 전용 Export와 권한 차단

As a 본사 사용자,
I want 허용된 리포트 데이터를 안전하게 export할 수 있기를 원한다,
So that 회의와 운영 점검에 필요한 자료를 외부 파일로 활용하면서도 민감 데이터 노출을 막을 수 있다.

**Acceptance Criteria:**

**Given** export 권한이 있는 본사 사용자가 리포트에서 export를 요청할 때
**When** 서버가 요청을 처리한다
**Then** 현재 사용자에게 허용된 지점, 기간, 컬럼만 포함한 export가 생성되어야 한다
**And** export 생성 시점의 권한을 서버에서 다시 확인해야 한다.

**Given** export 권한이 없는 본사 사용자나 지점장이 export를 요청할 때
**When** 서버가 요청을 처리한다
**Then** 요청은 거부되어야 한다
**And** export 파일 또는 민감 리포트 데이터는 생성되거나 반환되지 않아야 한다.

**Given** export 대상에 민감 회계 지표가 포함될 수 있을 때
**When** 서버가 컬럼을 구성한다
**Then** 권한 없는 사용자에게는 매출원가, 매출이익, 영업이익, 이익률, 인당생산성, 재고금액, lot 근거, 본사 고정비 같은 필드를 포함하지 않아야 한다
**And** 클라이언트에서 컬럼을 숨기는 방식에 의존하면 안 된다.

**Given** export 대상 데이터에 정정 기록이 있을 때
**When** export 파일이 생성된다
**Then** 기본 숫자는 정정 반영값을 사용해야 한다
**And** 필요 시 원본/정정 반영 구분 컬럼은 권한 있는 본사 사용자에게만 포함되어야 한다.

**Given** OQ-gated 계산 항목이 export 대상에 포함될 수 있을 때
**When** export 파일이 생성된다
**Then** 확정 숫자 대신 `기준 확인 필요`, `데이터 부족`, `계산 불가` 상태를 포함해야 한다
**And** 임시 계산값을 export하면 안 된다.

**Given** export가 생성될 때
**When** 서버가 작업을 완료한다
**Then** 생성자, 시각, 대상 리포트, 필터, 컬럼 범위는 감사 로그에 남아야 한다
**And** 파일명 또는 메타데이터에 권한 밖 정보가 포함되면 안 된다.

## Epic 7: 구현 전 정책 확정 (Discovery/Policy Track)

아직 구현 스토리로 만들면 안 되는 계산, 재고, 노출, 용어 정책을 닫는다.

### Story 7.1: 매출차액과 이상 신호 기준 정책 확정

As a PM과 본사 운영자,
I want 매출차액과 이상 신호 기준을 확정하고 싶다,
So that 관제판과 검토 화면이 임시 판단 없이 같은 기준으로 위험 지점을 표시할 수 있다.

**Acceptance Criteria:**

**Given** OQ-1이 열려 있을 때
**When** 이 discovery story를 수행한다
**Then** 매출차액 기준을 금액, 비율, 또는 둘 다로 볼지 결정해야 한다
**And** 부호 처리와 예시 계산을 포함해야 한다.

**Given** 매출/이익률 급락 기준을 정할 때
**When** 본사 운영자가 기준값을 검토한다
**Then** 기준값의 기본값, 지점별/전체 적용 여부, 변경 가능 범위를 문서화해야 한다
**And** 기준값 변경 시 감사 로그에 남겨야 할 항목을 정해야 한다.

**Given** 이상 신호 우선순위가 필요한 경우
**When** 여러 이상 신호가 한 지점에 동시에 발생한다
**Then** 표시 순서와 중복 표시 방식을 정해야 한다
**And** 임의로 신호를 숨기지 않는 원칙을 포함해야 한다.

**Given** 정책 문서가 완료되었을 때
**When** PM과 본사 운영자가 승인한다
**Then** 산출물에는 매출차액 기준표, 부호 예시, 임계값 예시, 표시 문구, 승인자가 포함되어야 한다
**And** MVP-S04를 구현 스토리로 승격할 수 있는지 명시해야 한다.

**Given** 정책이 아직 승인되지 않았을 때
**When** 구현 스토리를 만들려 한다
**Then** 매출차액 임계값 판정이나 이상 신호 확정 로직은 구현 스토리로 만들면 안 된다
**And** 기존 화면은 `기준 확인 필요` 표시를 유지해야 한다.

### Story 7.2: `30%단가` 의미와 화면 노출 정책 확정

As a PM과 본사 운영자,
I want `30%단가`의 업무 의미와 노출 범위를 확정하고 싶다,
So that 재고/품목 화면에서 잘못된 가격 기준을 구현하지 않을 수 있다.

**Acceptance Criteria:**

**Given** OQ-2가 열려 있을 때
**When** 이 discovery story를 수행한다
**Then** `30%단가`가 고정 마진 30% 판매가인지, 참고 판매가인지, 다른 내부 기준인지 결정해야 한다
**And** 계산식 또는 비계산 참고값 여부를 명확히 문서화해야 한다.

**Given** `30%단가`를 화면에 표시할지 검토할 때
**When** 본사와 지점장 권한별 노출 범위를 정한다
**Then** 지점장에게 노출 가능한지, 본사 전용인지, 숨김인지 결정해야 한다
**And** 민감 지표 역추정 가능성이 있으면 OQ-10A와 연결해야 한다.

**Given** `30%단가`가 계산에 쓰일 수 있을 때
**When** 정책을 확정한다
**Then** 적용 대상 품목, 기준 시점, 반올림/절사 기준, 과거 장부 보존 방식을 정해야 한다
**And** 기존 장부 원본값을 소급 변경하지 않는 원칙을 포함해야 한다.

**Given** 정책 산출물이 완료되었을 때
**When** PM과 본사 운영자가 승인한다
**Then** 산출물에는 의미 정의, 계산/비계산 여부, 권한별 표시 정책, 예시가 포함되어야 한다
**And** MVP-S05를 구현 스토리로 승격할 수 있는지 명시해야 한다.

**Given** 정책이 아직 승인되지 않았을 때
**When** 구현 스토리를 만들려 한다
**Then** `30%단가` 파생 표시나 계산은 구현 스토리로 만들면 안 된다
**And** 기존 화면은 `기준 확인 필요` 상태를 유지해야 한다.

### Story 7.3: 품목명/구분/규격 정규화 기준 확정

As a PM과 본사 운영자,
I want 기존 품목명을 품목명, 구분, 규격으로 나누는 기준을 확정하고 싶다,
So that 품목 마스터와 향후 이카운트 업로드 매핑을 같은 기준으로 만들 수 있다.

**Acceptance Criteria:**

**Given** OQ-3이 열려 있을 때
**When** 이 discovery story를 수행한다
**Then** 기존 품목 원문에서 품목명, 구분, 규격을 분리하는 기준을 정해야 한다
**And** 애매한 사례의 처리 원칙을 포함해야 한다.

**Given** 샘플 품목 데이터가 있을 때
**When** 본사 운영자가 정규화 기준을 검토한다
**Then** 최소 대표 샘플에 대해 원문, 정규 품목명, 구분, 규격, 비고를 담은 기준표를 만들어야 한다
**And** 매핑 실패 또는 검토 필요 상태를 표현할 수 있어야 한다.

**Given** 과거 장부 원본값이 있을 때
**When** 품목 마스터 정규화 정책을 확정한다
**Then** 과거 장부의 원문 품목명/규격은 보존해야 한다

### Story 8.5: 마감 운영, 작성자 표시명, 재고 용어, 그리드 리사이징 완료 기준 확정
