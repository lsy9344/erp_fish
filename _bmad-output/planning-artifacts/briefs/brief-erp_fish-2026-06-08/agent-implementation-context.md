# ERP Fish 기능 보완/개선/추가 에이전트 통합 문서

이 문서는 위 세 문서를 구현 에이전트가 바로 읽고 처리할 수 있게 하나로 합친 기준 문서다. 원문 문서는 배경 확인용이고, 기능 범위와 우선순위는 이 문서를 우선한다.

## 1. 작업 배경

ERP Fish 1차 MVP는 지점 일일 장부 입력, 본사 관제, 본사 마감/정정, 기본 리포트까지 구현하는 범위였다.

2026-06-08 회의와 의뢰자 제공 엑셀 확인 결과, 기존 엑셀에는 단순 장부보다 넓은 운영 업무가 들어 있었다. 이번 작업은 완전 신규 기능만 추가하는 일이 아니다. 기존 엑셀에 있던 업무를 웹 ERP에서 정식 기능으로 복원하거나, 얇게 구현된 MVP 기능을 실제 운영 수준으로 보완하는 일도 포함한다.

핵심 방향은 다음과 같다.

- 지점 장부 입력을 유지하면서 본사의 매입, 재고, 직원, 리포트, 손익 관리 능력을 높인다.
- 엑셀에 있던 업무 근거를 무시하지 않는다.
- 원본 장부값과 감사 이력을 보존한다.
- 데이터 구조가 흔들리는 기능부터 먼저 잡는다.
- AI 분석은 이번 구현 범위에 넣지 않는다.

## 3. 엑셀에서 확인된 업무 근거

### 3.1 `total_sales.xlsx`

이 파일은 전 지점 장기 원장과 본사 분석 리포트 역할을 한다.

- 일자, 요일, 매장별 매출 원장
- 현금, 카드, 매출이익, 이익률, 일지출, 영업이익
- 인당생산성, 최고매출품목, 최고매출액
- 근무인원, 팀장, 팀원
- 매출차액
- 기간 비교, 연도 비교, 월 비교
- 설날 같은 특수기간 비교
- 직원 이름 목록

### 3.2 `장부-202605현대 (1).xlsx`

이 파일은 지점 월간 장부 역할을 한다.

- `근무표`에서 직원별 월 근무일수 집계
- `이월`에서 월초/전일 재고 기준 관리
- `1`부터 `31`까지 일자별 장부
- 현금, 카드, 전일 이월금액, 비용, 비용합계
- 근무자 입력
- 냉동/생물 재고 입력
- 품목, 규격, 단가, 재고량, 재고금액
- 박스당 판매단가, `30%단가`
- 매입 작성칸
- 품목명/규격 정규화 수식
- 관리자 분석 영역의 품목규격, 전재고, 매입, 판매원가, 이익률, 판매금액, 판매량

## 4. 기능 계약

아래 `CAP-*` ID는 후속 작업, 스토리, 테스트에서 그대로 사용한다. 구현 중 범위가 바뀌면 ID를 재사용하지 말고 새 항목을 추가한다.

### CAP-1. 직원 마스터와 월간 근무일수 집계

의도: 본사와 지점은 직원 정보를 관리하고, 일일 장부의 근무자 선택 기록을 월별 근무일수로 집계할 수 있어야 한다.

구현 요구:

- 직원 등록, 수정, 비활성 관리
- 직원 소속 지점 관리
- 일일 장부에서 근무자 선택
- 일자별 근무자 저장
- 월간 직원별 근무일수 자동 집계
- 지점별/직원별 근무 현황 조회
- 같은 직원이 여러 지점에서 근무한 기록 저장

검수 조건:

- 지점장이 일일 장부에서 근무자를 선택할 수 있다.
- 같은 직원이 여러 지점에서 근무해도 기록이 저장된다.
- 본사는 월별 직원별 근무일수를 볼 수 있다.

### CAP-2. 특수기간 비교 리포트

의도: 본사는 설날 같은 특수기간을 기준으로 지점 실적을 연도별로 비교할 수 있어야 한다.

구현 요구:

- 특수기간 템플릿 관리
- D-day 기준 비교
- 연도별 매출, 마진, 근무인원 비교
- 지점별 특수기간 리포트

검수 조건:

- 본사가 특수기간과 기준일을 등록할 수 있다.
- 본사는 선택한 특수기간의 연도별 지점 실적을 볼 수 있다.

### CAP-3. 엑셀 보고 양식 수준의 리포트 고도화

의도: 본사는 기존 엑셀 수준으로 기간, 월, 연도 기준의 지점 실적을 비교할 수 있어야 한다.

구현 요구:

- 대조 기간과 현재 기간 비교
- 연도별 지점 비교
- 월별 매출, 이익률, 생산성 추이
- 보고용 화면 또는 출력 양식
- 기존 엑셀 항목과 신규 리포트 항목 매칭

검수 조건:

- 본사가 기간 두 개를 선택해 지점별 증감을 볼 수 있다.
- 본사가 연도별/월별 지점 실적을 볼 수 있다.
- 숫자 기준은 기존 서버 계산 로직을 사용한다.

### CAP-4. 상품별 관리자 분석

의도: 본사는 장부별 또는 기간별로 상품 판매, 마진, 최고매출품목을 분석할 수 있어야 한다.

구현 요구:

- 품목규격 기준 분석
- 전재고, 매입, 판매원가 계산
- 상품별 이익률 계산
- 판매금액과 판매량 계산
- 최고매출품목 자동 산출
- 품목별 판매/마진 분석 화면

검수 조건:

- 본사는 장부별 상품 분석을 볼 수 있다.
- 최고매출품목이 자동 계산된다.
- 계산 기준이 불확실한 항목은 임의로 확정하지 않고 `기준 확인 필요`로 표시한다.

### CAP-5. 품목명/규격 정규화

의도: 시스템은 원문 품목명과 분석용 품목/규격을 분리해, 다르게 입력된 같은 상품을 하나로 묶어 볼 수 있어야 한다.

구현 요구:

- 원문 품목명 저장
- 품목명/규격 분리
- 품목 별칭 관리
- 매입 품목과 재고 품목 매핑
- 매핑 실패 검수 화면
- 과거 장부 원문 품목명 보존
- 이카운트 엑셀 업로드와 연결 가능한 구조

검수 조건:

- 본사는 매핑 실패 품목을 확인하고 직접 연결할 수 있다.
- 과거 장부 원문 품목명은 보존된다.
- 같은 상품이 다르게 입력되어도 분석용 품목으로 묶을 수 있다.

### CAP-6. 이카운트 엑셀 업로드 기반 매입 자동 생성

의도: 본사는 이카운트 엑셀을 업로드해 장부 매입 데이터를 자동 생성할 수 있어야 한다.

이 기능은 추가견적에는 포함하지 않는 무료 제공 항목이지만, 실제 구현 범위에는 포함한다.

구현 요구:

- 이카운트 엑셀 업로드
- 업로드 파일 검증
- 품목명, 규격, 단가, 수량 파싱
- 품목 매핑
- 본사 업로드 데이터를 장부 매입으로 자동 생성
- 지점 화면에는 읽기 전용으로 노출
- 업로드 이력과 감사 로그 저장

검수 조건:

- 본사는 이카운트 엑셀을 업로드할 수 있다.
- 업로드 실패 사유가 사용자에게 보인다.
- 업로드된 매입은 해당 지점/일자 장부에 반영된다.
- 지점장은 본사 업로드 매입을 임의 수정할 수 없다.

### CAP-7. FIFO 재고 금액 계산과 매입 잔량 이력

의도: 시스템은 같은 품목의 매입 단가가 바뀌어도 선입선출 방식으로 재고 금액과 매입 잔량 근거를 추적할 수 있어야 한다.

이 기능은 적용 대상이다. 다만 추가견적 항목에는 포함하지 않는다.

구현 요구:

- 매입 라인별 잔여 수량 관리
- FIFO 방식으로 재고 금액 계산
- 매입일, 단가, 수량, 잔량 저장
- 전일 재고 또는 재고 금액 클릭 시 근거 팝업
- 기본 1개월 조회와 기간 필터
- 손실, 정정, 재고 조정과 연결

검수 조건:

- 같은 품목의 단가가 여러 번 바뀌어도 잔량별 원가가 추적된다.
- 본사는 특정 재고 금액의 매입 근거를 볼 수 있다.
- FIFO 계산이 불가능한 데이터는 조용히 넘어가지 않고 `확인 필요`로 표시된다.

### CAP-8. 본사 통합 전체 재고 뷰

의도: 본사는 모든 지점의 품목별 재고와 전체 합계를 한 화면에서 확인할 수 있어야 한다.

구현 요구:

- 전체 지점 품목별 재고 집계
- 지점별 잔량 비교
- 전체 합계
- 냉동/생물/품목/기간 필터
- 본사 발주 판단용 화면

검수 조건:

- 본사는 특정 품목이 각 지점에 얼마나 남았는지 한 화면에서 본다.
- 본사는 전체 지점 합산 수량을 볼 수 있다.

### CAP-9. 인사/급여 정산

의도: 본사는 직원별 월 근무 내역과 급여 정산 참고 자료를 관리할 수 있어야 한다.

구현 요구:

- 직원 입사일 관리
- 일당제 근무 기록
- 지각, 조퇴, 특수 상황 메모
- 급여 차액 수동 입력
- 여러 지점 근무 통합
- 개인별 월급 정산 페이지

검수 조건:

- 본사는 직원별 월 근무 내역을 볼 수 있다.
- 본사는 급여 차액을 수동으로 입력할 수 있다.
- 여러 지점에서 근무한 직원의 월 정산이 합산된다.

### CAP-10. 본사 고정비 입력과 월 손익계산서

의도: 본사는 매장별 월 고정비를 입력하고 일일 비용과 구분해 월 손익을 계산할 수 있어야 한다.

구현 요구:

- 매장별 월 고정비 입력
- 고정비 항목 관리
- 일일 비용과 월 고정비 구분
- 월 손익계산서 계산
- 매장별 손익 비교

검수 조건:

- 본사는 매장별 월세, 세금 등 고정비를 입력할 수 있다.
- 월 손익계산서에는 일일 비용과 월 고정비가 구분된다.

### CAP-11. LINE 또는 텔레그램 자동 알림

의도: 시스템은 정해진 시간에 본사나 담당자에게 주요 운영 위험을 자동으로 알려야 한다.

LINE을 쓰는 경우 `LINE Notify`를 사용하지 않는다. 원문 기준으로 LINE Notify는 2025-03-31 종료되었고, LINE Official Account와 LINE Messaging API를 사용해야 한다.

구현 요구:

- LINE Messaging API 또는 텔레그램 API 연동
- LINE 사용 시 LINE Official Account 생성 및 Messaging API 활성화
- LINE channel access token 관리
- LINE 대상 `userId`, `groupId`, `roomId` 수집 및 저장
- 발송 대상과 채널 설정
- 정기 스케줄러
- 적자 지점, 목표 마진율 미달, 장기 체화 재고 조건 계산
- 발송 실패, 재시도, 로그
- 운영 토큰 보안 관리

검수 조건:

- 지정 시간에 요약 알림이 발송된다.
- 발송 실패가 로그에 남는다.
- 본사는 알림 조건을 확인할 수 있다.
- LINE 사용 시 알림 대상은 Official Account 친구이거나, 알림용 그룹/room에 Official Account가 참여 중이어야 한다.

참고 링크:

- LINE Notify 종료 공지: `https://developers.line.biz/en/news/2025/04/01/line-notify/`
- LINE Messaging API 시작 문서: `https://developers.line.biz/en/docs/messaging-api/getting-started/`
- LINE 메시지 발송 문서: `https://developers.line.biz/en/docs/messaging-api/sending-messages/`
- LINE Messaging API 요금 문서: `https://developers.line.biz/en/docs/messaging-api/pricing/`

### CAP-12. AI 기반 경영 분석 확장성

의도: 이번 기능을 만들 때 향후 분석에 필요한 데이터 구조는 남기되, AI 기능 자체는 구현하지 않는다.

구현 요구:

- 주요 변경/입력 이력을 구조화해서 저장
- 분석에 필요한 날짜, 지점, 직원, 품목, 금액 기준을 명확히 유지
- 자연어 AI 분석은 별도 고도화 범위로 분리

검수 조건:

- 새 기능이 중요한 운영 데이터를 텍스트 메모에만 묻어두지 않는다.
- 향후 분석 가능한 구조로 저장한다.
- AI 화면, AI API 호출, AI 프롬프트, AI 분석 결과 저장 기능이 추가되지 않는다.

### CAP-13. 지점장 민감 회계 지표 숨김

의도: 지점장 화면에서는 현장 운영에 필요한 값만 보여주고 민감한 회계 지표는 숨겨야 한다.

숨길 지표:

- 매출원가
- 매출이익
- 인당생산성
- 영업이익

지점장 화면에는 현장 입력값, 매출 마진율, 재고 금액 중심으로 보여준다. 권한 제한은 UI만이 아니라 서버에서도 강제한다.

검수 조건:

- 지점장 권한으로 민감 지표를 화면이나 API 응답에서 볼 수 없다.
- 본사 권한은 필요한 회계 지표를 볼 수 있다.

### CAP-14. 희망 판매가 기준 손실액 계산

의도: 손실액 계산 정책은 단순 매입원가가 아니라 영업 전 희망 판매가와 실제 처리 금액의 차이를 반영할 수 있어야 한다.

필요 데이터:

- 품목별 희망 판매가
- 실제 판매/떨이/폐기 처리 금액
- 손실 사유

검수 조건:

- 희망 판매가 입력 주체와 입력 시점이 정해지기 전에는 계산 정책을 임의 확정하지 않는다.
- 손실액 계산 기준이 화면과 리포트에서 일관되게 적용된다.

### CAP-15. 본사 일괄 마감 또는 검토 생략 토글

의도: 본사는 지점 장부를 개별 검토하지 않고 바로 마감할 수 있는 옵션을 가져야 한다.

구현 요구:

- 본사 권한만 사용 가능
- 일괄 마감 전 위험 알림 표시
- 감사 로그 저장
- 기존 마감 후 원본 수정 제한 유지

검수 조건:

- 지점장 권한은 일괄 마감을 실행할 수 없다.
- 일괄 마감 실행 내역이 감사 로그에 남는다.

### CAP-16. 작성자 이름 세션 캐싱

의도: 장부 작성 과정에서 1단계에 입력한 작성자 이름이 7단계 전체 저장까지 유지되어야 한다.

검수 조건:

- 작성자 이름을 한 번 입력하면 같은 장부 저장 흐름 안에서 다시 입력하지 않아도 된다.
- 최종 저장 데이터에는 작성자명이 누락되지 않는다.

### CAP-17. 재고 테이블 문구 변경

의도: 기존 `차이` 항목 명칭을 사용자가 이해하기 쉬운 `당일 판매량`으로 바꿔야 한다.

검수 조건:

- 단순 라벨 변경인지 계산 의미 변경인지 먼저 확인한다.
- 계산 의미가 바뀌면 관련 리포트와 이상 신호 계산도 함께 맞춘다.

### CAP-18. 대시보드 그리드 리사이징

의도: 본사는 대시보드 테이블의 컬럼 또는 영역 크기를 업무에 맞게 조절할 수 있어야 한다.

검수 조건:

- 본사 대시보드에서 대상 컬럼 또는 영역 크기를 조절할 수 있다.
- 조절 기능이 주요 테이블 데이터 확인을 방해하지 않는다.

## 5. 구현 우선순위

권장 순서는 데이터 의존성이 큰 작업부터 잡는 방식이다.

1. 품목명/규격 정규화
2. 이카운트 업로드
3. FIFO 재고 금액 계산
4. 매입 잔량 이력 팝업
5. 본사 통합 전체 재고 뷰
6. 지점장 민감 지표 숨김 및 매입 수정 제한
7. 직원 마스터와 월간 근무일수 집계
8. 리포트 고도화와 특수기간 비교
9. 본사 고정비와 월 손익계산서
10. 인사/급여 정산
11. LINE 또는 텔레그램 알림

구현 에이전트는 한 번에 전부 만들지 말고, 위 순서를 기준으로 작은 작업 단위와 검수 기준을 나눈다.

## 6. 공통 제약

- 기존 PRD와 구현된 기능을 무시하지 않는다.
- 기존 장부 원본값을 덮어쓰지 않는다.
- 본사 마감 후 원본 수정은 계속 막아야 한다.
- 본사 수정, 정정, 업로드, 일괄 마감은 감사 로그를 남긴다.
- 지점장 권한 제한은 UI가 아니라 서버에서 강제한다.
- 엑셀 수식을 그대로 복사하지 말고 서버 계산 로직으로 재정의한다.
- 계산 기준이 불명확한 항목은 임의로 확정하지 않는다.
- 외부 연동 기능은 실패, 재시도, 로그를 반드시 포함한다.
- 운영 토큰과 외부 API 키는 안전하게 관리한다.
- LINE 알림은 LINE Notify가 아니라 LINE Messaging API와 LINE Official Account 기준으로 검토한다.
- AI 분석 기능은 이번 작업 범위가 아니다.

## 7. 제외 범위

이번 범위에서 제외한다.

- AI 분석 기능 구현
- AI 챗봇, 자연어 질의, 자동 분석 화면, AI 요약 기능
- POS, 카드사, 은행 자동 연동
- 과거 전체 엑셀 데이터 일괄 이관
- 서버 이용료
- LINE 공식 계정, 텔레그램 등 외부 서비스 운영 비용
- 오픈 후 장기 유지보수 비용

AI 분석은 향후 데이터가 충분히 쌓인 뒤 별도 고도화 단계에서 논의한다.

## 9. 구현 전 확인 질문

아래 질문은 구현 중 임의로 확정하면 안 된다.

- 이카운트 엑셀 양식은 고정인가, 바뀔 수 있는가?
- 품목명/규격 매핑은 본사가 직접 관리할 수 있어야 하는가?
- FIFO 계산은 모든 품목에 적용하는가, 일부 품목에만 적용하는가?
- 희망 판매가는 누가 언제 입력하는가?
- 지점장에게 보여줄 마진율은 어떤 기준인가?
- 특수기간 비교는 설날만 필요한가, 다른 명절/행사도 필요한가?
- 급여 정산은 실제 지급 확정까지 포함하는가, 참고 자료만 만드는가?

- `차이`를 `당일 판매량`으로 바꾸는 것은 라벨 변경인가, 계산 의미 변경인가?

## 10. 에이전트 작업 지침

구현 에이전트는 다음 순서로 작업한다.

1. 현재 코드와 데이터 모델을 먼저 확인한다.
2. 이미 구현된 MVP 기능과 PRD 내용을 기준으로 중복 구현을 피한다.
3. 기능마다 서버 권한, 감사 로그, 마감 후 수정 제한을 함께 확인한다.
4. 계산 기능은 테스트 가능한 단위로 분리한다.
5. 계산 기준이 불명확하면 `기준 확인 필요` 상태를 만들고 임의 계산을 넣지 않는다.
6. 엑셀 원본값은 보존하고, 분석용 정규화 값은 별도 필드나 매핑 구조로 둔다.
7. 외부 연동은 실제 발송 성공뿐 아니라 실패, 재시도, 로그까지 검수한다.
8. 기능 완료 보고 시 해당 `CAP-*` ID와 검수 조건 통과 여부를 함께 적는다.

## 11. 전체 성공 신호

본사는 ERP Fish에서 지점별 장부를 확인하는 수준을 넘어, 품목/재고/직원/리포트/월 손익을 운영 판단에 쓸 수 있다. 지점장은 본인 역할에 필요한 입력과 재고 중심 정보만 보며, 본사 업로드 매입과 민감 회계 지표는 권한으로 보호된다.

이번 작업이 끝났다고 말하려면 다음을 보여줘야 한다.

- 우선순위에 맞춰 구현된 기능의 `CAP-*` 검수 조건이 통과한다.
- 원본 장부값, 마감 상태, 감사 로그, 권한 제한이 깨지지 않는다.
- 이카운트 업로드, FIFO, 품목 정규화가 서로 충돌하지 않는다.
- AI 기능은 추가되지 않았고, 향후 분석 가능한 구조만 유지된다.
