# Product Brief: ERP Fish

## 목적

ERP Fish는 원드라이브 엑셀로 운영 중인 매장 일일 장부를 웹 기반 내부 ERP로 전환하는 제품이다. 이 브리프의 목적은 내부 개발 착수 전에 “무엇을 왜 만들지”를 개발팀이 같은 그림으로 이해하게 하는 것이다.

핵심은 엑셀 화면을 그대로 복제하는 것이 아니다. 현재 엑셀이 맡고 있는 `일일 마감 -> 재고 이월 -> 월간 근무/영업 집계 -> 본사 분석 리포트` 흐름을 데이터 모델, 계산 로직, 권한, 검증 규칙으로 다시 정의하는 것이다.

## 문제

현재 지점장은 퇴근 전 원드라이브 공유 엑셀에 당일 매출, 결제, 재고, 매입, 손실, 근무자 정보를 입력한다. 이 방식은 수식이 깨지기 쉽고, 항목 추가가 어렵고, 지점장이 제출 후 값을 바꿔도 통제하기 어렵다.

대표와 본사는 매일 아침 모든 지점 장부를 직접 확인한다. 매출차액, 이익률 급락, 과재고, 폐기/떨이 같은 문제를 찾아야 하지만, 모든 매장을 전수 확인하는 방식이라 시간이 많이 들고 중요한 신호를 놓칠 수 있다.

생선 유통 특성도 문제를 키운다. 같은 박스라도 마릿수와 신선도가 다르고, 폐기와 떨이가 자주 생기며, 이카운트 같은 일반 ERP만으로는 실제 매장 재고와 장부 재고를 맞추기 어렵다. 그 결과 지점장이 평균값으로 재고를 기록하거나, 손실 사유가 표준화되지 않은 채 남는다.

## 사용자

1차 사용자는 지점장과 본사 관리자다. 지점장은 모바일, 태블릿, PC에서 당일 장부를 입력하고 마감 제출한다. 본사 관리자는 지점별 마감 상태, 문제 매장, 장부 상세, 수정 요청, 검토 메모를 확인한다.

보조 사용자는 본사 스텝과 직원이다. 본사 스텝은 매입 입력 또는 수정 권한을 가진다. 직원은 우선 로그인 사용자라기보다 근무자 마스터로 관리하며, 근무일수와 인당생산성 계산에 쓰인다.

## 1차 제품 범위

1차 범위는 “매일 마감이 깨지지 않고 쌓이는 시스템”이다.

- 매장, 직원, 상품, 손실 유형 기준정보 관리
- 지점장 계정 기반 일일 장부 생성 및 입력
- 현금, 카드, 기타매출, 비용, 근무자 입력
- 냉동/생물 상품 재고 입력과 재고금액 계산
- 당일 매입 라인 입력과 합계 계산
- 떨이/폐기 손실 입력, 손실 유형 코드화, 사유 필수화
- 전일 마감 재고의 자동 이월
- 총매출, 매출원가, 매출이익, 이익률, 영업이익, 인당생산성 자동 계산
- 매출 정합성, 원가 정합성, 상품 분석 합계 차이, 재고 이월 정합성 검증
- 매장/일자 중복 방지, 빈 매장 차단, 미작성일과 휴무일 구분
- 매출 0일의 이익/재고 계산 차단 또는 별도 상태 처리
- 마감 제출, 제출 후 잠금, 본사 수정 요청/승인 흐름
- 본사 관리자용 일일 마감 현황과 문제 매장 필터
- 장부별 본사 검토 메모와 지점 확인 상태

## 명시적 제외 또는 후순위

1차 개발에서는 텔레그램 알림, 이카운트 연동, POS/카드 매출 연동, 입고일 기반 신선도 추적, AI 진열 이미지 식별을 기본 범위에서 제외한다. 고객 요구 문서에는 텔레그램 알림이 포함되어 있으나, 기능 분석 문서에서는 확장 기능으로 분류되어 있어 우선순위 확인이 필요하다.

과거 엑셀 데이터 이관도 “신규 마감 시스템이 안정화된 뒤 검증 범위를 정해 진행”하는 후순위 작업으로 본다. 과거 원장에는 현금, 카드, 최고매출품목, 매출차액 누락 구간이 있어 단순 이관하면 분석값을 오해할 수 있다.

## 성공 기준

- 지점장이 하루 장부를 엑셀 없이 제출할 수 있다.
- 제출된 장부는 잠기며, 사후 수정은 본사 승인 흐름을 거친다.
- 다음 영업일 장부에 전일 재고가 자동 반영된다.
- 본사는 전체 매장을 모두 열어보지 않아도 문제 매장을 먼저 볼 수 있다.
- 매출, 원가, 재고, 손실 계산식이 엑셀 수식이 아니라 서버 로직으로 재현된다.
- 폐기/떨이, 재고 조정, 매출차액 같은 문제 항목에 사유와 상태가 남는다.
- 신규 입력 데이터는 상품, 매장, 직원, 손실 유형 기준정보와 연결된다.

## 개발 착수 리스크

가장 큰 리스크는 범위다. 2~3주, 150만 원 가견적 안에 일일 장부, 재고수불, 권한, 검증, 대시보드, 알림, 외부 연동을 모두 넣으면 실패 가능성이 높다. 1차는 일일 마감과 본사 검증에 집중해야 한다.

두 번째 리스크는 계산 정책이다. `30%단가`, `전일 이월금액`, `매출차액 허용 기준`, 손실액 저장 방식, 상품별 판매량 산출 방식이 확정되지 않으면 데이터 모델과 계산 로직이 흔들린다.

세 번째 리스크는 상품 정규화다. 실제 장부에는 품목명과 규격 표기가 흔들리므로, 상품 alias 또는 변환 규칙이 없으면 상품별 매출, 손실, 과재고 분석이 틀어질 수 있다.

네 번째 리스크는 조용한 오류다. 실제 장부는 `IFERROR`, `VLOOKUP`, 품목명+규격 문자열 결합에 의존한다. 값이 안 맞아도 0으로 처리될 수 있으므로, 신규 시스템은 “계산 실패를 숨기지 않고 검증 상태로 드러내는 것”을 제품 요구사항으로 둬야 한다.

## 개발 전 확정 질문

1. `30%단가`는 고정 마진 30% 판매가인가, 참고 판매가인가?
2. `전일 이월금액`은 총매출에 어떤 회계 의미로 포함되는가?
3. `매출차액`은 얼마까지 허용할 것인가? 금액 기준인가, 비율 기준인가?
4. 매입 입력은 항상 본사 스텝만 하는가, 지점장 입력도 허용하는가?
5. 폐기/떨이 손실액은 양수 금액과 손실 유형으로 저장할 것인가, 음수 조정으로 저장할 것인가?
6. 상품별 판매량은 엑셀처럼 원가/단가로 추정할 것인가, POS 연동 전까지는 판매금액 중심으로 볼 것인가?
7. 텔레그램 알림은 1차 필수인가, 2차 확장인가?
8. 매출이 없는 날은 휴무, 미작성, 0원 영업일 중 어떤 상태로 구분할 것인가?
