# 수산물 유통 및 매장 통합 관리 시스템 컨셉

## 1. 에이전트 핸드오프 요약

- 제품 유형: 수산물 유통 본사와 직영/가맹 매장을 위한 웹 기반 운영 ERP.
- 핵심 문제: 매장별 엑셀 장부를 대표님이 매일 수작업으로 확인하면서 시간, 오류, 누락 리스크가 크다.
- 핵심 목표: `One-Screen, Exception-First` 방식으로 정상 매장은 요약하고 이상 매장만 즉시 확인하게 하여 아침 모니터링 시간을 크게 줄인다.
- MVP 초점: 일일 마감 입력, 예외 알림 대시보드, 수산물 특화 재고 보정, 매출/실정산 오차 처리, 권한/잠금, 일일 요약 리포트.
- 중요한 도메인 전제: 수산물 재고는 박스, 마리 수, 중량, 규격이 섞이고 감모/폐기/오기재가 자주 발생하므로 일반 ERP식 정형 재고 모델만으로는 부족하다.
- 후속 에이전트 주의점: 이 문서는 컨셉/요구사항 초안이다. 수치, 임계값, 승인 정책, 외부 연동 방식은 PRD 또는 아키텍처 단계에서 확인해야 한다.

## 2. 배경

현재 본사 및 여러 직영/가맹 매장은 OneDrive로 공유된 개별 엑셀 장부를 사용해 일일 마감을 진행한다. 대표님은 매일 아침 각 매장의 장부를 직접 열고 스크롤하며 매출, 재고, 폐기, 정산 이상 여부를 확인한다.

기존 일반 ERP는 수산물 매장의 비규격 재고, 일일 매출과 실정산액의 불일치, 매장별 마감 항목 변경 같은 운영 현실을 충분히 반영하지 못한다. 따라서 기존 엑셀 장부의 유연함은 유지하되, 본사 관점의 통합 관리와 예외 감지를 자동화하는 전용 시스템이 필요하다.

## 3. 사용자와 권한

| 역할 | 주요 목적 | 접근 범위 |
| --- | --- | --- |
| 대표님 | 전체 매장 예외 확인, 손익/재고 흐름 파악, 지시사항 전달 | 전체 매장 데이터, 통계, 예외 알림, 리포트 |
| 본사 스태프 | 매입/출고 데이터 관리, 장부 항목 관리, 예외 검토 지원 | 담당 업무 범위 내 전체 또는 일부 매장 |
| 지점장/매장 직원 | 일일 매출, 재고, 폐기, 정산 데이터 입력 | 본인 매장의 입력/조회 |
| 시스템 관리자 | 사용자, 권한, 마감 항목, 외부 연동 관리 | 시스템 설정 전체 |

## 4. 문제 정의

### PP-001. 매장별 장부 확인 비용이 과도함

대표님이 매일 아침 7개 이상의 엑셀 파일을 개별적으로 열어 이상 여부를 확인해야 한다. 매장이 13개 이상으로 늘어날 경우 확인 비용은 더 커진다.

### PP-002. 수산물 재고가 정형 단위로 맞지 않음

입고는 박스 단위, 판매와 가공은 마리 수 또는 중량 단위로 이루어진다. 참조기, 제주갈치처럼 규격과 실제 마리 수가 달라지는 품목은 잔여 재고가 `0.0001개`처럼 비현실적인 값으로 남거나 데이터가 유실될 수 있다.

### PP-003. 매출액과 실제 정산액의 차이를 추적하기 어려움

POS 또는 장부상 매출액과 마감 후 실제 현금/카드 정산액 사이에 차이가 발생한다. 현재는 오차 금액과 원인, 승인 여부를 체계적으로 기록하기 어렵다.

### PP-004. 엑셀 장부 변경과 유지보수가 취약함

새 항목 추가나 수식 수정이 필요할 때 모든 매장의 엑셀 폼을 수작업으로 바꿔야 한다. 매장별 포맷 불일치와 수식 오류가 발생하기 쉽다.

## 5. 성공 기준

| ID | 성공 기준 | 목표값 |
| --- | --- | --- |
| SC-001 | 대표님 아침 장부 확인 시간 | 하루 1~2시간에서 5분 이내로 단축 |
| SC-002 | 이상 매장 식별 방식 | 정상 매장은 요약, 이상 매장은 상단 알림으로 노출 |
| SC-003 | 재고 보정 추적성 | 모든 강제 보정에 사유, 작성자, 시각, 변경 전후 값 기록 |
| SC-004 | 매출/정산 오차 추적성 | 오차 금액, 사유, 승인 상태를 일일 마감 단위로 보관 |
| SC-005 | 장부 항목 변경 비용 | 본사 관리자 페이지에서 일괄 반영 |

## 6. 범위

### MVP 범위

- 웹 기반 일일 마감 입력 화면.
- 본사 예외 우선 대시보드.
- 수산물 특화 재고 조정과 사유 입력.
- 매출액과 실정산액 오차 기록 및 승인 요청.
- 마감 후 수정 잠금과 권한 제어.
- 일일 요약 리포트 자동 발송.
- 본사에서 마감 입력 항목을 일괄 관리하는 설정 화면.

### 후속 또는 선택 범위

- 이카운트(E-Count) 매입/출고 데이터 연동.
- 5개월 이상 누적 데이터 기반 반복 이상 패턴 분석.
- 매장 진열 사진 아카이브와 AI 키워드 검색/태깅.

### 명시적 비범위 초안

- 완전한 회계 ERP 대체.
- POS 자체의 결제 기능 대체.
- AI 사진 검색을 MVP 필수 기능으로 포함.
- 매장별 임의 엑셀 양식을 계속 유지하면서 모든 기능을 병행 지원.

## 7. 핵심 워크플로우

### WF-001. 매장 일일 마감

1. 지점장 또는 매장 직원이 모바일 또는 PC 브라우저로 로그인한다.
2. 당일 매출, 카드 결제액, 현금 시제, 재고, 폐기, 특이사항을 입력한다.
3. 시스템은 필수 항목, 숫자 형식, 재고 잔량, 정산 오차를 검증한다.
4. 이상이 있으면 사용자가 사유를 입력하거나 본사 승인 요청을 남긴다.
5. 마감을 제출하면 해당 일자의 장부는 잠금 상태가 된다.

### WF-002. 본사 아침 예외 검토

1. 대표님 또는 본사 스태프가 통합 대시보드에 접속한다.
2. 시스템은 정상 매장을 접고, 이상 매장을 상단에 우선 노출한다.
3. 사용자는 매출/정산 오차, 마진율 급감, 재고 불일치, 미마감 매장을 확인한다.
4. 필요한 경우 매장별 상세 화면에서 원인, 조정 내역, 승인 요청을 확인한다.

### WF-003. 재고 강제 보정

1. 사용자가 품목별 잔여 재고 불일치를 확인한다.
2. 보정 전 수량, 보정 후 수량, 사유 유형, 상세 메모를 입력한다.
3. 시스템은 보정 이력을 감사 로그로 저장한다.
4. 보정 결과는 다음 마감 또는 기초 재고에 반영된다.

### WF-004. 매출/실정산 오차 승인

1. 시스템이 장부상 매출액과 실제 카드 결제액 및 현금 시제 합계를 비교한다.
2. 허용 범위를 넘는 오차는 예외로 표시된다.
3. 지점장은 사유를 입력해 본사에 승인 요청을 보낸다.
4. 본사는 승인, 반려, 추가 확인 요청 중 하나로 처리한다.

### WF-005. 일일 요약 리포트

1. 마감 및 분석이 완료되면 시스템이 매장별 요약 데이터를 집계한다.
2. 대표님에게 전체 매장 요약, 이상 매장, 오차 금액, 주요 지시사항을 발송한다.
3. 지점장에게는 본인 매장의 결과와 조치 필요 사항만 발송한다.

## 8. 기능 요구사항

### FR-001. 웹 기반 일일 마감 입력

- 매장 사용자는 모바일 또는 PC 브라우저에서 일일 마감 데이터를 입력할 수 있어야 한다.
- 입력 항목은 최소한 매출, 카드 결제액, 현금 시제, 재고, 폐기, 특이사항을 포함한다.
- 본사 관리자는 입력 항목과 노출 순서를 일괄 제어할 수 있어야 한다.

### FR-002. 예외 우선 대시보드

- 정상 매장은 요약 상태로 표시해야 한다.
- 이상 매장은 대시보드 상단에 우선 노출해야 한다.
- 이상 유형은 최소한 마진율 급감, 매출/실정산 오차 과다, 재고 불일치, 미마감을 포함한다.

### FR-003. 수산물 특화 재고 단위 관리

- 품목은 박스, 마리 수, 중량, 규격 정보를 함께 다룰 수 있어야 한다.
- 시스템은 입고 단위와 판매/가공 단위가 다른 경우를 지원해야 한다.
- 소수점 잔량 같은 비현실적 수치가 발생할 수 있음을 전제로 보정 기능을 제공해야 한다.

### FR-004. 재고 보정 및 사유 기록

- 사용자는 잔여 재고가 맞지 않을 때 강제 보정을 할 수 있어야 한다.
- 모든 보정에는 사유 유형과 상세 메모 입력이 필요하다.
- 사유 유형의 초기 후보는 감모, 폐기, 오기재, 규격 차이, 기타다.
- 시스템은 보정 전 값, 보정 후 값, 작성자, 작성 시각을 저장해야 한다.

### FR-005. 이카운트 매입/출고 데이터 연동

- 본사 유통 부서가 사용하는 이카운트의 매입/출고 데이터를 가져올 수 있어야 한다.
- 가져온 데이터는 매장별 기초 재고 또는 입출고 내역과 매칭되어야 한다.
- 실제 연동 방식은 API, 파일 업로드, 엑셀 내보내기 중 가능한 방식을 확인해야 한다.

### FR-006. 매출액과 실정산액 오차 처리

- 시스템은 점장 장부상 매출액과 실제 카드 결제액 및 현금 시제 합계를 비교해야 한다.
- 오차 금액은 마감 화면에 명확히 표시되어야 한다.
- 허용 범위를 넘는 마이너스 또는 플러스 오차는 사유 입력과 본사 승인 요청이 가능해야 한다.

### FR-007. 자동 분석과 반복 이상 알림

- 누적 데이터가 충분할 경우 품목별, 매장별 반복 이상 패턴을 탐지해야 한다.
- 예: 특정 품목에서 반복적으로 수량 차이가 발생하면 거래처 컴플레인 또는 매장 관리 이슈 후보로 표시한다.
- 초기 분석 기준은 규칙 기반으로 시작하고, 누적 데이터 이후 고도화할 수 있다.

### FR-008. 일일 리포트 자동 발송

- 매일 아침 마감 및 분석이 완료되면 요약 리포트를 자동 발송해야 한다.
- 발송 채널 후보는 텔레그램, 알림톡, 문자, 이메일이다.
- 리포트는 매출, 오차 금액, 이상 유형, 주요 지시사항을 포함해야 한다.

### FR-009. 권한과 마감 잠금

- 지점장은 본인 매장의 데이터만 입력/조회할 수 있어야 한다.
- 대표님과 본사 스태프는 권한 범위에 따라 전체 또는 일부 매장을 조회할 수 있어야 한다.
- 마감 제출 후에는 임의 수정이 잠기며, 수정이 필요한 경우 승인 또는 보정 프로세스를 거쳐야 한다.

### FR-010. 진열 사진 아카이브 및 AI 검색

- 매장별, 날짜별 진열 사진을 아카이브할 수 있어야 한다.
- AI 모델 또는 태깅 기능을 통해 병어, 갈치 등 품목 키워드로 과거 진열 사례를 검색할 수 있어야 한다.
- 이 기능은 교육용 아카이브 성격이 강하므로 MVP 이후 선택 기능으로 둔다.

## 9. 비기능 요구사항

### NFR-001. 사용 편의성

- 매장 직원의 연령대와 IT 숙련도를 고려해 입력 흐름을 단순하게 설계해야 한다.
- 복잡한 엑셀 수식, 영어 메뉴, 숨겨진 탭 구조를 피한다.
- 숫자 입력, 확인, 제출 중심의 화면을 우선한다.

### NFR-002. 보안과 권한 제어

- 역할 기반 권한 제어(RBAC)를 적용한다.
- 매장 사용자는 본인 매장 데이터만 접근 가능해야 한다.
- 본사 관리자만 전체 통계, 예외 목록, 시스템 설정에 접근할 수 있어야 한다.

### NFR-003. 감사 가능성

- 마감 제출, 수정 요청, 승인, 반려, 재고 보정, 정산 오차 처리 이력을 추적해야 한다.
- 잠금 이후 변경은 원본 덮어쓰기보다 이력 기반 보정으로 처리한다.

### NFR-004. 안정성과 접근성

- 클라우드 기반으로 운영하여 특정 PC 장애나 정전에 의존하지 않도록 한다.
- 모바일, 태블릿, PC에서 주요 마감 입력과 확인이 가능해야 한다.

### NFR-005. 확장성

- 매장 수가 7개 수준에서 13개 이상으로 늘어나는 상황을 고려한다.
- 마감 항목, 품목, 사유 유형, 알림 임계값은 코드 수정 없이 관리 화면에서 조정 가능해야 한다.
