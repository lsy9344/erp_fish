## CAP-15 preflight 계약

### 기본 원칙

CAP-15 일괄 마감 또는 검토 생략 토글은 개별 마감보다 느슨한 예외를 허용하지 않는다. 후속 구현은 현재 `runHqLedgerClosePreflight`와 `closeHqLedger` 계약을 기준으로 삼고, 별도의 loose preflight를 만들지 않는다.

| 항목 | 계약 |
| --- | --- |
| 권한 | 모든 dry run과 실행은 본사 마감 권한 및 ledger scope 확인 후 수행한다. 권한 확인 전 상세 실패 사유를 반환하지 않는다. |
| dry run 입력 | 대상 장부 목록, 요청자, 실행 모드(`dryRun: true`), 요청 시각, 선택 필터, 선택 사유 후보를 포함한다. |
| dry run 출력 | 장부별 실행 가능 여부, `blocking`/`warning`/`exception-allowed`/`info` 분류, 실패 row 사유, 재시도 가능 여부, summary count를 반환한다. |
| 실행 입력 | dry run 기준 대상 목록, 대상별 stale `updatedAt` token 또는 동등 version, 예외 사유, 승인자/실행자, 실행 시각을 포함한다. |
| stale token | 실행 직전 장부 `updatedAt` 또는 version이 dry run 이후 변경되면 해당 장부는 실패 처리한다. 전체 batch를 자동 rollback하지 않는다. |
| 사유 입력 | `exception-allowed` 항목이 있으면 명시적 사유가 필요하다. 사유는 장부 단위 또는 batch 공통 사유를 허용할 수 있으나 audit payload에서 어떤 장부에 적용되었는지 추적 가능해야 한다. |
| 감사 이벤트 | dry run 요청, 실행 요청, 장부별 성공, 장부별 실패, 부분 실패 summary, 예외 사유, 실행자, 대상 장부 목록, 시각, preflight summary를 감사 이벤트 또는 연결 가능한 audit evidence로 남긴다. |
| 실패 row 분류 | 검증 오류, 이월 공백, 가격 기준 없음, 미확정 정정, 재고 조정 정책 위반, 이상 신호 차단은 PRD §4.5 마감 가능성 표와 현재 close preflight severity를 따른다. |
| 재시도 범위 | 실패한 장부만 재시도 대상으로 남긴다. 이미 성공한 장부는 재실행하지 않으며, 성공 건을 되돌리는 기능은 제공하지 않는다. |

### 부분 실패 처리

일괄 마감은 장부 단위 독립 처리다. 일부 장부가 성공한 뒤 다른 장부가 실패해도 성공 건을 자동 rollback하지 않는다. 실패 summary에는 실패 장부, 실패 사유, stale token 여부, 권한/scope 실패 여부, 재시도 가능 여부를 포함한다.

성공한 장부는 기존 HQ close와 동일하게 원본 장부를 잠그고, 이후 오류는 FR-20 append-only 정정 기록으로 처리한다. 잘못 마감한 장부를 silent reopen, bulk rollback, 원본 overwrite로 되돌리는 기능은 CAP-15 범위에 포함하지 않는다.

### 검토 생략 토글

검토 생략 토글이 후속 구현 story에 포함되더라도 preflight를 생략한다는 뜻이 아니다. 토글은 UI 흐름의 확인 단계를 줄일 수 있을 뿐이며, 서버 권한 확인, close preflight 재계산, blocking 차단, stale token 확인, 감사 이벤트, 예외 사유 요구는 그대로 유지한다.

## CAP-16 표시명 기준

### 값 분리

`authorDisplayName`은 장부 입력 화면에 표시하고 입력 흐름에서 유지하기 위한 편의 값이다. 인증 계정과 감사 actor의 source of truth가 아니다.

| 값 | 의미 | 대체 가능 여부 |
| --- | --- | --- |
| `authorDisplayName` | 화면 표시와 현장 작성자 식별 편의값 | 인증 계정, 제출자, 수정자, 마감자, 감사 actor를 대체할 수 없음 |
| 인증 계정 | 로그인한 사용자 계정과 권한 scope | 표시명으로 대체 불가 |
| `submittedById` | 제출한 인증 사용자 id | 표시명으로 대체 불가 |
| `updatedById` | 마지막 수정 인증 사용자 id | 표시명으로 대체 불가 |
| `closedById` | 본사 마감 인증 사용자 id | 표시명으로 대체 불가 |
| 감사 `actorId` | audit event 실행자 id | 표시명 캐시로 대체 불가 |

감사 로그, 권한 판단, 본사 마감자 기록은 항상 인증 actor id를 기준으로 한다. 표시명은 감사 payload에 보조 설명으로 들어갈 수 있지만 actor identity가 될 수 없다.

### 7단계 흐름 적용 범위

현재 `/app/store-entry`는 sales/cost/purchase/work/review와 inventory/losses 하위 route가 나뉘어 있다. CAP-16 후속 구현 story는 이 route 구조를 기준으로 유지 범위를 명확히 해야 한다.

| 범위 | 현재 기준 | 후속 구현에서 확정할 항목 |
| --- | --- | --- |
| sales/cost/purchase/work/review | 현재 저장된 ledger의 `authorDisplayName`을 각 단계 표시와 저장 상태에 사용한다. | 같은 장부 내 단계 이동, partial save, review return에서 값 유지 기준 |
| inventory/losses 하위 route | route가 분리되어 있으므로 동일 장부 작성자 표시명을 어떻게 읽고 보존할지 별도 기준 필요 | route 간 cache/server 저장 위치, 빈 값 처리, stale 값 갱신 기준 |
| 브라우저 세션 | 편의 cache 후보일 뿐 source of truth 아님 | sessionStorage/localStorage/server 저장 여부, 공동 기기 만료 기준 |
| 서버 저장 | 장부 row의 표시명 값 | 인증 actor와 분리한 validation, 수정 권한, 감사 payload 표현 |

### 공동 기기와 대리 입력

공동 기기 또는 대리 입력에서 표시명과 인증 계정이 다를 수 있다. 이 경우 장부 상세, 검토, 감사 화면은 최소한 다음을 추적 가능하게 해야 한다.

- 화면 표시 작성자: `authorDisplayName`
- 실제 인증 계정: `submittedById`, `updatedById`, `closedById`
- 감사 실행자: audit `actorId`
- 변경 시각과 변경 action

표시명이 비어 있거나 다른 사람 이름으로 남아 있어도 권한은 인증 계정으로만 판단한다. 후속 구현은 공동 기기에서 오래된 표시명이 자동으로 다음 장부에 오염되지 않도록 저장 위치와 만료 기준을 정의해야 한다.

## CAP-17 용어 결정

### Story 7.7 baseline

Story 7.7 정책 산출물의 결론을 Story 8.5의 baseline으로 둔다.

| 항목 | 결정 |
| --- | --- |
| MVP/CAP-17 범위 | 단순 라벨 변경 우선 |
| `당일 판매량` 의미 | 실제 POS 판매 수량이 아니라 기존 재고 `차이` 수량의 현장 친화 표시 용어 |
| 변경하지 않는 값 | `differenceQuantity`, `differenceAmount`, `salesDifference`, DB field, Prisma schema, 감사 action 이름, 기존 계산식 |
| 도움말 필요성 | `당일 판매량`이 실제 POS 판매 수량으로 오해되지 않도록 tooltip/help text를 제공해야 함 |
| 계산 의미 변경 | OQ-14 승인과 별도 계산/리포트/이상 신호/손실/조정 정책 전에는 구현 story로 승격하지 않음 |

권장 도움말 문구는 Story 7.7 기준을 따른다: 전일재고, 매입, 손실을 반영한 기준재고와 입력한 당일재고의 차이이며 실제 POS 판매 수량과 다를 수 있다는 점을 설명한다.

### 계산 의미 변경 차단

`당일 판매량`을 실제 판매 수량으로 재정의하려면 판매, 손실, 폐기/떨이, 조정, 반품, 이월 순서와 source of truth를 별도 정책으로 확정해야 한다. OQ-14와 별도 계산 정책이 닫히기 전에는 다음을 하지 않는다.

- `differenceQuantity = afterQuantity - beforeQuantity` 의미 변경
- `differenceAmount` 의미 변경
- `salesDifference` 라벨 또는 계산식 변경
- DB field 또는 Prisma schema 변경
- 이상 신호 threshold 재정의
- 리포트/export 계산 의미 변경

## CAP-18 완료 기준

### 대상 surface

CAP-18 리사이징 데이터 테이블은 데스크탑 본사 업무용 표에 한정한다.

| 포함 surface | 제외 surface |
| --- | --- |
| 본사 관제판, 전체 재고, 상품 분석, 본사 리포트 drilldown 표 | 지점장 모바일 장부 입력, 390px 모바일 단계형 입력, 카드형 모바일 관제판, 권한/민감 필드 차단 로직 |

지점장 장부 입력의 390px 모바일 지원 기준은 유지한다. 모바일에서는 리사이징보다 기본 반응형 배치, 카드형 요약, 가로 스크롤 최소화가 우선이다.

### 핵심 컬럼 최소 폭

핵심 컬럼은 숨길 수 없고 아래 최소 폭 아래로 줄일 수 없다.

| 컬럼 유형 | 최소 폭 | 최대/권장 기준 |
| --- | ---: | --- |
| 지점명 | 160px | 최대 280px 권장 |
| 장부 상태 | 96px | 상태 배지 전체 읽기 가능 |
| 마감 상태 | 96px | 상태 배지 전체 읽기 가능 |
| 이상 신호 | 160px | 주요 warning label 읽기 가능 |
| 최신 반영 시각 | 132px | 날짜/시각 축약 없이 읽기 가능 |
| 숫자 컬럼 | 112px | 부호와 단위가 겹치지 않아야 함 |
| 금액 컬럼 | 128px | 천 단위 구분과 원 단위 표시가 잘리지 않아야 함 |

### 키보드와 보조기기 조작

리사이징 핸들은 다음 완료 기준을 모두 만족해야 한다.

- 포커스 가능해야 한다.
- `role="separator"`를 사용한다.
- 현재 폭, 최소 폭, 최대 폭을 보조기기에 전달한다.
- 좌/우 화살표로 8px 단위 조정한다.
- `Shift`+화살표로 24px 단위 조정한다.
- hover/focus 상태를 제공하되 텍스트, 숫자, 상태 배지와 겹치지 않는다.
- 가로 스크롤 또는 고정 컬럼이 있는 경우 포커스 순서와 스크롤 위치가 예측 가능해야 한다.

### 사용자별 레이아웃 저장

컬럼 폭, 숨김 가능한 보조 컬럼, 고정 컬럼 후보, 정렬과 필터의 저장 여부는 사용자별 preference로 분리한다. 저장 모델은 후속 CAP-18 구현 story에서 확정하되 다음 기준을 만족해야 한다.

| 항목 | 기준 |
| --- | --- |
| 사용자별 저장 | 본사 사용자별로 저장한다. 다른 사용자 또는 지점장 화면에 전파하지 않는다. |
| 기본값으로 되돌리기 | 표 단위 reset을 제공하고 저장된 사용자 preference를 기본 레이아웃으로 되돌린다. |
| 핵심 컬럼 | 핵심 컬럼은 숨김 불가이며 최소 폭 아래로 축소 불가다. |
| 보조 컬럼 숨김 | 표시 편의 기능일 뿐 권한 차단이 아니다. 숨겨도 서버 응답 권한은 별도로 유지한다. |
| 민감 필드 | 리사이징/숨김 preference가 민감 필드 노출 정책을 바꾸지 않는다. 서버 DTO와 권한 helper가 boundary다. |

## 금지 사항

- 이 정책 산출물만으로 bulk close Server Action을 구현하지 않는다.
- 이 정책 산출물만으로 검토 생략 toggle을 구현하지 않는다.
- 개별 마감보다 느슨한 일괄 마감 예외를 허용하지 않는다.
- preflight 결과를 실행 직전에 재검증하지 않고 close하지 않는다.
- 부분 실패 시 성공 건을 자동 rollback, silent reopen, 원본 overwrite하지 않는다.
- 잘못 마감한 장부 되돌리기 기능을 CAP-15에 포함하지 않는다.
- `authorDisplayName`으로 인증 계정, `submittedById`, `updatedById`, `closedById`, 감사 `actorId`를 대체하지 않는다.
- Story 7.7 승인 없이 `당일 판매량`을 실제 POS 판매 수량으로 재정의하지 않는다.
- `differenceQuantity`, `differenceAmount`, `salesDifference`, DB field, 계산식을 CAP-17 단순 라벨 변경 범위에서 변경하지 않는다.
- dashboard layout preference schema, migration, seed를 추가하지 않는다.
- TanStack Table 또는 동등 table utility를 이 story에서 도입하지 않는다.
- inventory label 변경, author display cache 구현, unit/e2e tests를 이 story에서 추가하지 않는다.
- 컬럼 리사이징, 숨김, 저장 preference를 권한/민감 필드 차단 boundary로 사용하지 않는다.
- 지점장 모바일 장부 입력 390px 기준을 리사이징 테이블로 대체하지 않는다.

## 검증 기준

이 정책 산출물이 완료되려면 다음을 만족해야 한다.

- CAP-15 preflight 계약이 dry run, 실패 row 분류, 권한, stale token, 사유 입력, 감사 이벤트, 부분 실패, 재시도, 원본 잠금/append-only 정정 기준을 포함한다.
- CAP-16 표시명 기준이 표시명, 인증 계정, 제출/수정/마감자, 감사 actor를 분리한다.
- CAP-17 용어 결정이 Story 7.7 baseline과 OQ-14 차단 조건을 참조한다.
- CAP-18 완료 기준이 target surface, 핵심 컬럼 최소 폭, 키보드 조작, `role="separator"`, 사용자별 저장, `기본값으로 되돌리기`, 모바일 390px 제외를 포함한다.
- 승인자와 승격 가능 여부가 명시되어 있다.
- 금지 사항과 Traceability가 존재한다.
- `src/`, `prisma/`, `tests/` 변경이 없어야 한다.
