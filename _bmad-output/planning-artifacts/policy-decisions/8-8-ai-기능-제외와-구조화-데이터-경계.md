## AI 제외 선언

현재 릴리스에서 CAP-12는 AI 기능을 제공하지 않는다. 다음 항목은 명시적으로 금지한다.

| 금지 항목 | 결정 |
| --- | --- |
| AI 화면 | AI 대시보드, AI 분석 탭, 자연어 질의 화면, AI 요약 카드, AI 추천 CTA를 만들지 않는다. |
| AI API/Route Handler | `/api/ai`, `/api/chat`, `/api/analyze` 같은 Route Handler나 Server Action을 만들지 않는다. |
| model provider 호출 | OpenAI, Anthropic, Google, Azure OpenAI, Vercel AI SDK, LangChain, 기타 LLM/model provider API를 호출하지 않는다. |
| dependency | `package.json`에 AI/provider/vector 관련 dependency를 추가하지 않는다. 예: `openai`, `@anthropic-ai/*`, `ai`, `@ai-sdk/*`, `langchain`, vector DB client. |
| prompt 저장 | 프롬프트, system prompt, prompt template, prompt fixture, prompt execution log를 저장하지 않는다. |
| embedding/vector DB | embedding 생성, vector DB, semantic search index, RAG pipeline을 만들지 않는다. |
| 자동 분석 결과 | AI가 만든 경영 분석 결과, anomaly explanation, action plan, score, summary를 저장하거나 표시하지 않는다. |
| 추천/의사결정 자동화 | 발주 추천, 가격 추천, 인력 추천, 폐기/떨이 추천, 지점 운영 의사결정 자동화를 만들지 않는다. |
| AI 이미지 식별 | 영수증, 품목, 재고, 매장 사진을 AI로 식별하거나 분류하지 않는다. |
| 자연어 질의 UI | "질문하면 답하는" UI, chat input, natural-language report query를 만들지 않는다. |

AI 기능이 필요해지는 경우 다음 승인물이 먼저 필요하다.

| 선행 조건 | 필요 내용 |
| --- | --- |
| 별도 PRD 또는 change proposal | 사용자 가치, 비범위, 실패 fallback, human approval boundary, 배포 범위를 새로 정의한다. |
| provider/secrets 승인 | model provider, API key/secrets 저장소, rotation, quota/cost 책임자를 승인한다. |
| prompt/retention 승인 | prompt/response 보존 기간, 삭제 정책, logging redaction, replay 가능성을 승인한다. |
| 개인정보 검토 | 직원/급여/근무/수신자/계정 식별자 처리, data-subject/privacy 기준, export 만료 기준을 승인한다. |
| 민감 지표 leakage 테스트 | 원가, FIFO 원가, 이익, 마진율, 재고금액, lot 근거, 본사 고정비, 타 지점 비교가 prompt/log/cache/export/알림에 새지 않는 테스트를 포함한다. |
| 감사 정책 | 누가 어떤 입력으로 어떤 model 결과를 만들었고 누가 승인했는지 audit event로 남기는 기준을 승인한다. |

## 필드 승격 판정 기준

| 판정 | 기준 | 예시 |
| --- | --- | --- |
| 구현 후보 | 현재 FR/CAP/report/alert/compliance에서 조회, 필터, 집계, 권한 차단, audit에 필요하다. | `ledgerDate`, `storeId`, `report data status`, `dedupe key`, `redaction state` |
| 정책 후보 | 필요성은 있으나 OQ, 승인자, 계산 기준, 보존 기준이 아직 닫히지 않았다. | FIFO `valuation status`, hoped-sale-price version, 장기 체화 threshold version |
| 승격 불가/연구 후보 | 현재 제품 문제와 직접 연결되지 않고 미래 AI 분석 가능성만을 이유로 추가하려는 필드다. | LLM score, AI insight category, prompt token count, embedding id |
| 금지 | 민감 정보 leakage, 자동 의사결정, 기존 자유 메모 자동 파싱, provider 호출을 전제로 한다. | AI recommendation, prompt-derived employee issue, vector search match |

후속 구현 story는 구조화 필드가 필요한 CAP/FR, 화면 또는 API surface, 권한 allowlist, redaction, audit event, test 기준을 함께 가져야 한다. 이 문서만으로 schema/API/UI/test를 추가하지 않는다.

## 자유 텍스트와 구조화 필드의 경계

자유 메모 보존은 허용한다. 다만 리포트, 검색, 필터, 알림, 감사, 권한 차단에 필요한 값은 자유 텍스트에만 두지 않고 다음 중 하나로 남긴다.

| 구조화 방식 | 사용 기준 |
| --- | --- |
| code | 손실 유형, 근태 유형, 알림 조건, report status처럼 제한된 선택지가 있는 경우 |
| status | `needs_review`, `basis_missing`, `policy-unconfirmed`, `권한 차단`, `데이터 부족`처럼 상태 전이가 필요한 경우 |
| relation | `storeId`, `productId`, actor id, target id처럼 다른 entity와 연결해야 하는 경우 |
| event | upload commit, void, reprocess, close, correction, notification delivery처럼 감사 가능한 사건인 경우 |
| version | mapping version, threshold version, template version, policy gate version처럼 기준 변경을 추적해야 하는 경우 |
| timestamp | created/imported/committed/submitted/closed/effectiveAt처럼 순서와 적용 시점이 필요한 경우 |

다음 재해석은 금지한다.

| 자유 텍스트 | 금지 결정 |
| --- | --- |
| 기존 `workMemo` | 직원명, 근무자별 근태, 급여 사유로 자동 파싱하지 않는다. |
| 손실 `reason` | 손실 type, 원가성 손실액, 희망 판매가 손실액으로 자동 추출하지 않는다. |
| expense `memo` | 비용 category, 고정비, 월 손익 항목으로 자동 구조화하지 않는다. |
| upload `referenceInfo` | 품목 mapping, 거래처, 원가 근거로 자동 확정하지 않는다. |
| 기타 운영 메모 | 개인정보 또는 민감 회계 정보를 자동 추출해 구조화하지 않는다. |

기존 자유 텍스트에서 구조화 필드를 만들 필요가 생기면 별도 migration/import story가 필요하다. 그 story는 원본 보존, 수동 승인 근거, rollback, 개인정보/민감 지표 검토, audit event, 샘플링 검증을 포함해야 한다.

## 개인정보와 민감 데이터 차단 기준

### 재사용할 민감 taxonomy

Story 7.6/8.4의 민감 taxonomy를 CAP-12 구조화 registry에 그대로 적용한다.

| 민감 class | 예시 | 기본 정책 |
| --- | --- | --- |
| 원가/lot | 원가, FIFO 원가, 매출원가, unit price source, lot 근거 | 지점장/외부/share/export/cache/알림 기본 차단 |
| 이익/마진 | 매출이익, 영업이익, 마진율, 생산성 | 지점장 기본 차단. 본사도 allowlist 필요 |
| 재고금액 | 재고금액, FIFO 재고금액, valuation amount | 지점장 수량 중심 surface와 분리 |
| 희망 판매가 | 희망 판매가, 희망 판매가 기준 손실액 | Story 8.4 정책 gate 전 노출 금지 |
| 본사 정책 지표 | 본사 고정비, 타 지점 비교, 내부 threshold | 본사 권한과 store scope를 확인한 뒤 제한 노출 |
| 개인정보/급여 참고 | employee id/display name, 근무 가능 지점, 근무일, 급여 차액 메모 metadata | 개인정보 보존표, field-level 권한, export 만료, audit 필요 |

### Surface별 차단

| Surface | 기준 |
| --- | --- |
| 지점장 화면/API | 자기 지점 운영에 필요한 최소 정보만 허용한다. 원가, FIFO 원가, 이익, 마진율, 재고금액, lot 근거, 본사 고정비, 타 지점 비교는 차단한다. |
| 외부/비로그인/공유 링크 | 인증 전 store detail과 민감 지표를 제공하지 않는다. 링크 URL에 민감 key/value, token, 내부 계산 사유를 넣지 않는다. |
| 알림 템플릿 | 지점/외부 대상 템플릿에는 민감 key와 실제 value가 없어야 한다. 본사 템플릿도 channel scope와 수신자 권한을 확인한다. |
| export | 본사 export 권한, allowlist, no-store, audit log, 만료 기준을 적용한다. 민감 컬럼은 승인 전 포함하지 않는다. |
| cache/log | response cache, client state, data attribute, aria-label, provider log, debug log에 민감 value를 남기지 않는다. |
| 본사 내부 | 본사라는 이유만으로 전체 허용하지 않는다. 권한 프로파일과 store scope를 확인한 뒤 allowlist 방식으로 노출한다. |

권한 차단은 서버에서 강제한다. UI 숨김, table column hide, prompt/redaction copy, client-side filter는 보안 boundary가 아니다.

## 후속 구현 승격 조건

CAP-12 자체는 `approved discovery only`다. 구조화 필드 목록까지만 승인 가능하며, 이 문서만으로 AI 기능 제공이나 schema/API/UI/test 구현은 승인되지 않는다.

| 후속 후보 | 승격 조건 | 이 문서만으로 가능한가 |
| --- | --- | --- |
| 구조화 필드 schema/API/UI 구현 | 해당 CAP 구현 story에서 필요한 필드, 권한, validation, migration, audit, test 기준을 별도로 정의한다. | 아니오 |
| report/export 상태 확장 | FR/CAP report 계약, status/evidence 의미, export allowlist, 민감 차단 기준이 승인되어야 한다. | 아니오 |
| 알림 조건/발송 구현 | Story 8.7의 채널, template matrix, dedupe, failure/retry, token 보안 정책이 승인되어야 한다. | 아니오 |
| 직원/근무 구조화 | Story 8.1 개인정보 보존표, field-level 권한, export 만료, audit 기준이 승인되어야 한다. | 아니오 |
| 품목/upload/FIFO 구조화 | Story 8.2/8.3의 mapping, lot/evidence, 원가 정책, OQ 기준이 승인되어야 한다. | 아니오 |
| AI 기능 | 별도 PRD 또는 change proposal, provider/secrets/retention/redaction/audit/privacy/leakage test 승인이 필요하다. | 아니오 |

AI 기능 승격은 별도 PRD/change proposal 없이는 불가하다. 해당 proposal은 model provider, prompt, retention, redaction, audit, data-subject/privacy, 민감 지표 leakage 테스트, human approval, model failure fallback을 포함해야 한다.

## 금지 사항

- 이 정책 산출물만으로 Prisma model/migration, AI SDK, vector DB, search index, natural-language UI, recommendation engine, background AI job, unit/e2e tests를 추가하지 않는다.
- CAP-12를 이유로 `package.json` 또는 lockfile에 OpenAI, Anthropic, Vercel AI SDK, LangChain, embedding/vector 관련 dependency를 추가하지 않는다.
- 기존 자유 메모를 자동 파싱하거나 AI/자동 분석 입력으로 재해석하지 않는다.
- 민감 지표를 AI/prompt/log/cache/export/notification payload에 남기지 않는다.
- AI 분석 결과를 계산, 리포트, 운영 판단, 감사의 source of truth처럼 저장하지 않는다.
- 미래 AI 가능성만을 이유로 새 schema, dependency, provider SDK, prompt fixture, E2E route, smoke test를 추가하지 않는다.
- 승인자 없는 CAP-12 산출물로 AI 구현 story를 만들지 않는다.
