## 지점장 기본 차단 필드 taxonomy

아래 taxonomy는 key 이름, 화면 라벨, CSV 컬럼명, cache key/value, 알림 변수명에 모두 적용한다. camelCase, snake_case, 한글 라벨, 약어, nested key도 같은 의미이면 차단한다.

### 원가 계열

| 필드/개념 | 지점장 정책 | 비고 |
| --- | --- | --- |
| `unitPrice` | blocked | 재고, 매입, 손실 row의 단가가 원가 또는 원가 근거이면 차단한다. |
| `defaultUnitPrice` | blocked | 상품 기본 단가는 원가 역추정 근거가 될 수 있다. |
| `standardUnitPrice` | blocked | 표준 원가/기준 원가 의미이면 차단한다. |
| `costOfGoodsSold` | blocked | 매출원가 직접값이다. |
| `fifoCostOfGoodsSold` | blocked | FIFO 확정 원가 또는 후보값 모두 차단한다. |
| 매입 lot 원가 | blocked | lot별 단가, 잔량 원가, 차감 원가를 포함한다. |
| lot 잔량/근거 | blocked | 수량만으로 원가를 역추정하지 못하는 운영 상태는 허용 가능하지만 원가 근거와 결합하면 차단한다. |

### 애매한 파생값 처리

애매한 원가/이익 기반 파생값은 기본 차단한다. 예: "효율", "생산성", "기여도", "건전성", "수익성", "차액", "rate"가 실제로 원가, 이익, 마진율, 재고금액, lot 근거를 사용하면 지점장 허용값이 아니다. 허용하려면 다음을 별도 문서에 남긴다.

1. 계산식과 사용 원천 데이터
2. 지점장에게 공개해도 원가/이익을 역추정할 수 없는 근거
3. 서버 응답, export, cache, 알림 변수에서 key와 실제 value를 검증하는 테스트 기준
4. 승인자, 승인일, 롤백 기준

## 지점장 허용값 기준

지점장에게 허용되는 값은 현장 운영 수행에 필요한 비민감 데이터로 제한한다.

- 본인 지점의 날짜, 장부 상태, 단계 완료 여부, 제출/검토 상태
- 사용자가 입력한 비민감 수량, 상태, 메모, 사유
- 상품명, 카테고리, 규격처럼 원가와 직접 결합되지 않은 표시값
- 결제수단 합계, 총매출, 근무인원처럼 기존 MVP에서 허용된 현장 운영값
- `확인 필요`, `데이터 부족`, `계산 불가`, `재확인 필요` 같은 상태 레이블
- 손실 확인 후보의 비금액 수량/상태. 금액, 단가, 원가 근거는 제외한다.

## 본사 전용 shape와 지점장 shape 분리

본사 전용 응답 shape는 지점장 경로에서 재사용하지 않는다. 어쩔 수 없이 shared helper를 재사용할 때는 다음 중 하나 이상을 반드시 거친다.

- `requireStoreAccess`로 지점장 본인 지점 접근을 검증한다.
- `requireReportAccess`, `requireExportCreateAccess`, `requireSettingsAccess` 등 본사 action 권한을 다시 검증한다.
- 권한 scoped DTO를 별도로 생성한다.
- `omitSensitiveFields` 또는 동등한 allowlist/denylist 차단 helper를 서버에서 적용한다.
- 본사 전용 report/export DTO를 만들기 전에 지점장 경로인지 분기하고, 지점장 경로에서는 taxonomy key/value를 query select 또는 mapper 단계에서 제외한다.

UI 숨김, table column hide, CSS `display:none`, client conditional rendering만으로는 승인 기준을 만족하지 않는다. 클라이언트가 받지 않아야 할 key/value는 서버 응답에서 제거해야 한다.

## 서버 차단과 테스트 기준

후속 MVP-S09 구현 story는 지점장 세션에서 다음 surface에 taxonomy key/value가 포함되지 않음을 검증해야 한다.

| 대상 | 필수 검증 |
| --- | --- |
| 응답 JSON | `grossProfit`, `grossMarginRate`, `inventoryAmount`, `unitPrice`, `hopedSalePrice`, `fifoInventoryAmount`, `fixedCost`, `comparisonStore` 같은 key가 없어야 한다. |
| HTML/RSC payload | 화면 text뿐 아니라 response body에 seeded 민감 amount와 key가 없어야 한다. |
| export/다운로드 응답 | 지점장은 CSV를 받을 수 없어야 한다. forbidden body에도 taxonomy key/value, store id, scoped store ids, 내부 계산 사유가 없어야 한다. |
| error payload | forbidden/bad request 응답은 `error`, 비민감 message 정도로 제한한다. 민감 metric key, 계산 사유, scope detail을 넣지 않는다. |
| cache key/value | 권한별로 분리하거나 `no-store`를 사용한다. 본사 권한 결과를 지점장 cache key/value로 재사용하지 않는다. |
| 알림 템플릿 변수 | 지점/외부 대상 template variables에 taxonomy key/value를 추가하지 않는다. |

테스트는 key 이름과 실제 값을 모두 검사한다. 예: `grossProfit` key가 없어야 하고, seeded amount `123456789` 같은 민감 값도 body/CSV/cache payload에 없어야 한다. 클라이언트 화면 테스트만으로는 통과하지 않고, server/query/route/export response shaping 검증을 포함해야 한다.

### 후속 테스트 요구사항

| 테스트 유형 | 요구 기준 |
| --- | --- |
| unit | `omitSensitiveFields` 또는 권한 scoped DTO mapper가 nested key, 한글 라벨, key 변형, seeded value leakage를 막는지 검증한다. |
| server/query | 지점장 helper가 본사 DTO를 반환하지 않고 allowed DTO만 반환하는지 검증한다. |
| route/export | 지점장 또는 export 권한 없는 사용자가 CSV를 받지 못하고 forbidden body에 민감 key/value가 없는지 검증한다. |
| e2e | 지점장 화면과 network/body에서 민감 label/key/value가 보이지 않는지 검증한다. |
| cache | 권한별 cache 분리 또는 `Cache-Control: no-store`를 검증한다. |
| 알림 | 지점/외부 대상 template 변수 allowlist에 민감 변수가 없는지 검증한다. |

## Cache, export, 공유 링크, 알림 기준

### Cache

- 본사 report result와 지점장 result는 같은 cache key를 공유하지 않는다.
- 권한, role, store scope, action permission을 cache key에 포함하거나 `no-store`를 사용한다.
- export, forbidden response, 권한 경계가 있는 route는 `Cache-Control: no-store`를 기본으로 둔다.
- cache value에도 taxonomy key/value가 남으면 실패다. key만 권한별로 분리해도 value leakage가 있으면 실패다.

### Export

- 지점장은 report/export route에서 CSV를 받을 수 없다.
- 본사 조회 권한과 export 생성 권한을 분리한다. `HEADQUARTERS: report view`만으로 export를 생성하지 않는다.
- 본사 export allowlist에는 `grossProfit`, `grossMarginRate`, `operatingProfit`, `productivity` 같은 본사 지표가 포함될 수 있으나, 그 shape를 지점장/공유/외부 surface로 재사용하지 않는다.
- denied export 요청은 감사 로그 후보지만, 응답 body에는 민감 request detail을 넣지 않는다.

### 공유 링크

- 외부/비로그인 공유 링크는 기본 blocked다.
- 본사 내부 공유가 필요하면 링크 접근 시점에 권한을 재검사하고 만료 시간을 둔다.
- 공유 payload에는 taxonomy key/value가 없어야 한다. 본사 전용 export artifact URL을 지점장에게 노출하지 않는다.

### 알림 템플릿

- 본사 대상과 지점/외부 대상 템플릿을 분리한다.
- 지점/외부 대상에는 민감 변수 추가를 금지한다.
- 지점/외부 대상 알림은 `확인 필요`, `데이터 부족`, 제출 상태, 누락 단계 같은 비민감 상태 문구만 사용한다.
- 본사 내부 알림도 권한과 대상 channel을 기준으로 변수 allowlist를 둔다.

## 감사 로그와 운영 기준

OQ-10A MVP 최소 차단은 "노출 허용 기능"이 아니라 기본 차단 기준과 서버 테스트 기준의 승인이다. CAP-13 고도화/OQ-10B의 지표별 허용, 승인자, 롤백, 노출 정책 변경 UI는 이 story에서 열지 않는다.

후속 MVP-S09 구현 story는 다음 감사 로그 기준을 검토한다.

| 이벤트 | action 후보 | targetType 후보 | after payload 기준 |
| --- | --- | --- | --- |
| 지점장 export 차단 | `report.export.denied` | `ReportExport` | actor role, required action, report type, period 정도만. taxonomy key/value, scoped store ids, 내부 계산 사유는 제외한다. |
| 민감 field policy 변경 | `sensitive-field-policy.updated` | `SensitiveFieldPolicy` | changed field keys, allowed roles, approver, reason, effectiveAt, rollback reference. 실제 민감 금액은 제외한다. OQ-10B 전에는 구현하지 않는다. |
| 차단 helper 적용 실패 감지 | `sensitive-field-leak.detected` | `SecurityFinding` | finding id, surface, actor role, blocked key count. 실제 value는 redacted 처리한다. |
| 알림 변수 차단 | `notification.template-variable.denied` | `NotificationTemplate` | template id, variable key, target audience, actor. 민감 값은 제외한다. |

감사 로그 화면은 본사 권한자에게만 민감 정책 변경 이력을 보여준다. 지점장은 본인 action의 비민감 이력과 상태만 볼 수 있고, before/after payload에 taxonomy value가 있으면 redaction된 상태로 보여야 한다.

## OQ-10B 분리

OQ-10A 승인만으로 다음을 구현하지 않는다.

- 지표별 노출 허용 toggle
- 본사 승인자가 특정 지점장에게 민감 지표를 허용하는 workflow
- 민감 필드 정책 변경 UI
- 승인 취소 또는 rollback UI
- 역할별 동적 allowlist 관리
- 알림/공유/export별 세밀한 예외 허용

이 항목들은 OQ-10B 산출물에서 승인자, 변경 이력, rollback, 감사 로그, 테스트 기준이 별도로 확정된 뒤 구현한다.

## 후속 구현 story 범위

후속 MVP-S09 구현 story는 다음 범위를 포함한다.

- `omitSensitiveFields` 또는 권한 scoped mapper의 key coverage 확장
- 지점장 DTO shaping unit tests
- report/export forbidden body tests
- 지점장 화면과 server response e2e tests
- cache `no-store` 또는 권한별 cache 분리 검증
- 알림 template variable allowlist 검증
- 감사 로그 action/targetType/after payload의 redaction 검증

다음은 포함하지 않는다.

- OQ-10B 동적 노출 정책 UI
- Prisma schema migration
- 새 public API
- 본사 전용 report/export 구조를 지점장에게 공유하는 기능
- 민감 지표의 지점장 예외 허용 기능
