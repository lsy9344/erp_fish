## 채널 결정과 OQ-13 상태

### OQ-13 결정 상태

OQ-13은 현재 "결정 보류"다. PM과 본사 운영자가 LINE 우선, 텔레그램 우선, 둘 다 후보 유지, 또는 명시적 보류 중 하나를 승인하기 전에는 채널 연동 구현 story를 만들지 않는다.

| 후보 | 현재 상태 | 선택 시 준비 기준 | 구현 가능 여부 |
| --- | --- | --- | --- |
| LINE Official Account / Messaging API | 후보 | 공식 계정 소유자, Messaging API channel, channel access token, 월간 메시지 한도, 대상 수신자 등록 방식, test send 대상 승인 | OQ-13 승인 전 불가 |
| 텔레그램 Bot API | 후보 | bot token, target `chat_id`, group/channel 운영 방식, 봇 권한, 테스트 채팅방, 메시지 길이/formatting 기준 | OQ-13 승인 전 불가 |
| LINE + 텔레그램 병행 | 후보 | 채널별 토큰과 수신자 권한 분리, 중복 발송 방지, 채널 우선순위와 fallback 기준 | OQ-13 승인 전 불가 |
| 외부 알림 미사용 | 후보 | 내부 dashboard/리포트만으로 운영 위험을 전달할 수 있는지 PM 승인 | CAP-11 scope 재검토 필요 |

LINE을 선택하더라도 LINE Notify는 사용하지 않는다. LINE Notify 종료 이후 기준은 LINE Official Account와 Messaging API이며, 후속 구현 story는 공식 문서의 최신 발송 방식과 quota를 다시 확인해야 한다.

| 책임 항목 | 기준 |

| 운영 소유자 | 본사 운영자. 채널 계정 접근 권한과 수신자 등록 기준을 관리한다. |
| 기술 소유자 | 개발 리드. token 저장소, provider API client, idempotency, audit log, failure handling 기준을 구현 story에서 검증한다. |
| 승인자 | Noah Lee(PM), 본사 운영자, 개발 리드. test send 대상과 민감 템플릿 범위를 승인한다. |
| 토큰 보관 | 환경 변수 또는 승인된 secrets store. 저장소 커밋, 브라우저 노출, 문서 내 실제 token 기록 금지. |

OQ-13 승인 전에는 `src/features/notifications`, scheduled Route Handler, worker, Prisma notification models, provider SDK/API client, 외부 API 호출 코드를 구현하지 않는다.

## 알림 조건과 OQ-16 상태

### OQ-16 결정 상태

OQ-16은 현재 "결정 보류"다. 1주, 1개월, 품목별 기준을 모두 후보로 유지한다. 승인 전에는 장기 체화 재고 자동 알림, FIFO/lot 기반 aging 계산, 재고 신선도 schema를 구현하지 않는다.

| 장기 체화 후보 | 의미 | 장점 | 위험 | 현재 결정 |
| --- | --- | --- | --- | --- |
| 1주 기준 | 입고 또는 기준일 이후 7일 이상 움직임이 없는 품목 후보 | 빠른 신선도 위험 탐지 | 품목별 회전 특성 무시 가능 | 후보 |
| 1개월 기준 | 30일 또는 월 단위로 움직임이 없는 품목 후보 | 알림 피로가 낮고 운영 검토가 단순함 | 신선 품목에는 늦을 수 있음 | 후보 |
| 품목별 기준 | 품목 master 또는 정책표의 threshold를 사용 | 현실 운영에 가장 가까움 | master 품질과 승인 정책 필요 | 후보 |
| 결정 보류 | 자동 알림 대신 수동 리포트/검토로 유지 | 오발송 방지 | CAP-11 가치 실현 지연 | 현재 상태 |

| 조건 | 계산 source | 필요한 선행 정책 | 권한 대상 | 발송 주기 | 중복 억제 key | escalation 기준 | fallback |
| --- | --- | --- | --- | --- | --- | --- | --- |
| 당일 적자 발생 지점 | 장부 마감/정정 반영값, 비용/매출 계산 결과 | 적자 정의, 미마감 포함 여부, 민감 지표 권한 | 본사 내부 우선. 지점장 대상은 손익 민감 필드 제거 후 별도 승인 | 일 1회 또는 마감 후 1회 후보 | `daily_loss:{storeId}:{date}:{thresholdVersion}:{recipientGroup}:{templateId}` | 2영업일 연속 또는 손실액 threshold 초과 시 본사 운영자에게 escalation | `기준 확인 필요`, `데이터 부족`, `미마감 포함` |

| 장기 체화 재고 | 재고 snapshot, 입출고, FIFO/lot 또는 품목별 aging 후보 | OQ-16, CAP-8/CAP-7 재고/원가 근거, 품목 threshold | 본사 내부 우선. 지점장 대상은 품목명/수량 중심으로 제한 | 주 1회 또는 월 1회 후보 | `stale_inventory:{storeId}:{itemId}:{period}:{thresholdVersion}:{recipientGroup}:{templateId}` | 고위험 품목 수 또는 기준 초과 기간이 증가할 때 escalation | `기준 확인 필요`, `데이터 부족` |
| 제출/마감 지연 | ledger status, 제출 시간, 마감 deadline | 제출 deadline, 휴무/특수일 처리, 지점별 예외 | 지점장/지점 그룹 가능. 민감 지표 없음 | deadline 전 reminder와 deadline 후 1회 후보 | `submission_delay:{storeId}:{ledgerDate}:{deadlineVersion}:{recipientGroup}:{templateId}` | 반복 지연 또는 본사 마감 blocking 시 본사 운영자 escalation | `기준 확인 필요`, `휴무`, `이미 제출됨` |
| 정정 반영 재확인 후보 | correction request, correction-applied values, audit log | 정정 반영값 운영 기준, 본사 확인 책임 | 본사 내부 전용. 지점장 알림은 상태 안내만 별도 승인 | 정정 발생 후 1회 후보 | `correction_recheck:{storeId}:{eventId}:{thresholdVersion}:{recipientGroup}:{templateId}` | 미확인 상태가 SLA를 넘으면 escalation | `데이터 부족`, `권한 차단`, `이미 확인됨` |

알림은 발송량 자체가 성공 지표가 아니다. 중복 발송, 알림 피로, 오발송, 민감 지표 노출, 실패 로그 누락, 재시도 폭주를 실패 신호로 측정한다.

| link URL | 허용 | 허용 가능 | 인증 전 차단 | test URL | URL에 token, 민감 지표, store scope detail 포함 금지 |

지점장/외부 대상 템플릿에는 원가, FIFO 원가, 매출이익, 영업이익, 마진율, 재고금액, 원가 lot 근거, 본사 고정비, 타 지점 비교, 희망 판매가, 희망 판매가 기준 손실액을 넣지 않는다.

본사 대상 템플릿도 권한 프로파일과 channel scope를 확인한 뒤 변수 allowlist를 적용한다. 본사 권한이더라도 channel이 공유 group이거나 외부 구성원이 포함될 수 있으면 민감 변수는 blocked 처리한다.

### 링크 접근과 본문 redaction

알림 링크는 접근 시점에 인증/권한을 재검사해야 한다. 링크 자체에는 민감 지표, store scope detail, token, 내부 계산 사유, 원본/정정 차이, provider credential을 넣지 않는다.

알림 본문은 장기 보존 대상이 아니다. 발송 로그에는 template id, event id, recipient, status, provider response code, redaction state를 남기되, 민감한 메시지 본문 전체를 장기 보존하지 않는다.

OQ-10B 승인 전에는 알림 템플릿 예외 허용, dynamic variable allowlist UI, 지표별 수신자 예외 정책을 구현하지 않는다.

## 발송 실패, 재시도, 중복 억제, 감사 로그 계약

### `NotificationDeliveryLog` 후보 필드

이 표는 schema 확정이 아니라 후속 구현 story의 계약 후보다. Story 8.7에서는 Prisma model이나 migration을 만들지 않는다.

| 필드 | 의미 |
| --- | --- |
| `ruleId` | 어떤 알림 rule이 발송을 만들었는지 식별 |
| `recipientId` | 내부 수신자 또는 수신자 그룹 식별자 |
| `channel` | `line`, `telegram`, `internal_test` 등 채널 |
| `templateId` | 사용한 템플릿 id/version |
| `eventId` | 원인 이벤트 또는 계산 결과 id |
| `idempotencyKey` | 중복 발송 방지 key |
| `attemptNumber` | 발송 시도 번호 |
| `status` | `queued`, `sent`, `failed_retryable`, `failed_final`, `suppressed_duplicate`, `skipped_policy` 후보 |
| `providerResponseCode` | provider HTTP/status/error code |
| `nextRetryAt` | 다음 재시도 예정 시각 |
| `finalFailureReason` | 최종 실패 사유 |
| `redactionState` | `redacted`, `sensitive_blocked`, `safe_template`, `test_fixture` 후보 |

### 실패 분류

| 분류 | 예시 | 기본 처리 |
| --- | --- | --- |
| provider 인증 실패 | invalid token, revoked credential | 즉시 최종 실패 후보, 개발 리드와 본사 운영자 escalation |
| monthly quota/limit 초과 | LINE 월간 메시지 한도 초과 | retry만으로 해결하지 않음, 비용/한도 책임자 확인 |
| rate limit | provider 429 또는 유사 제한 | backoff 후 재시도 후보 |
| 비활성 채널 | account disabled, bot removed, channel missing | 최종 실패 후보, 채널 운영자 확인 |
| 수신자 차단/존재하지 않음 | blocked user, chat not found, forbidden | 해당 recipient 비활성 후보 |
| 부분 발송 실패 | multicast/group 일부 실패 | recipient별 delivery log와 재시도 분리 |
| 중복 발송 후보 | 같은 dedupe key 반복 생성 | `suppressed_duplicate`로 기록하고 발송하지 않음 |
| policy 차단 | 민감 변수 포함, 권한 불일치 | `skipped_policy`로 기록하고 발송하지 않음 |

### 재시도 정책 후보

재시도 횟수/간격/최종 실패 표시 정책은 아직 승인 전이다. 후속 구현 story는 고정 숫자를 임의로 만들지 않는다.

| 항목 | 후보 정책 |
| --- | --- |
| 재시도 횟수 | 0회, 3회, 또는 provider별 승인 횟수 중 선택 필요 |
| 재시도 간격 | 즉시 재시도 금지. exponential backoff 또는 고정 간격 후보 |
| 최종 실패 표시 | log status를 `failed_final`로 두고 알림 설정/로그 UI에서 확인 가능하게 함 |
| retry 대상 | rate limit, 일시적 network/provider 5xx 후보 |
| retry 제외 | 인증 실패, 권한/policy 차단, recipient blocked, quota 초과 후보 |

### 중복 억제 key

같은 위험 항목 반복 발송을 막기 위한 기본 후보는 다음 조합이다.

`condition type + store id + source date/period + threshold version + recipient group + template id`

조건별 상세 후보는 알림 조건표의 중복 억제 key를 따른다. dedupe key가 없거나 threshold version이 기록되지 않은 알림 rule은 발송하지 않는다.

### 감사 로그와 보존

알림 로그는 발송 성공/실패, 수신자, template id, 주요 event id, provider response code, redaction state를 남긴다. 민감한 메시지 본문 전체는 장기 보존하지 않는다.

감사 로그는 다음 질문에 답할 수 있어야 한다.

- 누가 어떤 rule/template/channel을 승인했는가
- 어떤 event가 어떤 수신자에게 발송되었거나 차단되었는가
- 실패가 provider 문제인지, 권한/policy 문제인지, 중복 억제인지
- 재시도 여부와 최종 실패 처리 시점은 언제인가
- 민감 변수는 redaction 또는 policy 차단을 통과했는가

## 후속 구현 승격 조건

CAP-11 구현 story는 아래 항목이 모두 승인된 뒤 생성할 수 있다.

| 조건 | 필요 승인 |
| --- | --- |
| OQ-13 | 우선 채널 또는 병행/보류 결정 |
| OQ-16 | 장기 체화 재고 기준 |
| 템플릿 field matrix | audience별 allowed/blocked 변수와 OQ-10B 영향 |
| 실패/재시도 정책 | 재시도 횟수, 간격, 최종 실패 표시, retry 제외 조건 |
| 중복 억제 정책 | condition별 dedupe key와 threshold version 관리 |
| 토큰 보안 정책 | token 저장소, rotation, 접근자, 유출 대응 |
| test send 기준 | test channel, test template, 승인자, 실제/fixture 데이터 분리 |
| 승인자 기록 | PM, 본사 운영자, 개발 리드 승인 기록 |

LINE 또는 텔레그램 중 하나를 먼저 구현할 때도 다른 채널 fallback을 암묵적으로 구현 범위에 포함하지 않는다. fallback 또는 병행 발송은 별도 승인과 별도 story가 필요하다.

## 금지 사항

- OQ-13 승인 전 CAP-11 채널 연동 구현 story를 만들지 않는다.
- OQ-16 승인 전 장기 체화 재고 자동 알림, FIFO/lot 기반 aging 계산, 재고 신선도 schema를 만들지 않는다.
- OQ-10B 승인 전 알림 템플릿 예외 허용, dynamic variable allowlist UI, 지표별 수신자 예외 정책을 만들지 않는다.
- 이 정책 산출물만으로 notification DB schema, scheduled job, Vercel Cron, queue worker, provider SDK/API client, test send UI, DeliveryLog table, unit/e2e tests를 추가하지 않는다.
- LINE Notify를 사용하지 않는다.
- browser-side scheduled delivery나 client-side provider token 노출을 만들지 않는다.
- 민감 메시지 본문 전체를 장기 보존하지 않는다.
- 지점장/외부 알림에 원가, FIFO 원가, 매출이익, 영업이익, 마진율, 재고금액, lot 근거, 본사 고정비, 타 지점 비교, 희망 판매가, 희망 판매가 기준 손실액을 포함하지 않는다.
- 중복 억제 없는 반복 발송, 실패 로그 없는 fire-and-forget 발송, provider quota/rate limit 무시를 허용하지 않는다.
