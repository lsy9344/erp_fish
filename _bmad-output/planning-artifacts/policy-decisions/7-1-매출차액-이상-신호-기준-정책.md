## 매출차액 기준표

| 항목 | 정책 |
| --- | --- |
| 계산식 | `totalSales - paymentTotal`, `paymentTotal = cashAmount + cardAmount + otherPaymentAmount` |
| 양수 의미 | 총매출이 결제합계보다 큼. 예: 매출 입력은 있으나 결제 입력이 부족한 방향 |
| 음수 의미 | 결제합계가 총매출보다 큼. 예: 결제 입력이 매출보다 많거나 총매출 입력이 부족한 방향 |
| 0 의미 | 총매출과 결제합계가 일치 |
| 표시 | `+₩48,000`, `-₩48,000`, `₩0`처럼 부호 있는 값으로 표시 |
| 임계값 판정 | 방향과 무관하게 `abs(salesDifference) > salesDifferenceAmount`이면 warning 후보 |
| 방향별 기준 | MVP에서는 미채택 |
| 비율 분모 | MVP에서는 비율 기준 미채택이므로 사용하지 않음 |
| 분모 0/결측 상태 | 후속에서 비율을 채택하면 총매출을 분모로 쓰고, 총매출 0 또는 결측은 `계산 불가`가 아니라 `데이터 부족`으로 표시한다. 승인 전에는 `기준 확인 필요`를 유지한다. |

## 부호 예시

| 케이스 | 총매출 | 현금 | 카드 | 기타 | 매출차액 | 표시 문구 | 판정 |
| --- | ---: | ---: | ---: | ---: | ---: | --- | --- |
| 차액 0 | 1,000,000 | 200,000 | 750,000 | 50,000 | 0 | `매출차액 ₩0` | 정상 |
| 양수 초과 | 1,000,000 | 200,000 | 700,000 | 0 | +100,000 | `매출차액 +₩100,000` | `abs(100,000)` 기준으로 warning 후보 |
| 음수 초과 | 1,000,000 | 300,000 | 760,000 | 0 | -60,000 | `매출차액 -₩60,000` | `abs(-60,000)` 기준으로 warning 후보 |
| 총매출 0 | 0 | 0 | 0 | 0 | 0 | `매출차액 ₩0` | 정상. 비율 기준은 데이터 부족 |
| 결제 입력 일부 누락 | 1,000,000 | 200,000 | 입력 전 | 0 | 계산 보류 | `매출차액 데이터 부족` | 결제 필수 입력 누락이면 임계값 판정 금지 |

## 임계값 기본값과 운영 범위

| 기준값 | 기본값 | 저장 단위 | 변경 가능 범위 | 음수 허용 | 소수 자리 | 비고 |
| --- | ---: | --- | ---: | --- | --- | --- |
| 매출 하락률 | 15.00% | bps 정수 `1500` | 0.00% - 100.00% | 아니오 | 입력 최대 2자리, 표시 1자리 | 전주 동일 요일 대비 |
| 이익률 하락폭 | 5.00%p | bps 정수 `500` | 0.00%p - 100.00%p | 아니오 | 입력 최대 2자리, 표시 1자리 | 기준 이익률 - 현재 이익률 |
| 매출차액 금액 | 50,000원 | KRW 정수 | 0원 - 2,147,483,647원 | 아니오 | 없음 | `abs(totalSales - paymentTotal)` |
| 매출차액 비율 | 미사용 | 해당 없음 | 해당 없음 | 해당 없음 | 해당 없음 | 채택 시 schema/model 변경 필요 |

원화 금액은 정수 KRW로 저장한다. 금액 입력의 소수점은 검증 오류로 처리한다.

## 매출/이익률 급락 비교 기준

| 우선순위 | 비교 기준 | 사용 조건 | fallback |
| ---: | --- | --- | --- |
| 1 | 전주 동일 요일 | 같은 지점의 7일 전 마감 장부가 있고 총매출/이익률 계산 가능 | 최근 7영업일 평균 |
| 2 | 최근 7영업일 평균 | 같은 지점의 최근 7영업일 중 4일 이상 계산 가능 | 전영업일 |
| 3 | 전영업일 | 직전 영업일 마감 장부가 있고 계산 가능 | 데이터 부족 |
| 4 | 데이터 부족 | 위 기준을 모두 만족하지 못함 | 이상 신호 확정 판정 금지 |

수동 기준값은 MVP에서 사용하지 않는다. 수동 baseline이 필요하면 별도 승인과 감사 로그 계약을 가진 후속 story로 분리한다.

## 기준값 변경 감사 로그 필드

기준값 변경은 `threshold.updated` 감사 이벤트로 남긴다. 필수 필드는 다음과 같다.

| 필드 | 필수 여부 | 설명 |
| --- | --- | --- |
| actor | 필수 | 변경 실행 사용자 ID와 표시명 |
| timestamp | 필수 | 서버 기준 변경 시각 |
| targetType | 필수 | `AnomalyThresholdSetting` 또는 동등한 도메인 대상 |
| targetId/scope | 필수 | MVP에서는 `GLOBAL` |
| before | 필수 | 변경 전 기준값 snapshot |
| after | 필수 | 변경 후 기준값 snapshot |
| reason | 필수 | 본사 운영자가 입력한 변경 사유 |
| approver | 필수 | 승인자. MVP에서는 Noah Lee(PM) 또는 위임 승인자와 본사 운영자 |
| effectiveFrom | 필수 | 적용 시작일. 기본은 다음 영업일 00:00, 긴급 변경 시 reason에 즉시 적용 근거 기록 |

## 이상 신호 우선순위와 중복 표시

복수 신호를 숨기지 않는다. 우선순위는 정렬 순서일 뿐 필터링 규칙이 아니다.

| 표시 순서 | 신호 | severity | 기본 문구 예시 | 비고 |
| ---: | --- | --- | --- | --- |
| 1 | 재고 이상 | critical | `재고 이상 · 광어 차이 19개` | 손실/재고는 운영 위험도가 높아 먼저 표시 |
| 2 | 손실 이상 | critical | `손실 이상 · ₩60,000` | |
| 3 | 매출차액 | warning | `매출차액 +₩48,000 · 기준 ₩50,000` | 승인 전에는 `기준 확인 필요` |
| 4 | 매출 급락 | warning | `매출 급락 · 18.0% 하락` | |
| 5 | 이익률 급락 | warning | `이익률 급락 · 6.0%p 하락` | |
| 6 | 기준 확인 필요 | info | `매출차액 +₩48,000 · 기준 확인 필요` | 정책 미확정 |
| 7 | 데이터 부족 | info | `매출 급락 데이터 부족` | 비교 기준 데이터 부족 |
| 8 | 계산 불가 | info | `이익률 계산 불가` | 계산 조건 불충족 |

### 동시 발생 예시

| 화면 | 표시 방식 |
| --- | --- |
| 관제판 행 | `재고 이상`, `손실 이상`, `매출차액 +₩48,000 · 기준 확인 필요`, `매출 급락`, `이익률 계산 불가` 순서로 chip을 모두 표시 |
| 장부 상세 | 상단 요약에 동일 순서로 chip을 표시하고, 각 chip tooltip/detail에 기준값과 원천 값을 표시 |
| 리포트/export | 같은 정렬 순서로 `anomalySignals` 또는 상태 컬럼에 모두 나열. 하나의 신호 때문에 다른 신호를 누락하지 않음 |

이상 신호는 기본적으로 마감 차단이 아니다. 필수 입력 누락, 권한, 원본 잠금, ClosePreflight hard-stop은 별도 정책에 따른다.
