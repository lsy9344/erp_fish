## 희망 판매가 미입력 처리

### OQ-9 결정

희망 판매가가 없는 손실 row의 `hopedSalePriceLossAmount`는 계산하지 않고 `데이터 부족`으로 반환한다. `계산 불가`는 값은 있으나 수량, 처리 금액, version 충돌, 승인 상태 때문에 산식 적용이 불가능한 경우에만 사용한다. 수동 입력은 기존 `LedgerLossItem.amount`에만 남고 희망 판매가 기준 손실액의 대체값이 아니다.

| 입력 상태 | 파생 손실액 상태 | 숫자값 | 표시 문구 | 비고 |
| --- | --- | ---: | --- | --- |
| 희망 판매가 version 없음 | `데이터 부족` | null | 희망 판매가 기준이 없어 계산할 수 없습니다. | 0원 대체 금지 |
| 희망 판매가는 있으나 처리 금액 없음 | `데이터 부족` | null | 실제 처리 금액 기준이 없어 계산할 수 없습니다. | 손실 유형별 처리 금액 정의 필요 |
| version 충돌 또는 적용 시점 불명확 | `계산 불가` | null | 가격 version 기준이 충돌해 계산할 수 없습니다. | split 또는 본사 승인 예외 필요 |
| 본사 승인 예외 | `계산 불가` 또는 `본사 예외 승인` | null | 예외 승인으로 계산값을 생성하지 않았습니다. | 사유/승인자/일자 필수 |
| 기존 손실 금액만 있음 | `데이터 부족` | null | 입력 손실액은 파생 손실액이 아닙니다. | `LedgerLossItem.amount` 보존 |

### 대체 금지 원칙

희망 판매가가 없을 때 다음 값을 기준 가격으로 사용하지 않는다.

- 0원
- `Product.defaultUnitPrice`
- `LedgerLossItem.unitPrice`
- FIFO 원가 또는 매입 lot 원가
- `30%단가`
- 이전 영업일 또는 오래된 stale 희망 판매가
- 사용자가 입력한 기존 `LedgerLossItem.amount`

이 값들은 화면 보조값이나 기존 MVP snapshot일 수 있지만, CAP-14의 희망 판매가 기준 손실액 산식의 기준 가격이 아니다.

### 저장, 제출, 마감, 리포트 영향

| 흐름 | 희망 판매가 미입력 시 허용 여부 | 처리 |
| --- | --- | --- |
| 장부 저장 | 허용 | 현재 MVP 손실 유형/수량/금액 기록은 저장한다. 파생 손실액은 `데이터 부족`으로 둔다. |
| 검토 제출 | 조건부 허용 | CAP-14가 필수 close gate로 승격되기 전에는 제출 가능하되 본사 검토 경고를 남긴다. |
| 본사 마감 | 조건부 허용 | 승인 전 MVP 마감은 기존 손실 금액 기준으로 가능하다. CAP-14 활성화 후에는 희망 판매가 version 또는 본사 예외 승인이 필요하다. |
| 본사 리포트 | 허용 | 숫자 0으로 집계하지 않고 `데이터 부족` 건수와 제외 사유를 별도 표시한다. |

## 기준 가격 시점과 version/lock 정책

### 기준 후보 비교

| 후보 | 채택 여부 | 이유 |
| --- | --- | --- |
| 영업 개시 전 지점장 입력값 | 채택 | CAP-14의 기본 의도와 일치하며 당일 손실 판단 기준을 안정적으로 lock할 수 있다. |
| 손실 발생 시점 가격 | 제한 채택 | 본사 override로 당일 적용 version이 명시된 경우에만 사용한다. 일반 지점장 변경에는 적용하지 않는다. |
| 장부 작성 시점 가격 | 미채택 | 늦은 입력이 손실 기준을 바꿔 손익을 왜곡할 수 있다. |
| 본사 마감 시점 가격 | 미채택 | 마감 직전 가격 변경이 과거 손실 row를 소급 변경할 수 있다. |
| 본사 override 가격 | 조건부 채택 | 승인자, 사유, 적용 시작 시각, 영향 row 범위를 감사 로그로 남길 때만 허용한다. |

### Lock 규칙

| 항목 | 정책 |
| --- | --- |
| 영업 개시 시각 | 지점 운영 설정의 business opening time을 사용한다. 설정이 없으면 CAP-14 구현 story에서 지점별 기본값을 먼저 정의해야 한다. |
| 가격 입력 마감 시각 | 영업 개시 전 마감이 기본이다. 예: 영업 개시 30분 전 또는 운영 설정값. |
| 기준 version | `storeId + businessDate + productId + normalized product key` 단위로 lock된 active version |
| 늦은 입력 | 장부 저장은 가능하지만 희망 판매가 기준 손실액은 `데이터 부족`이다. 본사 승인 override가 있으면 override version을 append한다. |
| 영업 중 지점장 변경 | 기본적으로 다음 영업일 version으로 생성한다. 당일 기존 row와 당일 후속 row 모두 자동 변경하지 않는다. |
| 영업 중 본사 override | override 승인 시 새 version을 append하고 적용 시작 시각을 기록한다. 적용 시작 전 row는 기존 version, 이후 row는 override version을 참조한다. |
| version 생성 기준 | 가격, 적용 시작 시각, 승인 상태, actor, reason이 바뀌면 새 version이다. 기존 version row를 수정하지 않는다. |
| 적용 시작 시각 | 일반 version은 business opening time, 본사 override는 승인된 effectiveAt |

### 변경 영향

| 대상 | 영향 |
| --- | --- |
| 이미 저장된 손실 row | row가 참조한 price version을 유지한다. 가격 변경으로 소급 재계산하지 않는다. |
| 마감 전 장부 | 아직 row version reference가 없으면 저장 또는 검토 시점에 lock된 기준 version을 연결해야 한다. 기준이 없으면 `데이터 부족`이다. |
| 마감 후 정정 | 원본 row를 덮어쓰지 않고 append-only correction 또는 recalculation evidence를 추가한다. |
| 월간 리포트 | 원본 row의 version reference 기준으로 집계한다. 정정이 있으면 정정 반영값과 원본값을 분리한다. |
| export | export 생성 시점의 권한과 row version reference를 함께 사용한다. stale price로 재계산하지 않는다. |

가격 version은 과거 장부 원본 row를 소급 덮어쓰지 않는다. 후속 구현은 append-only evidence 또는 `hopedSalePriceVersionId` 같은 version reference로 추적해야 한다.

## 손실 유형별 계산 정책과 예외

### 유형별 기준

후속 구현은 현 `LedgerInputCode.group = LOSS_TYPE`의 운영 코드명을 canonical type으로 매핑해야 한다. 코드명은 운영 설정이므로 이 문서에서는 정책 코드로 정의한다.

| 정책 코드 | 예시 운영명 | 희망 판매가 적용 | 실제 처리 금액 | 예외 상태 | 표시 문구 |
| --- | --- | --- | --- | --- | --- |
| `LOSS` | 손실, 분실, 파손 | 적용 | 회수/보상 금액이 있으면 해당 금액, 없으면 0원 | 처리 금액 불명확 시 `데이터 부족` | 희망 판매가 대비 회수 불가 손실 |
| `DISPOSAL` | 폐기 | 적용 | 0원 | 수량 또는 version 불명확 시 `계산 불가` | 폐기 처리로 회수 금액 없음 |
| `MARKDOWN` | 떨이 | 적용 | 실제 떨이 판매 금액 | 떨이 금액 미입력 시 `데이터 부족` | 희망 판매가 대비 떨이 차이 |
| `UNKNOWN` | 미분류 손실 유형 | 미적용 | 사용하지 않음 | `계산 불가` | 손실 유형 기준 확인 필요 |
| `HQ_EXCEPTION` | 본사 예외 | 조건부 | 승인 문서 기준 | 승인자/사유 누락 시 `계산 불가` | 본사 예외 승인 기준 |

기본 산식은 다음과 같다.

```text
hopedSalePriceLossAmount = max((lockedHopedSalePrice - actualDispositionUnitPrice) * quantity, 0)
```

단, 이 산식은 기준 price version, 수량, 실제 처리 금액 기준, 권한 노출 정책이 모두 승인된 후 shared server calculation module에서만 구현한다. 화면, export, report helper가 산식을 복제하지 않는다.

### 여러 version 또는 lot에 걸친 수량

| 상황 | 처리 |
| --- | --- |
| 하나의 손실 row가 하나의 price version에만 해당 | 계산 가능 |
| 하나의 손실 row가 여러 price version 적용 구간에 걸침 | 기본 차단하고 `계산 불가`로 둔다. 후속 구현에서 row split UI/API를 제공하면 split 계산 가능 |
| 하나의 손실 row가 여러 lot에 걸침 | 희망 판매가 손실액 자체는 lot 원가가 아니라 price version 기준이다. 단, 원가/마진 분석과 결합되는 경우 Story 7.4 FIFO 정책 승인 후 별도 계산한다. |
| 본사 승인 예외 | 승인자, 사유, 적용 범위, 영향 row, effectiveAt을 기록한 뒤 예외 상태로 둔다. 임의 숫자 대체는 금지 |

## 예시 계산

아래 예시는 정책 검증용이며 현재 코드 동작이 아니다. 모든 금액은 KRW 정수다.

### 예시 1: 희망 판매가 미입력

| 입력 | 값 |
| --- | ---: |
| 손실 유형 | 폐기 |
| 수량 | 3 |
| 희망 판매가 | 없음 |
| 실제 처리 단가 | 0 |
| 기존 `LedgerLossItem.amount` | 15,000 |

결과: `hopedSalePriceLossAmount = null`, 상태 `데이터 부족`. 기존 `LedgerLossItem.amount = 15,000`은 사용자가 입력한 손실 금액 snapshot으로 보존하지만 희망 판매가 기준 손실액으로 집계하지 않는다.

### 예시 2: 영업 중 가격 변경

| 시각 | 이벤트 | 가격 version | 처리 |
| --- | --- | --- | --- |
| 07:30 | 가격 입력 마감 | v1 = 10,000원 | 2026-06-12 영업일 기준 version lock |
| 10:00 | 손실 row 저장, 수량 2, 폐기 | v1 | `10,000 x 2 = 20,000` 후보 |
| 13:00 | 지점장이 12,000원으로 변경 | v2 next-day | 당일 기존 row와 당일 후속 row에 자동 적용하지 않음 |
| 14:00 | 본사 override로 12,000원 당일 14:00부터 승인 | v3 override | 14:00 이후 새 row만 v3 참조 가능 |

결과: 10:00 row는 v1을 유지한다. 14:00 이후 row가 v3를 참조하려면 본사 override 승인자, 사유, 적용 시작 시각이 있어야 한다.

### 예시 3: 손실, 폐기, 떨이 유형 차이

| 유형 | 수량 | lock 희망 판매가 | 실제 처리 단가 | 계산 |
| --- | ---: | ---: | ---: | ---: |
| `LOSS` | 2 | 10,000 | 0 | `(10,000 - 0) x 2 = 20,000` |
| `DISPOSAL` | 3 | 8,000 | 0 | `(8,000 - 0) x 3 = 24,000` |
| `MARKDOWN` | 4 | 12,000 | 7,000 | `(12,000 - 7,000) x 4 = 20,000` |

떨이의 실제 처리 단가가 없으면 숫자 0으로 대체하지 않고 `데이터 부족`이다. 폐기는 실제 처리 단가가 정책상 0원으로 정의된 유형이므로 0원 대체 금지 원칙과 충돌하지 않는다.

## 권한별 노출과 민감 필드 차단

희망 판매가와 희망 판매가 기준 손실액은 원가, 이익, 실제 마진율 차이를 역추정할 수 있다. Story 7.6/OQ-10A 승인 전에는 지점장에게 노출하지 않는다.

| Surface | 본사 권한 | 지점장 권한 | 필수 차단 방식 |

UI 숨김만으로는 승인 기준을 만족하지 않는다. 서버 응답 shaping 또는 `omitSensitiveFields` 계열 차단이 필요하다. Story 7.6/OQ-10A 승인 전에는 지점장에게 희망 판매가 기준 손실액 또는 역추정 가능한 파생값을 노출하지 않는다.
