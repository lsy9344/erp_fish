## CAP-2 특수기간 정의서

| --- | --- | --- |
| 설날 | baseline 후보 | OQ-11에서 기준일, D-day 범위, 휴무 처리 승인 후 가능 |

| 행사 기간 | 후보 | 행사 등록 주체와 승인자 기준 확정 전 구현 금지 |

후속 구현 story가 생성될 경우 최소 후보 필드는 다음과 같다. 이 표는 schema 확정이 아니라 계약 후보이며, migration은 OQ-11 승인 후 별도 story에서만 작성한다.

| `baseDate` | 기준일 | D0의 기준 날짜다. 음력/양력 변환 책임을 명확히 해야 한다. |

### D-day 기준

기준일은 `D0`으로 둔다. 기본 비교 단위는 달력일 기준 D-day offset이며, 영업일 기준 비교가 필요하면 OQ-11에서 별도 승인해야 한다.

| 기준 | 결정 |
| --- | --- |
| D0 | `baseDate` 당일이다. |
| 기본 offset | 달력일 기준 `D-7`부터 `D+3` 같은 연속 offset을 사용한다. |
| 영업일 기준 | 기본값이 아니다. 도입 시 휴무일/주말/공휴일 보정 규칙을 별도 승인한다. |
| 휴무일 표시 | 제외하지 않고 row 상태로 표시하는 것을 기본 후보로 둔다. 휴무 매출을 0으로 확정하지 않고 `휴무` 상태와 함께 표시한다. |
| 미입력일 | 0으로 대체하지 않고 `미입력` 또는 `데이터 부족` 상태를 표시한다. |

### 연도 비교 방식

연도 비교는 같은 D-day offset 간 비교를 기본으로 한다. 예를 들어 기준 연도의 `D-1`은 비교 연도의 `D-1`과 비교한다. 같은 양력 날짜 비교는 기본값이 아니며 별도 필터 옵션으로만 검토한다.

| 상황 | 표시 기준 |
| --- | --- |
| 윤년 | 실제 날짜가 달라도 D-day offset을 우선한다. 날짜 tooltip 또는 export 보조 컬럼에 실제 날짜를 표시한다. |
| 대체공휴일 | 대체공휴일 자체를 별도 row 상태로 표시한다. 비교 숫자를 임의 보정하지 않는다. |
| 기간 길이 차이 | 누락 offset은 `데이터 부족`으로 표시한다. 평균 또는 0으로 채우지 않는다. |
| 기준일 변경 | 변경 전후 기준일과 승인자를 표시하고, 다른 기준일끼리 비교할 때 `기준 확인 필요`를 표시한다. |
| 수동 제외일 | 제외 사유를 export에 포함한다. 집계에서 제외된 날짜 수를 함께 표시한다. |

### 지점 필터와 권한

특수기간 지점 필터는 현재 본사 report scope와 같은 서버 권한 범위를 사용한다. 후속 구현은 `requireReportAccess()`와 `getHeadquartersStoreScope()`를 통과한 지점만 반환해야 한다.

권한 밖 지점은 fallback, 다른 지점 데이터 자동 대체, 전체 평균 대체를 하지 않는다. 권한 밖 데이터가 요청되면 `권한 차단` 상태를 반환하거나 요청 자체를 거부하며, export에서도 같은 차단 기준을 적용한다.

### 상태 표시와 집계 처리

| 상태 | 집계 처리 | 화면 표시 | Export 표시 |
| --- | --- | --- | --- |
| 미입력 | 확정 숫자에서 제외 | `미입력` label, 원인 안내 | `dataStatus=미입력`, 숫자 blank |
| 입력중 | 확정 숫자에서 제외 | `입력중` label | `dataStatus=입력중`, 숫자 blank |
| 검토대기 | 미마감 포함 집계에만 포함 가능 | `검토대기`, 미마감 포함 표시 | `dataStatus=검토대기`, `includedAsUnclosed=true` |

OQ-11 승인 전에는 특수기간 등록 UI, DB migration, report route, chart, export 확장을 구현 story로 만들지 않는다.

### 계산 기준

기존 엑셀 수식은 참고 자료이며 권위 있는 계산 기준이 아니다. 서버 계산 기준은 `src/features/reports/queries.ts`, `src/features/reports/types.ts`, `src/server/calculations/*`, `src/features/reports/export.ts`의 기존 evidence/status 패턴을 따른다.

UI 컴포넌트, chart helper, export helper가 같은 계산식을 복제하지 않는다. 차트가 필요한 경우 표 원본값과 같은 서버 집계 결과를 사용하고, 차트만 별도 계산식을 만들지 않는다.

### 데이터 부족 표시

`데이터 부족`, `계산 불가`, `기준 확인 필요`, `정정 확인 필요`, `권한 차단`, `미마감 포함`은 0, 빈 문자열, 임시 숫자로 대체하지 않는다. export도 같은 상태를 보존해야 하며, 숫자 컬럼이 blank인 경우 status 컬럼으로 이유를 설명한다.

CAP-3은 `.xlsx` 업로드나 과거 엑셀 일괄 이관이 아니다. 과거 데이터 이관이 필요하면 OQ-5와 별도 story로 분리한다.

## CAP-10 월 손익 계산 정책

### 비용 범위

본사 관리 월 고정비와 지점장이 매일 입력하는 일일 비용을 분리한다.

| 비용 구분 | 예시 | 입력/관리 주체 | 정책 |
| --- | --- | --- | --- |
| 본사 월 고정비 | 월세, 세금, 관리비, 본사 지정 고정 수수료 | 본사 | 월 단위 승인·잠금·정정 기준 필요 |
| 지점 일일 비용 | 식대, 얼음, 주유비, 비품비, 기타 일일 운영비 | 지점장 또는 본사 보완 | 기존 장부 입력과 마감/정정 흐름을 따른다. |
| 매출원가 | FIFO 원가 또는 승인된 원가 기준 | 서버 계산 | 원가 정책 미확정이면 `기준 확인 필요`를 표시한다. |

### 고정비 적용 단위

| 항목 | 기준 |
| --- | --- |
| 적용 단위 | 지점별 월 |
| 적용 시작/종료월 | 월 단위 inclusive range로 기록한다. |
| 비용 항목 | 월세, 세금, 관리비 등 승인된 항목 code를 사용한다. |
| 금액 | 원 단위 정수 후보. 통화 단위 변경은 별도 정책이 필요하다. |
| 승인자 | 본사 운영자 또는 PM 승인자를 기록한다. |
| 사유 | 신규, 변경, 종료, 정정, rollback 사유를 기록한다. |
| lock/version | 월 손익 확정 또는 본사 승인 후 version lock을 둔다. |
| 정정/rollback | append-only 정정 기록을 기본으로 하며 silent overwrite를 금지한다. |

### 월 손익 필수 지표

| 지표 | 의미 | 노출 기준 |
| --- | --- | --- |
| 지점 | report scope 안의 지점 | 본사 권한 |
| 월 | 손익 기준 월 | 본사 권한 |
| 총매출 | correction-applied 매출 합계 | 본사 권한 |
| 매출원가 | 승인된 원가 기준에 따른 월 원가 | 지점장 기본 차단 |
| 매출이익 | 총매출 - 매출원가 | 지점장 기본 차단 |
| 일일 비용 | 장부의 일일 비용 합계 | 본사 권한, 지점장 노출은 별도 정책 |
| 월 고정비 | 본사 관리 월 고정비 합계 | 지점장 기본 차단 |
| 영업이익 | 매출이익 - 일일 비용 - 월 고정비 | 지점장 기본 차단 |
| 정정 반영 여부 | 정정 반영값 사용 여부 | 본사 권한 |
| 미마감 장부 수 | 해당 월 미마감 장부 수 | 본사 권한 |
| 고정비 적용 여부 | 고정비 정책 적용 상태 | 본사 권한 |
| export 가능 여부 | allowlist와 권한 기준 | 본사 export 권한 |

매출원가, 매출이익, 영업이익, 이익률, 본사 고정비는 지점장 surface에서 기본 차단한다. OQ-10B와 CAP-13 고도화 정책 승인 전에는 민감 월 손익 노출 구현 story를 만들지 않는다.

FIFO 원가가 확정되지 않았거나 가격 신뢰 상태가 `basis_missing`이면 월 손익은 확정 숫자 대신 `기준 확인 필요` 또는 `데이터 부족` 상태를 표시한다.

정정 반영값을 기본 운영 숫자로 사용하되, 원본/정정 반영/차이 컬럼은 본사 권한과 export allowlist 안에서만 제공한다.

미마감 장부를 포함할 경우 `미마감 포함` 상태와 미마감 장부 수를 표시하고, 마감 장부만의 확정 손익처럼 보이게 하지 않는다.

## Epic 6 report/export 기준과 재사용 방침

기존 report 경계는 `src/features/reports`, `src/app/app/reports`, `src/app/api/reports/export`, `src/server/calculations/*`이다. 후속 구현은 새 `src/features/excel-report`, `src/features/pnl-report`, `src/features/special-period-report`를 성급히 만들지 않고 기존 경계를 먼저 확장한다.

기존 report queries는 `requireReportAccess()`와 `getHeadquartersStoreScope()`를 사용한다. CAP-2/CAP-10 후속 구현도 같은 서버 권한과 store scope를 통과해야 한다.

기존 export route는 `requireExportCreateAccess()`, report type allowlist, CSV escaping, `Cache-Control: no-store`, `report.export.created` 감사 로그를 사용한다. 후속 export는 이 계약을 우회하지 않는다.

Excel 호환성이 필요해도 1차 계약은 CSV/export allowlist와 서버 계산 기준이다. XLSX dependency나 엑셀 서식 복제는 별도 제품 결정 전 추가하지 않는다.
