## CAP-19 계약/운영 범위 분리

CAP-19는 제품 기능 구현 범위가 아니다. 다음 항목은 운영 계약서 또는 견적서에서 결정한다.

| 항목 | 분류 | 결정 위치 | 제품 backlog 처리 |
| --- | --- | --- | --- |
| 유지보수 대응 시간 | 계약/운영 결정 | 운영 계약서 또는 견적서 | 제품 AC로 확정하지 않음 |
| 월 유지보수 비용 | 계약/운영 결정 | 운영 계약서 또는 견적서 | 청구/정산 기능을 만들지 않음 |
| 서버 이용료 포함 여부 | 계약/운영 결정 | 운영 계약서 또는 견적서 | 서버 비용 관리 화면을 만들지 않음 |
| 장애 보고 방식 | 계약/운영 결정 | 운영 계약서 또는 견적서 | 장애 보고 product surface가 별도 승인되기 전 구현 금지 |
| 백업 확인 책임 | 계약/운영 결정 | 운영 계약서 또는 견적서 | 백업 실행/복구 자동화를 만들지 않음 |
| 서버 세팅 책임 | 계약/운영 결정 | 운영 계약서 또는 견적서 | infra/deployment 설정 변경 금지 |
| 장애 대응 범위 | 계약/운영 결정 | 운영 계약서 또는 견적서 | 온콜/당직/티켓 시스템 구현 금지 |
| 시인성 개선 요청 처리 범위 | 계약/운영 결정 | 운영 계약서 또는 견적서 | 제품 변경 요청은 별도 PRD/change proposal 필요 |

회의 입력 비용 기준은 PRD의 "월 7-8만 원 수준, 서버 이용료 포함 10만 원 이하" 맥락을 참조할 수 있다. 다만 이 숫자는 제품 요구사항, 가격 계산 기능, 청구 모듈, 견적 자동화의 AC가 아니다. 최종 금액, 포함/제외 항목, 결제 주기, 세금/수수료, 서버비 부담 주체는 운영 계약서 또는 견적서에서만 닫는다.

## 제품에 남길 수 있는 최소 product surface 기준

제품 backlog에는 시스템이 직접 제공해야 하는 동작만 남길 수 있다. 운영자가 해야 할 계약상 업무, 대응 시간, 비용, 서버 이용료, 백업 실행 책임은 제품 surface가 아니다.

| 후보 surface | 허용 기준 | 후속 story 조건 |
| --- | --- | --- |
| 운영 로그/상태 | 업로드 실패, 알림 실패, 마감 실패, 권한 오류, 서버 계산 오류처럼 시스템 내부 사건을 사용자가 확인해야 한다. | 별도 PRD/change proposal, 화면/API/권한/감사/test AC 필요 |
| 설정 확인 화면 | 권한, 알림, 기준값, 운영 관련 제품 내부 설정의 현재 상태를 확인해야 한다. | 기존 설정/감사 패턴과 권한 범위 정의 필요 |
| 운영 점검 화면 | 제품 내부 상태, 최근 실패, 설정 누락, 감사 대상 이벤트처럼 시스템이 직접 보유한 점검 정보를 확인해야 한다. | 점검 항목, 권한, 감사, fallback, 테스트 AC를 별도 승인해야 함 |
| 백업 대상 데이터 식별 | 제품 데이터가 복구 대상인지 식별 가능해야 한다. | 해당 도메인 story에서 데이터 보존/감사 기준으로 구현 |
| 감사 로그 | 누가 제품 안에서 설정/마감/정정/업로드/알림 상태를 바꿨는지 추적해야 한다. | 서버 권한 강제, redaction, append-only audit 기준 필요 |
| 실패 상태 표시 | 계산 불가, 데이터 부족, 권한 차단, provider 실패처럼 제품이 알 수 있는 상태를 사용자에게 표시해야 한다. | 상태 의미와 fallback, 권한별 노출 기준 필요 |

다음 항목은 CAP-19만으로 제품 backlog에 남기지 않는다.

| 제외 항목 | 제외 이유 |
| --- | --- |
| 대응 시간 SLA 계산기 | 계약 조건이며 제품 source of truth가 아니다. |
| 비용 청구/정산 | 견적/계약/회계 업무이며 CAP-19 제품 기능이 아니다. |
| 서버 이용료 관리 | 인프라 비용 계약 변수이며 제품 backlog AC가 아니다. |
| 인프라 운영 대시보드 | 서버 운영 대행 범위이며 별도 운영 제품 PRD 전 구현 금지다. |
| 백업 실행/복구 자동화 | 운영 책임과 인프라 절차가 계약에서 확정되어야 한다. |
| 장애 당직 관리 | 운영 조직/계약 책임이며 제품 기능으로 간주하지 않는다. |
| 계약서 생성/전자서명 | 계약 업무 시스템이며 ERP Fish MVP/extension 제품 범위가 아니다. |
| 유지보수 티켓 시스템 | 운영 프로세스 도구이며 별도 PRD 전 구현 금지다. |

구체 product surface가 없으면 CAP-19만으로 구현 story를 만들지 않는다. 구체 product surface가 발생하면 다음 조건을 모두 만족해야 한다.

| 승격 조건 | 필요 내용 |
| --- | --- |
| 별도 PRD 또는 change proposal | 사용자 가치, 비범위, 운영 계약과 제품 기능의 경계, 실패 fallback을 정의한다. |
| 화면/API 계약 | 어떤 화면, API, Server Action/Route Handler가 필요한지 명시한다. |
| 권한/감사 기준 | 대표/소유자, PM, 본사 운영자, 지점장별 접근과 audit event를 정의한다. |
| 테스트 AC | unit/integration/e2e 또는 문서-only 검증 중 무엇이 필요한지 지정한다. |
| owner 승인 | 대표/소유자와 PM이 제품 구현으로 승격한다고 승인한다. |

## OQ-18 결정 상태와 승인 절차

OQ-18 질문은 다음과 같다.

> 유지보수 대응 시간, 월 비용, 서버 이용료 포함 여부는 어떤 계약 기준으로 확정할 것인가?

| 항목 | 기준 |
| --- | --- |
| 결정 등급 | 계약 전 필수 |
| 영향 범위 | CAP-19 |
| 승인 산출물 | 운영 계약서 또는 견적서 |
| 승인자 | 대표/소유자 + Noah Lee(PM) |
| 현재 상태 | 결정 전, 승인 대기 |
| 제품 처리 | 승인 전에는 운영 대행을 제품 기능으로 간주하지 않음 |

대표/소유자와 PM 승인 전에는 CAP-19를 `contract/ops only` 상태로 유지한다. 승인 완료 시에도 제품 구현이 자동으로 열리지 않는다. 승인은 운영 계약 조건을 닫는 행위이며, product surface가 별도로 정의된 경우에만 후속 구현 story를 만든다.

운영 계약서 또는 견적서에는 최소한 다음 항목이 있어야 한다.

| 계약 항목 | 문서화 기준 |
| --- | --- |
| 대응 시간 | 업무 시간/영업외/휴일 기준, 1차 응답과 해결 목표 구분 |
| 월 비용 | 기본 유지보수 비용, 추가 작업/범위 초과 비용, 세금/수수료 처리 |
| 서버 이용료 포함 여부 | 포함/별도 청구/고객 직접 결제 여부 |
| 장애 보고 방식 | 연락 채널, 보고 주기, 장애 종료 보고 기준 |
| 백업 확인 책임 | 누가 백업 상태를 확인하고 복구 실행을 승인하는지 |
| 범위 제외 | 제품 기능 개발, 신규 화면/API, 인프라 이전, 대량 데이터 복구 등 별도 견적 항목 |

## 백업/복구 최소선과 계약 변수 경계

제품이 식별 가능하게 보존해야 하는 데이터 범위는 다음과 같다.

| 데이터 범위 | 제품 최소 책임 |
| --- | --- |
| 일일 장부 | 장부 일자, 지점, 상태, 입력/제출/마감 이력 식별 가능 |
| 마감 | 마감 실행자, 마감 시각, 마감 전 점검 상태, 원본 잠금 기준 식별 가능 |
| 정정 | 원본/이전/정정 반영값, 사유, 작성자, 승인/반영 이력 식별 가능 |
| 감사 로그 | actor, action, target, before/after redaction 기준, reason, timestamp 식별 가능 |
| 업로드 이력 | 원본 파일/행/preview/commit/void/reprocess 상태와 감사 이력 식별 가능 |

PRD 최소선은 유지한다.

| 최소선 | 기준 |
| --- | --- |

| 영업외 장애 | 다음 영업일 아침 회의 전 복구 가능 |
| 감사/업로드 원본·파싱 결과 보존 | 최소 24개월 |

## 구현 금지 사항

- 이 정책 산출물만으로 Route Handler, Server Action, Prisma model/migration, cron/worker, Vercel config, monitoring SDK, backup script, billing module, ticketing module, unit/e2e tests를 추가하지 않는다.
- CAP-19를 이유로 `src/`, `prisma/`, `tests/`, `package.json`, lockfile, deployment/infra config를 수정하지 않는다.
- 운영 계약 비용, 대응 시간, 서버 이용료 포함 여부를 제품 backlog AC로 확정하지 않는다.
- 서버 세팅, 장애 대응, 시인성 개선 요청 처리 범위를 제품 기능 scope로 바꾸지 않는다.
- `운영 로그/상태` 후보만 보고 구현을 시작하지 않는다.
- 백업 주기, 보존 정책 세부값, 복구 실행 책임자 같은 계약 변수를 코드로 고정하지 않는다.
- 승인자 없는 견적/계약 조건을 완료로 표시하지 않는다.
