## CAP-5 품목 정규화 기준표

### 필드 계약

| 구분 | 필드 | 필수 여부 | 의미 | 처리 기준 |
| --- | --- | --- | --- | --- |
| 원문 | `rawProductName` | 필수 | 업로드 또는 기존 장부에서 들어온 원문 품목명 | trim 외 의미 변경 없이 보존한다. 정규 품목 변경으로 덮어쓰지 않는다. |
| 원문 | `rawProductSpec` | 조건부 | 업로드 또는 기존 장부에서 들어온 원문 규격 | 빈 값도 증거로 보존한다. 누락이면 검수/차단 사유가 될 수 있다. |
| 정규 | `name` | 승인 후 필수 | 분석과 품목 마스터 기준의 정규 품목명 | 핵심 상품명만 담고 산지, 등급, 규격, 냉동/생물 힌트는 원칙적으로 제외한다. |
| 정규 | `category` | 승인 후 필수 | 냉동/생물 구분 | MVP/현재 구현에서는 `냉동`, `생물`만 허용한다. 확정 불가 시 검수 상태로 둔다. |
| 정규 | `spec` | 승인 후 필수 | 단가와 재고 분석 단위를 구분하는 규격 | 중량, 수량, 포장 단위, 손질 단위가 단가를 바꾸면 포함한다. |
| 매핑 | `mappingId` | 후속 구현 후보 | 원문/alias와 정규 품목 연결 id | 승인된 mapping만 commit에 사용할 수 있다. |
| 매핑 | `mappingVersion` | 필수 | preview 시점의 mapping 버전 | preview에 고정하고, 변경 시 `revalidation_required`로 전환한다. |
| 검수 | `status` | 필수 | 매핑 검수 상태 | 아래 상태표 중 하나를 사용한다. |
| 검수 | `reviewReason` | 조건부 | 검수 또는 실패 사유 | 자동 추정 근거, 보류 사유, 실패 사유, 승인 근거를 남긴다. |

| 산지 | 제외 | 제외 | 원칙적으로 제외 | 가격/품질 차이가 크면 별도 속성 또는 spec 확장 여부를 후속 결정 |
| 등급 | 제외 | 제외 | 단가를 구분하는 운영 기준이면 후보 | 등급 체계가 불명확하면 `needs_review` |

| 손질 상태 | 기본 제외 | 제외 | 단가/재고 단위를 바꾸면 포함 | 예: 필렛, 절단, 순살 |
| 괄호 표기 | 괄호 밖 핵심명만 포함 | 상태 힌트면 반영 후보 | 규격이면 포함 | 나머지는 `reviewReason` |
| 약어/오탈자 | 직접 확정 금지 | 직접 확정 금지 | 직접 확정 금지 | alias 후보로 두고 본사 승인 필요 |
| 복수 상품 혼합 | 확정 금지 | 확정 금지 | 확정 금지 | `mapping_failed` |

### 본사 검수 흐름

| 상태값 | 한국어 표시 | 의미 | 사용자 액션 | Commit 영향 |
| --- | --- | --- | --- | --- |
| `approved` | 승인됨 | 원문과 정규 품목/규격 mapping이 승인됨 | 본사 검수자가 승인하거나 기존 승인 mapping 재사용 | commit 가능 후보 |
| `needs_review` | 검토 필요 | 후보는 있으나 `name`, `category`, `spec` 중 하나 이상 확신 부족 | 정규 품목 선택, alias 승인, 규격 보정, row 제외 | 기본 commit 차단 |
| `mapping_failed` | 매핑 실패 | 복수 상품, 필수 필드 결측, 후보 충돌, 단위 해석 실패 등으로 후보 생성 불가 | 원문 수정 요청, 수동 분리, row 제외, 재처리 | commit 차단 |
| `deferred` | 보류 | 운영/정책 결정이 필요해 지금 승인하지 않음 | 보류 사유와 재검토 날짜 기록 | commit 차단 |
| `revalidation_required` | 재검증 필요 | preview 이후 mapping, parser, 원본 파일, 정규 품목 기준이 바뀜 | reprocess 실행 후 새 preview 생성 | commit 차단 |

| 날짜 | `YYYY-MM-DD`, `YYYY.MM.DD`, Excel serial date | 파싱 실패, timezone으로 날짜가 바뀌는 변환, 대상 장부 일자 매칭 실패 |
| 금액 | 정수 KRW, 천단위 쉼표 포함 문자열 후보 | 파싱 실패, 소수 금액, 통화 단위 혼합, 수량 x 단가와 불일치 |
| 수량 | 정수 또는 소수 수량 후보 | 파싱 실패, 0/음수 정책 미확정, 단위 혼합으로 해석 불가 |
| 단가 | 정수 KRW 또는 소수 후보 | 파싱 실패, 0/음수 정책 미확정, 공급가/부가세 포함 여부 미확정 |

0 또는 음수 수량/단가는 반품, 보정, 취소와 연결될 수 있으므로 OQ 또는 별도 정책이 닫히기 전에는 commit 차단 사유로 둔다.

### 중복 판단 키

파일명만으로 중복을 판단하지 않는다.

| 레벨 | 키 | 기준 |
| --- | --- | --- |
| 파일 | 파일 해시 | 같은 파일 재업로드 후보 탐지에 사용한다. 파일명은 보조 표시값이다. |
| 배치 | 업로드 배치 id | preview/commit/void/reprocess 이력의 기준 id다. |
| 위치 | source sheet, source row number | 엑셀 원본 위치를 추적한다. |
| 문서 | 원본 문서번호, 공급처, 출고문서 | 있으면 행 identity 보강에 사용한다. |
| 매핑 | 정규화 mapping id/version | 같은 원문 행이라도 mapping version이 다르면 재검증 대상이다. |
| 보조 신호 | 수량, 단가, 금액, raw row hash | 자동 병합 기준이 아니라 중복 후보 표시 보조 신호다. |

## Preview/Commit/Void/Reprocess 상태표

| 상태 | 의미 | 진입 조건 | 전환 가능 상태 | 장부 영향 | 감사 로그 |
| --- | --- | --- | --- | --- | --- |
| `preview` | 파싱/매핑 검증 후 commit 대기 | 파일 업로드, 필수 헤더 매핑, row 파싱 시도 완료 | `committed`, `failed`, `revalidation_required`, `reprocessed` | 없음 | preview 생성, preview 편집, row 제외 권장 |
| `committed` | 매입 라인이 장부에 반영됨 | 미해결 차단 오류 없음, 본사 마감 전 장부, 권한 확인 완료 | `voided`, `reprocessed` | 매입 라인 생성 또는 연결 | 필수 |
| `voided` | commit된 업로드를 무효 처리 | 마감 전 장부 또는 정정 기록 흐름 승인 | `reprocessed` | 마감 전이면 무효 이벤트, 마감 후 직접 수정 금지 | 필수 |
| `reprocessed` | 같은 원본/수정 파일을 다시 처리 | mapping/parser/source 변경, 실패 재시도, 사용자의 재처리 선택 | 새 `preview`, `failed` | 직접 영향 없음. 새 배치로 추적 | 필수 |
| `failed` | 검증/파싱/commit 실패 | 필수 컬럼 누락, 파싱 실패, 권한 실패, DB commit 실패 등 | `reprocessed` | 없음 또는 실패 전 rollback | 필수 |
| `revalidation_required` | 기존 preview가 더 이상 commit 불가 | alias, 정규 품목, mapping version, 원본 파일, parser 변경 | `reprocessed`, `failed` | 없음 | 권장 |

### 실패와 재처리 기준

| 조건 | 상태/처리 | Commit 가능 여부 |
| --- | --- | --- |
| 필수 컬럼 누락 | `failed` 또는 row-level 차단 | 불가 |
| 날짜/금액/수량/단가 파싱 실패 | row-level 오류, batch `failed` 후보 | 불가 |
| 지점/일자 매칭 실패 | row-level 오류 | 불가 |
| `needs_review` 남음 | 검수 필요 | 기본 불가 |
| `mapping_failed` 남음 | 매핑 실패 | 불가 |
| `deferred` 남음 | 보류 | 불가 |
| 중복 후보 미해결 | 사용자 확인 필요 | 불가 |
| preview 이후 mapping 변경 | `revalidation_required` | 불가 |
| preview 이후 원본 파일 변경 | `revalidation_required` | 불가 |
| parser 버전 또는 헤더 매핑 변경 | `revalidation_required` | 불가 |
| commit 중 DB/권한/마감 상태 실패 | `failed`, rollback | 불가 |
| 본사 마감 후 장부 대상 | 정정 기록 흐름 필요 | 직접 commit/void 불가 |

### 마감 전후 처리

- preview는 장부 숫자에 영향을 주지 않는다.
- commit은 본사 마감 전 장부에만 직접 반영한다.
- 본사 마감 후에는 직접 commit/void하지 않고 FR-20 정정 기록 흐름으로 처리한다.
- void/reprocess는 원본 이력을 보존하는 추가 이벤트로만 처리하고 이전 배치와 연결한다.
- reprocess는 기존 배치를 덮어쓰지 않고 새 배치 또는 새 version으로 남긴다.

- 이 정책 산출물만으로 `ProductAlias`, `ProductMapping`, `ImportBatch`, upload parser, upload UI/API, migrations, seed, tests를 추가하지 않는다.

- 원문 품목명/규격을 삭제하거나 정규값으로 덮어쓰지 않는다.
- `PRODUCT_CATEGORY_VALUES`를 `냉동`, `생물` 외 값으로 확장하지 않는다.
- 파일명만으로 중복을 판단하지 않는다.
- preview를 장부 숫자에 반영하지 않는다.
- 본사 마감 후 직접 commit/void하지 않는다.
- 승인자 없는 산출물로 OQ-3/OQ-8/OQ-15 close 또는 CAP-6 구현 완료를 주장하지 않는다.

## 후속 CAP-6 구현 story 참조 문구

후속 CAP-6 구현 story는 반드시 다음 전제를 acceptance criteria 또는 Dev Notes에 포함한다.

1. CAP-5 mapping 계약과 원문 보존 원칙을 선행 조건으로 둔다.
2. 업로드 감사 계약을 참조하고 commit/void/reprocess 감사 로그를 필수로 둔다.
3. OQ-6 샘플 헤더 매핑표가 확정되지 않으면 parser와 header mapping 구현을 시작하지 않는다.
4. OQ-15가 닫히기 전 지점장 비상 매입 입력과 본사 업로드 행 임의 수정은 차단한다.
5. preview는 장부 숫자에 영향을 주지 않고 mapping version을 고정한다.
6. `needs_review`, `mapping_failed`, `deferred`, `revalidation_required`, 필수 컬럼 누락, 중복 후보 미해결, 지점/일자 매칭 실패가 남으면 기본 commit을 차단한다.
