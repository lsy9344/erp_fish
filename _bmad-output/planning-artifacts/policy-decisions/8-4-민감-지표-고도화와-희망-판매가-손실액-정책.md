## CAP-13 고도화 정책 메모

### OQ-10A 위에 쌓는 원칙

OQ-10A는 "지점장에게 기본적으로 보여주지 않을 것"을 닫는 최소 보안 기준이다. OQ-10B는 "특정 지표를 어떤 승인과 감사 조건에서 예외적으로 열 수 있는지"를 다루는 별도 정책이다. 따라서 OQ-10B가 닫히기 전에는 다음을 구현하지 않는다.

- 지점장 민감 지표 허용 toggle
- 민감 지표 dynamic allowlist UI
- 역할별 예외 정책
- 지점별/사용자별 configurable exposure
- 승인 취소 또는 rollback UI/API
- 본사 export/report shape를 지점장 surface로 재사용하는 shortcut

OQ-10B 승인 전 허용되는 범위는 차단 기본값 강화, audit hardening, 테스트 기준 작성, 문서화뿐이다.

### 조정 가능 후보 지표와 기본값

| 후보 지표 | 기본값 | 지점장 기본 허용 | OQ-10B 전 정책 |
| --- | --- | --- | --- |
| 원가 | `blocked` | 아니오 | 매입 단가, 장부 단가, lot 단가, 표준 단가 이름 변형 모두 차단 |
| FIFO 원가 | `blocked` | 아니오 | OQ-7/OQ-17과 OQ-10B 모두 닫히기 전 확정값 노출 금지 |
| 재고금액 | `blocked` | 아니오 | MVP 재고금액, FIFO 재고금액, 조정 전/후 금액 모두 차단 |
| 이익 | `blocked` | 아니오 | 매출이익, 영업이익, 생산성 등 이익 기반 파생값 차단 |
| 마진율 | `blocked` | 아니오 | gross margin, margin rate, 이익률 이름 변형 차단 |
| 희망 판매가 | `blocked` | 아니오 | 원가/마진 역추정 가능성이 있어 지점장 surface 차단 |
| 희망 판매가 기준 손실액 | `blocked` | 아니오 | OQ-9와 OQ-10B 전 `hopedSalePriceLossAmount` gate 유지 |
| 본사 고정비 | `본사 전용` | 아니오 | store allocation, fixed cost, overhead 이름 변형 차단 |
| 타 지점 비교 | `본사 전용` | 아니오 | 전체 지점 평균, 순위, 비교값, scoped store detail 차단 |
| 알림 템플릿 변수 | `blocked` 또는 `본사 전용` | 아니오 | 지점/외부 대상 template 변수에 민감 지표 추가 금지 |

후속 OQ-10B 구현 story가 예외 허용을 다루더라도 기본값은 항상 `blocked` 또는 `본사 전용`이어야 한다. 지점장 허용을 default로 두면 이 정책을 위반한다.

### 정책 변경 필수 필드

| 필드 | 필수 여부 | 기준 |
| --- | --- | --- |
| `approver` | 필수 | Noah Lee(PM), 개발 리드, 본사 운영자 중 정책에 지정된 승인자. 시스템 actor만으로 대체하지 않는다. |
| `reason` | 필수 | 허용하려는 지표, 대상 surface, 업무 필요성, 위험 수용 근거를 포함한다. |
| `effectiveAt` | 필수 | 정책 적용 시작 시각. 과거 소급 적용은 별도 본사 승인과 audit event 필요. |
| `expiresAt` 또는 `reviewBy` | 필수 | 영구 허용을 기본으로 두지 않는다. 만료일 또는 재검토일을 둔다. |
| `rollbackReference` | 필수 | 되돌릴 정책 version, rollback event id, incident/finding id 중 하나를 연결한다. |
| `affectedSurfaces` | 필수 | 화면, Server Action, Route Handler, report, export, cache, 알림 템플릿, 공유 링크 중 영향 범위를 나열한다. |
| `auditEventId` | 필수 | 정책 변경, 승인, rollback, 재노출 점검을 audit trail로 연결한다. |

정책 변경은 "삭제 후 재생성"으로 처리하지 않는다. append-only version 또는 rollback event를 남겨 어떤 기간에 어떤 지표가 어떤 surface에서 허용되었는지 추적 가능해야 한다.

### Rollback 기준

| 상황 | rollback 처리 |
| --- | --- |
| 잘못된 허용 정책 배포 | 기존 정책 row를 삭제하지 않고 새 blocked version 또는 rollback event를 append한다. |
| 지점장 surface에 민감 key/value 노출 | 즉시 blocked version을 적용하고 response/export/cache/알림 artifact 재노출 여부를 점검한다. |
| export artifact 생성 후 정책 오류 발견 | artifact 접근을 중지하거나 재생성하고, 접근 로그와 다운로드 여부를 audit에 연결한다. |
| cache에 본사 값이 저장됨 | 해당 권한/role/store scope cache를 purge하고 `no-store` 또는 권한별 key/value 분리 기준을 재검증한다. |
| 알림 템플릿에 민감 변수 포함 | 지점/외부 대상 템플릿을 rollback하고 발송 이력, 수신 대상, 변수 payload redaction 여부를 점검한다. |

Rollback 완료 기준은 "새 정책이 blocked로 보인다"가 아니라 기존 artifact와 재사용 cache까지 재노출 위험을 점검한 상태다.

## CAP-14 희망 판매가 기준 손실액 정책

### Story 7.5 baseline

| 항목 | Story 8.4 확정 후보 |
| --- | --- |
| 희망 판매가 미입력 | `데이터 부족`. 숫자 0이나 다른 단가로 대체하지 않는다. |
| version 충돌 | `계산 불가`. row split 또는 본사 override evidence 없이는 계산하지 않는다. |
| 대체 금지 | 0원, 기본 단가, 장부 단가, FIFO 원가, `30%단가`, stale 가격, 기존 `LedgerLossItem.amount` 대체 금지 |
| 기존 손실 금액 | `LedgerLossItem.amount`는 현재 사용자가 입력한 손실 금액 snapshot이다. `hopedSalePriceLossAmount`가 아니다. |
| 노출 정책 | 희망 판매가, version id, 실제 처리 단가, 파생 손실액은 지점장 surface에서 key와 value 모두 차단한다. |

### 기준 가격 version과 lock

| 항목 | 정책 |
| --- | --- |
| 기준 가격 | 영업 개시 전 lock된 hoped-sale-price version |
| 가격 입력 마감 | 영업 개시 전 운영 설정값. 설정이 없으면 CAP-14 구현 story에서 먼저 확정한다. |
| 지점장 영업 중 변경 | 다음 영업일 version으로 append한다. 당일 row에는 자동 적용하지 않는다. |
| 당일 적용 | 본사 override 승인, reason, effectiveAt, affected row scope, audit event가 있을 때만 허용한다. |
| 손실 발생 시점 | row의 발생/저장 시각이 version 적용 구간 안에 있어야 한다. 구간이 겹치거나 불명확하면 `계산 불가`다. |
| version reference | 후속 구현에서 손실 row 또는 append-only calculation evidence가 사용한 price version reference를 저장한다. |
| 기존 row 보존 | 가격 변경으로 기존 손실 row를 소급 재계산하거나 원본 row를 overwrite하지 않는다. |

### 손실 유형별 처리 금액 기준

| 정책 코드 | 적용 여부 | 실제 처리 금액 | 손실액 상태 | 비고 |
| --- | --- | --- | --- | --- |
| `LOSS` | 적용 후보 | 회수/보상 금액이 있으면 해당 금액, 없으면 0원 예외 | 기준 version과 처리 금액이 있으면 계산 가능 | 처리 금액 불명확 시 `데이터 부족` |
| `DISPOSAL` | 적용 후보 | 0원 | 기준 version과 수량이 있으면 계산 가능 | 폐기는 정책상 회수 금액 0원 |
| `MARKDOWN` | 적용 후보 | 실제 떨이 판매 단가 또는 금액 | 떨이 금액 없으면 `데이터 부족` | 0원 대체 금지 |
| `UNKNOWN` | 미적용 | 사용하지 않음 | `계산 불가` | 운영 코드 mapping 전 계산 금지 |
| `HQ_EXCEPTION` | 조건부 | 승인 문서 기준 | 승인자/사유/effectiveAt 누락 시 `계산 불가` | 임의 숫자 입력으로 대체 금지 |

후속 구현 산식 후보는 아래 기준을 공유 server calculation module에만 둔다.

```text
hopedSalePriceLossAmount = max((lockedHopedSalePrice - actualDispositionUnitPrice) * quantity, 0)
```

이 산식은 OQ-9, OQ-10B, 손실 유형 mapping, version/lock, 권한별 response shaping 테스트가 모두 승인된 뒤에만 구현한다. 화면, report, export helper가 산식을 복제하면 안 된다.

### `LedgerLossItem.amount` 분리

`LedgerLossItem.amount`는 현재 사용자가 입력한 손실 금액 snapshot이다. 이 값은 기존 MVP 손실 기록, 검토, 마감 흐름을 보존하기 위한 입력값이며 CAP-14의 희망 판매가 기준 파생 손실액이 아니다.

| 값 | 의미 | 후속 구현 정책 |
| --- | --- | --- |
| `LedgerLossItem.amount` | 사용자 입력 손실 금액 snapshot | 기존 row 보존. 파생 손실액으로 재해석 금지 |
| `hopedSalePriceLossAmount` | lock된 희망 판매가와 실제 처리 금액 차이의 파생값 | OQ-9/OQ-10B 승인 후 별도 계산/evidence로 생성 |
| `actualDispositionUnitPrice` | 손실 유형별 실제 처리 단가 | 지점장 surface 차단. 손실 유형별 기준 승인 필요 |
| `hopedSalePriceVersionId` | 계산 기준 price version reference | 지점장 surface 차단. append-only evidence 필요 |

## 서버 응답, export, cache, 알림 템플릿 계약

### DTO/mapper 분리

본사 전용 응답과 지점장 응답은 별도 DTO/mapper로 분리한다. 지점장 응답은 본사 DTO를 만든 뒤 UI에서 숨기는 방식이 아니라 서버 query/action/mapper 단계에서 민감 key와 실제 값을 제거해야 한다.

| surface | 본사 응답 | 지점장 응답 |
| --- | --- | --- |
| 화면/RSC payload | 권한과 store scope 확인 후 본사 전용 지표 후보 포함 가능 | 비민감 입력값, 수량/상태, `데이터 부족`/`계산 불가` label만 포함 |
| Server Action/Route Handler JSON | `requireReportAccess`, `requireExportCreateAccess`, `requireSettingsAccess` 등 action별 권한 확인 | `requireStoreAccess` 후 지점장 전용 DTO. 민감 key/value 없음 |
| report | 본사 report DTO | 지점장 report shape 없음 또는 비민감 상태 요약만 |
| export | 본사 export DTO | 생성 차단. forbidden body에도 민감 key/value 없음 |
| cache | 본사 권한/scope key-value | 권한별 key/value 분리 또는 `no-store` |
| 알림 | 본사 내부 template allowlist | 지점/외부 template에는 민감 변수 추가 금지 |

### 지점장 surface에서 제거할 key/value

지점장 surface에는 다음 key와 실제 값이 모두 남지 않아야 한다. camelCase, snake_case, 한글 라벨, CSV 컬럼명, nested key, cache key, template variable 이름 변형도 같은 의미이면 차단한다.

- `hopedSalePrice`
- `hopedSalePriceVersionId`
- `hopedSalePriceLossAmount`
- `actualDispositionUnitPrice`
- 원가, FIFO 원가, 매입 단가, 장부 단가
- 이익, 매출이익, 영업이익
- 마진율, 이익률
- 재고금액, FIFO 재고금액
- lot 근거, source row, 매입 단가 근거
- 본사 고정비, headquarters overhead, store allocation
- 타 지점 비교, 전체 지점 평균/순위
- 내부 계산 사유, scoped store detail, 본사-only policy reason

### Export 기준

| 항목 | 정책 |
| --- | --- |
| 생성 권한 | export 생성 시점에 `requireExportCreateAccess` 또는 동등 권한을 다시 확인한다. report view 권한만으로 export를 생성하지 않는다. |
| 지점장 요청 | CSV/다운로드 응답을 받지 못해야 한다. forbidden body에는 민감 key/value, store scope detail, 내부 계산 사유가 없어야 한다. |
| 본사 export | 본사 권한과 scope에 맞는 allowlist만 포함한다. 이 shape를 지점장/외부/share surface로 재사용하지 않는다. |
| artifact | 정책 rollback 또는 권한 변경 시 기존 artifact 접근과 재노출 위험을 점검한다. |
| cache header | export와 forbidden response는 `Cache-Control: no-store`를 기본으로 둔다. |

### Cache 기준

본사 권한 결과를 지점장 cache로 재사용하지 않는다. 권한, role, store scope, action permission을 cache key와 value 경계에 반영하거나 `no-store`를 사용한다. key만 분리하고 value 안에 본사 민감 key/value가 남으면 실패다.

| cache 대상 | 기준 |
| --- | --- |
| report result | role/store scope/action permission별 key와 value 분리 |
| export response | `no-store` 기본 |
| forbidden response | `no-store` 또는 민감 detail 없는 짧은 body |
| notification payload 후보 | 대상 audience별 변수 allowlist를 cache 전에 적용 |

### 알림 템플릿 기준

본사 대상과 지점/외부 대상 알림 템플릿을 분리한다. 지점/외부 대상에는 희망 판매가, 파생 손실액, 원가, 이익, 마진율, 재고금액, lot 근거, 본사 고정비, 타 지점 비교 변수를 추가하지 않는다.

| 대상 | 허용 변수 | 금지 변수 |
| --- | --- | --- |
| 본사 내부 | 승인된 본사 지표, 상태, finding id, action 필요 여부 | 승인되지 않은 raw 민감값, 권한 밖 store detail |
| 지점 | 제출 상태, 누락 단계, `확인 필요`, `데이터 부족`, `계산 불가` 등 비민감 상태 | 민감 taxonomy 전체 |
| 외부 | 비민감 공지 문구, 기간, 담당 연락 필요 여부 | 민감 taxonomy 전체, 내부 계산 사유 |

## 서버 차단 테스트 기준

후속 구현 story는 UI 표시 여부가 아니라 server response/export/cache/알림 payload를 검증해야 한다.

| 테스트 대상 | 필수 검증 |
| --- | --- |
| unit DTO/mapper | 지점장 mapper가 민감 key, 한글 라벨, nested key, seeded 민감 값을 제거한다. |
| server/query | 본사 DTO와 지점장 DTO가 분리되어 있고 지점장 경로에서 본사 DTO를 반환하지 않는다. |
| route/export | 지점장 또는 export 권한 없는 actor가 CSV를 받지 못하고 forbidden body에 민감 key/value가 없다. |
| HTML/RSC payload | 지점장 화면 response body에 label, key, seeded value가 없다. |
| cache | 권한별 key/value 분리 또는 `no-store`가 적용된다. 본사 value가 지점장 cache로 재사용되지 않는다. |
| 알림 | 지점/외부 template variable allowlist에 민감 변수가 없다. |
| audit/rollback | 정책 변경과 rollback이 audit event id, affected surface, artifact 재점검 기록을 남긴다. |

검증은 key 이름과 실제 seeded 값을 모두 확인한다. 예를 들어 `hopedSalePriceLossAmount` key가 없어야 하고, seeded amount `123456789` 같은 민감 값도 body/CSV/cache/template payload에 없어야 한다.
