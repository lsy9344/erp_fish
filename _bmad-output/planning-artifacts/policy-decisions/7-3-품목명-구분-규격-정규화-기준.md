### 예외 원칙

- `category`는 현 MVP에서 `냉동`, `생물`만 허용한다. `활어`, `선어`, `건어`, `가공`, `수입`, `국산`을 별도 구분으로 쓰려면 `prisma/schema.prisma`, `productFormSchema`, 품목 관리 UI, 관련 tests를 바꾸는 후속 구현 story가 필요하다.
- 산지와 등급은 원칙적으로 `name`에 넣지 않는다. 다만 운영자가 같은 어종을 산지/등급별로 별도 단가와 재고 단위로 관리하기로 승인하면 후속 story에서 별도 속성 또는 `spec` 확장 기준을 다시 정의한다.
- 규격이 비어 있지만 동일 원문 품목이 과거에 단일 규격으로만 쓰인 경우에도 자동 확정하지 않는다. 본사 검수자가 승인해야 `approved`가 된다.
- 자동 분리는 보조 제안일 뿐이다. 원문이 모호하면 정규값을 추정해 저장하지 않고 검수 상태를 유지한다.

## 검토 필요와 매핑 실패 기준

| 상태값 | 한국어 표시 | 의미 | 사용자 액션 | 후속 처리 |
| --- | --- | --- | --- | --- |
| `approved` | 승인됨 | 원문 품목명/규격이 정규 `name`, `category`, `spec`로 확정됨 | 검수자가 승인 또는 기존 승인 mapping 재사용 | 신규 장부/업로드 preview에 mapping id 적용 가능 |
| `needs_review` | 검토 필요 | 자동 제안은 가능하지만 category/spec/name 중 하나 이상 확신 부족 | 본사 검수자가 정규 품목 선택, 새 alias 승인, 보류 중 선택 | 승인 전 commit 차단 또는 해당 row 제외 |
| `mapping_failed` | 매핑 실패 | 원문이 복수 상품, 필수 필드 결측, 단위 해석 불가, 후보 충돌 등으로 제안 불가 | 원문 수정 요청, 수동 분리, 업로드 재처리 | row-level 오류로 남기고 감사 로그 기록 |
| `deferred` | 보류 | 정책/운영 결정이 더 필요해 지금 승인하지 않음 | 검수자가 보류 사유와 재검토 날짜 기록 | 분석 정규값 적용 금지 |

`검토 필요`는 사람이 확정할 수 있는 후보가 있는 상태이고, `매핑 실패`는 후보를 만들기 어려운 상태다. 두 상태 모두 정규 품목으로 자동 확정하지 않는다.

## 원문 보존과 분석용 정규 품목 원칙

- 기존 장부 snapshot 필드인 `productName`, `productCategory`, `productSpec`, `unitPrice`는 소급 변경하지 않는다.
- 과거 장부와 업로드 trace의 `rawProductName`, `rawProductSpec`는 증거 데이터다. 정규 품목 변경, alias 병합, mapping 변경으로 원문을 덮어쓰지 않는다.
- 분석용 정규 품목이나 mapping id가 도입되더라도 원문 입력값과 별도 컬럼 또는 별도 테이블로 추적한다.
- 품목 병합, alias 승인, mapping 변경이 필요한 경우 새 mapping/version을 만들고 적용 시작일 이후 신규 장부 또는 신규 업로드 preview에만 적용한다.
- 과거 리포트를 새 mapping 기준으로 재분석할 수는 있지만, 그 경우에도 "조회 시점의 분석 mapping version"을 별도로 표시해야 한다. 원본 장부 값 자체를 바꾸지 않는다.
- 비활성 품목, alias 삭제, mapping 병합은 과거 장부 표시명을 변경하지 않는다.

## 이카운트 업로드와 품목 매핑 검수 기준

CAP-6 업로드 필수 논리 필드와 품목 mapping 검수의 관계는 다음과 같다.

| 업로드 필드 | mapping 검수에서의 역할 | 실패 기준 |
| --- | --- | --- |
| 원문 품목명 | `rawProductName`으로 보존하고 alias/mapping 후보 탐색의 주 입력값 | 빈 값, 복수 상품 혼합, 후보 다중 충돌 |
| 규격 | `rawProductSpec`으로 보존하고 정규 `spec` 후보 탐색에 사용 | 빈 값인데 단가 단위 확정 불가, 단위 해석 불가 |
| 수량 | 단위/규격 해석의 sanity check에 사용 | 숫자 파싱 실패, 음수/0 정책 미확정 |
| 단가 | 같은 품목/규격 후보의 가격 범위 sanity check에 사용 | 숫자 파싱 실패, 기준 대비 이상치로 검수 필요 |
| 매입금액 | 수량 x 단가 검증과 row-level 오류 판단에 사용 | 계산 불일치, 숫자 파싱 실패 |

### Preview/commit 원칙

1. 업로드 preview 생성 시점에 `mappingVersion` 또는 동등한 version 식별자를 고정한다.
2. preview 이후 alias, 정규 품목, mapping version이 바뀌면 기존 preview는 `재검증 필요` 상태가 된다.
3. `재검증 필요` preview는 commit할 수 없다. 본사 운영자가 reprocess를 실행해 새 preview를 만든 뒤 commit해야 한다.
4. `needs_review`, `mapping_failed`, `deferred` row가 남아 있으면 기본 commit을 차단한다. 부분 commit이 필요하면 후속 story에서 승인 정책과 감사 로그를 별도로 정의한다.
5. commit된 import row는 원문 필드, 적용 mapping id/version, 검수 상태, 검수자, 승인 시각, 재처리 횟수를 추적해야 한다.

### 자동 매핑 실패 시 본사 검수 흐름

| 항목 | 기준 |

| 필요한 상태 | `needs_review`, `mapping_failed`, `deferred`, `revalidation_required` |
| 사용자 액션 | 정규 품목 선택, 신규 alias 제안 승인, 규격 수정, row 제외, 보류, 업로드 재처리 |
| 감사 로그 필드 | actor, timestamp, source import id, row id, rawProductName, rawProductSpec, before status, after status, selected mapping id/version, reason |

| 차단 조건 | 필수 필드 결측, 후보 충돌 미해결, preview mapping version 불일치, 승인자 없음 |

CAP-6 parser, upload preview/commit/reprocess UI/API, `ProductAlias`/`ProductMapping` schema는 이 story에서 구현하지 않는다.
