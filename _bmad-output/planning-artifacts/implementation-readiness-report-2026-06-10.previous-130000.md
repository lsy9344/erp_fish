FR2: 지점장은 자기 지점의 일일 장부만 조회하고 입력할 수 있어야 하며, 다른 지점 장부 목록과 상세에 접근할 수 없어야 한다. 본사 사용자는 모든 지점 장부 목록과 상세에 접근할 수 있어야 한다.

FR3: 시스템은 주요 입력과 수정에 대해 입력자, 수정자, 수정 시각, 변경 전 값, 변경 후 값을 기록해야 한다. 지점장 입력과 본사 입력/수정 내역은 구분되어 표시되어야 하며, 본사 마감 후 정정 기록은 일반 수정 이력과 구분되어야 한다.

FR4: 본사 사용자와 지점장은 지점+일자 기준으로 일일 장부를 열 수 있어야 한다. 같은 지점+일자에는 하나의 일일 장부만 존재해야 하며, 장부가 없으면 입력 시작 시 생성되고 기존 장부를 다시 열면 입력값이 유지되어야 한다.

FR5: 일일 장부 입력은 매출/결제, 비용, 매입, 재고, 손실/폐기/떨이, 근무인원/특이사항, 검토/제출 순서의 단계형 흐름으로 제공되어야 한다. 검토/제출 화면에서는 주요 누락 항목과 계산 요약을 확인할 수 있어야 하며, 검토 대기는 원본 잠금이 아니어야 한다.

FR6: 입력자는 총매출, 현금, 카드, 기타 결제수단, 매출차액 관련 값을 입력할 수 있어야 한다. 시스템은 결제수단별 금액과 총매출을 저장하고 매출차액을 계산하거나 표시하며, 기준값 초과 시 이상 신호 후보로 표시해야 한다.

FR7: 입력자는 비용 항목, 금액, 메모를 입력할 수 있어야 한다. 하나의 일일 장부에 여러 비용 항목을 등록할 수 있어야 하며, 비용 합계는 영업이익 계산에 반영되어야 한다.

FR8: 입력자는 품목, 규격, 단가, 수량, 매입금액을 입력할 수 있어야 한다. 지점장과 본사 사용자 모두 매입 정보를 입력할 수 있어야 하며, 본사 매입 기준 또는 품목 마스터를 선택해 입력할 수 있어야 한다.

FR9: 입력자는 품목, 규격, 단가, 전일재고, 매입, 판매/차감, 손실, 당일재고, 재고금액, 수량을 품목+규격 단위로 다룰 수 있어야 한다. 월 첫 장부의 전일재고는 월초 재고 스냅샷에서, 이후 영업일의 전일재고는 직전 마감 장부의 당일재고에서 자동 이월되어야 한다.

FR10: 본사 사용자와 지점장은 시스템 계산 재고와 실제 재고가 다를 때 재고 조정을 기록할 수 있어야 한다. 조정 전/후 수량과 금액, 차이, 사유, 작성자, 작성 시각이 남아야 하며, 본사 마감 후 재고 조정은 정정 기록으로만 추가되어야 한다.

FR11: 입력자는 손실, 폐기, 떨이 항목을 품목, 수량, 금액, 처리 유형, 사유/특이사항과 함께 입력할 수 있어야 한다. 손실 항목은 재고 흐름과 손실 리포트에 반영되고, 기준값 초과 시 이상 신호 후보로 표시되어야 한다.

FR12: 입력자는 근무인원과 근무 관련 특이사항을 기록할 수 있어야 한다. 근무인원은 인당생산성 계산에 반영되어야 한다.

FR13: 시스템은 매출, 매출원가, 매출이익, 이익률, 영업이익, 인당생산성, 평균재고, 평균매출, 매출대비 재고비율, 최고매출품목, 최고매출품목 매출액, 매출차액, 재고/손실 관련 지표를 계산해야 한다. 장부 상세, 관제판, 리포트는 같은 계산 기준을 사용해야 한다.

FR14: 시스템은 필수 입력값 누락, 결제수단 합계와 총매출 차이, 상품별 판매금액 합계와 총매출 차이, 재고/손실 이상 후보를 표시해야 한다. 검증 결과는 본사 마감 여부와 구분되어야 하며, 휴무일 장부는 미입력 장부와 구분되어야 한다.

FR18: 본사 사용자는 모든 지점의 본사 마감 전 일일 장부를 직접 입력하거나 수정할 수 있어야 한다. 본사 수정은 변경 이력에 본사 사용자 수정으로 기록되어야 하며, 수정 후 계산값과 이상 신호가 갱신되어야 한다.

FR20: 본사 마감 후 오류가 발견되면 본사 사용자는 원본을 수정하지 않고 정정 기록을 추가할 수 있어야 한다. 정정 기록에는 작성자, 작성 시각, 대상 항목, 원본값, 정정값, 사유가 남아야 하며 원본 수정으로 합쳐지면 안 된다.

FR21: 관제판과 리포트의 기본 숫자는 정정 반영값을 사용해야 한다. 정정 기록 추가 후 관제판과 리포트 숫자가 정정 반영값으로 표시되어야 하며, 상세에서는 원본값과 정정 반영값의 차이를 확인할 수 있어야 한다.

- CAP-11: 시스템은 LINE Messaging API 또는 텔레그램 API로 주요 운영 위험을 정해진 시간에 발송하고 실패/재시도/로그를 남겨야 한다.
- CAP-12: 새 기능은 향후 분석 가능한 구조화 데이터를 남기되, AI 화면, AI API 호출, AI 프롬프트, AI 분석 결과 저장은 구현하지 않아야 한다.
- CAP-13: 지점장 화면과 서버 응답은 매출원가, 매출이익, 인당생산성, 영업이익 같은 민감 회계 지표를 노출하지 않아야 한다.
- CAP-14: 손실액 계산은 희망 판매가와 실제 처리 금액 차이를 반영할 수 있어야 하며, 정책 미확정 항목은 임의 계산하지 않아야 한다.
- CAP-15: 본사는 개별 검토 없이 장부를 일괄 마감할 수 있는 옵션을 갖되, 권한 제한, 위험 알림, 감사 로그를 반드시 적용해야 한다.
- CAP-16: 장부 작성 과정에서 1단계에 입력한 작성자 이름은 7단계 전체 저장까지 유지되어야 한다.
- CAP-17: 재고 테이블의 `차이` 항목은 `당일 판매량`으로 바꾸되, 계산 의미 변경 여부는 리포트와 이상 신호까지 함께 확인해야 한다.
- CAP-18: 본사는 대시보드 테이블의 컬럼 또는 영역 크기를 업무에 맞게 조절할 수 있어야 한다.

### PRD 완성도 평가

PRD는 기본 MVP 요구사항, 비기능 요구사항, 제외 범위, 추가 구현 범위, 성공 지표, open questions를 명확히 포함한다. 다만 추가 구현 범위가 기존 1차 제외 범위 일부를 다시 승격하므로, 구현 계획에서는 MVP 기준선과 추가 구현 범위를 계속 구분해야 한다.

### Critical Violations

#### CR-1: 지점장 민감 회계 지표 제한이 늦게 배치되어 앞선 스토리와 충돌함

**위치:**

- `Story 2.2`는 지점장에게 비용/근무 단계에서 기본 영업이익과 인당생산성을 표시하도록 요구합니다.
- `Story 2.7`은 지점장 또는 본사 사용자 검토 화면에서 매출원가, 매출이익, 영업이익, 인당생산성을 표시하도록 요구합니다.
- `Story 8.1`은 지점장 권한 서버 응답에서 매출원가, 매출이익, 인당생산성, 영업이익을 포함하면 안 된다고 요구합니다.

**문제:**

CAP-13의 보안/권한 요구가 Epic 8에 늦게 배치되어, Epic 2 구현 시점에는 지점장 화면과 서버 응답이 민감 회계 지표를 노출하도록 구현될 수 있습니다. 이는 단순 UX 보완이 아니라 서버 응답 계약과 권한 경계 문제입니다.

**영향:**

Epic 2를 완료한 뒤 Epic 8에서 다시 고치면 보안 회귀와 재작업 위험이 큽니다. 특히 “UI 숨김만으로 보안을 처리하면 안 된다”는 요구와 직접 충돌합니다.

**권장 조치:**

- CAP-13의 서버 응답 제한을 Epic 1 또는 Epic 2의 공통 권한/쿼리 결과 shaping 기준으로 앞당깁니다.
- `Story 2.2`와 `Story 2.7`의 AC를 본사/지점장 역할별로 분리합니다.
- 지점장에게는 현장 입력값, 매출 마진율, 재고 금액, 확인 필요 상태만 반환하도록 명시합니다.
- `Story 8.1`은 신규 구현 스토리가 아니라 민감 지표 제한 검증/강화 스토리로 바꾸거나 제거합니다.

### Major Issues

#### MA-1: Story 11.3은 사용자 스토리가 아니라 구현 가드레일임

**위치:** `Story 11.3: 새 기능은 분석 가능한 구조를 남기되 AI 기능은 구현하지 않는다`

**문제:**

행위자가 `구현 에이전트`이며, “AI 기능을 만들지 않는다”는 범위 제한과 구현 원칙을 다룹니다. 이는 사용자 가치 스토리라기보다 아키텍처/Definition of Done/범위 가드레일입니다.

**영향:**

스토리로 남기면 구현자가 별도 기능처럼 처리하거나, 완료 기준이 코드 변경이 아닌 “하지 않음” 검증으로 모호해질 수 있습니다.

**권장 조치:**

- Story 11.3을 에픽/스토리 목록에서 제거하고, 아키텍처 강제 규칙 또는 모든 CAP 스토리의 공통 DoD로 이동합니다.
- 필요하면 Epic 11은 알림 설정, 알림 발송, 알림 로그까지만 남깁니다.
- “새 운영 데이터는 구조화한다”는 내용은 CAP-1~CAP-18 각 스토리의 데이터 모델 AC로 흡수합니다.

#### MA-2: 일부 스토리는 두 개 이상의 업무 단계를 한 번에 묶어 크기가 커짐

**예시:** `Story 2.2`는 비용 입력, 근무 정보 입력, 비용 합계, 영업이익, 인당생산성, 모바일 입력성까지 함께 포함합니다.

**문제:**

스토리 자체는 사용자 가치가 있지만, 비용 단계와 근무 단계는 데이터 모델, 폼, 검증, 계산 기준이 다릅니다. 민감 지표 제한 문제까지 겹쳐 구현 범위가 커질 수 있습니다.

**권장 조치:**

- 비용 입력과 근무/특이사항 입력을 분리하거나, 최소한 본사/지점장 계산 노출 범위를 AC에서 분리합니다.

### Minor Concerns

- UX 접근성 세부 항목은 여러 스토리에 흩어져 있으나, 공통 접근성 테스트 기준이 별도 체크리스트로 고정되어 있지 않습니다.
- `확인 필요`, `계산 불가`, `데이터 부족` 상태는 좋은 방어 장치지만, 구현 전에 공통 상태 enum과 표시 규칙을 정하지 않으면 화면마다 다른 문구가 나올 수 있습니다.
- Epic 10은 리포트 범위가 넓으므로 구현 스프린트에서는 기간 비교, 연도/월별, 특수기간, 월 손익, 엑셀 항목 매칭의 우선순위를 다시 확인하는 편이 좋습니다.

### Dependency Analysis

- 금지되는 명시적 forward dependency는 CR-1을 제외하면 발견되지 않았습니다.
- `Story 5.3`, `Story 5.4`, `Story 5.5`가 `Story 5.2`의 상세 확인 경로를 참조하는 것은 같은 에픽의 이전 스토리 참조라 허용 가능합니다.

- Epic 11의 알림 발송은 리포트/계산 기반이 필요하지만, 앞선 에픽 이후에 배치되어 순서상 문제는 없습니다.

### Database/Entity Creation Timing
