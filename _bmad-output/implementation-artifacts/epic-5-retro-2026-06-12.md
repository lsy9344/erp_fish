## Epic 요약

Epic 5는 본사가 지점, 품목, 매입 기준, 입력 코드, 이상 신호 기준값을 관리하는 기준정보 기반을 완성했다. 구현 결과는 PRD의 FR-17, FR-22, FR-24, FR-25, FR-26 및 감사/권한 원칙과 맞는다.

완료된 핵심 기반:

- 지점 관리가 생성 시각/수정 시각을 함께 표시하고, 지점 삭제 대신 활성/비활성 상태만 사용한다.
- 품목 관리는 품목명, 구분, 규격, 기본 단가, 활성 상태를 관리하되, 과거 장부 snapshot을 소급 변경하지 않는다.
- 매입 기준 관리는 활성 품목만 신규 기준으로 연결하고, 선택된 기준이 사용자가 편집한 매입 원본값을 조용히 덮어쓰지 않도록 정리됐다.
- 장부 입력 코드는 비용 항목과 손실 유형의 active-only 선택지와 서버 검증을 강화했다. 결제수단 코드는 관리/audit 대상이지만 현 sales/payment 입력은 고정 필드 계약을 유지한다.
- 이상 신호 기준값은 `GLOBAL` scope, `isActive`, 필수 변경 사유, `AuditLog.reason`, no-op audit suppression, 비활성 기준값 signal 차단을 갖게 됐다.
- 기준값이 저장되어도 OQ-1 미확정 항목은 확정 이상 판정으로 승격되지 않고 `기준 확인 필요` 흐름을 유지한다.

## 잘된 점

1. 기존 구현 재사용 원칙이 잘 지켜졌다.
   - 새 `src/features/products`, `src/features/codes`, `src/features/thresholds` 도메인이나 public API route를 만들지 않았다.
   - 기존 `src/features/master-data`, `src/features/dashboard`, Server Action, query, schema, component 패턴 안에서 확장했다.

2. 설정 권한 경계가 일관됐다.
   - master-data page/query/action은 `requireSettingsAccess()`를 기준으로 보호된다.
   - 지점장과 조회 전용 본사 권한은 mutation과 설정 화면 접근에서 차단된다.

3. active/inactive 정책이 삭제보다 안전한 운영 모델로 고정됐다.
   - 지점, 품목, 매입 기준, 입력 코드는 soft activation을 사용한다.
   - 비활성 기준정보는 신규 선택지에서 제외되지만 과거 장부, 감사 로그, snapshot 관계는 보존된다.

4. 감사 로그 품질이 좋아졌다.
   - create/update/status 변경이 before/after snapshot과 함께 남는다.
   - 이상 신호 기준값 변경은 필수 사유를 `AuditLog.reason`에 저장한다.
   - no-op 기준값 저장은 audit row를 늘리지 않는다.

5. Story 4의 정책 gate 학습이 Story 5.5에 적용됐다.
   - threshold row 존재 여부와 정책 승인 여부를 분리했다.
   - inactive threshold는 signal 소비자에게 설정 없음으로 normalize된다.

## 어려웠던 점

1. "기준정보를 바꾸면 신규 선택지는 바뀌지만 과거 원본은 바뀌면 안 된다"는 경계가 모든 story에 반복됐다.
   - 품목, 매입 기준, 입력 코드, threshold 모두 이 원칙을 각자 다르게 위반할 수 있었다.

2. 기존 구현이 이미 상당 부분 존재해, 새 기능 개발보다 구현 감사와 회귀 테스트 보강이 중요했다.
   - 특히 Story 5.2와 5.3은 기능 추가보다 기존 계약을 명시적으로 테스트로 고정하는 성격이 강했다.

3. 결제수단 코드의 제품 기대와 현재 ledger schema 사이에 경계가 있었다.
   - `PAYMENT_METHOD` 코드는 관리/audit/query 대상이지만, 현재 매출/결제 입력은 `cashAmount`, `cardAmount`, `otherPaymentAmount` 고정 필드다.
   - 동적 결제수단 line item 저장은 Epic 5.4에서 암묵적으로 추가하지 않는 것으로 정리했다.

4. E2E 실행 환경 제약이 반복됐다.
   - Story 5.1부터 5.5까지 Playwright webServer가 sandbox 포트 listen 제한으로 테스트 본문에 진입하지 못했다.
   - unit, lint, typecheck, build 또는 diff check는 story별로 통과했지만 브라우저 회귀는 포트 listen 가능한 환경에서 별도 확인이 필요하다.

## 핵심 학습

1. ERP Fish의 기준정보는 원본 장부를 소급 정리하는 도구가 아니라, 신규 입력과 운영 판단의 기준이다.
2. active/inactive는 삭제보다 중요한 감사 가능한 운영 상태다. 비활성 데이터는 신규 선택지에서만 제외하고 과거 조회에서는 살아 있어야 한다.
3. 기준값 저장은 정책 승인과 다르다. OQ-gated 항목은 저장된 값이 있어도 확정 warning/critical로 승격하면 안 된다.
4. payment method code management와 dynamic payment entry storage는 별도 범위다. 현재 ledger sales/payment schema는 고정 필드 계약을 유지한다.
5. 기준정보 변경 action은 권한, 서버 검증, transaction audit, route revalidation, field-level error를 한 묶음으로 다뤄야 한다.

- 기준값 구현은 OQ 승인 상태와 저장된 threshold 값을 분리한다: 완료. Story 5.5가 `isActive`와 OQ-gated downgrade 회귀 테스트를 추가했다.
- 마스터 비활성화 구현은 신규 선택지만 제한하고 과거 원본 표시는 보존한다: 완료. Store/Product/PurchaseStandard/LedgerInputCode 모두 soft activation으로 정리됐다.
- E2E 실행 환경 차단 추적: 미완료. 동일한 `listen EPERM` 제약이 Epic 5에서도 반복됐다.

## Epic 6 준비 평가

Epic 6는 본사 리포트와 안전한 Export를 다룬다. Epic 5의 결과는 Epic 6에 직접 선행 조건을 제공한다.

준비된 의존성:

- 활성 지점/품목/매입 기준/코드 선택지와 비활성 기준정보의 보존 경계
- settings 권한과 report 권한 분리
- audit history target label과 before/after/reason 표시 기반
- anomaly threshold의 저장/비활성/정책 gate 분리
- report/dashboard revalidation 경로

주의할 의존성:

- Epic 6 리포트는 반드시 shared calculation과 correction-applied values를 재사용해야 한다.
- 지점장 또는 권한 없는 본사 사용자는 본사 리포트와 export에서 민감 지표와 권한 밖 지점 데이터를 받을 수 없어야 한다.
- OQ-gated 계산 항목은 export에서도 임시 숫자가 아니라 `기준 확인 필요`, `데이터 부족`, `계산 불가` 상태를 내보내야 한다.
- `PAYMENT_METHOD` 코드가 있다고 해서 리포트/export가 동적 결제수단 line item을 가정하면 안 된다. 현재 매출/결제 원천은 고정 필드다.
- E2E는 현재 sandbox에서 막히므로 Epic 6의 export/권한 회귀는 unit/source-contract 테스트를 더 두껍게 가져가야 한다.

## 결정 사항

1. Epic 5는 story 기준 완료로 판단한다.
   - sprint-status의 Epic 5 story 5개가 모두 `done`이고 story file의 senior review가 승인 상태다.

2. `epic-5-retrospective`는 완료 상태를 유지한다.
   - sprint-status에는 이미 `done`으로 표시되어 있었고, 이번 실행은 새 회고 문서와 문서 감사를 추가하는 방식으로 처리한다.

3. Epic 6 계획은 유지하되, 리포트/export 구현 전 권한·민감 필드·OQ-gated 상태 매트릭스를 story별로 다시 확인한다.

4. 문서 업데이트 후보는 코드와 대조해 검증된 불일치만 수정한다.
   - Architecture는 Epic 5에서 확정된 master-data/settings contract와 현재 repo 구조 일부를 반영해야 한다.
   - README와 PRD는 현재 코드와 충돌하지 않아 수정하지 않는다.
