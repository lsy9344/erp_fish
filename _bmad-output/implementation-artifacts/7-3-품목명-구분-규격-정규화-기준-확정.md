# Story 7.3: 품목명/구분/규격 정규화 기준 확정

Status: done

## Story

As a PM과 본사 운영자,  
I want 기존 품목명을 품목명, 구분, 규격으로 나누는 기준을 확정하고 싶다,  
so that 품목 마스터와 향후 이카운트 업로드 매핑을 같은 기준으로 만들 수 있다.

## Acceptance Criteria

1. **Given** OQ-3이 열려 있을 때, **When** 이 discovery story를 수행한다, **Then** 기존 품목 원문에서 품목명, 구분, 규격을 분리하는 기준을 정해야 한다, **And** 애매한 사례의 처리 원칙을 포함해야 한다.
2. **Given** 샘플 품목 데이터가 있을 때, **When** 본사 운영자가 정규화 기준을 검토한다, **Then** 최소 대표 샘플에 대해 원문, 정규 품목명, 구분, 규격, 비고를 담은 기준표를 만들어야 한다, **And** 매핑 실패 또는 검토 필요 상태를 표현할 수 있어야 한다.
3. **Given** 과거 장부 원본값이 있을 때, **When** 품목 마스터 정규화 정책을 확정한다, **Then** 과거 장부의 원문 품목명/규격은 보존해야 한다, **And** 분석용 정규 품목과 원문 입력값을 구분하는 원칙을 포함해야 한다.
4. **Given** 후속 이카운트 업로드와 품목 매핑이 예정되어 있을 때, **When** 정규화 기준을 승인한다, **Then** 업로드 원문 품목명과 정규 품목을 연결하는 매핑 검수 기준을 포함해야 한다, **And** 자동 매핑 실패 시 본사 검수 흐름이 필요함을 명시해야 한다.
5. **Given** 정책 산출물이 완료되었을 때, **When** PM과 본사 운영자가 승인한다, **Then** 산출물에는 정규화 기준표, 검토 필요 기준, 원문 보존 원칙, 승인자가 포함되어야 한다, **And** MVP-S06 또는 후속 품목 정규화 구현 스토리로 승격할 수 있는지 명시해야 한다.

### Architecture Guardrails

- ERP Fish는 Next.js App Router, Server Components/Server Actions, Prisma, PostgreSQL, Auth.js/NextAuth, Tailwind/shadcn UI를 사용한다. 이 story에서는 새 library, public API, tRPC, client-only persistence, DB migration을 추가하지 않는다. [Source: `_bmad-output/planning-artifacts/architecture.md#Starter Template Evaluation`] [Source: `package.json`]
- Master-data changes must not rewrite historical ledger values. 품목 정규화 정책이 승인되더라도 과거 장부 snapshot은 증거 데이터로 보존하고, 적용 시작일 이후 신규 선택지/업로드 mapping에만 반영한다. [Source: `_bmad-output/planning-artifacts/architecture.md#Data Architecture`]
- 추가 scope의 `ProductAlias` 또는 `ProductMapping`, `ImportBatch`, upload row trace 모델은 architecture에서 후속 normalized structure 후보로 제시되었지만, 이 story의 결과물은 구현 코드가 아니라 승인 가능한 기준 문서다. [Source: `_bmad-output/planning-artifacts/architecture.md#Data Architecture`]
- Do not store business-critical reporting data only in free-text notes or unstructured JSON. 후속 mapping 구현은 원문, 정규 품목, 상태, version, 승인자, 적용 시작일, 감사 정보를 조회 가능한 구조로 남겨야 한다. [Source: `_bmad-output/planning-artifacts/architecture.md#Data Architecture`]
- CAP-6 업로드는 dedicated Route Handler 또는 server boundary를 통해 auth, file parsing, import/audit records, row-level validation errors를 처리해야 한다. Story 7.3은 CAP-6 parser/commit 흐름을 만들지 않고 mapping 검수 기준만 닫는다. [Source: `_bmad-output/planning-artifacts/architecture.md#Route Handler Pattern`]
- Feature code belongs under `src/features/{domain}`, shared server helpers under `src/server`, and persisted data in Prisma. 후속 story가 생기면 `product-mapping`, `imports`, `master-data`, `inventory`, `reports` 경계를 따라야 한다. [Source: `_bmad-output/planning-artifacts/architecture.md#Project Organization`]
- `확인 필요`, `계산 불가`, `데이터 부족` 같은 상태는 표면별로 달라지면 안 된다. 품목 mapping 상태도 `검토 필요`와 `매핑 실패`의 의미를 산출물에서 고정해야 한다. [Source: `_bmad-output/planning-artifacts/architecture.md#Error Handling Pattern`]
