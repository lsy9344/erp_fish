# Story 3.4: 계산 정책 미정 항목 차단과 테스트 기준

Status: done

## Story

As a 개발 리드,
I want 정책이 확정되지 않은 계산이 구현 스토리에 섞이지 않기를 원한다,
so that 잘못된 원가, 이익, 민감 지표가 제품에 먼저 들어가지 않는다.

## Acceptance Criteria

1. **OQ-gated 계산 차단**
   - **Given** 개발자가 계산 모듈을 구현할 때
   - **When** FIFO 원가, `30%단가`, 희망 판매가 기준 손실액, 민감 파생 지표 계산을 발견한다
   - **Then** 해당 계산은 구현하지 않아야 한다
   - **And** 관련 OQ가 닫히기 전에는 `확인 필요` 또는 discovery/policy story로 분리해야 한다.

2. **확정 숫자 대신 상태 반환**
   - **Given** OQ-gated 계산 항목이 UI에 표시될 수 있을 때
   - **When** 서버 응답이 계산 결과를 만든다
   - **Then** 확정 숫자 대신 상태값과 설명을 반환해야 한다
   - **And** 화면은 확정 계산처럼 보이게 표시하면 안 된다.

3. **지점장 민감 파생 지표 차단**
   - **Given** 지점장 권한으로 계산 요약을 조회할 때
   - **When** 민감 파생 지표가 포함될 수 있다
   - **Then** 서버는 해당 필드를 반환하지 않아야 한다
   - **And** OQ-10A 승인 전에는 애매한 원가/이익 기반 값도 숨김을 기본값으로 해야 한다.

4. **본사도 미확정 계산을 확정값으로 받지 않음**
   - **Given** 본사 권한으로 계산 요약을 조회할 때
   - **When** OQ가 닫히지 않은 계산 항목이 있다
   - **Then** 본사에도 확정 계산값처럼 제공하면 안 된다
   - **And** `기준 확인 필요`와 필요한 정책 질문을 함께 표시해야 한다.

5. **차단 회귀 테스트**
   - **Given** 개발자가 테스트를 작성할 때
   - **When** OQ-gated 항목에 대한 테스트를 실행한다
   - **Then** 임시 계산값이 아니라 차단 상태가 반환되는지 검증해야 한다
   - **And** 승인되지 않은 계산 로직이 추가되면 테스트가 실패해야 한다.

6. **정책 승인 후 우회 금지**
   - **Given** 이후 정책이 승인되어 계산이 구현 가능해질 때
   - **When** 새 구현 스토리가 생성된다
   - **Then** 이 스토리의 차단 규칙은 승인 문서와 테스트 기준에 맞게 갱신되어야 한다
   - **And** 기존 차단을 조용히 우회하면 안 된다.

## Dev Notes

### 핵심 구현 방향

- 이 story의 목적은 새 계산 기능을 늘리는 것이 아니라 **승인되지 않은 계산이 들어오지 못하게 코드와 테스트에 guardrail을 세우는 것**이다.
- 현재 `src/server/calculations/ledger.ts`는 `CalculationStatus`, `LedgerReviewMetric`, `calculateLedgerReviewSummary`를 갖고 있고, `policyUnconfirmed()`는 내부 helper로 존재한다. 이 helper를 재사용하거나 공개 gate helper로 정리한다.
- 기존 코드에는 `costOfGoodsSold`, `grossProfit`, `grossMarginRate`, `operatingProfit`, `productivity`, `inventoryAmount`, `salesDifference`가 있다. PRD는 MVP 운영 확인용 기본값과 FIFO 확정 원가를 구분하라고 요구한다. 따라서 모든 원가/이익 필드를 무조건 제거하지 말고, **FIFO/30%단가/희망 판매가/승인 전 민감 노출**을 차단하는 방향으로 좁혀 구현한다. [Source: `_bmad-output/planning-artifacts/prds/prd-erp_fish-2026-05-28-2/prd.md` §4.3]
- `salesDifference`는 OQ-1 임계값/확정 이상 판정과 다르다. 차이값 자체 또는 서버 상태는 유지할 수 있지만, OQ-1 전에는 "임계값 초과", "확정 이상"으로 분류하지 않는다. [Source: `_bmad-output/implementation-artifacts/3-3-검토-화면의-계산-검증-요약.md`]

### 현재 코드 상태

- `src/server/calculations/ledger.ts`:
  - 상태 label은 `ok`, `data-insufficient`, `policy-unconfirmed`, `calculation-unavailable`로 분리되어 있다.
  - `salesDifference`는 `inventoryAdjustments`와 `lossItems` context가 없으면 `policy-unconfirmed`을 반환한다.
  - `costOfGoodsSold`와 `inventoryAmount`는 현재 저장된 재고 수량/단가 기반으로 계산된다. 이 값은 FIFO 확정 원가로 명명하거나 표시하면 안 된다.
- `src/features/ledger/review-queries.ts`:
  - 검토 데이터는 `getStoreLedgerInTx`, `getInventoryStepDataInTx`, `getLossStepDataInTx`, `calculateLedgerReviewSummary`를 조합한다.
  - 파일 상단 주석에는 Story 3.4가 loss/inventory signal threshold를 연결한다고 되어 있지만, 현재 Story 3.4 요구는 OQ-gated 계산 차단과 테스트 기준이다. threshold 연결이 필요하면 이 story의 gate 규칙 안에서만 한다.
- `src/features/ledger/response-shaping.ts`:
  - 지점장 review metric allowlist가 있다. 새 metric을 추가할 때 기본 포함으로 두지 말고 의도적으로 허용한 항목만 추가한다.
- `src/server/sensitive-fields.ts`:
  - 공통 민감 key 차단 목록이 있다. review response shaping과 목록이 드리프트하지 않게 테스트한다.
- `tests/unit/calculation-status.test.mjs`, `tests/unit/ledger-review.test.mjs`, `tests/unit/sensitive-response-shaping.test.mjs`가 이번 story의 핵심 회귀 테스트 위치다.

### 반드시 보존할 동작

- 계산 주체는 서버다. 관제판, 상세, 리포트, export는 같은 서버 계산 함수 또는 같은 gate helper를 사용해야 한다. UI 전용 수식 복제는 금지다. [Source: `_bmad-output/planning-artifacts/architecture.md` "Calculation Strategy"]
- 정책 미정, 데이터 부족, 계산 실패는 `확인 필요`, `데이터 부족`, `계산 불가`로 명시하고 0 또는 오래된 값으로 숨기지 않는다. [Source: `_bmad-output/planning-artifacts/epics.md` NFR11]
- 지점장 화면/API에는 원가, 이익, 마진율, 영업이익, 인당생산성, 재고금액, 원가 lot 근거, 타 지점 비교를 응답하지 않는다. [Source: `_bmad-output/planning-artifacts/prds/prd-erp_fish-2026-05-28-2/prd.md` §4.1; `src/server/sensitive-fields.ts`]
- Story 3.3의 검토 요약 구조와 KST 단계 이동 링크를 깨지 않는다. `stepSummaries`, `missingItems`, `warnings`, `signals`, `summary`는 병렬로 유지한다. [Source: `_bmad-output/implementation-artifacts/3-3-검토-화면의-계산-검증-요약.md`]
- Story 2.9/3.2의 저장 충돌, 권한 우선 검증, field error path를 건드리지 않는다. 이 story는 계산 gate와 테스트 기준이 중심이다. [Source: `_bmad-output/implementation-artifacts/3-2-장부-단계별-입력-검증.md`]

### OQ Gate 기준

- OQ-1: 매출차액 허용 기준/임계값. 차이값 표시는 가능하지만 임계값 초과나 확정 이상 판정은 금지다.
- OQ-2: `30%단가` 의미. 단가 우선순위나 파생 계산에 포함하지 않는다.
- OQ-7/OQ-17: FIFO 적용 범위와 처리 순서. FIFO 확정 원가, FIFO 재고금액, lot 잔량 근거 계산은 discovery/policy story 전까지 금지다.
- OQ-9: 희망 판매가 기준 손실액. 현재 손실 기록은 수량/금액 기록에 한정하고 희망 판매가 파생 손실액은 금지다.
- OQ-10A: 지점장 민감 지표 노출 최소 기준. 애매한 원가/이익 기반 값은 서버 응답 차단을 기본값으로 둔다.
- OQ-14: `차이`를 `당일 판매량`으로 바꾸는 계산 의미 변경. 단순 라벨 변경은 별도 정책 확인 후 가능하지만 계산 의미 변경은 금지다.

- 외부 최신 API 지식이 필요한 작업은 아니다. 구현은 현재 저장소의 서버 계산, response shaping, unit/static test 패턴에 맞춘다.
- `pnpm`은 `packageManager: pnpm@10.31.0`로 고정되어 있다. 이전 story 기록상 PATH에 `pnpm` wrapper가 없을 수 있으므로 필요하면 `corepack pnpm ...` 또는 `./node_modules/.bin/...`를 사용한다.
