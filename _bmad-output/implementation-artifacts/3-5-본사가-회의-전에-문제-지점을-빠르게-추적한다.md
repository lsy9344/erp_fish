# Story 3.5: 본사가 회의 전에 문제 지점을 빠르게 추적한다

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 본사 사용자,
I want 관제판에서 문제 가능성이 높은 지점을 우선적으로 보고 상세로 빠르게 이동하고 싶다,
so that 아침 회의 전에 확인 순서를 정하고 필요한 장부를 먼저 열 수 있다.

## Acceptance Criteria

1. **Given** 관제판에 여러 지점 행이 표시될 때
   **When** 이상 신호 또는 검토 대기/미마감 상태가 있는 지점이 존재하면
   **Then** 문제 가능성이 높은 지점이 먼저 눈에 띄도록 정렬 또는 우선 표시되어야 한다
   **And** 정렬 기준은 사용자가 이해할 수 있는 한국어 레이블로 표시되어야 한다.

2. **Given** 심각 등급과 경고 등급 신호가 함께 있을 때
   **When** 관제판 행 우선순위를 정하면
   **Then** 심각 등급 신호가 있는 지점이 경고 등급만 있는 지점보다 우선되어야 한다
   **And** 같은 등급 안에서는 미마감/검토대기 상태를 구분할 수 있어야 한다.

3. **Given** 본사 사용자가 특정 지점 행을 선택할 때
   **When** 데스크탑에서 행을 클릭하면
   **Then** 장부 상세 Sheet 또는 상세 페이지로 이동해야 한다
   **And** 사용자가 다시 관제판으로 돌아왔을 때 기존 날짜 기준과 필터/정렬 상태가 유지되어야 한다.

4. **Given** 본사 사용자가 태블릿이나 모바일 폭에서 관제판을 볼 때
   **When** 지점 행을 선택하면
   **Then** 상세 페이지로 이동할 수 있어야 한다
   **And** 핵심 상태와 이상 신호 요약은 이동 전 목록에서도 확인 가능해야 한다.

5. **Given** 관제판 기본 조회가 수행될 때
   **When** 활성 지점이 10개 내외이면
   **Then** 상태, 계산 요약, 이상 신호, 우선순위 정보가 3초 안에 표시되는 것을 목표로 해야 한다
   **And** 지점별 상세 감사 이력, 전체 재고 행, 정정 상세 같은 무거운 데이터는 기본 조회에 포함하지 않아야 한다.

6. **Given** 관제판 행이 키보드로 탐색될 때
   **When** 사용자가 Tab, Enter, Space를 사용하면
   **Then** 문제 지점 행 또는 명확한 상세 보기 컨트롤에 접근할 수 있어야 한다
   **And** 포커스 표시가 보여야 한다.

7. **Given** 이상 신호가 없는 지점도 관제판에 표시될 때
   **When** 사용자가 전체 지점 상태를 확인하면
   **Then** 정상 또는 입력 상태를 구분해 볼 수 있어야 한다
   **And** 이상 신호가 없다는 이유로 지점 행이 숨겨지면 안 된다.

### Priority Rules

- 우선순위는 기존 `signals`와 `ledgerStatus`에서만 계산한다.
- `critical` 신호가 하나라도 있으면 최우선이다. 재고/손실 심각 신호가 여기에 해당한다.
- `warning` 신호만 있으면 다음 우선순위다. 매출/이익률/매출차액 경고 신호가 여기에 해당한다.
- 이상 신호가 없더라도 `IN_REVIEW`, `IN_PROGRESS`, `EMPTY` 같은 미완료 상태는 정상 완료/휴무보다 먼저 확인 가능해야 한다.
- `HEADQUARTERS_CLOSED`면서 신호가 없는 row와 `HOLIDAY` row는 낮은 우선순위지만 숨기지 않는다.
- 같은 priority bucket 안에서는 `storeName`과 `storeId`를 tie-breaker로 써서 순서가 안정되게 한다.
- 사용자가 `store-name` 정렬을 선택하면 우선순위 대신 지점명 순으로 볼 수 있어야 한다. 그래도 우선순위 레이블은 행에 남겨야 한다.
