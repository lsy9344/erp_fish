# Story 1.3: 역할 기반 로그인과 지점 접근 제한

Status: done

## Story

As a 지점장 또는 본사 사용자,
I want 로그인 후 내 권한에 맞는 메뉴와 데이터만 볼 수 있기를 원한다,
so that 지점 데이터와 본사 운영 정보가 잘못 노출되지 않는다.

## Acceptance Criteria

1. Given 로그인하지 않은 사용자가 보호된 `/app` 영역에 접근할 때, when 서버가 세션을 확인한다, then 사용자는 로그인 화면 또는 인증 흐름으로 이동해야 한다, and 보호 데이터는 서버 응답에 포함되지 않아야 한다.
2. Given 지점장 계정이 로그인되어 있을 때, when 지점장 앱 영역을 연다, then 연결된 지점의 장부 입력/조회 메뉴만 접근 가능해야 한다, and 타 지점 장부, 본사 설정, 본사 리포트 메뉴는 접근할 수 없어야 한다.
3. Given 본사 계정이 로그인되어 있을 때, when 본사 앱 영역을 연다, then 권한 프로파일에 맞는 본사 메뉴가 표시되어야 한다, and 권한이 없는 설정, 마감, export 같은 action은 UI와 서버에서 모두 차단되어야 한다.
4. Given 사용자가 URL을 직접 입력해 권한 밖 지점 또는 본사 화면에 접근할 때, when 서버 컴포넌트, 서버 액션, 또는 라우트 핸들러가 요청을 처리한다, then 접근은 거부되어야 한다, and 권한 없는 데이터가 클라이언트로 전달되지 않아야 한다.
5. Given 계정의 지점 접근권이나 권한 프로파일이 변경되었을 때, when 사용자가 다음 요청을 보낸다, then 변경된 권한이 서버 기준으로 즉시 반영되어야 한다, and 오래된 클라이언트 UI 상태만 믿고 접근을 허용하면 안 된다.

### Architecture Guardrails

- 인증 기반은 Create T3 App + NextAuth.js/Auth.js + Prisma-backed user/session persistence를 유지한다. 공개 self-signup은 만들지 않는다. [Source: `_bmad-output/planning-artifacts/architecture.md#Authentication & Security`]
- 모든 ledger, report, master-data, close, correction, settings action은 서버에서 authorization을 강제해야 한다. UI 숨김은 보조 수단일 뿐 보안 경계가 아니다. [Source: `_bmad-output/planning-artifacts/architecture.md#Authorization Pattern`]
- `src/server`는 auth session loading, authorization, audit logging, transactions, calculations 같은 cross-cutting server behavior의 소유 경계다. one-off 권한 검사를 feature 파일마다 만들지 않는다. [Source: `_bmad-output/planning-artifacts/architecture.md#Cross-Cutting Concerns`]
- 2026-06-11 authorization update는 `본사 사용자`를 unrestricted role이 아닌 umbrella term으로 보고, close/correction/upload/settings/report/export/store access에 명시적 action helper를 요구한다. [Source: `_bmad-output/planning-artifacts/architecture.md#Authorization Update`]
- App Router는 Server Components for reads, Server Actions for internal mutations, Route Handlers only for true HTTP endpoints 패턴을 따른다. [Source: `_bmad-output/planning-artifacts/architecture.md#API & Communication Patterns`]

- 2026-06-11 확인: Prisma explicit many-to-many relation은 relation table 자체가 model일 때 적합하다. 기존 `PermissionProfileAction`과 `UserPermissionProfile` explicit join을 재사용한다. [External: `https://www.prisma.io/docs/orm/prisma-schema/data-model/relations/many-to-many-relations`]
