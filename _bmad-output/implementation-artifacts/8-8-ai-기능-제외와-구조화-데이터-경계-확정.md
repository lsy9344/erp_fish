# Story 8.8: AI 기능 제외와 구조화 데이터 경계 확정

Status: done

## Story

As a PM과 개발 리드,
I want AI 기능을 구현하지 않으면서 향후 분석 가능한 구조화 데이터 경계를 확정하고 싶다,
so that CAP-12가 현재 제품 범위를 넘지 않으면서도 데이터 품질을 해치지 않게 할 수 있다.

## Acceptance Criteria

1. **Given** CAP-12가 AI 기반 경영 분석 확장성을 요구할 때, **When** 이 discovery story를 수행한다, **Then** 현재 릴리스에서 AI 화면, AI API, 자동 분석 결과, 추천 기능을 구현하지 않는다고 명시해야 한다, **And** 구조화해서 보존할 날짜, 지점, 직원, 품목, 금액, 상태, 이력 필드만 정의해야 한다.
2. **Given** 후속 분석을 고려해 데이터를 저장할 때, **When** 구조화 필드 목록을 작성한다, **Then** 자유 텍스트에만 중요한 업무 값을 묻지 않고, 조회 가능한 컬럼이나 이벤트로 남길 항목을 구분해야 한다, **And** 개인정보와 민감 회계 지표의 권한 차단 기준을 함께 표시해야 한다.
3. **Given** 개발자가 CAP-12를 구현 스토리로 승격하려 할 때, **When** 산출물을 검토한다, **Then** AI 기능 제공이 아니라 데이터 구조화, 감사 가능성, 계산 재사용성만 허용해야 한다, **And** AI 기능이 필요해지면 별도 PRD 또는 change proposal을 요구해야 한다.
4. **Given** 산출물이 완료되었을 때, **When** PM과 개발 리드가 승인한다, **Then** 산출물에는 AI 제외 선언, 구조화 필드 목록, 민감 데이터 차단 기준, 승격 가능 여부가 포함되어야 한다.

### Architecture Guardrails

- ERP Fish는 Next.js App Router, Server Components/Server Actions/Route Handlers, Prisma, PostgreSQL, Auth.js/NextAuth, Tailwind/shadcn UI 기반이다. Story 8.8은 새 library, provider SDK, Route Handler, DB migration, scheduled/background job을 추가하지 않는다. [Source: `_bmad-output/planning-artifacts/architecture.md#Starter Template Evaluation`] [Source: `package.json`]
- Architecture는 "AI 기능 제외와 분석 가능한 구조화 데이터 보존"을 cross-cutting concern으로 둔다. 이는 AI 기능을 만들라는 뜻이 아니라 모든 CAP 구현에서 중요한 업무 값을 자유 텍스트에만 묻지 말라는 데이터 품질 guardrail이다. [Source: `_bmad-output/planning-artifacts/architecture.md#Cross-Cutting Concerns Identified`]
- PostgreSQL/Prisma normalized relational tables가 source of truth다. JSON은 좁은 업로드 trace metadata처럼 보고가 임의 key에 의존하지 않는 경우에만 허용된다. business-critical reporting data를 free-text note 또는 unstructured JSON에만 저장하지 않는다. [Source: `_bmad-output/planning-artifacts/architecture.md#Data Architecture`]
- Server validation은 Zod boundary와 DB constraint가 기준이다. Store-scoped mutation은 identifier parse, authorization/scope check, full validation 순서를 지킨다. [Source: `_bmad-output/planning-artifacts/architecture.md#Validation Strategy`]
- Dashboard, detail, reports, future alert condition evaluation은 shared server calculation modules를 사용해야 한다. AI나 prompt 결과를 계산 source of truth로 두지 않는다. 기준이 없으면 `확인 필요`, `계산 불가`, `데이터 부족` 상태를 반환한다. [Source: `_bmad-output/planning-artifacts/architecture.md#Calculation Strategy`]
- Authorization은 서버에서 강제된다. UI 숨김, table column hide, cache preference, prompt/redaction copy는 보안 boundary가 아니다. [Source: `_bmad-output/planning-artifacts/architecture.md#Authorization Pattern`]
