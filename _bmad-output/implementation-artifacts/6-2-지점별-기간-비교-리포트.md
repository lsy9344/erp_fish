# Story 6.2: 지점별 기간 비교 리포트

Status: done

## Story

As a 본사 사용자,  
I want 선택한 기간의 지점별 실적을 비교할 수 있기를 원한다,  
so that 지점별 흐름과 차이를 회의나 운영 점검에 활용할 수 있다.

## Acceptance Criteria

1. **Given** 본사 사용자가 기간 비교 리포트 화면에 접근할 때, **When** 시작일, 종료일, 지점 필터를 선택한다, **Then** 권한 있는 지점의 기간별 기본 실적 요약이 표시되어야 한다, **And** 권한 밖 지점 데이터는 포함되지 않아야 한다.
2. **Given** 정정 기록이 있는 장부가 기간 안에 포함될 때, **When** 리포트가 집계 값을 계산한다, **Then** 기본 집계는 정정 반영값을 사용해야 한다, **And** 원본값과 정정 차이는 장부 상세에서 추적 가능해야 한다.
3. **Given** 비교 항목에 OQ-gated 계산이 포함될 수 있을 때, **When** 리포트가 결과를 표시한다, **Then** 매출이익, 이익률, 영업이익, 인당생산성, 재고금액 등 민감하거나 정책 미정인 항목은 승인 전 확정값으로 표시하지 않아야 한다, **And** `기준 확인 필요` 또는 권한 차단 상태로 표시해야 한다.
4. **Given** 지점장 권한 사용자가 기간 비교 리포트에 접근할 때, **When** 서버가 요청을 처리한다, **Then** 접근은 거부되어야 한다, **And** 타 지점 비교나 민감 지표는 반환되지 않아야 한다.
5. **Given** 본사 조회 전용 권한 사용자가 기간 비교를 조회할 때, **When** 서버가 응답을 만든다, **Then** 허용된 지점/기간/컬럼만 반환해야 한다, **And** export나 상세 drilldown action은 별도 권한이 없으면 수행할 수 없어야 한다.
6. **Given** 기간에 미마감 장부가 포함될 때, **When** 리포트가 집계를 표시한다, **Then** 미마감 포함 여부와 해당 상태를 표시해야 한다, **And** 마감 장부와 미마감 장부가 섞여 있음을 사용자가 알 수 있어야 한다.

### Gap 분석

- Epic 6.2 AC는 지점 필터를 명시하지만 현재 page props와 query는 `startDate`, `endDate`만 받는다. 지점 필터와 권한 밖 store 처리 테스트가 필요하다.
- PRD FR-28 최소 계약은 기간 A/B, 증감액, 증감률, 기간 A/B 필터와 "증감률 절대값 큰 순서" 정렬을 명시한다. 현재 구현은 단일 기간 집계에 가깝다. 구현자는 FR-28 계약을 충족하도록 A/B 비교를 추가하거나, 제품 결정상 단일 기간 비교로 제한한다면 그 제한을 명시적으로 문서화하고 테스트해야 한다.
- correction action은 `/app/reports/comparison`을 revalidate하지만 ledger save, HQ edit, HQ close, anomaly threshold, store master-data 변경 경로에는 comparison revalidation이 일부 누락되어 있다. 기간 리포트 stale data 위험을 감사해야 한다.
- 현재 `buildStoreComparisonReportRowForTest()`는 function name이 test-only처럼 보이지만 production query도 사용한다. 이름을 바꿀 경우 broad churn을 만들지 말고 export/test 계약과 호출부를 함께 정리한다.
- Story 6.2는 export 파일 생성이 아니다. 화면에 export 버튼을 추가하더라도 실제 다운로드/감사 로그/컬럼 차단은 Story 6.4 범위이므로 비활성 안내 또는 권한 설명까지만 허용한다.

### Architecture Guardrails

- ERP Fish는 Next.js App Router, Server Components, Server Actions, Prisma, PostgreSQL, NextAuth/Auth.js, Tailwind/shadcn UI를 사용한다. 이 story에서 tRPC, 별도 REST API, client-only data fetch layer를 추가하지 않는다. [Source: `_bmad-output/planning-artifacts/architecture.md#Technical Stack`] [Source: `package.json`]
- Report, dashboard, detail pages는 같은 server-side calculation functions를 공유해야 한다. 계산식을 UI 컴포넌트나 report 전용 helper에 복제하면 안 된다. [Source: `_bmad-output/planning-artifacts/architecture.md#Calculation Strategy`]
- Original ledger records는 본사 마감 후 보존된다. Reports and dashboards use correction-applied values by default while details keep original values visible. [Source: `_bmad-output/planning-artifacts/architecture.md#Audit and Correction Strategy`]
- `applyCorrectionValuesToLedgerReviewInput`는 dashboard, reports, close preflight가 공유하는 correction overlay다. Unsupported/not-yet-policy-approved corrections는 `정정 확인 필요` 또는 `기준 확인 필요`로 보여야 한다. [Source: `_bmad-output/planning-artifacts/architecture.md#Audit and Correction Strategy`]
- Authorization은 서버에서 강제한다. 모든 report query는 `REPORT_VIEW`와 store scope를 통과한 본사 사용자에게만 데이터를 반환해야 한다. [Source: `src/server/authz.ts`] [Source: `_bmad-output/planning-artifacts/architecture.md#Authentication & Security`]
- Route Handlers는 exports/health/integration 같은 실제 HTTP endpoint에 한정한다. 기간 비교 화면 조회는 server query + page render로 유지한다. [Source: `_bmad-output/planning-artifacts/architecture.md#API Boundaries`]
- `확인 필요`, `계산 불가`, `데이터 부족`은 shared status enum/display helper 또는 기존 evidence 패턴으로 일관되게 표현해야 한다. [Source: `_bmad-output/planning-artifacts/architecture.md#Process Patterns`]
