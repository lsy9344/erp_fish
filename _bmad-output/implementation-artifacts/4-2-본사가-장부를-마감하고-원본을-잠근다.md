## Story

### User Story

As a 본사 사용자,
I want 검토한 일일 장부를 본사 마감하고 원본을 잠그고 싶다,
So that 회의와 리포트 기준이 되는 원본 장부를 보존할 수 있다.

### Acceptance Criteria

**Given** 본사 사용자가 입력중 또는 검토대기 장부 상세에 접근할 때
**When** 마감 버튼을 선택하면
**Then** "장부를 마감합니다" 확인 Dialog가 표시되어야 한다
**And** 마감 후 지점장이 원본을 수정할 수 없고 정정 기록만 추가할 수 있다는 안내가 표시되어야 한다.

**Given** 본사 사용자가 마감 확인 Dialog에서 마감 확정을 선택할 때
**When** 서버가 요청을 처리하면
**Then** 장부 상태는 `본사마감`으로 변경되어야 한다
**And** `closedBy`, `closedAt`이 저장되어야 한다.

**Given** 장부가 `본사마감` 상태일 때
**When** 지점장 또는 본사 사용자가 원본 입력 저장 Server Action을 직접 호출해도
**Then** 서버에서 원본 수정이 거부되어야 한다
**And** UI disabled 상태만으로 잠금을 처리하면 안 된다.

**Given** 장부가 마감될 때
**When** 상태 변경과 감사 기록을 저장하면
**Then** 마감 상태 변경과 감사 기록은 같은 데이터베이스 트랜잭션 안에서 저장되어야 한다
**And** 감사 기록에는 actor, timestamp, target, before, after가 포함되어야 한다.

**Given** 이미 `본사마감` 상태인 장부에 마감 요청이 다시 들어올 때
**When** 서버가 요청을 처리하면
**Then** 중복 마감 기록이 무의미하게 쌓이지 않아야 한다
**And** 사용자는 이미 마감된 장부라는 안내를 봐야 한다.

**Given** 두 본사 사용자가 같은 장부를 거의 동시에 마감하려 할 때
**When** 첫 번째 요청이 성공한 뒤 두 번째 요청이 처리되면
**Then** 두 번째 요청은 현재 상태를 기준으로 안전하게 거부되거나 이미 마감됨으로 처리되어야 한다
**And** 원본 잠금 상태가 깨지면 안 된다.

**Given** 마감 Dialog가 열릴 때
**When** 키보드 사용자가 조작하면
**Then** Dialog에는 접근 가능한 제목, focus trap, 취소/확정 버튼, 닫힌 후 trigger 포커스 반환이 있어야 한다.

## Dev Notes (must-implement context)

### 핵심 목표
- Story 4.2는 Story 4.1의 지점/본사 편집 가능 로직을 깨지 않으면서, `HEADQUARTERS_CLOSED`(본사마감) 상태에서 원본 편집 경로를 강제 차단해야 한다.
- `본사마감`은 단순 UI 상태 표시가 아니라 저장/트랜잭션 정책에서 enforce되어야 한다(요청 직접 호출 포함).

### 현재 구현/제약 요약
- 4.1 구현에서 `ledger`, `inventory`, `losses`는 지금도 `ledgerId` 기반 HQ 액션과 `IN_PROGRESS/IN_REVIEW` 저장 경로를 갖고 있음.
- 상태 체계는 Enum으로 존재하므로 `본사마감`(Story 상의 `HEADQUARTERS_CLOSED`) 전환은 서버에서 명시적으로 저장해야 한다.
- 기존 4.1에서 `submittedById/submittedAt`은 보존되도록 변경했으므로 4.2에서 건드리지 않는다.
- 기존 dashboard는 summary 집계 조회와 detail 조회를 분리해 가볍게 유지하는 구조.
- `closedAt/closedBy`는 현재 스키마에 없다.

### 개발 가드레일 (실수 방지)
1. 확인 Dialog를 UI에만 둔 뒤 서버에서 검증하지 않으면 안 된다.
   마감 동작, 중복 호출 대응, 동시성 처리는 서버 action에서 보장한다.
2. 마감 액션을 `UPDATE` + `Audit`를 **단일 transaction**으로 처리한다.
3. `close` 요청은 idempotent하게 처리: 이미 마감이면 새 audit 무한 누적을 피하고 사용자에게 안내.
4. 동시 마감 경쟁 상태는 DB 상태 조건 (`where status != HEADQUARTERS_CLOSED`) + transaction with lock-aware update 또는 Prisma transactional isolation로 안정화.
5. 원본 저장 액션은 현재 상태 검사를 각 액션별로 수행:
   - HQ sales/payment action
   - HQ expense action
   - HQ purchase action
   - HQ inventory action
   - HQ loss action
   - HQ work info action
   - 기타 장부 저장 path 모두 대상.
6. 마감/잠금 안내는 4.1의 closed UI 처리 패턴을 확장해 일관화한다.

### 추천 구현 순서
1. `DailyLedger`에 `closedBy`/`closedAt` 추가 (Prisma model, migration).
2. 본사 마감 전용 Server Action을 신규 추가하거나 기존 액션을 분리:
   - 권한 검사: 본사 사용자만 가능
   - 상태 검사: 이미 `HEADQUARTERS_CLOSED`면 idempotent 반환
   - actor + before/after 계산
   - `closedAt`/`closedBy` 업데이트 + 감사 기록
3. 마감 버튼은 `/app/ledgers/[ledgerId]/page.tsx` 또는 하위 컴포넌트에서 처리.
4. 마감 Dialog 접근성: 제목/버튼 label/초점이동/닫힘 후 트리거 복귀 확인.
5. 기존 모든 HQ 저장 action에서 closed 상태 차단 및 closed 안내 메시지 통일.
6. E2E에서 동시 요청, 재호출, 직접 action 호출 차단을 source contract로 검증.
