# Story 4.2: 관제판 이상 상태와 기준 확인 필요 표시

Status: done

## Story

As a 본사 사용자,
I want 관제판에서 문제 가능성이 있는 지점과 기준 미확정 상태를 구분해서 보고 싶다,
so that 확정된 이상 신호와 아직 정책 확인이 필요한 항목을 혼동하지 않는다.

## Acceptance Criteria

1. **계산/검증 상태 구분**
   - **Given** 관제판 행에 장부 상태와 기본 계산 결과가 있을 때
   - **When** 서버가 표시 상태를 계산한다
   - **Then** 필수 누락, 데이터 부족, 계산 불가, 기준 확인 필요 상태를 구분해야 한다
   - **And** 하나의 상태가 다른 상태를 덮어 숨기면 안 된다.

2. **OQ-1 기준 미확정 표시**
   - **Given** 매출차액이나 이익률 급락 기준 OQ-1이 아직 닫히지 않았을 때
   - **When** 관제판이 관련 항목을 표시한다
   - **Then** 임계값 초과나 확정 이상 신호로 표시하지 않아야 한다
   - **And** `기준 확인 필요` 또는 동등한 문구로 정책 미정 상태를 보여줘야 한다.

3. **재고/손실 후보의 확정 이상 오인 방지**
   - **Given** 재고/손실 이상 후보가 있을 때
   - **When** 정책이 확정되지 않은 계산에 의존한다
   - **Then** 확정 이상으로 표시하지 않아야 한다
   - **And** 필요한 정책 질문 또는 관련 OQ를 확인할 수 있어야 한다.

4. **SignalChip 접근성**
   - **Given** SignalChip이 화면에 표시될 때
   - **When** 사용자가 행을 스캔한다
   - **Then** 색상만이 아니라 텍스트, 아이콘 또는 툴팁으로 의미를 알 수 있어야 한다
   - **And** 접근성 이름이 제공되어야 한다.

5. **복수 상태 동시 표시**
   - **Given** 한 지점에 여러 상태가 동시에 있을 때
   - **When** 관제판 행이 렌더링된다
   - **Then** 여러 SignalChip을 함께 표시할 수 있어야 한다
   - **And** 우선순위가 정해지지 않은 항목을 임의로 숨기면 안 된다.

6. **지점장 재사용 응답의 민감 정보 차단**
   - **Given** 지점장 경로에서 상태 요약이 재사용될 때
   - **When** 서버가 응답을 만든다
   - **Then** 민감 지표나 타 지점 비교 정보는 포함하지 않아야 한다
   - **And** 본사 관제판 데이터를 그대로 내려보내면 안 된다.

## Dev Notes

### 핵심 구현 방향

- 이 story는 "확정 이상 판정"을 늘리는 story가 아니다. 관제판이 이미 가진 계산/검증/정정 상태를 정확히 보여주고, 정책 미정 항목을 확정 이상으로 오해하지 않게 하는 story다.
- 새 dashboard query나 dashboard-v2를 만들지 않는다. Story 4.1에서 완성한 `src/features/dashboard` 경로를 확장한다.
- UI 컴포넌트 안에서 계산식을 만들지 않는다. `calculateLedgerReviewSummary`, `applyCorrectionValuesToLedgerReviewInput`, policy gate, anomaly helper 같은 서버 계산 경로를 통과한 결과만 표시한다.
- 확정 이상(`warning`, `critical`)과 확인 필요(`info`)의 의미를 섞지 않는다. OQ-gated 항목은 운영자가 보기에 중요하더라도 확정 이상이 아니라 `기준 확인 필요`다.

### 현재 코드 상태와 보존할 동작

- `src/features/dashboard/types.ts`
  - 현재 `DashboardSignalSummary`는 `id`, `label`, `severity`, `detail`만 가진다.
  - `DashboardSignalSeverity`는 `info | warning | critical`이다.
  - 보존: row shape에는 mutation action이 없다. 조회 전용 본사 row에 action 가능 여부를 암시하지 않는다.

- `src/features/dashboard/queries.ts`
  - 현재 `getHqDashboardRows()`는 `requireReportAccess()`, `getHeadquartersStoreScope()`, `dailyLedger.findMany()`, `getAnomalyThresholdSettingsForSignals()`, `calculateLedgerReviewSummary()`, `applyCorrectionValuesToLedgerReviewInput()`, `evaluateRevenueAnomalySignals()`, `evaluateInventoryLossAnomalySignals()`를 사용한다.
  - `getDashboardSignals()`는 revenue signal, inventory/loss signal, unapplied correction signal을 합친다.
  - `getDashboardPriority()`는 `critical`을 `심각 이상`, `warning`을 `경고 이상`, `info`를 `확인 필요`로 분류한다.
  - 보존: `미입력` 지점은 ledger null row로 남기며 query에서 장부를 자동 생성하지 않는다.
  - 위험: 현재 `anomaly.ts`는 threshold setting이 있으면 `sales-difference-exceeded`, `loss-amount-exceeded`, `inventory-difference-exceeded` 같은 warning/critical을 만들 수 있다. OQ 미승인 상태와 threshold 저장 상태를 분리하지 않으면 Story 4.2 AC 2/3을 위반할 수 있다.

- `src/server/calculations/ledger.ts`
  - `CALCULATION_STATUS_LABELS`는 `ok`, `data-insufficient`, `policy-unconfirmed`, `calculation-unavailable`을 정의한다.
  - `policy-unconfirmed`는 `unavailableReason: "계산 기준 확인 필요"`를 사용한다.
  - 보존: 금액은 integer KRW 기준이고 화면 formatter에서만 통화 표시한다.

- `src/features/ledger/review-validation.ts`
  - 제출 전 필수 누락을 `missingItems`로 산출한다.
  - 보존: 필수 누락은 입력 검증 상태이고, `data-insufficient`, `policy-unconfirmed`, `calculation-unavailable`은 계산 상태다. 관제판에서는 둘을 함께 보여줄 수 있지만 한 상태로 합치면 안 된다.

- `src/server/calculations/policy-gates.ts`
  - `salesDifferenceThresholdAnomaly`는 OQ-1에 묶여 있다.
  - `thirtyPercentUnitPrice`, `fifoCostOfGoodsSold`, `fifoInventoryAmount`, `hopedSalePriceLossAmount`, `storeManagerSensitiveDerivedMetrics`, `salesDifferenceMeaningChange`도 승인 전 차단 상태다.
  - 보존: 이 story에서 새 정책 확정이나 계산 구현으로 바꾸지 않는다.

- `src/server/calculations/anomaly.ts`
  - `evaluateRevenueAnomalySignals()`는 threshold가 없으면 `기준값 설정 전` info signal을 반환한다.
  - threshold가 있으면 매출 하락률, 이익률 하락폭, 매출차액을 평가할 수 있다. 이 로직을 관제판에서 그대로 확정 이상으로 쓰려면 OQ 승인 상태가 필요하다.
  - `evaluateInventoryLossAnomalySignals()`도 threshold가 있으면 재고/손실 초과를 critical로 만들 수 있다. 정책 미정 항목은 info 상태로 남겨야 한다.

- `src/features/dashboard/components/dashboard-signal-summary.tsx`
  - lucide `InfoIcon`, `TriangleAlertIcon`, `CircleAlertIcon`와 shadcn `Badge`를 사용한다.
  - `showDetails`가 true면 detail을 visible text로 표시하고, false면 `title`에 detail을 둔다.
  - 보존/보강: badge 자체의 접근성 이름이 detail까지 전달되는지 확인한다.

- `src/features/dashboard/components/hq-dashboard-table.tsx`
  - desktop table과 mobile card view가 모두 `DashboardSignalSummary`를 사용한다.
  - 숫자/시각 컬럼은 `tabular-nums`를 사용한다.
  - 보존: row click/keyboard activation과 상세 link query state를 깨지 않는다.

### 기술 스택과 최신 문서 확인

- `package.json` 기준 repo는 Next `^15.2.3`, React `^19.0.0`, Prisma `^6.6.0`, next-auth `5.0.0-beta.25`, Zod `^3.24.2`, Playwright `^1.60.0`, Tailwind `^4.0.15`, shadcn `^4.8.2`, pnpm `10.31.0`를 사용한다.
- 2026-06-12 공식 Next 문서 확인 결과 최신 문서 사이트는 Latest Version `16.2.2`를 표시하지만, 이 story에서 framework major upgrade는 하지 않는다. 현재 repo의 Next 15 App Router 패턴을 유지한다. [Source: https://nextjs.org/docs/app/getting-started/server-and-client-components]
- Next 공식 문서는 page/layout이 기본적으로 Server Components이고, event handler/browser API가 필요할 때 Client Components를 사용한다고 설명한다. 따라서 관제판 서버 데이터 계산은 Server Component/query helper에 두고, row click 같은 상호작용만 client component에 둔다. [Source: https://nextjs.org/docs/app/getting-started/server-and-client-components]
- Next `revalidatePath`는 Server Functions와 Route Handlers에서만 호출 가능하다. 이 story에서 mutation을 만들 가능성은 낮지만, threshold/설정 변경과 연결한다면 client component에서 직접 revalidate하지 않는다. [Source: https://nextjs.org/docs/app/api-reference/functions/revalidatePath]
- Prisma 공식 CRUD 문서는 `select`로 필요한 필드만 반환하는 패턴을 제시한다. 관제판 query는 Story 4.1처럼 필요한 ledger/inventory/loss/correction 필드만 select하고 감사 로그 대량 목록을 붙이지 않는다. [Source: https://www.prisma.io/docs/orm/prisma-client/queries/crud]
- Playwright 공식 문서는 `webServer`가 테스트 전에 로컬 dev server를 띄우는 옵션이라고 설명한다. 기존 환경에서 포트 바인딩 제한으로 E2E가 실패한 이력이 있으므로, 실패 시 원인이 테스트 코드인지 sandbox 포트 제한인지 로그에 분리 기록한다. [Source: https://playwright.dev/docs/test-webserver]

### 권한 및 보안 가드레일

- 관제판 query는 계속 본사 사용자 + `REPORT_VIEW` + store scope를 서버에서 확인해야 한다.
- 지점장 direct URL 접근은 전체 지점 목록이나 타 지점 데이터를 반환하지 않아야 한다.
- 지점장 경로에서 상태 요약을 재사용할 경우, 본사 `HqDashboardRow`를 그대로 serialize하지 않는다.
- 지점장 응답에는 이익률/마진율, 재고금액, 원가 lot 근거, 타 지점 비교, 본사 전체 ranking, threshold 수치 등 민감하거나 비교성 있는 값이 없어야 한다.

### UX / 접근성 가드레일

- SignalChip은 색상 단독 전달 금지다. 아이콘과 label은 필수이고 detail은 tooltip/title, visible text, 또는 aria name으로 전달한다.
- `기준 확인 필요`, `데이터 부족`, `계산 불가`는 서로 다른 상태로 보여야 한다.
- OQ-gated 항목은 "초과", "급락", "확정 이상"처럼 단정하지 않는다.
- 여러 SignalChip은 한 행에 함께 표시될 수 있어야 하며, 작은 화면에서 줄바꿈되어도 텍스트가 버튼/배지 밖으로 넘치면 안 된다.
- 관제판은 desktop 우선 `max-w-[1600px]` shell, table 가로 스크롤, mobile card view를 유지한다.
