# Story 7.6: 지점장 민감 필드 노출 차단 매트릭스 승인

Status: done

## Story

As a PM과 개발 리드,
I want 지점장에게 절대 노출하지 않을 민감 필드와 surface별 차단 방식을 승인하고 싶다,
so that 화면, API, export, 캐시, 알림에서 원가/이익 정보가 새지 않게 할 수 있다.

## Acceptance Criteria

1. **Given** OQ-10A가 열려 있을 때, **When** 이 discovery story를 수행한다, **Then** 지점장에게 차단할 원가, 이익, 마진율, 재고금액, lot 근거, 본사 고정비, 타 지점 비교 필드를 정의해야 한다, **And** 애매한 원가/이익 기반 파생값은 기본 차단으로 분류해야 한다.
2. **Given** surface별 노출 기준을 정할 때, **When** 화면, 서버 응답, export, 공유 링크, 캐시 응답, 알림 템플릿을 검토한다, **Then** 각 surface별 allowed/blocked field matrix를 작성해야 한다, **And** UI 숨김만으로는 승인 기준을 만족하지 않는다고 명시해야 한다.
3. **Given** 본사 권한과 지점장 권한이 같은 데이터를 다른 방식으로 볼 때, **When** 매트릭스를 승인한다, **Then** 본사 전용 필드와 지점장 허용 필드를 분리해야 한다, **And** 본사 전용 응답을 지점장 경로에서 재사용할 때 권한 재검사가 필요하다는 기준을 포함해야 한다.
4. **Given** 서버 차단 테스트 기준을 정할 때, **When** 승인 산출물을 작성한다, **Then** 지점장 세션에서 민감 필드가 응답 JSON, export, 캐시 응답에 포함되지 않는 테스트 기준을 포함해야 한다, **And** 클라이언트 숨김 테스트만으로 통과하면 안 된다.
5. **Given** 정책 산출물이 완료되었을 때, **When** PM과 개발 리드가 승인한다, **Then** 산출물에는 surface x 권한별 allowed/blocked field matrix, 서버 응답 차단 테스트, 감사 로그 기준, 승인자가 포함되어야 한다, **And** MVP-S09를 구현 스토리로 승격할 수 있는지 명시해야 한다.
6. **Given** OQ-10B가 아직 열려 있을 때, **When** OQ-10A가 승인된다, **Then** CAP-13 고도화의 지표별 노출 허용 기능은 자동으로 열리면 안 된다, **And** 고도화 노출 정책은 별도 OQ-10B 산출물로 남겨야 한다.
