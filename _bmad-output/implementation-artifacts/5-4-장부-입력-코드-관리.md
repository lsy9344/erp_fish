# Story 5.4: 장부 입력 코드 관리

Status: done

## Story

As a 본사 설정 관리자,  
I want 결제수단, 비용 항목, 손실 유형 같은 입력 코드를 관리할 수 있기를 원한다,  
so that 장부 입력 선택지가 본사 운영 기준에 맞게 유지된다.

## Acceptance Criteria

1. **Given** 설정 관리자 권한을 가진 본사 사용자가 코드 관리 화면에 접근할 때, **When** 코드 목록을 조회한다, **Then** 코드 유형, 표시명, 정렬 순서, 활성/비활성 상태를 볼 수 있어야 한다, **And** 권한 없는 사용자는 코드 관리 mutation을 수행할 수 없어야 한다.
2. **Given** 설정 관리자가 새 코드를 추가할 때, **When** 코드 유형과 표시명이 유효하다, **Then** 새 코드는 해당 입력 단계의 선택지로 사용할 수 있어야 한다, **And** 생성 이벤트는 감사 로그에 남아야 한다.
3. **Given** 설정 관리자가 코드 표시명이나 정렬 순서를 수정할 때, **When** 저장 요청이 성공한다, **Then** 신규 입력 화면에는 수정된 코드가 반영되어야 한다, **And** 과거 장부에 저장된 원본 코드 표시가 임의로 깨지면 안 된다.
4. **Given** 설정 관리자가 코드를 비활성화할 때, **When** 저장 요청이 성공한다, **Then** 비활성 코드는 신규 입력 선택지에서 제외되어야 한다, **And** 과거 장부에서 이미 사용된 코드는 조회 가능한 상태로 유지되어야 한다.
5. **Given** 장부 입력 화면이 선택지를 불러올 때, **When** 활성 코드가 존재한다, **Then** 매출/결제, 비용, 손실 유형 단계는 활성 코드만 신규 선택지로 보여줘야 한다, **And** 코드를 불러오지 못하면 사용자가 입력 오류를 이해할 수 있어야 한다.
6. **Given** 중복 코드나 잘못된 코드 유형이 입력될 때, **When** 서버가 요청을 검증한다, **Then** 저장은 거부되어야 한다, **And** UI는 해당 필드에 연결된 오류 메시지를 표시해야 한다.

### Architecture Guardrails

- ERP Fish는 Next.js App Router, Server Components, Server Actions, Prisma, PostgreSQL, Auth.js/NextAuth, Tailwind/shadcn UI를 사용한다. 이 story에서 public API route나 tRPC abstraction을 추가하지 않는다. [Source: `_bmad-output/planning-artifacts/architecture.md#Starter Template Evaluation`] [Source: `package.json`]
- master data and settings는 `src/features/master-data`와 `src/app/app/master-data`에 둔다. `src/components/ui`에는 shadcn primitive만 둔다. [Source: `_bmad-output/planning-artifacts/architecture.md#Project Structure & Boundaries`]
- 모든 settings query/action은 shared authorization helper를 먼저 호출해야 한다. UI 숨김은 보안 경계가 아니다. [Source: `_bmad-output/planning-artifacts/architecture.md#Authorization Pattern`]
- business-changing action은 business write와 audit write를 같은 transaction에 넣는다. 감사 이벤트는 actor, target, before, after를 포함해야 한다. [Source: `_bmad-output/planning-artifacts/architecture.md#Audit Pattern`]
- expected validation/business error는 `ActionResult<T>`로 반환한다. 새 result shape를 만들지 않는다. [Source: `_bmad-output/planning-artifacts/architecture.md#Action Response Formats`]
- master-data 변경은 과거 ledger snapshot을 임의로 rewrite하면 안 된다. 비활성화는 신규 선택지 제외를 의미하며 삭제가 아니다. [Source: `_bmad-output/planning-artifacts/architecture.md#Data Architecture`]
- money/date/calculation behavior를 결제수단 코드 때문에 UI 전용으로 갈라놓으면 안 된다. 결제 합계와 매출차액은 관제판, 검토, 마감, 리포트가 공유하는 서버 계산 경로와 호환되어야 한다. [Source: `_bmad-output/planning-artifacts/architecture.md#Calculation Strategy`]

### UX and Accessibility Notes

- 코드 관리 화면은 본사 기준정보 업무 화면이다. 조밀한 검색/필터/표/행 작업 중심으로 유지하고 마케팅형 hero/card layout을 만들지 않는다.
- 데이터가 없어도 세 코드 그룹 `결제수단`, `비용 항목`, `손실 유형`은 빠른 필터나 안내에서 확인 가능해야 한다.
- 폼 오류는 필드 하단 한국어 메시지와 `aria-invalid`, `aria-describedby`, 첫 오류 focus를 유지한다.
- 표시 순서 입력은 `inputMode="numeric"`과 서버 정수 검증을 모두 유지한다.
- 비활성 과거 코드는 기존 장부 행에서 원래 이름을 볼 수 있어야 하지만, 신규 선택지처럼 보이면 안 된다.
- 모바일 장부 입력에서 비용/손실 코드 select가 좁은 화면의 저장 버튼이나 금액 input을 덮지 않도록 `min-w-0`, wrapping, 안정적인 control height를 유지한다.
