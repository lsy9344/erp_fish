## Epic 요약

Epic 6는 본사 전용 일별, 기간 비교, 월간 리포트와 안전한 CSV export를 완성했다. 구현 결과는 PRD의 FR-27, FR-28, FR-29와 민감 지표 차단 원칙을 따른다.

완료된 핵심 기반:

- 일별 리포트는 최신 반영 시각, 상태 메시지, 장부/영업 상태, 정정 반영 evidence, OQ-gated 상태를 보여준다.
- 기간 비교 리포트는 권한 범위 내 지점 필터, 단일 선택 기간 비교, 권한 밖/비활성 지점 no-row 안내, 미마감 포함 신호를 제공한다.
- 월간 리포트는 권한 밖 지점 fallback을 제거하고, 정정 반영 건수와 실제 미마감 장부 포함 상태를 월간 KPI와 상태 요약에 드러낸다.
- Export는 `/api/reports/export` Route Handler와 `src/features/reports/export.ts` thin helper로 구현됐다.
- Export 생성은 `requireExportCreateAccess()`로 먼저 차단하고, 기존 report query가 다시 `requireReportAccess()`와 store scope를 수행한다.
- CSV는 서버 allowlist, sanitized filename, `no-store` response, `ReportExport` audit log, formula-leading value 방어를 갖는다.

## 잘된 점

1. 기존 리포트 경계 재사용이 일관됐다.
   - 새 `src/features/daily-report`, `src/features/comparison-report`, `src/features/monthly-report` 도메인을 만들지 않았다.
   - report 화면 조회는 Server Component와 `src/features/reports/queries.ts`에 남겼고, export만 Route Handler로 분리했다.

2. 권한 경계가 서버에서 반복 검증됐다.
   - 각 report query는 `requireReportAccess()`와 `getHeadquartersStoreScope()`를 사용한다.
   - export route는 `requireExportCreateAccess()`를 먼저 수행하고, query 재호출로 report 권한과 store scope를 다시 적용한다.
   - 조회 전용 본사와 지점장은 CSV 링크 노출과 직접 route 요청 모두에서 차단된다.

3. correction-applied value와 원본 추적 경계가 유지됐다.
   - 리포트 기본 숫자는 shared correction overlay를 사용한다.
   - 원본/정정 반영 차이는 evidence, 장부 상세, 정정 타임라인 기준으로 추적 가능하다.
   - export도 기존 evidence/status 값을 변환하고 DB raw field를 별도 계산하지 않는다.

4. OQ-gated 지표를 임시 숫자로 승격하지 않았다.
   - `기준 확인 필요`, `데이터 부족`, `계산 불가`, `정정 확인 필요`를 보존했다.
   - 최고매출품목, 평균재고, 재고비율, 이익률 같은 민감/정책 미정 지표는 확정 숫자처럼 보이지 않게 처리했다.

5. Story review가 실제 결함을 자동 수정했다.
   - E2E header assertion 오류 수정
   - rejected `storeId`가 selected state와 self-link에 남는 문제 수정
   - 월간 `미마감 포함`이 미입력일을 포함하던 문제 수정
   - export calendar validation, 권한 밖 `storeId` CSV 생성 차단, CSV formula injection 방어 수정

## 어려웠던 점

1. 기존 구현을 새 epic 번호와 계약으로 정리하는 작업이 반복됐다.
   - 일부 테스트와 story 맥락에 과거 Story 5.x 명칭이 남아 있어, 기능 추가보다 source-contract 명명과 범위 재확인이 중요했다.

2. PRD의 이상적 리포트 계약과 MVP 구현 범위를 명시적으로 분리해야 했다.
   - FR-28의 기간 A/B, 증감액, 증감률 계약은 현재 단일 선택 기간 비교로 제한됐다.
   - Story 6.2는 이 제한을 UI copy와 테스트로 고정했다.

3. 권한 밖 지점 필터 처리의 UX와 보안 경계가 미묘했다.
   - 권한 밖 `storeId`를 다른 지점으로 fallback하면 사용자에게 잘못된 데이터로 보일 수 있다.
   - Epic 6에서는 no-row와 명확한 안내를 선택하고, rejected id를 selected state에 남기지 않도록 수정했다.

4. Export는 파일, header, audit metadata까지 모두 노출 surface였다.
   - 단순히 화면 컬럼만 숨기는 것으로는 충분하지 않았다.
   - filename, response body, audit `after`, forbidden payload까지 민감 정보가 남지 않는지 테스트가 필요했다.

5. E2E 실행 환경 제약이 계속 반복됐다.
   - Story 6.1-6.4 모두 Playwright webServer가 `listen EPERM`으로 중단됐다.
   - unit/source-contract 테스트가 보강됐지만, 브라우저 다운로드와 감사 로그 흐름은 포트 바인딩 가능한 환경에서 별도 확인이 필요하다.

## 핵심 학습

1. 리포트와 export는 별도 계산 도메인이 아니라 기존 dashboard/detail 계산과 correction overlay를 소비하는 view/export layer다.
2. 권한 밖 필터는 조용한 fallback보다 명시적 no-row가 안전하다.
3. Export는 UI 기능이 아니라 보안 surface다. route, CSV cell, header, filename, audit metadata까지 같은 차단 원칙을 적용해야 한다.
4. OQ-gated 항목은 숫자를 비워 두는 것이 아니라 상태와 사유를 사용자와 파일에 명확히 남겨야 한다.
5. Story review는 구현 완료 후에도 scope mismatch와 테스트 오해를 잡아내는 품질 장치로 유효했다.

## Epic 7 준비 평가

Epic 7은 구현 전 정책 확정 트랙이다. Epic 6 구현은 Epic 7의 정책 질문을 대체하지 않고, 정책 미정 상태를 안전하게 표시하는 기반을 만들었다.

준비된 의존성:

- OQ-gated 계산 상태를 report/export에 노출하는 패턴
- 민감 필드 서버 차단과 export 권한 재검증 패턴
- 권한 밖 store filter no-row 패턴
- report/export audit target `ReportExport`
- CSV export의 안전한 파일 응답과 audit metadata

주의할 의존성:

- Epic 7.1에서 매출차액/이상 신호 기준이 승인되기 전까지 기존 화면과 export는 `기준 확인 필요`를 유지해야 한다.
- Epic 7.2와 7.6은 `30%단가`와 민감 필드 역추정 가능성을 함께 검토해야 한다.
- Epic 7.4가 FIFO 정책을 닫기 전에는 재고금액, 매출원가, 이익률을 FIFO 확정값처럼 표시하면 안 된다.
- Epic 7.7의 `차이` 용어 변경은 단순 라벨인지 계산 의미 변경인지 결정될 때까지 리포트/export 컬럼명 변경을 신중히 다뤄야 한다.
