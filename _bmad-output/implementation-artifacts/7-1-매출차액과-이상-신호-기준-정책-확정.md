# Story 7.1: 매출차액과 이상 신호 기준 정책 확정

Status: done

## Story

As a PM과 본사 운영자,  
I want 매출차액과 이상 신호 기준을 확정하고 싶다,  
so that 관제판과 검토 화면이 임시 판단 없이 같은 기준으로 위험 지점을 표시할 수 있다.

## Acceptance Criteria

1. **Given** OQ-1이 열려 있을 때, **When** 이 discovery story를 수행한다, **Then** 매출차액 기준을 금액, 비율, 또는 둘 다로 볼지 결정해야 한다, **And** 부호 처리와 예시 계산을 포함해야 한다.
2. **Given** 매출/이익률 급락 기준을 정할 때, **When** 본사 운영자가 기준값을 검토한다, **Then** 기준값의 기본값, 지점별/전체 적용 여부, 변경 가능 범위를 문서화해야 한다, **And** 기준값 변경 시 감사 로그에 남겨야 할 항목을 정해야 한다.
3. **Given** 이상 신호 우선순위가 필요한 경우, **When** 여러 이상 신호가 한 지점에 동시에 발생한다, **Then** 표시 순서와 중복 표시 방식을 정해야 한다, **And** 임의로 신호를 숨기지 않는 원칙을 포함해야 한다.
4. **Given** 정책 문서가 완료되었을 때, **When** PM과 본사 운영자가 승인한다, **Then** 산출물에는 매출차액 기준표, 부호 예시, 임계값 예시, 표시 문구, 승인자가 포함되어야 한다, **And** MVP-S04를 구현 스토리로 승격할 수 있는지 명시해야 한다.
5. **Given** 정책이 아직 승인되지 않았을 때, **When** 구현 스토리를 만들려 한다, **Then** 매출차액 임계값 판정이나 이상 신호 확정 로직은 구현 스토리로 만들면 안 된다, **And** 기존 화면은 `기준 확인 필요` 표시를 유지해야 한다.

## Dev Notes

### 현재 구현 상태

- 기존 매출/결제 입력은 총매출, 현금, 카드, 기타 결제수단, 매출차액 관련 값을 다룬다. Story 7.1은 이 값을 어떤 기준으로 이상 신호에 사용할지 닫는 정책 story다. [Source: `_bmad-output/planning-artifacts/epics.md#FR6`]
- `AnomalyThresholdSetting`은 이미 전역 `GLOBAL` 기준값을 저장한다. 저장 필드는 `salesDropRateBps`, `grossMarginDropBps`, `salesDifferenceAmount`, `lossAmount`, `inventoryDifferenceQuantity`, `isActive`, `updatedById`, timestamps다. [Source: `prisma/schema.prisma#model AnomalyThresholdSetting`] [Source: `_bmad-output/implementation-artifacts/5-5-이상-신호-기준값-설정-구조.md`]
- 기준값 화면과 Server Action은 이미 `requireSettingsAccess()`, Zod validation, Prisma transaction, advisory lock, `writeAuditLog`, `threshold.updated`, `AuditLog.reason`, route revalidation을 사용한다. Story 7.1은 이 구현을 바꾸지 않고 정책을 승인 가능한 문서로 만든다. [Source: `src/features/dashboard/threshold-actions.ts`] [Source: `src/features/dashboard/threshold-schemas.ts`]
- `getAnomalyThresholdSettingsForSignals()`는 `requireReportAccess()` 후 `normalizeAnomalyThresholdSignalSettings()`를 거쳐 활성 전역 기준값만 신호 계산 소비자에게 넘긴다. 비활성 기준값은 `null`로 취급된다. [Source: `src/features/dashboard/threshold-queries.ts`] [Source: `src/server/calculations/anomaly.ts`]
- `src/server/calculations/policy-gates.ts`에는 `salesDifferenceThresholdAnomaly`가 OQ-1 미확정 gate로 등록되어 있다. 정책 산출물이 승인되기 전에는 이 gate를 제거하거나 우회하면 안 된다. [Source: `src/server/calculations/policy-gates.ts`]
- `evaluateRevenueAnomalySignals()`는 threshold가 없으면 `기준값 설정 전` info를 반환하고, threshold가 있어도 매출/이익률 비교 정책이나 매출차액 계산 기준이 부족하면 info 상태를 반환한다. 후속 구현 story가 승인될 때까지 UI 전용 계산식을 만들지 않는다. [Source: `src/server/calculations/anomaly.ts`] [Source: `tests/unit/anomaly-sales-signals.test.mjs`]
- 현재 UX는 매출차액을 OQ-1 전까지 임계값 "초과"가 아니라 부호 있는 차이값과 `기준 확인 필요`로 표시하라고 정한다. 이상 신호는 아이콘+레이블을 사용하고 색상만으로 의미를 전달하지 않는다. ### Architecture Guardrails

- ERP Fish는 Next.js App Router, Server Components/Server Actions, Prisma, PostgreSQL, Auth.js/NextAuth, Tailwind/shadcn UI를 사용한다. 이 story에서는 새 library, public API, tRPC, client-only persistence, DB migration을 추가하지 않는다. [Source: `_bmad-output/planning-artifacts/architecture.md#API & Communication Patterns`] [Source: `package.json`]
- 정책 미정 항목은 계산 모델에 직접 영향을 주므로, 결정 전에는 `확인 필요`, `계산 불가`, `데이터 부족` 상태로 표현해야 한다. 0이나 임시값으로 대체하지 않는다. [Source: `_bmad-output/planning-artifacts/architecture.md#Technical Constraints & Dependencies`] [Source: `_bmad-output/planning-artifacts/architecture.md#Calculation Strategy`]
