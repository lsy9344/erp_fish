## Epic 요약

Epic 1은 인증, 사용자/지점/권한 모델, action-level authorization, 지점장 민감 필드 차단, 감사 로그 기반을 구축했다. 구현 결과는 PRD의 FR-1, FR-2, FR-3, FR-23과 아키텍처의 Authentication & Security, Authorization Pattern, Audit Pattern, Sensitive Field Gate 요구에 대체로 맞는다.

완료된 핵심 기반:

- Create T3 App 기반 Next.js, Prisma, NextAuth/Auth.js, Tailwind, shadcn/ui 앱 골격
- Credentials 기반 내부 계정 로그인과 보호된 `/app` 서버 가드
- `UserRole`, `Store`, `UserStoreAssignment`, `PermissionProfile`, `PermissionAction`, `UserPermissionProfile`
- DB 재조회 기반 `src/server/authz.ts` authorization helper
- action별 semantic helper: report, settings, user permission, ledger edit, close, correction, export
- 본사 `ALL_STORES`/`ASSIGNED_STORES` 지점 범위
- 지점장 응답의 민감 회계 필드 서버 차단
- shared `writeAuditLog()`와 사용자/지점 변경 감사 payload
- 변경 이력 조회 화면과 필터, 상세 dialog

## 잘된 점

1. 권한 경계를 서버 중심으로 정착시켰다.
   - Story 1.1에서 `/app` segment guard를 보강했고, Story 1.2~1.4에서 role-only check를 action/profile/store scope helper로 확장했다.
   - 권한 변경 후 기존 세션이 오래된 claim을 믿지 않도록 DB의 현재 사용자, 활성 profile, action, store assignment를 다시 읽는 패턴이 반복 적용됐다.

2. 보안 요구를 UI 숨김이 아니라 응답 shape까지 끌고 갔다.
   - Story 1.4에서 지점장 review, inventory, loss 응답에서 원가, 이익, 재고금액, 조정 금액, 손실 금액 계열이 서버 응답에서 제거됐다.
   - review 중 발견된 loss `amount` 누출을 자동 수정해 민감 필드 기준이 실제 타입과 action 응답까지 내려갔다.

3. 감사 로그가 별도 형식으로 흩어지지 않았다.
   - `src/server/audit.ts`의 shared helper와 dot notation action을 유지했다.
   - Story 1.5에서 actor permission context, target before/after snapshot, credential exclusion을 보강했다.

4. 리뷰 피드백이 실질적인 품질 개선으로 이어졌다.
   - 각 스토리에서 senior review가 HIGH/MEDIUM 이슈를 찾아 자동 수정했다.
   - 반복된 수정 유형은 권한 scope, 민감 응답, audit payload completeness, story file traceability였다.

## 어려웠던 점

1. 스토리 번호와 과거 산출물의 불일치가 혼선을 만들었다.
   - `implementation-artifacts`에는 과거 계획의 1-2, 1-3, 1-4, 1-5 성격 파일이 남아 있었고, 현재 sprint-status의 authoritative story와 제목/범위가 달랐다.
   - 구현 기록은 sprint-status key와 새 story file을 기준으로 정리됐지만, 후속 자동화가 오래된 파일을 잘못 선택할 위험이 남아 있다.

2. E2E 실행 환경이 반복 리스크였다.
   - Story 1.1~1.4는 Playwright가 로컬 포트 bind, PATH, test DB 부재, 포트 점유 문제로 반복 차단됐다.
   - Story 1.5에서는 WSL 앱 서버와 Windows Docker Postgres 조합으로 focused E2E가 통과했지만, 일반 개발 환경에서 재현 가능한 단일 명령 흐름은 아직 README에 충분히 정리되어 있지 않았다.

3. 권한 설계가 단순 role에서 profile/action으로 진화하면서 기존 코드 표면을 넓게 재점검해야 했다.
   - dashboard, report, master-data, ledger detail, correction, close, inventory, losses 등 여러 feature query/action에 같은 helper contract를 적용해야 했다.
   - 이 작업은 구조적으로 맞는 방향이지만, 새 feature가 broad role check로 회귀할 수 있어 정적 회귀 테스트가 중요해졌다.

## 핵심 학습

1. ERP Fish의 보안 source of truth는 세션 token이나 클라이언트 메뉴가 아니라 DB 기반 shared server helper여야 한다.
2. 민감 필드 차단은 component 조건부 렌더링보다 response mapper와 타입에서 먼저 보장해야 한다.
3. 감사 로그는 "무엇이 바뀌었는가"뿐 아니라 "어떤 권한 컨텍스트로 바뀌었는가"까지 포함해야 운영 신뢰를 만든다.
4. Story file과 sprint-status가 맞지 않으면 자동화가 잘못된 근거를 선택한다. Epic 2 전에는 authoritative story selection 규칙을 계속 명확히 해야 한다.
5. E2E는 테스트 코드보다 실행 조건이 더 자주 실패했다. 포트, DB URL, seed 계정, Playwright webServer 명령을 문서화하지 않으면 같은 실패가 반복된다.

## Epic 2 준비 평가

Epic 2는 지점+일자 장부 생성, 7단계 입력 흐름, 매출/결제, 비용, 매입, 재고, 손실, 제출, 저장 충돌을 구현한다. Epic 1의 기반은 Epic 2에 필요한 주요 선행 조건을 제공한다.

준비된 의존성:

- 지점장과 본사 사용자 로그인/진입 경로
- 지점 접근 제한과 본사 store scope
- `LEDGER_CREATE`, `LEDGER_EDIT` action helper
- 지점장 민감 응답 shaping 원칙
- audit helper와 actor context
- Prisma 기반 ledger 관련 모델의 상당 부분
- 장부 입력 관련 E2E 파일과 unit test 기반

주의할 의존성:

- README의 로컬 seed 안내가 실제 `.env.example`/seed 정책과 달라 신규 개발자가 로그인 재현에 실패할 수 있다.
- sprint-status의 Epic 2 story key는 backlog 구조이고, `implementation-artifacts`에는 과거 Epic 2 구현 파일들이 존재한다. Epic 2 실행 전 현재 backlog와 기존 구현 산출물의 대응 관계를 다시 확인해야 한다.
- Playwright focused suite는 환경 조합에 따라 통과했지만, 기본 `pnpm test:e2e`가 항상 같은 조건으로 성공한다고 볼 수 없다.

   - 권한, 감사, 민감 필드, server boundary 요구는 현재 구현 학습과 대체로 일치한다.

## Readiness Assessment

- Testing & Quality: unit, typecheck, lint, build 검증은 story별로 반복 수행됐다. E2E는 Story 1.5 focused suite에서 통과 기록이 있으나, 기본 로컬 실행 조건은 문서 보강이 필요하다.
- Deployment: production deploy 자체는 Epic 1 범위가 아니다. Prisma migration과 production env review는 별도 배포 준비에서 다룬다.
- Stakeholder Acceptance: 이 회고는 자동 회고이므로 직접 stakeholder acceptance는 확인하지 않았다. story status와 review 기록 기준으로 완료 판단했다.
- Technical Health: shared server boundary는 강화됐다. 가장 큰 잔여 리스크는 자동화가 stale story artifact를 잘못 읽는 문제와 E2E 환경 재현성이다.
- Unresolved Blockers: Epic 2 구현을 막는 코드 blocker는 확인되지 않았다. 문서/운영 blocker는 README seed/E2E 안내 불일치다.

## Significant Discovery Check

Epic 1 구현 과정에서 다음 Epic의 방향을 바꿀 정도의 제품/아키텍처 발견은 확인되지 않았다. 오히려 PRD와 architecture가 요구한 방향, 즉 server-side authorization, action permission, sensitive response shaping, append-only audit를 구현으로 확인했다.

다만 Epic 2 시작 전 반영해야 할 운영 발견은 있다.

- 장부 입력 story는 권한, 민감 응답, 감사 로그, 저장 충돌을 처음부터 acceptance criteria에 포함해야 한다.
- 과거 산출물이 섞여 있으므로, 구현 자동화는 현재 sprint-status와 story title을 기준으로 파일을 선택해야 한다.
- E2E 실행 환경은 README 수준에서 더 명확히 해야 한다.
