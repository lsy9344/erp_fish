# Story 2.2: 장부 입력 7단계 폼과 작성자 표시명 유지

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 지점장,
I want 하루 장부를 7단계 흐름으로 입력하고 작성자 표시명이 유지되기를 원한다,
so that 모바일에서도 빠뜨리지 않고 장부를 끝까지 작성할 수 있다.

## Acceptance Criteria

1. Given 지점장이 작성 가능한 장부를 열었을 때, when 장부 입력 화면이 표시된다, then 입력 흐름은 매출/결제, 비용, 매입, 재고, 손실/폐기/떨이, 근무인원/특이사항, 검토/제출의 7단계로 구성되어야 한다, and 현재 단계와 저장 상태를 사용자가 알 수 있어야 한다.
2. Given 사용자가 1단계에서 작성자 표시명을 입력했을 때, when 2~7단계로 이동하거나 부분 저장을 수행한다, then 작성자 표시명은 같은 장부 입력 흐름 안에서 유지되어야 한다, and 최종 저장 데이터에서 누락되면 안 된다.
3. Given 작성자 표시명과 인증 계정이 다를 때, when 장부 상세 또는 감사 관련 정보가 표시된다, then 작성자 표시명과 인증 계정 actor는 구분되어야 한다, and 작성자 표시명이 감사 로그 실행자를 대체하면 안 된다.
4. Given 모바일 390px 폭 화면에서 입력할 때, when 사용자가 각 단계를 이동한다, then 핵심 입력 필드와 단계 이동 버튼은 겹치지 않아야 한다, and 터치 대상은 최소 44x44px을 만족해야 한다.
5. Given 사용자가 단계 이동 중 미저장 변경을 가진 상태일 때, when 다음 단계로 이동하려 한다, then 저장, 취소, 계속 편집 중 하나를 선택하게 해야 한다, and 사용자가 선택하기 전에는 입력값을 조용히 버리면 안 된다.
6. Given 네트워크 오류나 저장 실패가 발생했을 때, when 사용자가 현재 단계에 머문다, then 마지막 저장 시각, 저장되지 않았을 수 있는 단계/필드, 재시도 액션을 보여줘야 한다, and 사용자가 입력값 손실 가능성을 이해할 수 있어야 한다.

- Prisma schema and migrations are the source of truth for persisted data. 작성자 표시명 persistence는 Prisma migration으로 반영한다. [Source: `_bmad-output/planning-artifacts/architecture.md#Data Architecture`]
- 대부분의 제품 동작은 Server Components와 Server Actions를 사용한다. Story 2.2를 위해 public API route를 만들지 않는다. [Source: `_bmad-output/planning-artifacts/architecture.md#API Boundaries`]
- 모든 ledger mutation은 Zod validation, shared authorization helper, transaction, audit, revalidation을 따라야 한다. [Source: `_bmad-output/planning-artifacts/architecture.md#Process Patterns`]
- Store managers must not receive sensitive accounting fields. 작성자 표시명 추가는 response shaping을 우회하지 않는다. [Source: `_bmad-output/planning-artifacts/architecture.md#Sensitive Field Gate`]
- Daily ledger records need a server version or edit token. Story 2.2의 모든 저장은 Story 2.1의 `version` guard를 계속 사용한다. [Source: `_bmad-output/planning-artifacts/architecture.md#Concurrent Editing And Save Conflicts`]
- CAP-16 guardrail은 display author name을 whole ledger entry flow에 저장하되 authenticated account와 audit actor와 분리하라고 요구한다. [Source: `_bmad-output/planning-artifacts/architecture.md#Additional Data And UI Guardrails`]

### Data and Security Guardrails

- 작성자 표시명은 화면 표시용 업무 값이다. 감사 로그 실행자, 권한 판단, `createdById`, `updatedById`, `submittedById`, `closedById`를 대체하지 않는다.
- 작성자 표시명을 session/localStorage-only 값으로만 두면 AC 2의 "최종 저장 데이터에서 누락되면 안 된다"를 만족하지 못한다. DB에 저장하고 query response로 재hydration해야 한다.
- 작성자 표시명은 외부 credential, session token, password, permission profile처럼 보안 경계에 쓰면 안 된다.
- `storeId + closingDate + version` 확인을 유지한다. `ledgerId`만 믿고 author display name을 저장하면 권한 밖 ledger id 주입을 놓칠 수 있다.
- `IN_PROGRESS`와 `IN_REVIEW`만 원본 장부 수정 가능 상태다. `HEADQUARTERS_CLOSED`와 `HOLIDAY`에서는 작성자 표시명도 원본 장부 필드로 보고 수정 차단한다.
- 미저장 변경 guard는 UX 보호 장치다. 서버 conflict guard를 대체하지 않는다.
- 작성자 표시명은 지점장에게 보여도 되는 비민감 업무 표시값이지만, 장부 상세/감사 정보에서 인증 actor와 나란히 표기할 때 두 값의 의미를 명확히 구분한다.
