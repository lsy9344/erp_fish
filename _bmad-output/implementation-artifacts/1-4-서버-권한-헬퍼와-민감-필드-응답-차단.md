# Story 1.4: 서버 권한 헬퍼와 민감 필드 응답 차단

Status: done

## Story

As a 개발자,
I want 권한 확인과 민감 필드 차단을 공통 서버 헬퍼로 사용할 수 있기를 원한다,
so that 이후 장부, 리포트, export 기능이 각자 다른 방식으로 보안을 구현하지 않는다.

## Acceptance Criteria

1. Given 서버 코드가 사용자 권한을 확인해야 할 때, when 기능 코드가 공통 authorization helper를 호출한다, then helper는 로그인 여부, 권한 프로파일, 지점 접근 범위, action 권한을 함께 확인해야 한다, and 권한 없는 요청은 업무 로직 실행 전에 중단되어야 한다.
2. Given 본사 action 권한이 필요한 요청이 있을 때, when close, correction, upload, settings, report/export 같은 action을 검사한다, then 각 action은 명시된 permission key 또는 동등한 구조로 확인되어야 한다, and 단순히 `본사 사용자`라는 큰 역할만으로 모든 action을 허용하면 안 된다.
3. Given 지점장 경로에서 장부나 리포트 데이터를 조회할 때, when 서버가 응답 데이터를 만든다, then 매출원가, 매출이익, 영업이익, 이익률, 인당생산성, 재고금액, lot 근거, 본사 고정비, 타 지점 비교 같은 민감 필드는 응답에서 제거되어야 한다, and 클라이언트에서 숨기는 방식에 의존하면 안 된다.
4. Given 본사 조회 전용 권한이 있는 사용자가 리포트나 export를 요청할 때, when 서버가 응답 필드와 대상 지점을 결정한다, then 허용된 지점/기간/컬럼만 포함해야 한다, and 권한 밖 지점이나 권한 없는 민감 컬럼은 반환하지 않아야 한다.
5. Given 새로운 feature action이 추가될 때, when 개발자가 권한 검사를 구현한다, then 기존 공통 helper 패턴을 재사용해야 한다, and one-off role check를 만들면 코드 리뷰에서 차단할 수 있어야 한다.

### Architecture Guardrails

- 선택 스택은 Create T3 App, Next.js App Router, TypeScript, Prisma, PostgreSQL, NextAuth.js/Auth.js, Tailwind, shadcn/ui, Vercel이다. 새 auth/API framework나 global client-state library를 추가하지 않는다. [Source: `_bmad-output/planning-artifacts/architecture.md#Technology Stack Selection`]
- 모든 ledger, report, master-data, close, correction, settings action은 서버에서 authorization을 강제한다. UI 숨김은 보조 수단일 뿐 보안 경계가 아니다. [Source: `_bmad-output/planning-artifacts/architecture.md#Authorization Pattern`]
- `src/server`는 auth session loading, authorization, audit logging, transactions, calculations 같은 cross-cutting server behavior의 소유 경계다. feature code는 shared helper를 호출하고 one-off 권한 검사를 만들지 않는다. [Source: `_bmad-output/planning-artifacts/architecture.md#Cross-Cutting Concerns`]
- reads는 Server Components와 server-side query helper, mutations는 Server Actions를 사용한다. Route Handler는 export 같은 실제 HTTP endpoint가 필요할 때만 둔다. [Source: `_bmad-output/planning-artifacts/architecture.md#API & Communication Patterns`]
- MVP는 correctness/freshness 우선이다. 권한별 민감 응답에는 공격적 캐싱을 적용하지 말고, 캐시가 필요하면 권한/profile/store scope를 cache key에 포함하거나 서버 재검사를 수행한다. [Source: `_bmad-output/planning-artifacts/architecture.md#Caching Strategy`]

### Project Structure Notes

- Shared authorization은 `src/server/authz.ts` 중심으로 둔다.
- Shared sensitive-field policy는 feature-local mapper가 중복되기 시작하면 `src/server`의 server-only helper로 올린다.
- Feature reads는 `src/features/{domain}/queries.ts`, mutations는 `src/features/{domain}/actions.ts`에서 shared helper를 호출한다.
- UI navigation filtering은 사용자 편의 기능이고, 실제 보안은 page/query/action/route handler/export의 서버 gate와 response shaping에서 완성한다.
