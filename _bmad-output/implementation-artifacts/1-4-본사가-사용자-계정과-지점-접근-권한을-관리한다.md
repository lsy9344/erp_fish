# Story 1.4: 본사가 사용자 계정과 지점 접근 권한을 관리한다

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 본사 사용자,
I want 본사 사용자와 지점장 계정, 지점 접근 권한을 관리하고 싶다,
so that 각 사용자가 맡은 범위의 업무 데이터만 사용할 수 있다.

## Acceptance Criteria

1. Given 본사 사용자가 사용자/권한 관리 화면에 접근할 때, when 화면이 열리면, then 사용자 목록에는 이름, 역할, 연결 지점, 활성/비활성 상태, 마지막 수정 시각이 표시되어야 한다, and 역할 또는 활성 상태로 필터링할 수 있어야 한다.
2. Given 본사 사용자가 새 사용자 계정을 만들 때, when 이름, 로그인 식별자, 역할을 입력하고 저장하면, then 해당 사용자는 지정된 역할로 생성되어야 한다, and 공개 self-signup 없이 본사 관리 흐름에서만 계정이 생성되어야 한다.
3. Given 본사 사용자가 지점장 계정을 만들거나 수정할 때, when 하나 이상의 활성 지점을 배정하고 저장하면, then 해당 지점장은 배정된 지점 데이터에만 접근할 수 있어야 한다, and 배정되지 않은 지점에는 서버에서 접근이 거부되어야 한다.
4. Given 본사 사용자가 사용자의 역할이나 지점 배정을 변경할 때, when 변경 내용을 저장하면, then 이후 서버 요청부터 변경된 접근 범위가 반영되어야 한다, and 화면 메뉴 표시도 새 역할과 접근 범위에 맞게 갱신되어야 한다.
5. Given 본사 사용자가 사용자를 비활성 처리할 때, when 비활성 상태로 저장하면, then 해당 사용자는 새 로그인이나 보호된 업무 요청을 수행할 수 없어야 한다, and 과거 감사 기록의 actor 참조는 보존되어야 한다.
6. Given 사용자/권한 정보가 변경될 때, when 생성, 역할 변경, 지점 배정 변경, 활성 상태 변경이 완료되면, then actor, timestamp, target, before, after를 포함하는 감사 기록이 남아야 한다.
7. Given 사용자/권한 관리 폼에 잘못된 값이 입력될 때, when 저장을 시도하면, then 필드 하단에 한국어 검증 메시지가 표시되어야 한다, and 포커스는 첫 번째 오류 필드로 이동해야 한다.

- 2026-05-29 확인: Next.js App Router mutation 문서는 Server Actions를 form action 또는 client event에서 호출할 수 있고, mutation 후 `revalidatePath` 또는 `revalidateTag`로 stale data를 갱신하라고 안내한다.
