# Story 2.7: 입력자가 검토 화면에서 계산값과 이상 후보를 확인한다

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 지점장 또는 본사 사용자,
I want 검토 화면에서 장부 계산값과 누락/이상 후보를 확인하고 싶다,
so that 제출 전에 장부의 주요 숫자와 문제 가능성을 한 번에 점검할 수 있다.

## Acceptance Criteria

1. **Given** 입력중 장부에 매출, 비용, 매입, 재고, 손실, 근무 정보가 일부 또는 전체 저장되어 있을 때  
   **When** 사용자가 검토/제출 단계에 접근하면  
   **Then** 총매출, 매출원가, 매출이익, 이익률, 영업이익, 인당생산성, 재고금액, 매출차액이 표시되어야 한다  
   **And** 계산은 서버 측 공유 계산 함수를 사용해야 한다.

2. **Given** 상품별 판매금액 합계 또는 판매/차감 값이 계산에 필요한 경우  
   **When** 해당 값이 아직 입력되지 않았거나 정책이 미정이면  
   **Then** 숨은 임의 계산 대신 "계산 기준 확인 필요" 또는 "계산 불가" 상태가 표시되어야 한다.

3. **Given** 총매출이 0이거나 비어 있을 때  
   **When** 이익률을 계산하면  
   **Then** 나누기 오류나 잘못된 무한대 값 대신 "계산 불가" 상태가 표시되어야 한다.

4. **Given** 근무인원이 0이거나 비어 있을 때  
   **When** 인당생산성을 계산하면  
   **Then** 나누기 오류 대신 "계산 불가" 상태가 표시되어야 한다.

5. **Given** 필수 입력값이 누락되어 있을 때  
   **When** 검토 요약이 표시되면  
   **Then** 누락 항목 목록이 표시되어야 한다  
   **And** 각 항목은 사용자가 해당 입력 단계로 돌아갈 수 있게 연결되어야 한다.

6. **Given** 결제수단 합계와 총매출이 다르거나 상품별 판매금액 합계와 총매출이 다를 때  
   **When** 검토 요약이 표시되면  
   **Then** 합계 불일치 경고가 표시되어야 한다  
   **And** 경고는 제출을 자동 차단하지 않아야 한다.

7. **Given** 매출차액, 재고 차이, 손실액 또는 손실 수량이 기준값을 넘을 때  
   **When** 검토 요약이 표시되면  
   **Then** 이상 후보가 아이콘과 레이블로 표시되어야 한다  
   **And** 색상만으로 의미를 전달하면 안 된다.

8. **Given** 사용자가 모바일 화면에서 검토 요약을 볼 때  
   **When** 계산값, 누락 항목, 이상 후보가 표시되면  
   **Then** 390px 폭에서도 내용이 겹치지 않고 읽을 수 있어야 한다.

### Architecture Guardrails

- 스택은 Next.js App Router + Server Components/Server Actions + Prisma + PostgreSQL + Zod + shadcn/ui + Tailwind CSS를 유지한다. [Source: `_bmad-output/planning-artifacts/architecture.md` > `Core Architectural Decisions`]
- 이 스토리는 read/query 중심이다. 제출 mutation과 상태 전이는 Story 2.8에서 처리한다.
- Route Handler를 만들지 않는다. 내부 UI 조회는 Server Component + feature query로 충분하다.
- 모든 민감한 장부 조회는 서버에서 권한을 확인한다. page layer에서 `requireStoreAccess(storeId)` 또는 `getStoreManagerWorkspace()` 흐름을 유지한다.
- 계산 규칙은 `src/server/calculations`에 둔다. client component는 표시와 interaction만 담당한다.
- 돈은 integer KRW로 계산하고 UI에서만 포맷한다.
- 원본 장부 잠금 정책은 바꾸지 않는다. `IN_REVIEW`는 원본 잠금이 아니며, `HEADQUARTERS_CLOSED`만 원본 수정이 잠긴다. [Source: `_bmad-output/planning-artifacts/prds/prd-erp_fish-2026-05-28-2/review-resolution.md` > `지점장 제출 후 잠금 vs 본사 마감 잠금`]
- 새로운 이상 기준값 관리 UI나 DB 모델은 이 스토리 범위가 아니다. Story 3.2에서 다룬다.

### UX Guardrails

- 화면은 기존 장부 입력 화면과 같은 운영 도구 톤을 유지한다. 과한 hero, 장식 카드, 설명형 랜딩을 만들지 않는다.
- 상태 배지는 텍스트 기반이어야 하며, 이상 후보는 아이콘 + 레이블 + 실제 수치로 표시한다.
- 금액은 원 단위와 천 단위 콤마로, 이익률은 소수 첫째 자리까지 표시한다.
- `ReviewSummary`는 계산값, 누락 항목, 이상 후보를 한 화면에서 빠르게 훑을 수 있어야 한다.
- 390px 모바일에서 텍스트와 숫자가 버튼 또는 card 밖으로 넘치면 실패다.
- 누락 항목 링크는 사용자가 바로 해당 입력 단계로 돌아갈 수 있게 실제 route/query string을 사용한다.

### Latest Technical Information

- Next.js 공식 문서에 따르면 App Router는 Server Components, Suspense, Server Functions를 사용하는 file-system router다. 이 스토리는 App Router page + Server Component query 흐름을 유지한다. [External: `https://nextjs.org/docs/app`]
- Next.js `revalidatePath`는 Server Functions와 Route Handlers에서 호출할 수 있다. 2.7 자체는 mutation이 아니므로 새 revalidation이 필요하지 않지만, 2.8 제출 action은 기존 저장 action처럼 영향을 받는 path를 갱신해야 한다. [External: `https://nextjs.org/docs/15/app/api-reference/functions/revalidatePath`]
- Prisma interactive transaction은 여러 Prisma query와 user code를 하나의 transaction 안에서 실행할 수 있지만 오래 열어두면 성능과 deadlock 위험이 있다. Review query가 transaction을 쓴다면 DB 조회와 계산만 짧게 수행한다. [External: `https://www.prisma.io/docs/orm/v6/prisma-client/queries/transactions`]
- Zod `safeParse`는 throw 없이 성공/실패 result를 반환한다. 이 스토리에 입력 mutation은 없지만, 계산 threshold나 query input schema를 만들 경우 기존 `safeParse` + `ActionResult`/field error 패턴을 따른다. [External: `https://zod.dev/?id=parse`]
