# Story 7.5: 희망 판매가 기준 손실액 정책 확정

Status: done

## Story

As a PM과 본사 운영자,  
I want 희망 판매가 기준 손실액 계산 정책을 확정하고 싶다,  
so that 손실/폐기/떨이 입력이 잘못된 기준으로 손익에 반영되지 않는다.

## Acceptance Criteria

1. **Given** OQ-9가 열려 있을 때, **When** 이 discovery story를 수행한다, **Then** 희망 판매가가 미입력된 경우 손실액을 어떻게 처리할지 결정해야 한다, **And** `계산 불가`, `데이터 부족`, 수동 입력 중 어떤 상태를 사용할지 명시해야 한다.
2. **Given** 희망 판매가가 영업 중 변경될 수 있을 때, **When** 계산 정책을 확정한다, **Then** 손실 발생 시점, 장부 작성 시점, 마감 시점 중 어떤 가격을 기준으로 할지 정해야 한다, **And** 가격 version 또는 lock 정책을 포함해야 한다.
3. **Given** 손실 유형별 계산 방식이 다를 수 있을 때, **When** 정책 산출물을 작성한다, **Then** 손실, 폐기, 떨이 유형별로 희망 판매가 적용 여부와 예외를 정의해야 한다, **And** 예시 계산을 포함해야 한다.
4. **Given** 지점장 노출 범위를 정할 때, **When** 희망 판매가 또는 손실액이 민감 지표를 역추정하게 만들 수 있다, **Then** 권한별 표시 정책을 OQ-10A와 연결해야 한다, **And** 지점장에게 노출하지 않을 필드를 명시해야 한다.
5. **Given** 정책 산출물이 완료되었을 때, **When** PM과 본사 운영자가 승인한다, **Then** 산출물에는 기준 가격, version/lock 정책, 유형별 예외, 예시 계산, 승인자가 포함되어야 한다, **And** MVP-S08 또는 후속 구현 스토리로 승격할 수 있는지 명시해야 한다.

### Architecture Guardrails

- ERP Fish는 Next.js App Router, Server Components/Server Actions, Prisma, PostgreSQL, Auth.js/NextAuth, Tailwind/shadcn UI를 사용한다. 이 story에서는 새 library, public API, tRPC, client-only persistence, DB migration을 추가하지 않는다. [Source: `_bmad-output/planning-artifacts/architecture.md#Starter Template Evaluation`] [Source: `package.json`]
- 계산은 dashboard, detail, report, export가 공유하는 server calculation module에 있어야 한다. 희망 판매가 손실 계산도 후속 구현 시 shared server calculation으로 들어가야 하며 UI-only 계산이나 report 전용 복제 계산을 만들면 안 된다. [Source: `_bmad-output/planning-artifacts/architecture.md#Calculation Strategy`]
- 정책이나 source data가 없으면 `확인 필요`, `계산 불가`, `데이터 부족` 상태를 반환해야 한다. 희망 판매가 미입력이나 version 불명확을 0원 또는 stale 단가로 대체하면 안 된다. [Source: `_bmad-output/planning-artifacts/architecture.md#Calculation Strategy`]
- 본사 마감 후 원본 장부는 보존되어야 하고 변경은 append-only 정정 기록으로만 반영된다. 희망 판매가 version/lock 정책도 과거 `LedgerLossItem` row를 소급 overwrite하지 않아야 한다. [Source: `_bmad-output/planning-artifacts/architecture.md#Audit and Correction Strategy`]
- 서버 권한은 모든 ledger, report, correction, master-data, settings action에서 강제된다. 지점장 응답은 원가, 이익, 마진율, 재고금액 같은 민감 회계 필드를 서버 응답에서 제거해야 한다. [Source: `_bmad-output/planning-artifacts/architecture.md#Authorization Pattern`]
- 후속 구현이 가격 version, override, lock, audit를 추가한다면 Zod server boundary validation, database constraints, transaction 내 audit write, route revalidation 패턴을 따라야 한다. [Source: `_bmad-output/planning-artifacts/architecture.md#Validation Strategy`] [Source: `_bmad-output/planning-artifacts/architecture.md#Event and Audit Patterns`]
