# Story 3.4: 본사가 재고/손실 이상 신호를 본다

Status: review

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 본사 사용자,
I want 관제판에서 재고 차이와 손실 이상 신호를 확인하고 싶다,
so that 재고 흐름이나 손실이 비정상적인 지점을 먼저 점검할 수 있다.

## Acceptance Criteria

1. 본사 사용자가 관제판을 열 때 장부의 재고 조정, 실제 재고, 손실 데이터와 기준값이 존재하면 재고 차이와 손실 이상 여부가 서버 측 공유 이상 신호 계산으로 평가되어야 한다.
2. UI 컴포넌트 안에서 재고/손실 threshold 공식을 따로 만들면 안 된다.
3. 재고 차이 수량이 본사 기준값을 넘으면 관제판 행에 재고 이상 신호 칩이 표시되어야 한다.
4. 재고 이상 신호의 칩 또는 상세에는 기준값과 실제 차이값이 표시되어야 한다.
5. 손실액이 본사 기준값을 넘으면 관제판 행에 손실 이상 신호 칩이 표시되어야 한다.
6. 손실 이상 신호 칩에는 아이콘과 한국어 레이블이 함께 표시되어야 한다.
7. 재고/손실 신호가 심각 등급이면 destructive 스타일의 신호 칩과 행 강조가 적용되어야 하며, 색상만으로 의미를 전달하면 안 된다.
8. 재고 또는 손실 데이터가 아직 미입력일 때는 미입력 상태와 정상 0 값을 구분해야 하며, 미입력을 정상으로 취급해 조용히 숨기면 안 된다.
9. 재고 계산에 필요한 전일재고, 매입, 당일재고, 조정값 중 일부가 없으면 "계산 불가" 또는 "입력 필요" 상태가 표시되어야 하며, 임의의 0으로 대체해 정상 처리하면 안 된다.
10. 한 지점에 재고 이상과 손실 이상이 동시에 발생하면 두 신호 칩이 모두 표시될 수 있어야 한다.
11. 행 선택 시 장부 상세에서 같은 재고/손실 신호 요약과 관련 근거를 확인할 수 있어야 한다.

### Data Model and Calculation Rules

- `AnomalyThresholdSetting.lossAmount`는 손실액 기준값이다. 손실 여부(`hasLoss`)가 true인 것만으로 이상 신호를 만들면 안 된다. [Source: `C:\Code\Project\erp_fish\prisma\schema.prisma#AnomalyThresholdSetting`]
- `AnomalyThresholdSetting.inventoryDifferenceQuantity`는 재고 차이 수량 기준값이다. 이름 그대로 수량 기준이며 `differenceAmount`와 혼동하지 않는다. [Source: `C:\Code\Project\erp_fish\prisma\schema.prisma#AnomalyThresholdSetting`]
- 현재 schema에는 `inventoryDifferenceAmount` threshold가 없다. AC의 금액 표현은 `LedgerInventoryAdjustment.differenceAmount`를 detail 근거로 보여주는 수준으로 처리하고, 금액 기준 trigger를 위해 새 schema 필드를 추가하지 않는다. 금액 기준 trigger가 꼭 필요하면 별도 product decision으로 남긴다.
- 재고 차이의 우선 source는 `LedgerInventoryAdjustment.differenceQuantity`다. 이 값은 조정 전/후 차이를 이미 저장한다. [Source: `C:\Code\Project\erp_fish\prisma\schema.prisma#LedgerInventoryAdjustment`]
- 손실 이상 source는 `LedgerLossItem.amount` 합계 또는 품목별 합계다. `LedgerLossItem.quantity`는 detail과 후보 설명에 사용할 수 있지만, 현재 전역 threshold에는 손실 수량 기준값이 없다. [Source: `C:\Code\Project\erp_fish\prisma\schema.prisma#LedgerLossItem`]
- 정상 0은 signal 없음이다. null, missing, item 없음, 계산 불가는 info 신호 또는 detail 상태로 표현한다.
- ledger 없는 row는 장부 미입력 상태다. 이 상태에서 "정상"으로 보이게 만들지 않고, 기존 `미입력` row 표시와 pending/info signal 흐름을 보존한다.
- 여러 신호가 동시에 있으면 revenue warning과 inventory/loss critical이 같은 `signals` 배열에 공존해야 한다. 행 강조는 existing `critical > warning > info` 우선순위를 따른다.
