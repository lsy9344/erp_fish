# Story 5.2: 본사가 리포트 숫자의 근거와 정정 차이를 추적한다

Status: done

## Story

As a 본사 사용자,  
I want 리포트 숫자의 원본값, 정정 반영값, 계산 상태를 확인하고 싶다,  
so that 회의에서 사용하는 숫자를 신뢰하고 필요하면 근거를 추적할 수 있다.

## Acceptance Criteria

1. **Given** 리포트 숫자에 정정 기록이 반영되어 있을 때  
   **When** 본사 사용자가 해당 숫자의 상세를 열면  
   **Then** 원본값, 정정 반영값, 정정 여부가 구분되어 표시되어야 한다  
   **And** 정정 반영값에는 "정정 반영" 레이블이 표시되어야 한다.

2. **Given** 리포트 숫자가 계산 불가 상태일 때  
   **When** 본사 사용자가 상세를 열면  
   **Then** 계산 불가 사유가 한국어로 표시되어야 한다  
   **And** 미입력, 휴무, 0, 데이터 부족 상태가 구분되어야 한다.

3. **Given** 리포트 숫자가 장부 상세 데이터에서 나온 경우  
   **When** 본사 사용자가 근거 확인 경로를 선택하면  
   **Then** 해당 지점+일자의 장부 상세 또는 정정 타임라인으로 이동할 수 있어야 한다.

4. **Given** 같은 항목에 여러 정정 기록이 있을 때  
   **When** 리포트 상세가 표시되면  
   **Then** 최신 정정값이 현재 정정 반영값으로 표시되어야 한다  
   **And** 이전 정정 기록은 타임라인에서 확인할 수 있어야 한다.

5. **Given** 리포트 계산이 수행될 때  
   **When** 원본값과 정정 반영값을 비교하면  
   **Then** shared server calculation이 두 값을 명시적으로 구분해 제공해야 한다  
   **And** UI에서 별도 보정 계산을 만들면 안 된다.

6. **Given** 좁은 화면에서 숫자 근거 상세를 볼 때  
   **When** 원본값과 정정 반영값이 함께 표시되면  
   **Then** 두 값이 겹치지 않고 텍스트 레이블로 구분되어야 한다  
   **And** 색상이나 취소선만으로 의미를 전달하면 안 된다.

## Dev Notes

### 핵심 목표

- Story 5.2는 5.1에서 만든 일별 아침 회의 리포트에 "왜 이 숫자인지"를 설명하는 숫자별 근거 레이어를 추가한다.
- 리포트의 기본 표시 숫자는 계속 정정 반영값이다. 사용자가 근거를 열었을 때 원본값, 정정 반영값, 정정 여부, 계산 불가 사유, 장부 상세/정정 타임라인 경로를 확인할 수 있어야 한다.
- 이 story는 기간 비교(5.3), 월간 마감/이상(5.4), 월간 손실/재고 흐름(5.5)을 미리 만들지 않는다. 다만 5.3-5.5가 재사용할 수 있는 숫자 근거 타입과 표시 패턴을 만든다.

### 현재 구현 상태

- `src/features/reports`와 `src/app/app/reports/daily`는 Story 5.1에서 추가되었다.
- `src/features/reports/queries.ts`는 active store와 선택 일자 ledger를 읽고, `getLatestCorrectionValuesForLedgers()`, `applyCorrectionValuesToLedgerReviewInput()`, `calculateLedgerReviewSummary()`로 정정 반영값을 만든다.
- `src/features/reports/types.ts`는 현재 `DailyMeetingReportRow = HqDashboardRow`로 관제판 row 타입을 그대로 재사용한다. 5.2의 근거 데이터는 리포트 전용 확장 타입으로 두는 편이 안전하다.
- `src/features/reports/components/daily-meeting-report-table.tsx`는 현재 숫자와 장부 상세 링크만 표시한다. 숫자별 근거 상세 UI는 아직 없다.
- `src/app/app/ledgers/[ledgerId]/page.tsx`는 장부 주요 숫자, 현재 정정 반영값, `CorrectionPanel` 정정 기록 타임라인을 이미 보여준다. 5.2의 상세 이동 목적지로 재사용한다.

### 구현 가드레일

1. 본사 전용 page/query는 반드시 `requireHeadquartersUser()`를 호출한다.
2. 원본 장부 row는 절대 덮어쓰지 않는다. 정정은 `CorrectionRecord` append-only 기록이다.
3. 최신 정정값 선택 규칙은 `getLatestCorrectionValueMap()`/`getLatestCorrectionValuesForLedgers()`의 기존 계약을 따른다.
4. 리포트, 관제판, 장부 상세이 서로 다른 계산식을 갖게 만들면 안 된다.
5. 금액은 integer KRW 상태로 계산하고 UI에서만 원 단위 포맷을 적용한다.
6. 날짜는 기존 KST 운영일 helper를 유지한다. 브라우저 로컬 날짜 계산으로 바꾸지 않는다.
7. 미입력은 ledger 없음, 휴무는 `DailyLedger.status === "HOLIDAY"`, 0은 실제 값 0이다. 세 상태를 같은 "계산 불가"로 뭉개지 않는다.
8. Story 5.1 관련 파일이 현재 작업 트리에 미커밋 변경으로 남아 있을 수 있다. dev agent는 이 변경을 되돌리지 말고 그 위에 맞춰 작업한다.
