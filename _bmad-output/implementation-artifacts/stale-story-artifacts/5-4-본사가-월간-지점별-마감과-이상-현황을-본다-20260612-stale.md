# Story 5.4: 본사가 월간 지점별 마감과 이상 현황을 본다

Status: done

## Story

As a 본사 사용자,
I want 월과 지점을 선택해 마감 현황과 주요 이상 항목을 보고 싶다,
so that 월간 운영 점검에서 누락 장부와 반복 문제를 확인할 수 있다.

## Acceptance Criteria

1. **Given** 본사 사용자가 월간 요약 리포트 화면에 접근할 때
   **When** 월과 지점을 선택하면
   **Then** 선택한 지점의 월간 마감 현황과 주요 이상 항목이 표시되어야 한다
   **And** 월/지점 필터는 URL 상태로 유지되어야 한다.

2. **Given** 월간 마감 현황이 표시될 때
   **When** 해당 월의 일별 장부 상태를 집계하면
   **Then** 본사마감, 검토대기, 입력중, 미입력, 휴무일이 구분되어 표시되어야 한다
   **And** 마감되지 않은 장부가 있으면 명확히 구분되어야 한다.

3. **Given** 선택한 월에 이상 신호가 있는 장부가 있을 때
   **When** 월간 주요 이상 항목이 표시되면
   **Then** 매출/이익률 급락, 매출차액, 재고/손실 이상 항목이 날짜와 지점 맥락으로 표시되어야 한다
   **And** 이상 신호는 아이콘과 텍스트 레이블로 표시되어야 한다.

4. **Given** 월간 이상 항목을 계산할 때
   **When** 장부에 정정 기록이 포함되어 있으면
   **Then** 기본 숫자는 정정 반영값을 사용해야 한다
   **And** 원본값과 정정 반영값 차이가 있는 항목은 Story 5.2의 상세 확인 경로를 제공해야 한다.

5. **Given** 월간 상태 또는 이상 항목이 표시될 때
   **When** 사용자가 특정 날짜를 선택하면
   **Then** 해당 일자의 장부 상세로 이동할 수 있어야 한다.

6. **Given** 선택한 월에 장부 데이터가 없을 때
   **When** 월간 요약 화면이 열리면
   **Then** 간결한 한국어 빈 상태 메시지가 표시되어야 한다.

7. **Given** 월간 리포트 계산이 수행될 때
   **When** 마감 현황과 이상 항목을 계산하면
   **Then** 관제판/장부 상세와 같은 shared server calculation과 이상 신호 평가를 사용해야 한다
   **And** UI 전용 계산식을 만들면 안 된다.

## Dev Notes

### 핵심 목표

- Story 5.4는 월간 요약 중 "마감 현황과 주요 이상 항목"만 구현한다.
- 월간 핵심 성과, 손실/재고 흐름, 최고매출품목은 Story 5.5 범위다. 5.4에서 미리 만들지 않는다.
- 기본 숫자와 이상 신호는 정정 반영값 기준이어야 한다. 원본 DB row를 덮어쓰거나 UI에서 보정 계산을 만들지 않는다.
- 가장 중요한 결과는 본사가 특정 지점의 한 달 중 어떤 날짜가 미입력/입력중/검토대기인지, 어떤 날짜에 반복 이상이 있는지 바로 찾는 것이다.

### 현재 구현 상태

- `src/app/app/reports/daily/page.tsx`와 `src/app/app/reports/comparison/page.tsx`는 존재한다. `monthly` route는 아직 없다.
- `src/features/reports/queries.ts`는 KST 날짜 helper, active store 조회, 기간 범위 ledger 조회, 정정 batch 조회, `metricEvidence`, shared calculation 기반 기간 집계를 이미 갖고 있다.
- `src/features/reports/types.ts`는 `DailyMeetingReportMetricEvidence`와 `StoreComparisonReportData`를 갖고 있다. 월간 리포트는 별도 타입을 추가해 일별/기간 row 타입을 억지로 오염시키지 않는다.
- `src/features/reports/components/daily-meeting-report-table.tsx`와 `store-comparison-report-table.tsx`는 데스크탑 표, 모바일 카드, `details` 기반 근거 보기, `tabular-nums`, `break-words` 패턴을 제공한다.
- `src/features/corrections/actions.ts`는 정정 생성 후 `/app/reports/daily`와 `/app/reports/comparison`을 revalidate한다. 월간 리포트를 만들면 `/app/reports/monthly`도 stale하지 않게 추가해야 한다.
- 현재 작업 트리에는 Story 5.3 관련 미커밋 변경이 남아 있다. dev agent는 이를 되돌리지 말고 현재 파일 상태 위에서 작업한다.

### 구현 가드레일

1. 본사 전용 page/query는 반드시 `requireHeadquartersUser()`를 호출한다. UI 메뉴 숨김은 보안 경계가 아니다.
2. 이 story에서 public API route, tRPC, client-side fetch layer를 새로 만들지 않는다. 읽기는 Server Component + feature query helper 패턴을 따른다.
3. 새 Prisma 모델이나 migration을 만들지 않는다. `Store`, `DailyLedger`, `CorrectionRecord`, ledger detail relations를 조회해 계산한다.
4. 미입력 날짜를 위해 ledger를 생성하지 않는다. ledger 없음은 `미입력`이다.
5. 최신 정정값 선택 규칙은 기존 correction query 계약인 `createdAt desc, id desc`를 따른다.
6. 금액은 integer KRW 상태로 계산하고 UI에서만 원 단위 포맷을 적용한다.
7. 날짜는 KST 운영일 helper를 사용한다. 브라우저 로컬 날짜 계산으로 월 시작/끝을 바꾸지 않는다.
8. 미입력은 ledger 없음, 휴무는 `DailyLedger.status === "HOLIDAY"`, 0은 실제 값 0이다. 세 상태를 같은 "계산 불가"로 뭉개지 않는다.
9. 리포트, 관제판, 장부 상세가 서로 다른 계산식을 갖게 만들면 안 된다.
10. 월간 매출/손실/재고 성과 지표는 Story 5.5에서 다룬다. 5.4는 상태와 이상 항목에 집중한다.

### UX and Accessibility Notes

- 본사 리포트는 업무 표면이다. 설명용 hero나 큰 카드 레이아웃을 만들지 말고 조밀하지만 읽기 쉬운 표와 리스트를 우선한다.
- 월간 화면은 "상태 요약 -> 주요 이상 항목 -> 날짜별 상세" 순서가 좋다. 누락 날짜와 이상 날짜를 먼저 찾게 해야 한다.
- 상태 배지는 텍스트 중심이다. 이상 신호는 아이콘/텍스트 레이블을 함께 둔다. 색상 단독 전달 금지다.
- 모바일에서는 한 달 날짜 목록이 길어질 수 있다. 핵심 상태 카운트를 먼저 보여주고 날짜별 목록은 읽기 쉬운 리스트나 가로 스크롤 표로 둔다.
- `tabular-nums`를 유지하고, 금액/비율 포맷은 기존 일별/기간 리포트와 일관되게 한다.
- 긴 지점명, 계산 불가 사유, 신호 detail, 정정 상태 텍스트가 버튼이나 숫자를 덮지 않게 responsive wrapping을 적용한다.
- Dialog/Sheet 중첩은 금지다. 월간 리포트에서는 날짜 선택 시 장부 상세 페이지로 이동하는 방식이 단순하고 안전하다.
