# Story 2.5: 지점장 또는 본사가 실제 재고 차이를 조정 사유와 함께 기록한다

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 지점장 또는 본사 사용자,
I want 시스템 계산 재고와 실제 재고가 다를 때 조정 사유와 함께 기록하고 싶다,
so that 재고 차이를 숨기지 않고 장부와 리포트에 추적 가능한 값으로 반영할 수 있다.

## Acceptance Criteria

1. **Given** 입력중 또는 검토 대기 장부의 재고 단계에서 시스템 계산 재고와 실제 입력 재고가 다를 때
   **When** 사용자가 재고 조정을 추가하면
   **Then** 품목, 규격, 조정 전 수량/금액, 조정 후 수량/금액, 차이 수량/금액이 기록되어야 한다
   **And** 작성자와 작성 시각이 함께 저장되어야 한다.

2. **Given** 사용자가 재고 조정을 저장하려 할 때
   **When** 차이 사유가 비어 있으면
   **Then** 저장이 차단되어야 한다
   **And** 사유 필드에 한국어 검증 메시지가 표시되어야 한다.

3. **Given** 재고 조정이 저장될 때
   **When** 장부 재고 흐름을 다시 계산하면
   **Then** 조정값이 재고 흐름과 재고/손실 이상 후보에 반영되어야 한다.

4. **Given** 재고 조정이 저장된 장부를 다시 열 때
   **When** 사용자가 해당 품목 행을 보면
   **Then** 조정 전 값과 조정 후 값, 차이 사유를 확인할 수 있어야 한다
   **And** 조정된 행은 색상만이 아니라 텍스트 또는 아이콘 레이블로도 구분되어야 한다.

5. **Given** 본사 마감 전 장부일 때
   **When** 권한 있는 지점장 또는 본사 사용자가 조정을 추가하거나 수정하면
   **Then** 원본 장부의 재고 조정 기록으로 저장되어야 한다.

6. **Given** 본사 마감된 장부일 때
   **When** 사용자가 재고 조정을 시도하면
   **Then** 이 스토리의 원본 재고 조정 기능으로는 저장되지 않아야 한다
   **And** 마감 후 정정 기록 흐름을 사용해야 한다는 안내가 표시되어야 한다.

7. **Given** 재고 조정이 생성되거나 변경될 때
   **When** 저장이 완료되면
   **Then** actor, timestamp, target, before, after, reason을 포함하는 감사 기록이 남아야 한다.

## Dev Notes

### Source Context

- Story 2.5는 Epic 2의 다섯 번째 장부 입력 스토리이며, Story 2.4의 재고 입력 화면 위에 "차이 사유가 있는 실제 재고 조정"을 추가한다.
- 요구사항 원문은 `FR10`으로도 정의되어 있다. 본사 사용자와 지점장은 시스템 계산 재고와 실제 재고가 다를 때 조정 전/후 값, 차이, 사유, 작성자, 작성 시각을 남겨야 한다.
- 이 스토리는 "값을 그냥 수정하고 끝"이 아니라 "차이를 추적 가능한 기록으로 남기는 것"이 핵심이다.
- 검토 대기(`IN_REVIEW`)는 원본 장부가 아직 잠기지 않은 상태다. 본사 마감(`HEADQUARTERS_CLOSED`) 후에는 원본 조정이 아니라 Story 4의 정정 기록 흐름을 사용해야 한다.

### Current Repository State

- `src/features/inventory`는 Story 2.4에서 이미 실제 재고 입력 기능을 갖고 있다.
- `LedgerInventoryItem` 현재 필드:
  - `previousQuantity`, `purchasedQuantity`, `currentQuantity`, `quantity`, `inventoryAmount`, `isModified`
  - 품목 스냅샷: `productName`, `productCategory`, `productSpec`, `unitPrice`
  - 이월 출처: `carryoverSource`, `carryoverLedgerId`
- 현재 `saveLedgerInventoryItems`는 `IN_PROGRESS` 장부만 저장한다. Story 2.5의 AC는 `입력중 또는 검토 대기`를 요구하므로, 새 조정 action은 `IN_PROGRESS`와 `IN_REVIEW`를 명확히 허용해야 한다.
- 현재 `getInventoryStepDataInTx`는 기존 재고 행이 있으면 active product와 purchase-only product를 병합한다. 이 병합 동작은 새 조정 조회를 넣어도 유지해야 한다.
- 현재 `calculateInventoryAmount(quantity, unitPrice)`는 safe integer 범위를 넘으면 `null`을 반환한다. 조정 저장에서는 계산값이 필수이므로 overflow를 저장 차단 오류로 바꾸는 별도 helper가 필요하다.
- 현재 `AuditLog`는 action, target, actor, before, after, createdAt을 저장하지만 별도 reason 필드는 없다. 이 스토리에서는 사유가 감사 요구사항의 핵심이므로 감사 모델과 helper를 함께 확장해야 한다.

### Architecture Guardrails

- 스택은 Next.js App Router + Server Components/Server Actions + Prisma + PostgreSQL + Zod + shadcn/ui + Tailwind CSS를 유지한다.
- Route Handler를 새로 만들지 않는다. 이 기능은 내부 UI mutation이므로 Server Action이 맞다.
- 모든 business-changing action은 서버 권한 검사, Zod 검증, transaction, audit log, revalidation 순서를 따른다.
- 감사 기록은 별도 이벤트 버스가 아니라 `AuditLog` row로 남긴다.
- 원본 장부와 정정 기록은 분리한다. `HEADQUARTERS_CLOSED` 상태에서 이 스토리의 action이 원본 장부 값을 바꾸면 안 된다.
- 돈은 integer KRW로 저장하고 UI에서만 원화 포맷을 적용한다.
- 계산 규칙은 `src/server/calculations`에 둔다. UI와 report 후보가 서로 다른 계산식을 갖지 않게 한다.

### UX Guardrails

- 재고 테이블은 냉동/생물 탭, 품목/규격/전일재고/매입/당일재고/재고금액/수량 구조를 유지한다.
- 수정/조정/이상 상태는 색상만으로 구분하지 않는다. "수정됨", "조정 필요", "조정됨"처럼 텍스트 또는 아이콘+텍스트 레이블을 같이 둔다.
- 숫자 데이터는 `tabular-nums`를 적용한다.
- 모바일 390px에서 주요 입력과 버튼은 최소 44px 높이를 유지한다.
- 재고 행이 30개 이상일 수 있으므로 표는 가로 스크롤 또는 paging/windowing 전략을 계속 유지한다.
- 조정 사유 입력은 한국어 업무 문장에 맞게 짧고 직접적인 label/error를 사용한다.

### Latest Technical Information

- Next.js 공식 문서는 mutation 뒤 `revalidatePath` 또는 `revalidateTag`로 업데이트된 데이터를 다시 검증하는 흐름을 안내한다. 현재 저장소는 path revalidation을 쓰므로 새 action도 같은 방식을 따른다.
  [External: `https://nextjs.org/docs/app/building-your-application/data-fetching/server-actions-and-mutations`]
- Prisma 공식 문서는 interactive `$transaction`이 여러 query와 user code를 하나의 transaction으로 묶을 수 있다고 설명한다. 조정 upsert, 재고 행 갱신, 감사 로그는 같은 transaction 안에 둔다.
  [External: `https://www.prisma.io/docs/orm/prisma-client/queries/transactions`]
- Zod 공식 문서는 validation issue를 field별 오류 구조로 포맷하는 방식을 제공한다. 현재 저장소의 `toFieldErrors` 패턴을 유지한다.
  [External: `https://zod.dev/error-formatting`]
