# Story 5.1: 본사가 일별 아침 회의 리포트를 본다

Status: done

## Story

As a 본사 사용자,  
I want 특정 일자의 전체 지점 회의용 요약을 보고 싶다,  
so that 아침 회의에서 마감 상태와 이상 항목을 기준으로 지점별 이야기를 진행할 수 있다.

## Acceptance Criteria

1. **Given** 본사 사용자가 아침 회의 리포트 화면에 접근할 때  
   **When** 날짜를 선택하면  
   **Then** 해당 일자의 전체 지점 요약이 표시되어야 한다  
   **And** 기본 날짜는 오늘 또는 운영 기준일로 설정되어야 한다.

2. **Given** 일별 회의 리포트가 표시될 때  
   **When** 지점별 행이 로드되면  
   **Then** 지점별 마감 상태, 이상 신호, 매출, 이익률, 매출차액, 손실 현황이 표시되어야 한다.

3. **Given** 장부에 정정 기록이 있을 때  
   **When** 리포트 숫자가 계산되면  
   **Then** 기본 숫자는 정정 반영값을 사용해야 한다  
   **And** 원본값과 정정 반영값 차이가 있으면 상세 확인 경로가 제공되어야 한다.

4. **Given** 장부가 미입력, 입력중, 검토대기, 본사마감, 휴무일 중 하나일 때  
   **When** 리포트 행이 표시되면  
   **Then** 상태가 명확한 텍스트 배지로 표시되어야 한다  
   **And** 미입력과 휴무일은 구분되어야 한다.

5. **Given** 이상 신호가 있는 지점이 있을 때  
   **When** 리포트 행이 표시되면  
   **Then** 이상 신호는 아이콘과 레이블로 표시되어야 한다  
   **And** 색상만으로 의미를 전달하면 안 된다.

6. **Given** 리포트 계산이 수행될 때  
   **When** 매출, 이익률, 매출차액, 손실 현황을 계산하면  
   **Then** 관제판/장부 상세와 같은 shared server calculation을 사용해야 한다  
   **And** UI 전용 계산식을 만들면 안 된다.

7. **Given** 리포트 데이터가 로딩 중일 때  
   **When** 화면이 데이터를 기다리면  
   **Then** 실제 표와 유사한 Skeleton 상태가 표시되어야 한다.

8. **Given** 선택한 날짜에 표시할 지점 데이터가 없을 때  
   **When** 리포트가 열리면  
   **Then** 간결한 한국어 빈 상태 메시지가 표시되어야 한다.

## Dev Notes

### 핵심 목표

- Story 5.1은 Epic 5 리포트 영역의 첫 화면이다. "일별 아침 회의 리포트"를 만들되, 관제판/장부 상세과 다른 계산 기준을 만들면 안 된다.
- 회의용 숫자는 기본적으로 정정 반영값이어야 한다. 원본 DB row를 덮어쓰거나, UI에서 임시로 보정 계산을 만들지 않는다.
- 이 story는 Story 5.2의 깊은 숫자 근거/정정 차이 분석을 모두 구현하는 범위가 아니다. 다만 정정 차이가 있는 행은 기존 장부 상세로 이동할 수 있어야 한다.

### 현재 구현 상태

- `src/features/reports`와 `src/app/app/reports`는 아직 없다. Story 5.1에서 새로 만드는 것이 맞다.
- `src/components/app-sidebar.tsx`의 리포트 항목은 현재 임시로 다른 경로를 가리킨다. `/app/reports/daily`로 바꿔야 한다.
- `src/features/dashboard/queries.ts`는 active store 전체, 날짜별 ledger 조회, 정정 반영 계산, 이상 신호 계산을 이미 한 곳에서 처리한다. 리포트는 이 패턴을 출발점으로 삼아야 한다.
- `src/app/app/dashboard/loading.tsx`는 실제 표와 모바일 카드 형태에 가까운 Skeleton 패턴을 제공한다. 리포트 loading도 이 수준의 구조적 Skeleton을 사용한다.

### 구현 가드레일

1. 본사 전용 query/page는 반드시 `requireHeadquartersUser()`를 호출한다. UI에서 메뉴를 숨기는 것은 보안 경계가 아니다.
2. 읽기는 Server Component + feature query helper 패턴을 따른다. 이 story에서 새 public API route, tRPC, client-side fetch layer를 만들지 않는다.
3. `Store`, `DailyLedger`, `CorrectionRecord`, `LedgerExpense`, `LedgerInventoryItem`, `LedgerInventoryAdjustment`, `LedgerLossItem`, `AnomalyThresholdSetting` 모델을 조회한다. 새 Prisma 모델이나 migration은 필요하지 않다.
4. 최신 정정값 선택 규칙은 기존 correction query 계약을 따른다: `createdAt desc, id desc`.
5. 금액은 integer KRW, 날짜는 KST 운영일 기준, JSON 경계는 ISO 문자열 원칙을 유지한다.
6. 미입력과 휴무일은 다르다. 휴무 ledger는 `HOLIDAY`, 미입력은 ledger 없음이다.
7. 월간/기간 비교 리포트 범위(Story 5.3-5.5)를 미리 만들지 않는다. 5.1은 일별 회의용 요약에 집중한다.

### UX and Accessibility Notes

- 본사 화면은 데스크탑 스캔 속도가 우선이다. 불필요한 hero나 설명용 카드 대신 조밀하지만 읽기 쉬운 업무 표를 사용한다.
- 데스크탑은 표 중심, 모바일은 겹치지 않는 간소 카드 또는 가로 스크롤 표를 허용한다.
- 숫자는 `tabular-nums`를 유지하고, 금액은 원 단위와 천 단위 콤마로 표시한다.
- 이상 신호는 색상만으로 의미를 전달하지 않는다. 아이콘 또는 텍스트 레이블을 반드시 함께 둔다.
- 상태 배지는 명확한 한국어 텍스트를 포함한다.
- 좁은 화면에서 긴 지점명, 이상 신호, 정정 상태 텍스트가 버튼이나 숫자를 덮지 않게 `min-w-0`, `break-words`, wrapping을 사용한다.
