# Story 3.1: 본사가 전체 지점 장부 상태를 관제판에서 본다

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 본사 사용자,
I want 오늘 또는 어제 기준 전체 지점의 장부 상태를 한 화면에서 보고 싶다,
so that 아침 회의 전에 어떤 지점부터 확인할지 빠르게 판단할 수 있다.

## Acceptance Criteria

1. **Given** 본사 사용자가 관제판에 접근할 때
   **When** 화면이 열리면
   **Then** 오늘 날짜 기준 전체 활성 지점의 장부 상태 행이 표시되어야 한다
   **And** 사용자는 오늘/어제 기준을 전환할 수 있어야 한다.

2. **Given** 관제판 행이 표시될 때
   **When** 지점별 데이터가 로드되면
   **Then** 각 행에는 영업 상태, 장부 상태, 매출, 이익률, 매출차액, 손실 여부, 마지막 수정자, 마지막 수정 시각, 본사 마감 여부가 표시되어야 한다.

3. **Given** 지점+일자 장부가 아직 없을 때
   **When** 해당 지점 행이 표시되면
   **Then** 장부 상태는 `미입력`으로 표시되어야 한다
   **And** 휴무일 상태와 시각적으로 구분되어야 한다.

4. **Given** 장부 상태가 입력중, 검토대기, 본사마감, 휴무 중 하나일 때
   **When** 상태 배지가 표시되면
   **Then** 상태 텍스트가 명확히 표시되어야 한다
   **And** 색상만으로 상태 의미를 전달하면 안 된다.

5. **Given** 관제판 데이터가 로딩 중일 때
   **When** 화면이 데이터를 기다리면
   **Then** 실제 테이블 행과 비슷한 Skeleton 상태가 표시되어야 한다.

6. **Given** 활성 지점이 없거나 표시할 데이터가 없을 때
   **When** 관제판이 열리면
   **Then** 빈 표만 보여주는 대신 간결한 한국어 빈 상태 메시지를 표시해야 한다.

7. **Given** 본사 관제판 기본 조회가 수행될 때
   **When** 활성 지점이 10개 내외이면
   **Then** 3초 안에 지점 목록과 기본 상태 정보가 표시되는 것을 목표로 해야 한다
   **And** 상세 감사 이력이나 무거운 drilldown 데이터는 기본 조회에 포함하지 않아야 한다.

8. **Given** 390px 모바일 또는 좁은 화면에서 관제판을 볼 때
   **When** 화면이 렌더링되면
   **Then** 지점명, 상태, 이상 신호 요약, 매출 같은 핵심 정보가 겹치지 않게 표시되어야 한다
   **And** 숨겨진 컬럼은 상세 화면에서 확인할 수 있어야 한다.

### Latest Technical Information

- 현재 `package.json` 기준 로컬 스택은 Next.js `^15.2.3`, React `^19.0.0`, Prisma `^6.6.0`, shadcn `^4.8.2`, Playwright `^1.60.0`이다. 2026-05-31 npm registry 조회 기준 최신 major는 Next `16.2.6`, React `19.2.6`, Prisma Client `7.8.0`, shadcn `4.9.0`, Playwright `1.60.0`이다. 이 스토리에서 major upgrade를 하지 않는다.
- Next.js App Router는 file-system router이며 Server Components, Suspense, Server Functions를 지원한다. 관제판 조회는 Server Component + server query helper로 유지한다. [External: `https://nextjs.org/docs/app`]
- Next.js `revalidatePath`는 Server Functions와 Route Handlers에서 호출할 수 있고 Client Component에서는 호출할 수 없다. 기존 ledger mutation의 dashboard revalidation 패턴을 유지한다. [External: `https://nextjs.org/docs/app/api-reference/functions/revalidatePath`]
- Next.js data fetching 문서는 Server Components에서 ORM/database client query logic이 클라이언트 bundle에 포함되지 않는다고 설명한다. dashboard read query는 이 서버 경계에 둔다. [External: `https://nextjs.org/docs/app/getting-started/fetching-data`]
- Prisma Client는 기본적으로 scalar field만 반환하고 relation은 반환하지 않는다. dashboard query는 `select`/nested `select`로 필요한 필드와 relation count만 명시한다. [External: `https://www.prisma.io/docs/orm/prisma-client/queries/select-fields`]
- Prisma interactive transaction은 오래 열어두면 성능/교착 문제가 생길 수 있다. Story 3.1은 read-only focused query가 핵심이므로 긴 transaction이나 네트워크 작업을 transaction 안에 넣지 않는다. [External: `https://www.prisma.io/docs/orm/prisma-client/queries/transactions`]
- shadcn/ui Skeleton은 content loading placeholder 용도이며 Table 예시가 있다. `loading.tsx`는 실제 관제판 행 구조와 비슷하게 만든다. [External: `https://ui.shadcn.com/docs/components/skeleton`]
