# Story 4.6: 마감 후 정정 기록과 정정 반영값

Status: done

## Story

As a 본사 사용자,
I want 마감 후 오류를 원본 수정이 아니라 정정 기록으로 반영할 수 있기를 원한다,
so that 원본 감사 추적을 잃지 않으면서 운영 숫자를 바로잡을 수 있다.

## Acceptance Criteria

1. **정정 기록 추가와 원본 보존**
   - **Given** 장부가 본사 마감 상태일 때
   - **When** 권한 있는 본사 사용자가 정정 기록을 추가한다
   - **Then** 정정 기록에는 대상 항목, 원본값, 직전 정정 반영값, 새 정정값, 사유, 작성자, 작성 시각이 저장되어야 한다
   - **And** 원본 장부 값은 변경되지 않아야 한다.

2. **정정 사유 필수와 필드 오류**
   - **Given** 정정 사유가 비어 있을 때
   - **When** 사용자가 정정 기록을 저장하려 한다
   - **Then** 저장은 거부되어야 한다
   - **And** UI는 사유 입력 오류를 해당 필드와 연결해 표시해야 한다.

3. **상세 화면의 원본/정정 반영 차이와 시간순 이력**
   - **Given** 정정 기록이 추가되었을 때
   - **When** 장부 상세를 조회한다
   - **Then** 원본값과 정정 반영값의 차이를 확인할 수 있어야 한다
   - **And** 정정 이력은 시간순으로 추적 가능해야 한다.

4. **관제판/리포트 기본 숫자의 정정 반영값 사용**
   - **Given** 관제판이나 리포트가 장부 숫자를 조회할 때
   - **When** 정정 기록이 존재한다
   - **Then** 기본 숫자는 정정 반영값을 사용해야 한다
   - **And** 원본값이 필요한 상세 화면에서는 원본값도 함께 확인할 수 있어야 한다.

5. **권한 없는 정정 차단과 민감 정보 비노출**
   - **Given** 권한 없는 본사 사용자 또는 지점장이 정정 기록을 추가하려 할 때
   - **When** 서버가 요청을 처리한다
   - **Then** 요청은 거부되어야 한다
   - **And** 정정 가능한 민감 필드 정보는 반환되지 않아야 한다.

6. **정정 반영 기준 계산과 OQ-gated 상태 유지**
   - **Given** 정정 기록이 계산에 영향을 줄 때
   - **When** 서버 계산 모듈이 결과를 만든다
   - **Then** 승인된 기본 계산은 정정 반영값 기준으로 갱신되어야 한다
   - **And** OQ-gated 계산은 여전히 승인 전까지 `기준 확인 필요` 상태를 유지해야 한다.

## Dev Notes

### 핵심 구현 방향

- 이 story는 새 정정 시스템을 처음부터 만드는 작업이 아니다. 현재 작업트리에는 `src/features/corrections/actions.ts`, `queries.ts`, `schemas.ts`, `types.ts`, `components/correction-panel.tsx`, `CorrectionRecord` schema, dashboard/report overlay, unit/e2e 테스트가 이미 있다. Dev agent는 이 흐름을 검증하고 빠진 계약을 보강해야 한다.
- 정정 기록은 append-only다. 원본 장부 row를 직접 바꾸거나 "정정 후 원본값"으로 덮어쓰는 구현은 FR-20, NFR2, AR13 위반이다.
- PRD는 정정을 replacement 방식으로 정의한다. 수량/금액 증감(delta)만 저장하는 방식은 별도 승인 전 구현하지 않는다.
- 관제판/리포트 기본 숫자는 최신 정정 반영값을 사용하지만, 상세와 감사 추적에서는 원본값, 직전 정정 반영값, 최신 정정값을 구분해야 한다.
- 정정 영향 전파가 즉시 확정되지 않는 대상은 숫자를 조용히 바꾸지 말고 `정정 확인 필요` 또는 `hasUnappliedCorrections`로 표면화한다.

### 현재 코드 상태와 보존할 동작

- `prisma/schema.prisma`
  - `CorrectionTargetType`은 `LEDGER_FIELD`, `PAYMENT_FIELD`, `EXPENSE_ROW`, `PURCHASE_ROW`, `INVENTORY_ROW`, `LOSS_ROW`, `CALCULATED_METRIC`을 포함한다.
  - `CorrectionRecord`는 `dailyLedgerId`, `targetType`, `targetId`, `fieldKey`, `originalValue`, `previousAppliedValue`, `correctedValue`, `reason`, `createdById`, `createdAt`을 가진다.
  - 보존: `dailyLedgerId + targetType + targetId + fieldKey + createdAt` index와 `createdBy`/`dailyLedger` FK를 유지한다.

- `src/features/corrections/actions.ts`
  - 현재 `createCorrectionRecord()`는 `requireCorrectionCreateAccess()`, `requireHeadquartersLedgerScope()`, `HEADQUARTERS_CLOSED` 확인, 서버 원본값 재조회, 정정값 normalize, advisory transaction lock, `CorrectionRecord` create, `correction.created` audit, 관련 path revalidation을 수행한다.
  - 보존: transaction isolation은 `Serializable`, 같은 target lock은 `pg_advisory_xact_lock(hashtext(key))`, revalidation은 ledger/detail/dashboard/daily/comparison/monthly를 유지한다.
  - 주의: 이 파일에서 원본 business table update가 생기면 regression이다.

- `src/features/corrections/queries.ts`
  - `getLatestCorrectionValueMap()`은 target별 최신 정정을 고르고 `originalValue`, `previousAppliedValue`, `latestAppliedValue`를 제공한다.
  - `getLatestCorrectionValuesForLedgers()`는 report/dashboard용 batch helper다. 반복 ledger별 query로 되돌리면 성능 회귀다.
  - `getStoreReadableCorrectionRecordsForLedger()`는 지점장 읽기 경계를 다루지만 생성 권한과 target 목록 노출은 별개로 차단해야 한다.

- `src/server/calculations/ledger.ts`
  - `applyCorrectionValuesToLedgerReviewInput()`은 원본 input을 clone한 뒤 지원 가능한 정정만 overlay한다.
  - 현재 적용 가능 대상은 payment fields, expense amount, workerCount, inventory currentQuantity/quantity, loss quantity/amount다.
  - `CALCULATED_METRIC`, inventoryAmount, 지원하지 않는 fieldKey, 잘못된 numeric payload는 unapplied로 남겨 `정정 확인 필요`를 유발해야 한다.

- `src/app/app/ledgers/[ledgerId]/page.tsx`
  - `getCorrectionRecordsForLedger()`, `getLatestCorrectionValueMap()`, `createCorrectionRecord`, `CorrectionPanel` 연결이 있다.
  - `getCorrectionTargetOptions()`는 현재 payment, workerCount, expense amount, purchase amount 중심이다. inventory/loss target까지 UI에서 노출할지 검토하되 서버가 이미 지원하는 target과 불일치하지 않게 한다.
  - `ledger.status === "HEADQUARTERS_CLOSED"`일 때 원본 입력 탭 disabled, 닫힌 장부 안내, CorrectionPanel 조건을 보존한다.

- `src/features/dashboard/queries.ts`, `src/features/reports/queries.ts`
  - 기본 숫자는 `applyCorrectionValuesToLedgerReviewInput()` 결과로 계산한다.
  - report evidence는 원본/정정 반영/차이 구분을 유지해야 한다.
  - `hasUnappliedCorrections`는 dashboard/report signal에 남겨야 하며, 숨기거나 정상으로 취급하면 안 된다.

### 권한 및 보안 가드레일

- `CORRECTION_CREATE` 권한 없는 사용자는 생성 action을 실행할 수 없다.
- scope 밖 ledger나 지점장 요청에는 correction target list, fieldKey 후보, originalValue, previousAppliedValue, 계산 상세를 반환하지 않는다.
