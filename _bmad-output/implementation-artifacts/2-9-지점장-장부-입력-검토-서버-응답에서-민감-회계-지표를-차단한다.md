# Story 2.9: 지점장 장부 입력/검토 서버 응답에서 민감 회계 지표를 차단한다

Status: review

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 본사 사용자,
I want 지점장 권한의 장부 입력/검토 응답에서 민감 회계 지표가 빠지길 원한다,
so that 이미 구현된 Epic 2 장부 흐름이 CAP-13 보안 요구와 충돌하지 않는다.

## Acceptance Criteria

1. **Given** 지점장 권한으로 비용/근무 단계와 검토/제출 단계를 조회할 때
   **When** 서버가 응답을 만들면
   **Then** 매출원가, 매출이익, 영업이익, 인당생산성은 응답에 포함되면 안 된다
   **And** UI 숨김만으로 보안을 처리하면 안 된다.

2. **Given** 지점장 화면에 장부 요약이 표시될 때
   **When** 비용/근무 또는 검토/제출 단계가 열리면
   **Then** 현장 입력값, 비용 합계, 허용된 매출 마진율/재고 금액, 누락 항목, 이상 후보, 확인 필요 상태만 표시되어야 한다.

3. **Given** 본사 권한으로 장부를 조회할 때
   **When** 본사 전용 장부 상세 또는 검토 화면이 표시되면
   **Then** 본사 운영에 필요한 회계 지표를 볼 수 있어야 한다.

4. **Given** 기존 Story 2.2와 Story 2.7 테스트를 보정할 때
   **When** 지점장 권한 E2E와 서버 계약 테스트가 실행되면
   **Then** 지점장 화면과 응답에서 민감 회계 지표가 보이지 않는 회귀 테스트가 포함되어야 한다.

5. **Given** 역할별 응답 shaping을 구현할 때
   **When** 장부 입력/검토 query 또는 action result가 반환되면
   **Then** 공통 응답 shaping helper 또는 동등한 서버-side mapper를 사용해야 한다
   **And** 화면 컴포넌트가 민감 필드를 받은 뒤 숨기는 방식으로 처리하면 안 된다.

### Architecture Guardrails

- 보안 경계는 서버다. Client Component 조건부 렌더링은 보조 UI일 뿐, 민감 데이터 차단 근거가 될 수 없다.
- 내부 mutation은 기존처럼 Server Action을 사용한다. 이 스토리에서 새 Route Handler를 만들 필요는 없다.
- `requireStoreAccess`와 기존 store-manager page 흐름을 유지한다. 필요하면 helper에 actor role 또는 view mode를 명시적으로 넘긴다.
- 돈/재고/노무 계산은 shared calculation module을 유지한다. 역할별 mapper는 계산 기준을 바꾸는 곳이 아니라 반환 shape를 제한하는 곳이다.
- 새 DB migration은 예상되지 않는다. 이 스토리는 persisted data 변경이 아니라 response contract 변경이다.
- 공통 상태 문구 `확인 필요`, `계산 불가`, `데이터 부족`은 화면마다 다른 문자열로 흩어지지 않게 한다.
- CAP-12 가드레일도 유지한다. 이 스토리에서 AI 챗봇, 자연어 질의, AI API 호출, AI 프롬프트 저장, AI 분석 결과 저장을 만들지 않는다.

### UX Guardrails

- 지점장 비용/근무/검토 화면은 민감 지표가 빠져도 업무 흐름을 잃으면 안 된다. 남겨야 할 것은 입력값, 비용 합계, 누락 항목, 이상 후보, 제출 흐름이다.
- 지점장에게 보여줄 매출 마진율 기준이 불확실하면 숫자를 임의로 만들지 말고 `확인 필요`로 남긴다.
- 제출 버튼, 성공/오류 피드백, 상태 배지는 Story 2.8 결과를 유지한다.
- 390px 모바일에서 요약 카드가 줄어든 뒤에도 버튼, 상태 배지, 경고 목록이 겹치지 않아야 한다.

### Latest Technical Information

- Next.js data security guidance는 서버에서 데이터 접근과 노출 범위를 의도적으로 좁히는 패턴을 강조한다. 이 스토리는 Client Component가 받은 데이터를 숨기는 방식이 아니라 서버 응답 shape를 제한한다. [External: `https://nextjs.org/docs/app/guides/data-security`]

- Next.js `revalidatePath`는 Server Functions와 Route Handlers에서 호출할 수 있다. 기존 저장/제출 action의 revalidation 흐름은 유지한다. [External: `https://nextjs.org/docs/app/api-reference/functions/revalidatePath`]
- Prisma interactive transaction은 여러 query와 사용자 코드를 하나의 transaction 안에서 실행할 수 있다. 이번 스토리는 새 transaction 패턴보다 기존 action/query transaction 안의 반환 mapper 적용이 중심이다. [External: `https://www.prisma.io/docs/orm/prisma-client/queries/transactions`]
