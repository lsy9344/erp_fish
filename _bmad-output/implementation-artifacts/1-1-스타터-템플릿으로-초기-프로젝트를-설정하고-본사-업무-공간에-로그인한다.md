# Story 1.1: 스타터 템플릿으로 초기 프로젝트를 설정하고 본사 업무 공간에 로그인한다

Status: done

## Story

As a 본사 사용자,
I want 초기 설정된 ERP Fish의 보호된 업무 공간에 로그인하고 기본 본사 화면에 진입하고 싶다,
so that 인증된 사용자만 본사 업무 화면을 사용할 수 있다.

## Acceptance Criteria

1. Given ERP Fish 프로젝트가 초기 구현 상태일 때, when 이 스토리가 완료되면, then `pnpm create t3-app@latest erp_fish` 기반 초기 프로젝트 생성, 의존성 설치, Prisma, NextAuth.js, Tailwind CSS 초기 설정이 완료되어야 한다, and shadcn/ui 초기화와 필요한 환경 변수 예시의 `.env.example` 문서화가 완료되어야 한다.
2. Given 로그인하지 않은 사용자가 보호된 업무 경로에 접근할 때, when `/app` 또는 본사 업무 화면을 직접 열면, then 장부나 기준정보 데이터가 보이지 않아야 한다, and 사용자는 로그인 화면으로 안내되어야 한다.
3. Given 본사 계정이 존재할 때, when 올바른 인증 정보로 로그인하면, then 본사 업무 셸에 진입해야 한다, and 좌측 사이드바에는 홈, 리포트, 기준정보, 설정 진입점이 보여야 한다.
4. Given 로그인한 본사 사용자가 업무 셸을 볼 때, when 화면이 렌더링되면, then shadcn/ui 기반 레이아웃과 ERP Fish 기본 디자인 토큰이 적용되어야 한다, and 라이트/다크 모드에서 기본 텍스트와 주요 버튼 대비가 깨지지 않아야 한다.
5. Given 로그인 폼에 잘못된 값이 입력될 때, when 사용자가 로그인을 시도하면, then 간결한 한국어 오류 메시지가 표시되어야 한다, and 오류는 필드 또는 폼 수준에서 접근 가능하게 연결되어야 한다.
6. Given 로그인과 세션 처리가 서버에서 수행될 때, when 인증 상태를 확인하면, then 클라이언트 UI 표시 여부가 아니라 서버 세션을 기준으로 보호 여부를 판단해야 한다.

- 모든 민감한 서버 진입점은 세션을 로드하고 역할/지점 접근을 확인해야 한다. Story 1.1에서 최소 `src/server/auth.ts`, `src/server/authz.ts` 또는 생성 스택의 동등한 위치에 공유 서버 헬퍼를 둔다. [Source: `_bmad-output/planning-artifacts/architecture.md:180`]
- NextAuth.js와 Auth.js 용어/API를 섞어 쓰지 않는다. T3 생성 결과가 사용하는 패턴을 기준으로 하나의 인증 어휘를 유지한다. [Source: `_bmad-output/planning-artifacts/architecture.md:739`]
- Story 1.1의 로그인은 내부 본사 계정으로 검증 가능해야 한다. Credentials provider, email-based login, 또는 T3 생성 스택이 권장하는 동등한 내부 계정 방식을 선택하되, 외부 OAuth 예제 계정만으로 AC 3을 만족시키면 안 된다.
- Credentials provider를 선택하면 Prisma `User`에 `role`과 `passwordHash` 또는 동등한 서버 검증 필드를 추가하고, seed는 환경 변수의 개발용 비밀번호를 해시해서 저장한다. 평문 비밀번호를 저장소나 DB seed 파일에 직접 커밋하지 않는다.
- `/app` 보호는 선행 redirect만으로 끝내지 않는다. 보호된 서버 페이지와 서버 query/helper에서 세션과 role을 다시 확인해 데이터 조회 전 차단한다.
- 최신 Next.js 인증 가이드는 미들웨어/프록시 같은 선행 체크가 유용해도 데이터 보호의 유일한 방어선이 되면 안 된다고 설명한다. Story 1.1도 `/app` 선행 redirect와 서버 페이지/쿼리의 세션 확인을 함께 둔다. [External: https://nextjs.org/docs/app/guides/authentication]
- Auth.js 최신 예시는 `auth.ts`에서 `auth`와 `handlers`를 내보내고 route handler에서 `GET`, `POST`를 연결하는 패턴을 보여준다. T3 생성 코드가 이 패턴을 쓰면 그대로 따른다. [External: https://authjs.dev/]

### API, Server, And Data Boundaries

- MVP는 Next.js App Router 서버 경계를 직접 사용한다. Server Components는 인증된 read를 담당하고, Server Actions는 내부 mutation을 담당한다. [Source: `_bmad-output/planning-artifacts/architecture.md:191`]
- Route Handlers는 인증, health check, export, future webhook처럼 실제 HTTP endpoint가 필요한 경우에만 사용한다. [Source: `_bmad-output/planning-artifacts/architecture.md:203`]
- 모든 mutation은 Zod schema로 검증한다. Story 1.1의 로그인 폼도 입력 검증과 한국어 오류 표시를 둔다. [Source: `_bmad-output/planning-artifacts/architecture.md:211`]
- 예상 오류는 표준 결과 형태를 사용한다. 공통 타입은 `src/lib/action-result.ts`에 둘 수 있다. [Source: `_bmad-output/planning-artifacts/architecture.md:390`]
