## Epic 요약

Epic 4는 본사 운영자가 전체 지점 상태를 빠르게 보고, 마감 전 장부를 직접 보완하고, 마감 전에 위험을 점검하고, 본사 마감 후 원본을 잠근 뒤 정정 기록으로 운영 숫자를 반영하는 흐름을 완성했다. 구현 결과는 FR-15, FR-18, FR-19, FR-20, FR-21, NFR2, NFR10, NFR11의 핵심 방향과 맞는다.

완료된 핵심 기반:

- `/app/dashboard`의 본사 관제판이 활성 지점, 장부 상태, 최신 반영 시각, 마지막 수정자, 기본 요약, 정정 상태, signal을 서버 계산 기준으로 표시
- OQ-gated 이상 후보를 확정 warning/critical로 승격하지 않고 `기준 확인 필요`로 유지
- 본사 원본 보완 수정 action에 권한, store scope, version/edit token, 사유, audit before/after, 닫힌 장부 차단을 적용
- ClosePreflight가 권한, 상태, 필수 누락, 계산 상태, 이월/가격 기준, 정정 반영 상태, 이상 신호를 표 항목으로 요약
- `closeHqLedger()`가 stale token을 먼저 판정하고, transaction 안에서 preflight를 다시 실행한 뒤 마감 상태와 audit log를 기록
- 개별 마감 예외는 `exception-allowed` 항목이 있고 blocking 항목이 없을 때 사유를 요구하도록 정리
- `CorrectionRecord` 기반 append-only 정정 흐름과 `applyCorrectionValuesToLedgerReviewInput()` overlay를 관제판, 리포트, preflight가 공유

## 잘된 점

1. Epic 3의 계산/정책 gate 학습이 실제로 적용됐다.
   - 관제판과 ClosePreflight 모두 `policy-unconfirmed`, `data-insufficient`, `calculation-unavailable`을 구분했다.
   - OQ-1 미확정 상태에서 기준값 저장 여부와 확정 이상 판정을 분리했다.

2. 본사 권한 경계가 UI 표시가 아니라 server action 중심으로 유지됐다.
   - 조회 전용 본사는 상세 조회와 mutation 권한이 분리됐다.
   - preflight와 correction target detail은 권한과 ledger scope 통과 후에만 구성되도록 강화됐다.

3. 마감 흐름이 snapshot 의존에서 transaction 재검증으로 성숙했다.
   - UI preflight 통과를 그대로 믿지 않고 `closeHqLedger()`가 같은 transaction 안에서 다시 검증한다.
   - stale token은 preflight blocker보다 먼저 `LEDGER_CONFLICT`로 반환하도록 계약이 고정됐다.

4. 원본 보존과 정정 반영값의 경계가 명확해졌다.
   - 본사 마감 후 원본 row update/delete 대신 `CorrectionRecord`만 append한다.
   - 관제판과 리포트는 적용 가능한 최신 정정값을 기본 숫자로 쓰되, 적용 불가 정정은 `정정 확인 필요`로 표면화한다.

5. 반복 review가 실제 결함을 줄였다.
   - ClosePreflight 정상 상태 행 누락, stale token 순서, inventory adjustment policy status, 모바일 overflow, correction target option 권한 게이트 같은 문제가 review에서 잡히고 수정됐다.

## 어려웠던 점

1. "조회 가능"과 "수정/마감/정정 가능" 권한을 계속 분리해야 했다.
   - 본사 조회 전용 계정은 관제판과 상세를 볼 수 있지만 mutation은 차단되어야 한다.
   - correction target option처럼 화면에 렌더링되지 않아도 서버 렌더 중 민감 원본값을 준비하는 것 자체가 위험이었다.

2. ClosePreflight는 단순 경고 목록이 아니라 마감 계약이었다.
   - 필수 누락, 이월 공백, 가격 기준 없음은 개별 예외 사유가 있으면 허용될 수 있지만, 숫자 오류와 정정 영향 미확정은 차단으로 유지해야 했다.
   - 이 분류가 UI, close action, audit payload에 함께 반영되어야 했다.

3. 정정은 "숫자 바꾸기"가 아니라 감사 가능한 overlay였다.
   - 지원 가능한 target만 운영 기본값에 반영하고, 계산 metric 직접 정정이나 지원 불가 field는 조용히 반영하지 않도록 해야 했다.
   - 최신 정정값 tie-breaker와 batch 조회가 필요해 관제판/리포트 성능 경계도 같이 고려했다.

4. E2E 실행 환경 제약이 반복됐다.
   - Story 4.1부터 4.6까지 Playwright webServer가 sandbox 포트 listen 제한으로 실행되지 못했다.
   - unit, typecheck, lint, build 검증은 통과했지만 브라우저 수준 회귀는 실행 가능한 환경에서 별도 확인이 필요하다.

## 핵심 학습

1. ERP Fish의 본사 관제와 리포트 숫자는 shared server calculation과 correction overlay를 기준으로 해야 한다.
2. ClosePreflight 결과는 UI 안내가 아니라 마감 실행의 서버 계약이다. close action은 반드시 재검증해야 한다.
3. 권한 없는 사용자에게는 action 결과뿐 아니라 target 후보, 원본값, closeability detail도 반환하면 안 된다.
4. 본사 마감 후 변경은 append-only 정정 기록이며, 원본 row를 수정하는 편의 구현은 제품 원칙과 맞지 않는다.
5. 적용 불가 정정과 OQ-gated 계산은 조용히 숫자로 바꾸지 말고 `정정 확인 필요`와 `기준 확인 필요`로 남겨야 한다.

## Epic 5 준비 평가

Epic 5는 본사 기준 정보와 운영 설정을 다룬다. Epic 4의 결과는 Epic 5에 직접 선행 조건을 제공한다.

준비된 의존성:

- permission action별 server guard와 read-only HQ 경계
- audit log의 before/after/reason 패턴
- active/inactive 상태가 신규 입력 선택지에 미치는 영향에 대한 관제판 precedent
- OQ-gated threshold와 확정 이상 판정을 분리하는 signal contract
- report/dashboard revalidation과 correction-applied calculation pattern

주의할 의존성:

- Sprint status에는 Epic 5 story가 backlog인데 `epic-5-retrospective`가 done으로 표시되어 있다. Epic 5 착수 전 sprint-status 정합성을 정리해야 한다.
- Story 5.5의 기준값 저장 구조는 만들 수 있어도 OQ-1이 닫히기 전 확정 이상 판정에 사용하면 안 된다.
- 지점/품목/매입 기준/코드 비활성화는 과거 장부 원본 표시를 깨면 안 된다.
- 마스터 변경은 신규 선택지와 운영 기준을 바꾸지만 과거 원본 장부 값을 조용히 덮어쓰면 안 된다.

## Readiness Assessment

- Testing & Quality: unit, typecheck, lint, build 검증은 story별로 반복 수행됐다. E2E는 포트 listen 제한으로 실행되지 못했다.
- Deployment: production deploy 자체는 Epic 4 범위가 아니다. 서버 계산, 권한, 마감, 정정 흐름은 테스트 가능한 단위 계약으로 준비됐다.
- Stakeholder Acceptance: 자동 회고이므로 직접 stakeholder acceptance는 확인하지 않았다. story status와 review 기록 기준으로 완료 판단했다.
- Technical Health: shared calculation, correction overlay, preflight 재검증, 권한 게이트가 강화됐다. 가장 큰 잔여 리스크는 Epic 5가 threshold/마스터 변경을 과거 장부나 OQ-gated signal에 잘못 적용하는 것이다.
- Unresolved Blockers: Epic 5 제품 방향을 바꿀 코드 blocker는 확인되지 않았다. 추적 blocker는 Epic 5 sprint-status/retro 상태 불일치와 E2E 실행 환경이다.

## Significant Discovery Check

Epic 4 구현 과정에서 Epic 5 계획을 전면 재작성할 수준의 발견은 없었다. 다만 다음 구현 기준은 Epic 5 story에 반영되어야 한다.

- 저장된 기준값은 정책 승인 상태와 다르다.
- 기준/마스터 변경은 신규 운영 기준에만 적용하고 과거 원본 장부를 덮어쓰지 않는다.
- 설정 관리자 권한은 본사 조회 권한과 별도로 mutation action 단위로 검증해야 한다.
- audit reason은 본사 수정, 마감 예외, 정정, 기준 변경에 일관되게 필요하다.
