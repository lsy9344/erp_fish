# Story 1.5: 감사 로그 기반 구축

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 본사 관리자,
I want 주요 변경 이력이 누가 언제 무엇을 바꿨는지 남기를 원한다,
so that 장부, 권한, 기준값 변경을 나중에 신뢰할 수 있게 추적할 수 있다.

## Acceptance Criteria

1. Given 감사 대상 이벤트가 발생할 때, when 서버 mutation이 실행된다, then 감사 로그에는 actor 계정, 권한 컨텍스트, 대상 엔티티, 이벤트 유형, 발생 시각이 기록되어야 한다, and 가능한 경우 변경 전 값, 변경 후 값, 사유도 함께 기록되어야 한다.
2. Given 권한 변경이나 지점 접근권 변경이 발생할 때, when 본사 설정 관리자가 변경을 저장한다, then 변경 전/후 권한과 대상 사용자가 감사 로그에 남아야 한다, and 변경 실행자와 변경 시각을 확인할 수 있어야 한다.
3. Given 아직 장부 입력 기능이 완성되지 않았을 때, when 감사 로그 기반 스토리가 완료된다, then 사용자/권한/지점 접근 변경 이벤트로 감사 로그 기능을 독립 검증할 수 있어야 한다, and 장부 전체 모델을 미리 구현하지 않아야 한다.
4. Given 본사 사용자가 감사 로그를 확인할 때, when 감사 로그 목록을 조회한다, then 최근 이벤트를 사람이 읽기 쉬운 문장과 기본 필터로 볼 수 있어야 한다, and 권한 없는 사용자는 감사 로그에 접근할 수 없어야 한다.
5. Given 향후 장부, 마감, 정정, 기준값 변경 기능이 추가될 때, when 개발자가 새 mutation을 구현한다, then 기존 감사 로그 helper를 재사용할 수 있어야 한다, and 각 기능이 서로 다른 감사 로그 형식을 만들지 않아야 한다.

### Architecture Guardrails

- 선택 스택은 TypeScript, Next.js App Router, Prisma, PostgreSQL, NextAuth.js/Auth.js, Tailwind CSS, shadcn/ui다. 이 스토리에서 auth framework, ORM, UI primitive 체계를 바꾸지 않는다. [Source: `_bmad-output/planning-artifacts/architecture.md#Technology Stack Selection`]
- 감사 이벤트는 별도 event bus가 아니라 `AuditLog` row로 표현한다. action 이름은 dot notation을 사용하고 payload에는 actor, target entity, before, after, reason where applicable을 포함한다. [Source: `_bmad-output/planning-artifacts/architecture.md#Event and Audit Patterns`]
- 모든 sensitive server action/query는 shared authorization helper를 먼저 호출해야 한다. 변경 이력 조회는 settings 권한이 있는 본사 사용자만 볼 수 있어야 한다. [Source: `_bmad-output/planning-artifacts/architecture.md#Process Patterns`]
- Server Components는 read, Server Actions는 mutation을 담당한다. public API route를 만들지 않는다. [Source: `_bmad-output/planning-artifacts/architecture.md#API & Communication Patterns`]
- `src/server`는 auth, authorization, audit logging, transactions 같은 cross-cutting server behavior의 소유 경계다. feature code는 helper를 호출하고 audit writer를 복제하지 않는다. [Source: `_bmad-output/planning-artifacts/architecture.md#Architectural Boundaries`]
- Prisma schema와 migration은 persistence source of truth다. 기존 migration을 고치지 말고 필요한 경우 새 migration을 추가한다. [Source: `_bmad-output/planning-artifacts/architecture.md#Database Architecture`]

### Data and Security Guardrails

- Audit row는 append-only다. audit history 화면에서 수정/삭제 action을 만들지 않는다.
- actor는 인증 계정 id여야 한다. 장부 입력 작성자 표시명, 대상 사용자명, 화면 표시 문자열을 actor 대신 쓰지 않는다.
- 권한 컨텍스트는 최소 actor의 권한 action 또는 profile/store scope를 추적 가능하게 해야 한다. 현 스키마에 별도 컬럼이 없다면 `after`/`before` payload 또는 helper 입력에 과도하지 않은 context를 담는 방식을 검토하되 password/session/token은 절대 담지 않는다.
- 권한 변경 감사는 대상 사용자와 실행자를 혼동하면 안 된다. `targetId`는 변경 대상 사용자, `actorId`는 변경 실행자다.
- 사용자/권한 변경 action은 `requireUserPermissionAccess()`를 사용하고, 변경 이력 read는 `requireSettingsAccess()` 또는 동등한 설정 권한 helper를 사용한다. one-off `role === "HEADQUARTERS"` 검사만으로 통과시키지 않는다.
- 예외/실패 시 감사 로그를 거짓으로 남기지 않는다. business write가 실패하면 audit write도 rollback되어야 한다.

### UX Guardrails

- 본사 화면은 기존 `HeadquartersShell`, 좌측 sidebar, `PageHeader`, shadcn table/form primitive를 따른다.
- 변경 이력 목록은 내부 ERP 표면이어야 한다. 마케팅형 설명, 카드 중첩, 별도 랜딩 화면을 만들지 않는다.
- 필터는 URL query로 유지한다: `targetType`, `actorId`, `from`, `to`.
- 날짜 필터는 한국 시간 기준 일자 범위로 해석한다.
- 긴 JSON 상세는 dialog 안에서 넘침 없이 스크롤되어야 한다. 버튼 텍스트와 테이블 텍스트가 모바일/데스크탑에서 겹치지 않아야 한다.
- 권한 없음은 빈 화면이 아니라 `/app/unauthorized` 패턴을 따른다.

### Latest Technical Information

- 2026-06-11 확인: Next.js `revalidatePath`는 Server Functions와 Route Handlers에서 호출할 수 있고 Client Components나 Proxy에서는 사용할 수 없다. 감사/사용자/지점 mutation 후 revalidation은 Server Action 안에서 수행한다. [External: `https://nextjs.org/docs/app/api-reference/functions/revalidatePath`]
- 2026-06-11 확인: Prisma transaction은 read/write 묶음이 전체 성공 또는 전체 실패하도록 보장한다. business write와 audit write는 같은 interactive transaction으로 묶는다. [External: `https://www.prisma.io/docs/orm/prisma-client/queries/transactions`]
- 2026-06-11 확인: Auth.js Credentials provider의 `authorize`는 앱의 사용자 검증 로직을 호출하고 실패 시 null/error를 반환하는 구조다. Story 1.5는 인증 방식을 바꾸지 않고 actor id를 기존 auth/session 기반으로만 사용한다. [External: `https://authjs.dev/getting-started/authentication/credentials`]
