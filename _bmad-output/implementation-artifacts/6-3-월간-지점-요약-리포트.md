# Story 6.3: 월간 지점 요약 리포트

Status: done

## Story

As a 본사 사용자,  
I want 지점별 월간 마감 현황과 운영 요약을 볼 수 있기를 원한다,  
so that 월 단위로 지점 운영 상태를 점검할 수 있다.

## Acceptance Criteria

1. **Given** 본사 사용자가 월간 지점 요약 화면에 접근할 때, **When** 대상 월과 지점 필터를 선택한다, **Then** 권한 있는 지점의 월간 마감 현황, 주요 상태, 손실/재고 흐름 요약이 표시되어야 한다, **And** 권한 밖 지점 데이터는 포함되지 않아야 한다.
2. **Given** 해당 월에 마감 장부와 미마감 장부가 섞여 있을 때, **When** 월간 요약이 표시된다, **Then** 마감/미마감 건수와 미마감 포함 여부를 보여줘야 한다, **And** 사용자가 숫자의 확정 상태를 알 수 있어야 한다.
3. **Given** 정정 기록이 있는 장부가 월간 집계에 포함될 때, **When** 리포트가 기본 숫자를 계산한다, **Then** 정정 반영값을 사용해야 한다, **And** 원본값 확인은 장부 상세에서 가능해야 한다.
4. **Given** 최고매출품목, 평균재고, 재고비율 같은 항목이 정책 미정 또는 데이터 부족일 때, **When** 월간 요약이 표시된다, **Then** `기준 확인 필요`, `데이터 부족`, `계산 불가` 상태를 구분해 보여야 한다, **And** 확정 숫자처럼 표시하면 안 된다.
5. **Given** 지점장 권한 사용자가 월간 본사 요약에 접근할 때, **When** 서버가 요청을 처리한다, **Then** 접근은 거부되어야 한다, **And** 타 지점 비교 또는 민감 지표는 반환되지 않아야 한다.
6. **Given** 월간 요약 화면이 렌더링될 때, **When** 숫자와 상태가 표시된다, **Then** 숫자 컬럼은 읽기 쉽게 정렬되어야 한다, **And** 상태는 색상만이 아니라 텍스트로도 의미를 알 수 있어야 한다.

### Gap 분석

- PRD FR-29는 "지점별 월간 실적"과 기본 정렬 "지점명, 월 순서"를 명시하지만 현재 route는 단일 선택 지점 요약이다. 구현자는 다중 지점 표가 필요한지 결정하고, 단일 지점 요약을 유지할 경우 화면 설명과 테스트에서 제품 제한을 명시해야 한다.
- PRD FR-29 필수 컬럼의 "정정 반영 건수"가 화면에서 명시적으로 드러나는지 확인해야 한다. 현재 metric evidence와 anomaly items는 정정 타임라인을 보여주지만, 월 전체 정정 반영 건수 요약이 없을 수 있다.
- AC2는 "마감/미마감 건수와 미마감 포함 여부"를 요구한다. 현재 `statusCounts`는 closed/review/inProgress/missing/holiday를 분리하지만 KPI 자체에 `미마감 포함` 신호가 충분히 붙는지 확인해야 한다.
- 최고매출품목은 현재 `계산 기준 확인 필요`로 의도적으로 막혀 있다. Dev agent가 이를 미완성으로 오판해 임의 계산을 넣으면 CAP-4/OQ-gated 상품 분석을 앞당기는 회귀가 된다.
- invalid or out-of-scope `storeId` fallback은 권한 밖 데이터 포함은 아니지만, 잘못된 URL이 다른 지점 데이터를 보여줄 수 있어 Story 6.2의 no-row guardrail과 UX가 다르다. 이 차이는 의식적으로 정해야 한다.

### Architecture Guardrails

- ERP Fish는 Next.js App Router, Server Components, Server Actions, Prisma, PostgreSQL, NextAuth/Auth.js, Tailwind/shadcn UI를 사용한다. 이 story에서 tRPC, 별도 REST API, client-only data fetch layer를 추가하지 않는다. [Source: `_bmad-output/planning-artifacts/architecture.md#API & Communication Patterns`] [Source: `package.json`]
- Report, dashboard, detail pages는 같은 server-side calculation functions를 공유해야 한다. 계산식을 UI 컴포넌트나 report 전용 helper에 복제하면 안 된다. [Source: `_bmad-output/planning-artifacts/architecture.md#Calculation Strategy`]
- Original ledger records는 본사 마감 후 보존된다. Reports and dashboards use correction-applied values by default while details keep original values visible. [Source: `_bmad-output/planning-artifacts/architecture.md#Audit and Correction Strategy`]
- `applyCorrectionValuesToLedgerReviewInput`는 dashboard, reports, close preflight가 공유하는 correction overlay다. Unsupported/not-yet-policy-approved corrections는 `정정 확인 필요` 또는 `기준 확인 필요`로 보여야 한다. [Source: `_bmad-output/planning-artifacts/architecture.md#Audit and Correction Strategy`]
