# Story 5.3: 매입 기준 관리

Status: done

## Story

As a 본사 설정 관리자,  
I want 매입 입력에서 참조할 기준 정보를 관리할 수 있기를 원한다,  
so that 지점장과 본사 사용자가 장부 매입을 더 일관되게 입력할 수 있다.

## Acceptance Criteria

1. **Given** 설정 관리자 권한을 가진 본사 사용자가 매입 기준 관리 화면에 접근할 때, **When** 매입 기준 목록을 조회한다, **Then** 품목, 규격, 기준 단가 또는 기준 설명, 활성/비활성 상태를 볼 수 있어야 한다, **And** 권한 없는 사용자는 매입 기준 mutation을 수행할 수 없어야 한다.
2. **Given** 설정 관리자가 매입 기준을 추가하거나 수정할 때, **When** 필수 값이 유효하다, **Then** 기준 정보는 저장되어야 한다, **And** 변경 전/후 값과 실행자는 감사 로그에 남아야 한다.
3. **Given** 매입 기준이 비활성화되었을 때, **When** 사용자가 신규 매입 라인을 입력한다, **Then** 비활성 기준은 신규 선택지에서 제외되어야 한다, **And** 과거 장부에서 이미 참조한 기준 정보는 원본 표시를 유지해야 한다.
4. **Given** 지점장 또는 본사 사용자가 장부 매입 입력을 할 때, **When** 활성 매입 기준이 존재한다, **Then** 사용자는 기준 정보를 선택하거나 참조할 수 있어야 한다, **And** 기준 정보는 입력 편의와 검토 기준으로만 쓰이며, 사용자가 입력한 장부 원본값을 조용히 덮어쓰면 안 된다.
5. **Given** 이카운트 업로드 기능이 후속 범위에 있을 때, **When** 이 스토리를 구현한다, **Then** 업로드 preview/commit, 매핑 실패 검수, reprocess 기능은 포함하지 않아야 한다, **And** 후속 업로드 기능이 같은 기준 정보를 참조할 수 있는 구조는 허용된다.
6. **Given** 단가나 금액 값이 입력될 때, **When** 서버가 값을 검증한다, **Then** 금액은 integer KRW 기준이어야 한다, **And** 잘못된 숫자, 음수, 필수 누락은 거부되어야 한다.

### 현재 구현 상태

- `PurchaseStandard` 모델은 이미 `id`, `productId`, `standardUnitPrice`, `referenceInfo`, `isActive`, `createdAt`, `updatedAt`, `updatedById`를 갖고 `Product`와 `LedgerPurchaseItem`에 연결된다. 새 table migration을 다시 만들지 않는다. [Source: `prisma/schema.prisma#model PurchaseStandard`]
- `PurchaseStandard.product`는 `onDelete: Restrict`이고 `LedgerPurchaseItem.purchaseStandard`는 `onDelete: SetNull`이다. 기준 삭제를 운영 기능으로 만들면 과거 장부 참조/감사 추적을 깨뜨릴 수 있으므로 비활성화가 기본 방식이다. [Source: `prisma/schema.prisma#model PurchaseStandard`] [Source: `prisma/schema.prisma#model LedgerPurchaseItem`]

- `src/features/master-data/purchase-standard-actions.ts`에는 생성/수정/상태 변경 action이 이미 있고 `requireSettingsAccess()`, Prisma transaction, `writeAuditLog`, `ActionResult`, route revalidation을 사용한다. 이 파일을 확장하고 새 action family를 만들지 않는다. [Source: `src/features/master-data/purchase-standard-actions.ts`]

### Architecture Guardrails

- ERP Fish는 Next.js App Router, Server Components, Server Actions, Prisma, PostgreSQL, Auth.js/NextAuth, Tailwind/shadcn UI를 사용한다. 이 story에서 public API route나 tRPC abstraction을 추가하지 않는다. [Source: `_bmad-output/planning-artifacts/architecture.md#Technology Stack`] [Source: `package.json`]
- business code는 `src/features/master-data`, ledger input code는 `src/features/ledger`, shared auth/audit는 `src/server`, UI primitive는 `src/components/ui`에 둔다. [Source: `_bmad-output/planning-artifacts/architecture.md#Project Structure & Boundaries`]
- 모든 settings query/action은 shared authorization helper를 먼저 호출해야 한다. 매입 기준 관리는 `PermissionAction.SETTINGS_MANAGE` 기반의 `requireSettingsAccess()`가 서버 경계다. [Source: `_bmad-output/planning-artifacts/architecture.md#Authorization Pattern`]
- expected validation/business error는 `ActionResult<T>`로 반환한다. field error는 UI에서 `aria-invalid`, `aria-describedby`, 첫 오류 focus와 연결한다. [Source: `_bmad-output/planning-artifacts/architecture.md#Action Response Formats`]
- business-changing action은 business write와 audit write를 같은 transaction에 넣는다. 감사 이벤트는 actor, target, before, after를 포함해야 한다. [Source: `_bmad-output/planning-artifacts/architecture.md#Audit Pattern`]
- 돈은 DB와 서버 boundary에서 integer KRW로 유지하고 UI에서만 한국어 금액 포맷을 적용한다. [Source: `_bmad-output/planning-artifacts/architecture.md#Data Exchange Formats`]
- master-data 변경 뒤에는 dashboard, store-entry, reports가 stale 값을 보지 않도록 Server Action에서 revalidation을 수행한다. [Source: `_bmad-output/planning-artifacts/architecture.md#Revalidation Strategy`]
- Store-manager ledger input은 최소 390px 모바일 폭에서도 동작해야 한다. 매입 기준 UI 변경이 `PurchaseStepClient`의 모바일 입력성과 기존 unsaved/conflict dialogs를 깨면 안 된다. [Source: `_bmad-output/planning-artifacts/architecture.md#Responsive Requirements`]
