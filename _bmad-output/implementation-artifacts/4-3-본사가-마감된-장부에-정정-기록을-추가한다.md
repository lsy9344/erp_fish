# Story 4.3: 본사가 마감된 장부에 정정 기록을 추가한다

Status: done

## Story

As a 본사 사용자,
I want 마감된 장부에 원본을 바꾸지 않는 정정 기록을 추가하고 싶다,
so that 원본 감사 추적을 보존하면서 운영 숫자를 바로잡을 수 있다.

## Acceptance Criteria

1. **Given** 장부가 `본사마감` 상태일 때
   **When** 본사 사용자가 정정 기록 추가를 선택하면
   **Then** 정정 입력 영역이 열려야 한다
   **And** 원본 입력 필드는 계속 disabled 상태여야 한다.

2. **Given** 본사 사용자가 정정 기록을 작성할 때
   **When** 대상 항목, 원본값, 정정값, 사유를 입력하고 저장하면
   **Then** append-only `CorrectionRecord`가 생성되어야 한다
   **And** 원본 장부 값은 변경되면 안 된다.

3. **Given** 정정 대상 항목을 선택할 때
   **When** 사용자가 정정 대상을 지정하면
   **Then** 대상은 안정적인 식별자로 저장되어야 한다
   **And** 대상 유형은 장부 필드, 결제/비용/매입/재고/손실 행, 계산 표시값 중 어떤 범위인지 구분되어야 한다.

4. **Given** 정정 기록이 저장될 때
   **When** 원본값을 기록하면
   **Then** 정정 당시의 원본값 또는 이전 반영값을 비교할 수 있도록 저장해야 한다
   **And** 원본 장부 자체를 덮어쓰면 안 된다.

5. **Given** 같은 대상 항목에 여러 정정 기록이 있을 때
   **When** 정정 반영값을 계산하면
   **Then** 최신 정정 기록의 `정정값`을 해당 대상의 최종 반영값으로 사용해야 한다
   **And** 이전 정정 기록은 삭제되지 않고 타임라인에 남아야 한다.

6. **Given** 정정 사유가 비어 있을 때
   **When** 저장을 시도하면
   **Then** 저장이 차단되어야 한다
   **And** 사유 필드에 한국어 검증 메시지가 표시되어야 한다.

7. **Given** 지점장 또는 권한 없는 사용자가 정정 기록 추가를 시도할 때
   **When** 서버가 요청을 처리하면
   **Then** 정정 기록은 생성되지 않아야 한다
   **And** 사용자는 "접근 권한이 없습니다." 화면 또는 권한 오류 안내를 봐야 한다.

8. **Given** 정정 기록이 생성될 때
   **When** 저장이 완료되면
   **Then** 정정 기록과 감사 기록은 같은 데이터베이스 트랜잭션 안에서 저장되어야 한다
   **And** 감사 기록에는 actor, timestamp, target, before, after, reason이 포함되어야 한다.

## Dev Notes

### 핵심 목표

- Story 4.3의 핵심은 "마감 후 운영 숫자를 바로잡되 원본 장부를 절대 덮어쓰지 않는다"이다.
- `CorrectionRecord`는 append-only다. 정정 취소/삭제/수정은 이번 story에서 만들지 않는다.
- `AuditLog`는 별도 요구다. `CorrectionRecord`가 있어도 `correction.created` 감사 로그를 같은 transaction에 남겨야 한다.
- Story 4.4/4.5가 원본값/정정 반영값 표시와 관제판 반영 계산을 이어받을 수 있게 데이터 구조와 최신값 선택 helper를 남긴다.

### 현재 구현 상태

- `CorrectionRecord` 모델과 `src/features/corrections` 폴더는 아직 없다.
- `DailyLedgerStatus`는 `IN_PROGRESS`, `IN_REVIEW`, `HEADQUARTERS_CLOSED`, `HOLIDAY`를 가진다.
- Story 4.2로 `DailyLedger.closedById`, `closedAt`, `closedBy` relation, `ledger.hq.closed` 감사 기록이 추가됐다.
- 기존 원본 저장 action은 `IN_PROGRESS`/`IN_REVIEW`만 허용하고, `HEADQUARTERS_CLOSED`에서는 정정 기록 안내를 반환한다.
- `src/app/app/ledgers/[ledgerId]/page.tsx`는 마감 장부에서 원본 입력 컨트롤을 disabled로 보여주며, 아직 실제 정정 입력 UI는 없다.
- `src/features/dashboard/queries.ts`는 현재 원본 `DailyLedger` 값으로 관제판/상세 요약을 계산한다. 4.3에서 이 쿼리를 무겁게 바꾸지 말고 정정 생성과 최신값 helper에 집중한다.

### 구현 가드레일

1. 원본 테이블 update/delete 금지: 정정 저장은 `CorrectionRecord.create`와 `AuditLog.create`만 수행한다.
2. 본사 권한은 서버에서 강제한다. 클라이언트 UI 숨김은 보안 경계가 아니다.
3. 정정 가능 상태는 `HEADQUARTERS_CLOSED`뿐이다. 마감 전 장부 수정은 Story 4.1의 원본 편집 흐름을 사용한다.
4. 정정 대상은 사람이 읽는 label이 아니라 안정적인 target metadata로 저장한다.
5. 같은 대상의 여러 정정은 최신 record가 적용값이다. 이전 record는 삭제하지 않는다.
6. 감사 payload에는 target metadata, 이전값, 새 정정값, reason을 넣는다.
7. `revalidatePath` 실패가 이미 커밋된 정정 저장을 실패처럼 보이게 만들지 않도록 4.2의 best-effort 패턴을 참고한다.
8. dashboard/report 계산 반영 전체는 4.5/Epic 5 범위지만, 4.3에서 만든 helper가 그 기준이 되어야 한다.

### Data Model Notes

- Prisma `Json` is acceptable for `originalValue`, `previousAppliedValue`, `correctedValue` because 정정 대상은 금액, 수량, 텍스트, 계산 표시값을 모두 포함한다.
- JSON을 쓰더라도 shape를 좁게 유지한다. 예: `{ "kind": "money", "value": 45000 }`, `{ "kind": "quantity", "value": 12 }`, `{ "kind": "text", "value": "..." }`.
- 보고/계산에 직접 필요한 target/value는 helper에서 안전하게 해석할 수 있어야 한다. 애매한 형식이면 계산에 임의 반영하지 말고 "정정 확인 필요" 상태를 후속 story로 넘긴다.
- `createdAt desc, id desc` 정렬을 써서 같은 시각 정정의 tie를 안정적으로 처리한다.

### UX Notes

- `CorrectionPanel`은 마감 장부 상세의 원본 입력 탭과 별도 영역이다.
- 원본 입력 필드는 계속 disabled 상태여야 한다.
- 대상 선택 UI는 처음에는 제한된 항목부터 시작해도 된다. 단, 저장 구조는 비용/매입/재고/손실 row target까지 확장 가능해야 한다.
- 색상이나 취소선만으로 원본/정정 의미를 전달하지 않는다. 텍스트 label을 사용한다.
- 사유 필드는 필수이며 첫 오류 focus 대상이 되어야 한다.
