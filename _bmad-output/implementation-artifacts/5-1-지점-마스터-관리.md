# Story 5.1: 지점 마스터 관리

Status: done

## Story

As a 본사 설정 관리자,  
I want 지점명과 활성/비활성 상태를 관리할 수 있기를 원한다,  
so that 운영 중인 지점만 신규 장부 입력 대상이 되고 과거 데이터는 보존할 수 있다.

## Acceptance Criteria

1. **Given** 설정 관리자 권한을 가진 본사 사용자가 지점 관리 화면에 접근할 때, **When** 지점 목록을 조회한다, **Then** 지점명, 활성/비활성 상태, 생성/수정 시각을 볼 수 있어야 한다, **And** 권한 없는 사용자는 지점 관리 화면과 mutation에 접근할 수 없어야 한다.
2. **Given** 설정 관리자가 새 지점을 추가할 때, **When** 필수 값이 유효하다, **Then** 새 지점이 생성되어야 한다, **And** 생성 이벤트는 감사 로그에 남아야 한다.
3. **Given** 설정 관리자가 지점명을 수정할 때, **When** 저장 요청이 성공한다, **Then** 새 지점명이 신규 화면에 반영되어야 한다, **And** 변경 전/후 지점명은 감사 로그에 남아야 한다.
4. **Given** 설정 관리자가 지점을 비활성화할 때, **When** 저장 요청이 성공한다, **Then** 비활성 지점은 신규 장부 생성 선택지에서 제외되어야 한다, **And** 과거 장부와 감사 로그는 보존되어야 한다.
5. **Given** 이미 장부가 있는 지점을 삭제하려 할 때, **When** 서버가 요청을 처리한다, **Then** 실제 삭제는 차단되어야 한다, **And** 사용자는 비활성화를 사용해야 한다는 안내를 받아야 한다.
6. **Given** 지점명 또는 상태 값이 잘못 입력되었을 때, **When** 서버가 요청을 검증한다, **Then** 저장은 거부되어야 한다, **And** UI는 해당 필드에 연결된 오류 메시지를 보여줘야 한다.

### Architecture Guardrails

- MVP 기능은 Server Components와 Server Actions가 기본이다. 이 story는 public Route Handler를 추가하지 않는다. [Source: _bmad-output/planning-artifacts/architecture.md#API Boundaries]
- business code는 `src/features/master-data`, shared auth/audit는 `src/server`, UI primitive는 `src/components/ui`에 둔다. [Source: _bmad-output/planning-artifacts/architecture.md#Structure Patterns]
- action result는 `ActionResult<T>` shape를 유지한다. expected validation/business error는 throw하지 말고 `actionError`로 반환한다. [Source: _bmad-output/planning-artifacts/architecture.md#Action Response Formats]
- 모든 민감 query/action은 shared authorization helper를 먼저 호출한다. 설정 관리는 `PermissionAction.SETTINGS_MANAGE` 기반의 `requireSettingsAccess()`가 경계다. [Source: _bmad-output/planning-artifacts/architecture.md#Authorization Pattern]
- business-changing action은 같은 transaction 안에서 business write와 audit write를 처리한다. [Source: _bmad-output/planning-artifacts/architecture.md#Audit Pattern]
- Next.js `revalidatePath`는 server 환경에서 호출해야 한다. 현재 action 파일처럼 Server Action에서 호출하고 Client Component로 옮기지 않는다. [Source: https://nextjs.org/docs/app/api-reference/functions/revalidatePath]
- Prisma read-modify-write는 atomicity가 필요하면 transaction 안에 둔다. 현 store action의 duplicate check, update/create, audit write transaction 구조를 유지한다. [Source: https://www.prisma.io/docs/orm/prisma-client/queries/transactions]
- Playwright e2e는 접근성 locator(`getByRole`, `getByLabel`, `getByText`) 중심 패턴을 유지한다. [Source: https://playwright.dev/docs/locators]

### Data and UX Requirements

- 지점 생성은 항상 활성 상태로 시작한다. 생성 dialog에 비활성 선택을 추가하지 않는다.
- 지점명 수정 dialog에서는 기존 `name`, `isActive` 편집 흐름을 유지한다.
- row-level 활성 상태 변경은 이름 수정과 분리된 `updateStoreStatus`를 사용한다.
- 비활성 지점은 master-data list에서는 볼 수 있어야 하지만 신규 장부/지점장 작업 선택지에서는 제외되어야 한다.
- 과거 장부, 감사 로그, 지점 배정은 지점 비활성화 후에도 보존한다.
- 중복명/검증 오류는 해당 field와 연결된 한국어 오류로 표시하고 첫 오류 input에 focus를 유지한다.
