# Story 5.3: 본사가 선택 기간의 지점별 실적을 비교한다

Status: ready-for-dev

## Story

As a 본사 사용자,  
I want 선택 기간의 지점별 실적을 비교하고 싶다,  
so that 어느 지점의 매출, 이익, 생산성, 재고 흐름이 좋은지 운영 기준으로 판단할 수 있다.

## Acceptance Criteria

1. **Given** 본사 사용자가 기간 비교 리포트 화면에 접근할 때  
   **When** 시작일과 종료일을 선택하면  
   **Then** 선택 기간의 지점별 실적 비교 표가 표시되어야 한다  
   **And** 기간 필터는 URL 상태로 유지되어 새로고침해도 같은 조건을 볼 수 있어야 한다.

2. **Given** 기간 비교 표가 표시될 때  
   **When** 지점별 데이터가 로드되면  
   **Then** 매출, 매출이익, 이익률, 영업이익, 인당생산성, 평균재고, 평균매출, 매출대비 재고비율, 손실이 표시되어야 한다.

3. **Given** 선택 기간에 정정 기록이 있는 장부가 포함될 때  
   **When** 기간 비교 지표가 계산되면  
   **Then** 기본 숫자는 정정 반영값을 사용해야 한다  
   **And** 원본값과 정정 반영값 차이가 있는 지표는 Story 5.2의 상세 확인 경로를 제공해야 한다.

4. **Given** 선택 기간에 미입력, 입력중, 검토대기, 본사마감, 휴무일 데이터가 섞여 있을 때  
   **When** 지표가 집계되면  
   **Then** 각 상태는 구분되어 집계 또는 표시되어야 한다  
   **And** 미입력과 휴무일을 같은 값으로 취급하면 안 된다.

5. **Given** 평균재고, 평균매출, 재고비율을 계산할 때  
   **When** 필요한 일별 데이터가 부족하거나 평균매출이 0이면  
   **Then** 나누기 오류 대신 "계산 불가" 또는 "데이터 부족" 상태가 표시되어야 한다.

6. **Given** 기간 비교 계산이 수행될 때  
   **When** 리포트 지표를 계산하면  
   **Then** shared server calculation을 사용해야 한다  
   **And** 관제판이나 월간 리포트와 다른 UI 전용 계산식을 만들면 안 된다.

7. **Given** 본사 사용자가 비교 표를 볼 때  
   **When** 숫자 컬럼이 표시되면  
   **Then** 금액은 원 단위와 천 단위 콤마로, 이익률은 소수 첫째 자리까지 표시되어야 한다  
   **And** 숫자 데이터는 tabular-nums로 정렬되어야 한다.

8. **Given** 좁은 화면에서 기간 비교 리포트를 볼 때  
   **When** 전체 컬럼을 표시하기 어렵다면  
   **Then** 핵심 컬럼을 우선 표시하고 나머지는 가로 스크롤 또는 상세로 확인할 수 있어야 한다.

## Dev Notes

### 핵심 목표

- Story 5.3은 일별 아침 회의 리포트(5.1)와 숫자 근거/정정 추적(5.2)을 기간 단위 지점 비교로 확장한다.
- 기본 표시 숫자는 정정 반영값이다. 원본값과 정정 반영값의 차이는 Story 5.2의 evidence 패턴으로 추적 가능해야 한다.
- 이 story는 월간 요약(5.4)이나 월간 손실/재고 흐름(5.5)을 미리 만들지 않는다. 다만 5.4/5.5가 재사용할 수 있는 기간 집계 helper는 `reports/queries.ts` 내부에서 작게 분리할 수 있다.

### 현재 구현 상태

- `src/app/app/reports/daily/page.tsx`와 `loading.tsx`는 존재한다. `comparison` route는 아직 없다.
- `src/features/reports/queries.ts`는 일별 active store + ledger 조회, KST 날짜 helper, 정정 batch 조회, shared calculation, metric evidence 생성 패턴을 이미 갖고 있다.
- `src/features/reports/types.ts`는 `DailyMeetingReportMetricEvidence`와 `DailyMeetingReportRow`를 갖고 있다. 기간 비교는 별도 타입을 추가해 일별 row 타입을 억지로 오염시키지 않는다.
- `src/features/reports/components/daily-meeting-report-table.tsx`는 데스크탑 표, 모바일 카드, `details` 기반 근거 보기, `tabular-nums`, `break-words` 패턴을 제공한다.
- `src/features/corrections/actions.ts`는 5.2에서 `/app/reports/daily` revalidation을 추가했다. 비교 리포트를 만들면 `/app/reports/comparison`도 stale하지 않게 추가해야 한다.

### 구현 가드레일

1. 본사 전용 page/query는 반드시 `requireHeadquartersUser()`를 호출한다. UI 메뉴 숨김은 보안 경계가 아니다.
2. 이 story에서 public API route, tRPC, client-side fetch layer를 새로 만들지 않는다. 읽기는 Server Component + feature query helper 패턴을 따른다.
3. 새 Prisma 모델이나 migration을 만들지 않는다. `Store`, `DailyLedger`, `CorrectionRecord`, ledger detail relations를 조회해 계산한다.
4. 기간 비교 row를 위해 누락 ledger를 생성하지 않는다. ledger 없음은 미입력 상태로 계산한다.
5. 최신 정정값 선택 규칙은 기존 correction query 계약인 `createdAt desc, id desc`를 따른다.
6. 금액은 integer KRW 상태로 계산하고 UI에서만 원 단위 포맷을 적용한다.
7. 날짜는 KST 운영일 helper를 사용한다. 브라우저 로컬 날짜 계산으로 기준일을 바꾸지 않는다.
8. 미입력은 ledger 없음, 휴무는 `DailyLedger.status === "HOLIDAY"`, 0은 실제 값 0이다. 세 상태를 같은 "계산 불가"로 뭉개지 않는다.
9. 리포트, 관제판, 장부 상세가 서로 다른 계산식을 갖게 만들면 안 된다.
10. Story 5.2는 sprint status상 아직 `review`다. dev agent는 5.2 구현 파일을 현재 작업 트리 기준으로 읽되, 리뷰에서 바뀐 내용이 있으면 그 결과를 우선한다.

### UX and Accessibility Notes

- 본사 리포트는 업무 표면이다. 설명용 hero나 큰 카드 레이아웃을 만들지 말고 조밀하지만 읽기 쉬운 표를 우선한다.
- 비교 표는 지점별 스캔이 목적이다. 데스크탑에서는 지점명, 상태 카운트, 매출, 이익률, 영업이익이 먼저 보여야 한다.
- 모바일에서는 전체 9개 지표를 한 화면에 억지로 밀어 넣지 않는다. 핵심 지표 우선, 나머지는 가로 스크롤 또는 details 영역을 사용한다.
- 숫자는 `tabular-nums`를 유지하고, 금액/비율 포맷은 기존 일별 리포트와 일관되게 한다.
- 정정 반영, 계산 불가, 데이터 부족, 손실 있음/없음은 색상 단독 전달 금지다. 텍스트 레이블을 함께 둔다.
- 긴 지점명, 계산 불가 사유, 정정 상태 텍스트가 버튼이나 숫자를 덮지 않게 responsive wrapping을 적용한다.
