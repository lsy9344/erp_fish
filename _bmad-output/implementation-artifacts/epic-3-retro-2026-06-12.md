## Epic 요약

Epic 3은 승인된 기본 계산을 서버 기준으로 정리하고, 저장/제출 검증, 검토 화면 요약, 정책 미정 계산 차단을 구축했다. 구현 결과는 FR-13, FR-14, NFR10, NFR11, UX-DR12, UX-DR13의 핵심 방향과 맞는다.

완료된 핵심 기반:

- `CalculationStatus`와 `LedgerReviewMetric`으로 `정상`, `데이터 부족`, `확인 필요`, `계산 불가` 상태를 분리
- 장부 검토, 관제판, 리포트가 `calculateLedgerReviewSummary`를 재사용하도록 정적 회귀 테스트 추가
- 서버 검증 helper를 공통화하고 field error path를 단계 UI와 연결
- 권한 밖 store 요청이 상세 field validation보다 먼저 거부되도록 store access 최소 파싱 후 `requireStoreAccess()` 실행
- 검토 화면의 6단계 `stepSummaries`와 결제 차액 부호 표시 추가
- 지점장 검토 응답의 metric allowlist와 shared `omitSensitiveFields()` 강화
- OQ-1, OQ-2, OQ-7, OQ-9, OQ-10A, OQ-14, OQ-17 관련 계산 gate registry 추가

## 잘된 점

1. 계산 상태가 제품 언어로 정리됐다.
   - Story 3.1에서 `data-insufficient`, `policy-unconfirmed`, `calculation-unavailable`을 분리했다.
   - `0원`, 빈 값, 오래된 값으로 계산 실패를 숨기는 위험을 테스트로 막았다.

2. 검증과 계산 상태를 섞지 않았다.
   - Story 3.2는 field validation을 강화하면서도 계산 상태를 필수 입력 오류로 합치지 않았다.
   - 검토 화면은 "입력해야 하는 것"과 "계산 기준/데이터 상태"를 병렬로 보여준다.

3. 권한 우선 검증이 실제 보안 요구에 맞게 정리됐다.
   - store-scoped action은 `storeId`만 최소 파싱하고 권한을 먼저 확인한 뒤 상세 Zod schema를 실행한다.
   - 권한 밖 요청에 상세 field error나 장부 스냅샷이 섞여 나가는 위험을 줄였다.

4. 지점장 민감 지표 차단이 한 번 더 강화됐다.
   - Story 3.3 review에서 `stepSummaries.metrics` allowlist가 추가됐다.
   - Story 3.4 review에서 exact key만 제거하던 shared helper가 파생 key와 OQ-gated key variant까지 제거하도록 강화됐다.

5. OQ-gated 계산이 구현 흐름에 섞이지 않도록 guardrail이 생겼다.
   - `src/server/calculations/policy-gates.ts`가 metric id, label, reason, OQ id를 구조화했다.
   - dashboard/report/review query에 `30%`, FIFO, 희망 판매가 손실액 같은 직접 수식이 들어오면 테스트가 실패한다.

## 어려웠던 점

1. 정책 미정과 기본 운영 확인값의 경계가 섬세했다.
   - 저장된 운영 단가 기반 기본 확인값은 유지해야 했지만 FIFO 확정 원가처럼 보이면 안 됐다.
   - 해결은 명명, reason, policy gate metadata, 테스트를 함께 둔 것이다.

2. 검토 화면 확장은 민감 응답 회귀 위험이 컸다.
   - 새 `stepSummaries`가 안전한 metric만 만든다는 사실에 의존하면 이후 확장에서 민감 지표가 새로 새어 나갈 수 있었다.
   - response shaping에서 positive allowlist를 적용해 구조적으로 막았다.

3. 서버 action 순서가 문서 표현보다 더 구체적으로 진화했다.
   - 아키텍처 문서는 "schema validation 후 authorization-sensitive logic"처럼 읽혔다.
   - 구현은 권한 밖 데이터 노출 방지를 위해 `storeId` 최소 파싱, authorization, 상세 validation 순서를 택했다.

4. E2E 실행 환경 제약이 반복됐다.
   - Story 3.1부터 3.4까지 Playwright webServer가 `127.0.0.1:3000` bind 단계에서 `EPERM`으로 차단됐다.
   - unit, typecheck, lint는 통과했지만 브라우저 회귀는 실행 환경 보강이 필요하다.

## 핵심 학습

1. ERP Fish의 계산 source of truth는 UI나 query별 helper가 아니라 shared server calculation module이어야 한다.
2. 정책 미정은 제품 상태다. 개발 편의를 위한 임시 숫자로 대체하면 본사 관제와 리포트 신뢰를 해친다.
3. 권한 밖 요청은 상세 validation보다 authorization을 먼저 통과해야 한다. 단, 이를 위해 필요한 최소 identifier parsing은 허용된다.
4. 지점장 민감 지표 차단은 "만들 때 안전한 데이터만 넣기"와 "응답 shaping에서 allowlist로 제거하기"가 함께 필요하다.
5. 정적 회귀 테스트는 이 프로젝트에 잘 맞는다. 계산 수식, 민감 key, action 순서를 빠르게 지키는 안전망이 됐다.

## Epic 4 준비 평가

Epic 4는 본사 관제, 보완 수정, ClosePreflight, 본사 마감, 정정 기록을 다룬다. Epic 3 결과는 Epic 4의 직접 선행 조건이다.

준비된 의존성:

- 관제판과 리포트가 재사용할 `calculateLedgerReviewSummary`
- `기준 확인 필요`, `데이터 부족`, `계산 불가` 상태 모델
- OQ-gated 계산 registry와 정책 미정 metadata
- store manager 민감 응답 차단 helper와 review metric allowlist
- field-level validation error와 접근성 연결 패턴
- version/edit token 기반 저장 충돌 흐름과 검토 제출 경계

주의할 의존성:

- Epic 4 관제판의 이상 신호는 OQ-1과 threshold 설정 상태를 확정 이상으로 오해하면 안 된다.
- ClosePreflight는 필수 누락과 OQ-gated `기준 확인 필요`를 구분해야 한다.
- 본사 보완 수정은 store manager action 흐름을 단순 재사용하지 말고 본사 action permission, audit reason, version guard를 함께 적용해야 한다.
- 정정 반영값은 이미 계산 overlay helper가 있으므로, 관제판과 리포트에서 원본값과 정정 반영값의 기준을 혼동하지 않아야 한다.

4. 아키텍처 문서는 권한 우선 검증 흐름을 반영해 수정한다.

## Readiness Assessment

- Testing & Quality: unit, typecheck, lint 검증은 story별로 반복 수행됐다. E2E는 실행 환경의 port bind 제약으로 차단됐다.
- Deployment: production deploy 자체는 Epic 3 범위가 아니다. 계산/검증 코드는 서버 경계와 테스트 기준으로 준비됐다.
- Stakeholder Acceptance: 자동 회고이므로 직접 stakeholder acceptance는 확인하지 않았다. story status와 review 기록 기준으로 완료 판단했다.
- Technical Health: shared calculation, validation, response shaping, policy gate가 강화됐다. 가장 큰 잔여 리스크는 Epic 4가 이를 우회해 별도 계산/상태 표현을 만드는 것이다.
- Unresolved Blockers: Epic 4 시작을 막는 코드 blocker는 확인되지 않았다. 문서 blocker는 아키텍처 validation/data flow 설명의 순서 불일치다.

## Significant Discovery Check

Epic 3 구현 과정에서 Epic 4의 제품 방향을 바꿀 수준의 발견은 없었다. 다만 구현 방식의 중요한 세부 결정이 확인됐다.

- 권한 밖 store 요청은 상세 입력값 검증보다 먼저 authorization으로 차단해야 한다.
- OQ-gated 계산은 본사 화면에서도 확정값처럼 제공하면 안 되며, OQ id와 정책 질문을 구조화해야 한다.
- 지점장 응답은 safe construction만 믿지 말고 response shaping allowlist로 다시 제한해야 한다.

이 발견은 Epic 4 계획을 재작성할 정도는 아니지만, ClosePreflight, dashboard signal, HQ edit, close/correction story의 구현 기준으로 반영해야 한다.
