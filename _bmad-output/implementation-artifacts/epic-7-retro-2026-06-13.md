## 잘된 점

1. 정책-only 경계가 반복적으로 강화됐다.
   - 각 story는 산출물 작성, 승인자/승인 상태, 후속 구현 승격 조건을 남겼다.
   - Story 7.3부터 7.7까지 review가 범위 밖 `src/`/`tests/` drift를 발견하고 제거했다.

2. 후속 구현을 위한 조건부 승격 기준이 구체화됐다.
   - 매출차액, `30%단가`, FIFO, 희망 판매가 손실액, 민감 필드, 재고 용어 변경 모두 구현 가능 조건과 금지 조건을 분리했다.
   - Epic 8 story가 참조할 수 있는 파일 경로와 code surface 후보가 story별로 남았다.

3. 기존 architecture guardrail과 충돌하지 않았다.
   - shared server calculation, server authorization, audit, correction, policy gate 원칙을 유지했다.
   - 승인 전에는 `기준 확인 필요`, `데이터 부족`, `계산 불가` 상태를 숫자나 확정 계산으로 승격하지 않았다.

4. 지점장 민감 필드 차단 기준이 더 선명해졌다.
   - Story 7.6은 화면, 서버 응답, export, 공유 링크, cache, 알림 템플릿을 한 matrix로 다뤘다.
   - UI 숨김만으로는 승인 기준을 만족하지 않는다는 원칙이 반복 확인됐다.

5. Epic 6 회고의 학습이 적용됐다.
   - 정책 미정 지표를 숫자로 승격하지 않는다는 Epic 6 action item이 Epic 7 전체에 반영됐다.
   - 문서 업데이트는 검증된 불일치만 반영한다는 기준도 유지됐다.

## 어려웠던 점

1. 정책 산출물과 구현 착수를 계속 분리해야 했다.
   - 여러 story에서 신규 unit/e2e test 또는 source 변경 drift가 발생했고, review에서 제거됐다.
   - 이 패턴은 자동화가 "검증을 위해 코드를 추가한다"는 방향으로 과잉 보정될 수 있음을 보여준다.

2. 승인 대기 상태와 정책 결정 내용을 동시에 표현해야 했다.
   - 산출물은 권장 정책과 예시를 포함하지만, 실제 PM/본사 운영자 승인 전에는 OQ가 완전히 닫혔다고 주장하면 안 된다.
   - 후속 story 생성자는 "정책 초안 존재"와 "승인 완료"를 구분해야 한다.

3. 용어 충돌 위험이 반복됐다.
   - `차이`는 재고 차이와 매출 차이에 모두 걸칠 수 있다.
   - `당일 판매량`은 실제 POS 판매 수량으로 오해될 수 있어 도움말과 계산식 보존 원칙이 필요했다.

4. 민감 지표의 파생 가능성이 넓었다.
   - `30%단가`, 희망 판매가, FIFO lot, 재고금액, 금액 기반 차이는 모두 원가/마진 역추정 가능성이 있다.
   - 단일 화면 기준이 아니라 API/export/cache/알림까지 같이 막아야 한다.

5. Playwright E2E 실행 환경 제약이 반복됐다.
   - 여러 story record에서 Next.js/Playwright webServer가 `127.0.0.1` bind 시 `listen EPERM`으로 중단됐다.
   - 문서-only story에서는 최종 source/test drift가 제거됐지만, 브라우저 경계 검증은 포트 바인딩 가능한 환경에서 계속 별도 확인 대상이다.

## 핵심 학습

1. Discovery epic도 review가 필요하다. 정책 문서만 만드는 작업에서도 테스트 파일, test-summary, story status drift가 생길 수 있다.
2. 승인 상태는 기능 상태와 다르다. 정책 산출물이 있어도 승인 대기면 구현 story는 조건부 또는 차단 상태로 남아야 한다.
3. 민감 필드 차단은 필드명만 지우는 문제가 아니다. 실제 seeded value, forbidden payload, export body, cache key/value, 알림 변수까지 포함한다.
4. 정책 미정 항목은 0, 빈 값, 임시 계산으로 대체하지 않는다. 명시 상태를 유지해야 나중에 숫자 신뢰가 깨지지 않는다.
5. Epic 8은 Epic 7 산출물을 바로 구현 입력으로 쓰기보다, 필요한 승인 증거를 먼저 확인해야 한다.

## 이전 회고 연속성

Epic 6 회고의 action item은 Epic 7에서 다음처럼 반영됐다.

- Epic 7 discovery story 산출물은 구현 가능한 정책과 계속 차단해야 할 정책을 분리한다: 완료. 모든 Story 7.x 산출물에 승격 가능 여부와 조건이 있다.
- E2E 실행 가능 환경에서 브라우저 흐름을 재검증한다: 진행 필요. Epic 7은 문서-only라 최종 source/test 변경은 없지만, 환경 제약 자체는 남아 있다.
- OQ-gated report/export 상태를 숫자로 승격하지 않는다: 완료. 정책 산출물은 승인 전 gate 유지와 `기준 확인 필요` 유지 원칙을 반복한다.
- Export route/helper 경계를 유지한다: 해당 없음. Epic 7은 export code를 변경하지 않았다.
- 권한 밖 `storeId` fallback 없이 명시적 차단 또는 no-row를 유지한다: 해당 없음. Epic 7은 권한 정책 문서만 보강했다.

## Epic 8 준비 평가

Epic 8은 승인 추가 구현 백로그 정렬 track이며, Epic 7의 정책 산출물을 많이 참조한다. 다만 현재 sprint-status 기준 Epic 8 story는 모두 `backlog`이고, Epic 7 산출물 대부분은 "승인 대기" 상태다.

준비된 의존성:

- Story 8.2는 Story 7.3의 품목 정규화 기준을 참조할 수 있다.
- Story 8.3은 Story 7.4의 FIFO 적용 범위와 처리 순서 산출물을 참조할 수 있다.
- Story 8.4는 Story 7.5와 7.6의 희망 판매가 손실액/민감 필드 matrix를 참조할 수 있다.
- Story 8.5는 Story 7.7의 재고 용어 정책을 참조할 수 있다.
- 모든 Epic 8 story는 승인 증거와 구현 승격 조건을 다시 확인해야 한다.

주의할 의존성:

- OQ-10B, OQ-8, OQ-15, OQ-11, OQ-12, OQ-13, OQ-16, OQ-18은 Epic 7에서 닫히지 않았다.
- Epic 7 산출물의 "조건부 구현 가능"은 formal approval 없이 자동 구현 가능을 뜻하지 않는다.
- Epic 8은 구현 착수보다 승인 정리와 backlog 경계 확정이 먼저다.

## 결정 사항

1. Epic 7은 story 기준 완료로 판단한다.
   - sprint-status의 Epic 7 story 7개가 모두 `done`이고, 각 story file에 senior review 또는 검증 기록이 남아 있다.

2. `epic-7-retrospective`를 완료로 전환한다.
   - retrospective artifact를 생성하고 sprint-status를 `done`으로 갱신한다.

3. Epic 8 계획은 유지하되, 구현 착수 조건을 강화한다.
   - Epic 8은 Epic 7 정책 산출물을 참조할 수 있지만, 승인 대기 항목은 구현 story로 자동 승격하지 않는다.

4. 문서 업데이트는 검증된 불일치만 반영한다.
   - Epic 7은 app source, Prisma schema, tests를 최종 변경하지 않은 정책 epic이므로, code와 기존 architecture/PRD/README/config가 실제로 어긋나는지 먼저 확인한다.
