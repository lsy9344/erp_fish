# Story 5.5: 본사가 월간 손실/재고 흐름과 핵심 성과를 본다

Status: done

## Story

As a 본사 사용자,  
I want 월간 손실/재고 흐름과 핵심 성과 지표를 보고 싶다,  
so that 한 지점의 월간 운영 품질과 재고 흐름을 요약해서 판단할 수 있다.

## Acceptance Criteria

1. **Given** 본사 사용자가 월간 요약 리포트에서 월과 지점을 선택할 때  
   **When** 데이터가 로드되면  
   **Then** 월간 매출, 매출이익, 이익률, 영업이익, 손실 합계, 평균재고, 평균매출, 매출대비 재고비율이 표시되어야 한다.

2. **Given** 월간 손실/재고 흐름이 표시될 때  
   **When** 해당 월의 장부 데이터를 집계하면  
   **Then** 손실/폐기/떨이 합계와 재고 흐름 요약이 표시되어야 한다  
   **And** 손실 유형별 구분이 가능한 경우 유형별 요약을 볼 수 있어야 한다.

3. **Given** 월간 최고매출품목을 계산할 때  
   **When** 상품별 판매금액 데이터가 충분히 존재하면  
   **Then** 최고매출품목과 최고매출품목 매출액이 표시되어야 한다  
   **And** 동률이 있으면 일관된 tie-breaking 규칙 또는 "공동" 표시를 사용해야 한다.

4. **Given** 최고매출품목 계산에 필요한 상품별 판매금액 산출 기준이 미정이거나 데이터가 부족할 때  
   **When** 월간 요약이 표시되면  
   **Then** 임의로 품목을 선택하지 않아야 한다  
   **And** "계산 기준 확인 필요" 또는 "데이터 부족" 상태를 표시해야 한다.

5. **Given** 월간 평균재고, 평균매출, 매출대비 재고비율을 계산할 때  
   **When** 마감되지 않은 장부, 미입력, 휴무일이 포함되어 있으면  
   **Then** 어떤 일자가 계산에 포함 또는 제외됐는지 확인할 수 있어야 한다  
   **And** 평균매출이 0이면 나누기 오류 대신 "계산 불가" 상태를 표시해야 한다.

6. **Given** 월간 지표에 정정 기록이 반영될 때  
   **When** 월간 요약 숫자가 계산되면  
   **Then** 기본 숫자는 정정 반영값을 사용해야 한다  
   **And** 원본값과 정정 반영값 차이가 있는 항목은 Story 5.2의 상세 확인 경로를 제공해야 한다.

7. **Given** 월간 리포트 계산이 수행될 때  
   **When** 손실, 재고, 최고매출품목, 평균 지표를 계산하면  
   **Then** shared server calculation을 사용해야 한다  
   **And** 기간 비교 리포트와 다른 UI 전용 계산식을 만들면 안 된다.

8. **Given** 월간 요약을 좁은 화면에서 볼 때  
   **When** 핵심 성과와 손실/재고 흐름이 표시되면  
   **Then** 주요 숫자는 겹치지 않고 읽을 수 있어야 한다  
   **And** 세부 표는 가로 스크롤 또는 상세 영역으로 확인할 수 있어야 한다.

## Dev Notes

### 핵심 목표

- Story 5.5는 Story 5.4가 만든 월간 요약 화면을 확장해 월간 핵심 성과, 손실/폐기/떨이, 재고 흐름을 한 지점 기준으로 보여준다.

- 최고매출품목은 현재 데이터 모델에서 산출 기준이 부족할 수 있다. 부족하면 임의 계산보다 `계산 기준 확인 필요`가 정답이다.

### Architecture and Implementation Guardrails

1. 리포트 읽기는 Server Component + feature query helper 패턴을 따른다. public API route, tRPC, client-side fetch layer를 새로 만들지 않는다.
2. 모든 리포트 query는 서버에서 `requireHeadquartersUser()` 또는 같은 권한 helper로 본사 권한을 확인한다.
3. Prisma schema는 이 story에서 바꾸지 않는다. 새 모델이나 migration 없이 기존 `DailyLedger`, `LedgerInventoryItem`, `LedgerInventoryAdjustment`, `LedgerLossItem`, `CorrectionRecord`, `Store`를 사용한다.
4. `Store.storeId + DailyLedger.closingDate` 범위 조회와 기존 relation select를 사용한다. 월간 누락 날짜를 위해 `DailyLedger`를 생성하지 않는다.
5. 정정 overlay는 `getLatestCorrectionValuesForLedgers()`와 `applyCorrectionValuesToLedgerReviewInput()`를 사용한다. 정정 적용 순서는 기존 correction query 계약인 `createdAt desc, id desc`를 따른다.
6. 계산은 서버에서 integer KRW로 수행하고 UI에서는 `Intl.NumberFormat("ko-KR")` 등 기존 포맷만 적용한다.
7. 평균재고와 평균매출은 기간 비교 리포트와 같은 기준을 사용한다. 휴무는 business aggregate에서 제외하고, 미입력은 ledger가 없어 제외되되 포함/제외 상태를 사용자가 확인할 수 있어야 한다.
8. 평균매출이 0이거나 분모가 없으면 `계산 불가`를 표시한다. `Infinity`, `NaN`, `0으로 대체`는 금지다.
9. 최고매출품목은 상품별 판매금액 source가 없으면 `계산 기준 확인 필요` 또는 `데이터 부족`이다. 재고금액, 손실금액, 매입금액을 최고매출품목의 대체 source로 쓰지 않는다.
10. 손실 유형별 요약은 `ledgerLossItems.lossTypeName`을 사용한다. 손실 코드 관리 명칭이 변경되어도 historical row의 저장된 이름을 유지한다.
11. 기존 dirty worktree가 있을 수 있다. 관련 없는 변경을 되돌리지 않는다. 특히 `store-comparison-report-table.tsx`, E2E global setup, 5.3 story 문서가 dirty일 수 있으니 편집 전 읽고 현재 상태 위에서 작업한다.
