# Story 5.2: 품목 마스터 기본 관리

Status: done

## Story

As a 본사 설정 관리자,  
I want 품목명, 구분, 규격, 기본 단가를 관리할 수 있기를 원한다,  
so that 장부 입력자가 일관된 품목 기준을 참조할 수 있다.

## Acceptance Criteria

1. **Given** 설정 관리자 권한을 가진 본사 사용자가 품목 관리 화면에 접근할 때, **When** 품목 목록을 조회한다, **Then** 품목명, 구분, 규격, 기본 단가, 활성/비활성 상태를 볼 수 있어야 한다, **And** 권한 없는 사용자는 품목 관리 mutation을 수행할 수 없어야 한다.
2. **Given** 설정 관리자가 새 품목을 추가할 때, **When** 필수 값이 유효하다, **Then** 새 품목이 생성되어야 한다, **And** 생성 이벤트는 감사 로그에 남아야 한다.
3. **Given** 설정 관리자가 품목의 기본 정보를 수정할 때, **When** 저장 요청이 성공한다, **Then** 신규 장부 입력 선택지에는 수정된 값이 반영되어야 한다, **And** 과거 장부에 저장된 원본 품목명/규격/단가를 임의로 바꾸면 안 된다.
4. **Given** 설정 관리자가 품목을 비활성화할 때, **When** 저장 요청이 성공한다, **Then** 비활성 품목은 신규 선택지에서 제외되어야 한다, **And** 과거 장부에서는 원본 표시가 유지되어야 한다.
5. **Given** OQ-3 품목명/구분/규격 정규화 기준이 아직 닫히지 않았을 때, **When** 이 스토리를 구현한다, **Then** 초기 엑셀 데이터 정규화나 자동 매핑 확정 기능은 구현하지 않아야 한다, **And** 기본 CRUD와 원문 보존 구조에 집중해야 한다.
6. **Given** 품목 기본 단가가 입력될 때, **When** 서버가 값을 검증한다, **Then** 금액은 integer KRW 기준으로 저장되어야 한다, **And** 잘못된 숫자나 음수는 거부되어야 한다.

## Dev Notes

### 현재 구현 상태

- `Product` 모델은 이미 `id`, `name`, `category`, `spec`, `defaultUnitPrice`, `isActive`, `createdAt`, `updatedAt`, `updatedById`와 `@@unique([name, category, spec])`를 갖는다. 새 product table migration을 다시 만들지 않는다. [Source: `prisma/schema.prisma#model Product`]
- `Product`는 `PurchaseStandard`, `LedgerPurchaseItem`, `LedgerInventoryItem`, `LedgerInventoryAdjustment`, `LedgerLossItem`, `InventoryOpeningSnapshot`과 relation을 가진다. 이 relation 때문에 hard delete는 위험하며, 비활성화가 기본 운영 방식이다. [Source: `prisma/schema.prisma#model Product`]
- 장부 라인들은 `productName`, `productCategory`, `productSpec`, `unitPrice` snapshot을 별도 저장한다. 품목 마스터 수정은 이 snapshot들을 소급 update하면 안 된다. [Source: `prisma/schema.prisma#model LedgerPurchaseItem`] [Source: `prisma/schema.prisma#model LedgerInventoryItem`] [Source: `prisma/schema.prisma#model LedgerLossItem`]
- `src/features/master-data/product-actions.ts`에는 `createProduct`, `updateProduct`, `updateProductStatus`가 이미 있고 모두 `requireSettingsAccess()`, Prisma transaction, `writeAuditLog`, `ActionResult`, route revalidation을 사용한다. 이 파일을 확장하고 새 action family를 만들지 않는다. [Source: `src/features/master-data/product-actions.ts`]
- `src/features/master-data/product-queries.ts`의 `getProductsForHeadquarters()`는 settings 권한으로 목록을 조회하고, `getActiveProductOptions()`는 활성 품목만 신규 입력 선택지로 반환한다. [Source: `src/features/master-data/product-queries.ts`]
- `src/features/master-data/product-schemas.ts`는 `PRODUCT_CATEGORY_VALUES = ["냉동", "생물"]`, trim, KRW integer 검증, 한국어 field error를 이미 정의한다. OQ-3이 닫히기 전에는 이 범위를 임의 확장하지 않는다. [Source: `src/features/master-data/product-schemas.ts`]
- `src/features/master-data/components/product-management-client.tsx`는 품목 목록, 검색/구분/상태 필터, 생성/수정 dialog, row-level 상태 적용, field error focus를 제공한다. UI를 새로 만들기보다 이 컴포넌트를 점검/보강한다. [Source: `src/features/master-data/components/product-management-client.tsx`]
- `/app/master-data/products` route와 sidebar 항목은 이미 있다. route는 `requireSettingsAccess()`, `getHeadquartersNavigationItems()`, `HeadquartersShell`, `PageHeader`, `ProductManagementClient` 조합을 유지한다. [Source: `src/app/app/master-data/products/page.tsx`] [Source: `src/components/app-sidebar.tsx`]
- 기존 product 테스트는 `master-data-products`에 있으나 일부 문자열과 assertion 설명이 Story 1.5/`스토리15` 시절을 가리킨다. Story 5.2 구현 시 테스트 이름과 fixture prefix가 개발자를 오도하지 않게 정리하는 것이 좋다. [Source: `tests/unit/master-data-products.test.mjs`] [Source: `tests/e2e/master-data-products.spec.ts`]

### Architecture Guardrails

- ERP Fish는 Next.js App Router, Server Components, Server Actions, Prisma, PostgreSQL, Auth.js/NextAuth, Tailwind/shadcn UI를 사용한다. 이 story에서 새 API abstraction이나 public Route Handler를 추가하지 않는다. [Source: `_bmad-output/planning-artifacts/architecture.md#Technology Stack`]
- business code는 `src/features/master-data`, shared auth/audit는 `src/server`, UI primitive는 `src/components/ui`에 둔다. [Source: `_bmad-output/planning-artifacts/architecture.md#Project Structure & Boundaries`]
- 모든 민감 query/action은 shared authorization helper를 먼저 호출해야 한다. 설정 관리는 `PermissionAction.SETTINGS_MANAGE` 기반의 `requireSettingsAccess()`가 서버 경계다. [Source: `_bmad-output/planning-artifacts/architecture.md#Authorization Pattern`]
- expected validation/business error는 `ActionResult<T>`로 반환한다. field error는 UI에서 `aria-invalid`, `aria-describedby`, 첫 오류 focus와 연결한다. [Source: `_bmad-output/planning-artifacts/architecture.md#Action Response Formats`]
- business-changing action은 business write와 audit write를 같은 transaction에 넣는다. audit payload에는 변경 전/후 `name`, `category`, `spec`, `defaultUnitPrice`, `isActive`가 포함되어야 한다. [Source: `_bmad-output/planning-artifacts/architecture.md#Audit Pattern`]
- master-data 변경 뒤에는 dashboard, reports, store-entry가 stale 값을 보지 않도록 Next.js revalidation을 Server Action에서 수행한다. [Source: `_bmad-output/planning-artifacts/architecture.md#Caching Strategy`]
