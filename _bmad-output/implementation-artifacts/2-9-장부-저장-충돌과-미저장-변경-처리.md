# Story 2.9: 장부 저장 충돌과 미저장 변경 처리

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 지점장 또는 본사 사용자,
I want 같은 장부를 여러 사람이 다룰 때 내 입력이 조용히 덮어써지지 않기를 원한다,
so that 본사와 지점장이 함께 수정해도 장부 데이터 신뢰를 잃지 않는다.

## Acceptance Criteria

1. Given 사용자가 장부를 불러올 때, when 서버가 장부 데이터를 반환한다, then 장부 또는 입력 섹션의 version/edit token이 함께 제공되어야 한다, and 저장 요청은 마지막으로 읽은 token을 함께 보내야 한다.
2. Given 다른 사용자가 같은 섹션의 같은 필드를 먼저 저장했을 때, when 사용자가 오래된 token으로 저장을 시도한다, then 서버는 저장을 거부해야 한다, and 내 입력값, 서버 최신값, 마지막 수정자, 수정 시각을 비교할 수 있는 충돌 정보를 반환해야 한다.
3. Given 서로 다른 섹션의 변경이 충돌 없이 병합 가능한 경우, when 서버가 저장 요청을 처리한다, then 병합은 감사 로그를 남긴 뒤 허용할 수 있어야 한다, and 변경된 섹션, 필드, actor, 시각이 추적 가능해야 한다.
4. Given 본사가 강제 수정 중인 장부 또는 섹션이 있을 때, when 지점장이 해당 영역을 저장하려 한다, then 서버는 최신 상태 재확인을 요구해야 한다, and UI는 `본사 수정 중` 또는 동등한 상태를 표시할 수 있어야 한다.
5. Given 사용자가 단계 이동 또는 하단 탭 이동을 시도할 때, when 현재 단계에 미저장 변경이 있다, then 저장, 취소, 계속 편집 중 하나를 선택하게 해야 한다, and 선택 전에는 입력값을 버리거나 자동으로 덮어쓰면 안 된다.
6. Given 모바일 또는 브라우저 세션이 오래된 장부 상태를 가지고 있을 때, when 사용자가 저장을 시도한다, then 서버는 오래된 저장을 거부하고 최신 장부를 다시 불러오게 해야 한다, and 사용자가 어떤 값이 저장되지 않았는지 알 수 있어야 한다.

## Dev Notes

### Source Context

- Epic 2는 지점장 또는 본사가 하루 장부를 만들고 매출/비용/매입/재고/손실/근무 정보를 모바일 우선 7단계 흐름으로 입력하는 영역이다. [Source: `_bmad-output/planning-artifacts/epics.md#Epic 2: 지점 일일 장부 입력`]
- Story 2.9는 동시 편집과 저장 충돌을 다룬다. 조용한 마지막 저장 덮어쓰기를 막고 version/edit token, conflict 비교, 미저장 변경 guard를 요구한다. [Source: `_bmad-output/planning-artifacts/epics.md#Story 2.9: 장부 저장 충돌과 미저장 변경 처리`]
- PRD §3.2는 장부에 서버 version 또는 edit token을 두고, 부분 저장과 최종 제출이 마지막으로 읽은 token을 함께 보내야 한다고 정의한다. 같은 섹션 같은 필드 충돌은 사용자 비교/해결이 필요하고, 오래된 세션은 최신 장부를 다시 불러와야 한다. [Source: `_bmad-output/planning-artifacts/prds/prd-erp_fish-2026-05-28-2/prd.md#3.2 동시 편집과 저장 충돌`]
- PRD의 장부 상태 모델은 `검토 대기`를 원본 잠금으로 보지 않는다. `IN_REVIEW` 저장은 계속 허용하되 충돌 token과 audit은 유지해야 한다. [Source: `_bmad-output/planning-artifacts/prds/prd-erp_fish-2026-05-28-2/prd.md#3.1 장부 상태 모델`]
- UX는 `SaveConflictDialog`를 명시하고, 내 입력값/서버 최신값/마지막 수정자/수정 시각을 나란히 보여주라고 요구한다. [Source: `_bmad-output/planning-artifacts/ux-designs/ux-erp_fish-2026-05-28/EXPERIENCE.md#Component Patterns`]
- UX는 모바일 하단 탭 `장부`/`재고`/`손실` 이동 시 미저장 변경이 있으면 저장, 취소, 계속 편집 중 하나를 먼저 선택해야 한다고 정의한다. [Source: `_bmad-output/planning-artifacts/ux-designs/ux-erp_fish-2026-05-28/EXPERIENCE.md#Navigation Model`]
- 디자인 토큰은 저장 충돌 행에 warning border를 쓰고, 마지막 수정자와 시각을 값과 같은 시각 계층에 두라고 정의한다. [Source: `_bmad-output/planning-artifacts/ux-designs/ux-erp_fish-2026-05-28/DESIGN.md#Conflict Row State`]

### Validation Notes

- Checklist 재분석 결과 핵심 위험은 9가지다.
- Critical 1: 현재 지점장 경로는 `version`, 본사 경로는 `updatedAt`을 token처럼 사용한다. 두 경로가 같은 conflict response contract를 반환하지 않으면 AC 2/6 UI가 일관되게 구현되지 않는다.
- Critical 2: 현재 `mapLedgerConflictError`/`conflictError`는 대부분 generic refresh message다. AC 2가 요구하는 내 입력값, 서버 최신값, 마지막 수정자, 수정 시각이 없다.
- Critical 3: `DailyLedger.version`은 ledger-level이라 서로 다른 섹션 변경도 stale token으로 막힌다. 병합을 허용하려면 section/field diff와 audit가 필요하다. 구현하지 못하면 명시 거부가 안전하다.
- Critical 4: 본사 상세 shadcn `Tabs`는 미저장 변경 guard가 명확하지 않다. 탭 전환으로 draft가 사라지면 AC 5 위반이다.
- Critical 5: store-manager shell 하단 탭 `장부`/`재고`/`손실`은 step navigation 밖의 이동이다. 미저장 변경 guard 범위를 검증해야 한다.
- Critical 6: conflict payload는 서버 응답이므로 지점장에게 민감 회계 필드를 포함하면 보안 회귀다.
- Critical 7: submit/review conflict를 generic error로만 보여주면 사용자가 어떤 제출 상태가 저장되지 않았는지 알 수 없다.
- Critical 8: HQ close 중복 요청은 audit 1회 테스트가 있지만, stale close UI가 structured conflict/reload guidance를 보여주는지 별도 검증이 필요하다.
- Critical 9: conflict dialog에서 `내 값으로 다시 저장`을 제공할 경우 최신 token 재확인 없이 같은 payload를 반복하면 다시 last-write-wins 위험이 생긴다.
