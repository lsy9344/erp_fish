# Story 3.1: 기본 계산 모듈과 상태 표현

Status: done

## Story

As a 본사 사용자,
I want 승인된 기본 계산값이 서버 기준으로 일관되게 계산되기를 원한다,
so that 장부 상세, 관제판, 리포트가 서로 다른 숫자를 보여주지 않는다.

## Acceptance Criteria

1. **기본 계산 재사용**
   - **Given** 장부에 매출/결제, 비용, 근무인원 데이터가 저장되어 있을 때
   - **When** 서버 계산 모듈이 기본 계산을 수행한다
   - **Then** 총매출, 결제수단 합계, 비용합계, 근무인원, 평균매출 같은 OQ 비의존 계산값을 반환해야 한다
   - **And** 같은 계산 함수는 장부 상세, 관제판, 리포트에서 재사용 가능해야 한다.

2. **누락값 상태**
   - **Given** 계산에 필요한 입력값이 누락되었을 때
   - **When** 서버 계산 모듈이 결과를 만든다
   - **Then** 누락된 계산은 `데이터 부족` 상태로 반환해야 한다
   - **And** 0이나 이전 값을 조용히 대체하면 안 된다.

3. **정책 미정 상태**
   - **Given** 정책이 확정되지 않은 계산 항목이 있을 때
   - **When** 서버 계산 모듈이 결과를 만든다
   - **Then** 해당 항목은 `확인 필요` 상태로 반환해야 한다
   - **And** 확정 계산값처럼 표시되면 안 된다.

4. **계산 오류 상태**
   - **Given** 계산 중 예상하지 못한 오류가 발생했을 때
   - **When** 서버가 결과를 반환한다
   - **Then** 해당 계산은 `계산 불가` 상태로 반환되어야 한다
   - **And** 서버에는 디버깅 가능한 오류 로그가 남아야 한다.

5. **KRW 값과 공통 포맷**
   - **Given** 금액 값이 표시될 때
   - **When** UI가 계산 결과를 렌더링한다
   - **Then** 내부 값은 integer KRW 기준이어야 한다
   - **And** 화면 표시 포맷은 공통 formatter를 통해 처리되어야 한다.

6. **테스트와 중복 방지**
   - **Given** 개발자가 계산 테스트를 작성할 때
   - **When** 기본 계산 함수의 unit test를 실행한다
   - **Then** 정상값, 누락값, 정책 미정, 오류 상태를 각각 검증해야 한다
   - **And** UI 전용 계산 복제 로직을 만들면 안 된다.

## Dev Notes

### 핵심 구현 방향

- 이 story는 새 UI 기능을 크게 만드는 작업이 아니라, 이미 생긴 계산 함수를 **공통 상태 모델로 정리하고 재사용 경계를 강화하는 작업**이다.
- 기존 계산 파일:
  - `src/server/calculations/ledger.ts`: `calculatePaymentDifference`, `calculateExpenseTotal`, `calculatePurchaseTotal`, `calculateGrossProfit`, `calculateProductivity`, `calculateLedgerReviewSummary`, 정정 overlay 적용 함수가 있다.
  - `src/server/calculations/inventory.ts`: 재고금액, 시스템 재고 수량, 손실 요약, 재고 조정 계산이 있다.
  - `src/server/calculations/anomaly.ts`: 이상 신호 상태와 계산 후보가 있다.
- 새 계산 로직을 UI 컴포넌트, dashboard query, report query 안에 흩뿌리지 말고 `src/server/calculations` 아래에서 시작한다. [Source: `_bmad-output/planning-artifacts/architecture.md` "Calculation Strategy", "File Structure Patterns"]

### 현재 코드 상태

- `calculateLedgerReviewSummary`는 현재 다음 값을 반환한다: `totalSales`, `costOfGoodsSold`, `grossProfit`, `grossMarginRate`, `operatingProfit`, `productivity`, `inventoryAmount`, `paymentDifference`, `salesDifference`.
- 현재 상태 표현은 `LedgerReviewMetric = { value: number | null; unavailableReason?: "계산 불가" | "계산 기준 확인 필요" }`로 제한되어 있다.
- 3.1의 요구에는 `데이터 부족`, `확인 필요`, `계산 불가`가 분리되어야 하므로, 기존 `unavailableReason` 문자열만으로 후속 story까지 끌고 가면 상태가 모호해진다.
- `tests/unit/ledger-review.test.mjs`는 이미 계산 정상값, salesDifference 정책 미정, divide-by-zero/hidden inventory 계산 불가, 지점장 민감 지표 차단을 검증한다. 이 테스트를 버리지 말고 확장한다.

### 반드시 보존할 동작

- 지점장 응답에는 민감 지표가 포함되면 안 된다. 기존 테스트는 `toStoreManagerLedgerReviewStepData`가 `totalSales`, `paymentDifference`만 반환함을 확인한다.
- 지점장 화면/API/export/알림에는 원가, 이익, 마진율, 재고금액, 원가 lot 근거, 타 지점 비교를 서버 응답부터 차단한다. [Source: `_bmad-output/planning-artifacts/ux-designs/ux-erp_fish-2026-05-28/EXPERIENCE.md` "SensitiveMetricGuard"]
- 검토 대기는 원본 잠금이 아니다. 본사 마감 후에만 원본 수정이 잠기며 정정 기록 흐름으로 넘어간다. 이 story에서 계산 상태를 바꾸면서 장부 상태 전이를 변경하지 않는다. [Source: `_bmad-output/planning-artifacts/architecture.md` "Important Handoff Notes"]
- 결제수단 합계 불일치는 현재 `getWarnings(summary.paymentDifference.value ?? 0)`로 검토 화면 경고가 된다. 상태 모델 변경 후에도 결제 차이값이 사라지면 안 된다.
- 기존 충돌 처리와 edit token 흐름을 건드리지 않는다. Story 2.9에서 확립된 저장 충돌 처리는 계산 helper와 별개다.

### 기본 계산 정책

- PRD의 1차 계산 기준은 운영 확인용 기본 계산값이다. FIFO 매입 lot, 잔여 수량, 재고 원가 근거가 도입되기 전에는 매출원가/재고금액/이익률/영업이익을 FIFO 기준 확정 원가처럼 표시하지 않는다. [Source: `_bmad-output/planning-artifacts/prds/prd-erp_fish-2026-05-28-2/prd.md` "4.3 계산과 검증"]
- MVP 단가 기준은 실제 장부 라인 단가, 업로드/본사 작성 매입 단가, 유효한 품목 마스터 단가, 없음 순이다. 기준이 없으면 0 대체가 아니라 `기준 확인 필요`다. [Source: `_bmad-output/planning-artifacts/prds/prd-erp_fish-2026-05-28-2/prd.md` "MVP 단가/가격 기준"]
- OQ-1 매출차액 기준, OQ-2 `30%단가`, OQ-7 FIFO, OQ-9 희망 판매가 손실액, OQ-14 `차이` 의미 변경, OQ-17 FIFO 처리 순서는 아직 관련 계산을 확정값처럼 만들 수 없는 게이트다. [Source: `_bmad-output/planning-artifacts/prds/prd-erp_fish-2026-05-28-2/prd.md` "10. Open Questions"]

### 상태 모델 권장안

개발자는 정확한 타입명은 코드에 맞춰 정하되, 의미는 아래와 같이 유지한다.

```ts
type CalculationStatus =
  | "ok"
  | "data-insufficient"
  | "policy-unconfirmed"
  | "calculation-unavailable";

type CalculationMetric = {
  value: number | null;
  status: CalculationStatus;
  label?: "데이터 부족" | "확인 필요" | "계산 불가";
  reason?: string;
};
```

- `data-insufficient`: 저장된 입력 자체가 부족하다. 예: 근무인원 없음, 비용 라인 없음이 계산 요구상 필요한 경우, 재고 입력 없음.
- `policy-unconfirmed`: 데이터는 있어도 정책/OQ가 닫히지 않았다. 예: FIFO 확정 원가, `30%단가`, 희망 판매가 손실액, 매출차액 임계값 해석.
- `calculation-unavailable`: 0으로 나누기, 안전 정수 범위 초과, 유효하지 않은 숫자, 서버 계산 예외. 예외 로그에는 metric id, ledger/report context id, 원인 코드를 남기되 민감한 원가/이익 상세값을 불필요하게 남기지 않는다.
- 기존 화면 호환을 위해 `policy-unconfirmed`은 현재 표시상 `확인 필요` 또는 기존 문자열 `계산 기준 확인 필요`로 매핑할 수 있다. 단, 새 내부 상태 코드에서는 `data-insufficient`와 분리한다.

### 최신 기술 정보

- `package.json` 기준 현재 프로젝트는 Next `^15.2.3`, React `^19.0.0`, Prisma `^6.6.0`, Node test runner 기반 unit test를 사용한다. 이 story에서 프레임워크 업그레이드를 하지 않는다.
- Next 공식 문서는 2026-06 확인 시 최신을 16.1.7로 표시하지만, 현재 repo 버전과 패턴을 유지한다. Server Actions는 서버에서 실행되는 async mutation이며 호출 후 revalidation과 함께 쓰는 패턴을 따른다. [Source: `https://nextjs.org/docs/app/getting-started/updating-data`]
- Prisma 트랜잭션은 여러 DB 작업을 하나의 성공/실패 단위로 묶는 공식 패턴이다. 이 story는 DB 쓰기가 주목적이 아니지만, 계산 호출부가 기존 트랜잭션 query 안에서 쓰이는 점을 깨지 않는다. [Source: `https://www.prisma.io/docs/orm/prisma-client/queries/transactions`]
- Node test runner는 현 저장소 unit test 명령과 맞는다. Jest/Vitest 신규 도입은 범위 밖이다. [Source: `https://nodejs.org/api/test.html`]
