# Story 8.5: 마감 운영, 작성자 표시명, 재고 용어, 그리드 리사이징 완료 기준 확정

Status: done

## Story

As a PM과 UX 리드,
I want 운영 편의 확장과 데스크탑 그리드 UX의 완료 기준을 확정하고 싶다,
so that CAP-15/CAP-16/CAP-17/CAP-18이 안전한 운영 흐름 안에서 구현될 수 있다.

## Acceptance Criteria

1. **Given** CAP-15가 일괄 마감 또는 검토 생략 토글을 요구할 때, **When** 마감 운영 기준을 정한다, **Then** dry run, preflight 실패 사유, 실행 권한, 사유 입력, 감사 이벤트, 부분 실패 처리 기준을 포함해야 한다, **And** 원본 장부 잠금과 append-only 정정 원칙을 약화시키면 안 된다.
2. **Given** CAP-16이 작성자 이름 세션 캐싱을 요구할 때, **When** 작성자 표시명 기준을 정한다, **Then** 표시명, 인증 계정, 감사 actor를 분리하고 장부 입력 7단계 흐름에서 유지되는 범위를 정의해야 한다, **And** 감사 로그 actor를 표시명 캐시로 대체하면 안 된다.
3. **Given** CAP-17이 재고 테이블 문구 변경을 요구할 때, **When** `차이`와 `당일 판매량`의 의미를 검토한다, **Then** 단순 라벨 변경과 계산 의미 변경을 분리하고, 계산 의미 변경은 OQ-14 승인 후에만 승격해야 한다, **And** Epic 7 Story 7.7의 용어 승인 산출물을 참조해야 한다.
4. **Given** CAP-18과 UX-DR23이 데스크탑 리사이징 데이터 테이블을 요구할 때, **When** 완료 기준을 정한다, **Then** 키보드 조작, 최소/최대 폭, 핵심 컬럼 숨김 방지, 사용자별 레이아웃 저장, `기본값으로 되돌리기`를 포함해야 한다, **And** 모바일 장부 입력의 390px 지원 기준을 깨지 않도록 데스크탑 전용으로 범위를 제한해야 한다.
5. **Given** 산출물이 완료되었을 때, **When** PM과 UX 리드가 승인한다, **Then** 산출물에는 CAP-15 preflight 계약, CAP-16 표시명 기준, CAP-17 용어 결정, CAP-18 완료 기준, 승격 가능 여부가 포함되어야 한다.

### Architecture Guardrails

- ERP Fish는 Next.js App Router, Server Components/Server Actions, Prisma, PostgreSQL, Auth.js/NextAuth, Tailwind/shadcn UI를 사용한다. 이 story에서는 새 library, public API, tRPC, client-only global state, DB migration을 추가하지 않는다. [Source: `_bmad-output/planning-artifacts/architecture.md#Starter Template Evaluation`] [Source: `package.json`]
- 권한은 서버에서 강제한다. 마감, 정정, report, settings action은 shared authorization helper를 먼저 통과해야 한다. 일괄 마감도 UI toggle이나 hidden control이 아니라 server action 권한과 ledger scope로 통제해야 한다. [Source: `_bmad-output/planning-artifacts/architecture.md#Authorization Pattern`]
- 본사 마감 후 원본 장부는 보존되고 변경은 append-only 정정 기록으로만 반영된다. CAP-15는 이 원칙을 약화시키는 bulk rollback, reopen, silent overwrite를 포함하면 안 된다. [Source: `_bmad-output/planning-artifacts/architecture.md#Audit and Correction Strategy`]
- Store-scoped mutation은 접근 제어에 필요한 최소 identifier를 먼저 parse하고 권한/scope 확인 후 상세 validation을 수행한다. 일괄 마감 dry run도 상세 실패 사유를 권한 확인 전에 반환하면 안 된다. [Source: `_bmad-output/planning-artifacts/architecture.md#Data Architecture`]
- Resizable dashboard and analysis tables may justify TanStack Table or equivalent utility, but resizing is not a security boundary. Column visibility/width preferences must not expose sensitive fields or hide required error states. [Source: `_bmad-output/planning-artifacts/architecture.md#Tables and Reports`]
- 지점장 ledger input은 390px 모바일 폭에서 동작해야 한다. CAP-18은 데스크탑 HQ data table 범위이며 모바일 장부 입력의 sticky header, pagination/virtualization, 44px touch target 기준을 깨면 안 된다. [Source: `_bmad-output/planning-artifacts/architecture.md#Responsive Requirements`]
