# Story 1.2: 사용자, 지점, 권한 프로파일 모델링

Status: done

## Story

As a 본사 설정 관리자,
I want 사용자, 지점, 지점 접근권, 권한 프로파일의 기본 데이터 구조가 준비되기를 원한다,
so that 본사와 지점장이 각자 허용된 범위에서만 ERP Fish를 사용할 수 있다.

## Acceptance Criteria

1. Given 개발자가 Prisma schema를 작성할 때, when 사용자와 지점 모델을 정의한다, then 사용자는 본사 사용자 또는 지점장 계정으로 구분될 수 있어야 한다, and 지점은 활성/비활성 상태와 기본 식별 정보를 가져야 한다.
2. Given 한 사용자가 지점 데이터에 접근할 때, when 서버가 접근 범위를 확인한다, then 지점장은 연결된 지점만 접근할 수 있어야 한다, and 본사 사용자는 권한 프로파일에 따라 전체 또는 지정 지점에 접근할 수 있어야 한다.
3. Given 본사 권한이 단일 역할로 충분하지 않을 때, when 권한 프로파일을 정의한다, then 마감, 정정, 업로드, 설정, 리포트/export 같은 action 단위 권한을 표현할 수 있어야 한다, and 이후 스토리에서 action-level authorization helper가 이 구조를 사용할 수 있어야 한다.
4. Given 개발자가 초기 테스트 데이터를 준비할 때, when seed 데이터를 실행한다, then 최소 본사 관리자 계정, 지점장 계정, 샘플 지점, 지점 접근 연결이 생성되어야 한다, and 초기 비밀번호 또는 secret은 운영용 실제 값으로 저장소에 커밋되지 않아야 한다.
5. Given 품목/장부/감사 기능이 아직 구현되지 않았을 때, when 이 스토리가 완료된다, then 사용자, 지점, 권한 프로파일 구조만 독립적으로 검증 가능해야 한다, and 미래 장부 모델 전체를 미리 만들지 않아야 한다.

### Latest Technical Information

- 2026-06-11 확인: Prisma의 explicit many-to-many relation은 relation table 자체를 schema model로 표현하며, relation metadata를 저장해야 할 때 적합하다. 권한 profile assignment와 profile-action 연결은 system flag, createdAt, assignedBy 같은 확장 여지가 있으므로 explicit join model을 사용한다. [External: https://www.prisma.io/docs/orm/prisma-schema/data-model/relations/many-to-many-relations]
- 2026-06-11 확인: Auth.js Credentials provider는 username/password 흐름을 지원하지만 password hashing, rate limiting, reset 등은 애플리케이션 책임이다. 이 스토리는 기존 Credentials 흐름을 유지하고 rate limiting debt를 해결한 것처럼 표시하지 않는다. [External: https://authjs.dev/getting-started/authentication/credentials]
- 2026-06-11 확인: Next.js App Router 인증 가이드는 authorization을 session/cookie 기반 optimistic check와 DB 기반 secure check로 나누고, sensitive data/action은 data source 가까이에서 검사하라고 안내한다. 이 스토리도 profile/action/store authorization을 DB 기반 helper에 둔다. [External: https://nextjs.org/docs/app/guides/authentication]
