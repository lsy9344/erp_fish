# Story 3.3: 검토 화면의 계산/검증 요약

Status: done

## Story

As a 지점장 또는 본사 사용자,
I want 제출 전 검토 화면에서 장부의 기본 계산과 검증 상태를 한눈에 보고 싶다,
so that 누락과 이상 후보를 확인한 뒤 제출하거나 보완할 수 있다.

## Acceptance Criteria

1. **단계별 저장 상태와 기본 요약**
   - **Given** 사용자가 검토/제출 단계에 도달했을 때
   - **When** 검토 화면이 렌더링된다
   - **Then** 매출/결제, 비용, 매입, 재고, 손실, 근무 정보의 저장 상태와 기본 요약이 표시되어야 한다
   - **And** 각 요약은 서버 계산/검증 결과를 기준으로 해야 한다.

2. **결제 차액의 부호 표시**
   - **Given** 결제수단 합계와 총매출이 다를 때
   - **When** 검토 화면이 차이값을 표시한다
   - **Then** 부호 있는 차이값과 설명을 보여줘야 한다
   - **And** OQ-1 전에는 이상 신호 임계값 초과로 표현하지 않아야 한다.

3. **누락/검증 오류의 복귀 경로**
   - **Given** 필수 누락 또는 검증 오류가 있을 때
   - **When** 검토 화면이 표시된다
   - **Then** 사용자는 어느 단계로 돌아가야 하는지 알 수 있어야 한다
   - **And** 오류 항목을 눌러 해당 단계/필드로 이동할 수 있어야 한다.

4. **계산 상태 보존**
   - **Given** 정책 미정 또는 데이터 부족 계산이 있을 때
   - **When** 검토 화면이 결과를 표시한다
   - **Then** `기준 확인 필요`, `데이터 부족`, `계산 불가` 상태를 그대로 보여줘야 한다
   - **And** 0원 또는 빈 값으로 숨기면 안 된다.

5. **지점장 민감 지표 차단**
   - **Given** 지점장 권한으로 검토 화면을 볼 때
   - **When** 서버가 요약 응답을 만든다
   - **Then** 민감 회계 지표나 원가 기반 파생값은 응답에서 제외되어야 한다
   - **And** 지점장에게 필요한 입력 확인 정보만 제공되어야 한다.

6. **본사 권한 검토 요약**
   - **Given** 본사 사용자가 검토 화면을 볼 때
   - **When** 권한 있는 본사 계정이 접근한다
   - **Then** 본사 권한에 허용된 요약과 검증 상태를 볼 수 있어야 한다
   - **And** 권한 프로파일에 없는 action은 수행할 수 없어야 한다.

## Dev Notes

### 핵심 구현 방향

- 이 story는 계산 엔진을 새로 만드는 작업이 아니다. Story 3.1의 `calculateLedgerReviewSummary`, Story 3.2의 `getLedgerReviewMissingItems`/서버 검증 결과를 검토 화면이 더 잘 보여주게 만드는 작업이다. [Source: `_bmad-output/implementation-artifacts/3-1-기본-계산-모듈과-상태-표현.md`; `_bmad-output/implementation-artifacts/3-2-장부-단계별-입력-검증.md`]
- 검토 화면의 기준 데이터는 `src/features/ledger/review-queries.ts`에서 서버 transaction 안에 모은다. `ReviewSummaryClient`는 받은 결과를 렌더링하고 제출 action을 호출하는 클라이언트 컴포넌트로 유지한다. [Source: `src/features/ledger/review-queries.ts`; `src/features/ledger/components/review-summary-client.tsx`]
- 지점장 민감 지표 차단은 MVP 필수 보안 기준이다. 지점장 화면/API에는 매출원가, 매출이익, 영업이익, 이익률/마진율, 인당생산성, 재고금액, 원가 lot 근거, 재고/손실 금액 근거를 노출하지 않는다. [Source: `_bmad-output/planning-artifacts/prds/prd-erp_fish-2026-05-28-2/prd.md` §4.1, §4.3; `_bmad-output/planning-artifacts/epics.md` UX-DR12]
- 본사와 지점장 요구를 한 타입에 억지로 섞지 않는다. 지점장용 `StoreManagerLedgerReviewStepData`는 안전한 subset이고, 본사 전용 요약은 별도 mapper나 기존 HQ detail response를 사용한다. [Source: `src/features/ledger/review-types.ts`; `src/features/ledger/response-shaping.ts`]
- OQ-1이 닫히기 전에는 결제/매출차액을 "임계값 초과" 또는 "확정 이상"으로 표현하지 않는다. 차이값 자체와 확인 설명만 표시한다. [Source: `_bmad-output/planning-artifacts/epics.md` Story 3.3; `_bmad-output/planning-artifacts/prds/prd-erp_fish-2026-05-28-2/prd.md` Open Questions OQ-1]

### 반드시 보존할 동작

- 권한은 서버에서 강제한다. Server Action은 렌더링된 페이지가 인증 화면이어도 내부에서 인증/인가를 확인해야 한다. [Source: Next Forms guide `https://nextjs.org/docs/app/guides/forms`]
- `submitLedgerForReview`의 optimistic concurrency/version conflict와 `SaveConflictDialog` 흐름을 깨지 않는다. 제출 성공 시 `version`, `updatedAt`, `submittedById`, `submittedAt` 갱신을 유지한다. [Source: `src/features/ledger/actions.ts`; `src/features/ledger/components/review-summary-client.tsx`]
- 저장/제출 revalidation은 서버 전용 API 경계에서 유지한다. `revalidatePath`는 서버 함수/Route Handler에서 호출하는 API다. [Source: Next `revalidatePath` docs `https://nextjs.org/docs/app/api-reference/functions/revalidatePath`]
- Zod major upgrade를 하지 않는다. repo는 Zod `^3.24.2`를 사용하고 있으며, 기존 `fieldErrors`/issue path 변환 계약을 유지한다. [Source: `package.json`; Zod error formatting `https://zod.dev/error-formatting`]
- 첫 오류 focus나 이동 보조를 구현할 때 React ref 기반 focus 패턴을 유지할 수 있다. [Source: React `useRef` docs `https://react.dev/reference/react/useRef`]
- DB migration은 기본적으로 필요 없다. 이번 story는 검토 화면 data shape/UI/test 확장이다. persisted schema 변경은 요구사항 밖이다.

### UX / 표시 가드레일

- 내부 ERP/SaaS 톤을 유지한다. 검토 화면은 장식보다 빠른 스캔, 단계별 상태, 명확한 복귀 액션이 우선이다. [Source: `_bmad-output/planning-artifacts/ux-designs/ux-erp_fish-2026-05-28/DESIGN.md`]
- `ReviewSummary`는 필수 누락, 단계별 완료 여부, 이상 후보, 이월/저장 상태를 한 번에 보여주며 지점장에게 민감 지표를 표시하지 않는다. [Source: `_bmad-output/planning-artifacts/ux-designs/ux-erp_fish-2026-05-28/EXPERIENCE.md` Component Patterns]
- 모바일 390px 폭에서 카드/버튼 텍스트가 겹치면 안 된다. 기존 `min-h-11`, `tabular-nums`, wrap/break-word 패턴을 유지한다. [Source: `_bmad-output/planning-artifacts/epics.md` NFR5, UX-DR19, UX-DR22]
- 상태는 색상만으로 전달하지 않는다. `기준 확인 필요`, `데이터 부족`, `계산 불가`, `입력 필요`, `검토`는 텍스트 배지와 설명을 함께 제공한다. [Source: `_bmad-output/planning-artifacts/epics.md` UX-DR9, UX-DR13]
- 빈 상태도 명시한다. 경고/이상 후보가 없으면 기존처럼 "현재 표시할 합계 불일치나 이상 후보가 없습니다." 같은 문구를 유지하거나 더 구체화한다.
