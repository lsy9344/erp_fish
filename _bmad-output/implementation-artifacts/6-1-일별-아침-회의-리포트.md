# Story 6.1: 일별 아침 회의 리포트

Status: done

## Story

As a 본사 사용자,  
I want 특정 일자의 전체 지점 회의용 요약을 볼 수 있기를 원한다,  
so that 아침 회의 전에 지점별 상태와 주요 확인 대상을 빠르게 파악할 수 있다.

## Acceptance Criteria

1. **Given** 본사 사용자가 일별 리포트 화면에 접근할 때, **When** 날짜를 선택한다, **Then** 지점별 마감 상태, 장부 상태, 최신 반영 시각, 기본 매출 요약, 손실 여부, 상태 메시지가 표시되어야 한다, **And** 권한 있는 지점/컬럼만 포함되어야 한다.
2. **Given** 정정 기록이 있는 장부가 있을 때, **When** 리포트가 숫자를 표시한다, **Then** 기본 숫자는 정정 반영값을 사용해야 한다, **And** 원본값 확인은 장부 상세에서 가능해야 한다.
3. **Given** OQ-gated 계산이나 이상 기준이 필요한 항목이 있을 때, **When** 리포트가 해당 항목을 표시한다, **Then** 확정 숫자나 확정 이상으로 표시하지 않아야 한다, **And** `기준 확인 필요`, `데이터 부족`, `계산 불가` 상태를 구분해 보여야 한다.
4. **Given** 지점장 권한 사용자가 일별 본사 리포트 URL에 접근할 때, **When** 서버가 요청을 처리한다, **Then** 접근은 거부되어야 한다, **And** 전체 지점 리포트 데이터는 반환되지 않아야 한다.
5. **Given** 리포트가 조회될 때, **When** 서버가 데이터를 만든다, **Then** 장부 상세/관제판과 같은 서버 계산 함수를 사용해야 한다, **And** 리포트 전용으로 계산식을 복제하면 안 된다.
6. **Given** 일별 리포트 화면이 렌더링될 때, **When** 숫자 컬럼과 상태 배지가 표시된다, **Then** 숫자 컬럼은 읽기 쉽게 정렬되어야 한다, **And** 상태는 색상만이 아니라 텍스트로도 의미를 알 수 있어야 한다.

### Architecture Guardrails

- ERP Fish는 Next.js App Router, Server Components, Server Actions, Prisma, PostgreSQL, NextAuth/Auth.js, Tailwind/shadcn UI를 사용한다. 이 story에서 tRPC, 별도 REST API, client-only data fetch layer를 추가하지 않는다. [Source: `_bmad-output/planning-artifacts/architecture.md#Technical Stack`] [Source: `package.json`]
- Most product behavior는 Server Components와 Server Actions를 사용하고, Route Handlers는 exports/health/integration 같은 실제 HTTP endpoint에 한정한다. Daily report 조회는 public API route가 아니라 server query + page render로 유지한다. [Source: `_bmad-output/planning-artifacts/architecture.md#API Boundaries`]
- Report, dashboard, detail pages는 같은 server-side calculation functions를 공유해야 한다. 계산식을 UI 컴포넌트나 report 전용 helper에 복제하면 안 된다. [Source: `_bmad-output/planning-artifacts/architecture.md#Calculation Strategy`]
- Original ledger records는 본사 마감 후 보존된다. Reports and dashboards use correction-applied values by default while details keep original values visible. [Source: `_bmad-output/planning-artifacts/architecture.md#Audit and Correction Strategy`]
- `applyCorrectionValuesToLedgerReviewInput`는 dashboard, reports, close preflight가 공유하는 correction overlay다. Unsupported/not-yet-policy-approved corrections는 `정정 확인 필요` 또는 `기준 확인 필요`로 보여야 한다. [Source: `_bmad-output/planning-artifacts/architecture.md#Audit and Correction Strategy`]
- Authorization은 서버에서 강제한다. `REPORT_VIEW`와 store scope를 통과한 본사 사용자만 일별 리포트를 조회한다. [Source: `src/server/authz.ts`] [Source: `_bmad-output/planning-artifacts/architecture.md#Authentication & Security`]
- 변경 후 stale data를 막기 위해 ledger, close, correction, threshold, master-data 변경 action의 existing revalidation을 보존한다. Next.js 공식 문서는 `revalidatePath`가 Server Functions와 Route Handlers에서 호출 가능하고 Client Components에서는 호출할 수 없다고 한다. [Source: `src/features/corrections/actions.ts`] [Source: https://nextjs.org/docs/app/api-reference/functions/revalidatePath]
