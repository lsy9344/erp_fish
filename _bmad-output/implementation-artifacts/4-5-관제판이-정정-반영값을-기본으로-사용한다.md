# Story 4.5: 관제판이 정정 반영값을 기본으로 사용한다

Status: done

## Story

As a 본사 사용자,
I want 정정 기록을 추가한 뒤 관제판 숫자와 이상 신호가 정정 반영값으로 갱신되길 원한다,
so that 리포트 화면이 없어도 본사 운영 판단에 최신 보정 숫자를 사용할 수 있다.

## Acceptance Criteria

1. **Given** 마감 장부에 정정 기록이 추가되어 있을 때
   **When** 관제판 행이 표시되면
   **Then** 매출, 이익률, 매출차액, 손실 여부, 이상 신호 계산은 정정 반영값을 기본으로 사용해야 한다
   **And** 원본값이 필요한 경우 상세에서 구분해 확인할 수 있어야 한다.

2. **Given** 정정 기록이 새로 저장될 때
   **When** 저장이 완료되면
   **Then** 관련 장부 상세와 관제판 경로가 갱신되어야 한다
   **And** 사용자가 낡은 계산값을 계속 보지 않도록 revalidation 또는 동등한 갱신 처리가 수행되어야 한다.

3. **Given** 정정 반영값을 계산할 때
   **When** 같은 대상에 여러 정정 기록이 있으면
   **Then** 최신 정정 기록의 `정정값`을 최종 반영값으로 사용해야 한다
   **And** 이전 정정 기록은 계산에서 제외되더라도 이력에서는 보존되어야 한다.

4. **Given** 정정 기록 대상이 계산 지표에 영향을 줄 때
   **When** 매출이익, 이익률, 영업이익, 매출차액, 이상 신호를 계산하면
   **Then** shared server calculation이 원본값과 정정 반영값을 명시적으로 구분해 계산해야 한다
   **And** UI에서 별도 보정 계산을 만들면 안 된다.

5. **Given** 정정 대상이나 정정값 형식이 계산에 적용할 수 없을 때
   **When** 관제판이 계산을 수행하면
   **Then** 임의로 원본값을 정상값처럼 사용하지 않아야 한다
   **And** "정정 확인 필요" 또는 계산 불가 상태를 표시해야 한다.

6. **Given** 지점장이 마감 장부를 조회할 때
   **When** 정정 반영값이 있는 항목을 보면
   **Then** 원본 수정은 할 수 없어야 한다
   **And** 정정 반영값과 본사 정정 이력을 읽기 전용으로 확인할 수 있어야 한다.

## Dev Notes

### 핵심 목표

- Story 4.5는 정정 기록 생성이나 장부 상세의 원본/정정 표시를 다시 만드는 story가 아니다. Story 4.3/4.4에서 만든 append-only `CorrectionRecord`, 최신 정정값 map, 상세 표시 패턴을 재사용해 관제판 기본 계산 기준을 정정 반영값으로 바꾸는 story다.
- 관제판 행의 `매출`, `이익률`, `매출차액`, `손실 여부`, `이상 신호`는 정정 반영값을 기본으로 사용해야 한다.
- 원본값은 감사 기준으로 계속 보존한다. 정정 반영값을 만들기 위해 원본 DB row를 덮어쓰면 안 된다.
- Epic 5 리포트가 같은 계산 기준을 재사용해야 하므로, 임시 UI 계산이나 dashboard-only 계산식을 만들면 안 된다.

### 현재 구현 상태

- `src/features/dashboard/queries.ts`의 `getHqDashboardRows`는 active store와 해당 일자 ledger를 조회하고, `toDashboardRow`에서 원본 `DailyLedger` 값을 `calculateLedgerReviewSummary`에 넣는다.
- `src/features/dashboard/queries.ts`의 `getHqLedgerDetail`도 같은 방식으로 원본 ledger 값 기준의 상세 요약과 이상 신호를 계산한다.
- `src/server/calculations/ledger.ts`는 `calculateLedgerReviewSummary`를 제공하지만, 아직 원본값과 정정 반영값을 구분하는 input/result shape는 없다.
- `src/features/corrections/queries.ts`에는 `getLatestCorrectionValueMap`, `getLatestCorrectionValuesForLedger`, `buildCorrectionTargetKey`가 있다. 단건 ledger 표시에는 충분하지만, 관제판 다건 ledger에 그대로 쓰면 N+1 query가 될 수 있다.
- `src/features/corrections/actions.ts`는 정정 저장 후 `/app/ledgers/${ledgerId}`와 `/app/dashboard`를 revalidate한다. 이 동작은 Story 4.5 AC 2에 직접 해당한다.
- `src/app/app/ledgers/[ledgerId]/page.tsx`는 Story 4.4에서 `getLatestCorrectionValueMap(correctionRecords)`로 상세 화면의 원본/정정 반영 표시를 제공한다.

### 구현 가드레일

1. 기존 `CorrectionRecord` 모델을 재사용한다. 새 정정 테이블이나 dashboard 전용 보정 테이블을 만들지 않는다.
2. 최신 정정값 선택 규칙은 Story 4.3/4.4와 동일하게 `createdAt desc, id desc`다.
3. 원본 `DailyLedger`, `LedgerExpense`, `LedgerPurchaseItem`, `LedgerInventoryItem`, `LedgerLossItem` 값은 정정 반영 계산 때문에 변경하지 않는다.
4. 정정 반영 overlay와 계산은 서버 helper에 둔다. React 컴포넌트에서 마진율, 매출차액, 손실 이상 신호를 계산하지 않는다.
5. 적용 가능한 정정값만 반영한다. `kind`, `targetType`, `fieldKey`, 대상 row가 계산에 맞지 않으면 "정정 확인 필요" 또는 계산 불가 상태를 명시한다.
6. 관제판 query는 빠르게 유지한다. correction records는 필요한 ledger id 범위로 한 번에 조회하고, 상세 timeline/audit 데이터는 dashboard row에 싣지 않는다.
7. `검토대기`는 원본 수정 가능, `HEADQUARTERS_CLOSED` 이후만 정정 기록 사용이라는 정책을 유지한다.

### Data and Calculation Notes

- `CorrectionRecord.correctedValue`는 `{ kind, value, label }` JSON shape를 사용한다.
- `PAYMENT_FIELD.totalSalesAmount` 정정은 `totalSalesAmount` 계산 입력에 직접 반영할 수 있다.
- `PAYMENT_FIELD.cashAmount`, `cardAmount`, `otherPaymentAmount` 정정은 `paymentDifference`와 결제 관련 검토값에 영향을 준다.
- `LEDGER_FIELD.workerCount` 정정은 `productivity` 계산 입력에 영향을 준다.
- `INVENTORY_ROW.currentQuantity` 또는 `quantity`, `inventoryAmount` 정정은 재고 흐름과 재고 이상 신호 입력에 영향을 줄 수 있다. 현재 `calculateLedgerReviewSummary`는 `currentQuantity ?? quantity`와 `unitPrice`로 재고 금액과 매출원가를 계산한다.
- `LOSS_ROW.amount` 또는 `quantity` 정정은 손실 여부와 손실 이상 신호 입력에 영향을 준다.
- `CALCULATED_METRIC` 정정은 계산 결과를 직접 덮어쓰는 위험이 크다. 계산에 적용할 수 없으면 "정정 확인 필요"로 다루고, 원본 계산값을 정상값처럼 보여주지 않는다.
- `salesDifference`는 현재 `calculateLedgerReviewSummary`에서 "계산 기준 확인 필요"로 남아 있다. Story 4.5에서 이 기준 자체를 새로 정의하지 말고, 정정 반영 overlay가 기존 unavailable 상태를 숨기지 않게 한다.

### UX and Accessibility Notes

- 관제판 행은 전체 클릭으로 장부 상세에 진입해야 한다. 정정 반영값의 원본 차이는 상세에서 확인하게 연결한다.
- 이상 신호는 `SignalChip`처럼 아이콘과 텍스트 레이블을 함께 사용한다. 색상만으로 의미를 전달하지 않는다.
- 숫자 데이터는 `tabular-nums`를 유지한다.
- 데스크탑에서는 전체 컬럼, 태블릿/모바일에서는 기존 간소화 동작을 유지한다.
- "정정 반영"이나 "정정 확인 필요" 상태를 추가할 때 버튼/배지 텍스트가 좁은 화면에서 겹치지 않게 줄바꿈과 `break-words` 패턴을 유지한다.
