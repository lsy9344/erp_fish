# Story 8.6: 특수기간, 엑셀 매핑, 월 손익 리포트 계약 확정

Status: done

## Story

As a PM과 본사 운영자,
I want 리포트 고도화의 컬럼, 필터, 집계 계약을 확정하고 싶다,
so that CAP-2/CAP-3/CAP-10이 본사 판단에 맞는 숫자를 제공하게 할 수 있다.

## Acceptance Criteria

1. **Given** CAP-2가 특수기간 비교 리포트를 요구할 때, **When** 특수기간 기준을 정한다, **Then** 설날 외 기간 포함 여부, D-day 기준, 연도 비교 방식, 지점 필터, 미마감 장부 처리 기준을 정의해야 한다, **And** OQ-11이 닫히기 전에는 특수기간 구현 스토리를 만들면 안 된다.
2. **Given** CAP-3이 기존 엑셀 보고 항목 매핑을 요구할 때, **When** 리포트 항목 계약을 작성한다, **Then** 기존 엑셀 항목명, 시스템 지표명, 계산식, 데이터 부족 표시, export 컬럼을 매핑해야 한다, **And** 엑셀 수식 복제가 아니라 서버 계산 기준을 사용해야 한다.
3. **Given** CAP-10이 월 손익계산서를 요구할 때, **When** 월 손익 정책을 정한다, **Then** 본사 고정비 입력, 매출원가, 영업이익, 정정 반영, 미마감 장부 수, 지점장 차단 기준을 포함해야 한다, **And** OQ-10B와 고정비 정책 전에는 민감 월 손익 노출 구현 스토리를 만들면 안 된다.
4. **Given** 산출물이 완료되었을 때, **When** PM과 본사 운영자가 승인한다, **Then** 산출물에는 리포트 컬럼·집계 계약서, 특수기간 정의서, 월 손익 계산 정책, 승격 가능 여부가 포함되어야 한다, **And** 후속 구현 스토리는 Epic 6의 본사 전용 export 차단 기준을 유지해야 한다.

### Architecture Guardrails

- ERP Fish는 Next.js App Router, Server Components/Server Actions/Route Handlers, Prisma, PostgreSQL, NextAuth/Auth.js, Tailwind/shadcn UI를 사용한다. 이 story에서는 새 library, public API, tRPC, client-only fetch layer, DB migration을 추가하지 않는다. [Source: `_bmad-output/planning-artifacts/architecture.md#Decision Impact Analysis`] [Source: `package.json`]
- Dashboard, detail pages, reports, export는 같은 server-side calculation functions를 공유해야 한다. CAP-2, CAP-3, CAP-10 계산식은 UI 컴포넌트, chart helper, export helper에 복제하지 않는다. [Source: `_bmad-output/planning-artifacts/architecture.md#Calculation Strategy`]
- 특수기간 비교, 월 손익, 알림 조건 같은 추가 계산도 shared server calculation module에 있어야 한다. 정책이나 원천 데이터가 없으면 `확인 필요`, `계산 불가`, `데이터 부족` 상태를 반환하고 0이나 stale 단가로 대체하지 않는다. [Source: `_bmad-output/planning-artifacts/architecture.md#Calculation Strategy`]
- 본사 마감 후 원본 장부는 보존되고, reports/dashboard/export 기본 숫자는 정정 반영값을 사용한다. 원본값과 정정 반영값은 권한 있는 본사 사용자에게만 추적 가능해야 한다. [Source: `_bmad-output/planning-artifacts/architecture.md#Audit and Correction Strategy`]
- Authorization은 모든 ledger, report, master-data, close, correction, settings action에서 서버가 강제한다. CAP-2/CAP-10 후속 report query는 `requireReportAccess()`와 store scope 확인 없이 데이터를 반환하면 안 된다. [Source: `_bmad-output/planning-artifacts/architecture.md#Authorization Pattern`]
- Route Handlers는 exports, health, integration endpoint처럼 실제 HTTP endpoint가 필요할 때만 사용한다. report 화면 조회는 existing server query + page render를 유지하고, 다운로드만 export Route Handler에 둔다. [Source: `_bmad-output/planning-artifacts/architecture.md#API Boundaries`]
