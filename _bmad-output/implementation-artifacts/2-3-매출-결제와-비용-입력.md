# Story 2.3: 매출/결제와 비용 입력

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 지점장,
I want 총매출, 결제수단, 비용 항목을 입력할 수 있기를 원한다,
so that 하루 장부의 기본 매출과 비용 정보를 본사에 제출할 수 있다.

## Acceptance Criteria

1. Given 사용자가 매출/결제 단계에 있을 때, when 총매출, 현금, 카드, 기타 결제수단 값을 입력한다, then 값은 장부에 저장되어야 한다, and 금액은 서버에서 숫자/범위 검증을 거쳐야 한다.
2. Given 결제수단 합계와 총매출이 다를 때, when 사용자가 매출/결제 단계를 저장하거나 검토 단계로 이동한다, then 시스템은 부호 있는 차이값을 표시해야 한다, and OQ-1이 닫히기 전에는 이상 신호 임계값 초과 판정으로 처리하지 않아야 한다.
3. Given 사용자가 비용 단계에 있을 때, when 비용 항목, 금액, 메모를 여러 줄 입력한다, then 각 비용 라인은 장부에 저장되어야 한다, and 비용 합계는 서버 기준으로 계산되어야 한다.
4. Given 비용 항목 코드가 비활성 상태일 때, when 사용자가 새 비용 라인을 추가한다, then 비활성 코드는 신규 선택지에 표시되지 않아야 한다, and 과거 장부에 이미 저장된 비활성 코드는 원본 표시를 유지해야 한다.
5. Given 지점장 또는 본사가 매출/비용 입력값을 수정할 때, when 저장 요청이 성공한다, then 변경 이력은 감사 로그 helper를 통해 남아야 한다, and 장부의 version 또는 edit token이 갱신되어야 한다.
6. Given 사용자가 권한 없는 지점의 매출/비용 저장을 시도할 때, when 서버 action이 요청을 처리한다, then 저장은 거부되어야 한다, and 권한 밖 장부 데이터는 반환되지 않아야 한다.

### Architecture Guardrails

- 선택 스택은 Next.js App Router, TypeScript, Prisma, PostgreSQL, NextAuth/Auth.js, Tailwind CSS, shadcn/ui, Vercel이다. Story 2.3에서 ORM, auth framework, UI primitive 체계를 바꾸지 않는다. [Source: `_bmad-output/planning-artifacts/architecture.md#Starter Template Evaluation`]
- Prisma schema and migrations are the source of truth for persisted data. 이 story는 기존 `DailyLedger`와 `LedgerExpense` 모델을 확장/보강하며 별도 parallel ledger table을 만들지 않는다. [Source: `_bmad-output/planning-artifacts/architecture.md#Data Architecture`]
- 대부분의 제품 동작은 Server Components와 Server Actions를 사용한다. 매출/비용 저장을 위해 public API route를 만들지 않는다. [Source: `_bmad-output/planning-artifacts/architecture.md#API & Communication Patterns`]
- 모든 mutation은 Zod validation, shared authorization helper, transaction, audit, revalidation을 따라야 한다. [Source: `_bmad-output/planning-artifacts/architecture.md#Process Patterns`]
- Store manager responses must omit sensitive accounting fields. 비용 합계는 허용하되 매출원가, 매출이익, 영업이익, 이익률, 인당생산성, 재고금액, lot 근거는 지점장 응답에서 제외한다. [Source: `_bmad-output/planning-artifacts/architecture.md#Sensitive Field Gate`]
- 검토 대기는 원본 잠금 상태가 아니다. `IN_REVIEW`에서는 원본 수정이 가능하고, `HEADQUARTERS_CLOSED`만 원본 수정 차단 상태다. [Source: `_bmad-output/planning-artifacts/architecture.md#Implementation Handoff`]

### Data and Security Guardrails

- 저장 대상은 `ledgerId` 단독이 아니라 `storeId + closingDate + ledgerId + version` 조합으로 검증한다.
- 금액은 DB와 계산 경계에서 integer KRW로 유지하고 UI formatting 문자열은 서버로 그대로 저장하지 않는다.
- `paymentDifferenceAmount`는 서버 계산 결과를 authoritative response로 사용한다. UI의 실시간 draft 계산은 보조 표시다.
- OQ-1 전에는 매출/결제 차이를 anomaly threshold 초과, warning/danger signal, close-blocking condition으로 확정하지 않는다.
- 비용 코드 선택지는 active `EXPENSE_ITEM`만 신규 저장 가능해야 한다. 비활성 코드 표시 보존은 조회/UI 책임이며, 신규 저장 허용과 혼동하지 않는다.
- 비용 라인 전체 교체 방식은 현재 구현과 테스트를 따르되, audit payload에 before/after expenseItems가 남아야 한다.
- `HEADQUARTERS_CLOSED`와 `HOLIDAY`에서는 client disabled뿐 아니라 server action도 원본 수정 요청을 거부해야 한다.
- 지점장 응답은 기존 `toStoreManagerLedgerCostStepData` response shaping을 거쳐야 한다. 민감 필드를 client에서만 숨기면 안 된다.

### UX Guardrails

- 장부 입력은 반복 업무 task surface다. 매출/결제와 비용 단계는 현재 선택된 지점+영업일 장부 context 안에서 바로 입력 가능해야 한다.
- 7단계 순서와 label은 유지한다: 매출/결제, 비용, 매입, 재고, 손실/폐기/떨이, 근무인원/특이사항, 검토/제출.
- 숫자 입력은 `inputMode="numeric"`과 `tabular-nums`를 유지한다.
- 결제 차이값은 `+₩`, `-₩`, `₩0`처럼 부호가 사용자가 이해 가능하게 표시되어야 한다. 0과 미입력을 혼동시키지 않는다.
- 비용 행 추가/삭제 button은 lucide icon + 명확한 label을 사용하고, 390px에서 텍스트가 넘치거나 행 내부 control이 겹치지 않아야 한다.
- 비활성 과거 비용 코드는 기존 행에서 원래 이름으로 보존하되, 사용자가 새 행을 추가할 때 선택 가능한 옵션처럼 강조하지 않는다.
- 저장 상태는 색상만으로 전달하지 않는다. "저장됨", "저장 중", "저장 실패", "마지막 저장" 텍스트를 포함한다.
- 지점장에게 원가, 이익/마진, 인당생산성, 재고금액, lot 근거를 노출하지 않는다.
