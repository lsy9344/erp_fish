# Story 2.6: 지점장이 손실/폐기/떨이를 입력하고 재고 흐름에 반영한다

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 지점장,
I want 손실, 폐기, 떨이 항목을 품목별로 입력하고 싶다,
so that 실제 재고 감소와 손실 금액이 장부와 검토 요약에 반영될 수 있다.

## Acceptance Criteria

1. **Given** 지점장이 입력중 장부의 손실/폐기/떨이 단계에 접근할 때
   **When** 화면이 열리면
   **Then** active 품목 목록과 active 손실 유형 코드 목록을 선택지로 사용할 수 있어야 한다
   **And** 비활성 품목이나 비활성 손실 유형은 신규 입력 선택지에서 제외되어야 한다.

2. **Given** 지점장이 손실 항목을 입력할 때
   **When** 품목, 수량, 금액, 처리 유형, 사유/특이사항을 저장하면
   **Then** 해당 손실 항목이 장부에 저장되어야 한다
   **And** 하나의 장부에 여러 손실 항목을 저장할 수 있어야 한다.

3. **Given** 손실 항목이 저장될 때
   **When** 재고 흐름을 다시 계산하면
   **Then** 손실 수량과 금액이 품목별 재고 흐름에 반영되어야 한다
   **And** 손실 합계는 검토 요약과 이후 손실 리포트에 사용할 수 있어야 한다.

4. **Given** 손실액 또는 손실 수량이 본사 기준값을 넘을 때
   **When** 손실 단계 또는 검토 요약이 표시되면
   **Then** 재고/손실 이상 후보로 표시될 수 있어야 한다
   **And** 이 경고는 저장을 막지 않아야 한다.

5. **Given** 사유/특이사항이 필요한 처리 유형일 때
   **When** 사용자가 사유 없이 저장을 시도하면
   **Then** 저장이 차단되어야 한다
   **And** 사유 필드에 한국어 검증 메시지가 표시되어야 한다.

6. **Given** 기존 손실 항목이 저장되어 있을 때
   **When** 손실 단계가 다시 열리면
   **Then** 저장된 손실 항목과 손실 합계가 표시되어야 한다.

7. **Given** 390px 모바일 화면에서 손실 단계를 사용할 때
   **When** 품목 선택, 유형 선택, 수량/금액 입력을 수행하면
   **Then** 터치 가능한 선택지와 숫자 입력 모드가 제공되어야 한다
   **And** 저장 후 성공 또는 실패 피드백이 표시되어야 한다.

## Dev Notes

### Source Context

- Story 2.6은 Epic 2의 손실/폐기/떨이 단계다. 앞의 재고 입력과 뒤의 검토/제출 사이에서 손실 수량/금액을 장부와 재고 흐름에 연결한다. [Source: `_bmad-output/planning-artifacts/epics.md` > `### Story 2.6`]
- 관련 요구사항은 `FR5`, `FR9`, `FR11`, `FR13`, `FR14`, `FR24`, `FR26`이다. 핵심은 품목+규격 단위 재고 흐름, 손실 리포트용 합계, 비활성 코드 신규 입력 제외, 이상 후보 표시다. [Source: `_bmad-output/planning-artifacts/prds/prd-erp_fish-2026-05-28-2/prd.md` > `FR-11`, `FR-14`, `FR-26`]
- UX 흐름은 7단계 입력 중 5단계를 손실로 둔다. 모바일 하단 탭에는 `장부` / `재고` / `손실`이 있고, 손실 단계는 품목 선택, 손실 유형 "폐기", 금액, 필수 사유 입력을 포함한다. [Source: `_bmad-output/planning-artifacts/ux-designs/ux-erp_fish-2026-05-28/EXPERIENCE.md` > `Flow 2`]

### Current Repository State

- `src/app/app/store-entry/losses/page.tsx`는 지금 `손실 입력 준비` placeholder만 표시한다. 이 파일은 권한/지점 선택 흐름을 보존하면서 실제 화면으로 교체해야 한다.
- `LedgerInputCodeGroup.LOSS_TYPE`은 이미 존재한다. `getActiveLedgerInputCodeOptions("LOSS_TYPE")`를 재사용하면 active 손실 유형 선택지를 얻을 수 있다.
- `DailyLedger`에는 `ledgerExpenses`, `ledgerPurchaseItems`, `ledgerInventoryItems`, `ledgerInventoryAdjustments` 관계만 있고 손실 관계는 아직 없다.
- `LedgerInputCode`에는 `ledgerExpenses` 관계만 있다. 손실 유형 relation을 추가해야 한다.
- `Product`와 `User`도 손실 라인 relation이 없다. 생성/수정자 추적을 위해 relation을 추가한다.
- `src/server/calculations/inventory.ts`의 `calculateSystemInventoryQuantity`는 현재 `previousQuantity + purchasedQuantity`만 계산한다. 손실 수량 차감을 여기에서 확장해야 UI와 재고/검토 계산이 갈라지지 않는다.
- `src/features/inventory/adjustment-reconciliation.ts`는 매입 변경 뒤 조정 기준을 다시 맞추지만, 아직 손실 집계는 모른다. 손실 저장 뒤 stale 조정이 남지 않게 같은 경로를 확장한다.

### Architecture Guardrails

- 스택은 Next.js App Router + Server Components/Server Actions + Prisma + PostgreSQL + Zod + shadcn/ui + Tailwind CSS를 유지한다. [Source: `_bmad-output/planning-artifacts/architecture.md` > `Core Architectural Decisions`]
- Route Handler를 새로 만들지 않는다. 이 기능은 내부 UI mutation이므로 Server Action이 맞다.
- 모든 민감한 조회/저장 경로는 서버에서 `requireStoreAccess` 또는 동등한 shared auth helper를 통과해야 한다. UI에서 탭을 숨기는 것만으로는 권한 처리가 아니다.
- business-changing action은 Zod 검증, transaction, audit row, revalidation을 같은 기존 패턴으로 처리한다.
- 마스터 데이터 변경은 과거 장부 값을 바꾸면 안 된다. 손실 라인도 매입/재고처럼 품목과 손실 유형 이름 snapshot을 갖는다.
- 돈은 integer KRW로 저장하고 UI에서만 원화 문자열로 표시한다.
- 계산 규칙은 `src/server/calculations`에 둔다. 손실 합계, 재고 차감, 이상 후보 기준을 client-only 계산으로 흩뜨리지 않는다.

### UX Guardrails

- 손실 화면은 운영 도구다. 마케팅형 hero/card layout을 만들지 말고, 기존 장부/재고 입력 화면처럼 조용하고 촘촘한 입력 UI를 유지한다.
- 동적 행 추가/삭제는 기존 비용/매입 화면의 `PlusIcon`, `Trash2Icon`, 한국어 오류, 첫 오류 포커스 패턴을 따른다.
- 이상 후보는 색상만으로 전달하지 않는다. "기준 초과" 같은 텍스트 레이블과 수치를 함께 보여준다.
- 저장 실패 문구는 기존과 맞춰 "저장에 실패했습니다. 다시 시도해 주세요."를 사용한다.

### Implementation Boundaries

- 이 스토리는 손실 입력과 재고 흐름 반영까지다. 본사 기준값 관리 화면, 월간 손실 리포트 전체 화면, 검토/제출 화면 완성은 각각 이후 스토리 범위다.
- 다만 이후 스토리가 사용할 수 있게 손실 합계와 이상 후보 계산 결과는 query/type으로 노출한다.
- 손실 저장은 경고 때문에 막히면 안 된다. 저장 차단은 필수 입력 누락, 잘못된 숫자, 권한 없음, 마감 상태 같은 명확한 오류에 한정한다.
- `IN_PROGRESS` 장부 저장은 필수다. `IN_REVIEW` 허용 여부는 Story 2.5와 같은 정책을 따른다. 검토 대기는 잠금이 아니므로, 손실도 본사 마감 전에는 수정 가능하게 두는 편이 현재 PRD와 맞다. `HEADQUARTERS_CLOSED`는 원본 저장을 막고 정정 흐름 안내를 반환한다.

### Latest Technical Information

- Next.js 공식 문서는 mutation 뒤 `revalidatePath` 또는 `revalidateTag`를 Server Function 안에서 호출해 업데이트된 데이터를 다시 보여주는 흐름을 안내한다. 현재 저장소는 `revalidatePath` 패턴을 쓰므로 손실 저장도 같은 방식으로 `/app/store-entry/losses`, `/app/store-entry/inventory`, `/app/store-entry`, `/app/dashboard` 중 영향 화면을 갱신한다.
  [External: `https://nextjs.org/docs/app/building-your-application/data-fetching/server-actions-and-mutations`]
- Prisma 공식 문서는 복잡한 read/modify/write에는 interactive `$transaction`을 쓰고, transaction은 짧게 유지하라고 안내한다. 손실 저장, 재고 조정 재계산, 감사 로그 기록은 같은 transaction 안에서 처리하되 외부 네트워크 작업은 넣지 않는다.
  [External: `https://www.prisma.io/docs/orm/prisma-client/queries/transactions`]
- Zod 공식 문서는 `safeParse`, `transform`, `pipe`, refinement 계열 API를 제공한다. 저장소는 Zod v3.24.2를 사용하므로 기존 `safeParse` + `toFieldErrors` 패턴을 유지한다.
  [External: `https://zod.dev/packages/zod`]
