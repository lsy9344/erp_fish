# Story 1.7: 본사가 기준정보와 권한 변경 이력을 확인한다

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 본사 사용자,
I want 기준정보와 권한 변경 이력을 확인하고 싶다,
so that 누가 언제 무엇을 바꿨는지 추적하고 운영 문제를 빠르게 확인할 수 있다.

## Acceptance Criteria

1. Given 본사 사용자가 변경 이력 화면에 접근할 때, when 화면이 열리면, then 지점, 사용자/권한, 품목, 매입 기준, 코드 변경 이력이 시간 역순으로 표시되어야 한다, and 각 행에는 변경 시각, 변경자, 대상 유형, 대상 이름, 변경 유형이 표시되어야 한다.
2. Given 본사 사용자가 특정 이력 행을 선택할 때, when 상세가 열리면, then 변경 전 값과 변경 후 값을 구분해서 볼 수 있어야 한다, and 값이 긴 경우에도 읽을 수 있는 형태로 표시되어야 한다.
3. Given 본사 사용자가 변경 이력을 찾을 때, when 대상 유형, 변경자, 기간으로 필터링하면, then 조건에 맞는 이력만 표시되어야 한다.
4. Given 지점장 또는 비권한 사용자가 변경 이력 화면에 접근할 때, when 서버가 요청을 처리하면, then 변경 이력 데이터는 반환되지 않아야 한다, and 사용자는 "접근 권한이 없습니다." 화면을 봐야 한다.
5. Given 변경 이력 데이터가 없는 상태일 때, when 본사 사용자가 이력 화면을 열면, then 빈 표만 보여주는 대신 간결한 빈 상태 메시지를 표시해야 한다.
6. Given 변경 이력 목록이 로딩 중일 때, when 화면이 데이터를 기다리면, then 실제 표 레이아웃과 비슷한 Skeleton 상태가 표시되어야 한다.

### Audit Target and Action Mapping

- 대상 유형 표시:
  - `Store` -> `지점`
  - `User` -> `사용자/권한`
  - `Product` -> `품목`
  - `PurchaseStandard` -> `매입 기준`
  - `LedgerInputCode` -> `코드`
- 변경 유형 표시:
  - `*.created` -> `생성`
  - `*.updated` -> `수정`
  - `*.activated` -> `활성화`
  - `*.deactivated` -> `비활성화`
  - `user.role_changed` -> `역할 변경`
  - `user.store_assignments_changed` -> `지점 배정 변경`
  - `ledger_input_code.reordered` -> `표시 순서 변경`
- 알 수 없는 action은 숨기지 말고 원문 action을 fallback으로 보여준다. 그래야 후속 스토리에서 새 audit action이 추가되어도 데이터가 사라져 보이지 않는다.
- 대상 이름 fallback 우선순위는 current record name/email -> `after` snapshot -> `before` snapshot -> `targetId`이다. 과거 actor/target 보존을 위해 삭제 기능을 만들면 안 된다.

### Latest Technical Information

- 2026-05-29 확인: Next.js `revalidatePath`는 Server Functions와 Route Handlers에서 사용할 수 있고 Client Components에서는 사용할 수 없다. 이번 화면은 `requireHeadquartersUser()`/`noStore()` 경로를 거치는 읽기 화면이므로 history 표시를 위해 기존 mutation action에 무리하게 revalidation을 추가하지 않는다. [External: `https://nextjs.org/docs/app/api-reference/functions/revalidatePath`]
- 2026-05-29 확인: Next.js App Router는 route folder의 `loading.js`/`loading.tsx`로 즉시 loading UI를 제공하며 Skeleton 같은 fallback을 둘 수 있다. Story 1.7은 `/app/master-data/history/loading.tsx`를 사용한다. [External: `https://nextjs.im/docs/15/app/api-reference/file-conventions/loading/`]
- 2026-05-29 확인: Prisma Client는 `where`로 필터링하고 `orderBy`로 정렬한다. audit history query는 targetType/actor/date filter와 stable order를 Prisma query에 명시한다. [External: `https://www.prisma.io/docs/orm/prisma-client/queries/filtering-and-sorting`]
- 2026-05-29 확인: Prisma pagination은 offset과 cursor 기반을 모두 지원하며, 큰 데이터셋에는 cursor 방식이 더 안정적이다. 기본 50행 제한을 두고 후속 확장 시 cursor를 사용한다. [External: `https://www.prisma.io/docs/orm/prisma-client/queries/pagination`]
- 2026-05-29 확인: Prisma PostgreSQL connector는 Prisma `Json`을 PostgreSQL `jsonb`로 매핑한다. `AuditLog.before/after`는 JSON으로 유지하고 UI 표시에서만 pretty formatting한다. [External: `https://docs.prisma.io/docs/v6/orm/overview/databases/postgresql`]
