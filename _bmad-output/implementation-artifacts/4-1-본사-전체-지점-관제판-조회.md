# Story 4.1: 본사 전체 지점 관제판 조회

Status: done

## Story

As a 본사 사용자,
I want 오늘/어제 기준 전체 지점 장부 상태를 한 화면에서 보고 싶다,
so that 아침 회의 전에 문제 가능성이 있는 지점을 빠르게 찾을 수 있다.

## Acceptance Criteria

1. **권한 있는 본사 관제판 조회**
   - **Given** 본사 사용자가 관제판에 접근할 때
   - **When** 권한 있는 세션으로 오늘/어제 기준 관제판을 연다
   - **Then** 지점별 영업 상태, 장부 상태, 본사 마감 여부, 최신 반영 시각, 마지막 수정자, 기본 요약이 표시되어야 한다
   - **And** 지점 행 전체 클릭 시 해당 장부 상세 페이지로 이동해야 한다.

2. **조회 전용 본사 권한 제한**
   - **Given** 조회 전용 본사 권한 사용자가 관제판을 볼 때
   - **When** 서버가 관제판 데이터를 반환한다
   - **Then** 허용된 지점과 컬럼만 포함해야 한다
   - **And** 저장/마감/정정 같은 mutation action은 수행할 수 없어야 한다.

3. **기본 조회 성능 경계**
   - **Given** 본사 관제판이 기본 조회를 수행할 때
   - **When** 활성 지점이 10개 내외인 경우
   - **Then** 지점 목록과 기본 상태는 3초 안에 표시되는 것을 목표로 해야 한다
   - **And** 무거운 감사 내역이나 리포트 drilldown은 별도 페이지나 패널로 분리해야 한다.

4. **지연 로딩 상태**
   - **Given** 관제판 데이터 로딩이 3초를 초과할 때
   - **When** 화면이 대기 상태를 보여준다
   - **Then** 부분 로드, 마지막 갱신 시각, 재시도 액션을 제공해야 한다
   - **And** 사용자가 빈 화면을 오류 없는 정상 상태로 오해하면 안 된다.

5. **지점장 접근 차단**
   - **Given** 지점장 권한 사용자가 본사 관제판 URL에 접근할 때
   - **When** 서버가 요청을 처리한다
   - **Then** 접근은 거부되어야 한다
   - **And** 전체 지점 목록 또는 타 지점 데이터는 반환되지 않아야 한다.

6. **숫자 표시와 상태 접근성**
   - **Given** 관제판 행에 숫자 컬럼이 표시될 때
   - **When** UI가 금액이나 수량을 렌더링한다
   - **Then** 숫자 컬럼은 `tabular-nums` 또는 동등한 고정폭 숫자 표시를 사용해야 한다
   - **And** 상태 배지는 텍스트만으로도 의미를 알 수 있어야 한다.

## Dev Notes

### 핵심 구현 방향

- 이 story는 본사 아침 회의용 **조회 관제판**이다. 새 마감, 정정, 보완 수정 기능을 여기서 구현하지 않는다. 해당 mutation은 Story 4.3~4.6 범위다.
- 기존 `src/features/dashboard` 구현을 확장한다. 새 `dashboard-v2`, 별도 리포트 query, UI 전용 계산 layer를 만들지 않는다.
- 본사 관제판은 `REPORT_VIEW` 권한과 본사 store scope가 전제다. 조회 전용 본사는 볼 수 있지만 저장/마감/정정 action은 수행할 수 없어야 한다.
- `미입력` row는 반드시 조회 결과에 남겨야 한다. 단, 관제판 조회 query가 장부를 생성하면 read-only 권한과 성능 경계를 깨므로 금지한다.

### 기술 스택과 버전 기준

- `package.json` 기준 현재 repo는 Next `^15.2.3`, React `^19.0.0`, Prisma `^6.6.0`, next-auth `5.0.0-beta.25`, Zod `^3.24.2`, Playwright `^1.60.0`, Tailwind `^4.0.15`, shadcn `^4.8.2`, pnpm `10.31.0`를 사용한다.
- 이 story에서 framework/library major upgrade를 하지 않는다. 구현은 현재 Next App Router, Server Components, Server Actions/queries, Prisma, shadcn/ui 패턴에 맞춘다.
- 외부 최신 API 지식이 필요한 작업은 없다. 관제판은 현재 repo의 서버 계산, 권한, UI 컴포넌트 계약을 따르는 작업이다.

### 데이터와 계산 가드레일

- Prisma `DailyLedger`는 `storeId + closingDate` unique 제약을 갖고, `status`, `version`, `authorDisplayName`, 금액/근무/마감/정정 관련 필드를 보유한다.
- 금액은 integer KRW 기준이다. 화면에서만 `Intl.NumberFormat("ko-KR", { style: "currency", currency: "KRW" })` 또는 기존 formatter를 사용한다.
- `calculateLedgerReviewSummary`가 반환하는 `data-insufficient`, `policy-unconfirmed`, `calculation-unavailable` 상태를 0원이나 빈 문자열로 숨기지 않는다.
- Story 3.4의 policy gate를 우회하지 않는다. OQ-1 전에는 매출차액을 임계값 초과/확정 이상으로 표현하지 않는다. FIFO 원가, `30%단가`, 희망 판매가 손실액, 지점장 민감 파생 지표 노출은 승인 전 구현하지 않는다.

### 권한 및 보안 가드레일

- 관제판 query는 본사 사용자 + `REPORT_VIEW` permission + store scope가 모두 충족되어야 한다.
- 지점장 direct URL 접근은 `/app/unauthorized` 또는 동등한 서버 차단으로 끝나야 하며, 전체 지점 목록이나 타 지점 데이터가 응답에 포함되면 안 된다.
- 조회 전용 본사는 허용 지점과 허용 컬럼만 조회한다. 수정/마감/정정 버튼 또는 form action을 활성화하지 않는다.
- store scope가 `ASSIGNED_STORES`인 본사 사용자는 배정 지점만 봐야 한다. `ALL_STORES` 프로필과 혼동하지 않는다.

### UX / 접근성 가드레일

- 관제판은 본사 desktop 우선, `max-w-[1600px]` shell 기준이다. 단, 모바일 card view도 읽을 수 있어야 한다.
- 상태 배지는 5종: `미입력`, `입력중`, `검토대기`, `본사마감`, `휴무`를 텍스트 pill로 표시한다.
- 이상 신호는 `SignalChip` 패턴을 따른다. 색상 단독 전달은 금지하고, 아이콘/레이블/툴팁 또는 detail text를 제공한다.
- 숫자 데이터는 `tabular-nums`를 적용한다. 특히 매출, 매출 차이, 마진율, 손실 금액/수량, 최신 시각이 table/card에서 흔들리지 않아야 한다.
- 모달 스택은 이 story 범위에 필요 없다. 관제판의 row click과 상세 link가 주 이동 방식이다.
