# Story 3.2: 장부 단계별 입력 검증

Status: done

## Story

As a 지점장,
I want 각 입력 단계에서 잘못된 값과 누락을 바로 확인할 수 있기를 원한다,
so that 제출 전에 장부 오류를 줄일 수 있다.

## Acceptance Criteria

1. **단계별 필수 입력 누락 차단**
   - **Given** 사용자가 장부 입력 단계의 값을 저장할 때
   - **When** 필수 입력값이 누락되어 있다
   - **Then** 서버는 저장 또는 제출을 거부해야 한다
   - **And** 응답은 어느 단계의 어느 필드가 문제인지 알려야 한다.

2. **숫자 필드 검증**
   - **Given** 사용자가 금액, 수량, 인원 같은 숫자 필드를 입력할 때
   - **When** 음수, 문자, 허용 범위 밖 값 등 잘못된 값이 들어온다
   - **Then** 서버 검증은 요청을 거부해야 한다
   - **And** UI는 해당 필드에 연결된 오류 메시지를 보여줘야 한다.

3. **오류 포커스와 접근성**
   - **Given** 폼 오류가 화면에 표시될 때
   - **When** 사용자가 제출을 시도한다
   - **Then** 첫 번째 오류 필드로 포커스가 이동해야 한다
   - **And** 오류 메시지는 `aria-describedby` 또는 동등한 접근성 연결을 가져야 한다.

4. **권한 우선 검증**
   - **Given** 지점장이 권한 밖 장부 값을 저장하려 할 때
   - **When** 서버가 요청을 검증한다
   - **Then** 권한 오류는 입력값 검증보다 먼저 처리되어야 한다
   - **And** 권한 밖 장부 데이터는 응답에 포함되지 않아야 한다.

5. **검증 오류와 계산 상태 분리**
   - **Given** 검증 오류와 계산 상태가 동시에 존재할 때
   - **When** UI가 검토 정보를 표시한다
   - **Then** 필수 오류, `기준 확인 필요`, `데이터 부족`, `계산 불가`는 구분되어야 한다
   - **And** 한 상태가 다른 상태를 덮어 숨기면 안 된다.

6. **스키마 동시 갱신**
   - **Given** 개발자가 검증 스키마를 구현할 때
   - **When** 새로운 장부 입력 필드를 추가한다
   - **Then** Zod 또는 동등한 서버 검증 스키마를 함께 갱신해야 한다
   - **And** 클라이언트 검증만으로 저장을 허용하면 안 된다.

## Dev Notes

### 핵심 구현 방향

- 이 story는 새 장부 단계를 추가하는 작업이 아니다. 기존 7단계 입력 흐름의 서버 검증, 오류 shape, UI focus/accessibility, submit-time 필수 검증을 일관되게 만드는 작업이다.
- 검증은 서버 action의 Zod schema가 권위다. 클라이언트 포맷터와 입력 mask는 사용성 보조일 뿐 저장 허용 기준이 아니다. [Source: `_bmad-output/planning-artifacts/architecture.md` "Validation Strategy", "Validation Pattern"]
- `ActionResult`는 이미 `fieldErrors`와 충돌 payload를 지원한다. 새 action별 오류 shape를 만들지 말고 기존 `actionError` 계약을 재사용한다. [Source: `src/lib/action-result.ts`]
- 금액/수량 검증을 UI 계산과 섞지 않는다. 계산 상태는 Story 3.1의 서버 계산 결과이며, 입력 검증 실패와 별개로 표시되어야 한다. [Source: `_bmad-output/implementation-artifacts/3-1-기본-계산-모듈과-상태-표현.md` "상태 모델 권장안"]

### 현재 코드 상태

- `src/features/ledger/schemas.ts`에는 매출/결제, 비용, 매입, 근무, 제출 schema가 있고 integer KRW 파서와 `toFieldErrors`가 있다.
- `src/features/inventory/schemas.ts`와 `src/features/losses/schemas.ts`에도 별도 숫자 파서와 `toFieldErrors`가 있다. 세 파일의 정책이 드리프트하면 같은 숫자가 단계마다 다르게 통과할 수 있다.
- `src/features/ledger/actions.ts`는 `parseLedger*Input` 후 `requireStoreAccess`를 호출한다. AC 4 때문에 상세 필드 검증보다 권한 확인이 먼저 필요한 권한 밖 요청을 따로 처리해야 한다.
- `src/features/ledger/review-validation.ts`는 제출 전 누락 항목을 `sales`, `expenses`, `purchases`, `inventory`, `work`로 만든다. 손실은 "없음으로 검토 가능" 상태이므로 필수 누락으로 막지 않는다.
- 현재 단계 UI들은 대부분 `fieldErrors`를 받고 `focusFirstError`를 구현한다. 특히 동적 행이 있는 비용/매입/재고/손실은 field path와 DOM ref 배열/맵이 어긋나면 포커스가 실패한다.

### 반드시 보존할 동작

- 권한은 서버에서 강제한다. UI 숨김이나 클라이언트 라우팅만으로 권한을 처리하지 않는다. [Source: `_bmad-output/planning-artifacts/architecture.md` "Authorization Pattern"]
- 지점장 응답은 민감 회계 필드를 서버 response shaping에서 차단한다. 이 story에서 검증 오류를 상세화하면서 원가, 이익, 마진율, 재고금액, lot 근거를 노출하지 않는다. [Source: `_bmad-output/planning-artifacts/architecture.md` "Authentication & Security"; `src/features/ledger/response-shaping.ts`]
- Story 2.9의 optimistic concurrency/version/edit token과 `LEDGER_CONFLICT` 흐름을 유지한다. 검증 실패와 충돌 실패를 같은 오류로 합치지 않는다. [Source: `_bmad-output/implementation-artifacts/3-1-기본-계산-모듈과-상태-표현.md` "Previous Story Intelligence"]
- 본사 마감 또는 휴무 장부의 원본 수정 차단을 유지한다. 검증 스키마가 통과해도 closed/holiday 상태면 원본 저장은 허용되지 않는다. [Source: `src/features/ledger/actions.ts`, `src/features/inventory/actions.ts`, `src/features/losses/actions.ts`]
- 제출 전 필수 누락은 저장 중 임시 누락과 다를 수 있다. 예: 근무인원 null은 저장 가능하지만 제출은 차단한다.

### 테스트 기준

- Unit test는 각 schema의 `safeParse` 실패 결과와 `toFieldErrors` 경로를 검증한다.
- Static regression test는 action들이 `requireStoreAccess`를 상세 단계 schema 검증보다 먼저 수행하는 구조를 확인해야 한다. 정적 테스트가 취약하면 helper 함수 단위로 실행 가능한 테스트를 만든다.
- UI 정적 테스트는 모든 오류 표시가 `FieldError id`, 입력의 `aria-describedby`, 첫 오류 focus helper에 연결되는지 확인한다.
- E2E가 실행 가능한 환경이면 각 단계에서 잘못된 값 저장 시 필드 오류와 focus 이동을 확인한다. 이 저장소의 이전 3.1 기록처럼 샌드박스 포트 바인딩 제한이 있으면 e2e 불가 사유를 story Dev Agent Record에 남긴다.

### 최신 기술 정보

- `package.json` 기준 현재 repo는 Next `^15.2.3`, React `^19.0.0`, Zod `^3.24.2`, Prisma `^6.6.0`, Node test runner를 사용한다. 이 story에서 프레임워크나 Zod major upgrade를 하지 않는다.
- Next 공식 Forms 문서는 Server Action이 서버에서 실행되며, action 내부에서 인증/인가를 항상 확인하라고 안내한다. 현 repo의 client-action 호출 패턴을 유지하되 이 원칙을 권한 우선 검증에 반영한다. [Source: `https://nextjs.org/docs/app/guides/forms`]
- Next `revalidatePath`는 Server Functions/Route Handlers에서 호출하는 서버 전용 API다. 저장 성공 후 현재처럼 관련 장부/관제/리포트 경로 revalidation을 유지한다. [Source: `https://nextjs.org/docs/app/api-reference/functions/revalidatePath`]
- Zod 공식 오류 문서는 error issue의 `path`와 field error 변환을 제공한다. 현 repo는 Zod 3이므로 `error.flatten().fieldErrors` 또는 기존 `issues` 순회 방식을 유지하고, Zod 4 전용 API로 바꾸지 않는다. [Source: `https://zod.dev/error-formatting`]
- React 공식 문서는 `useRef`로 DOM input에 접근해 `focus()`를 호출하는 패턴을 문서화한다. 현재 form clients의 ref 기반 첫 오류 focus 패턴은 유지 가능한 접근이다. [Source: `https://react.dev/reference/react/useRef`]

### UX / 표시 가드레일

- 폼 오류는 `aria-describedby`로 연결하고 제출 시 첫 오류 필드로 포커스를 이동해야 한다. [Source: `_bmad-output/planning-artifacts/epics.md` "Story 3.2"; `_bmad-output/planning-artifacts/epics.md` "UX-DR24"]
