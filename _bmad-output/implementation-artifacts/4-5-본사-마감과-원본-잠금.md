# Story 4.5: 본사 마감과 원본 잠금

Status: done

## Story

As a 본사 마감 담당자,
I want 검토가 끝난 일일 장부를 본사 마감 처리할 수 있기를 원한다,
so that 원본 장부가 보존되고 이후 운영 숫자는 정정 기록으로만 바뀌게 할 수 있다.

## Acceptance Criteria

1. **본사 마감 성공과 감사 로그**
   - **Given** 권한 있는 본사 사용자가 마감 가능한 장부를 열었을 때
   - **When** ClosePreflight가 통과한 장부를 마감한다
   - **Then** 장부 상태는 `본사마감`으로 변경되어야 한다
   - **And** 마감 실행자, 시각, 대상 장부, 검증 요약이 감사 로그에 남아야 한다.

2. **마감 직전 서버 재검증**
   - **Given** 마감 요청이 들어왔을 때
   - **When** 서버가 요청을 처리한다
   - **Then** 권한, 장부 상태, version/edit token, 필수 검증을 다시 확인해야 한다
   - **And** ClosePreflight만 통과했다는 이유로 재검증을 생략하면 안 된다.

3. **마감 후 원본 수정 차단**
   - **Given** 장부가 본사 마감 상태가 된 뒤
   - **When** 지점장 또는 본사 사용자가 원본 입력값을 직접 수정하려 한다
   - **Then** 원본 수정은 차단되어야 한다
   - **And** 마감 후 오류는 정정 기록 흐름으로 안내되어야 한다.

4. **중복 마감 거부**
   - **Given** 이미 마감된 장부에 대해 다시 마감 요청이 들어올 때
   - **When** 서버가 요청을 처리한다
   - **Then** 중복 마감은 거부되어야 한다
   - **And** 기존 마감 정보는 보존되어야 한다.

5. **마감 상태 상세 표시와 잠금 UI**
   - **Given** 마감 후 장부 상세를 조회할 때
   - **When** 화면이 렌더링된다
   - **Then** 원본값, 마감 상태, 마감자, 마감 시각을 확인할 수 있어야 한다
   - **And** 수정 가능한 입력 UI는 잠긴 상태로 보여야 한다.

6. **권한 없는 마감 차단**
   - **Given** 권한 없는 사용자가 마감을 시도할 때
   - **When** 서버 action이 요청을 처리한다
   - **Then** 요청은 거부되어야 한다
   - **And** 마감 가능성 상세나 민감 데이터는 반환되지 않아야 한다.

7. **개별 마감 예외 사유 처리**
   - **Given** ClosePreflight 결과에 `exception-allowed` 항목만 있고 차단 항목이 없을 때
   - **When** 본사 마감 담당자가 개별 마감을 진행한다
   - **Then** 예외 사유 입력 없이는 마감을 실행할 수 없어야 한다
   - **And** 사유, 예외 항목 요약, 실행자는 감사 로그에 남아야 한다.

## Dev Notes

### 핵심 구현 방향

- Story 4.4가 이미 ClosePreflight server action, `buildHqLedgerClosePreflightInTx()`, close transaction 재검증, conflict dialog, preflight 표 UI를 만들었다. 4.5는 이를 버리고 새 마감 엔진을 만드는 story가 아니다.
- 4.5의 미완성 지점은 PRD §4.5의 개별 마감 예외 계약이다. 현재 `buildHqLedgerClosePreflightInTx()`는 `exceptionAllowedCount > 0`이면 `canClose = false`로 둔다. 4.5에서는 차단 항목은 계속 막고, 사유가 있는 개별 마감만 예외 허용한다.
- 본사 마감은 되돌리는 기능을 포함하지 않는다. 잘못 마감한 장부는 Story 4.6의 정정 기록 흐름으로 처리한다.
- 원본 보존이 최우선이다. 마감 성공 후 원본 `DailyLedger`, `LedgerExpense`, `LedgerPurchaseItem`, `LedgerInventoryItem`, `LedgerInventoryAdjustment`, `LedgerLossItem` 값을 정정 반영 목적으로 직접 수정하면 안 된다.

### 권한 및 보안 가드레일

- `LEDGER_HQ_CLOSE` 없는 사용자는 preflight와 close를 모두 실행할 수 없다.
- `REPORT_VIEW`만 있는 조회 전용 본사는 close button이 숨겨져도 server action 직접 호출을 반드시 차단해야 한다.
- 권한/scope 실패 응답에는 preflight `items`, `summary`, `storeName`, 계산값, missing item detail을 넣지 않는다.
- 지점장에게 본사 closeability 상세가 내려가면 보안 회귀다.
- 본사 마감 성공 후에도 지점장과 본사의 원본 수정 server action은 닫힌 장부를 수정하면 안 된다.

### UX / 접근성 가드레일

- Dialog 안의 ClosePreflight 표는 조건명, 상태, 설명, 필요한 조치를 유지한다.
- `차단`, `사유 필요`, `경고`, `정보` 상태는 색상뿐 아니라 텍스트로 표시한다.
