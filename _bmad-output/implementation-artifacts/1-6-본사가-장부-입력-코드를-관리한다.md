# Story 1.6: 본사가 장부 입력 코드를 관리한다

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 본사 사용자,
I want 결제수단, 비용 항목, 손실 유형 같은 장부 입력 코드를 관리하고 싶다,
so that 지점장과 본사가 일관된 선택지로 장부를 입력할 수 있다.

## Acceptance Criteria

1. Given 본사 사용자가 코드 관리 화면에 접근할 때, when 화면이 열리면, then 코드 그룹 목록에는 결제수단, 비용 항목, 손실 유형이 포함되어야 한다, and 각 코드에는 코드명, 표시 순서, 활성/비활성 상태, 마지막 수정 시각이 표시되어야 한다.
2. Given 본사 사용자가 새 코드를 등록할 때, when 코드 그룹과 코드명을 입력하고 저장하면, then 해당 코드는 활성 상태로 생성되어야 한다, and 장부 입력에서 사용할 수 있는 active 코드 목록에 포함될 수 있어야 한다.
3. Given 본사 사용자가 코드명이나 표시 순서를 수정할 때, when 변경 내용을 저장하면, then 신규 입력 선택지에는 변경된 코드 정보가 반영되어야 한다, and 과거 장부에 이미 저장된 코드 표시와 감사 추적은 보존되어야 한다.
4. Given 본사 사용자가 코드를 비활성 처리할 때, when 비활성 상태로 저장하면, then 해당 코드는 신규 장부 입력 선택지에서 제외될 수 있어야 한다, and 과거 장부 상세와 리포트에서는 기존 코드 의미를 확인할 수 있어야 한다.
5. Given 코드 정보가 변경될 때, when 생성, 수정, 표시 순서 변경, 활성 상태 변경이 완료되면, then actor, timestamp, target, before, after를 포함하는 감사 기록이 남아야 한다.
6. Given 코드 관리 폼에 잘못된 값이 입력될 때, when 저장을 시도하면, then 필드 하단에 한국어 검증 메시지가 표시되어야 한다, and 포커스는 첫 번째 오류 필드로 이동해야 한다.

- 코드 그룹은 운영자가 자유 생성하는 별도 그룹 테이블이 아니라 고정된 세 그룹이다: 결제수단, 비용 항목, 손실 유형.
- 코드는 삭제하지 않는다. 비활성 코드는 신규 장부 입력 선택지에서 제외하지만, 과거 장부 상세/리포트와 audit에서 기존 의미를 확인할 수 있어야 한다.
- 미래 장부 모델은 코드 선택 시 당시의 코드명 또는 표시값을 ledger row에 snapshot으로 저장해야 한다. Story 1.6은 그 ledger 구현을 만들지 않지만, code update가 과거 ledger snapshot을 갱신하는 helper나 cascade성 로직을 만들면 안 된다.
- `displayOrder`는 정렬 힌트다. 같은 표시 순서가 임시로 생겨도 `name asc`, `id asc`를 추가 정렬해 화면 순서를 안정적으로 유지한다.
- 중복 코드명은 같은 그룹 안에서 막는다. 다른 그룹 간 같은 이름은 허용된다.

### Latest Technical Information

- 2026-05-29 확인: Next.js 공식 문서의 최신 표시 버전은 16.2.2지만, 이 저장소는 `next` `^15.2.3`을 사용한다. 이번 스토리 중 Next.js 업그레이드는 하지 않는다. `revalidatePath`는 Server Functions와 Route Handlers에서 호출할 수 있고 Client Components에서는 호출할 수 없으므로 revalidation은 Server Action 안에서 수행한다. [External: `https://nextjs.org/docs/app/api-reference/functions/revalidatePath`]
- 2026-05-29 확인: Prisma transaction은 read/write 작업 묶음이 전체 성공 또는 전체 실패하도록 보장한다. 코드 변경과 audit write는 interactive transaction으로 묶는다. [External: `https://www.prisma.io/docs/orm/prisma-client/queries/transactions`]
- 2026-05-29 확인: Prisma referential action은 관련 record 삭제/수정 시 DB 동작을 정하고 `@relation`에서 정의된다. 코드와 updater 관계는 cascade delete가 아닌 보존 중심으로 설계한다. [External: `https://www.prisma.io/docs/orm/prisma-schema/data-model/relations/referential-actions`]
- 2026-05-29 확인: Zod 공식 문서는 coercion이 입력값을 대상 타입으로 변환한다고 설명한다. 표시 순서 같은 numeric form 값은 빈 문자열/음수/소수/초과값을 한국어 오류로 돌려주도록 명시적으로 변환/검증한다. [External: `https://zod.dev/api#coercion`]
