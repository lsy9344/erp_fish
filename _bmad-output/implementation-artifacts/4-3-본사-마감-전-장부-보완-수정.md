# Story 4.3: 본사 마감 전 장부 보완 수정

Status: done

## Story

As a 권한 있는 본사 사용자,
I want 마감 전 지점 장부의 누락 또는 오류를 직접 보완 수정할 수 있기를 원한다,
so that 아침 회의 전에 장부를 본사 기준으로 정리할 수 있다.

## Acceptance Criteria

1. **권한 있는 본사 직접 수정**
   - **Given** 본사 사용자가 마감 전 장부 상세를 열었을 때
   - **When** 권한 프로파일이 직접 수정 action을 허용한다
   - **Then** 본사는 매출/비용/매입/재고/손실/근무 입력값을 보완 수정할 수 있어야 한다
   - **And** 권한이 없는 본사 계정은 수정 action을 수행할 수 없어야 한다.

2. **본사 수정 감사 로그**
   - **Given** 본사가 장부 값을 수정할 때
   - **When** 저장 요청이 성공한다
   - **Then** 변경 전/후 값, 수정자, 시각, 사유가 감사 로그에 남아야 한다
   - **And** 지점장 입력과 본사 수정은 구분되어야 한다.

3. **지점장 동시 저장 재확인**
   - **Given** 본사가 특정 장부 또는 섹션을 수정 중일 때
   - **When** 지점장이 같은 영역을 저장하려 한다
   - **Then** 서버는 최신 상태 재확인을 요구해야 한다
   - **And** UI는 `본사 수정 중` 또는 동등한 상태를 표시할 수 있어야 한다.

4. **본사 stale token 충돌 처리**
   - **Given** 본사가 오래된 version/edit token으로 저장하려 할 때
   - **When** 서버가 저장 요청을 처리한다
   - **Then** 저장은 거부되어야 한다
   - **And** 서버 최신값과 본사 입력값을 비교할 수 있는 충돌 정보를 제공해야 한다.

5. **본사 마감 후 원본 수정 차단**
   - **Given** 장부가 이미 본사 마감 상태일 때
   - **When** 본사 사용자가 원본 필드를 직접 수정하려 한다
   - **Then** 원본 수정은 차단되어야 한다
   - **And** 마감 후 변경은 정정 기록 흐름으로 안내되어야 한다.

6. **수정 후 계산/검증 최신화**
   - **Given** 보완 수정 후 계산/검증 결과가 바뀔 때
   - **When** 장부 상세 또는 관제판이 다시 조회된다
   - **Then** 승인된 기본 계산과 검증 상태는 최신 값 기준으로 갱신되어야 한다
   - **And** OQ-gated 계산은 여전히 `기준 확인 필요`로 유지되어야 한다.

## Dev Notes

### 핵심 구현 방향

- 이 story는 본사 마감 전 원본 장부를 보완 수정하는 story다. 마감 후 정정 기록 생성은 Story 4.6 범위이며, Story 4.3에서는 마감 장부 원본 수정 차단과 정정 흐름 안내까지만 다룬다.
- 기존 코드에는 HQ 편집 action과 상세 페이지 탭이 이미 있다. 새 병렬 구현을 만들지 말고 현재 경로의 누락을 보강한다.
- 가장 큰 누락 위험은 "본사 수정 사유"다. PRD와 AC는 사유를 감사 로그에 요구하지만, 현재 일반 HQ 원본 수정 action은 손실/재고 조정 외에는 `AuditLog.reason`이 비어 있을 가능성이 있다.
- 구현 완료 선언 전에 각 저장 경로가 권한, store scope, edit token, 닫힌 장부 차단, audit before/after/reason, revalidation을 모두 만족하는지 확인한다.

### 기술 스택과 버전 기준

- `package.json` 기준 repo는 Next `^15.2.3`, React `^19.0.0`, Prisma `^6.6.0`, next-auth `5.0.0-beta.25`, Zod `^3.24.2`, Playwright `^1.60.0`, Tailwind `^4.0.15`, shadcn `^4.8.2`, pnpm `10.31.0`를 사용한다.
- 이 story에서 framework/library major upgrade를 하지 않는다. 현재 Next App Router, Server Actions, Prisma transaction, shadcn/ui, Sonner toast, Playwright 테스트 패턴을 유지한다.
- 외부 최신 API 지식이 필요한 신규 라이브러리 도입은 없다. 기존 repo 패턴만 사용한다.

### 권한 및 보안 가드레일

- 본사 직접 수정은 `PermissionAction.LEDGER_EDIT`이 필요하다. `REPORT_VIEW`만 있는 사용자는 상세 조회는 가능해도 원본 저장 action은 거부되어야 한다.
- 본사 사용자라도 store scope 밖 장부는 `requireHeadquartersStoreScope()`로 차단한다.
- 지점장 경로에서 HQ detail/action shape를 재사용하지 않는다. 지점장에게 본사 전체 관제판 row나 민감 지표, 타 지점 비교, threshold detail을 내려보내면 안 된다.
- 마감 후 원본 수정은 절대 허용하지 않는다. `HEADQUARTERS_CLOSED` 상태에서 수정해야 할 값은 `CorrectionPanel`과 `createCorrectionRecord()`로만 처리한다.

### UX / 접근성 가드레일

- 본사 상세은 내부 ERP 톤을 유지한다. shadcn/ui `Tabs`, `Alert`, `Button`, `Dialog`, `Badge`, `Field`를 우선 사용하고 새 styled primitive를 만들지 않는다.
- 본사 수정 사유 입력은 명확한 label, `aria-describedby`, field error, 첫 오류 focus를 제공한다.
- 닫힌 장부에서는 disabled 컨트롤만 보여주지 말고 "본사 마감된 장부는 원본 항목을 수정할 수 없습니다. 정정 기록을 사용해 주세요." 안내를 유지한다.
